Dead Letter Queue (DLQ)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the role of dead letter queues as a safety net for messages that exhaust all retry attempts. Be able to design DLQ topology, monitoring, alerting, and manual reprocessing workflows.

Definition

Dead Letter Queue (DLQ) is a dedicated queue (or topic) that holds messages that have failed processing after the maximum number of retry attempts. These messages require human intervention, code fixes, or upstream data correction before they can be successfully processed.

Where the DLQ Fits in the Processing Pipeline

Producer ──► Main Queue (orders) │ └─► Consumer Group │ ├─ SUCCESS ──► commit offset, write to DB │ └─ FAILURE │ ├─ Attempt retry (backoff) │ └─ FAIL again? ──► retry again... │ └─ Max retries exhausted │ └─► DLQ (orders-dlq) │ ├─ Store original message ├─ Store error metadata (exception, timestamp, retry count) └─ Trigger alert

What Should the DLQ Contain?

An effective DLQ stores not just the failed message, but enough context to diagnose and reprocess:

Field Purpose
Original message payload The message that failed processing
Error / exception details Stack trace, error code, error message
Failure timestamp When the final failure occurred
Retry count Number of attempts before landing in DLQ
Source topic / queue Which topic the message came from
Consumer group ID Which consumer failed
Partition / offset Original position for potential re-injection
Message headers Trace ID, correlation ID, tenant ID

DLQ Topology Patterns

1. Single Shared DLQ

orders-topic ──► consumer ──┐ payments-topic ──► consumer ─┼──► shared-dlq inventory-topic ──► consumer ┘

Simple to set up. Harder to manage -- no isolation between failure domains, one toxic message can clog alerting for all services.

2. Per-Topic DLQ

orders-topic ──► consumer ──► orders-dlq payments-topic ──► consumer ──► payments-dlq inventory-topic ──► consumer ──► inventory-dlq

Better isolation. Each team manages their own DLQ. Recommended for most production systems.

3. Per-Error-Type DLQ

orders-topic ──► consumer ──┬──► orders-dlq-validation (400, schema mismatch) ├──► orders-dlq-timeout (downstream timeout) └──► orders-dlq-permission (403, auth failure)

Advanced pattern. Routes to different DLQ topics based on error classification, enabling specialized handling per failure type.

Monitoring and Alerting

A DLQ with no monitoring is worse than no DLQ at all. Messages silently accumulate, data silently degrades, and no one knows until a customer reports missing data weeks later.

Essential monitoring:

  • DLQ depth (message count): Alert when depth exceeds a threshold (e.g., > 0 for critical topics, > 100 for non-critical).
  • DLQ ingestion rate: Alert when new messages land at a rate exceeding N/minute.
  • DLQ age: Alert when oldest message in DLQ exceeds SLA (e.g., 1 hour for P1, 24 hours for P2).
  • Error type distribution: Dashboard showing breakdown by error code helps identify systemic issues.
  • DLQ consumer health: Ensure the DLQ itself has active consumers/monitors.

Reprocessing Workflow

DLQ (orders-dlq) │ ├─ Investigate: inspect message + error metadata │ ├─ Classify failure: │ ├─ Transient (service was down, now recovered) ──► reprocess automatically │ ├─ Data issue (bad payload) ──► fix data, reprocess manually │ ├─ Code bug (consumer crash) ──► fix code, deploy, then reprocess │ └─ Poison message (unrecoverable) ──► discard with audit trail │ └─ Reprocess: ├─ Re-inject to main topic (preferred for Kafka) ├─ Replay via consumer group offset reset └─ Direct API call (for single-message fixes)

Use Cases

System DLQ Use
Kafka Default DLQ suffix: {original-topic}.DLT (Spring Kafka), or custom routing via DeadLetterPublishingRecoverer
AWS SQS Redrive policy: specify source queue, max receive count, and DLQ ARN
RabbitMQ Dead letter exchange (DLX) with routing key, TTL-based expiry to DLX
Azure Service Bus Built-in dead-lettering on max delivery count or TTL expiry
Google Cloud Pub/Sub Dead letter topic configured per subscription with max delivery attempts

Common Interview Questions

  1. What is a dead letter queue and when would you use one?
  2. How would you design monitoring for a DLQ to prevent silent data loss?
  3. A payment processing consumer keeps sending messages to the DLQ. How do you diagnose the issue?
  4. Describe the workflow for reprocessing messages from a DLQ back into the main pipeline.
  5. What metadata should you store in a DLQ message and why?
  6. How do you handle a "poison message" that crashes every consumer that attempts it?
  7. Compare DLQ patterns across Kafka, RabbitMQ, and SQS.

What to Skip

In an interview, avoid:
  • Designing a system without a DLQ. Every at-least-once consumer needs one.
  • Proposing to silently drop failed messages. Even if the message is "not important," you need observability.
  • Ignoring the reprocessing workflow. A DLQ is not a graveyard -- you must explain how messages get rescued.
  • Storing DLQ messages without error context. Without the exception, stack trace, and retry count, diagnosis is nearly impossible.
