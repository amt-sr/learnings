8.3.2 At Least Once

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand the at-least-once delivery guarantee — the most common Kafka default — where messages are never lost but may be delivered multiple times, and how to handle duplicates through idempotent consumers.

Definition

At least once delivery means each message is delivered one or more times. A message is never lost, but it may appear multiple times due to retries on failure. The producer retries until it receives an acknowledgment, and the consumer reprocesses messages if it crashes before committing offsets. This is Kafka's default delivery guarantee.

Guarantee: No data loss, but potential duplicates. The system prioritizes ensuring every message is processed over preventing duplicate processing.

How Duplicates Occur

Producer-Side Duplicate (acks=1 or acks=all, retries enabled): ┌─────────────────────────────────────────────────────────────┐ │ │ │ 1. Producer ──send(msg1)──────► Leader Broker │ │ │ │ │ 2. Leader writes msg1 to log │ │ │ (msg1 is now persisted) │ │ │ │ │ │ 3. Leader ──ACK──────► ✗ Network failure (ACK lost) │ │ │ │ │ 4. Producer: "No ACK received" │ │ │ → RETRY ──send(msg1)──► Leader Broker │ │ │ │ │ 5. Leader: "msg1 already at offset 0, but I'll accept │ │ the retry" (without idempotent producer) │ │ OR with idempotent: "dup detected, silently discard" │ │ │ │ Result without idempotence: msg1 appears TWICE in partition│ └─────────────────────────────────────────────────────────────┘ Consumer-Side Duplicate (commit after processing): ┌─────────────────────────────────────────────────────────────┐ │ 1. Consumer ──poll()──► receives msg1, msg2, msg3 │ │ │ │ 2. Consumer processes msg1 ✓ │ │ 3. Consumer processes msg2 ✓ │ │ 4. Consumer processes msg3 — CRASH! │ │ │ │ 5. Consumer restarts │ │ Last committed offset = 0 (before msg1) │ │ │ │ 6. Consumer re-reads: msg1, msg2, msg3 │ │ → msg1 and msg2 are REPROCESSED (duplicates) │ └─────────────────────────────────────────────────────────────┘

Kafka Default Configuration

Kafka's default settings provide at-least-once semantics:

# Producer defaults → at-least-once
acks=1                          // Wait for leader acknowledgment
retries=Integer.MAX_VALUE       // Retry forever on retriable errors
enable.idempotence=false        // No deduplication on producer side

# Consumer defaults → at-least-once
enable.auto.commit=true         // Auto-commit offsets periodically
// Risk: message processed but not yet committed → reprocessed on crash



Handling Duplicates: Idempotent Consumers



Since at-least-once allows duplicates, consumers must be designed to handle them gracefully. An idempotent consumer produces the same result regardless of how many times it processes the same message.



Common Idempotency Strategies




  
StrategyHow It WorksExample

  
Deduplication tableStore processed message IDs; skip if already processedINSERT INTO processed(msg_id) ... ON CONFLICT IGNORE

  
Natural idempotencyOperation is inherently idempotentUPDATE users SET status='active' WHERE id=123

  
Conditional writeOnly write if state matches expected preconditionUPDATE orders SET status='paid' WHERE id=X AND status='pending'

  
Unique constraintDatabase unique constraint rejects duplicate insertsUnique constraint on (event_id) column

  
Optimistic lockingUse version column; reject write if version mismatchUPDATE ... WHERE version=N






  Interview Tip: "How do you handle duplicates in at-least-once delivery?" is extremely common. The best answer uses a concrete example: "For an order payment system, I would use a conditional update — UPDATE orders SET status='paid' WHERE id=? AND status='pending'. If the row is already 'paid', the update affects 0 rows, and I know it was already processed."

  Idempotent producer (optional but recommended)

  Full pipeline:
  ✓ Producer retries → message never lost
  ✓ Kafka persists → message survives broker failure
  ✓ Consumer re-reads on crash → message never skipped
  ✓ Idempotent consumer → duplicates are harmless




When At Least Once Is the Right Choice



    
  
  • Most business-critical operations: Order processing, payment handling, inventory updates.
    • 
  
  • Financial transactions: Every transaction must be recorded; duplicates are handled via idempotency.
    • 
  
  • Database synchronization: Changes must be propagated; duplicate apply is safe with idempotent operations.
    • 
  
  • Notification delivery: Every notification must be sent; receiving two emails is better than missing one.
    • 





  Industry Standard: At least once is the most widely used delivery guarantee in production Kafka deployments. It is the default for both producers and consumers. The cost (potential duplicates) is managed through idempotent consumers, which is a well-understood pattern.
