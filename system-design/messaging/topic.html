8.2.2 Topic (Kafka Topic)

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand what a Kafka topic is, how it is partitioned for parallelism, how retention works, and how to design topic naming conventions for a production system.

Definition

A topic in Kafka is a named category or feed to which messages (called records or events) are published. Topics are the fundamental unit of organization in Kafka — producers write to topics, and consumers read from topics. Each topic is divided into one or more partitions, which enable parallel processing.

Analogy: A topic is like a table in a database. It has a name (schema), contains records, and is organized for efficient reads and writes. Partitions are like shards — they split the data for parallel access.

Key Properties

Partitions for Parallelism

A topic with N partitions supports up to N consumers reading in parallel (within a consumer group). Each partition is consumed by exactly one consumer in the group. The number of partitions is set at topic creation time and is critical for system design — too few partitions limit throughput; too many increase overhead.

Retention Period

Kafka retains messages for a configurable period (e.g., 7 days) or until a size limit is reached. Unlike traditional queues where messages are deleted after consumption, Kafka messages persist regardless of whether they have been consumed. This enables replay, reprocessing, and multiple independent consumer groups reading the same topic at different rates.

Topic Compaction

With compacted topics, Kafka keeps only the latest value for each key. Earlier records with the same key are removed. This is useful for table-like state (e.g., latest user profile, latest account balance).

Topic Naming Conventions

Interview Tip: When designing a Kafka-based system, topic naming comes up often. Use a consistent, hierarchical naming convention that encodes the domain and event type.
PatternExampleNotes

Choosing the Number of Partitions

  • Throughput target: Total desired throughput / throughput per partition ≈ partition count. A single Kafka partition typically handles 10-100 MB/s depending on hardware.
  • Consumer count: Partition count is the upper bound on parallel consumers per group. If you need 10 consumers processing in parallel, you need at least 10 partitions.
  • Key-based ordering: If you need ordering for a specific key (e.g., all events for user 123 in order), the key must always hash to the same partition. More partitions mean more keys sharing a partition, but also more potential for hot partitions.
  • Operational overhead: More partitions = more open file handles, more leader election overhead, larger recovery time. A reasonable starting point is 6-12 partitions per topic in production.

What to Skip

Do not dive into:
  • Kafka's internal storage format (segment files, index files, time-index) — too implementation-specific for most interviews.
  • Topic-level authorization ACLs and security configuration unless the question focuses on security.
  • KRaft/ZooKeeper topic metadata storage details — focus on the conceptual model.
