Exactly-Once Delivery

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand how exactly-once semantics (EOS) work in distributed messaging systems, particularly Kafka's approach using idempotent producers and transactional consumers. Be able to explain why true exactly-once is hard, where it applies, and how to reason about trade-offs between delivery guarantees.

Definition

Exactly-Once Semantics (EOS) means each message is delivered to the consumer precisely one time -- neither lost nor duplicated. The effect of processing the message is applied exactly once to the downstream state.

In practice, "exactly-once" in Kafka means exactly-once processing: the message may be delivered multiple times over the network, but the combination of producer idempotence, transactional writes, and consumer offset management ensures the outcome is equivalent to processing each message once.

Delivery Guarantee Comparison

Guarantee Message Lost? Duplicates? Throughput Latency Complexity
At-Most-Once Yes (on failure) No Highest Lowest Lowest
At-Least-Once No Yes (on retry) High Medium Medium
Exactly-Once No No Lower (coordination) Higher Highest
Key Insight: At-most-once trades correctness for speed. At-least-once is the default for most systems but requires idempotent consumers. Exactly-once shifts the deduplication burden to the infrastructure layer so application code can be simpler.

How Kafka Achieves Exactly-Once

Kafka's EOS is built on two cooperating mechanisms: idempotent producers and transactions. Together they provide end-to-end exactly-once processing in Kafka Streams.

1. Idempotent Producer

Even a correctly-behaved producer can retry a send if the broker ack is lost (network timeout). Without protection, this produces duplicates on the broker. The idempotent producer prevents this.

Producer (PID=42, SeqNum=0) ──msg──► Broker (Partition 0) ACK lost (timeout) Producer (PID=42, SeqNum=0) ──msg──► Broker (Partition 0) │ ├─ SeqNum 0 already seen for PID 42 └─ Silently deduplicate, return ACK
  • Producer ID (PID): A unique identifier assigned by the broker to each producer instance (epoch-based, refreshed on fencing).
  • Sequence Number: Monotonically increasing per <PID, Partition> pair.
  • Broker-side dedup buffer: Retains the last N sequence numbers per PID per partition. Duplicate sequence numbers are silently dropped.
  • Enable with enable.idempotence=true (also sets acks=all, max.in.flight.requests.per.connection <= 5).
Scope: Idempotent producer guarantees no duplicates per partition for a single producer session. It does not handle cross-partition atomicity or consumer-side read-committed semantics on its own.

2. Transactions (Atomic Writes Across Partitions)

Many use cases require writing to multiple partitions atomically -- for example, a stream processor consuming from one topic and producing to two output topics. Without transactions, a crash mid-write leaves inconsistent state (some partitions updated, others not).

Producer begins Transaction (T1) │ ├─ Write msg to orders-topic (P0) [status: UNCOMMITTED] ├─ Write msg to inventory-topic (P2) [status: UNCOMMITTED] ├─ Write msg to notification-topic (P1) [status: UNCOMMITTED] │ ├─ Consumer offsets committed to __consumer_offsets (as part of T1) │ └─ Producer commits Transaction T1 │ └─ All messages + offsets become visible to read-committed consumers On failure: Producer aborts Transaction T1 └─ All UNCOMMITTED messages are discarded

Key concepts:

  • Transaction Coordinator: A broker-appointed coordinator manages the two-phase commit protocol for each transaction.
  • Transaction Log (__transaction_state): An internal topic storing transaction lifecycle states (Ongoing, PrepareCommit, PrepareAbort, CompleteCommit, CompleteAbort).
  • Transactional ID: Application-level identifier (e.g., "payment-processor-1") that survives producer restarts. Enables fencing -- if a zombie producer with the same TID tries to write, it is rejected.
  • isolation.level=read_committed: Consumers only read messages from committed transactions (plus non-transactional messages for backward compatibility).

3. Exactly-Once in Kafka Streams

Kafka Streams combines idempotent producer + transactions + consumer offset committing into a single atomic unit:

Kafka Streams (instance-1, TID="stream-app-1") │ ├─ consume(topic-A) ──► process() ──► produce(topic-B) │ │ ├─ consume(topic-A) ──► process() ──► produce(topic-C) │ │ └─ commit offsets for topic-A + produce markers for topic-B,C (all within ONE transaction)
  • The stream processor's consumer offsets are committed within the same transaction as the produced output records.
  • If the process crashes, uncommitted outputs are rolled back, and the consumer restarts from the last committed offset -- guaranteeing exactly-once processing.
  • Enable with processing.guarantee=exactly_once_v2 (Kafka 3.0+).

When Exactly-Once Matters (and When It Does Not)

Scenario Recommended Guarantee Rationale
Financial ledger / double-entry bookkeeping Exactly-Once Duplicate transactions have real monetary cost
Event sourcing / CQRS projections Exactly-Once (or at-least-once + idempotent consumer) Duplicates cause projection state drift
Log aggregation / metrics At-Least-Once Mild over-counting is tolerable
Real-time notifications (email, push) At-Most-Once or at-least-once with dedup Sending twice is annoying but not catastrophic
IoT sensor data ingestion At-Most-Once Volume is high, occasional data loss is acceptable

Trade-Offs and Costs

Performance cost: Exactly-once in Kafka imposes roughly 15-30% throughput reduction compared to at-least-once due to transaction coordinator overhead, two-phase commit protocol, and additional round-trips. For latency-sensitive paths at massive scale, at-least-once with idempotent consumers may be more practical.
  • Throughput: Transaction boundaries add commit latency. Batch sizes should be tuned (e.g., transaction.timeout.ms, transaction.max.timeout.ms).
  • Complexity: Requires understanding of producer epochs, zombie fencing, transaction coordinator failover, and read_committed consumer behavior.
  • Operational: Transaction log topic must be healthy. Coordinator failures can stall new transactions until a new coordinator is elected.

Exactly-Once vs. Idempotent Consumer

Alternative approach: Many systems achieve "effectively exactly-once" by using at-least-once delivery plus an idempotent consumer that deduplicates using a natural or synthetic key (e.g., INSERT ... ON CONFLICT DO NOTHING). This avoids transaction coordination overhead but shifts complexity to the consumer and downstream storage.

Common Interview Questions

  1. What does "exactly-once" actually mean in Kafka, and how does it differ from the theoretical definition?
  2. How does Kafka's idempotent producer prevent duplicate messages on retry?
  3. Explain the two-phase commit protocol Kafka uses for transactions.
  4. What is a zombie producer and how does the transactional ID + epoch prevent it?
  5. When would you choose at-least-once with idempotent consumers over Kafka's exactly-once?
  6. What happens to in-flight messages when a producer crashes mid-transaction?
  7. How does isolation.level=read_committed affect consumer behavior and latency?
  8. Design a payment processing pipeline -- when is exactly-once mandatory vs. over-engineered?

What to Skip

In an interview, avoid:
  • Confusing idempotent producer with exactly-once processing -- they are necessary but not sufficient on their own.
  • Claiming true exactly-once over an unreliable network is possible without qualification. It is always about effective exactly-once via deduplication or transactional processing.
  • Going deep into Kafka's __transaction_state topic internal layout unless asked. Focus on the conceptual protocol.
  • Ignoring the performance cost. Interviewers want to see you understand the trade-off.
