8.2.7 Consumer Group

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand Kafka consumer groups as the mechanism for parallel consumption, partition assignment, scaling by adding consumers, and the rebalancing process when consumers join or leave.

Definition

A consumer group is a group of one or more consumers that jointly consume a set of Kafka topics. Each partition within a subscribed topic is assigned to exactly one consumer within the group. Consumer groups enable horizontal scaling — add more consumers to increase processing capacity, up to the number of partitions.

Core Rule: Within a consumer group, each partition is consumed by exactly one consumer. If you have more consumers than partitions, the excess consumers are idle. If you have fewer consumers than partitions, some consumers handle multiple partitions.

Rebalancing

When consumers join or leave a group (or when topics/partitions change), Kafka triggers a rebalance — a process of reassigning partitions among consumers in the group. During rebalancing:

Before: Consumer 3 crashes P0 ──► C1, P1 ──► C2, P2 ──► C3 │ ▼ Rebalance triggered: ┌─────────────────────────────────────────────┐ │ 1. Consumer 3 stops sending heartbeats │ │ 2. Group coordinator detects failure │ │ 3. All consumers stop processing (stop-the- │ │ world for eager rebalance) │ │ 4. Partitions reassigned: │ │ P0 ──► C1, P1 ──► C2, P2 ──► C1 │ │ 5. Consumers resume processing │ └─────────────────────────────────────────────┘ After: P0 ──► C1, P1 ──► C2, P2 ──► C1 (C1 now handles P0 and P2)

Eager vs. Cooperative Rebalancing

AspectEager (default legacy)Cooperative (incremental)
RevocationAll partitions revoked from all consumersOnly moving partitions are revoked
DowntimeProcessing stops for all consumers during rebalanceOnly affected consumers pause briefly
ComplexitySimpleMore complex but better for large groups
Use caseSmall consumer groupsLarge consumer groups, frequent joins/leaves
Interview Tip: Rebalancing is a common pain point in Kafka deployments. Mention that frequent rebalances (due to misconfigured timeouts or unstable consumers) cause processing pauses and increased latency. Solutions: use cooperative sticky assignor, tune heartbeat and session timeouts, and stabilize consumer deployments.

Multiple Consumer Groups

Different consumer groups can independently read from the same topic. Each group maintains its own committed offsets. This enables multiple downstream services to process the same data independently at their own pace.

Topic: "order-events" (3 partitions) Consumer Group A ("payment-processors"): P0 ──► A-1, P1 ──► A-2, P2 ──► A-3 Processes payments for all orders Consumer Group B ("analytics"): P0 ──► B-1, P1 ──► B-1, P2 ──► B-2 Counts and aggregates order metrics Consumer Group C ("fraud-detection"): P0 ──► C-1, P1 ──► C-1, P2 ──► C-1 Screens every order for fraud signals Each group processes all 3 partitions independently Each group has its own committed offsets

Scaling Strategy

Practical Approach:
  1. Start with partition count based on target throughput and max expected consumers.
  2. Add consumers to a group to scale out processing capacity.
  3. If consumers are CPU-bound, one consumer per CPU core per partition is a good starting point.
  4. Monitor consumer lag to determine when more consumers are needed.
  5. Remember: adding consumers beyond the partition count does not help.

What to Skip

Do not dive into:
  • Group coordinator protocol internals (JoinGroup, SyncGroup, Heartbeat request format and binary protocol).
  • Partition assignor algorithms beyond a brief comparison (Range, RoundRobin, Sticky, CooperativeSticky).
  • Static membership protocol details unless asked about advanced consumer group tuning.
