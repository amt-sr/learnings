Retry

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand retry strategies for failed operations in distributed systems, including fixed interval vs. exponential backoff, max retries, retry budgets, and when to stop retrying and escalate to a dead letter queue.

Definition

Retry is the automatic re-execution of a failed operation. In messaging systems, a consumer that fails to process a message (due to a transient error) re-attempts processing before giving up and moving the message to a dead letter queue.

Why Retries Matter

Failures in distributed systems are routine, not exceptional. Network glitches, downstream service unavailability, temporary resource exhaustion, and schema mismatches are all common. A well-designed retry strategy:

  • Recovers from transient failures without human intervention.
  • Provides back pressure to upstream producers when downstream is degraded.
  • Prevents cascading failures by spacing out retry attempts.
  • Guarantees eventual processing or explicit escalation via a dead letter queue.

Retry Strategies

Fixed Interval

Retry at a constant delay after each failure.

Attempt 1: FAIL ──► wait 1s ──► Attempt 2: FAIL ──► wait 1s ──► Attempt 3: FAIL ──► wait 1s ──► FAIL
Problem: Fixed intervals do not adapt to load. If 10,000 messages all fail simultaneously, they all retry at the same moment, causing a thundering herd that can take down the recovery target.

Exponential Backoff

Double the delay after each failure: 1s, 2s, 4s, 8s, 16s...

Attempt 1: FAIL ──► wait 1s ──► Attempt 2: FAIL ──► wait 2s ──► Attempt 3: FAIL │ wait 4s ─────────────┘ │ Attempt 4: FAIL ──► wait 8s ──► Attempt 5

Exponential backoff naturally spaces retries further apart, reducing load on the recovering system. See 8.4.3 Exponential Backoff for full details.

Retry Budget

A per-client or per-service budget that limits total retry attempts over a time window, preventing retry storms.

Retry Budget = 10% of total requests / second Incoming: 1000 req/s ──► Allow 100 retries/s │ ├─ If budget exhausted: new failures go directly to DLQ │ (no retry, fast-fail) └─ Budget refills each second
  • Popularized by Google's Site Reliability Engineering book.
  • Typical budget: 10-20% of primary request capacity.
  • Prevents a degraded downstream from being hammered by retries on top of new traffic.

Retry Parameters

Parameter Description Typical Values
Max Retries Maximum number of retry attempts before giving up 3-5 for user-facing, 5-10 for batch
Initial Delay Delay before the first retry 100ms - 1s
Backoff Multiplier Factor to increase delay each retry 2x (exponential)
Max Delay (cap) Upper bound on backoff delay 10s - 60s
Jitter Random variance added to delay +/- 10-25% of delay
Retry Budget Max retry rate relative to primary traffic 10-20%
Retryable vs. Non-Retryable Classification of error types Network timeout: retry; validation error: no retry

Distinguishing Retryable vs. Non-Retryable Errors

Critical design decision: Not all errors should be retried. Retrying a deterministic failure (invalid data, missing schema, permission denied) wastes resources and delays the inevitable move to the DLQ.
Error Type Retry? Examples
Transient / recoverable Yes Network timeout, connection reset, 503 Service Unavailable, rate limit (429)
Deterministic / permanent No 400 Bad Request, 403 Forbidden, 404 Not Found, deserialization error, schema mismatch
Resource contention Yes (with backoff) Database deadlock, connection pool exhaustion, out-of-memory (if recoverable)
Data-dependent Maybe Unique constraint violation -- retry only if idempotent or root cause is known

Retry in Practice

Kafka Consumer Retry Pattern

Consumer processes message │ ├─ SUCCESS ──► commit offset, continue │ └─ FAILURE │ ├─ Retryable? ──► No ──► send to DLQ immediately │ └─ Yes │ ├─ backoff delay (exponential + jitter) ├─ retry count < max_retries? │ ├─ Yes ──► reprocess │ └─ No ──► send to DLQ, alert │ └─ If retry budget exhausted ──► send to DLQ immediately

AWS SDK Retry Defaults

  • Standard mode: Max 3 retries, exponential backoff (base 2s), jitter.
  • Adaptive mode: Client monitors latency and adjusts retry behavior dynamically based on observed response times.
  • Legacy mode: Max 5 retries, fixed 5s delay between attempts (deprecated).

Common Interview Questions

  1. How would you implement a retry strategy for a message consumer that calls a third-party API?
  2. What is a retry budget and why is it important in large-scale systems?
  3. How do you distinguish between retryable and non-retryable errors?
  4. What happens if all consumers retry simultaneously? How do you prevent thundering herd?
  5. Design a retry policy for a payment processing service -- what parameters would you tune?
  6. How does retry interact with message ordering?

What to Skip

In an interview, avoid:
  • Proposing unlimited retries without a max retry count or timeout.
  • Retrying non-retryable errors (400, 403, 404, validation failures).
  • Ignoring retry storms -- always mention backoff, jitter, or retry budgets.
  • Forgetting to mention the dead letter queue as the final fallback for exhausted retries.
