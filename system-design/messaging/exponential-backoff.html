Exponential Backoff

Difficulty: вӯҗвӯҗвҳҶвҳҶвҳҶ Interview Frequency: вҳ…вҳ…вҳ…вҳ…вҳ…

Goal

Understand exponential backoff with jitter as a fundamental reliability pattern. Be able to explain why naive fixed-interval retries cause thundering herd problems, how exponential backoff mitigates them, and why jitter is essential for correctness at scale.

Definition

Exponential Backoff is a retry strategy where the delay between attempts doubles after each failure (e.g., 1s, 2s, 4s, 8s, 16s...). It gives a failing system progressively more time to recover while bounding total retry duration via a maximum backoff cap.

The Problem: Thundering Herd

When a shared dependency fails and many clients retry simultaneously at the same fixed interval, they create synchronized retry storms that can overwhelm the recovering service.

Exponential Backoff Solution

The Formula

delay = min(base * 2^attempt, max_delay)

Where base is the initial delay (e.g., 1 second), attempt is the retry number (0-indexed), and max_delay is an upper bound (e.g., 60 seconds).

Jitter: Preventing Synchronized Retries

Even with exponential backoff, clients that start at the same time still retry at correlated intervals. Jitter adds randomness to break synchronization.

Without jitter, exponential backoff only helps if clients started failing at different times. If 1,000 consumers fail simultaneously (e.g., downstream DB restarts), they all follow the same 1s, 2s, 4s schedule and still create synchronized waves. Jitter is not optional -- it is required for correctness at scale.

Jitter Strategies

Strategy Formula Behavior
Full Jitter random(0, base * 2^attempt) Widest distribution, lowest average delay. Best for throughput recovery.
Equal Jitter base * 2^attempt / 2 + random(0, base * 2^attempt / 2) Guarantees minimum wait of half the backoff interval. More conservative.
Decorrelated Jitter random(base, prev_delay * 3) Each retry picks a random value between base and 3x the previous delay. Good distribution, no correlated waves.

Max Backoff Cap

Without a cap, delays grow unboundedly: 1s, 2s, 4s, ... 17 minutes, 34 minutes, 68 minutes. After 20 retries, the delay exceeds 12 days. A cap prevents:

  • Excessive latency for long-running recovery scenarios.
  • Outlasting the useful lifetime of the operation (e.g., a user request timeout).
  • Zombie retries that fire hours after the original request was abandoned.
Rule of thumb: Set max backoff to a value that is long enough for the downstream to recover (e.g., a DB failover may take 30-60 seconds) but short enough to not exceed the operation's SLA.

Where Exponential Backoff Is Used

System / Protocol Usage
TCP Congestion control: Karn's algorithm with exponential backoff for retransmission timeout
Wi-Fi (802.11) CSMA/CA: binary exponential backoff for channel contention
AWS SDK Standard retry mode: exponential backoff with full jitter, max 3 retries
gRPC Configurable backoff with jitter for client-side retries (MAX_BACKOFF default: 120s)
Kafka Producer retries: retry.backoff.ms (default 100ms), consumer rebalance backoff: retry.backoff.ms
Ethernet Truncated binary exponential backoff for collision resolution
HTTP/2 Not built-in, but client libraries (OkHttp, HttpClient) implement it

Exponential Backoff vs. Circuit Breaker

Complementary patterns: Exponential backoff handles individual operation retries. Circuit breakers handle service-level degradation. A circuit breaker stops all traffic to a failing service and lets it recover, while exponential backoff spaces out retries for individual requests. In production, both are used together.
  • Exponential backoff: "This specific request failed; I will wait longer before trying again."
  • Circuit breaker: "This service is failing broadly; I will stop sending it traffic for a cooldown period."
  • When the circuit opens, individual retries stop (fast-fail). When the circuit closes (half-open state), exponential backoff governs the probe requests.

What to Skip

In an interview, avoid:
  • Proposing exponential backoff without mentioning jitter. Without jitter, it still fails at scale.
  • Forgetting the max backoff cap. Unbounded backoff is effectively "never retry."
  • Confusing exponential backoff with circuit breakers -- they solve different problems.
  • Ignoring the interaction between backoff and message ordering (long backoff delays block subsequent messages in a partition).
