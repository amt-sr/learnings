8.2.4 Offset

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the concept of Kafka offsets, how consumers track their position in a partition, and the critical difference between committed and current offsets.

Definition

An offset is a unique, monotonically increasing integer assigned to each message within a Kafka partition. It represents the message's position in the partition's append-only log. Offsets are immutable — once assigned, an offset for a given message never changes, even if messages before it are compacted away.

Key Insight: The offset is the consumer's bookmark. It tells the consumer where it left off, enabling it to resume processing after a crash or restart without missing messages or reprocessing too many.

How Offsets Work

Partition 0 (append-only log): ┌───────┬───────┬───────┬───────┬───────┬───────┬───────┐ │ msg1 │ msg2 │ msg3 │ msg4 │ msg5 │ msg6 │ msg7 │ │ off=0 │ off=1 │ off=2 │ off=3 │ off=4 │ off=5 │ off=6 │ └───────┴───────┴───────┴───────┴───────┴───────┴───────┘ ▲ ▲ committed offset current position Consumer State: ┌─────────────────────────────────────────────────────┐ │ Committed Offset: 3 │ │ → Consumer has acknowledged processing up to msg4 │ │ → On restart, resumes reading from offset 4 │ │ │ │ Current Position: 5 │ │ → Consumer is currently processing msg6 │ │ → If it crashes NOW, it resumes from offset 4 │ │ (committed offset), reprocessing msg5 and msg6 │ └─────────────────────────────────────────────────────┘

Committed Offset vs. Current Offset

ConceptDefinitionPurpose
Committed offsetLast offset the consumer has successfully processed and acknowledgedRecovery point after crash/restart
Current positionOffset the consumer is currently reading/processingTracks how far ahead the consumer has read
LagDifference between log end offset and committed offsetMeasures how far behind the consumer is
Critical Distinction: If a consumer crashes after reading msg5 (current position = 5) but before committing, it will restart from the committed offset (3) and reprocess msg4 and msg5. This is the root cause of duplicate processing in at-least-once delivery semantics.

Offset Management

Automatic Commit

Kafka can auto-commit offsets at regular intervals (default: every 5 seconds). This is simple but risky — if the consumer crashes between processing and auto-commit, messages are reprocessed.

Auto-Commit Timeline: ──► process msg3 ──► process msg4 ──► CRASH ──► restart (not committed) (not committed) │ ▼ resumes from offset 0 (last committed offset) Result: msg1, msg2, msg3, msg4 all reprocessed

Manual Commit

The consumer explicitly commits offsets after processing. This gives precise control over what has been acknowledged:

  • Commit after processing: Process the message, then commit. If processing succeeds but commit fails, the message is reprocessed (at-least-once).
  • Commit the next offset before processing: Risky — if processing fails, the message is skipped (at-most-once).
Best Practice: Use manual synchronous commits after successfully processing each batch. For higher throughput, commit after processing a batch of N messages, accepting that a crash may reprocess up to N messages.

Storing Committed Offsets

Committed offsets are stored in a special Kafka topic called __consumer_offsets. Each consumer group has its own entry. Kafka periodically compacts this topic, keeping only the latest committed offset per group-partition pair.

Offset Reset Strategies

When a new consumer group starts or when a committed offset is invalid (e.g., the offset was deleted by retention), Kafka needs to decide where to start reading:

StrategyBehaviorUse Case
earliestStart from the oldest available message in the partitionBackfilling, reprocessing all history
latestStart from the newest message (skip old messages)Real-time consumers that only care about new data
noneThrow an exception if no committed offset existsStrict mode — fail fast if offset is missing

Consumer Lag Monitoring

Consumer lag (the delta between the log end offset and the committed offset) is a critical metric. High lag means the consumer cannot keep up with producers.

Monitoring: Use tools like Kafka Consumer Group Lag Monitor, Burrow, or Grafana dashboards to track lag per partition. Alert when lag exceeds a threshold. Common causes: consumer too slow, consumer crashed, partition reassignment in progress.

What to Skip
