8.2.9 ISR (In-Sync Replicas)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the ISR (In-Sync Replicas) concept in Kafka, how it relates to data durability and availability, and how min.insync.replicas provides a safety net.

Definition

The ISR (In-Sync Replicas) is the set of all replicas for a partition that are fully caught up with the leader. A replica is considered "in-sync" if it has fetched all messages up to the leader's current end-of-log offset within a configured time window. The leader is always part of the ISR.

Key Insight: The ISR is the set of replicas that are trusted to have the latest data. When acks=all, the producer waits for acknowledgments from all ISR members. Only replicas in the ISR are eligible to become leader during leader election. The ISR shrinks and grows dynamically as followers catch up or fall behind.

How ISR Works

Partition 0 (replication factor = 3) ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Broker 1 │ │ Broker 2 │ │ Broker 3 │ │ LEADER │ │ FOLLOWER │ │ FOLLOWER │ │ offset: 100 │ │ offset: 98 │ │ offset: 65 │ └──────┬──────┘ └──────┬──────┘ └──────┬──────┘ │ │ │ │ ISR = {Broker 1, Broker 2} │ │ Broker 3 NOT in ISR (offset 65, │ │ too far behind leader's 100) │ │ │ ▼ │ ┌──────────────────────────────────────────┐│ │ Leader tracks: ││ │ ISR = [Broker 1 (leader), Broker 2] ││ │ ││ │ Broker 3 is catching up: ││ │ offset 65 → 70 → 75 → 80 → 90 → 98 ││ │ ││ │ Once Broker 3 reaches offset 100: ││ │ ISR = [Broker 1 (leader), Broker 2, ││ │ Broker 3] ││ └──────────────────────────────────────────┘│ │ Broker 3 catches up → added back to ISR ────┘

ISR Dynamics

Removing from ISR

A follower is removed from the ISR if it falls behind the leader by more than the configured threshold. The primary mechanism is the replica.lag.time.max.ms configuration (default: 10 seconds). If a follower has not sent a fetch request or has not caught up within this time, it is shrunk out of the ISR.

Adding Back to ISR

When a follower catches up to the leader's log end offset, it is added back into the ISR. This happens automatically as the follower continues to pull messages from the leader.

min.insync.replicas

This topic-level configuration defines the minimum number of ISR replicas (including the leader) that must acknowledge a write for it to be considered successful. It works in conjunction with acks=all:

ConfigValueBehavior
acks=all-Producer waits for all ISR replicas to ack
min.insync.replicas2 (recommended)Require at least 2 ISR replicas to proceed
Critical Interaction: If min.insync.replicas=2 and the ISR shrinks to 1 (only the leader remains), Kafka rejects all writes to that partition. This prevents data loss by refusing to accept writes when there are not enough replicas to guarantee durability.
Normal operation (ISR = 3): min.insync.replicas = 2, acks = all Producer ──write──► Leader ──► F1 ACK, F2 ACK ✓ Write succeeds (3 >= 2 in-sync replicas) One follower falls behind (ISR = 2): ISR = {Leader, F1}, F2 removed Producer ──write──► Leader ──► F1 ACK ✓ Write succeeds (2 >= 2 in-sync replicas) Another follower falls behind (ISR = 1): ISR = {Leader only}, F1 removed Producer ──write──► Leader ✗ Write REJECTED (1 < 2 min.insync.replicas) → Producer receives NotEnoughReplicasException → Partition is unavailable for writes until ISR recovers

Recommended Configuration

# Topic-level configuration for high durability:
acks=all                        # Wait for all ISR replicas
min.insync.replicas=2           # Require at least 2 (leader + 1 follower)
replication.factor=3            # 3 total copies (leader + 2 followers)

# Broker-level:
replica.lag.time.max.ms=10000   # Remove follower from ISR after 10s behind
unclean.leader.election.enable=false  # Never elect a non-ISR replica
Production Best Practice: Set min.insync.replicas=2 with replication.factor=3 and acks=all. This gives you: durability (data written to leader + at least 1 follower before ack), availability (can tolerate 1 broker failure), and safety (writes rejected if ISR drops below 2).
