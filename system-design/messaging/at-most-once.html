8.3.1 At Most Once

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand the at-most-once delivery guarantee — where messages may be lost but are never duplicated — and the scenarios where it is appropriate.

Definition

At most once delivery means each message is delivered zero or one times. A message may be lost (never delivered) but will never be delivered more than once. There are no duplicates. This is the "fire-and-forget" approach — the system does not retry delivery on failure.

Guarantee: No duplicates, but potential data loss. The system prioritizes avoiding duplicate processing over guaranteeing delivery.

How It Works

At Most Once — Producer Side (acks=0): Producer ──send(msg1)──► Broker ──✓ received Producer ──send(msg2)──► Broker ──✗ network error (msg2 LOST) Producer ──send(msg3)──► Broker ──✓ received Producer does not retry → msg2 is gone forever At Most Once — Consumer Side (commit before processing): Consumer ──poll(msg1)──► Broker Consumer ──commit(offset)──► Broker ← commit BEFORE processing Consumer ──process(msg1)──► CRASH! ← msg1 processed? unknown Consumer ──poll(msg2)──► Broker ← resumes from msg2, never retries msg1 Result: msg1 may or may not have been processed, but it is never reprocessed

Two Ways to Achieve At Most Once

Producer-Side: Fire and Forget

The producer sends the message and does not wait for an acknowledgment. If the message is lost in transit or the broker fails before writing, the message is gone.

acks=0  // Fire-and-forget, no acknowledgment
retries=0  // Do not retry

Consumer-Side: Commit Before Processing

The consumer commits the offset immediately after receiving the message, before processing it. If processing fails, the consumer does not reprocess the message because the offset has already advanced past it.

consumer.poll()
consumer.commitSync()   // Commit BEFORE processing
process(record)          // If this fails, offset already committed → message lost

When Is At Most Once Acceptable?

Interview Tip: The interviewer often asks "when is losing a message okay?" Have concrete examples ready. The key criterion: the business can tolerate the loss, or the data is regenerated/repeated frequently enough that a single loss is negligible.
Use CaseWhy At Most Once Is OK
IoT sensor telemetryNext reading arrives in seconds/minutes; one lost reading is negligible
Real-time analytics countersMissing one page view does not materially affect metrics
Log aggregationLosing a few log lines is acceptable; logs are best-effort diagnostics
Heartbeat/ping messagesMissing one heartbeat triggers a timeout check on the next one
Stock ticker pricesPrice updates arrive continuously; missing one tick is acceptable
Social media metricsLosing a few "like" events is tolerable for approximate counts

Trade-offs

AspectAt Most Once
DuplicatesNever — zero duplicates guaranteed
Data lossPossible — messages can be lost
LatencyLowest — no retry or waiting for acks
ThroughputHighest — no backpressure from acks
ComplexitySimplest — no retry logic, no idempotency needed
Consumer idempotencyNot required

What to Skip

Do not dive into:
  • Kafka transaction API internals (that belongs to the exactly-once section).
  • Idempotent producer implementation details — at-most-once does not require idempotency.
  • Specific broker retry policies — the focus is on the guarantee semantics, not implementation.
