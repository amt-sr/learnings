8.2.5 Producer

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand how Kafka producers publish messages, including key-based partitioning, acknowledgment levels, idempotent producers, and batching strategies.

Definition

A Kafka producer is a client application that publishes messages (records) to Kafka topics. The producer determines which partition each message goes to, manages acknowledgment of successful writes, and can batch multiple messages together for efficiency.

Partition Selection

When a producer sends a message, it must decide which partition to route it to:

ScenarioStrategyOrdering
Key is providedhash(key) % num_partitionsAll messages with same key → same partition → ordered
No key providedRound-robin (sticky batching)No ordering guarantee
Custom partitionerUser-defined logicDepends on implementation
Sticky Partitioner: Kafka's default round-robin "sticky" partitioner batches messages to a single partition until the batch is full or a timeout expires, then switches. This improves batching efficiency compared to strict round-robin.

Acknowledgment Levels (acks)

The acks configuration controls how many replicas must acknowledge the write before the producer considers it successful:

SettingBehaviorDurabilityLatency
acks=0Fire-and-forget. Producer does not wait for any acknowledgment.Lowest — data can be lost if broker failsLowest
acks=1Wait for leader to write. Does not wait for follower replication.Moderate — leader may fail before followers replicateModerate
acks=all (or -1)Wait for leader + all ISR replicas to acknowledge.Highest — data survives leader failureHighest
Interview Tip: The "acks=1 vs. acks=all" trade-off is a frequent question. The critical nuance: acks=1 acknowledges when the leader writes to its log, but followers may not have replicated yet. If the leader crashes immediately after ack, the follower (now new leader) may not have the message. acks=all requires the leader and all in-sync replicas to acknowledge, preventing this data loss scenario.

Idempotent Producer

When a producer retries a send (e.g., due to a timeout), the message may have been written but the acknowledgment was lost. Without idempotency, retries cause duplicates.

enable.idempotence=true  (requires acks=all)

With idempotent producer enabled, Kafka assigns each producer a Producer ID (PID) and tracks a sequence number per partition. If the broker detects a duplicate sequence number (same PID + sequence), it silently discards the duplicate.

Without Idempotence: Producer ──send(msg1)──► Broker (writes msg1) ──ack lost──► Producer Producer ──retry(msg1)────────────────────────────► Broker (writes msg1 AGAIN) Result: msg1 appears twice in the partition With Idempotence: Producer ──send(PID=42, seq=0, msg1)──► Broker (writes msg1) ──ack lost──► Producer Producer ──retry(PID=42, seq=0, msg1)────────────────────► Broker Broker: "I already have PID=42, seq=0" → discard duplicate Result: msg1 appears exactly once
Important: Idempotent producer prevents duplicates within a single producer instance. It does not prevent duplicates across different producer instances or after a producer restart (new PID is assigned).

Batching

Producers batch messages to reduce the number of round-trips to the broker:

  • linger.ms: Wait up to N milliseconds to accumulate messages before sending. Default: 0 (send immediately). Increasing to 5-20ms significantly improves throughput.
  • batch.size: Maximum batch size in bytes. Default: 16KB. When the batch is full, it is sent immediately regardless of linger.ms.
  • Compression: Producers can compress batches (gzip, snappy, lz4, zstd). Compression reduces network bandwidth and storage but adds CPU overhead.

Retries + Ordering: If you need strict ordering per key and retries are enabled, set max.in.flight.requests.per.connection=1 (with idempotent producer, you can use up to 5). Otherwise, a retry for message B may succeed before the original send of message A, violating ordering.

What to Skip

Do not dive into:
  • Producer buffer pool internals (RecordAccumulator, ByteBuffer allocation).
  • Network client thread model details (Selector, NIO internals).
  • Transaction coordinator internals unless specifically asked about Kafka transactions.
