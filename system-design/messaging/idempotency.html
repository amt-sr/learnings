Idempotency

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand idempotency as a core pattern for building reliable message consumers. Be able to design idempotent consumers using deduplication keys, database constraints, and idempotency key stores, and articulate why idempotency is essential when using at-least-once delivery.

Definition

Idempotency means that executing the same operation multiple times produces the same result as executing it once. An idempotent consumer can safely process a duplicate message without corrupting state or causing side effects (e.g., charging a credit card twice, sending a notification twice, or incrementing a counter twice).

Why Idempotency Matters in Messaging

Producer sends msg A │ ├─ Broker receives msg A, writes to partition │ ├─ Consumer processes msg A → SUCCESS │ └─ ACK lost in network (consumer crash before commit) │ └─ Consumer restarts, reads msg A again (at-least-once) │ ├─ WITHOUT idempotency: duplicate side effect (double charge!) └─ WITH idempotency: detected as duplicate, skipped
Core principle: At-least-once delivery (the default and most common guarantee) implies that duplicates are not just possible but expected. Idempotent consumers are the standard mechanism for handling this -- either via application-level deduplication or infrastructure-level exactly-once semantics.

Idempotency in Practice: Design Patterns

1. Natural Idempotency

Some operations are inherently idempotent by their nature. No additional mechanism needed.

Operation Idempotent? Reason
SET x = 10 Yes Same value regardless of how many times executed
INSERT ... ON CONFLICT DO NOTHING Yes Duplicate insert is a no-op
UPSERT (key, value) Yes Same final state regardless of retries
INCREMENT counter by 1 No Each execution changes the value
CHARGE credit card $50 No Each execution charges the customer again
SEND email No Each execution sends another email

2. Database Unique Constraints

Use a unique constraint on a deduplication key to make non-idempotent operations idempotent.

-- Deduplication table
CREATE TABLE processed_events (
  event_id     VARCHAR(128) PRIMARY KEY,   -- idempotency key
  entity_type  VARCHAR(64),
  entity_id    VARCHAR(128),
  processed_at TIMESTAMP DEFAULT NOW()
);

-- Consumer logic (pseudocode):
INSERT INTO processed_events (event_id)
VALUES ('evt-abc-123')
ON CONFLICT (event_id) DO NOTHING
RETURNING event_id;

-- If no row returned → first time seeing this event → process business logic
-- If row returned (conflict) → duplicate → skip
Why this works: The database's unique constraint provides atomic check-and-set semantics. Two concurrent consumers processing the same message cannot both insert the same event_id -- one succeeds, one gets a constraint violation and skips.

3. Distributed Idempotency Key Store

For systems that cannot rely on a single database, use a distributed store (Redis, DynamoDB, Memcached) to track processed message IDs.

Consumer processes message │ ├─ Extract idempotency key (message ID, dedup header, etc.) │ ├─ CHECK: EXISTS(idempotency_key) in Redis │ ├─ EXISTS → skip (duplicate) │ └─ NOT EXISTS → SET idempotency_key with TTL │ │ │ ├─ SET NX succeeded → process business logic │ └─ SET NX failed (race) → skip (duplicate) │ └─ Business logic: write to DB, call API, etc. 
-- Redis: atomic check-and-set
result = redis.set("dedup:evt-abc-123", "1", NX, EX 86400)
if result == "OK":
    process_message()    # first time
else:
    return               # duplicate, skip

-- DynamoDB: conditional write
dynamodb.put_item(
    table="processed_events",
    item={"event_id": "evt-abc-123", "processed_at": now()},
    condition="attribute_not_exists(event_id)"
)

4. Idempotency Key Design

The choice of idempotency key is critical. It must be unique per logical operation.

Scenario Good Idempotency Key Why
Payment processing payment_id (generated by client) Each payment attempt gets a unique ID; retries use the same ID
Order status update order_id:version Same order, different update versions are distinct operations
User profile update user_id:timestamp or user_id:request_id Distinguishes between retries of the same update vs. new updates
Event sourcing event_id (UUID or ULID) Every event has a globally unique identifier
Kafka message topic:partition:offset Natural position-based key, but changes on re-consumption
Warning: Using topic:partition:offset as the idempotency key works for simple cases but breaks on consumer group rebalancing (offsets reset), topic compaction (offsets change), or reprocessing from a DLQ (new offset). Prefer application-level keys like event_id or request_id.

TTL and Cleanup

Idempotency key stores grow unboundedly. A TTL (time-to-live) policy is essential.

System TTL Mechanism Typical TTL
Redis Key expiry (EX flag) 24 hours - 7 days
DynamoDB TTL attribute on table 24 hours - 7 days
PostgreSQL Background DELETE WHERE processed_at < now() - interval '7 days' 7 - 30 days
Memcached Per-key expiration 24 hours
Risk: If the TTL expires before all retries are complete, a legitimate retry will be treated as a new message and cause a duplicate side effect. Set TTL significantly longer than your maximum retry window (including DLQ reprocessing time).

Idempotency vs. Exactly-Once

Relationship: Kafka's exactly-once semantics (idempotent producer + transactions) prevent duplicates at the broker level. But if the consumer writes to an external system (MySQL, DynamoDB, third-party API), the broker's EOS does not help. You still need consumer-side idempotency for the downstream write. In most real-world systems, both are used: EOS at the broker for internal consistency, and idempotent consumers for external writes.

Common Interview Questions

  1. What is idempotency and why is it essential for at-least-once delivery?
  2. Design an idempotent consumer for a payment processing system. What data structures and mechanisms would you use?
  3. How would you choose an idempotency key for different types of operations?
  4. What is the trade-off between using database unique constraints vs. a distributed key store (Redis) for deduplication?
  5. How do you handle TTL expiry of idempotency keys? What happens if a retry arrives after the key expires?
  6. Design an idempotency solution that works across multiple consumers in a consumer group.
  7. How does Kafka's idempotent producer relate to consumer-side idempotency?
  8. A user clicks "Submit Payment" twice quickly. How does idempotency prevent double charges?

What to Skip

In an interview, avoid:
  • Assuming at-least-once delivery is sufficient without idempotency. It is not -- duplicates are guaranteed eventually.
  • Using Kafka offsets as idempotency keys without acknowledging the limitations (offset reset, compaction, reprocessing).
  • Proposing an idempotency key store without a TTL strategy. The store grows without bound.
  • Confusing producer idempotence (Kafka's PID + sequence number) with consumer idempotency (deduplication of business logic). They solve different problems.
  • Ignoring the race condition: two consumers processing the same message concurrently. Use atomic check-and-set (SET NX, conditional insert) -- never separate read-then-write.
