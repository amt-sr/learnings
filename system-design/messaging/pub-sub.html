8.1.2 Pub/Sub (Publish/Subscribe)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand the publish/subscribe messaging pattern, how it differs from a queue, and when to apply the fan-out pattern in system design.

Definition

Pub/Sub (Publish/Subscribe) is a messaging pattern where a publisher sends a message to a topic (or channel), and all registered subscribers to that topic receive a copy of the message. Unlike a queue where each message is consumed by one consumer, pub/sub delivers each message to every subscriber — this is the fan-out pattern.

Core Idea: Publishers and subscribers are fully decoupled. The publisher has no knowledge of which subscribers exist, how many there are, or whether any are currently listening. Subscribers express interest in a topic and receive all messages published to it.

Architecture

Publisher(s) Topic Subscriber(s) ┌──────────┐ ┌──────────────┐ ┌──────────┐ │ Order Svc ├─publish──►│ │─copy──►│ Inventory│ │ User Svc ├─publish──►│ "orders" │─copy──►│ Analytics│ └──────────┘ │ Topic │─copy──►│ Notif. │ │ │─copy──►│ Search │ └──────────────┘ └──────────┘ • One message published → all subscribers receive a copy • Subscribers are independent — one's failure does not affect others • New subscribers can be added without any publisher changes

Key Characteristics

Fan-Out Delivery

Every message published to a topic is delivered to all active subscribers. This is fundamentally different from a queue's competing-consumer model. Fan-out is what enables event-driven architectures where multiple downstream systems react to the same event independently.

Topic-Based Routing

Messages are categorized by topic. Publishers write to specific topics, and subscribers register interest in specific topics. Topics provide a natural organization boundary — e.g., user.created, order.placed, payment.processed.

Temporal Decoupling

Subscribers do not need to be online when the message is published. Depending on the implementation, subscribers can receive messages from when they were offline (durable subscriptions) or only messages published while actively connected (non-durable).

Pub/Sub vs. Queue

AspectQueuePub/Sub
Delivery modelOne message → one consumerOne message → all subscribers
RelationshipPoint-to-pointFan-out (one-to-many)
Consumer couplingCompeting consumers share workSubscribers are fully independent
Typical use caseTask distribution, work stealingEvent notification, data propagation
OrderingNatural FIFO per queuePer-topic ordering (implementation dependent)
Scaling consumersAdd consumers → faster processingAdd subscribers → more systems react (not faster)
ExamplesSQS, RabbitMQ (direct exchange)SNS, Kafka topics, Google Pub/Sub
Interview Tip: A common follow-up is "can you combine queue and pub/sub?" Answer: yes. AWS SNS (pub/sub) can fan out to multiple SQS queues. This lets each downstream service have its own queue with independent retry/DLQ logic.

SNS + SQS Fan-Out Pattern

Event Producer SNS Topic SQS Queues ┌──────────┐ ┌──────────────┐ ┌──────────┐ │ Order ├─publish──►│ │─send──►│ Email Q │──► Email Worker │ Service │ │ order.placed│─send──►│ Inventory│──► Inventory Worker └──────────┘ └──────────────┘─send──►│ Analytics│──► Analytics Worker └──────────┘ Benefits: • Each subscriber has its own queue → independent retry, DLQ, processing rate • SNS handles fan-out; SQS handles reliable delivery • Adding a new subscriber = add a new SQS subscription (no publisher changes)

Challenges

  • Message explosion: With many subscribers, a single publish results in N deliveries. Monitor throughput and cost.
  • No backpressure: The publisher is unaware of subscriber health. Slow subscribers fall behind. Mitigate with per-subscriber queues.
  • Ordering across subscribers: Each subscriber sees messages in order, but ordering between subscribers is not guaranteed.
  • Exactly-once delivery: Most pub/sub systems provide at-least-once. Subscribers must be idempotent.
