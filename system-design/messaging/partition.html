8.2.3 Partition (Kafka Partition)

Difficulty: вӯҗвӯҗвӯҗвҳҶвҳҶ Interview Frequency: вҳ…вҳ…вҳ…вҳ…вҳ…

Goal

Understand Kafka partitions as the unit of parallelism and ordering, how key-based partitioning works, and the implications for consumer scalability.

Definition

A partition is an ordered, append-only sequence of records within a Kafka topic. Each topic is divided into one or more partitions. Messages within a single partition are strictly ordered by their offset. Partitions are distributed across Kafka brokers, enabling parallel read and write operations.

Key Insight: The partition is the fundamental trade-off point in Kafka. It gives you parallelism (more partitions = more concurrent consumers) but limits ordering (ordering is only guaranteed within a partition, not across partitions).

Key-Based Partition Assignment

When a producer sends a message with a key, Kafka uses a hash of the key to determine which partition the message goes to:

partition = hash(key) % num_partitions

This ensures that all messages with the same key always go to the same partition, preserving ordering for that key across the entire topic.

Practical Example: If you publish order events keyed by order_id, all events for the same order will appear in the same partition in order. Event A (order created) will always be before Event B (order paid) for the same order вҖ” even though other orders' events are interleaved.

Interview Tip: A common follow-up: "How would you pick the number of partitions?" Answer: start with a number based on target throughput (partition throughput / desired throughput) and the maximum number of consumers you need. You can increase partitions later, but you cannot decrease them without recreating the topic.

Partition and Ordering: Critical Nuance

  • Within a partition: Messages are ordered by offset. If producer sends m1 then m2 to the same partition, every consumer sees m1 before m2.
  • Across partitions: No ordering guarantee. Message m1 in partition 0 and message m2 in partition 1 have no defined relative order.
  • Key-based ordering: Use the same key for all messages that need relative ordering. The hash ensures they land in the same partition.
  • No key: Messages are distributed round-robin across partitions. No ordering guarantee at all.

Partition Reassignment

Kafka supports reassigning partitions between brokers (e.g., when adding a new broker to the cluster). This is done via partition reassignment commands and moves data between brokers. It does not change partition count.
