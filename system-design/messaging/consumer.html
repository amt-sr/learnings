8.2.6 Consumer

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand how Kafka consumers read messages, the pull vs. push delivery model, the role of consumer groups, and offset commit strategies.

Definition

A Kafka consumer is a client application that reads messages from one or more Kafka topics. Consumers subscribe to topics and pull messages from partitions. Each consumer tracks its position (offset) within each partition it reads from, enabling it to resume after failure.

Pull vs. Push Model

Push Model (RabbitMQ): Broker ──push──► Consumer Broker decides when and how fast to push messages. Risk: Overwhelms slow consumer → consumer buffers overflow → messages dropped. Pull Model (Kafka): Consumer ──pull──► Broker Consumer decides when to fetch and how many messages to pull. Consumer pulls at its own pace → never overwhelmed. Trade-off: If no data available, pull results in empty response (polling overhead). Mitigation: Long polling — broker waits until data is available or timeout.
AspectPush (RabbitMQ)Pull (Kafka)
Flow controlBroker pushes — can overwhelm consumerConsumer pulls — self-paced
Idle consumersNo overhead (broker waits)Polling overhead (empty pulls)
BackpressureConsumer must signal (prefetch count)Natural — consumer pulls only when ready
ThroughputGood for low-latency scenariosExcellent for high-throughput batching
Interview Tip: "Why does Kafka use a pull model?" is a common question. The answer: it gives consumers control over their own processing rate, avoids overwhelming slow consumers, and enables efficient batching. Kafka mitigates the polling overhead with long polling and consumer group coordination.

Consumer Groups

Consumers are organized into consumer groups. Each consumer in a group is assigned one or more partitions. Within a group, each partition is consumed by exactly one consumer. This is the competing-consumer pattern — partitioned across consumers.

Key Rule: If a consumer group has more consumers than partitions, some consumers will be idle. The partition count is the upper bound on parallelism within a group. Consumer groups are covered in depth in the 8.2.7 Consumer Group section.

Offset Commit Strategies

Automatic Commit

enable.auto.commit = true  (default)
auto.commit.interval.ms = 5000  (default: 5 seconds)

Offsets are committed periodically in the background. Simple but can cause message loss or duplicate processing on failure.

Manual Synchronous Commit

enable.auto.commit = false
// After processing:
consumer.commitSync();

Explicitly commit after processing. Blocks until the broker acknowledges. Safest option but adds latency per commit.

Manual Asynchronous Commit

enable.auto.commit = false
// After processing:
consumer.commitAsync(callback);

Fire-and-forget commit. Does not block. Risk: commit may fail and you will not know before the next poll.

Best Practice: Use commitAsync for normal processing (low latency) and commitSync in a finally block or on close to guarantee the last batch is committed before shutdown.

Processing Loop

while (true) { ConsumerRecords records = consumer.poll(Duration.ofMillis(1000)); for (ConsumerRecord record : records) { process(record); // 1. Deserialize and process the message trackProgress(record); // 2. Update local state } consumer.commitSync(); // 3. Commit offsets after batch // (or commitAsync for lower latency) }

Common Failure Modes

  • Consumer too slow: Processing takes longer than max.poll.interval.ms → consumer kicked out of group → rebalance triggered.
  • Commit before processing: If processing fails after commit, messages are lost (at-most-once).
  • Process before commit, crash before commit: Messages reprocessed on restart (at-least-once, duplicates possible).
  • Poison message: A message that consistently causes consumer to crash → consumer stuck in crash-loop → falls out of group. Need DLQ pattern.
