URL Shortener

Difficulty: ⭐⭐ Interview Frequency: ★★★★★

Goal

Design a service that takes a long URL and returns a short, unique alias. When a user visits the short URL, they are redirected to the original long URL. The system must handle high read throughput for redirections and support analytics on click data.

Functional Requirements

  • Shorten a given long URL to a compact alias (e.g., https://sho.rt/abc123)
  • Redirect the short URL to the original long URL via HTTP 301/302
  • Support optional custom aliases (e.g., mybrand)
  • Provide click analytics: timestamp, IP, referrer, browser, location
  • Support link expiration and deactivation
  • Provide API access for programmatic shortening

Non-Functional Requirements

  • High availability: redirection must be highly available (99.99%+ uptime)
  • Low latency: redirect latency < 50ms for reads
  • Scalability: handle 100M+ URLs stored, 10B+ clicks/month
  • Predictable URLs: short URLs should be unguessable to prevent enumeration
  • Durability: no data loss for stored mappings

Deep Dive: Short Code Generation

Approach 1: Counter + Base62 Encoding

Use a distributed counter (Snowflake-like or database sequence) to generate a unique 64-bit integer, then encode it in base62 (0-9, a-z, A-Z). A 7-character base62 string supports 62^7 = 3.5 trillion combinations.

Base62 encoding function:
Given counter ID = 125,000,000: repeatedly divide by 62 and map remainders to charset 0-9a-zA-Z. Result: 8M0k (padded to desired length).

Approach 2: Hash + Base62 (MD5/SHA-256)

Hash the long URL, take first N bytes, encode in base62. Risk of collisions — must check DB and retry with a different prefix if collision occurs.

ApproachProsCons
Counter + Base62Guaranteed unique, simplePredictable (sequential), counter coordination needed
Hash + Base62No shared counter neededCollision risk, collision handling complexity
Pre-generated random IDsUnguessable, no runtime generationMust pre-generate and store pool, coordination
Recommended: Pre-generate unique IDs using a KGS (Key Generation Service) that batches random IDs from a distributed ID generator (Snowflake/UUID v7) and stores them in a database. Workers fetch batches from the KGS to avoid hot ID generation at scale.

Deep Dive: Redirection Flow

  1. Client requests GET /{short_code}
  2. Read service checks Redis cache for short_code → long_url mapping
  3. Cache hit: return HTTP 302 with Location header immediately
  4. Cache miss: query PostgreSQL, populate cache, return 302
  5. Fire-and-forget analytics event to Kafka (click metadata)
301 vs 302: Use 301 for permanent redirects (browser caches the redirect, reducing server load, but analytics lose clicks). Use 302 for temporary redirects (every click hits the server, better analytics). For a production shortener with analytics requirements, use 302.

Scalability Considerations

  • Cache layer: Redis cluster with ~99% cache hit rate for reads. Cache hot URLs aggressively. Use LRU eviction. TTL of 24h for cache entries.
  • Database sharding: Shard by short_code hash across PostgreSQL instances when single node cannot handle the write load. 100M URLs with metadata is ~50-100GB — fits on a single large instance initially.
  • Analytics pipeline: Decouple click analytics from the redirect path. Write to Kafka, consume with stream processors into ClickHouse for fast OLAP queries. Do not block redirects on analytics writes.
  • Rate limiting: Per-user rate limits on URL creation to prevent abuse. Token bucket algorithm via Redis.
  • Security: Validate long URLs to prevent redirects to malicious sites. Block known phishing/malware domains. URL scanning service for enterprise plans.

Common Interview Questions

  • How would you prevent enumeration of short codes? A: Use random pre-generated IDs with sufficient entropy (7 chars base62 = 62^7 space). Add rate limiting on lookups.
  • How do you handle collisions with hash-based approaches? A: Double-write: check DB before insert. On collision, append random suffix or retry with different hash prefix.
  • What happens when the cache goes down? A: Reads fall through to DB. Deploy Redis in primary-replica or cluster mode for HA. Circuit breaker prevents cascade failures.
  • How would you support custom aliases? A: Add uniqueness check on alias column. Reserve namespace prefixes for premium users. Rate limit custom alias creation.
  • How do you scale the write path? A: Batch inserts, use write-through cache, consider Cassandra for unlimited horizontal scaling at the cost of consistency.
  • How would you detect and block malicious URLs? A: URL reputation service, Google Safe Browsing API integration, content-type checking on redirect, CAPTCHA for suspicious patterns.
  • Why not use a NoSQL database? A: You could — Cassandra or DynamoDB would work well. The relational model is simpler for the metadata and single-key lookups. The choice depends on team expertise and operational requirements.
What to Skip:
  • Don't over-engineer the analytics pipeline initially — mention it as a separate concern
  • Don't spend time on URL parsing or validation edge cases unless asked
  • Don't design user authentication from scratch — assume an existing auth service
  • Don't get stuck on base conversion math — explain the concept and move on
