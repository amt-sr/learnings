Chat System

Difficulty: вӯҗвӯҗвӯҗвӯҗ Interview Frequency: вҳ…вҳ…вҳ…вҳ…

Goal

Design a real-time messaging system similar to WhatsApp or Slack. The system must support one-on-one and group chats, message history, online status, read receipts, and message delivery guarantees with low latency across millions of concurrent connections.

Functional Requirements

  • Send and receive messages in real-time (1:1 and group chats)
  • View message history with pagination
  • Display online/offline status of users
  • Support read receipts (message seen by recipient)
  • Support message types: text, image, video, file, emoji reactions
  • Search message history
  • Message delivery guarantees (at-least-once, no message loss)

Non-Functional Requirements

  • Real-time: message delivery < 500ms p99
  • High concurrency: 5M+ simultaneous WebSocket connections
  • Scalability: horizontal scaling of WebSocket servers
  • Reliability: at-least-once delivery with message ordering
  • Storage: persistent message storage with efficient retrieval

Deep Dive: Message Delivery Flow

  1. Send: Client sends message via WebSocket to its connected server.
  2. Validate & Persist: WS server validates, assigns a message ID (Snowflake), writes to the message store (Cassandra), and publishes to Kafka.
  3. Fan-out: A fan-out service consumes from Kafka. For each recipient, looks up their connected WS server via the presence service, and publishes to that server's Redis channel.
  4. Deliver: The target WS server receives the message from Redis Pub/Sub and pushes it to the client via WebSocket.
  5. Acknowledge: Client sends an ack back via WebSocket. If no ack within timeout, the message is marked as undelivered for later retrieval.
Message ordering: Use a per-conversation sequence number (monotonically increasing) to maintain message order. Each message is assigned a sequence ID within its conversation. Clients request messages with sequence numbers greater than their last seen. This decouples ordering from the distributed system's clock.

Deep Dive: Presence Service

Track which users are online and which WebSocket server they are connected to.

# Redis presence data
# Map: user_id вҶ’ {server_id, last_heartbeat}
HSET presence:user:12345 server_id "ws_server_3" last_heartbeat 1701234567

# Presence TTL: 30 seconds (renewed by heartbeat)
EXPIRE presence:user:12345 30

# Online friends of user 12345
SUNION online_friends:user:12345
  вҶ’ {456, 789, 1011, ...}

# When user 12345 comes online:
SADD online_friends:user:456 12345
SADD online_friends:user:789 12345
# Fan-out online status to all friends via Redis Pub/Sub
PUBLISH presence:user:12345 '{"type": "online", "user_id": 12345}'
Heartbeat mechanism: Clients send a heartbeat ping every 15 seconds. The server updates the Redis TTL on each heartbeat. If the TTL expires (server crash, network failure), the user is automatically marked offline. This avoids explicit "logout" dependency.

Deep Dive: Message Storage

Use Cassandra for message storage due to its high write throughput and natural partitioning by conversation.

Why Cassandra? Messages are write-heavy (10B/day), need to be partitioned by conversation for efficient retrieval, and have a natural access pattern (time-ordered within a partition). Cassandra handles this well with its wide-row model and tunable consistency.

Deep Dive: Group Chat Fan-out

For group chats with many members, fan-out is expensive. Two strategies:

StrategyDescriptionBest For
Fan-out on deliveryStore one copy per conversation. Deliver to all members when they connect.Small groups (<100 members)
Fan-out on writeWrite one copy per recipient. Each user has their own inbox.Large groups (1000+ members) with selective read
Hybrid approach: For groups < 100 members, fan-out on delivery (one row per conversation). For groups > 100, use fan-out on write (one row per recipient) to avoid reading the full conversation for each user. Threshold is configurable based on read:write ratio.

Common Interview Questions

  • How do you handle message delivery when the recipient is offline? A: Store messages in the recipient's inbox (fan-out on write). When they come online, send all pending messages since their last seen sequence number. Use push notifications for a subset (last message, unread count).
  • How do you prevent message duplication? A: Use idempotent message IDs. Client includes a client-side generated message ID. Server deduplicates using this ID. On reconnect, client sends last acknowledged message ID.
  • How do you handle WebSocket server scaling? A: Stateless WebSocket servers behind a load balancer. Use Redis Pub/Sub for cross-server message routing. Presence service maps user_id to server_id for direct routing.
  • How do you support message search? A: Index messages in Elasticsearch. Use a firehose from Kafka to Elasticsearch. Support full-text search with filters (sender, date range, conversation).
  • How do you handle read receipts at scale? A: Read receipts are separate lightweight events. Batch them (e.g., send a read receipt for the latest message, not every individual message). Store per-user last_read_seq per conversation.
  • What happens during a network partition? A: Messages are buffered in Kafka. Presence may become stale (show online users as offline). Accept eventual consistency for presence. Message delivery resumes when partition heals.
