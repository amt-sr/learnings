Google Docs

Difficulty: ⭐⭐⭐⭐⭐ Interview Frequency: ★★★

Goal

Design a real-time collaborative document editing system where multiple users can edit the same document simultaneously and see each other's changes in real-time. The system must handle conflict resolution, maintain document consistency, support offline editing with reconnection sync, and provide low-latency propagation of changes across all connected clients.

Functional Requirements

  • Real-time collaborative editing: multiple users edit the same document simultaneously
  • Every user sees a consistent view of the document at all times
  • Support text formatting (bold, italic, headings, lists) and rich content (images, tables)
  • Handle conflict resolution when concurrent edits affect the same region
  • Support offline editing with merge on reconnection
  • Version history with undo/redo per user
  • Cursor and selection sharing (see where others are typing)

Non-Functional Requirements

  • Real-time: changes visible to other users within 100ms
  • Consistency: no data loss, all users converge to the same final document
  • Scalability: support millions of concurrent editing sessions
  • Low latency: typing latency not noticeably worse than local editing
  • Availability: survive server failures without data loss

Deep Dive: Operational Transformation (OT)

OT is the algorithm behind Google Docs. When two users make concurrent edits, the server transforms one operation against the other so both can be applied without conflict.

OT Example

Document state: "Hello world" (length 11) User A: inserts "beautiful " at position 6 → "Hello beautiful world" User B: inserts "Dear " at position 0 → "Dear Hello world" Both operations based on revision 0 (concurrent edits). Server receives A's op first: Apply A: "Hello beautiful world" (rev 0 → rev 1) Transform B's op against A's op: B inserts at 0 → still inserts at 0 (A's insert was after position 0) Apply transformed B: "Dear Hello beautiful world" (rev 1 → rev 2) Server receives B's op first: Apply B: "Dear Hello world" (rev 0 → rev 1) Transform A's op against B's op: A inserts at 6 → insert at 6+5=11 (shifted by B's 5-char insert) Apply transformed A: "Dear Hello beautiful world" (rev 1 → rev 2) Same final result: "Dear Hello beautiful world" ✓ (consistency guaranteed)
OT properties: The transformation function must satisfy two properties: (1) TP1(a, b) applied after b equals TP2(b, a) applied after a (convergence), and (2) applying a transformation never loses operations (intention preservation). Google's Wave protocol formalized these guarantees.

Deep Dive: OT Server Algorithm

# Server-side OT processing

On receive operation op from client at revision client_rev:

  # 1. Transform against all operations the client hasn't seen
  for each pending_op in operations[client_rev + 1 ... server_rev]:
      op = transform(op, pending_op)

  # 2. Apply to current document state
  document = apply(document, op)
  server_rev += 1

  # 3. Broadcast to all OTHER clients
  for each other_client in connected_clients:
      # Each client may be at a different revision
      broadcast_op = op
      for each op that other_client has NOT seen:
          broadcast_op = transform(broadcast_op, that_op)
      send(broadcast_op) to other_client

  # 4. Persist operation to operation log
  append_to_log(op, server_rev)
Centralized OT server: The server acts as the single source of truth. Clients send operations, the server transforms and applies them, then broadcasts. This avoids N-way transformation complexity (which would be O(N^2) for N concurrent editors). The downside is that the server is a single point of coordination.

Deep Dive: CRDT (Alternative to OT)

Conflict-Free Replicated Data Types (CRDTs) provide an alternative approach where each client can apply operations independently and all replicas will converge to the same state without a central server.

AspectOTCRDT
ArchitectureCentral server requiredPeer-to-peer capable
ComplexityHigh (transformation logic)Moderate (merge functions)
Document modelLinear text (array)RGA (Replicated Growable Array), tree, etc.
Rich text supportMature (Google Docs proven)Research phase (proven for text, complex for rich)

Hybrid approach (used by modern editors): Use OT for real-time collaborative editing with a central server, and CRDT concepts for offline-first editing and reconnection merge. Notion and some modern editors take this hybrid approach. The OT server handles the happy path; CRDT-style tombstones and version vectors handle reconnection.

Deep Dive: Persistence and History

  • Operation log: Append-only log of all operations, stored in Kafka for durability and replay. Each operation has a monotonically increasing revision number.
  • Document snapshots: Periodically (every 100 operations or 10 seconds) snapshot the current document state. Store in the database. This enables fast document loading (load latest snapshot + replay operations since snapshot).
  • Version history: Users can view the document at any previous revision. Walk backwards through the operation log and apply inverse operations (or use snapshots to skip large jumps).
Compaction: The operation log grows unboundedly. Periodically compact: create a new snapshot from the current state, then truncate the operation log before the snapshot. Keep a configurable retention window (e.g., last 1000 revisions) for undo support.

Common Interview Questions

  • Why not just send the entire document on every keystroke? A: Bandwidth and latency. A 50KB document updated 10 times/sec = 500KB/sec per editor. With OT/CRDT, each operation is ~20 bytes, so 200 bytes/sec — 2500x more efficient. Also, sending the full document makes conflict resolution impossible.
  • How does OT handle three or more concurrent editors? A: The central server serializes all operations. Each operation is transformed against all pending operations in order. The complexity is O(N * K) where N = number of pending ops, K = transformation cost. For typical documents with a few editors, this is fast.
  • How do you handle the server going down? A: All operations are persisted to the operation log (Kafka) before acknowledging. On server restart, replay from the last snapshot + operations. Clients buffer operations locally and retry on reconnect.
  • How do you support undo/redo in a collaborative document? A: Per-user undo stack. Each user tracks their own operations. Undo removes the user's last operation (transformed against intervening operations from others). This is different from a single global undo — each user undoes only their own changes.
  • How does this scale to very large documents (100+ pages)? A: Use a hierarchical document model (tree structure) instead of a flat array. This enables selective syncing of only the visible portions of the document (viewport optimization). Operations are applied to the relevant subtree.
  • What is the difference between optimistic updates and OT? A: Optimistic updates apply locally and hope for no conflicts (retry on conflict). OT guarantees convergence — every concurrent edit is mathematically resolved. For collaborative editing, OT/CRDT is required; optimistic updates are insufficient.
What to Skip:
  • Don't implement the full OT transformation function — describe the concept and properties
  • Don't design the rich text editor UI (Quill, ProseMirror) — assume a client-side library
  • Don't design the document sharing/permission system in detail
  • Don't get into specific CRDT implementations (RGA, LWW-Element-Set) unless asked
