Pastebin

Difficulty: ⭐⭐ Interview Frequency: ★★★★

Goal

Design a web application where users can store text snippets (code, notes, configuration) and share them via a unique URL. Pastes should support expiration, syntax highlighting, and privacy controls.

Functional Requirements

  • Create a new paste with text content and optional metadata (title, language, expiration)
  • View a paste via a unique URL, rendered with syntax highlighting
  • Support raw text view (no highlighting, plain text)
  • Delete a paste (with or without authentication)
  • List recent public pastes
  • Support private/unlisted pastes

Non-Functional Requirements

  • Low latency: paste creation < 500ms, view < 200ms
  • Scalability: handle 10M+ pastes, 100M+ daily views
  • Storage efficiency: compress large pastes, handle up to 512KB per paste
  • Reliability: no data loss, eventual consistency acceptable for listing

Deep Dive: Storage Strategy

Separation of Metadata and Content

Store paste metadata (ID, title, language, expiration, visibility, creator) in a database (Cassandra or PostgreSQL). Store the actual text content in object storage (S3) or a key-value store. This separation allows the database to remain lightweight while object storage handles variable-size content efficiently.

Why separate metadata from content? Database rows with large TEXT columns degrade query performance and increase replication cost. Object storage is cheaper at scale and handles variable-size blobs natively. The metadata DB only needs to serve small, indexed rows for lookups and listing.

Content Compression

Compress paste content before storing. Use gzip or zstd for 60-80% size reduction on text. Decompress on read for rendering. This reduces storage costs significantly (900GB raw becomes ~200GB compressed).

Deep Dive: Paste Expiration

  1. TTL at the application level: Store expires_at timestamp in metadata. On read, check if paste has expired and return 404/Gone if so. A background cleanup job deletes expired content from object storage and metadata from the database.
  2. TTL at the storage level: Use S3 lifecycle policies to automatically delete objects after a set time. For database records, use Cassandra TTL or a scheduled job to purge expired rows.
Expiration cleanup: Do not delete synchronously on read. Use a batch cleanup job that runs periodically (e.g., every hour) to scan and delete expired pastes. This avoids read-path latency impact from garbage collection.

Deep Dive: Rate Limiting

Prevent abuse by rate-limiting paste creation:

  • IP-based rate limiting: 10 pastes/hour per unauthenticated IP
  • User-based rate limiting: 50 pastes/hour per authenticated user
  • Content size limit: 512KB per paste, reject larger submissions
  • Implement via token bucket in Redis with a sliding window counter

Scalability Considerations

  • Object storage: S3 or equivalent for paste content. Handles variable-size content, replication, and lifecycle policies natively.
  • Cache: Cache rendered paste views in Redis for popular pastes. Use paste_id as key. TTL aligned with paste expiration.
  • Syntax highlighting: Perform server-side using Pygments or client-side using Prism.js/Highlight.js. Server-side allows caching the rendered output. Client-side shifts compute to the browser.
  • Sharding: Shard pastes table by paste_id prefix or hash for horizontal scaling. Cassandra naturally supports this with its partition key.
  • CDN: Serve raw paste views and rendered HTML through a CDN for geographic distribution. Cache-Control headers aligned with expiration.

Common Interview Questions

  • How do you handle paste deletion without authentication? A: Generate a random delete token on creation, hash it, and store the hash. Return the plaintext token to the creator. On delete, verify the hash.
  • What happens when object storage is temporarily unavailable during paste creation? A: Store content in a local buffer/queue, retry with exponential backoff. Return the paste URL immediately if metadata was saved.
  • How do you support syntax highlighting efficiently? A: Server-side rendering with Pygments, cache the rendered HTML in Redis. Or push highlighting to the client with Prism.js.
  • How would you prevent spam/abuse? A: Rate limiting, content scanning for malware/phishing, CAPTCHA for anonymous users, auto-expire suspicious pastes quickly.
  • How do you handle the "recent pastes" listing at scale? A: Maintain a materialized view or a Redis sorted set (ZADD with timestamp). For larger scale, use a dedicated listing service with denormalized data.
  • Why not store everything in a relational database? A: You could for smaller scale. Separating content from metadata allows independent scaling. Object storage is 10x cheaper for large blobs than database storage.
What to Skip:
  • Don't implement full user authentication — mention it as an enhancement
  • Don't spend time on specific syntax highlighting library internals
  • Don't design a WYSIWYG editor — the core problem is storage and retrieval
  • Don't over-design the listing/recent pastes feature unless asked
