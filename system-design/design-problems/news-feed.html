News Feed

Difficulty: вӯҗвӯҗвӯҗвӯҗ Interview Frequency: вҳ…вҳ…вҳ…вҳ…вҳ…

Goal

Design a news feed system similar to Facebook, Twitter, or Instagram. The system must generate a personalized feed of posts from a user's social graph, support ranking and ranking updates, handle high write throughput (new posts), and serve millions of feed reads per second.

Functional Requirements

  • Display a personalized, ranked feed of posts from followed users and pages
  • Support creating new posts (text, images, videos)
  • Support likes, comments, and shares on posts
  • Update the feed in near real-time when new posts are created
  • Support infinite scroll pagination
  • Handle feed ranking based on relevance, recency, and social signals

Non-Functional Requirements

  • Low latency: feed generation < 500ms p99
  • High throughput: 100K+ writes/sec (new posts), 10M+ reads/sec (feed views)
  • Consistency: eventual consistency acceptable (feed may lag by seconds)
  • Scalability: support billions of users, trillions of posts
  • Personalization: ranking must consider individual user preferences

Deep Dive: Fan-out on Write vs Fan-out on Read

Fan-out on Write (Push Model)

When a user creates a post, the system immediately pushes (writes) the post to the feed of all their followers. Feed reads are fast (just read from cache) but writes are expensive (one write per follower).

Alice posts вҶ’ Write to Alice's outbox вҶ’ Fan-out: Write to feed of Follower 1, Follower 2, ..., Follower N Write cost: O(N) where N = follower count Read cost: O(1) вҖ” just read from pre-computed feed Best for: Users with few followers (most users)

Fan-out on Read (Pull Model)

When a user requests their feed, the system gathers the latest posts from all followed users and merges/ranks them on the fly. Writes are cheap (one write) but reads are expensive (query N inboxes).

User opens feed вҶ’ Gather posts from: Following 1, Following 2, ..., Following M вҶ’ Merge, rank, paginate вҶ’ Return feed Write cost: O(1) вҖ” just write to outbox Read cost: O(M) where M = following count Best for: Users with many followers (celebrities, influencers)

Hybrid Approach (Recommended)

Production strategy: Use fan-out on write for most users (followers < 10K). For celebrity accounts with massive followings (followers > 1M), use fan-out on read. The threshold between the two strategies is configurable. This avoids writing a single post to millions of inboxes.

Deep Dive: Feed Ranking

Rank posts in the feed using a scoring function that considers multiple signals:

score = f(recency, engagement, social_affinity, content_type, freshness)

Where:
  recency       = time_decay(post_timestamp, half_life=4h)
  engagement    = log(likes + 2*comments + 3*shares + 1)
  social_affinity = interaction_frequency(viewer, author)
  content_type   = weight(video=1.5, photo=1.2, text=1.0, link=0.8)
  freshness     = penalty for posts already seen by the user
Ranking at scale: Pre-compute a ranked feed for each user asynchronously (offline ranking). When the user loads their feed, serve the pre-computed feed from cache. Periodically re-rank based on updated engagement signals. This decouples the expensive ranking computation from the latency-critical feed read path.

Deep Dive: Feed Caching

  • Pre-computed feed cache: Store the top 100 ranked posts for each user in Redis. TTL of 5-15 minutes. Invalidate on new post creation (fan-out updates the cache).
  • Feed assembly: The fan-out workers write new post IDs to each follower's sorted set in Redis (ZADD with score = rank). On feed read, ZRANGE returns the top N posts.
  • Miss handling: If the cache is cold (new user, cache expiry), generate the feed on-the-fly from the user's following list. This is slower but only happens rarely.
# Redis feed structure (per user)
ZADD feed:user:12345 1701234567 "post_abc"  # score = timestamp or rank
ZADD feed:user:12345 1701234560 "post_def"
ZADD feed:user:12345 1701234555 "post_ghi"

# Get top 25 posts (most recent)
ZREVRANGE feed:user:12345 0 24

# Trim to keep only latest 1000 posts per user
ZREMRANGEBYRANK feed:user:12345 0 -1001

Deep Dive: Pagination

Use cursor-based pagination for efficient infinite scroll:

  • Each post has a unique, ordered ID (e.g., timestamp-based Snowflake ID)
  • Client sends the last seen post ID as the cursor
  • Server returns posts with IDs < cursor, ordered descending
  • Avoids offset-based pagination problems (skipped duplicates, inconsistent results)
Offset pagination pitfall: If new posts arrive between page loads, offset-based pagination can skip posts or show duplicates. Cursor-based pagination with deterministic ordering avoids this entirely.

Common Interview Questions

  • How do you handle a celebrity with 100M followers? A: Fan-out on read for that user's posts. Store the post once, and when followers load their feed, pull in the celebrity's latest posts. Use a special "celebrity" flag in the follower graph to trigger this path.
  • How do you handle feed consistency? A: Accept eventual consistency. Feed updates propagate within seconds. If a user sees stale data, a refresh (pull-to-refresh) fetches the latest. For critical updates (e.g., post deletion), use an invalidation event broadcast via WebSocket.
  • How do you handle the write fan-out latency for popular users? A: Fan-out is async via Kafka. The user sees their post immediately (write to outbox). Fan-out workers process followers asynchronously. Followers see the post within seconds. This avoids blocking the write on fan-out completion.
  • What happens when Redis cache runs out of memory? A: Use LRU eviction вҖ” less active users' feeds are evicted first. They are regenerated on next access. Monitor cache hit rate and scale Redis horizontally when hit rate drops below 95%.
  • How would you support a "Stories" feature (ephemeral, 24h content)? A: Separate data path. Store stories in Redis with 24h TTL. On feed load, merge top stories into the feed. Use a separate carousel UI component. Simpler lifecycle than permanent posts.
