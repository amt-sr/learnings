Distributed Cache

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design a distributed in-memory caching system that provides fast key-value lookups across a cluster of machines. The system must handle data partitioning, node failures, cache eviction, and replication while maintaining low latency and high throughput.

Functional Requirements

  • Store and retrieve key-value pairs with sub-millisecond latency
  • Support TTL (time-to-live) for automatic expiration
  • Distribute data across multiple nodes with rebalancing on topology changes
  • Replicate data for fault tolerance
  • Support configurable eviction policies (LRU, LFU, FIFO)
  • Support optional persistence to disk

Non-Functional Requirements

  • Low latency: reads/writes < 1ms for single key operations
  • High throughput: 1M+ operations/second per cluster
  • Scalability: linear scaling with node additions
  • Availability: survive N/2 - 1 node failures (with replication factor N)
  • Consistency: eventual consistency acceptable; tunable for strong consistency

Deep Dive: Consistent Hashing

Use consistent hashing to map keys to nodes. Each node is placed on a virtual circle (hash ring). Keys are hashed and placed on the ring; each key is assigned to the nearest node clockwise. When nodes are added or removed, only K/N keys are remapped (K = total keys, N = total nodes).

Virtual nodes: Each physical node is represented by multiple virtual nodes (e.g., 150-200 vnodes per physical node) on the hash ring. This ensures more uniform data distribution even when physical nodes have different capacities or when hashing produces uneven distribution.

Deep Dive: Replication

For a replication factor of N, each key is stored on N distinct nodes. The primary node is the first node clockwise on the hash ring. The next N-1 distinct nodes clockwise serve as replicas.

StrategyConsistencyAvailabilityDescription
synchronousStrongLowerWait for all replicas to acknowledge before returning
asynchronousEventualHighReturn immediately, replicate in background
quorum (W < N)TunableTunableWait for W of N replicas (R + W > N for strong reads)
Read repair: When a read detects stale data on a replica (version mismatch), trigger a background read repair that updates the stale replica. This is a lazy anti-entropy mechanism. For more aggressive consistency, use a gossip-based anti-entropy protocol (Merkle trees) to detect and repair inconsistencies proactively.

Redis approach: Redis uses a combination of LRU and LFU via the maxmemory-policy configuration. It samples a small number of keys (default 5) and evicts the best candidate from the sample, avoiding O(N) scans. This probabilistic approach achieves near-optimal eviction with O(1) amortized cost.

Deep Dive: Cache Invalidation

  • Cache-aside (lazy loading): Application checks cache first; on miss, reads from DB and populates cache. Pro: simple. Con: stale data until TTL expires.
  • Write-through: Write to cache and DB simultaneously. Pro: data is never stale. Con: write latency increased.
  • Write-behind (write-back): Write to cache immediately, batch-write to DB asynchronously. Pro: low write latency. Con: data loss risk on crash.
  • Cache invalidation: On DB update, delete or update the corresponding cache key. Pro: immediate consistency. Con: thundering herd on repopulation.
Thundering herd: When a popular cache key expires, many requests simultaneously try to repopulate it. Mitigate with: request coalescing (only one request fetches from DB, others wait), probabilistic early expiration (expire ~10% of keys slightly before TTL to stagger refreshes), or a lock-per-key approach.

Common Interview Questions

  • How does consistent hashing minimize data movement when nodes are added/removed? A: Only keys in the affected range (K/N) are remapped. Virtual nodes ensure balanced redistribution.
  • How do you handle a node failure? A: Detect via heartbeat timeout. Promote a replica to primary for the affected key ranges. Trigger rebalancing to maintain the replication factor.
  • What is the "hot key" problem and how do you solve it? A: A single key receives disproportionate traffic. Solutions: micro-sharding (split the key across multiple cache nodes), local caching at the application layer, or request-level fanout.
  • How do you achieve strong consistency in a distributed cache? A: Quorum reads/writes (R + W > N), synchronous replication, and read repair. Trade off availability for consistency.
  • When would you choose Memcached over Redis? A: When you need simple caching, multi-threaded performance for large values, and don't need data structures or persistence. Memcached has lower per-key overhead.
  • How do you handle cache warming? A: Pre-populate cache from DB during startup or low-traffic periods. Use a dedicated warmup service that reads hot keys from analytics and loads them into the cache.
What to Skip:
  • Don't implement the actual cache server internals (memory allocator, I/O threads) unless asked
  • Don't get deep into Redis Sentinel vs Cluster unless the interviewer asks
  • Don't design a general-purpose database — stay focused on caching semantics
  • Don't skip the eviction policy discussion — it's a key differentiator at L5/L6 level
