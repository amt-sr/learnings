Logging System

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★

Goal

Design a centralized logging system that ingests log events from thousands of services at high throughput, stores them for search and analysis, and provides a query interface for debugging, monitoring, and compliance. The system must handle variable volume with backpressure handling and tiered storage for cost-effective retention.

Functional Requirements

  • Ingest structured log events from multiple services and hosts
  • Support log levels (DEBUG, INFO, WARN, ERROR) and metadata (service, host, trace_id)
  • Provide full-text search across logs with filters (service, time range, level, trace_id)
  • Support real-time log streaming (tail -f across services)
  • Retain logs based on configurable retention policies per service
  • Support log aggregation and pattern-based alerting

Non-Functional Requirements

  • High throughput: ingest 10TB+ of logs per day
  • Low latency (ingest): logs searchable within 10 seconds of generation
  • Low latency (query): search results returned within 1-5 seconds
  • Durability: no log data loss (at-least-once delivery)
  • Cost-effective: tiered storage (hot/warm/cold) for different retention needs

Deep Dive: Log Collection

Agents run on every host (or as a sidecar in Kubernetes) to collect logs.

  • Fluent Bit: Lightweight agent (minimal CPU/memory). Runs on every node. Batches and compresses logs before sending to Kafka or directly to Elasticsearch.
  • Fluentd: Heavier agent with rich plugin ecosystem. Used for log parsing, enrichment, and routing when complex processing is needed.
  • Filebeat: Alternative agent (Elastic Stack). Reads log files and ships to Logstash or Elasticsearch.
Buffering and backpressure: When the downstream (Kafka/Elasticsearch) is slow or down, the collector buffers logs locally on disk. Fluent Bit uses a filesystem buffer (configurable size, e.g., 1GB). When the buffer is full, either drop oldest logs (lossy) or block the application (lossless, but risks affecting the application). For critical logs, prefer lossless with a large buffer.

Deep Dive: Ingest Pipeline

Raw logs arrive as semi-structured text. The pipeline must parse, enrich, and index them.

  1. Parse: Extract structured fields from log format (JSON, key=value, regex for unstructured). Most services should emit structured JSON logs.
  2. Enrich: Add metadata: host IP, service version, datacenter, availability zone, Kubernetes pod name, trace_id (from distributed tracing context).
  3. Index: Route to the appropriate Elasticsearch index based on service, date, and log level. Use index templates for consistent mapping.
# Index naming pattern:
logs-{service}-{date}

# Examples:
logs-payment-service-2026.06.28
logs-api-gateway-2026.06.28
logs-auth-service-2026.06.28

Deep Dive: Tiered Storage

TierStorageRetentionCostUse Case
HotElasticsearch SSD7-30 daysHigh ($/GB)Active debugging, real-time dashboards
WarmElasticsearch HDD30-90 daysModerateLess frequent queries, post-incident analysis
ColdS3 + Athena/Parquet1-7 yearsLow ($/GB)Compliance, audit, rare investigation
Index lifecycle management (ILM): Use Elasticsearch ILM policies to automatically transition indices: Hot (0-7 days) → Warm (7-30 days, reduce replicas) → Cold (30+ days, move to S3) → Delete (after retention period). This automates storage management and reduces cost.

Deep Dive: Search at Scale

  • Full-text search: Elasticsearch inverted index provides fast full-text search across log messages. Supports wildcard queries, regex, proximity search.
  • Structured filters: Exact match on service, level, host, trace_id via keyword fields. These are cacheable and fast.
  • Time-range queries: Always include a time range filter. This limits the search to specific indices (date-based), reducing the search space dramatically.
  • Aggregations: Group by service, level, or host for dashboards. Pre-computed for hot data, on-the-fly for warm/cold.
Query routing: When a query spans hot and cold storage, the query service routes time-range segments to the appropriate backend. Recent data goes to Elasticsearch, older data goes to Athena/S3. Results are merged client-side.

Common Interview Questions

  • How do you handle log volume spikes (10x normal during incidents)? A: Kafka absorbs spikes as a buffer. Collectors batch and compress to reduce per-message overhead. Scale Elasticsearch horizontally by adding nodes. Accept increased query latency during spikes.
  • How do you prevent logs from affecting application performance? A: Async logging (application writes to an in-process buffer, background thread ships logs). Separate log disks. If the buffer is full, drop logs (lossy) to protect the application. Sidecar pattern isolates the collector from the application container.
  • How do you ensure log integrity for compliance? A: Write-once semantics in the log store. Hash-chain log entries (blockchain-like) to detect tampering. Store audit-critical logs in an append-only store. Use S3 Object Lock for immutable cold storage.
  • How do you handle sensitive data in logs (PII, passwords)? A: Parse and redact sensitive fields at collection time. Use regex patterns to detect SSN, credit cards, emails. Hash or mask the values. Never log authentication tokens or passwords. Implement as a collector plugin for consistency.
  • What is the cost breakdown of this system? A: 80% storage cost, 15% compute (Elasticsearch), 5% network. Storage is the dominant cost. Mitigate with: compression (5-10x), tiered storage, aggressive retention policies, sampling for high-volume DEBUG logs.
What to Skip:
  • Don't design log visualization/dashboards — focus on ingestion and search
  • Don't design the application-side logging libraries
  • Don't get into specific Elasticsearch query DSL syntax
  • Don't design alerting rules — mention it as a consumer of the log system
