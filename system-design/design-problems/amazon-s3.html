Amazon S3

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★

Goal

Design an object storage system similar to Amazon S3 that provides virtually unlimited storage, high durability, read-after-write consistency, versioning, and lifecycle management for arbitrary objects. The system must serve as a foundational storage layer for other applications (backup, analytics, CDN origin, application data).

Functional Requirements

  • Store and retrieve arbitrary objects (0 bytes to 5TB) using a flat namespace (bucket + key)
  • Support read-after-write consistency for PUTs of new objects
  • Support object versioning — keep multiple versions of the same key
  • Support lifecycle policies: transition between storage tiers, expiration
  • Support range requests (read partial content)
  • Support conditional operations (ETag-based If-Match, If-None-Match)
  • Provide strong access control (bucket policies, IAM, pre-signed URLs)

Non-Functional Requirements

  • Durability: 99.999999999% (11 nines) — effectively zero data loss
  • Availability: 99.99% for PUT operations, 99.9% for GET
  • Scalability: unlimited number of objects, exabytes of storage
  • Throughput: support 100K+ PUTs/sec, 1M+ GETs/sec per bucket
  • Latency: GET latency < 100ms p99 for cached objects
  • Consistency: read-after-write for new objects, eventual for overwrites

# Upload object

Body:  (object data)

Resp:  200 OK (ETag: "\"d41d8cd98f00b204e9800998ecf8427e\"")

GET    /{bucket}/{key}
Headers: Range: bytes=0-1023  (optional range request)
Resp:  200 OK / 206 Partial Content

# Head object (metadata only, no body)

Resp:  200 OK (Content-Length, Content-Type, ETag, Last-Modified, Version-Id)

# Delete object

Resp:  204 No Content

# List objects (with prefix filter)

Resp:  <ListBucketResult>...</ListBucketResult>

# Versioning operations
PUT    /{bucket}?versioning  Body: <VersioningConfiguration><Status>Enabled</Status>...
GET    /{bucket}/{key}?versionId=12345  (fetch specific version)

Deep Dive: Flat Namespace Architecture

S3 uses a flat namespace: buckets are the top-level container, and objects are addressed by keys within a bucket. There are no real directories — the "/" in keys is just a naming convention.

Why flat namespace? Hierarchical filesystems have scaling limits (inode tables, directory locking). A flat namespace allows unlimited objects per bucket without structural bottlenecks. "Folders" are virtual — implemented as prefix-based queries (WHERE key LIKE 'photos/%'). 
# These are all independent flat objects in the same bucket:
photos/2026/vacation.jpg
photos/2026/birthday.jpg
documents/report.pdf
documents/notes.txt

# "Listing the photos/2026/ folder" is actually:
# SELECT key FROM objects WHERE bucket='mybucket' AND key LIKE 'photos/2026/%'
# LIMIT 1000

Deep Dive: Consistency Model

S3 provides strong read-after-write consistency for PUTs of new objects. For PUTs that overwrite existing objects, S3 provides eventual consistency (you might read the old version briefly after a write).

Consistency details:
  • New object PUT → GET: Strongly consistent. After a successful PUT of a new key, any GET immediately returns the new object.
  • Overwrite PUT → GET: Eventually consistent. After overwriting an existing key, a GET might return the old version for a brief period (typically sub-second).
  • DELETE → GET: Eventually consistent. A GET after DELETE might return the deleted object briefly.
  • PUT → LIST: Eventually consistent. A new object might not appear in LIST immediately.

Deep Dive: Replication and Placement

Objects are replicated across multiple storage nodes in different availability zones.

Object PUT flow: Client ──► Front-End ──► Placement Engine │ Determine replication targets │ ┌────────────────┼────────────────┐ ▼ ▼ ▼ Node A Node B Node C (AZ-1a) (AZ-1b) (AZ-1c) [Object data] [Object data] [Object data] Write quorum: W=2 (acknowledge after 2 of 3 nodes confirm) Read quorum: R=1 (read from any 1 node, but prefer closest) Consistency: R + W > N → 2 + 1 > 3 (strong for new objects) For overwrites: asynchronous replication (W=1, eventual consistency) This explains the eventual consistency for overwrites.

Deep Dive: Object Versioning

When versioning is enabled, every PUT creates a new version. Every DELETE creates a delete marker (not actual deletion). Old versions are retained until explicitly removed or by lifecycle policy.

PUT    mybucket/photo.jpg  → Version 1 (2026-01-01)
PUT    mybucket/photo.jpg  → Version 2 (2026-06-28) (overwrite)
DELETE mybucket/photo.jpg  → Delete marker, Version 3

# Listing all versions:
GET    mybucket/photo.jpg?versions
→ Version 3: Delete marker (2026-06-28)
  Version 2: 2.5MB photo (2026-06-28)
  Version 1: 1.2MB photo (2026-01-01)

# Regular GET returns 404 (latest version is a delete marker)
# GET ?versionId=1 returns the original photo
Versioning cost: Every overwrite doubles the storage for that key. Use lifecycle policies to automatically expire old versions after N days or keep only the last M versions. This is critical for cost control in write-heavy workloads.

Deep Dive: Lifecycle Management

Automate storage optimization with lifecycle rules:

RuleTransitionCost Impact
Standard → IAAfter 30 days, not accessed recently~60% cheaper than Standard
IA → GlacierAfter 90 days, rarely accessed~75% cheaper than IA
Glacier → Glacier Deep ArchiveAfter 1 year, archival only~50% cheaper than Glacier
Any → DeleteAfter 365 days (temporary data)Free up storage
Non-current versions → DeleteAfter 30 days (keep 1 version)Free up storage
Intelligent tiering: Automatically move objects between access tiers based on access patterns. No lifecycle configuration needed — the system monitors access frequency and moves infrequently accessed objects to lower-cost tiers. Best for unpredictable workloads.

Deep Dive: Erasure Coding for Cold Storage

For Glacier-class storage, use erasure coding instead of replication for cost efficiency.

# Erasure coding (e.g., 12+4 Reed-Solomon):
# Split object into 12 data chunks + 4 parity chunks
# Store each chunk on a different node
# Can reconstruct from any 12 of 16 chunks
# Storage overhead: 16/12 = 1.33x (vs 3x for replication)
# Durability: still 11 nines (with appropriate redundancy)

Replication: 3x storage = 300 PB for 100 PB data
Erasure coding: 1.33x = 133 PB for 100 PB data
Savings: 167 PB (56% reduction in storage cost)

Common Interview Questions

  • How does S3 achieve 11 nines durability? A: Replication across multiple failure domains (availability zones), continuous integrity checking with checksums, automated scrubbing that detects and repairs corruption, and quick recovery from failed nodes. The per-object durability of 99.999999999% means you lose 1 object per 100 billion objects stored per year on average.
  • How do you handle the metadata bottleneck for billions of objects? A: Shard metadata by (bucket, key_prefix_hash) across many metadata nodes. Use a DynamoDB-like consistent hashing ring. Each metadata node handles a subset of keys. Metadata is relatively small (~1KB per object), so billions of objects = ~1TB metadata, easily sharded.
  • How does S3 handle multipart uploads for large objects? A: Client uploads parts independently (each up to 5GB). The service stores each part as a separate object. On completion, S3 creates a manifest that references all parts. The object is assembled on read (logical concatenation) or physically if needed. Failed parts can be retried independently.
  • Why is S3 eventually consistent for overwrites? A: For performance. Synchronous replication for overwrites would require all replica nodes to acknowledge the update before returning success, adding latency (especially for cross-region replication). Since most workloads create new objects (not overwrites), this trade-off is acceptable. S3 has been moving toward strong consistency over time.
  • How do you implement range requests efficiently? A: The storage node stores objects as fixed-size chunks. For a range request, the node reads only the relevant chunks and returns them with a 206 Partial Content response. Combined with CDN support for Range headers, this enables efficient video streaming from S3.
What to Skip:
  • Don't design the physical data center infrastructure
  • Don't implement the full IAM/access control system
  • Don't design cross-region replication in detail unless asked
  • Don't implement the event notification system (S3 events → Lambda)
