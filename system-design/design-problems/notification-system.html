Notification System

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design a multi-channel notification system that sends notifications to users via email, SMS, push notifications (iOS/Android), and in-app notifications. The system must support templating, user preferences, delivery tracking, and high throughput across all channels.

Functional Requirements

  • Send notifications via multiple channels: email, SMS, push, in-app
  • Support notification templates with dynamic content (user name, action, link)
  • Allow users to configure channel preferences (opt-in/opt-out per channel per category)
  • Track delivery status (sent, delivered, read, failed) and support retry logic
  • Support scheduled notifications and batch sending (marketing campaigns)
  • Rate limit notifications per user to prevent spam

Non-Functional Requirements

  • High throughput: handle 10M+ notifications/hour
  • Low latency: push/in-app < 1s, email < 5s, SMS < 10s
  • Reliability: at-least-once delivery guarantee with deduplication
  • Scalability: independent scaling per channel
  • Fault tolerance: retry failed deliveries with exponential backoff

Deep Dive: Notification Flow

  1. Trigger: A service posts a notification event to Kafka with user_id, template_id, channel hints, and template data.
  2. Router: A consumer reads from Kafka, resolves user preferences (which channels are enabled), selects the delivery channels, and routes to per-channel topics.
  3. Template rendering: Template service renders the notification using the template_id and data. Produces a subject, body, and optional metadata per channel.
  4. Delivery: Channel-specific workers pull from their respective Kafka topics and send via external providers (SES for email, Twilio for SMS, FCM/APNs for push).
  5. Tracking: Delivery tracking service records status (sent, delivered, bounced, read) from provider webhooks/callbacks.
Why Kafka between stages? Decouples producers from consumers, enables independent scaling per channel, provides buffering during provider outages, and ensures at-least-once delivery semantics. If a downstream provider (e.g., Twilio) is down, messages accumulate in Kafka and are processed when it recovers.

Deep Dive: Template System

Templates use a placeholder system for dynamic content. Store templates in a database with versioned content.

Template: "order_shipped" (v2)
  Email subject: "Your order #{{order_id}} has shipped!"
  Email body:    "Hi {{user.name}}, your order #{{order_id}} was shipped via
                  {{carrier}}. Track it here: {{tracking_url}}"
  Push title:    "Order Shipped"
  Push body:     "Your order #{{order_id}} is on its way!"
  SMS body:      "Your order #{{order_id}} has shipped. Track: {{tracking_url}}"
Template versioning: Support multiple versions of templates for A/B testing and gradual rollouts. Store the template version in the notification metadata for auditability. When a user reports a notification issue, you can reproduce exactly what was sent.

Deep Dive: Retry and Deduplication

  • Retry strategy: Exponential backoff with jitter. Retry 3 times for transient failures (timeout, 5xx). Do not retry for permanent failures (invalid email, unsubscribed). Increase delay: 1s, 5s, 30s.
  • Dead letter queue: After max retries, move failed notifications to a DLQ topic. Manual inspection and reprocessing via an operations dashboard.
  • Deduplication: Generate a deterministic notification ID based on (event_type, user_id, entity_id). Store in a Redis set with a TTL equal to the dedup window (e.g., 5 minutes). Check before sending to prevent duplicate notifications from at-least-once delivery.

Common Interview Questions

  • How do you handle rate limiting for notifications? A: Per-user daily caps per channel (e.g., max 10 push notifications/hour, max 5 emails/day for marketing). Use Redis counters with TTL. Enforce in the router before queuing.
  • What happens when an email provider (SES) is down? A: Messages accumulate in the Kafka topic. Configure alerting on consumer lag. Auto-retry with backoff. If outage is extended, use a backup provider.
  • How do you ensure notifications are not lost? A: At-least-once delivery with Kafka (producer acks=all, consumer manual commit). Deduplication on the consumer side using notification IDs. Dead letter queue for permanent failures.
  • How would you support real-time (in-app) notifications? A: WebSocket connection per user. Maintain a connection manager that maps user_id to WebSocket server. Use a pub/sub system (Redis Pub/Sub or Kafka) to broadcast to the correct server. Client reconnects with exponential backoff.
  • How do you handle notification localization? A: Store the user's locale preference. Templates have locale variants. The router selects the appropriate template version based on locale. Fallback to English if locale template is missing.
What to Skip:
  • Don't implement the actual email sending (SMTP, DKIM) — assume a provider like SES
  • Don't design the user authentication or profile system
  • Don't get deep into push notification certificate management
  • Don't over-design the template editor — a simple placeholder system suffices
