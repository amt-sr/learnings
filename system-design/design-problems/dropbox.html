Dropbox

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design a file synchronization service similar to Dropbox that keeps files across multiple devices in sync. The system must efficiently detect changes, transfer only modified portions of files (delta sync), handle conflict resolution when multiple devices edit the same file simultaneously, and provide offline support with reconnection sync.

Functional Requirements

  • Automatically sync files between user's devices (desktop, mobile, web)
  • Efficiently detect local file changes (new, modified, deleted)
  • Transfer only changed blocks (delta sync) for modified files
  • Handle conflict resolution when the same file is modified on multiple devices
  • Support offline editing with sync on reconnection
  • Provide file versioning and restore capabilities
  • Share files and folders with other users

Non-Functional Requirements

  • Bandwidth efficiency: transfer only changed data, not entire files
  • Low latency: sync notification within seconds of a change
  • Scalability: 500M+ users, 500B+ files
  • Storage efficiency: deduplication (across users and within user history)
  • Reliability: no data loss, sync eventual consistency

Deep Dive: Change Detection

Each device must detect local file changes efficiently:

ApproachHow it WorksProsCons
Filesystem events (inotify/FSEvents)OS notifies on file changeImmediate, efficientEvent loss during sleep, platform-dependent
Periodic scanScan all files, compare against metadata DBReliable, cross-platformCPU-intensive for large directories
HybridUse events for real-time + periodic scan as fallbackBest of bothComplex
Recommended: Hybrid approach. Use OS filesystem events (inotify on Linux, FSEvents on macOS, ReadDirectoryChangesW on Windows) for real-time change detection. Run a periodic full-scan as a fallback every few minutes to catch missed events (e.g., during sleep, network disconnect, or event queue overflow).

Deep Dive: Delta Sync

Instead of uploading entire files, only upload changed blocks.

# Step 1: Client hashes local file blocks
File: report.docx (20MB, split into 4MB blocks)
  Block 1: SHA256=a1b2c3 (unchanged)
  Block 2: SHA256=d4e5f6 (CHANGED)
  Block 3: SHA256=g7h8i9 (unchanged)
  Block 4: SHA256=j0k1l2 (unchanged)
  Block 5: SHA256=m3n4o5 (NEW - file grew)

# Step 2: Client sends block hashes to server
# Step 3: Server responds with missing blocks
  Missing blocks: [d4e5f6, m3n4o5]  (server doesn't have these)

# Step 4: Client uploads only the 2 missing blocks (8MB instead of 20MB)
# Bandwidth savings: 60%
Rolling checksum (rsync algorithm): For smaller files, use a rolling checksum approach. Divide the file into variable-sized blocks based on content boundaries (not fixed offsets). This allows efficient sync even when data is inserted in the middle of a file (blocks after the insertion are shifted but remain intact).

Deep Dive: Conflict Resolution

When the same file is modified on two devices simultaneously:

  1. Both devices upload their changes (different block sets).
  2. Server detects that the base version has diverged (both devices edited version N, creating versions N+1a and N+1b).
  3. Server creates a "conflicted copy" — the file from the second device is saved as report (Device B's conflicted copy 2026-06-28).docx.
  4. Both devices download the conflict copy and notify the user.
Alternative: Operational Transform (OT) for text documents. Instead of file-level conflict copies, use character-level merge (similar to Google Docs). This is more complex but provides a better user experience for text files. For binary files (images, videos, PDFs), conflict copies are the only viable approach.

Deep Dive: Notification Service

When a device uploads changes, other connected devices must be notified promptly.

Device A uploads change │ ▼ Notification Service (per-user channels) │ ├──► WebSocket to Device B → "File changed, sync now" ├──► WebSocket to Device C → "File changed, sync now" └──► Push notification to Device D (mobile, offline) → "Sync on next open"
  • Use WebSocket for devices currently connected.
  • Use push notifications (APNs/FCM) for mobile devices that are offline.
  • Per-user notification channels (Redis Pub/Sub partitioned by user_id).
  • If all devices are offline, they will discover changes on next connection via the cursor mechanism.

Deep Dive: Metadata Model

cursorBIGINTPer-user sync cursor (monotonic)
Cursor-based sync: Each user has a monotonically increasing cursor. When a device requests changes, it sends its last cursor. The server returns all changes with a cursor greater than the client's. This ensures the device receives all changes, even if it was offline for an extended period.

Common Interview Questions

  • How does delta sync handle small changes in large files? A: Fixed-block approach (4MB blocks) may waste bandwidth for small changes within a block. The rolling checksum (rsync) approach splits on content boundaries, so small insertions only affect the block where they occur.
  • How do you handle large directories with millions of files? A: Batch sync: process changes in batches with pagination. Use efficient filesystem APIs (readdir with stat). Consider hierarchical sync (sync subdirectories independently) to parallelize and limit per-operation scope.
  • How do you handle selective sync (not all files on all devices)? A: Allow users to choose which folders to sync per device. The metadata service tracks per-device sync preferences. Devices only receive change notifications for their subscribed paths.
  • How do you handle initial sync for a new device? A: Download file metadata first (fast, small). Then download file contents on demand or in background, prioritizing recently accessed files. Show a "syncing" indicator for files not yet downloaded.
  • What happens if the user is offline and modifies many files? A: All changes are recorded locally. On reconnection, the sync client uploads all changes sequentially. Use the cursor mechanism to discover server-side changes made during the offline period and merge.
  • How do you prevent unauthorized access to files? A: End-to-end encryption (user holds the key). Server stores encrypted blocks but cannot decrypt. Metadata is encrypted per-user. Sharing requires re-encrypting with the recipient's key.
What to Skip:
  • Don't design end-to-end encryption in detail unless asked
  • Don't design the file sharing/collaboration workflow — focus on sync
  • Don't design the mobile app UI or desktop app architecture
  • Don't implement filesystem watcher internals — describe the interface
