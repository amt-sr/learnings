File Storage

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design a scalable file storage system similar to Amazon S3 that stores, retrieves, and manages arbitrary files (photos, videos, documents). The system must handle large files efficiently, support metadata management, provide data deduplication, and deliver files through a CDN for fast global access.

Functional Requirements

  • Upload files of any type and size (up to 5GB per file)
  • Download files via public or authenticated URLs
  • Support file metadata (name, size, content-type, custom tags)
  • Support hierarchical namespace (folders/directories) or flat object model
  • Provide data deduplication to save storage costs
  • Support file versioning and soft delete

Non-Functional Requirements

  • Durability: 99.999999999% (11 nines) — no data loss
  • Availability: 99.99% for read operations
  • Scalability: exabytes of storage, petabytes of daily throughput
  • Throughput: support high upload/download bandwidth
  • Consistency: read-after-write consistency
  • Security: encryption at rest and in transit, access control

Deep Dive: Block Storage Architecture

Files are split into fixed-size blocks (e.g., 4MB-64MB) and distributed across storage nodes. This enables parallel uploads, efficient deduplication, and fault tolerance per block.

Why block storage? Storing files as monolithic objects limits upload parallelism and makes deduplication impossible (a 1-byte change to a 1GB file requires re-storing the entire 1GB). Block storage enables content-addressable deduplication and parallel I/O.

Content-Addressable Storage (CAS)

Each block is identified by the hash of its content (SHA-256). Blocks with the same hash are stored once, regardless of how many files reference them.

# Block identification
block_hash = SHA256(block_content)

# File = ordered list of block hashes
File "photo.jpg" (5MB):
  Block 1: hash=a1b2c3... (4MB)
  Block 2: hash=d4e5f6... (1MB)

File "photo_copy.jpg" (5MB, identical):
  Same blocks: a1b2c3, d4e5f6
  → Zero additional storage (deduplication)
Deduplication savings: For photo storage services, deduplication can save 30-50% of storage (duplicate photos, shared files, backups). For code repositories, savings can be 80%+ (shared dependencies across projects).

Deep Dive: Multipart Upload

For large files (> 100MB), use multipart upload:

  1. Initiate: Client requests a multipart upload ID.
  2. Upload parts: Client splits the file into parts (typically 5-100MB each) and uploads each part independently in parallel. Each part gets an ETag (MD5).
  3. Complete: Client sends a complete-multipart request with the ordered list of parts and their ETags. Server assembles the metadata (no physical concatenation needed if using block storage).
  4. Abort: If upload fails, client can abort. Unreferenced blocks are garbage collected.
Resumable uploads: Multipart upload enables resumable uploads. If the connection drops during part 5 of 10, the client only needs to retry parts 5-10. This is critical for large files on unreliable networks.

Flat namespace vs hierarchical: S3 uses a flat object model (object keys can contain "/" but are not real directories). This is simpler and more scalable than a true hierarchical filesystem. Directory listing is implemented as a prefix query on the object key (WHERE key LIKE 'photos/2026/%').

Deep Dive: CDN Integration

  • Origin pull: CDN edge nodes pull files from the origin storage cluster on cache miss. Configure origin shielding to avoid multiple edge nodes hitting the origin for the same file.
  • Pre-signed URLs: Generate time-limited, signed URLs for private files. The CDN validates the signature and expiration. No need for backend authentication on every request.
  • Cache control: Set Cache-Control headers based on file type. Immutable files (by content hash) can be cached indefinitely. Mutable files get short TTLs.
  • Invalidation: For cache invalidation, either (a) change the URL (version the key), or (b) use a purge API. Versioning the URL is preferred as it is more reliable and scales better.

Deep Dive: Garbage Collection

When files are deleted, blocks must be cleaned up. But shared blocks may still be referenced by other files.

  1. Mark file as deleted in metadata (soft delete).
  2. GC job scans unreferenced blocks: for each block, check if any file references it.
  3. Delete blocks with zero references.
  4. Use a reference count or a mark-and-sweep algorithm.
Reference counting pitfall: Simple reference counting can cause issues if deletions are missed (orphaned blocks accumulate). Use a periodic mark-and-sweep: enumerate all referenced blocks from metadata, compare with all stored blocks, delete the difference. Run weekly to catch orphaned blocks.

Common Interview Questions

  • How do you ensure data durability? A: Replicate each block across 3+ nodes in different failure domains (racks, availability zones). Use erasure coding (e.g., 10+4) for cold storage to reduce replication overhead while maintaining durability. Checksum each block and verify on read (read-repair if checksum mismatch).
  • How does block storage handle concurrent updates to the same file? A: Versioning — each update creates a new version with its own block list. Old versions are retained based on the versioning policy (keep last N versions, or keep for N days).
  • How do you handle storage node failures? A: Detect via heartbeat. Replicate missing blocks from surviving replicas to maintain the replication factor. Rebalance the cluster to distribute load. Use consistent hashing for block placement.
  • What is erasure coding and when would you use it? A: Erasure coding splits data into N data chunks and computes M parity chunks. Any N of (N+M) chunks can reconstruct the data. Reduces storage overhead from 3x (replication) to 1.33x (10+4). Used for cold data where read latency is less critical.
  • How do you handle very small files (< 1KB)? A: Store small files in the metadata record directly (inline storage). Avoid the overhead of block allocation for tiny files. Threshold: files < 128KB stored inline in the metadata database.
What to Skip:
  • Don't design the physical storage hardware — focus on the logical architecture
  • Don't design the CDN network topology — treat it as a service
  • Don't implement file compression — mention as an optimization
  • Don't design access control policies in detail unless asked
