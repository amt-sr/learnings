YouTube

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Design a video streaming platform similar to YouTube that handles video upload, transcoding, storage, streaming via CDN, thumbnail generation, search, and personalized recommendations. The system must handle massive bandwidth for video delivery and a high-throughput upload/transcode pipeline.

Functional Requirements

  • Upload videos of various sizes and formats (up to 10GB, 12 hours)
  • Transcode uploaded videos into multiple resolutions (360p, 480p, 720p, 1080p, 4K)
  • Stream videos with adaptive bitrate (DASH/HLS) based on network conditions
  • Generate thumbnails at multiple timestamps for video preview
  • Search videos by title, description, and metadata
  • Personalized recommendation feed
  • Support likes, comments, view count, and video statistics

Non-Functional Requirements

  • High bandwidth: support 1M+ concurrent video streams
  • Low latency for upload: start processing within seconds of upload completion
  • Smooth streaming: adaptive bitrate, no buffering for most connections
  • Scalability: 800M+ videos, 500 hours uploaded per minute
  • Cost efficiency: minimize storage and bandwidth costs via CDN and transcoding

Deep Dive: Upload Pipeline

  1. Request: Client calls the upload API with video metadata (title, size, format).
  2. Presigned URLs: Server generates pre-signed URLs for each part of a multipart upload to the object storage (S3). Client uploads directly to S3, bypassing the application server.
  3. Complete: Client confirms all parts uploaded. Server triggers the transcode pipeline.
  4. Metadata: Store video metadata in the database: title, description, channel, upload time, status (processing/ready).
Direct-to-S3 upload: Never route large video files through the application server. Use pre-signed URLs so the client uploads directly to object storage. This avoids saturating the application server's bandwidth and allows S3 to handle the upload at scale.

Deep Dive: Transcoding Pipeline

Convert uploaded video (typically one format/resolution) into multiple resolutions and formats for adaptive streaming.

Upload: video.mp4 (1080p, 30fps, H.264, 500MB) │ ▼ Transcode Workers (FFmpeg, running on GPU instances) │ ├──► 360p (HLS segments, ~50MB) ├──► 480p (HLS segments, ~100MB) ├──► 720p (HLS segments, ~200MB) ├──► 1080p (HLS segments, ~500MB) └──► 4K (HLS segments, ~1.5GB) (if uploaded in 4K) Also generate: ├──► Thumbnails: frame at 0s, 10s, 30s, midpoint (3 thumbnails) ├──► Preview GIF: 3-second animated preview └──► Metadata: duration, resolution, bitrate, codec 
# Transcode with FFmpeg (example for 720p HLS)
ffmpeg -i input.mp4 \
  -c:v libx264 -preset fast -b:v 2.5M -maxrate 2.5M \
  -bufsize 5M -vf "scale=1280:720" \
  -c:a aac -b:a 128k -ac 2 \
  -f hls -hls_time 10 -hls_list_size 0 \
  -hls_segment_filename "720p_seg_%03d.ts" \
  720p_master.m3u8
Transcoding at scale: Use a distributed task queue (Kafka + worker pool) for transcoding jobs. Scale horizontally by adding GPU-optimized worker instances. Priority queue: popular videos and recently uploaded videos get priority. Use auto-scaling to handle upload spikes.

Deep Dive: Adaptive Streaming (HLS/DASH)

Video is split into small segments (typically 2-10 seconds each). The player downloads segments one at a time and can switch quality mid-stream.

Master Playlist (.m3u8): ┌─────────────────────────────────────────────────┐ │ #EXTM3U │ │ #EXT-X-STREAM-INF:BANDWIDTH=800000,RESOLUTION=640x360 │ │ 360p/playlist.m3u8 │ │ #EXT-X-STREAM-INF:BANDWIDTH=1400000,RESOLUTION=854x480│ │ 480p/playlist.m3u8 │ │ #EXT-X-STREAM-INF:BANDWIDTH=2500000,RESOLUTION=1280x720│ │ 720p/playlist.m3u8 │ │ #EXT-X-STREAM-INF:BANDWIDTH=5000000,RESOLUTION=1920x1080│ │ 1080p/playlist.m3u8 │ └─────────────────────────────────────────────────┘ Player logic: 1. Fetch master playlist 2. Start with 720p (based on estimated bandwidth) 3. Measure actual download speed for each segment 4. If bandwidth drops → switch to lower quality 5. If bandwidth increases → switch to higher quality 6. All switching is seamless (no buffering)

Deep Dive: CDN Strategy

  • Origin shielding: All CDN edge nodes fetch from a single origin shield node (not directly from S3). This prevents multiple edge nodes from duplicating origin requests for the same content.
  • Cache hierarchy: Edge cache (1-5 minutes TTL for hot segments) → Regional cache (1 hour TTL) → Origin (S3).
  • Pre-warming: For highly anticipated content (new popular video), pre-populate CDN edge caches before the traffic arrives.
  • P2P assist: For live streaming, use WebRTC-based P2P between viewers to reduce CDN bandwidth. This is an advanced optimization.
Cache key design: Use content-hash-based URLs for video segments. This ensures cache consistency (same content = same URL) and enables browser-level caching. Example: /videos/{video_id}/1080p/seg_{hash}.ts.

Deep Dive: Recommendation System

YouTube's recommendation is one of the most sophisticated ML systems. At a high level:

  1. Candidate generation: From billions of videos, narrow down to hundreds of candidates using collaborative filtering (users who watched X also watched Y), content-based filtering (similar tags/topics), and trending signals.
  2. Ranking: Score each candidate using a deep learning model that considers: watch probability, expected watch time, user satisfaction prediction, diversity, and freshness.
  3. Re-ranking: Apply business rules: freshness boost, diversity (avoid showing 10 similar videos in a row), deduplication, and policy filters (content safety).
What to Skip: In an interview, describe the recommendation system at a high level. Don't design the ML model architecture or the training pipeline. Focus on the system integration: how candidates flow through the pipeline and how the feed API serves results.

Common Interview Questions

  • How do you handle video uploads from mobile devices with unreliable networks? A: Resumable multipart upload. Client uploads parts independently. If a part fails, retry only that part. Track progress locally. Resume from the last successfully uploaded part on reconnection.
  • How do you prevent copyrighted content from being uploaded? A: Content ID system — fingerprint uploaded videos against a database of copyrighted content. Use audio fingerprinting (Audio ID) and video fingerprinting. Flag matching content for review or block. This is a separate, complex system.
  • How do you handle the transcoding queue during traffic spikes? A: Priority queue: paid/premium content first, then recently uploaded, then standard. Auto-scale transcoder workers based on queue depth. Use spot/preemptible GPU instances for cost efficiency.
  • How do you count views accurately? A: Count views in a streaming aggregator (Kafka Streams/Flink) with a deduplication window. A "view" requires watching at least 30 seconds or 50% of the video (whichever is shorter). Use HyperLogLog for approximate unique viewer counts at scale.
  • What resolution/codec choices do you make for transcoding? A: H.264 for broad compatibility (all devices). VP9/AV1 for efficiency (30-50% bandwidth savings) served to compatible browsers. Store multiple codecs; the player selects based on device capability.
What to Skip:
  • Don't design the actual video player (JavaScript/ExoPlayer) — assume a standard HLS player
  • Don't design the monetization/advertising system
  • Don't design content moderation or the copyright system in detail
  • Don't implement the recommendation ML model — describe the pipeline at a high level
