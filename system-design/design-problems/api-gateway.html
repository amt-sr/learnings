API Gateway

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design an API gateway that serves as the single entry point for all client requests to backend microservices. The gateway handles routing, authentication, rate limiting, caching, request/response transformation, protocol translation, and observability (logging, metrics, tracing) — decoupling cross-cutting concerns from individual services.

Functional Requirements

  • Route requests to the correct backend service based on path, method, headers
  • Authenticate and authorize requests (JWT validation, API keys, OAuth)
  • Rate limit requests per client, per endpoint, per tier
  • Cache responses for GET requests (configurable TTL per route)
  • Transform requests and responses (add/remove headers, body mapping)
  • Translate between protocols (HTTP → gRPC, WebSocket proxy)
  • Circuit breaking and load balancing across backend instances
  • Comprehensive logging, metrics, and distributed tracing

Non-Functional Requirements

  • Low latency: add < 5ms of overhead per request (gateway must be thin)
  • High throughput: handle 100K+ requests/sec per gateway instance
  • High availability: 99.99% uptime (gateway failure = all services down)
  • Scalability: horizontally scalable, stateless
  • Extensibility: add new routes/services without redeploying the gateway
  • Security: TLS termination, WAF integration, DDoS protection

Deep Dive: Request Processing Pipeline

Each request passes through a chain of filters/middleware before reaching the backend. The pipeline is configurable per route.

Request Pipeline (ordered filters): 1. [Request ID] → Generate unique request ID, inject into header (X-Request-ID) 2. [Logging] → Log request start (method, path, client_ip, request_id) 3. [Rate Limit] → Check rate limit, reject if exceeded (429) 4. [Auth] → Validate JWT/API key, inject user_id into header 5. [CORS] → Validate/insert CORS headers 6. [Transform] → Modify request (add headers, map body fields) 7. [Cache Lookup] → Check cache for GET requests, return 304/200 if hit 8. [Circuit Breaker]→ Check circuit state, reject if open (503) 9. [Load Balance] → Select backend instance (round-robin, least-conn) 10.[Proxy] → Forward request to selected backend 11.[Cache Store] → Store response in cache (GET 200 responses only) 12.[Transform] → Modify response (strip internal headers, add custom) 13.[Logging] → Log request completion (status, latency, backend) 14.[Metrics] → Record request metrics (latency histogram, status code) Response: ←─────────────────────────────────────────────────────────

Deep Dive: Routing and Configuration

Route configuration is stored centrally and pushed to gateway instances via a configuration service.

# Route matching (priority order):
# 1. Exact path match:   /api/v1/users/123
# 2. Prefix match:       /api/v1/users/  (for listing)
# 3. Regex match:         /api/v1/{service}/{resource} (dynamic routing)
# 4. Method match:       GET /api/v1/users/123 → user-service
#                        POST /api/v1/users → user-service

# Dynamic configuration reload:
# - Config Service (consul, etcd, or custom) stores route configs
# - Gateway instances watch for config changes (long-polling or push via gRPC)
# - On change: hot-reload routes without restart (atomic swap)
# - No rolling deployment needed for route changes
Hot reload: Gateway routes must be dynamically configurable without restarting. Use a watch-based configuration system (etcd watch, Consul template, or a custom config push service) that triggers hot-reload. New routes appear within seconds of configuration changes.

Deep Dive: Authentication

The gateway validates authentication before requests reach backend services.

MethodHow it WorksLatencyBest For
JWT (local validation)Gateway verifies JWT signature using public key cached locally< 1msService-to-service, internal APIs
API Key (Redis lookup)Gateway looks up API key in Redis to get associated identity and scopes~1-2msThird-party developers, public APIs
OAuth introspectionGateway calls auth service to validate token~10-50msExternal OAuth tokens
mTLSClient certificate validation< 1msService mesh, high-security
JWT optimization: Cache the auth server's public key (JWKS) locally and refresh periodically. Validate JWT signature locally without calling the auth server on every request. This reduces auth latency from ~10ms to < 1ms. The JWKS refreshes every 5 minutes via background fetch.

Deep Dive: Circuit Breaking

Protect backend services from cascading failures. When a backend is unhealthy, the circuit breaker opens and returns fast failures (503) instead of slow timeouts.

Deep Dive: Protocol Translation

The gateway can translate between protocols, allowing REST clients to communicate with gRPC services and vice versa.

REST Client ──► API Gateway ──► gRPC Service REST: POST /api/v1/users Body: { "name": "Alice", "email": "alice@example.com" } Accept: application/json Gateway translates to: gRPC: UserService.CreateUser proto: { name: "Alice", email: "alice@example.com" } Gateway translates response back: REST: 201 Created Body: { "user_id": "123", "name": "Alice" }
Protocol translation trade-off: Protocol translation adds complexity to the gateway (must understand both protocols). Alternative: use HTTP/2 with gRPC-Web, which allows browsers to call gRPC services directly. Or use a dedicated sidecar (Envoy) per service for translation. Keep the central gateway simple.

Common Interview Questions

  • How do you handle configuration updates without downtime? A: Hot reload — gateway watches for config changes (etcd watch, polling, or push via gRPC). New config is applied atomically without dropping in-flight requests. In-flight requests complete with the old config; new requests use the new config.
  • What happens when the gateway itself becomes the bottleneck? A: Scale horizontally (stateless instances). If individual features (rate limiting, auth) become bottlenecks, offload to dedicated services. Use a control plane / data plane split (like Envoy xDS) where the data plane is lightweight and the control plane handles heavy computation.
  • How do you prevent the gateway from being a single point of failure? A: Deploy multiple instances behind a load balancer. Health check all instances. Use blue-green or canary deployments for gateway updates. Circuit break on every backend route to prevent cascade failures from affecting the gateway process.
  • How does the gateway add tracing headers? A: Extract trace context from incoming request (W3C Trace Context, B3 headers). Inject trace ID into backend request headers. Pass through existing trace context for distributed tracing (Jaeger, Zipkin). Add the gateway's span to the trace.
  • When should you use an API gateway vs a service mesh? A: API gateway: north-south traffic (external clients → internal services), cross-cutting concerns for public APIs. Service mesh: east-west traffic (service-to-service), per-service features (mTLS, retry policies). In practice, use both: gateway for external edge, mesh for internal communication.
  • How do you handle long-running requests (streaming, WebSocket)? A: The gateway proxies WebSocket connections as a bidirectional tunnel. For SSE (Server-Sent Events), the gateway buffers the initial response headers and then streams the body. Set longer timeouts for streaming routes.
What to Skip:
  • Don't design the backend microservices — focus on the gateway itself
  • Don't implement specific filter plugins in code — describe the interface and behavior
  • Don't design the configuration management system in detail unless asked
  • Don't get into Web Application Firewall (WAF) rules — mention as an integration point
