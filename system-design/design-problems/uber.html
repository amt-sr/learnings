Uber

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Design a ride-sharing system similar to Uber that matches passengers with nearby drivers, handles real-time GPS tracking, calculates dynamic surge pricing, and manages trip lifecycle from request to payment. The system must handle geospatial queries at scale with sub-second latency.

Functional Requirements

  • Passenger requests a ride with pickup and drop-off locations
  • System finds and matches the nearest available drivers
  • Real-time GPS tracking of drivers and trip progress
  • Dynamic pricing (surge) based on supply/demand
  • Trip lifecycle: request → match → pickup → en-route → drop-off → payment
  • Driver earnings and passenger billing
  • Rating and feedback for both drivers and passengers

Non-Functional Requirements

  • Low latency: driver matching < 3s, GPS updates < 1s
  • High availability: 99.99% (riders stranded is a critical failure)
  • Scalability: handle millions of concurrent trips and GPS updates
  • Accuracy: driver location updates with < 50m accuracy
  • Reliability: no trip data loss, accurate billing

Deep Dive: Geospatial Index

The core challenge: efficiently find the nearest available drivers to a given pickup location.

Option 1: Geohash

Encode latitude/longitude into a short string. Nearby locations share prefix. Partition the world into grid cells.

Geohash precision levels: Precision 1: ~5,000 km × 5,000 km (continent) Precision 2: ~1,250 km × 625 km (country) Precision 4: ~39 km × 19.5 km (city) Precision 5: ~4.9 km × 4.9 km (district) Precision 6: ~1.2 km × 600 m (neighborhood) Precision 7: ~150 m × 150 m (street block) Precision 8: ~37 m × 19 m (building) Pickup at (37.7749, -122.4194): Geohash = "9q8yy" (precision 5: ~5km area) Search: find all drivers in "9q8yy" + adjacent cells (8 neighbors) → 9 cells × ~50 drivers/cell = ~450 candidate drivers Sort by actual Haversine distance → top 5

Option 2: Quadtree

Recursively subdivide 2D space into 4 quadrants. Each leaf node holds a bounded number of points. Search by traversing from root to the relevant quadrants.

Quadtree for San Francisco: Level 0: [ Entire SF Bay Area ] ┌────────┬────────┐ Level 1: │ NW SF │ NE SF │ ├────────┼────────┤ │ SW SF │ SE SF │ └────────┴────────┘ Subdivide dense areas further: Level 2: [Downtown] splits into 4 sub-quadrants Level 3: [Financial District] splits again Each leaf: max 50 drivers → Search: traverse to leaf containing pickup point → If leaf is sparse, expand to neighboring leaves
Recommended: Redis GEO (sorted sets with geohash). Redis provides built-in geospatial commands: GEOADD to index driver locations, GEORADIUS / GEORADIUSBYMEMBER to find nearby drivers within a radius, and GEOSEARCH for more flexible queries. This handles millions of driver locations with sub-millisecond latency. 
# Redis geospatial commands for driver location

# Driver updates location (every 3-5 seconds)
GEOADD drivers:available 37.7749 -122.4194 "driver:12345"
EXPIRE drivers:available:driver:12345 15  # TTL for stale locations

# Find 10 nearest available drivers to pickup location
GEORADIUS drivers:available 37.7749 -122.4194 3000 m COUNT 10 ASC
# Returns: ["driver:12345", "driver:67890", ...] sorted by distance

# When driver goes offline or gets matched
ZREM drivers:available "driver:12345"

Deep Dive: Driver Matching Algorithm

  1. Search: Query the geospatial index for available drivers within a radius (e.g., 3km) of the pickup location.
  2. Score: Rank candidates by a composite score: distance (40%), rating (20%), acceptance rate (20%), time since last trip (10%), surge zone bonus (10%).
  3. Dispatch: Send ride offers to top 5 candidates simultaneously (batched dispatch). First driver to accept gets the ride.
  4. Timeout: If no driver accepts within 15 seconds, expand search radius and re-dispatch.
  5. Fallback: After 3 rounds, show "No drivers available" with surge pricing indication.
Batch dispatch vs sequential: Sending offers to multiple drivers simultaneously reduces matching time but increases system load (driver receives multiple offers, may accept one and reject others). Sequential dispatch (one at a time) is fairer but slower. Hybrid: batch 3-5, sequential beyond that.

Deep Dive: Real-Time Location Tracking

Drivers update their GPS location every 3-5 seconds. Passengers see real-time driver movement.

  • Ingestion: Drivers POST location updates via HTTP (more reliable than WebSocket for high-frequency updates).
  • Storage: Redis GEO for the current state (latest location only). GPS history for active trips stored in a time-series store or Kafka (for replay, ETA calculation, and dispute resolution).
  • Propagation: When a driver in an active trip updates location, the trip service publishes to a Kafka topic. A WebSocket server subscribed to the topic pushes the update to the passenger.
  • Deduplication: Skip location updates if the driver hasn't moved significantly (< 10m) to reduce load.
Location accuracy challenges: GPS can be inaccurate in urban canyons, tunnels, and underground. Use sensor fusion (GPS + accelerometer + gyroscope) on the client to improve accuracy. Detect "jumps" (sudden large position changes) and filter them out server-side.

Deep Dive: Surge Pricing

Dynamic pricing based on real-time supply/demand imbalance.

surge_multiplier = f(demand, supply, time, weather, events)

Algorithm:
  1. Divide the city into surge zones (geohash precision 6, ~1km areas)
  2. For each zone:
     - demand = ride requests in last 10 minutes
     - supply = available drivers in the zone
     - ratio = demand / max(supply, 1)
  3. surge_multiplier = min(max(1.0, ratio * base_factor), 3.5)

  Smoothing:
  - Apply time-based smoothing to prevent rapid fluctuation
  - Never decrease surge by more than 20% per 5-minute interval
  - Allow immediate increase to respond to sudden demand spikes
Fairness concern: Surge pricing can lead to discrimination if not implemented carefully. Apply the same surge multiplier to all passengers in the same zone. Do not personalize pricing. Ensure surge zones are small enough for local precision but large enough to avoid gaming.

Common Interview Questions

  • How do you handle the case where a driver accepts a ride but doesn't move? A: Monitor driver movement after acceptance. If the driver doesn't start moving toward the pickup within 2 minutes, mark them as unresponsive, cancel the match, and re-dispatch to another driver. Penalize the driver's acceptance rate metric.
  • What happens when Redis (location store) fails? A: Deploy Redis in cluster mode (primary + replicas). If the entire cluster fails, fall back to database queries (PostgreSQL with PostGIS) — slower but functional. Buffer location updates locally on drivers and batch-upload on recovery.
  • How do you handle ETA calculations? A: Use a road network graph with real-time traffic data. Distance + average speed for the route. Use a routing engine (OSRM, Google Maps API) for turn-by-turn estimates. Cache route segments for common pickup-dropoff pairs.
  • How do you handle long-polling vs WebSocket for passenger tracking? A: WebSocket for active trips (real-time driver location). Long-polling for pre-trip (available drivers in area). Fallback to polling (every 3 seconds) if WebSocket disconnects.
  • How would you support shared rides (UberPool)? A: Additional matching constraints: route compatibility, pickup order optimization, detour time limit (max 5 min extra). Use a routing engine to compute the merged route. Dynamic pricing considers the route sharing discount.
What to Skip:
  • Don't design the payment system in detail — it's covered in a separate problem
  • Don't design driver onboarding, background checks, or vehicle inspection
  • Don't implement the actual routing/mapping algorithms — assume a service
  • Don't design the rating algorithm in detail unless asked
