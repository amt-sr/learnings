Distributed Lock

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design a mechanism for mutual exclusion across multiple machines/processes in a distributed system. The lock must prevent concurrent access to shared resources, handle node failures gracefully, and avoid deadlocks and race conditions.

Functional Requirements

  • Acquire an exclusive lock on a named resource
  • Release the lock when work is complete or on lease expiration
  • Support lease/TTL to prevent permanent lock holding on crash
  • Support re-entrant locks (same owner can reacquire)
  • Provide fairness (FIFO ordering) or priority-based acquisition
  • Support lock metadata (owner ID, purpose, timestamp)

Non-Functional Requirements

  • Safety: mutual exclusion must be guaranteed — at most one holder at any time
  • Liveness: lock must eventually become available even under failures
  • Low latency: acquire/release < 10ms in normal operation
  • Availability: lock service remains available during partial network partitions
  • Fault tolerance: survive minority node failures (with quorum)

Deep Dive: Redis SETNX-Based Lock

# Acquire lock (atomic SET NX with expiration)
SET lock:job:1 "client_A:abc123" NX EX 30

# Release lock (Lua script for atomic check-and-delete)
EVAL "
  if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('DEL', KEYS[1])
  else
    return 0
  end
" 1 lock:job:1 "client_A:abc123"

# Renew lease (extend TTL while still holding)
EVAL "
  if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('PEXPIRE', KEYS[1], ARGV[2])
  else
    return 0
  end
" 1 lock:job:1 "client_A:abc123" 30000
Why use Lua for release? A naive GET + DEL is not atomic. Between the GET (confirming ownership) and DEL, the lock might expire and another client could acquire it. The Lua script runs atomically in Redis, preventing this race condition.

Deep Dive: Redlock Algorithm

Redis single-instance locks have a critical flaw: if the Redis master crashes after a client acquires the lock, a promoted replica might not yet have the lock data. The Redlock algorithm addresses this by acquiring the lock on N independent Redis instances (typically 5) and requiring a majority (N/2 + 1 = 3) to succeed.

Client ──┬──► Redis-1 ── acquired ✓ ├──► Redis-2 ── acquired ✓ ├──► Redis-3 ── acquired ✓ (majority: 3/5) ├──► Redis-4 ── timeout ✗ └──► Redis-5 ── acquired ✓ Lock is granted (3 of 5 nodes agreed) Effective TTL = original_TTL - time_to_acquire
Redlock criticism (Martin Kleppmann): The Redlock algorithm relies on clock synchronization (GC pauses can cause lease expiry). Distributed systems cannot guarantee lock safety with time-based leases alone. For true safety, use fencing tokens (see below). In practice, Redlock works well for most use cases where approximate safety is acceptable.

Deep Dive: Fencing Tokens

Even with distributed locks, resource servers (databases, storage) need a mechanism to reject stale lock holders. A fencing token is a monotonically increasing number that the lock service assigns to each lock acquisition. The client includes the fencing token in every request to the resource server, which rejects requests with tokens older than the highest seen.

Lock Service Client Resource Server │ │ │ │ ACQUIRE("lock") │ │ │──────────────────►│ │ │ token=42 │ │ │◄──────────────────│ │ │ │ WRITE(data, tok=42) │ │ │─────────────────────►│ │ │ │ ✓ Accepted │ │ │ │ (network delay, │ │ │ client retries) │ │ │ │ WRITE(data, tok=42) │ │ │─────────────────────►│ │ │ │ ✗ Rejected (42 < 43)
Key insight: Fencing tokens move the safety guarantee to the resource server. Even if a client holds a stale lock (due to GC pause, network partition, or lease expiry), the resource server will reject its requests because the fencing token is outdated. This is the most robust approach for distributed mutual exclusion.

Deep Dive: ZooKeeper-Based Locks

ZooKeeper provides a different approach using ephemeral sequential znodes:

  1. Client creates an ephemeral sequential znode under a lock path: /locks/resource/lock-
  2. ZooKeeper appends a monotonically increasing sequence number: /locks/resource/lock-0000000001
  3. Client reads all children of /locks/resource
  4. If the client's znode has the lowest sequence number, it holds the lock
  5. Otherwise, it watches (sets a watcher on) the znode with the next-lower sequence number
  6. When that predecessor is deleted (lock released), the client is notified and checks again
Herd effect mitigation: Instead of watching the lock directory (which triggers all waiting clients on every change), each client watches only its immediate predecessor. This ensures O(1) notifications per lock change rather than O(N).

Common Interview Questions

  • What happens if a client acquires a lock and then crashes? A: The TTL/lease expires automatically, releasing the lock. The lock service does not rely on the client sending a RELEASE.
  • How do you prevent deadlock? A: TTL on every lock (lease). If a client crashes or is partitioned, the lease expires. Also, implement lock ordering (always acquire locks in the same order).
  • What is the difference between a lock and a lease? A: A lease is time-bounded authority to perform an action. A lock is mutual exclusion. A lease can expire; a lock should be explicitly released. In practice, distributed locks use leases for safety.
  • Why not use a database for locking? A: Databases are slower (disk-based commits), harder to make highly available, and don't natively support ephemeral nodes or automatic lease expiry. Use them only for low-frequency, coarse-grained locks.
  • How would you implement a re-entrant lock? A: Track the owner ID and a re-entrant count. On acquire by the same owner, increment count. On release, decrement; when count reaches 0, delete the lock key.
  • When would you choose ZooKeeper over Redis for locking? A: When you need strong consistency guarantees, FIFO fairness, automatic failure recovery, or the lock is used for leader election. ZooKeeper's ZAB consensus protocol provides stronger safety than Redis.
What to Skip:
  • Don't implement a full ZooKeeper or etcd cluster design — focus on the lock semantics
  • Don't design the resource server that uses the lock — only the lock service itself
  • Don't get into the full Redlock debate unless the interviewer steers there
  • Don't forget to discuss fencing tokens — it's a senior-level differentiator
