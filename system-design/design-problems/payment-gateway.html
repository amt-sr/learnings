Payment Gateway

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design a payment gateway that processes financial transactions between merchants and payment processors (banks, card networks). The system must guarantee ACID properties, handle idempotency, manage retries with exactly-once semantics, and support reconciliation with external financial systems.

Functional Requirements

  • Process payment requests: charge, refund, authorize, capture
  • Support multiple payment methods: credit card, debit card, bank transfer, digital wallet
  • Guarantee idempotency — duplicate requests must not result in duplicate charges
  • Handle retries for transient failures with exactly-once processing
  • Provide transaction status: pending, completed, failed, refunded
  • Support daily reconciliation with external payment processors
  • Webhook notifications for async transaction status updates

Non-Functional Requirements

  • ACID compliance: no double charges, no lost transactions, no inconsistent states
  • High availability: 99.99% uptime (revenue-critical system)
  • Low latency: synchronous charge < 3s, but prioritize correctness over speed
  • Auditability: complete audit trail for every state change
  • Compliance: PCI-DSS Level 1, SOC 2 Type II, data encryption at rest and in transit

Deep Dive: Idempotency

The single most important property of a payment system. A duplicate request (network retry, user double-click) must never result in a duplicate charge.

# Idempotency table
CREATE TABLE idempotency_keys (
    idempotency_key VARCHAR(128) PRIMARY KEY,
    merchant_id     BIGINT          NOT NULL,
    request_params  JSONB           NOT NULL,    -- Original request fingerprint
    transaction_id  VARCHAR(64)     NOT NULL,    -- Resulting transaction
    response_body   JSONB           NOT NULL,    -- Cached response
    status          VARCHAR(32)     NOT NULL,    -- processed, expired
    created_at      TIMESTAMP       NOT NULL,
    expires_at      TIMESTAMP       NOT NULL     -- Auto-cleanup after 24h
    -- Index for cleanup job
);
CREATE INDEX idx_idempotency_expiry ON idempotency_keys(expires_at);

# On every request:
# 1. Check idempotency_key table
#    - EXISTS + status='processed' → return cached response immediately
#    - EXISTS + status='processing' → return 409 Conflict (in-flight)
#    - NOT EXISTS → proceed with payment
# 2. INSERT idempotency_key with status='processing' (atomic)
# 3. Process payment
# 4. UPDATE idempotency_key with transaction_id + response + status='processed'
Key design decision: Use the client-provided idempotency key (in the header) as the deduplication mechanism. This shifts the responsibility to the client to generate a unique key per operation, which is simpler than server-side fingerprinting and works across retries caused by network failures (where the server never saw the first request).

Deep Dive: Transaction State Machine

┌──────────────┐ │ INITIATED │ └──────┬───────┘ │ ┌────────────┴────────────┐ │ │ ┌─────▼──────┐ ┌──────▼─────┐ │ PROCESSING │ │ FAILED │ └─────┬──────┘ │ (init │ │ │ failure) │ ┌─────┴──────┐ └─────────────┘ │ │ ┌────────▼──┐ ┌─────▼────────┐ │ COMPLETED │ │ FAILED │ │ (charged) │ │ (processor │ └─────┬─────┘ │ rejected) │ │ └──────────────┘ │ ┌─────┴──────┐ │ REFUNDED │ │ (partial │ │ or full) │ └───────────┘
Critical invariant: A transaction can only transition to specific states. Use a state machine with enforced transitions in the database (CHECK constraints or application-level guards). No transaction can go from COMPLETED back to PROCESSING. Every state transition is recorded in an immutable audit log.

Deep Dive: ACID Guarantees

Every payment operation must be atomic:

  1. Begin database transaction
  2. INSERT transaction record with status = INITIATED
  3. INSERT idempotency key record
  4. Commit — at this point, the intent is durably recorded
  5. Call payment processor API (external call)
  6. Begin new database transaction
  7. UPDATE transaction status to COMPLETED or FAILED based on processor response
  8. Commit
Two-phase commit pattern: The external processor call cannot be in a single database transaction. Instead, use a "prepare" phase (step 1-4) that records intent, then a "commit" phase (step 6-8) that records the result. If the system crashes between phases, a reconciliation job detects and resolves the inconsistency ( INITIATED records that never transitioned to COMPLETED/FAILED).

Deep Dive: Reconciliation

Detect and resolve discrepancies between internal records and external processor records.

  • Daily reconciliation: Download daily settlement files from the processor. Compare against internal transaction records. Flag discrepancies: amounts mismatch, missing transactions, phantom transactions.
  • Real-time monitoring: Track transaction age in PROCESSING state. Alert if a transaction has been processing for more than 5 minutes. Auto-retry or manual resolution based on configurable rules.
  • Settlement matching: Processor deposits funds in batches. Match settlements to internal transaction records to ensure the correct amount was received.
Reconciliation as a safety net: Even with perfect code, network failures, bugs, and edge cases will cause inconsistencies. Reconciliation is the last line of defense. Design it from day one, not as an afterthought. Automated reconciliation catches 99% of issues; manual review handles the rest.

Deep Dive: Retry with Exactly-Once

# Retry queue design
# Use a dedicated "retry" table instead of a message queue for payment operations
# (database-backed = ACID guarantees, no message loss)

CREATE TABLE retry_queue (
    retry_id        BIGSERIAL PRIMARY KEY,
    transaction_id  VARCHAR(64) NOT NULL,
    operation       VARCHAR(32) NOT NULL,  -- charge, refund, capture
    attempt_count   INT          NOT NULL DEFAULT 0,
    max_attempts    INT          NOT NULL DEFAULT 5,
    next_retry_at   TIMESTAMP    NOT NULL,
    params          JSONB        NOT NULL,
    created_at      TIMESTAMP    NOT NULL
);

# Retry worker:
# 1. SELECT * FROM retry_queue WHERE next_retry_at <= NOW() LIMIT 100
# 2. For each: execute the operation
#    - Success: DELETE from retry_queue
#    - Retryable failure: UPDATE attempt_count++, next_retry_at = NOW() + backoff
#    - Non-retryable failure: DELETE from retry_queue, mark transaction as FAILED
# 3. Use idempotency key on every retry to ensure exactly-once

Common Interview Questions

  • How do you handle a scenario where the processor charges the card but your system crashes before recording the result? A: The idempotency key ensures that on retry, we detect a potential duplicate. But since we don't have a record, we need to: (1) try a refund/reversal if the amount is known, (2) check the processor's API for the transaction status, (3) create a manual review ticket if unclear. This is why reconciliation exists.
  • What is the difference between authorize and capture? A: Authorize places a hold on funds (validity: 5-7 days). Capture charges the held funds. This two-step process allows merchants to verify order details before finalizing the charge. If not captured within the validity window, the hold expires automatically.
  • How do you handle multi-currency transactions? A: Store amounts in the merchant's base currency. Convert using a real-time FX rate at authorization time. Lock the rate for 24 hours to prevent discrepancies between authorize and capture. Use a separate FX rate service with daily rate updates.
  • How do you handle partial refunds? A: Track multiple refund records per transaction. Each refund reduces the refundable balance. Enforce: sum of refunds <= original charge amount. Use a separate refunds table linked to the transaction.
  • What security measures do you implement? A: PCI-DSS compliance (tokenize card data, never store CVV), TLS 1.3 for all communication, field-level encryption for PII, HMAC signatures on webhooks, IP allowlisting for processor callbacks, fraud detection scoring before processing.
What to Skip:
  • Don't design the full merchant onboarding flow
  • Don't design fraud detection in detail — mention it as a pre-processing step
  • Don't implement PCI-DSS compliance procedures — mention requirements and move on
  • Don't design the processor integration (Stripe API details) — treat it as a black box
