Metrics System

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★

Goal

Design a time-series metrics system (similar to Prometheus, Datadog, or CloudWatch) that collects, stores, aggregates, and visualizes metrics from thousands of services. The system must handle high cardinality data, support downsampling for long-term retention, and provide fast query performance across various time ranges.

Functional Requirements

  • Collect numeric metrics (counters, gauges, histograms, summaries) from services
  • Support arbitrary labels/tags (high cardinality) for dimensional queries
  • Support PromQL / similar query language for aggregation and analysis
  • Provide dashboards and alerting based on metric thresholds
  • Downsample data for cost-effective long-term storage
  • Support recording rules for pre-computed aggregations

Non-Functional Requirements

  • High ingestion rate: 10M+ data points/sec
  • Low query latency: < 1s for common dashboards, < 5s for complex queries
  • Storage efficiency: 10x compression over raw data
  • Scalability: handle millions of time series
  • Durability: no data loss for ingested metrics

Deep Dive: Data Model

A time series is uniquely identified by a metric name and a set of label-value pairs (dimensions).

Time series: {
  metric_name: "http_request_duration_seconds",
  labels: { method: "GET", path: "/api/users", status: "200" }
}

# Data points over time:
timestamp    | value
1701234550000 | 0.032
1701234560000 | 0.045
1701234570000 | 0.038
1701234580000 | 0.051
High cardinality is the enemy: Each unique combination of labels creates a new time series. If http_request_duration_seconds has labels for method (4), path (10K), status (100), and user_id (1M), the cardinality is 4 * 10K * 100 * 1M = 4 trillion time series. This will crash any metrics system. Enforce cardinality limits (e.g., max 100K per metric) and avoid high-cardinality labels (user_id, request_id) in metrics.

Deep Dive: Storage Engine

Time-series data has unique properties that enable efficient storage:

  • Append-only: New data points are always appended (time moves forward). No in-place updates.
  • Sorted by time: Data is naturally ordered by timestamp within each series.
  • Delta encoding: Values change slowly between consecutive points. Store deltas instead of absolute values.
  • Gorilla compression: Facebook's Gorilla paper describes XOR-based compression for floating-point time series, achieving 10-20x compression.
TechniqueCompression RatioDescription
Delta-of-delta timestamps~5xStore the difference between timestamp deltas
XOR floating-point values~10xGorilla: XOR current value with previous, store leading zeros + XOR value
Dictionary encoding (labels)~20xMap label values to integers, store integers instead of strings
General compression (Snappy/Zstd)2-3xApplied after column-specific encoding
Column-oriented storage: Store timestamps in one column and values in another. This enables efficient compression (homogeneous data in each column) and selective reading (e.g., read only the value column for aggregation, skip timestamps).

Deep Dive: Downsampling

Raw data at 15s resolution is expensive to store long-term. Downsample (roll up) to coarser resolutions over time.

Downsampling Pipeline: Raw data (15s intervals) │ │ After 7 days ▼ 1-minute rollups (avg, min, max, sum, count) │ │ After 30 days ▼ 5-minute rollups │ │ After 90 days ▼ 1-hour rollups │ │ After 1 year ▼ 1-day rollups (kept indefinitely) Storage reduction: - 15s raw: 4 points/min (baseline) - 1m: 1 point/min (4x reduction) - 5m: 0.2 points/min (20x reduction) - 1h: 0.017 points/min (240x reduction)
Rollup functions: Different aggregation functions are appropriate for different metric types. Counters → sum. Gauges → avg, min, max. Histograms → pre-computed percentile buckets (p50, p95, p99) from the raw histogram data.

Deep Dive: Query Engine

Support a PromQL-like query language for flexible metric analysis:

# Rate of HTTP requests (per second, over 5 min window)
rate(http_requests_total[5m])

# 95th percentile latency by service
histogram_quantile(0.95, rate(http_request_duration_seconds_bucket[5m]))

# Top 10 services by error rate (aggregation)
topk(10, sum(rate(http_requests_total{status=~"5.."}[5m])) by (service))

# CPU usage > 80% alert
100 * (1 - avg(rate(node_cpu_seconds{mode="idle"}[5m]))) > 80
Query optimization: Pre-computed recording rules avoid re-evaluating expensive queries on every dashboard refresh. For example, pre-compute rate(http_requests_total[5m]) every 15 seconds and store as a new time series. Dashboard queries read the pre-computed series instead of re-scanning raw data.

Common Interview Questions

  • How is this different from a logging system? A: Metrics are structured numeric data points indexed by time and labels. Logs are unstructured/semi-structured text. Metrics are highly compressible and support mathematical operations (rate, sum, percentile). Logs support full-text search. They serve different purposes.
  • How do you handle counter resets (process restart)? A: Use the rate() or increase() function which detects counter resets and handles them correctly by extrapolating the rate across the reset boundary.
  • What is the difference between pull and push collection? A: Pull (Prometheus-style): server scrapes /metrics endpoints at intervals. Pros: server controls load, easy service discovery. Cons: missed scrapes during server restart. Push (StatsD/Telegraf-style): clients send metrics to a gateway. Pros: no missed data. Cons: backpressure management, harder service discovery.
  • How do you handle a sudden spike in time series count? A: Cardinality limiter: reject new time series when total exceeds a threshold. Alert on cardinality growth. Auto-detect high-cardinality labels using sampling. Kill the offending time series and alert the team.
  • How do you support histograms? A: Store cumulative bucket counts: _bucket{le="0.1"}, _bucket{le="0.5"}, _bucket{le="1.0"}, _bucket{le="+Inf"}. The histogram_quantile() function interpolates percentiles from bucket counts. This avoids storing every individual value.
What to Skip:
  • Don't design the full dashboard UI — focus on the data pipeline
  • Don't implement the full PromQL parser — describe the key operations
  • Don't design distributed tracing (Jaeger/Zipkin) — it's a separate observability pillar
  • Don't get into service discovery mechanisms unless asked
