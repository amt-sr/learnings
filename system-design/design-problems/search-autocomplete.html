Search Autocomplete

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Design a search autocomplete system that suggests completions as the user types. The system must return ranked suggestions in under 50ms, handle millions of queries per second, and support personalization, trending topics, and multilingual content.

Functional Requirements

  • Return top-K suggestions (typically 5-10) as the user types each character
  • Rank suggestions by popularity, recency, and personal relevance
  • Support prefix matching (typing "goo" suggests "google", "google maps", "goodbye")
  • Support multilingual and Unicode content
  • Update suggestions in near real-time as new trending queries emerge
  • Support personalization based on user history and location

Non-Functional Requirements

  • Ultra-low latency: < 50ms p99 per keystroke
  • High throughput: 1M+ queries/sec
  • High availability: 99.99% uptime
  • Freshness: trending suggestions update within minutes
  • Scalability: millions of terms in the index

Deep Dive: Trie Data Structure

A trie (prefix tree) is the natural data structure for prefix search. Each node represents a character. A path from root to a node represents a prefix. Terminal nodes store the complete terms and their frequencies.

Trie for autocomplete: (root) ├── g ──── o ──── o ──── g ──── l ──── e "google" (freq: 9500000) │ └─── d ──── w ──── i ──── l ── l "goodwill" (900000) ├── g ──── o ──── o ──── g ──── l ──── e ──── " " ──── m ... "google maps" │ ... "google drive" │ ... "google translate" ├── a ──── p ──── p ──── l ──── e ──── " " ──── s ──── t ... "apple store" └── ... Query: "goo" → traverse g→o→o → collect all terms under this node Result: "google", "google maps", "google drive", "google translate", "goodwill" Ranked by frequency (descending)
Optimization: Top-K at each node. Instead of storing all terms under a node, store only the top-K (e.g., 10) most frequent completions at each trie node. This limits the traversal to O(K) regardless of how many terms share a prefix. The trie becomes a "top-K trie" where every node pre-computes its best suggestions.

Deep Dive: In-Memory Trie with Top-K

For sub-50ms latency, the trie must be entirely in memory.

class TrieNode:
    children: Map
    top_suggestions: PriorityQueue  # Top 10 by rank
    is_terminal: bool

# Lookup: "goo" → traverse 3 nodes → return top_suggestions
# Time: O(prefix_length) for traversal + O(1) for retrieval
# Space: O(N * avg_word_length) where N = number of unique terms
# For 50M terms, ~50M * 20 chars * 50 bytes/node ≈ 50 GB (with top-K storage)
Memory optimization: 50 GB for a single machine is large but feasible with modern servers (256GB RAM). For horizontal scaling, shard the trie by first character (26 shards for English, more for Unicode). Each shard holds ~2GB. Replicate each shard for availability.

Deep Dive: Ranking Strategy

Combine multiple signals to rank suggestions:

rank = w1 * log(query_frequency)           # Global popularity
     + w2 * recency_score                   # How recently the query became popular
     + w3 * personal_frequency              # How often this user searches this term
     + w4 * location_relevance              # Location-based boost
     + w5 * click_through_rate              # Historical CTR for this suggestion

Where:
  w1 = 0.4, w2 = 0.2, w3 = 0.2, w4 = 0.1, w5 = 0.1  (tunable weights)
Two-stage ranking: First stage (in the trie) uses pre-computed global rank. Second stage (in the ranking merger) adjusts scores based on personal signals. This separates the data structure from personalization and allows independent updates.

Deep Dive: Update Pipeline

Search query logs drive trie updates. The pipeline must refresh trending data frequently without disrupting reads.

  1. Collect: Query logs are written to Kafka in real-time.
  2. Aggregate: Stream processor (Flink) aggregates query frequencies over a time window (e.g., last 24 hours, last 7 days).
  3. Build: Build a new trie snapshot from aggregated data. This is done offline.
  4. Deploy: Atomic swap: replace the old trie with the new one. No downtime — use pointer swap (double buffering). The new trie is pre-loaded in memory before the swap.

Deep Dive: Handling Typo Tolerance

Users make typos. The autocomplete should still return relevant suggestions.

  • Edit distance: For short prefixes (< 3 chars), compute edit distance to known prefixes. Allow 1 edit for prefixes of length 2, 0 edits for single-char prefixes.
  • Phonetic matching: For proper nouns, use Soundex or Metaphone to match terms that sound similar.
  • Popular correction: If the user's prefix has no suggestions but a close prefix (1 edit away) has popular suggestions, suggest the corrected prefix.
Balancing typo tolerance and latency: Full edit distance computation for every keystroke is expensive. Pre-compute a "fuzzy index" that maps common misspellings to correct prefixes. Or use a probabilistic data structure (BK-tree) for bounded-distance searches.

Common Interview Questions

  • How do you handle Unicode and non-Latin scripts? A: The trie is character-agnostic — it works with any character set. For CJK languages, segment text into words before indexing. Use UTF-8 aware trie nodes. Shard by Unicode range for non-Latin scripts.
  • How do you personalize suggestions? A: Maintain per-user query history in Redis. On suggestion request, merge global suggestions with personal history. Boost personal terms. Use a separate lightweight "personal trie" per active user, or just a top-100 list per user.
  • What happens when the trie update is slow? A: The old trie continues serving. Updates are not blocking. If the pipeline is delayed (e.g., aggregation taking too long), serve stale results. Alert on pipeline lag. Accept eventual consistency for suggestions.
  • How do you handle trending spikes (sudden surge for a new term)? A: Use a separate "real-time trending" layer with a shorter time window (last hour). Merge trending results into the trie results with a boost. Trending terms are stored in Redis sorted sets with short TTLs.
  • Why not use Elasticsearch for autocomplete? A: You could — ES supports prefix queries and completion suggesters. However, for ultra-low latency (< 50ms) at 1M QPS, an in-memory trie is faster because it avoids network overhead and index lookup. ES is better when the corpus is too large for memory.
What to Skip:
  • Don't design the full search engine — focus only on autocomplete/prefix search
  • Don't implement spell-checking or "did you mean" — mention as a related but separate system
  • Don't design the query logging pipeline in detail
  • Don't over-focus on edge cases like empty queries or single-character inputs
