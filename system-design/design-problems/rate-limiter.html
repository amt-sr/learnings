Rate Limiter

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Design a distributed rate limiting service that controls the rate of incoming requests to protect backend services from overload, abuse, and DDoS attacks. The system must enforce limits across multiple server instances and support various rate limiting algorithms.

Functional Requirements

  • Enforce per-user, per-IP, or per-API-key rate limits
  • Support different rate limiting algorithms (token bucket, sliding window, fixed window)
  • Return appropriate HTTP responses (429 Too Many Requests) with retry-after headers
  • Support configurable rate limit rules (X requests per Y time window)
  • Provide rate limit status headers (X-RateLimit-Remaining, X-RateLimit-Reset)
  • Support hierarchical rate limits (per-user, per-tier, per-endpoint, global)

Non-Functional Requirements

  • Low latency: rate limit check < 2ms (must not add significant latency to request path)
  • High throughput: handle 1M+ checks/second
  • Distributed: consistent enforcement across all instances
  • Low memory: O(N) where N is the number of unique rate-limited entities
  • Available: fail-open (allow requests) when rate limiter is unavailable

Deep Dive: Rate Limiting Algorithms

1. Token Bucket

Each entity has a bucket that holds tokens. Tokens are added at a fixed rate (refill). Each request consumes one token. If the bucket is empty, the request is rejected.

Token Bucket (capacity=10, refill_rate=5/min) Time 0: [██████████] 10 tokens → Request: OK (9 remaining) Time 1m: [████████] 8 tokens → Request: OK (7 remaining) Time 2m: [██████] 6 tokens → Request: OK (5 remaining) Time 3m: [████] 4 tokens → Burst of 6: 4 OK, 2 rejected Key properties: - Allows bursty traffic up to bucket capacity - Average rate limited by refill rate - Memory: O(1) per entity (token count + last_refill_time) 
# Redis token bucket implementation
EVAL "
  local rate = tonumber(ARGV[1])
  local capacity = tonumber(ARGV[2])
  local now = tonumber(ARGV[3])
  local requested = tonumber(ARGV[4])

  local fill_time = capacity / rate
  local tokens = tonumber(redis.call('GET', KEYS[1]) or capacity)
  local last_time = tonumber(redis.call('GET', KEYS[2]) or now)

  local delta = math.max(0, now - last_time)
  tokens = math.min(capacity, tokens + (delta * rate))

  if tokens >= requested then
    tokens = tokens - requested
    redis.call('SET', KEYS[1], tokens)
    redis.call('SET', KEYS[2], now)
    redis.call('EXPIRE', KEYS[1], math.ceil(fill_time))
    redis.call('EXPIRE', KEYS[2], math.ceil(fill_time))
    return {1, tokens}
  else
    redis.call('SET', KEYS[1], tokens)
    redis.call('SET', KEYS[2], now)
    return {0, tokens}
  end

2. Sliding Window Log

Maintain a sorted log of all request timestamps within the current window. On each request, prune timestamps older than the window and check if the count exceeds the limit.

Pros: Exact enforcement (no boundary burst problem). Cons: O(N) memory where N = requests per window per entity. For 10M users at 100 requests/window, this is 1B entries — too expensive.

3. Sliding Window Counter (Recommended)

Hybrid of fixed window and sliding window. Weighted average of the current window and the previous window to approximate a sliding window with O(1) memory.

Sliding Window Counter (limit=100/hour): Previous window (60-120 min ago): 80 requests → 20% weight → 16 weighted Current window (0-60 min ago): 60 requests → 80% weight → 48 weighted Combined estimate: 64 requests Rate = 64/100 → 64% utilized → ALLOWED Formula: weighted_count = (current_count * elapsed_pct) + (prev_count * (1 - elapsed_pct)) where elapsed_pct = time_elapsed_in_current_window / window_size
Recommended for production: Sliding window counter provides a good balance of accuracy and memory efficiency. It eliminates the boundary burst problem of fixed windows while using only 2 counters per entity (current and previous window). Redis can store both counters in a single hash.

Deep Dive: Distributed Rate Limiting

With multiple server instances, each instance must enforce the same rate limit consistently. Two main approaches:

Centralized (Redis-based)

All instances check and update counters in a shared Redis cluster. Single source of truth, but adds network latency and creates a bottleneck at Redis.

Local + Sync

Each instance maintains a local rate limiter. Periodically sync state to peers. Lower latency, but may allow slightly more requests than the limit (acceptable for many use cases).

Fail-open vs Fail-closed: If Redis is unavailable, should requests be allowed (fail-open) or rejected (fail-closed)? For most API services, fail-open is preferred — availability over strict rate limiting. For security-sensitive endpoints (login, password reset), fail-closed is safer. Make this configurable per rule.

Common Interview Questions

  • How do you handle rate limiting when a user is behind a NAT (many users share one IP)? A: Don't rely solely on IP. Use API keys, tokens, or user IDs. For anonymous users, combine IP with browser fingerprinting or use a CAPTCHA.
  • How do you handle the "thundering herd" when many requests are unblocked simultaneously? A: Use jitter: add random delay (0-500ms) to the retry-after time. Or use token bucket which naturally smooths out re-admission.
  • How would you implement rate limiting at the API gateway level? A: Deploy rate limiter as a filter/middleware in the gateway. Use Redis cluster for state. Configure rules per route. Pass rate limit headers to clients.
  • What is the boundary burst problem with fixed windows? A: A user can make 2x the limit by sending requests at the boundary of two windows (e.g., 100 requests at :59 and 100 at :00 = 200 in 2 seconds). Sliding window algorithms eliminate this.
  • How do you handle rate limit propagation delays in a distributed setup? A: Accept approximate enforcement. Use a slightly conservative limit (e.g., enforce 95 instead of 100) to account for propagation lag. For strict enforcement, use Redis Lua scripts for atomic operations.
What to Skip:
  • Don't design the full API gateway — focus on the rate limiting component
  • Don't implement all algorithms in code — explain the tradeoffs and pick one
  • Don't over-design the management API — keep it simple
  • Don't forget to discuss fail-open vs fail-closed — this is a critical operational decision
