CQRS

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★

Goal

Understand the Command Query Responsibility Segregation pattern, how separating read and write models enables independent optimization, and when CQRS adds value versus unnecessary complexity.

Definition

CQRS (Command Query Responsibility Segregation) is a pattern that separates the data model for reads from the data model for writes. Commands mutate state; queries return data without side effects. Each model can use a different database, schema, and optimization strategy.

In a traditional CRUD system, the same model handles both reads and writes. CQRS splits this into two sides: a write side optimized for consistency and validation, and a read side optimized for query performance and flexibility.

Write Side vs Read Side

DimensionWrite SideRead Side
OptimizationConsistency, validation, business rulesQuery performance, flexibility
Data modelNormalized (3NF)Denormalized (star schema, flat docs)
DatabasePostgreSQL, MySQL (ACID)Elasticsearch, Redis, MongoDB
OperationsINSERT, UPDATE, DELETE (commands)SELECT only (queries)
ThroughputLower (write optimization)Higher (read optimization, caching)
ConsistencyStrong (single source of truth)Eventually consistent (synced via events)

Event Synchronization

The write side publishes events (e.g., OrderCreated, OrderUpdated). A projector subscribes to these events and updates the read model.

# Write side emits event:
event = OrderCreated { orderId: 123, userId: 456, items: [...], total: 99.99 }
eventBus.publish(event)

# Projector updates read model:
class OrderReadProjector:
    on(OrderCreated):
        readDB.insert({
            orderId: 123,
            userId: 456,
            itemCount: 3,
            total: 99.99,
            status: "pending"
            // denormalized: pre-computed fields for fast queries
        })

When CQRS Helps

  • Read-heavy workloads: Read:write ratio > 10:1. Optimize reads with denormalization, caching, and specialized databases.
  • Complex queries: Read queries require joins across many tables. Denormalize for fast access.
  • Different scaling needs: Reads need horizontal scaling with Elasticsearch; writes need ACID with PostgreSQL.
  • Different teams: Read optimization team and write logic team can work independently.

When CQRS Is Overkill

Do not use CQRS when: The read and write models are similar (simple CRUD), your system is small, your team is small, or you cannot tolerate eventual consistency. CQRS adds complexity: two databases, event synchronization, projector logic, and eventual consistency handling.
Interview tip: "For a system where 95% of traffic is reads and queries are complex (multi-field search, aggregations), I would use CQRS with PostgreSQL for writes and Elasticsearch for reads. The write side would publish events to Kafka, and a projector would update the Elasticsearch index. This gives me millisecond read latency with strong write consistency."

Common Interview Questions

  1. What is CQRS and what problem does it solve?
  2. How does the write side synchronize with the read side?
  3. What databases would you use for the write vs read side?
  4. When would you NOT use CQRS?
  5. How do you handle eventual consistency in CQRS?
  6. What is the relationship between CQRS and event sourcing?
  7. Compare CQRS with a traditional single-database approach.

What to Skip

Avoid diving into: Specific CQRS framework code (Axon, MediatR), event store implementation details, or projector error handling strategies. Focus on the architectural pattern, the separation rationale, and when to apply it.
