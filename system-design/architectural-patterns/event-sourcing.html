Event Sourcing

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★

Goal

Understand event sourcing as a persistence model where events are the source of truth, how to rebuild state from an event log, projections, snapshots, and when event sourcing provides value over traditional CRUD.

Definition

Event sourcing is a persistence pattern where instead of storing the current state of an entity, you store a sequence of events that led to that state. The current state is derived by replaying (rehydrating) events from the beginning.

In traditional CRUD, you overwrite state: UPDATE orders SET status='shipped' WHERE id=123. The previous state is lost. In event sourcing, you append an immutable event: OrderShipped{orderId: 123, timestamp: ...}. The full history is preserved forever.

Traditional State vs Event Sourced

TRADITIONAL (CRUD): ┌──────────────────────┐ │ orders table │ │ id: 123 │ │ status: "shipped" │ ← only current state │ total: 99.99 │ ← previous state LOST └──────────────────────┘ EVENT SOURCED: ┌──────────────────────────────────────────────────┐ │ event_log (append-only) │ │ [1] OrderCreated {orderId:123, items:[...]} │ │ [2] PaymentReceived {orderId:123, amount:99.99} │ │ [3] OrderShipped {orderId:123, trackingId:...} │ │ │ │ Current state = replay events 1→2→3: │ │ Order {id:123, status:"shipped", paid:true, ...}│ └──────────────────────────────────────────────────┘

Snapshots

Replaying thousands of events for every read is expensive. Snapshots periodically persist the current state, so replay only needs events since the last snapshot.

# Without snapshot: replay 10,000 events (slow)
# With snapshot: replay snapshot + 50 events (fast)

Snapshot at event 9950:
  Order {id: 123, status: "processing", items: [...], total: 450.00}

To get current state at event 10000:
  1. Load snapshot (event 9950)
  2. Replay events 9951-10000 (50 events)
  3. Current state = result

Snapshot frequency: every N events or time-based

Event Log Properties

PropertyDescriptionWhy It Matters
Append-onlyEvents are only appended, never modified or deletedImmutable audit trail
OrderedEvents for a single entity are sequencedReplay produces deterministic state
ImmutableOnce written, events cannot changePrevents data corruption, enables time travel
Schema versionedEvents include version for compatibilitySupports schema evolution over time

Advantages and Disadvantages

AdvantageDisadvantage
Complete audit trailIncreased complexity
Time travel (rebuild state at any point)Eventual consistency for projections
Natural fit for event-driven architectureEvent schema evolution is hard
Debugging: replay events to reproduce bugsStale events: correcting an event requires a compensating event
Multiple projections from same event logStorage growth: events accumulate forever
No data loss: all changes preservedLearning curve for the team
When to use: Financial systems (transactions), audit-critical domains (healthcare, compliance), domains where "what happened" is as important as "what is the current state," and systems that benefit from temporal queries ("what was the balance on March 15?").
Interview tip: "For a banking system, I would use event sourcing because every transaction must be auditable. The event log is the source of truth, and I can rebuild any account balance at any point in time. For read-heavy queries, I would project into a materialized view." This shows you know when to apply advanced patterns.

Common Interview Questions

  1. What is event sourcing and how does it differ from traditional CRUD?
  2. How do you rebuild state from events?
  3. What is a projection and why do you need multiple projections?
  4. What are snapshots and when do you use them?
  5. How do you handle event schema evolution?
  6. How do you correct a wrong event in an event-sourced system?
  7. When would you NOT use event sourcing?
  8. How does event sourcing relate to CQRS?

What to Skip

Avoid diving into: Event store implementation details (Kafka log compaction, custom event store), aggregate design (DDD terminology), or specific event store libraries (EventStoreDB). Focus on the conceptual model: events as source of truth, replay, projections, and trade-offs.
