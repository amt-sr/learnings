Materialized View

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand materialized views as pre-computed query results that avoid expensive joins and aggregations at read time. Learn refresh strategies (on-write, periodic, on-demand), denormalization patterns, and when materialized views are the right optimization.

Definition

Materialized view is a database object that stores the result of a complex query as a pre-computed table. Unlike a regular view (which is a saved query that runs on every access), a materialized view physically stores the computed data, making reads fast at the cost of maintaining the data through refresh operations.

Materialized views trade write overhead for read performance. Instead of joining 5 tables with aggregations on every read, the result is pre-computed and stored. When the source data changes, the view is refreshed to reflect the updates.

Database-Level Materialized Views

-- PostgreSQL materialized view
CREATE MATERIALIZED VIEW monthly_revenue AS
SELECT
  p.id AS product_id,
  p.name,
  SUM(oi.quantity * oi.price) AS revenue,
  COUNT(DISTINCT o.id) AS order_count
FROM products p
JOIN order_items oi ON oi.product_id = p.id
JOIN orders o ON o.id = oi.order_id
WHERE o.created_at >= date_trunc('month', now())
GROUP BY p.id, p.name;

-- Refresh periodically (e.g., every 5 minutes via cron)
REFRESH MATERIALIZED VIEW monthly_revenue;

-- Or concurrently (does not block reads)
REFRESH MATERIALIZED VIEW CONCURRENTLY monthly_revenue;

Application-Level Materialized Views

In distributed systems, materialized views are often maintained at the application level rather than using database features.

When to Use Materialized Views

  • Expensive reads: Queries that join many tables, compute aggregations, or scan large datasets.
  • Read-heavy workloads: 100:1 read:write ratio. Amortize write cost across many reads.
  • Dashboard/analytics data: Leaderboards, counters, summary statistics that update infrequently.
  • Search optimization: Pre-compute search indexes from normalized source data.
Interview tip: "For a Twitter-style feed, I would use a materialized view. When a user posts a tweet, I write the tweet to the tweets table AND fan-out to a pre-computed home timeline for their followers. This makes the feed query O(1) — just read the pre-computed timeline — instead of querying all followed users and sorting by time."

Common Interview Questions

  1. What is a materialized view and how is it different from a regular view?
  2. Compare the different refresh strategies.
  3. How would you design a materialized view for a news feed?
  4. What is the trade-off between read and write performance?
  5. How do you handle consistency between source data and materialized views?
  6. When would you use an application-level vs database-level materialized view?
  7. How does a materialized view relate to CQRS?

What to Skip

Avoid diving into: Database-specific syntax (Oracle, SQL Server materialized view options), view index internals, or CDC tool configuration. Focus on the pattern: pre-computed results, refresh strategies, and the read/write trade-off.
