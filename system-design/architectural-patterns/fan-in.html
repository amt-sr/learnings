Fan-In

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★

Goal

Understand the fan-in pattern where a single service or process aggregates results from multiple sources, the techniques for parallel aggregation, handling partial failures, and the relationship to scatter-gather.

Definition

Fan-in is a pattern where a service collects and aggregates results from multiple upstream services or data sources into a single unified response. It is the counterpart to fan-out: after scattering requests to multiple services, you fan-in the responses to form a complete answer.

Fan-in is inherent in most microservices architectures. When an API Gateway or BFF (Backend for Frontend) needs data from 3-5 downstream services, it fans out requests and fans in the responses to compose the final result.

Parallel Execution for Low Latency

SEQUENTIAL (slow): 150ms + 200ms + 80ms = 430ms total Aggregator → User Service (150ms) → Order Service (200ms) → Wallet (80ms) │ │ └───────────────────── 430ms ─────────────────────────┘ PARALLEL (fast): max(150ms, 200ms, 80ms) = 200ms total Aggregator → User Service (150ms) ─────┐ → Order Service (200ms) ─────┤── 200ms total → Wallet Service (80ms) ────┤ ┌──────────────────────────────────────┘ Implementation (Go-style): go fetchUser() → channel go fetchOrders() → channel go fetchWallet() → channel select { all done } → merge & return
Latency optimization: Always execute fan-in requests in parallel, not sequentially. The total latency equals the slowest response, not the sum. This is the most common interview optimization for APIs that aggregate data from multiple services.

Handling Partial Failures

When some upstream services fail, the aggregator must decide how to handle the incomplete response.

  • Fail the entire request: If any dependency fails, return 500. Simple but poor UX.
  • Return partial data: Return what is available with null/default for failed services. Client can handle gracefully.
  • Cached fallback: Serve stale cached data when the live service is unavailable. Maintains UX at the cost of staleness.
  • Circuit breaker + fallback: If a service is consistently failing, short-circuit and use fallback without waiting for timeout.
Interview tip: "When aggregating from 5 services, I would execute all requests in parallel with individual timeouts. If one service is slow, I would return the data from the other 4 services plus cached data for the slow service, rather than making the user wait for all 5."

Common Interview Questions

  1. What is fan-in and how does it relate to fan-out?
  2. How would you aggregate data from multiple services with minimal latency?
  3. What happens when one upstream service is slow?
  4. How do you handle partial failures in a fan-in aggregation?
  5. Compare sequential vs parallel fan-in.
  6. What is the relationship between fan-in and the BFF pattern?

What to Skip

Avoid diving into: Specific language concurrency primitives (goroutines, async/await syntax), or specific RPC framework call patterns. Focus on the architectural pattern, parallel execution, and failure handling strategies.
