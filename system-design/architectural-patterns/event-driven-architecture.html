Event-Driven Architecture

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Understand the event-driven architectural style, the difference between events and commands, producers and consumers, event bus/broker patterns, and the trade-offs of asynchronous communication in distributed systems.

Definition

Event-driven architecture (EDA) is a pattern where services communicate by producing and consuming events rather than making direct synchronous calls. An event represents a fact that something happened ("OrderPlaced", "PaymentProcessed"). Services react to events asynchronously, enabling loose coupling and independent scaling.

Event vs Command

DimensionEventCommand
Semantics"Something happened" (past tense)"Do something" (imperative)
ExampleOrderPlaced, UserRegisteredCreateOrder, SendEmail
Producer expectationNo expectation of response (fire-and-forget)May expect a response or acknowledgment
CouplingProducer does not know about consumersCommand sender knows (or expects) a handler
Failure handlingConsumer must handle failure independentlySender handles failure directly
Key insight: Events are facts. They describe what happened. Commands are requests. They ask for something to happen. In event-driven architecture, prefer events for state changes and commands for actions with expected outcomes.

Communication Patterns

Point-to-Point (Queue)

Producer ──▶ Queue ──▶ Consumer A (competing consumers) ──▶ Consumer B (one message goes to ONE consumer) Use case: Task distribution (email sending, image processing) Tools: SQS, RabbitMQ queues, Kafka with consumer groups

Publish-Subscribe (Topic)

Producer ──▶ Topic ──▶ Consumer Group A (all get every message) ──▶ Consumer Group B (all get every message) ──▶ Consumer Group C (all get every message) Use case: Event notification (OrderPlaced → inventory, shipping, analytics) Tools: Kafka topics, SNS, RabbitMQ fanout exchange, Google Pub/Sub

Advantages and Disadvantages

AdvantageWhy It Matters
Loose couplingProducer does not know about consumers. Add new consumers without changing producer.
Independent scalingProducer and consumers scale independently based on their own load.
ResilienceIf a consumer is down, events accumulate in the queue/broker. No lost messages.
ExtensibilityAdd new event consumers without touching existing services.
Asynchronous processingProducer does not wait for consumers. Lower latency for the caller.
DisadvantageWhy It Matters
Eventual consistencyNo immediate consistency. Consumers may be seconds/minutes behind.
Complex error handlingFailed events need retry, dead letter queues, manual intervention.
Observability challengeHard to trace event flow across multiple services.
Event schema evolutionChanging event format breaks consumers. Need versioning strategy.
Debugging difficultyHard to reproduce issues; event order and timing matter.
Infrastructure overheadNeed a message broker (Kafka, RabbitMQ) adding operational complexity.

Choosing Between Sync and Async

Use synchronous (REST/gRPC) when: The caller needs an immediate result (e.g., "what is this user's balance?"), the operation is fast and critical, or you need strong consistency guarantees.
Use asynchronous (events) when: The operation takes a long time (email, video processing), the caller does not need an immediate result, multiple services need to react independently, or you want to decouple the producer from consumers.
Interview tip: When designing a system, explicitly state your communication strategy. "For user-facing reads, I would use synchronous REST calls. For state-change operations like order placement, I would use events so that inventory, notification, and analytics services can react independently." This shows thoughtful architectural decision-making.

Common Interview Questions

  1. What is event-driven architecture and when would you use it?
  2. What is the difference between an event and a command?
  3. Compare point-to-point vs publish-subscribe messaging.
  4. What are the trade-offs of event-driven architecture?
  5. How do you handle failures in an event-driven system?
  6. What is a dead letter queue and when do you need one?
  7. How do you ensure exactly-once processing?
  8. When would you choose synchronous over asynchronous communication?

What to Skip

Avoid diving into: Kafka internals (partitions, ISR, ZooKeeper), RabbitMQ exchange types, or specific broker configurations. Focus on the architectural pattern and its trade-offs.
