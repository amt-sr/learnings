Fan-Out on Write

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Understand the fan-out-on-write pattern: pre-computing data for multiple consumers at write time to make reads fast. This is a critical pattern for designing news feeds, notification systems, and any read-heavy system.

Definition

Fan-out on write is a pattern where, when data is written, it is actively distributed (fanned out) to multiple pre-computed stores or consumers. The write is slower and more complex, but every read is fast because the data is already pre-computed and stored in the read-optimized location.

The core trade-off: do the work at write time (when one user acts) so that reads are cheap (when many users consume). This is the principle behind Twitter's home timeline, Facebook's news feed, and most notification systems.

Fan-Out on Write vs Fan-Out on Read

DimensionFan-Out on WriteFan-Out on Read
When work happensAt write timeAt read time
Write latencyHigher (fan-out to N destinations)Low (single write)
Read latencyLow (pre-computed, O(1))Higher (query + aggregate N sources)
Write throughputLimited by fan-out factorHigh
Read throughputHigh (simple reads)Limited by aggregation cost
ConsistencyStrong (data pre-written)Read-your-writes may lag
StorageHigher (denormalized copies)Lower (single source)
Best forCelebrity followers, read-heavyLow-follower accounts, write-heavy
Twitter's hybrid approach: Use fan-out-on-write for users with fewer followers (say, under 10K) and fan-out-on-read for celebrities (millions of followers). Writing to 1 million timelines on every tweet is impractical. Reading from 1000 followed users for a celebrity is feasible.

Implementation Strategies

Synchronous Fan-Out

# Writer directly pushes to all follower timelines
def post_tweet(user, tweet):
    db.save_tweet(tweet)                    # source of truth
    followers = db.get_followers(user.id)   # N followers
    for follower in followers:
        redis.lpush(f"timeline:{follower.id}", tweet.to_json())
    return success  # must wait for all N writes

Asynchronous Fan-Out (via Queue)

# Writer publishes to queue; workers handle fan-out
def post_tweet(user, tweet):
    db.save_tweet(tweet)
    queue.publish("tweet_fanout", {user_id: user.id, tweet_id: tweet.id})
    return success  # immediate response

# Worker processes queue message
def handle_fanout(message):
    followers = db.get_followers(message.user_id)
    for follower in followers:
        redis.lpush(f"timeline:{follower.id}", tweet.to_json())

Handling Scale

Interview tip: When designing a Twitter-style feed, explain the hybrid approach: "For users with under 10K followers, I fan-out on write to pre-compute home timelines in Redis. For celebrities with millions of followers, I fan-out on read — their tweets are stored in a special 'celebrity' bucket, and followers pull from it on read. This balances write cost with read performance."

What to Skip

Avoid diving into: Specific Redis list operations, Kafka partitioning for fan-out workers, or timeline pagination strategies. Focus on the architectural pattern, the write vs read trade-off, and the hybrid approach for handling varying fan-out factors.
