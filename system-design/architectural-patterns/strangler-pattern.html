Strangler Pattern

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★

Goal

Understand the strangler fig pattern as a strategy for incrementally migrating from a monolith to microservices without a big-bang rewrite, reducing risk and enabling continuous delivery during migration.

Definition

Strangler fig pattern is a migration strategy where new functionality is built as new services, and the old monolith is gradually "strangled" by redirecting traffic from its endpoints to the new services piece by piece. Named after strangler fig trees that gradually envelop and replace their host tree.

The pattern avoids the risk of a big-bang rewrite. Instead of rebuilding everything at once, you incrementally extract pieces of the monolith into standalone services, routing traffic to the new services while keeping the monolith running for the remaining functionality.

Migration Flow

PHASE 1: Monolith handles everything ┌──────────────────┐ │ Monolith │ │ /users │ │ /orders │ │ /payments │ │ /shipping │ │ /notifications │ └──────────────────┘ PHASE 2: Extract user service, route /users to new service ┌──────────────┐ │ API Gateway │──/users───▶ User Service (NEW) └──────┬───────┘ │ └──/orders, /payments, /shipping, /notifications──▶ Monolith (remaining) PHASE 3: Extract more services ┌──────────────┐ │ API Gateway │──/users───────▶ User Service │ │──/orders──────▶ Order Service (NEW) │ │──/payments────▶ Payment Service (NEW) └──────┬───────┘ │ └──/shipping, /notifications──▶ Monolith (remaining) PHASE 4: Everything extracted, monolith retired ┌──────────────┐ │ API Gateway │──/users──────────▶ User Service │ │──/orders─────────▶ Order Service │ │──/payments───────▶ Payment Service │ │──/shipping───────▶ Shipping Service │ │──/notifications──▶ Notification Service └──────────────┘

Key Mechanism: Routing Layer

The routing layer (API Gateway, reverse proxy, or feature flags) determines which requests go to the new service and which go to the old monolith. This can be path-based, feature-flagged, or percentage-based.

Routing strategies: 1. Path-based routing: /api/v2/users/* → User Service (new) /api/v1/users/* → Monolith (old, still handling some edge cases) 2. Feature flag: if (feature_enabled("new_user_service", user.id)): route to User Service else: route to Monolith 3. Percentage-based (canary): 10% of traffic → User Service (new) 90% of traffic → Monolith (old)

Migration Steps

StepActionRisk
1. Identify boundaryFind a well-defined module/feature to extractLow (planning only)
2. Create new serviceBuild the extracted functionality as a standalone serviceLow (parallel development)
3. Duplicate writesWrite to both monolith and new service; read from monolithMedium (dual writes)
4. Switch readsRead from new service; fall back to monolith on errorMedium (production traffic)
5. Switch writesWrite to new service only; monolith becomes read-only for this moduleHigh (full cutover)
6. Remove old codeDelete the extracted module from the monolithLow (cleanup)

Challenges

  • Shared database: The hardest part. The new service needs to share data with the monolith until the database is also split. Strategies: shared database initially, then split.
  • Transactional boundaries: Operations that span multiple modules in the monolith (in a single transaction) become distributed transactions when extracted.
  • Data migration: Moving data from monolith schema to new service schema requires careful migration with zero downtime.
  • Testing in production: Use canary releases and feature flags to validate new services with real traffic before full cutover.
Interview tip: When asked "how would you migrate a monolith to microservices?", the strangler pattern is the answer. "I would use the strangler fig pattern: incrementally extract modules into services, routing traffic via an API gateway. Start with the lowest-risk, most independent module. Use the parallel-write → switch-read → switch-write → remove-old-code sequence for each extraction."

Common Interview Questions

  1. What is the strangler fig pattern?
  2. How would you migrate a monolith to microservices?
  3. How do you handle the shared database during migration?
  4. What is the role of the API Gateway in the strangler pattern?
  5. How do you minimize risk during the migration?
  6. What is the order of module extraction?
  7. How do you handle transactions that span the old and new system?

What to Skip

Avoid diving into: Specific data migration tooling (AWS SCT, Flyway), database schema migration strategies, or organization change management. Focus on the incremental migration strategy and the routing mechanism.
