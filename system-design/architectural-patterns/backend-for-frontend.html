Backend for Frontend (BFF)

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand the BFF pattern where each frontend (web, mobile, partner API) has its own dedicated backend service that aggregates and tailors data for that specific client, avoiding a one-size-fits-all API.

Definition

Backend for Frontend (BFF) is a pattern where each frontend application (web, iOS, Android, partner API) has its own dedicated backend service. The BFF aggregates data from multiple backend services, transforms it, and returns it in the exact format the frontend needs.

Instead of a single API that tries to serve all clients, BFF creates per-client backends. The web BFF returns HTML-friendly responses with full data; the mobile BFF returns lightweight JSON with minimal data to conserve bandwidth; the partner BFF returns a different schema with rate limiting.

BFF vs API Gateway

DimensionAPI GatewayBFF
ScopeShared across all clientsOne per client type
ResponsibilityInfrastructure concerns (auth, rate limiting, routing)Business logic (aggregation, transformation)
Data shapingMinimal (pass-through)Significant (tailored per client)
OwnershipPlatform/infra teamFrontend team
CustomizationLow (one-size-fits-all)High (per-client)
They complement each other: An API Gateway handles cross-cutting infrastructure concerns for all clients. BFFs sit behind the gateway and handle client-specific business logic. In production, you typically have both: API Gateway → BFF → Core Services.

Why BFF Helps

  • Different data needs: Web needs 50 fields per page; mobile needs 15. A shared API either over-fetches for mobile (wasting bandwidth) or under-fetches for web (more round trips).
  • Different performance profiles: Mobile has high latency, low bandwidth. BFF can batch requests, compress more aggressively, and return minimal payloads.
  • Different features: iOS supports push notifications; web does not. BFF can handle push registration for iOS only.
  • Team autonomy: Each frontend team owns their BFF. No dependency on a shared API team for client-specific changes.
  • Versioning: Mobile apps cannot force users to update. BFF allows versioned endpoints so old mobile versions continue working.

Trade-offs

AdvantageDisadvantage
Tailored API per clientCode duplication across BFFs (similar aggregation logic)
Frontend team autonomyMore services to deploy and maintain
Reduced client-side complexityAdditional network hop (client → BFF → services)
Independent evolution per clientShared logic may diverge across BFFs
Interview tip: "I would use a BFF pattern when I have distinct frontend clients with very different needs — for example, a web app that needs rich data for rendering and a mobile app that needs minimal payloads. The web BFF would aggregate from 5 services with full data; the mobile BFF would aggregate the same services but return only the 10 fields needed."

Common Interview Questions

  1. What is the BFF pattern and why does it exist?
  2. Compare BFF with API Gateway — how are they different?
  3. What are the trade-offs of having a BFF per client type?
  4. How does the BFF pattern help mobile app development?
  5. What happens when BFFs share a lot of common logic?
  6. How do you handle API versioning with BFF?

What to Skip

Avoid diving into: GraphQL as an alternative to BFF (that is a separate topic), specific framework implementations (Next.js API routes as BFF), or team organizational structures. Focus on the architectural pattern and trade-offs.
