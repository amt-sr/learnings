Cache Miss

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What a cache miss is and why it happens
  • Origin pull / origin shield architecture
  • How cache population works after a miss
  • The cold start problem and how to mitigate it
  • Miss penalty and its impact on performance and cost

Definition

A cache miss occurs when the CDN edge server does not have the requested resource in its local cache and must fetch it from the origin server. Every cache miss results in a full round trip to the origin, higher latency for the requesting user, and increased load on the origin infrastructure.

Origin Pull

When a cache miss occurs, the CDN performs an origin pull — it makes an HTTP request to the origin server on behalf of the client, receives the response, caches it locally, and then serves it to the client.

Edge sends request to origin: │ │ GET /images/hero.jpg HTTP/1.1 │ │ Host: example.com │ │ X-Forwarded-For: 203.0.113.50 │ │ X-CDN-Request-ID: req_abc123 │ │ │ │ 2. Origin processes and responds: │ │ HTTP/1.1 200 OK │ │ Cache-Control: max-age=86400 │ │ ETag: "xyz789" │ │ Content-Type: image/jpeg │ │ [binary data...] │ │ │ │ 3. Edge stores response in local cache with metadata: │ │ Key: /images/hero.jpg │ │ TTL: 86400s (from Cache-Control) │ │ Size: 245 KB │ │ ETag: "xyz789" │ │ CachedAt: 2026-06-28T10:00:00Z │ │ │ │ 4. Edge serves cached response to waiting client │ └──────────────────────────────────────────────────────────────┘

Origin Shield

Without an origin shield, a popular new resource can cause a thundering herd — hundreds of edge servers all requesting the same resource from origin simultaneously after a cache invalidation.

WITHOUT Origin Shield (thundering herd): Edge (Tokyo) ─────────────────┐ Edge (London) ────────────────┤ Edge (Frankfurt) ─────────────┼── 500 concurrent requests for same asset! Edge (Sydney) ────────────────┤ Origin gets crushed Edge (Singapore) ─────────────┘ ▼ ┌────────┐ │ Origin │ ← 500 simultaneous requests! └────────┘ WITH Origin Shield: Edge (Tokyo) ──┐ Edge (London) ──┤ Edge (Frankfurt) ──┼──► ┌──────────────┐ ┌────────┐ Edge (Sydney) ──┤ │ Origin Shield │───►│ Origin │ Edge (Singapore) ──┘ │ (1 regional │ │ │ │ PoP caches) │ │ │ └──────────────┘ └────────┘ │ First request: shield MISS → origin fetch → cache in shield Next 499 requests: shield HIT → served instantly Origin receives: 1 request (not 500)
Origin shield is critical for large-scale systems. Cloudflare calls it the "Tiered Cache," Akamai calls it "Midgress." The origin shield sits between all edge PoPs and the origin. Even if 10,000 edge servers need the same resource after invalidation, only ONE request reaches the origin. The shield deduplicates concurrent requests and caches the response regionally.

Request Coalescing

Even within a single PoP, multiple clients may request the same uncached
resource simultaneously. CDNs use request coalescing to handle this:

  Client A ──GET /new-product.jpg──┐
  Client B ──GET /new-product.jpg──┤  All arrive within 100ms
  Client C ──GET /new-product.jpg──┘

  Edge Server behavior:
  1. Client A's request triggers origin fetch
  2. Client B and C's requests are QUEUED (not sent to origin)
  3. When origin responds, all three clients get the same response
  4. Result: 1 origin request instead of 3

  Cloudflare: "Request Coalescing"
  Fastly:     "Stale If Error + shielding"
  Akamai:     "SureRoute with prefetch"

Cache Population

After a cache miss, the CDN must store the response for future hits. The population strategy matters for both performance and cost.

StrategyDescriptionTrade-off
Populate on read (pull)Cache is populated when a client requests a resource that is not cachedFirst user pays latency penalty; simple to implement
Pre-warm / prefetch (push)Proactively fetch popular resources into edge caches before users request themRequires predicting demand; uses bandwidth but eliminates cold starts
Background refreshPeriodically re-fetch popular resources before TTL expiresUsers never see a miss for popular content; uses extra bandwidth

Cold Start Problem

The cold start problem occurs when a resource has never been requested at a particular edge PoP, so every new user at that location experiences a cache miss on their first request. This is especially painful after:

  • A new deployment or release (new asset URLs)
  • A cache purge that clears entries across all PoPs
  • Adding a new edge PoP to the CDN
  • A viral event that drives traffic from a region that has not accessed the content before
Cold Start Timeline after cache purge: Time Event User Latency Origin Load ────────────────────────────────────────────────────────────────── T+0s Cache purge issued — — T+1s First user requests asset ~300ms (miss) 1 request T+2s Second user requests asset ~300ms (miss) 2 concurrent T+3s Third user requests asset ~10ms (hit!) 0 requests T+4s+ All subsequent users ~10ms (hit) 0 requests Problem: During T+1s to T+2s, origin may receive a burst of requests before any edge server has populated its cache. Solution: Pre-warm the cache BEFORE purging: 1. Deploy new assets with new URLs (content hashing) 2. Trigger pre-warm requests to all major PoPs 3. THEN invalidate old asset URLs 4. Users seamlessly get new assets from edge (no cold start)
Interview tip: When asked about cold start, the strongest answer combines three techniques: (1) content-hashed URLs so old and new assets coexist, (2) pre-warming PoPs via the CDN's API before switching traffic, and (3) origin shield to absorb any remaining thundering herd.

Miss Penalty

Miss penalty is the cost incurred when a cache miss occurs. It has three dimensions:

DimensionImpact of MissTypical Cost
LatencyUser waits for full round trip to origin instead of local edge response200ms+ extra (cross-region), vs 10ms on cache hit
Origin loadEvery miss generates real work for the origin (CPU, DB queries, bandwidth)10-100x more expensive than serving from edge
Availability riskIf origin is degraded, all misses result in errors or timeoutsCascading failure if origin goes down
Example: E-commerce site during a flash sale

  Normal traffic: 10,000 requests/sec
  Cache hit ratio: 90%
  Origin load:    1,000 requests/sec (manageable)

  Flash sale traffic: 100,000 requests/sec
  Cache hit ratio: 90%  (cold content is cached)
  Origin load:     10,000 requests/sec (strain)

  If cache hit ratio drops to 70% during the sale (new products listed,
  many first-time requests):
  Origin load:     30,000 requests/sec → origin may crash

  With origin shield + request coalescing:
  Shield absorbs most duplicates
  Origin load:     ~3,000 requests/sec (manageable)
  Even at 70% hit ratio, the shield deduplicates concurrent requests

Reducing Cache Misses

  1. Content hashing in URLs — app.a1b2c3.js changes URL when content changes, allowing infinite TTL and near-100% hit ratio
  2. Pre-warming — use CDN APIs to pre-fetch into edge caches before traffic arrives
  3. Origin shield — deduplicate concurrent miss requests before they hit origin
  4. Longer TTLs with stale-while-revalidate — serve stale content while refreshing in the background
  5. Normalize cache keys — strip tracking parameters, sort query strings, ignore case differences
  6. Larger edge cache capacity — reduce LRU evictions for long-tail content

Common Interview Questions

  • "What happens on a cache miss?"
    The edge server checks its local cache, finds nothing, and performs an origin pull: it forwards the request to the origin server, receives the response, caches it locally (respecting Cache-Control headers), and serves it to the client. The miss penalty is an extra ~200ms of latency and origin server load.
  • "What is the cold start problem and how do you solve it?"
    Cold start happens when a resource is not cached at a particular edge PoP, so the first user at that location always experiences a miss. Solutions: pre-warm caches via CDN APIs, use origin shields to absorb thundering herds, deploy content with hashed URLs (so new assets don't invalidate old ones), and use background refresh for popular content.
  • "How would you handle a thundering herd after a cache purge?"
    Three layers of defense: (1) origin shield / tiered cache to deduplicate requests before they reach origin, (2) request coalescing within each PoP so only one origin request per resource is made, (3) serve stale content with stale-while-revalidate so users see instant responses while the cache repopulates in the background.
  • "Design a system where the origin must never be overwhelmed by cache misses."
    Use origin shield (regional caching layer), request coalescing at every PoP, circuit breaker pattern to shed load if origin is degraded, pre-warm caches before deploying new content, and implement rate limiting at the edge. The origin should handle a predictable, bounded load regardless of client traffic spikes.
