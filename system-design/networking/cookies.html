Cookies

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What cookies are and how they work
  • Cookie attributes (HttpOnly, Secure, SameSite) and their security implications
  • Cookies vs tokens (JWT) for authentication
  • Cookie-based session management

Definition

Cookies are small pieces of data stored by the browser and sent with every HTTP request to the originating domain. They're the primary mechanism for maintaining state (sessions, preferences, auth) in HTTP's otherwise stateless protocol.

How Cookies Work

1. Server sends Set-Cookie header with response: HTTP/1.1 200 OK Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict 2. Browser stores the cookie 3. Browser sends Cookie header with every subsequent request: GET /api/user HTTP/1.1 Cookie: session_id=abc123 4. Server reads cookie, looks up session in Redis/DB → Returns user data

Cookie Attributes

AttributePurposeSecurity Impact
HttpOnlyCookie inaccessible to JavaScript (document.cookie)Prevents XSS theft
SecureCookie only sent over HTTPSPrevents interception on HTTP
SameSite=StrictCookie only sent with same-site requestsPrevents CSRF
SameSite=LaxCookie sent with top-level navigations, not cross-siteBalances CSRF protection and usability
Path=/Cookie sent for all pathsScope limitation
Domain=.example.comShared across subdomainsScope control
Max-Age=3600Cookie expires in 1 hourLimits exposure window
Security rule: Always set HttpOnly and Secure on auth cookies. Set SameSite=Lax at minimum. Without these, your cookies are vulnerable to XSS (HttpOnly), interception (Secure), and CSRF (SameSite).

Cookies vs Tokens (JWT)

CookiesJWT Tokens
StorageBrowser-managed (auto-sent)Client-managed (localStorage/sessionStorage)
CSRFVulnerable (unless SameSite)Immune (not auto-sent)
XSSSafe with HttpOnlyVulnerable if in localStorage
MobilePoor supportWorks everywhere
StateServer-side (lookup session in store)Self-contained (stateless)
ScalabilityRequires session store (Redis)No server-side storage needed
InvalidationEasy (delete from store)Hard (need blacklist or short expiry)
Size limit4KB per cookieLarger (typically 1-2KB JWT)
Best practice for APIs: Use JWT in Authorization header for mobile/third-party APIs. Use HttpOnly cookies for browser-based web apps. For SPAs, consider both: HttpOnly cookie for auth + short-lived JWT for API calls.

Common Interview Questions

  • "What is SameSite cookie attribute?"
    Prevents the browser from sending cookies with cross-site requests. Strict = never sent cross-site. Lax = sent with top-level navigations (like clicking a link). Protects against CSRF attacks.
  • "Cookies vs JWT — which would you use?"
    Cookies for browser apps (HttpOnly + Secure + SameSite). JWT for mobile APIs or third-party integrations. For SPAs, both: HttpOnly cookie for session + JWT for API authorization.
  • "How do you prevent session hijacking via cookies?"
    HttpOnly (no XSS access), Secure (HTTPS only), SameSite (no CSRF), short session timeout, IP binding, rotate session ID after login, use HTTPS everywhere.

What to Skip

  • Third-party cookies (being phased out by browsers)
  • Cookie size limits per browser
  • document.cookie API details
