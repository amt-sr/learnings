WebSockets

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What WebSockets are and how full-duplex communication works
  • The HTTP upgrade handshake mechanism
  • WebSocket vs SSE vs HTTP polling trade-offs
  • Connection management at scale

Definition

WebSocket is a protocol providing full-duplex, bidirectional communication between client and server over a single TCP connection. Unlike HTTP's request-response model, once a WebSocket connection is established, either side can send messages at any time without the overhead of HTTP headers.

The Handshake

WebSocket connections start as an HTTP request that upgrades to the WebSocket protocol.

Step 1: HTTP Upgrade Request (client → server) ┌──────────────────────────────────────────────────────┐ │ GET /chat HTTP/1.1 │ │ Host: example.com │ │ Upgrade: websocket │ │ Connection: Upgrade │ │ Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ== │ │ Sec-WebSocket-Version: 13 │ └──────────────────────────────────────────────────────┘ Step 2: HTTP 101 Switching Protocols (server → client) ┌──────────────────────────────────────────────────────┐ │ HTTP/1.1 101 Switching Protocols │ │ Upgrade: websocket │ │ Connection: Upgrade │ │ Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo= │ └──────────────────────────────────────────────────────┘ Step 3: WebSocket connection established ┌────────┐ TCP connection ┌────────┐ │ Client │◄═══════ full-duplex link ═══════►│ Server │ │ │ send frames anytime, both ways │ │ └────────┘ └────────┘

After the handshake, the connection switches from HTTP to the WebSocket protocol. Messages are sent as lightweight frames (2-14 byte headers vs ~500-800 bytes for HTTP). The connection stays open until either side closes it.

Choosing the right technology: Use HTTP polling only for legacy compatibility. Use SSE for server-to-client push (notifications, live feeds) — simpler, auto-reconnects, works through proxies. Use WebSocket for true bidirectional communication (chat, multiplayer, collaborative editing) where the client also sends frequent messages.

Use Cases

Chat and Messaging

Client A ──WebSocket──► Chat Server ◄──WebSocket── Client B │ Message Broker (Kafka/Redis Pub-Sub) │ Chat DB (MongoDB) Flow: 1. Clients connect via WebSocket 2. Client A sends message → server 3. Server publishes to message broker 4. Server pushes to Client B via WebSocket 5. Server persists message to DB

Real-Time Dashboards

Stock prices, live sports scores, system metrics. Server pushes updates as they happen. WebSocket allows the client to also subscribe/unsubscribe to specific channels.

Collaborative Editing

Tools like Google Docs, Figma. Each keystroke/action is sent via WebSocket to an operational transform (OT) or CRDT server, which broadcasts to other collaborators.

Connection Management at Scale

Persistent WebSocket connections are the hardest part of scaling real-time systems. Each connection holds a TCP socket open on the server.

Connection servers — hold WebSocket connections, lightweight 2. Pub/Sub broker (Redis, Kafka) — route messages between servers 3. Presence service — track who is online (Redis sorted sets) 4. Load balancer — sticky sessions or connection-aware routing

Scaling Challenges

ChallengeSolution
Millions of connectionsMultiple WebSocket servers behind a load balancer. Each server holds ~10K-100K connections.
Cross-server messagingRedis Pub/Sub or Kafka to route messages between connection servers
Connection failuresExponential backoff reconnect, message queue for offline delivery
Load balancer timeoutsLong timeout configs, TCP health checks (not HTTP), ALB with WebSocket support
Memory per connection~10-50KB per connection — 1M connections = ~10-50GB. Use async/non-blocking I/O (Node.js, Go, Netty).
State managementExternalize connection state to Redis. Connection servers are stateless from a deploy perspective.
Production tip: Use TCP keep-alives and application-level heartbeats (ping/pong frames) to detect dead connections. Set heartbeat intervals to ~30s. Without heartbeats, half-open connections accumulate and consume resources. If a user's phone goes to sleep, the TCP connection may appear alive but be dead on one side.

Common Interview Questions

  • "How does the WebSocket handshake work?"
    Client sends an HTTP GET request with Upgrade: websocket and Connection: Upgrade headers, plus a Sec-WebSocket-Key. Server responds with HTTP 101 Switching Protocols and a Sec-WebSocket-Accept hash. The TCP connection then switches from HTTP to the WebSocket protocol.
  • "WebSocket vs HTTP polling vs SSE?"
    HTTP polling wastes bandwidth (repeated full HTTP requests). SSE is server-push only (good for notifications). WebSocket is bidirectional (good for chat, games). SSE auto-reconnects; WebSocket requires manual reconnection. See: Server-Sent Events.
  • "How would you scale WebSocket connections to millions of users?"
    Horizontal scaling: multiple WebSocket servers behind a load balancer. Redis Pub/Sub (or Kafka) to route messages between servers so that a message from a user on Server A reaches a user on Server B. Connection servers hold only connections — all state is externalized. Use sticky sessions at the LB so reconnects go to the same server when possible.
  • "How do you handle reconnection?"
    Client-side exponential backoff (1s, 2s, 4s, 8s, max 30s). Server-side: buffer messages for a short window (e.g., 30s) after disconnect. On reconnect, client sends last-seen sequence ID to catch up on missed messages.
  • "What happens when a WebSocket server goes down?"
    All connections on that server are lost. Clients detect the close and reconnect (exponential backoff). Load balancer routes reconnected clients to a healthy server. Message queues buffer pending messages. For mission-critical systems, use a persistent message queue (Kafka) so no messages are lost during failover.

What to Skip

  • WebSocket frame format details (opcode, masking, FIN bit) — know frames exist, not their binary structure
  • WebSocket compression (permessage-deflate) — mention it exists, skip implementation
  • WebSocket over HTTP/2 vs HTTP/1.1 debate — too niche
  • Socket.IO specific API — know it's a library built on WebSockets with fallbacks, not its API
