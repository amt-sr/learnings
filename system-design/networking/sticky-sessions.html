Sticky Sessions

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What sticky sessions (session affinity) are and how they work
  • Why they are generally an anti-pattern for stateless architectures
  • Implementation: cookie-based affinity, IP hash
  • Problems with scaling, failures, and load distribution

Definition

Sticky sessions (also called session affinity) ensure that a client's requests are always routed to the same backend server. The load balancer remembers which server handled a client's first request and sends all subsequent requests from that client to the same server.

Without Sticky Sessions (normal): Request 1 ──▶ Server A Request 2 ──▶ Server B ← Different server (session lost!) Request 3 ──▶ Server C ← Different server (session lost!) With Sticky Sessions: Request 1 ──▶ Server A (set cookie: SERVER=A) Request 2 ──▶ Server A (cookie: SERVER=A → route to A) Request 3 ──▶ Server A (cookie: SERVER=A → route to A) The LB "sticks" the client to Server A for all subsequent requests.

Why Sticky Sessions Exist

Historically, applications stored session state in local server memory. When a user logged in on Server A, their session data existed only on Server A. If the next request went to Server B, Server B had no session data and the user appeared logged out.

# Legacy (session in local memory):
Server A memory:
  session_id=abc123 → { user: "alice", cart: [...] }

# If request goes to Server B:
Server B memory:
  session_id=abc123 → NOT FOUND  → Redirect to login ❌

# This is why sticky sessions were invented:
# Force all requests with session_id=abc123 to Server A.

Implementation Methods

Cookie-Based Affinity

The load balancer sets a cookie on the first response (e.g., AWSALB, SERVERID). On subsequent requests, the client sends this cookie back, and the LB uses it to route to the same server.

# AWS ALB sticky session configuration:
# Stickiness enabled with duration-based cookie
# Cookie name: AWSALB
# Expiration: 1 hour (can be up to 7 days)

# Nginx sticky session:
upstream backend {
    ip_hash;  # or use sticky module
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}

# HAProxy:
# cookie SERVERID insert indirect nocache
# backend api
#     cookie SERVERID
#     server api1 10.0.1.10:8080 cookie api1
#     server api2 10.0.1.11:8080 cookie api2

IP Hash Affinity

The LB hashes the client's IP address and maps it to a specific server. This doesn't require cookies but breaks when clients are behind NAT (many clients share one IP) or change IPs (mobile networks).

MethodProsCons
Cookie-basedWorks with NAT, accurate per-clientRequires cookies, can't work with non-browser clients
IP HashNo cookies needed, simpleBreaks with NAT, mobile IP changes, unbalanced distribution

Why Sticky Sessions Are an Anti-Pattern

In modern stateless architectures, sticky sessions introduce significant problems:

Uneven Load Distribution

Problem: Some servers get more traffic than others Without stickiness (even distribution): Server A: ████████ (33%) Server B: ████████ (33%) Server C: ████████ (33%) With stickiness (uneven distribution): Server A: ████████████████████ (60%) ← High-traffic users Server B: ████████ (25%) Server C: ████ (15%) ← Idle

Server Failure Problems

When a server goes down, all clients "stuck" to it lose their session. The LB must redistribute them, which causes session loss anyway — defeating the purpose of stickiness.

Scaling Issues

Adding or removing servers invalidates existing sticky mappings. The load balancer must redistribute clients, and any in-memory sessions on removed servers are lost.

Deployments Become Harder

Rolling updates become risky. When you deploy a new version to Server A, all stuck clients experience a disruption. You can't gracefully migrate sessions between servers.

The right approach: Design your application to be stateless. Store session data in an external store (Redis, Memcached, DynamoDB) instead of local server memory. This eliminates the need for sticky sessions entirely and enables true horizontal scaling.

Stateless Architecture (The Solution)

Stateful (bad — requires sticky sessions): ┌────────┐ ┌──────────────────────┐ │ Client │───▶│ Server A (local mem) │ ← session data lives here │ │───▶│ Server B (local mem) │ ← no session for this client │ │───▶│ Server C (local mem) │ ← no session for this client └────────┘ └──────────────────────┘ Stateless (good — no sticky sessions needed): ┌────────┐ ┌──────────┐ ┌─────────────┐ │ Client │───▶│ Any Srv │───▶│ Redis │ ← session data │ │───▶│ Any Srv │───▶│ (shared) │ │ │───▶│ Any Srv │───▶│ │ └────────┘ └──────────┘ └─────────────┘ Session data is stored in Redis, not in server memory. Any server can handle any request because it reads session data from the shared store.
# Stateless session example:
# Login: server writes session to Redis
redis.set("session:abc123", json.dumps({
    "user_id": 42,
    "username": "alice",
    "roles": ["admin"]
}), ex=3600)

# Any subsequent request to any server:
# Server reads session from Redis
session = redis.get("session:abc123")
# Works on Server A, B, or C — no sticky session needed

When Sticky Sessions Are Acceptable

There are a few scenarios where sticky sessions are reasonable:

  • WebSocket connections — Long-lived connections benefit from affinity to avoid re-establishing state
  • File upload progress — Large uploads that can't be easily resumed
  • Legacy applications — When refactoring to stateless is not feasible
  • Temporary migration — During the transition from stateful to stateless architecture

Common Interview Questions

  • "What are sticky sessions and why are they problematic?"
    Sticky sessions (session affinity) route all requests from a client to the same backend server. They are problematic because they cause uneven load distribution, fail badly when a server goes down (all stuck clients lose sessions), make scaling harder, and complicate deployments. The better approach is to make the application stateless by storing session data in Redis or a database.
  • "How do you avoid needing sticky sessions?"
    Store all session state in an external data store (Redis, Memcached, DynamoDB) instead of local server memory. Each request reads session data from the shared store. Any server can handle any request, enabling proper load distribution and seamless scaling.
  • "When might you still use sticky sessions?"
    WebSocket connections (long-lived, stateful), large file uploads, legacy apps that can't be refactored, or as a temporary measure during migration to a stateless architecture.
  • "How do sticky sessions affect load distribution algorithms?"
    They override them. Even if you're using least connections or weighted round robin, sticky sessions will route returning clients to their "stuck" server regardless of current load. This can lead to hot spots where one server is overwhelmed while others are idle.
