Authoritative DNS

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • The role of authoritative DNS servers and how they store zone data
  • DNS record types: A, AAAA, CNAME, MX, NS, TXT, SRV, SOA
  • Major DNS providers: Route53, Cloudflare DNS, Google Cloud DNS
  • Time-to-live (TTL) and its impact on propagation and failover
  • How to choose DNS providers for high availability and low latency

What is Authoritative DNS?

An authoritative DNS server is the source of truth for a DNS zone. When a recursive resolver asks for api.example.com, the authoritative server for example.com is the one that returns the final answer. Unlike recursive resolvers, authoritative servers do not iterate — they only answer queries for domains they are explicitly configured to serve.

Key insight: Authoritative DNS is your control plane for routing traffic. In system design interviews, you use authoritative DNS to direct users to the right data center, implement load balancing, failover between regions, and integrate with CDNs. Choosing the wrong DNS provider or misconfiguring TTL can cause extended outages during failures.

CNAME gotcha: You cannot have a CNAME at the same level as other records (with rare exceptions like DNSSEC). For example, you cannot have both example.com CNAME cdn.cloudflare.com and example.com MX mail.example.com. This is a frequent interview trap. Use A records at the apex and CNAMEs for subdomains.

Multi-provider strategy: In a production system design, never rely on a single DNS provider. Use your primary provider (e.g., Route 53) for all zones and configure a secondary provider (e.g., Cloudflare, NS1) as a slave. If the primary goes down, the secondary can still serve queries. NS records should list servers from both providers.

Time to Live (TTL)

TTL controls how long a DNS record can be cached by resolvers, browsers, and operating systems. It is set by the authoritative server and communicated with every DNS response.

TTL ValueDurationUse CaseTrade-off
60s1 minuteFailover zones, active-active DRHigher resolver query volume
300s5 minutesServices needing frequent updatesModerate query volume
3600s1 hourStandard web servicesGood balance
86400s1 dayStatic infrastructure, apex domainsLow query volume, slow propagation
604800s1 weekNS records, infrastructure that never changesMinimal queries, very slow changes
TTL is the most dangerous knob in DNS: A TTL of 86400 means that during a failover, clients may continue hitting a dead server for up to 24 hours. For services requiring fast failover, TTL should be 60–300 seconds. But low TTL means higher query volume to your authoritative servers, increasing cost and risk of throttling.

TTL in Practice: Failover Scenario

Scenario: Your API server in us-east-1 fails. You need to failover to us-west-2.

  1. Health check detects failure (seconds)
  2. DNS provider updates A record: us-east-1 → us-west-2 IP (seconds)
  3. Cached resolvers still serve old IP until TTL expires:
     - TTL = 60s  → max 60s of stale traffic    (acceptable for most systems)
     - TTL = 300s → max 5min of stale traffic   (may cause errors)
     - TTL = 3600s → max 1hr of stale traffic    (extended outage)
     - TTL = 86400s → max 24hr of stale traffic  (catastrophic)

  4. Even after TTL expires, some clients re-cache from other resolvers
     that still have the stale record (cascading propagation delay)

  Total failover time = health check + DNS update + TTL + propagation
  With TTL=60s: ~70-90 seconds total
  With TTL=86400s: ~24+ hours total

DNSSEC

DNSSEC adds cryptographic signatures to DNS records, allowing resolvers to verify that the response has not been tampered with. It protects against DNS spoofing and cache poisoning attacks.

ComponentPurpose
DNSKEYPublic key for verifying signatures in the zone
RRSIGDigital signature attached to each record set
DSDelegation signer — links child zone to parent zone's trust chain
NSEC / NSEC3Proves a record does NOT exist (authenticated denial)
DNSSEC in interviews: Know that it exists and what problem it solves (integrity). Do not deep-dive into the key management ceremony or rollover mechanics unless explicitly asked. Most major DNS providers (Cloudflare, Route 53, Google Cloud DNS) offer one-click DNSSEC.

Common Interview Questions

  • "What DNS record types do you use when designing a web service?"
    A record for the main domain and API, CNAME for CDN-hosted assets, NS records for zone delegation, TXT for SPF/DKIM email security and domain verification. Example: example.com → A record, cdn.example.com → CNAME to Cloudflare, api.example.com → A record pointing to load balancer.
  • "How would you implement DNS-based load balancing?"
    Use weighted routing policies for canary/blue-green deployments, latency-based routing for global services, failover routing with health checks for disaster recovery. Route 53 and Cloudflare both support these natively.
  • "What TTL would you set for a production API?"
    300 seconds (5 minutes) as a starting point. Low enough for reasonable failover time, high enough to keep query volume manageable. For critical failover scenarios (active-active), use 60 seconds. For NS records, use higher values (86400+).
  • "How do you handle DNS failover in a multi-region system?"
    Use health-checked failover routing (Route 53 / Cloudflare), set TTL to 60–300 seconds, deploy application servers in both regions with independent databases (or replicated), use anycast DNS for global query routing, and configure secondary DNS provider as backup.
  • "Why can't you put a CNAME at the apex (example.com)?"
    RFC 1034 prohibits CNAME records from coexisting with any other record type at the same name. Since the apex usually has NS, SOA, and MX records, a CNAME is technically invalid there. Workarounds: use an A record directly, or use ANAME/ALIAS (Cloudflare/Route 53 specific) which resolves the CNAME at the DNS provider level and returns an A record to the client.

What to Skip

  • DNS zone transfer protocol (AXFR/IXFR) internals
  • RFC 1034/1035 specification details
  • Dynamic DNS (DDNS) update protocol
  • Reverse DNS (PTR records) unless asked about email deliverability
