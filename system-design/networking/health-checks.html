Health Checks

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What health checks are and why they are essential for load balancing
  • Active vs passive health checks and when to use each
  • Configuration parameters: interval, timeout, threshold, unhealthy detection
  • Circuit breaker pattern at the load balancer level

Definition

A health check is a mechanism where the load balancer periodically probes backend servers to determine if they are healthy and capable of serving traffic. When a server is detected as unhealthy, the load balancer stops routing requests to it. When it recovers, the LB gradually reintroduces traffic. Without health checks, a failing backend would receive traffic and cause errors for clients.

Active Health Checks

The load balancer proactively sends requests to each backend on a schedule to verify it is healthy. This is the most common approach.

HTTP Health Check

# Health check endpoint on your application
GET /health
Response: 200 OK {"status": "healthy", "db": "up", "cache": "up"}

GET /health/ready       → Deep health check (checks DB, Redis, etc.)
GET /health/live        → Liveness check (process is running)

# HAProxy active health check configuration
backend api_backend
    balance roundrobin
    server api1 10.0.1.10:8080 check inter 5s fall 3 rise 2
    server api2 10.0.1.11:8080 check inter 5s fall 3 rise 2
    server api3 10.0.1.12:8080 check inter 5s fall 3 rise 2

    option httpchk GET /health
    http-check expect status 200

TCP Health Check

The LB attempts to establish a TCP connection to the backend. If the connection succeeds, the server is healthy. Useful for non-HTTP services like databases and caches.

Passive Health Checks

The load balancer monitors actual client traffic to detect failures without sending dedicated probes. If a backend returns too many 5xx errors or takes too long to respond, the LB marks it as unhealthy.

Trade-off: Passive health checks are zero-overhead but have slower detection (depends on client traffic volume). If traffic is low, a failing server might not be detected quickly. Active checks add overhead but detect failures immediately regardless of traffic patterns. In production, use both together.

Design your health check endpoint carefully: A /health endpoint should be fast (under 100ms), lightweight (no expensive queries), and check only the essentials. Separate liveness (am I running?) from readiness (am I ready to serve?) — Kubernetes uses this pattern. A server that is alive but not ready should be removed from the LB pool until it becomes ready (e.g., waiting for cache warmup or DB migration).

Circuit Breaker at LB Level

A circuit breaker is an advanced form of health checking where the load balancer stops sending traffic to a backend after detecting a failure pattern, without waiting for explicit health check failures.

Circuit Breaker States: ┌───────────┐ failures exceed ┌───────────┐ │ CLOSED │ ──── threshold ──────▶│ OPEN │ │ (normal) │ │ (blocking)│ └───────────┘ └───────────┘ ▲ │ │ timeout expires │ │ (half-open test) │ └────────────────────────────────────┘ CLOSED: Traffic flows normally. Track failure rate. OPEN: All traffic is rejected/circuit-broken. No requests to backend. HALF-OPEN: Allow a few test requests. If they succeed → CLOSED. If they fail → back to OPEN. Example thresholds: - Trip circuit: 50% error rate in last 10 seconds - Reset timeout: 30 seconds - Half-open: allow 3 test requests
Why circuit breakers matter: Without a circuit breaker, when a backend is struggling, the LB keeps sending requests that fail. This wastes resources, increases latency for clients, and can cascade failures. A circuit breaker "fails fast" — it immediately rejects requests or routes to fallback, giving the struggling backend time to recover.

Graceful Draining

When a server needs to be taken offline (deploy, scaling down), the LB should drain it gracefully:

  1. Mark the server as draining (no new sessions/connections)
  2. Allow in-flight requests to complete (with a timeout)
  3. After the drain timeout, remove the server from the pool
# Nginx graceful drain:
# 1. Set server to draining mode
# 2. Existing connections complete
# 3. No new connections are routed to this server

# Kubernetes pod termination:
# 1. Send SIGTERM to pod
# 2. Pod marks itself not-ready (readiness probe fails)
# 3. LB stops routing new traffic
# 4. Pod finishes in-flight requests (grace period: 30s default)
# 5. Pod exits

Common Interview Questions

  • "What is a health check and why is it important?"
    The LB periodically probes backends to verify they can serve traffic. If a backend fails health checks, the LB stops routing to it. Without health checks, a crashing server would continue receiving requests and returning errors to clients.
  • "Active vs passive health checks — when would you use each?"
    Active checks send dedicated probes on a schedule — fast detection but adds overhead. Passive checks monitor real traffic — no overhead but slow to detect failures on low-traffic services. In production, use both: active for guaranteed detection speed, passive for supplementary coverage.
  • "How would you design a health check endpoint?"
    Separate /health/live (lightweight — process running check) from /health/ready (deep check — DB, Redis, external deps). The readiness check should be fast (<100ms). Return 200 for healthy, 503 for unhealthy. The LB should use the readiness endpoint so it routes away from servers that are alive but unable to serve requests.
  • "Explain the circuit breaker pattern."
    Monitor the error rate from a backend. When it exceeds a threshold (e.g., 50% errors), trip the circuit to OPEN — stop sending traffic. After a timeout, allow a few test requests (HALF-OPEN). If they succeed, close the circuit. If they fail, keep it open. This prevents cascading failures and gives struggling servers time to recover.
  • "How do you handle deployments without downtime?"
    Rolling update: deploy new version one server at a time. Each server is drained from the LB before the new version starts. Health checks ensure the new version is healthy before moving to the next server. Blue/green deployment: maintain two identical environments; switch LB traffic from blue to green after health checks pass on green.

What to Skip

  • Specific health check response body formats
  • DNS-based health checking (SRV records)
  • Gossip protocols for health checking (SWIM, Serf)
