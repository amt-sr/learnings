DNS Caching

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • DNS caching at browser, OS, and resolver levels — how each works and their typical TTL bounds
  • How TTL controls cache expiry and the trade-offs of short vs. long TTL
  • dns-prefetch and preconnect hints to reduce perceived DNS latency
  • DNS caching strategy in system design: reducing latency, controlling failover speed, managing query volume

DNS Caching Hierarchy

DNS is cached at multiple layers between the client and the authoritative server. A cache hit at any layer eliminates all downstream lookups. The layers are checked in order — closest to the client first.

DNS Caching Layers (checked in order): ┌──────────────────────────────────────────────────────────────────────┐ │ Client Machine │ │ │ │ Layer 1: Browser Cache │ │ ├─ Chrome: chrome://net-internals/#dns │ │ ├─ Firefox: network.dnsCacheExpiration (default: 60s) │ │ ├─ Cache size: ~1000 entries │ │ └─ TTL: capped by browser (often 60s to 5min) │ │ Hit → 0ms (no network at all) │ │ │ │ Layer 2: OS Resolver Cache │ │ ├─ macOS: mDNSResponder (scutil --dns) │ │ ├─ Linux: nscd, systemd-resolved │ │ ├─ Windows: DNS Client service │ │ ├─ /etc/hosts check (static override) │ │ └─ TTL: respects authoritative TTL │ │ Hit → 0ms (no network at all) │ │ │ └──────────────────────────┬───────────────────────────────────────────┘ │ Cache miss ▼ ┌──────────────────────────────────────────────────────────────────────┐ │ Layer 3: Recursive Resolver Cache │ │ ├─ ISP resolver or public resolver (8.8.8.8, 1.1.1.1) │ │ ├─ Shared cache — benefits ALL users of this resolver │ │ ├─ Cache size: millions of entries (Google: 1T+ queries/day) │ │ └─ TTL: respects authoritative TTL (may honor minimum) │ │ Hit → 1-5ms (single RTT to resolver) │ │ │ └──────────────────────────┬───────────────────────────────────────────┘ │ Cache miss ▼ ┌──────────────────────────────────────────────────────────────────────┐ │ Layer 4: Full Resolution │ │ ├─ Root server → TLD server → Authoritative server │ │ └─ Latency: 30-100ms (3 sequential RTTs) │ └──────────────────────────────────────────────────────────────────────┘

TTL and Cache Expiry

TTL (Time to Live) is the authoritative server's directive for how long any caching layer may store a record. Each layer independently tracks the remaining TTL and discards the record when it reaches zero.

TTL Expiry Example (TTL = 300 seconds): t=0s Authoritative returns: api.example.com → 203.0.113.42 (TTL=300) All caches store: remaining TTL = 300s t=60s Browser query → Browser cache HIT (remaining TTL: 240s) Resolver query → Resolver cache HIT (remaining TTL: 240s) t=180s Browser query → Browser cache HIT (remaining TTL: 120s) Resolver query → Resolver cache HIT (remaining TTL: 120s) t=300s Browser query → Browser cache EXPIRED → asks OS OS query → OS cache EXPIRED → asks Resolver Resolver query → Resolver cache EXPIRED → full resolution → 30-100ms After re-query at t=300s: New TTL clock starts: remaining TTL = 300s again Important: Some resolvers cap minimum TTL (e.g., Google 8.8.8.8 minimum = 60s) and browsers may cap maximum TTL (e.g., Chrome caps at 5 minutes).
Stale-while-revalidate pattern: Some resolvers (and modern browsers) implement stale-while-revalidate: when TTL expires, they immediately return the stale cached value while asynchronously re-querying the authoritative server. This masks the DNS resolution latency from the user at the cost of potentially serving a briefly outdated record.

Interview strategy: When asked "what TTL should I use?", the answer is always "it depends." State the trade-off clearly: low TTL for fast failover (with higher query volume and cost), high TTL for performance and cost savings (with slower failover). Ask the interviewer about the service's availability requirements.

dns-prefetch and preconnect

Browsers support resource hints that initiate DNS resolution (and TCP/TLS connections) before the browser actually needs the resource. This eliminates DNS latency from the critical rendering path.

dns-prefetch

Resolves the domain name only. The browser performs the DNS lookup and caches the result, so when the actual request is made, DNS resolution is already complete.

<!-- In <head> -->
<link rel="dns-prefetch" href="//api.example.com">
<link rel="dns-prefetch" href="//cdn.example.com">
<link rel="dns-prefetch" href="//analytics.example.com">

  What happens:
  1. Browser parses <head>, sees dns-prefetch
  2. Immediately sends DNS query for api.example.com
  3. Result cached in browser DNS cache
  4. Later, when JavaScript calls fetch("https://api.example.com/data"):
     → DNS cache HIT → skip DNS resolution → save 1-100ms

preconnect

Goes further than dns-prefetch — it also establishes the TCP connection and performs the TLS handshake, so the HTTP request can start immediately.

<!-- In <head> -->
<link rel="preconnect" href="https://api.example.com" crossorigin>

  What happens:
  1. Browser sees preconnect
  2. DNS lookup for api.example.com          (~0-100ms saved)
  3. TCP 3-way handshake to resolved IP      (~30-60ms saved)
  4. TLS handshake (if HTTPS + crossorigin)  (~50-150ms saved)
  5. Connection kept open (briefly)
  6. When actual request is made → uses existing connection

  preconnect = dns-prefetch + TCP handshake + TLS handshake
HintDNSTCPTLSLatency SavedBest For
dns-prefetchYesNoNo~1–100msDomains used later in page load (analytics, lazy images)
preconnectYesYesYes~80–300msCritical domains needed early (API calls, fonts, key resources)
preconnect cost: Each preconnect consumes a TCP connection slot. Browsers typically limit concurrent connections to 6 per origin. Over-using preconnect wastes resources. Rule of thumb: use preconnect for 2–4 critical origins, and dns-prefetch for the rest. Remove hints for domains loaded quickly anyway.

DNS Caching in System Design

In a system design interview, DNS caching decisions affect availability, performance, and cost at the architecture level.

Strategy: CDN Integration

Typical CDN DNS setup:

  static.example.com  CNAME  →  cdn.cloudflare.com.         (TTL: 300s)
  cdn.cloudflare.com  A      →  104.16.x.x (anycast IP)    (TTL: 300s)

  Client → Resolver → Cloudflare authoritative
            → Returns anycast IP for nearest edge PoP
            → Client connects to edge PoP → cache HIT → content served

  DNS caching here is critical:
  - The CNAME to cdn.cloudflare.com gets cached at the resolver
  - The A record for the anycast IP gets cached
  - Both should have moderate TTL (300s): long enough for cache hits,
    short enough that Cloudflare can shift traffic between PoPs

Strategy: Service Discovery in Microservices

Traditional DNS: TTL of 300s between microservices

  service-a.internal.example.com  →  10.0.1.10  (TTL: 300s)

  Problem: If service-a scales horizontally, new instances are not
  discoverable for up to 5 minutes due to cached DNS.

  Solutions:
  1. Short TTL (5-10s) — more queries but fast service discovery
  2. Client-side load balancing with DNS + local cache refresh
  3. Use a dedicated service mesh (Envoy, Istio) instead of DNS for
     internal service discovery — avoids TTL limitations entirely
  4. Use a DNS provider with push-based updates (Consul Template,
     Kubernetes CoreDNS with watch)

How to Flush DNS Cache

Browser:
  Chrome:  chrome://net-internals/#dns → "Clear host cache"
  Firefox: network.dnsCacheExpiration → set to 0 → refresh

macOS:
  sudo dscacheutil -flushcache && sudo killall -HUP mDNSResponder

Linux (systemd-resolved):
  sudo systemd-resolve --flush-caches

Linux (nscd):
  sudo nscd -i hosts

Windows:
  ipconfig /flushdns

Common Interview Questions

  • "How many layers of DNS cache exist?"
    Three main layers: browser cache (fastest, smallest scope), OS cache (machine-wide, respects TTL), and recursive resolver cache (shared across all clients using that resolver, largest scope). Each layer independently checks TTL before returning a cached result.
  • "What is the difference between dns-prefetch and preconnect?"
    dns-prefetch only performs the DNS lookup. preconnect performs DNS + TCP handshake + TLS handshake. Use preconnect for critical resources needed early in page load (saves ~80–300ms). Use dns-prefetch for non-critical domains to at least eliminate DNS latency.
  • "Your DNS provider goes down. What happens?"
    Clients continue working if they have a cached record (until TTL expires). After TTL expiry, all new and expired queries fail. Mitigation: use multiple DNS providers (primary + secondary NS records), use anycast for geographic redundancy, set low TTL on critical services, and use a DNS monitoring service with alerts.
  • "How do you prevent DNS caching from causing outages during deployment?"
    Before a planned IP change or failover: lower TTL to 60s several hours in advance (wait for existing caches to expire). Make the change. Verify propagation. Raise TTL back after confirming stability. For emergency failover, accept the TTL-window of stale traffic and ensure your application layer handles errors gracefully.
  • "Design a system that needs sub-second failover across regions."
    DNS alone cannot achieve sub-second failover (TTL is the floor). Combine DNS failover (TTL=60s) with client-side retry logic (try next region on connection failure), use anycast routing at the IP layer for instant failover, or use a service mesh that can reroute traffic at the L7 proxy layer within milliseconds.

What to Skip

  • Browser-specific DNS cache internals (Chrome's async DNS resolver, Mozilla TRR)
  • OS-specific resolver daemon configuration (nscd.conf, resolved.conf)
  • HTTP Resource Hints beyond dns-prefetch and preconnect (modulepreload, prefetch)
  • DNS cache poisoning attack details (unless asked about DNS security)
