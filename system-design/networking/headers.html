Headers

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • The most important HTTP headers for system design
  • Request headers vs response headers
  • How headers affect caching, security, and performance

Definition

HTTP headers are key-value pairs sent with requests and responses that convey metadata — content type, caching rules, authentication, compression, and more.

Headers in System Design

Request Flow: Client → CDN → LB → App Server → Database Key headers at each layer: Client → CDN: Accept-Encoding: gzip, br (compression) Cookie: session_id=abc123 (auth) Authorization: Bearer token (API auth) CDN → App: X-Forwarded-For: 10.0.1.5 (original client IP) X-Request-ID: req-12345 (tracing correlation) Host: api.example.com (virtual host routing) App → Database: (not HTTP — uses TCP protocol like PostgreSQL wire protocol)
Interview tip: When asked about API design, mention key headers: "For authentication, we'd use Authorization: Bearer. For rate limiting, we'd return 429 with Retry-After header. For caching, we'd set Cache-Control and ETag headers. For tracing, we'd propagate X-Request-ID across all services."

Common Interview Questions

  • "What headers would you set for a REST API?"
    Request: Authorization (auth), Content-Type (JSON), Accept (JSON). Response: Content-Type, Cache-Control, ETag, X-Request-ID for tracing.
  • "What's the difference between Content-Type and Accept?"
    Content-Type describes what you're sending (request body format). Accept describes what you want back (response format). Client sets Accept: application/json. Server responds with Content-Type: application/json.
  • "What's X-Forwarded-For?"
    A header added by proxies containing the original client IP. Without it, the server only sees the proxy's IP. Multiple proxies chain: X-Forwarded-For: client, proxy1, proxy2.

What to Skip

  • Full HTTP/1.1 header specification (50+ headers)
  • Obsolete headers (Pragma, TE, Transfer-Encoding details)
  • Cookie-specific headers (covered in Cookies)
