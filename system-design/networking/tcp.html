TCP

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What TCP is and how it differs from UDP
  • The TCP handshake (3-way) and teardown (4-way)
  • How TCP guarantees reliable delivery
  • Flow control, congestion control, and their system design implications
  • Why TCP matters for database connections, APIs, and microservices

Definition

TCP (Transmission Control Protocol) is a connection-oriented, reliable, ordered, byte-stream protocol at the transport layer. It guarantees that data arrives intact, in order, and without duplication.

Analogy: TCP is like certified mail — you get acknowledgment of delivery, the receiver can request retransmission if something is lost, and items arrive in order. UDP is like regular mail — faster but no guarantees.

Three-Way Handshake

Client Server ────── ────── 1. SYN ────────────────→ "I want to connect" seq=x (client's initial sequence number) 2. ←──────────────── SYN+ACK seq=y, ack=x+1 "OK, I acknowledge. Let's talk." (server's initial sequence number) 3. ACK ────────────────→ "Confirmed. Let's start." seq=x+1, ack=y+1 Connection established! Data transfer begins. Cost: 1 RTT for setup. Same datacenter: ~0.5ms. Cross-region: ~50-200ms. This is why connection reuse matters.
Why 3-way (not 2-way)? Prevents stale connections. A delayed SYN from a previous connection won't confuse the server — the server's SYN-ACK requires a matching ACK from the client.

Four-Way Teardown

Client Server ────── ────── 1. FIN ────────────────→ "I'm done sending" seq=u 2. ←──────────────── ACK ack=u+1 "Got it" 3. ←──────────────── FIN seq=w "I'm done too" 4. ACK ────────────────→ "Goodbye" ack=w+1 Client enters TIME_WAIT (2×MSL, typically 60s). Why? Ensures last ACK reached server. Prevents stale packets.

Reliability Mechanisms

MechanismHow It Works
Sequence numbersEvery byte is numbered. Receiver reorders if packets arrive out of order.
Acknowledgments (ACKs)Receiver confirms received bytes. Unacknowledged = retransmit.
RetransmissionSender retransmits unacknowledged segments after timeout.
ChecksumsEvery segment has a checksum for corruption detection.
Duplicate detectionReceiver ignores duplicate segments using sequence numbers.

Flow Control

Prevents a fast sender from overwhelming a slow receiver.

Sender → Receiver Receiver advertises a "window" (rwnd) = how much buffer space it has. rwnd = 64KB → sender can send up to 64KB without ACK Receiver ACKs 32KB → rwnd updated → sender sends more If receiver buffer fills: rwnd = 0 → sender pauses (zero window) Sender sends periodic probes to detect when window reopens.

Congestion Control

Prevents a sender from overwhelming the network itself.

Phases: 1. Slow Start: cwnd starts at 1 MSS, doubles each RTT 2. Congestion Avoidance: cwnd increases linearly (1 MSS per RTT) 3. Fast Retransmit: 3 duplicate ACKs → retransmit immediately 4. Fast Recovery: halve cwnd, continue with adjusted rate Congestion event (timeout or 3 dup ACKs): → cwnd drops to 1 MSS (timeout) or halves (fast recovery) → Slow start or congestion avoidance restarts
System design impact: When designing high-throughput systems, TCP congestion control means you can't just add bandwidth. The network between your services constrains throughput. Multi-hop microservice architectures suffer because each hop resets the congestion window. This is why connection pooling and keep-alive matter.

TIME_WAIT and Port Exhaustion

Problem: Server closing connection enters TIME_WAIT for 60s (2×MSL).
  At 10K connections/sec: up to 600K sockets in TIME_WAIT.
  Each uses memory and a port tuple.

Solutions:
  1. SO_REUSEADDR on server socket
  2. Use connection pooling (avoid closing/reopening)
  3. Increase ephemeral port range
  4. TCP_tw_reuse kernel parameter (careful — can cause issues)

Common Interview Questions

  • "How does TCP guarantee reliable delivery?"
    Sequence numbers (ordering), ACKs (acknowledgment), retransmission (timeout or 3 duplicate ACKs), checksums (corruption detection). Together they ensure all data arrives, in order, without corruption.
  • "What's the 3-way handshake?"
    SYN → SYN-ACK → ACK. Establishes both sides' sequence numbers and confirms both can send and receive. Costs 1 RTT.
  • "Why does HTTP need TCP?"
    HTTP needs reliable, ordered delivery. A web page's HTML, CSS, and JS must arrive completely and in order. TCP provides this guarantee.
  • "What's TIME_WAIT?"
    State after active close. Lasts 2×MSL (~60s). Ensures last ACK reached the peer and absorbs stale delayed packets. At high connection rates, TIME_WAIT accumulation can exhaust ports.
  • "TCP vs UDP — when would you use UDP?"
    When speed matters more than reliability: live video/audio streaming, online gaming, DNS queries, IoT sensor data, where occasional packet loss is acceptable but latency is critical.
