UDP

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What UDP is and why it exists alongside TCP

Definition

UDP (User Datagram Protocol) is a connectionless, best-effort transport protocol. It sends packets (datagrams) without establishing a connection, guaranteeing delivery, or preserving order.

Analogy: UDP is like shouting across a room. You say something and hope it's heard. You don't wait for acknowledgment, and you don't retry if someone missed it. It's fast but unreliable.

Key insight: UDP is not "worse" than TCP — it's a different tool for different requirements. When you need speed and can tolerate loss, UDP is the right choice. Many reliable protocols (QUIC, WebRTC, RTP) build their own reliability on top of UDP to avoid TCP's limitations (head-of-line blocking, congestion control overhead).

Building Reliability on UDP

Many protocols add reliability on top of UDP when they need both speed and guarantees.

QUIC:     Reliability + ordering + congestion control on UDP
          (used by HTTP/3, WebTransport)

WebRTC:   RTP/RTCP on UDP for real-time video/audio
          DTLS for encryption

RTP:      Real-time Transport Protocol (ordering + timing info)

Reliable UDP libraries:
  - ENet (gaming)
  - UDT (bulk data transfer)
  - QUIC libraries (quiche, ngtcp2)

Common Interview Questions

  • "When would you choose UDP over TCP?"
    When latency matters more than reliability: live streaming, gaming, VoIP, DNS queries. Also when you want to build your own reliability (QUIC) or when occasional data loss is acceptable (metrics, logs).
  • "Does UDP guarantee delivery?"
    No. Packets can be lost, reordered, or duplicated. The application must handle this if needed. Many reliable protocols (QUIC, RTP) implement their own delivery guarantees on top of UDP.
  • "Why does HTTP/3 use UDP?"
    To avoid TCP's head-of-line blocking across streams, reduce handshake overhead (0-RTT), enable connection migration, and implement congestion control in user-space for faster iteration.
  • "Why does DNS use UDP?"
    DNS queries are small (fits in one packet), single request-response, and speed matters. If the response is lost, the client re-queries. Large responses (>512 bytes) fall back to TCP.
