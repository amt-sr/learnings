HTTPS

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What HTTPS is and why it's non-negotiable in production
  • The layered stack: HTTP over TLS over TCP
  • Performance cost and why it's negligible with TLS 1.3
  • HTTP Strict Transport Security (HSTS)
  • How HTTPS fits into system architecture

Definition

HTTPS (HTTP Secure) is HTTP running over TLS. It encrypts all communication between client and server, preventing eavesdropping and tampering.

HTTPS = HTTP + TLS HTTP application layer → TLS encryption layer → TCP transport layer → IP network layer
In 2024, HTTPS is not optional: Google ranks HTTP sites lower in search. Browsers show "Not Secure" warnings. Mixed content (HTTP resources on HTTPS page) is blocked by default. All production APIs must use HTTPS.

The HTTPS Stack

Application Layer: HTTP Request/Response ───────────────────────────────────────── Security Layer: TLS (encryption + authentication) ───────────────────────────────────────── Transport Layer: TCP (reliable delivery) ───────────────────────────────────────── Network Layer: IP (routing) ───────────────────────────────────────── Link Layer: Ethernet/WiFi What changes with HTTPS: - HTTP headers are encrypted - Request/response body is encrypted - URL path and query params are encrypted (TLS 1.3) - Server identity is verified via certificate - Connection to port 443 instead of 80
Note: In TLS 1.3, the entire URL (including path and query string) is encrypted. In TLS 1.2, the SNI (Server Name Indication) was visible in plaintext — which leaks which domain you're visiting.

HTTPS Performance

Overhead:
  TCP handshake:       1 RTT
  TLS 1.2 handshake:  2 RTTs (total: 3 RTTs before first byte)
  TLS 1.3 handshake:  1 RTT  (total: 2 RTTs before first byte)
  TLS 1.3 0-RTT:      0 RTTs (cached session, total: 1 RTT)

  CPU overhead: ~1-2% with AES-NI hardware acceleration
  Memory overhead: ~10-50KB per connection (negligible)

  Conclusion: The performance argument against HTTPS is dead.
  TLS 1.3 + session resumption = nearly identical to plain HTTP.

HSTS (HTTP Strict Transport Security)

HSTS tells browsers to only use HTTPS for a domain, preventing downgrade attacks.

Strict-Transport-Security: max-age=31536000; includeSubDomains; preload

max-age=31536000  →  1 year: browser only uses HTTPS
includeSubDomains →  applies to all subdomains
preload            →  added to HSTS preload list (shipped in browsers)

Without HSTS: attacker can intercept first HTTP request and
redirect to HTTP version (strip TLS). HSTS prevents this.

Common Interview Questions

  • "What's the difference between HTTP and HTTPS?"
    HTTPS = HTTP + TLS. Everything HTTP does, plus encryption (confidentiality), server authentication (identity verification via certificates), and integrity (tamper detection). Uses port 443 instead of 80.
  • "Is HTTPS slower than HTTP?"
    Marginally — TLS 1.3 adds 1 RTT for handshake (0 with session resumption). CPU overhead is ~1-2% with AES-NI. The performance difference is negligible for all practical purposes.
  • "What's HSTS?"
    HTTP Strict Transport Security — a header telling the browser to only use HTTPS for this domain. Prevents SSL stripping attacks. max-age specifies how long to remember.
  • "Where does TLS termination happen?"
    At the load balancer or CDN. Backend servers receive plain HTTP. For internal zero-trust, use mutual TLS (mTLS) via service mesh.

What to Skip

  • TLS 1.0 and 1.1 (deprecated, insecure)
  • SSL (predecessor, completely broken)
  • Certificate pinning details
