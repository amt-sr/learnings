HTTP/1.1

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • Key features of HTTP/1.1 that made it the standard for 20+ years
  • Head-of-line blocking and its impact on performance
  • Pipelining — the failed attempt to solve HOL blocking
  • Why HTTP/2 was needed

Definition

HTTP/1.1 (1997, RFC 7230) is the most widely used HTTP version. Its key innovation over HTTP/1.0: persistent connections (keep-alive) by default and pipelining support.

Key Features

FeatureDescription
Persistent connectionsDefault keep-alive — reuse TCP connection for multiple requests
Chunked transfer encodingSend response in chunks without knowing total size upfront (streaming)
Host headerRequired — enables virtual hosting (multiple domains on one IP)
PipeliningSend multiple requests without waiting for responses (theoretically)
Cache controlCache-Control header for fine-grained caching
Byte rangesRequest partial content (Range header — resumable downloads)

Head-of-Line Blocking

HTTP/1.1's biggest limitation. A single TCP connection can only handle one request at a time. If a large response is being sent, all subsequent requests must wait.

TCP Connection 1 (HTTP/1.1): Request 1 (large image) → [...............sending...............] Request 2 (small JS) → BLOCKED (waiting for Request 1 to finish) Request 3 (CSS) → BLOCKED (waiting for Requests 1 and 2) Solution: Open 6 TCP connections (browser default): Connection 1: Request 1 → [sending image] Connection 2: Request 2 → [small JS sent, response received ✓] Connection 3: Request 3 → [CSS sent, response received ✓] Connection 4: Request 4 → ... Connection 5: Request 5 → ... Connection 6: Request 6 → ... Problem: 6 connections per domain × multiple domains = hundreds of TCP connections. Wasteful and slow to set up.
Why this matters: Head-of-line blocking is THE reason HTTP/2 exists. HTTP/2 multiplexing solves this by allowing unlimited concurrent requests on a single TCP connection. Browsers open 6 connections per domain specifically to work around HTTP/1.1's HOL blocking.

Pipelining (Failed Solution)

HTTP/1.1 pipelining: Send multiple requests without waiting for responses.

  Request 1 ──→
  Request 2 ──→  (sent before Response 1 arrives)
  Request 3 ──→
  ←──────── Response 1
  ←──────── Response 2
  ←──────── Response 3

Why it failed:
  ✗ Responses must arrive in order (still HOL blocked)
  ✗ Many servers/proxies didn't implement it correctly
  ✗ No performance benefit if responses arrive out of order
  → Pipelining is effectively dead. No major browser uses it.

Common Interview Questions

  • "What's head-of-line blocking in HTTP/1.1?"
    On a single TCP connection, requests are queued. A slow response blocks all subsequent requests. Browsers work around this by opening 6 parallel connections, but that wastes resources and doesn't scale.
  • "Why do browsers open 6 connections per domain?"
    To work around HTTP/1.1's head-of-line blocking. With 6 connections, 6 requests can be in-flight simultaneously. RFC 7230 recommends this limit. HTTP/2 eliminates this need with multiplexing.
