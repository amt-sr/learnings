GraphQL

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What GraphQL is and how queries, mutations, and subscriptions work
  • Schema-first design and the resolver pattern
  • The N+1 problem and DataLoader solution
  • When to choose GraphQL over REST or gRPC

Definition

GraphQL is a query language for APIs and a runtime for executing those queries. Clients request exactly the data they need — no more, no less. Unlike REST's multiple fixed endpoints, GraphQL exposes a single endpoint with a typed schema that describes all available data and relationships.

Core Operations

Three operation types: QUERY — read data (like GET) ┌──────────────────────────────────────────────────────┐ │ query { │ │ user(id: "123") { │ │ name │ │ email │ │ posts(first: 5) { │ │ title │ │ comments { author { name } } │ │ } │ │ } │ │ } │ └──────────────────────────────────────────────────────┘ MUTATION — write data (like POST/PUT/DELETE) ┌──────────────────────────────────────────────────────┐ │ mutation { │ │ createPost(input: { title: "Hello", userId: 123 })│ │ { post { id, title } } │ │ } │ └──────────────────────────────────────────────────────┘ SUBSCRIPTION — real-time data (like WebSocket) ┌──────────────────────────────────────────────────────┐ │ subscription { │ │ onNewPost { id, title, author { name } } │ │ } │ └──────────────────────────────────────────────────────┘

Schema-First Design

The schema is the contract between client and server. It defines types, fields, and relationships. Both sides agree on the schema first, then implement independently.

type User {
  id: ID!
  name: String!
  email: String!
  posts: [Post!]!
}

type Post {
  id: ID!
  title: String!
  author: User!
  comments: [Comment!]!
}

type Query {
  user(id: ID!): User
  users(filter: UserFilter): [User!]!
  post(id: ID!): Post
}

type Mutation {
  createPost(input: CreatePostInput!): Post!
  updatePost(id: ID!, input: UpdatePostInput!): Post!
}

type Subscription {
  onNewPost: Post!
  onUserUpdated(id: ID!): User!
}

Resolvers

Resolvers are functions that resolve the actual data for each field in the schema. For each field, GraphQL calls its resolver to fetch the data.

const resolvers = {
  Query: {
    user: (parent, { id }, context) => context.db.users.findById(id),
  },
  User: {
    posts: (parent, args, context) =>
      context.db.posts.findByAuthorId(parent.id),
    // This causes the N+1 problem if called per-user!
  },
  Post: {
    comments: (parent, args, context) =>
      context.db.comments.findByPostId(parent.id),
  },
};

N+1 Problem and DataLoader

The N+1 problem is the most common GraphQL performance pitfall. When resolving nested fields, a naive implementation fires one query per parent record.

Request: { users(first: 100) { posts { title } } }

Naive resolvers execute:
  1 query → SELECT * FROM users LIMIT 100       (returns 100 users)
  100 queries → SELECT * FROM posts WHERE user_id = ?  (one per user!)
  = 101 total queries

With DataLoader:
  1 query → SELECT * FROM users LIMIT 100
  1 query → SELECT * FROM posts WHERE user_id IN (1,2,3,...,100)
  = 2 total queries
DataLoader batches and deduplicates database calls within a single GraphQL request tick. It collects all IDs for a given field, executes a single batched query, then distributes results back to individual resolvers. 
const userLoader = new DataLoader(async (ids) => {
  const users = await db.users.findAll({ where: { id: ids } });
  return ids.map(id => users.find(u => u.id === id));
});

const resolvers = {
  User: {
    posts: (user, args, context) =>
      context.postLoader.loadMany(user.postIds),
  },
};

When to Use GraphQL vs REST vs gRPC

Decision framework:
  • REST: Public APIs, simple CRUD, when HTTP caching matters, small teams. Fastest to get started.
  • GraphQL: Frontend-heavy apps with complex data needs, mobile apps reducing round trips, when multiple clients need different data shapes. See REST.
  • gRPC: Internal microservice-to-service communication, polyglot environments, when latency and throughput are critical. See gRPC.
  • Hybrid: Use gRPC between services, GraphQL as the BFF (Backend for Frontend), and REST for third-party integrations.

Production Examples

  • GitHub — migrated REST API v3 to GraphQL (v4) for flexible data access
  • Shopify — GraphQL Storefront API for custom storefronts
  • Twitter/X — uses GraphQL internally for frontend data layer
  • Facebook — invented GraphQL in 2012 for mobile app performance

Common Interview Questions

  • "What is GraphQL and how does it differ from REST?"
    GraphQL is a query language with a single endpoint where clients specify exactly what data they need. REST uses multiple endpoints with fixed response shapes. GraphQL solves over-fetching and under-fetching but sacrifices native HTTP caching.
  • "Explain the N+1 problem in GraphQL."
    When resolving nested fields (e.g., users with posts), a naive resolver fires one DB query per parent record — 100 users = 101 queries. Solved with DataLoader, which batches all IDs into a single query within one request tick.
  • "How do you secure a GraphQL API?"
    Authentication via token in HTTP header (same as REST). Authorization in resolvers per field. Rate limiting based on query complexity/depth (not just request count). Use persisted queries to prevent arbitrary query execution.
  • "When would you NOT use GraphQL?"
    Simple CRUD APIs with few resources. When HTTP caching is critical. When your team lacks GraphQL expertise. For binary data or file uploads. When the API is consumed by simple scripts (cURL, webhooks).
  • "How does GraphQL handle real-time data?"
    Subscriptions use WebSockets under the hood. The client subscribes to events, and the server pushes updates. Implementations: Apollo, graphql-ws, or AWS AppSync.

What to Skip

  • GraphQL SDL advanced features (directives, union types, input types) — know they exist, not their syntax
  • Federation / Apollo Federation details — mention the concept of schema stitching, not implementation
  • Relay client spec — historical, rarely used in interviews
  • Specific library APIs (Apollo Client details, URQL, etc.)
