Weighted Round Robin

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • How weighted round robin extends round robin with server capacity weights
  • Weight assignment strategies and their trade-offs
  • The smooth weighted round robin algorithm (Nginx implementation)
  • When to use weighted round robin vs other algorithms

Definition

Weighted round robin is a load balancing algorithm that distributes requests proportionally to each server's assigned weight. Servers with higher weights receive more requests, allowing you to account for differences in server capacity (CPU, memory, network) or to perform gradual traffic shifting (canary deployments).

Problem with plain Round Robin: Server A: 8 CPU, 32 GB RAM (high capacity) Server B: 4 CPU, 16 GB RAM (low capacity) Plain round robin: A: ████████████ (50%) ← Can handle more, underutilized B: ████████████ (50%) ← Weak server, overloaded Weighted Round Robin (weight A=5, B=2): Total weight = 7, so A gets 5/7 = 71%, B gets 2/7 = 29% Request distribution (over 7 requests): Req 1 ──▶ A Req 2 ──▶ A Req 3 ──▶ A Req 4 ──▶ A Req 5 ──▶ A Req 6 ──▶ B Req 7 ──▶ B A: ████████████████████████ (71%) ← Utilized properly B: ████████████ (29%) ← Not overloaded

Smooth Weighted Round Robin

The naive implementation (send all requests to A 5 times, then B 2 times) creates bursts. Nginx uses smooth weighted round robin to distribute requests evenly over time while maintaining the correct weight proportions.

Naive weighted round robin (bursts): A A A A A B B | A A A A A B B | A A A A A B B └── 5 to A ──┘└ 2 to B┘ (A gets hammered for 5 requests, then idle for 2) Smooth weighted round robin (even spread): A B A A B A A | B A A B A A B | A A B A A B A (Properly interleaved, no bursts) Algorithm (Nginx): ┌──────────────────────────────────────────┐ │ servers: A(weight=5), B(weight=1) │ │ │ │ Before each selection: │ │ current_weight += effective_weight │ │ │ │ Select server with highest current_weight│ │ selected.current_weight -= total_weight│ │ │ │ Step 1: cw(A)=0+5=5, cw(B)=0+1=1 │ │ Select A. cw(A)=5-6=-1 │ │ Step 2: cw(A)=-1+5=4, cw(B)=1+1=2 │ │ Select A. cw(A)=4-6=-2 │ │ Step 3: cw(A)=-2+5=3, cw(B)=2+1=3 │ │ Tie → Select A (or B). │ │ ... │ │ Over 6 requests: A gets 5, B gets 1 │ └──────────────────────────────────────────┘

Configuration Examples

# Nginx — weighted round robin
upstream backend {
    server 10.0.1.10:8080 weight=5;  # High-capacity server
    server 10.0.1.11:8080 weight=3;  # Medium-capacity server
    server 10.0.1.12:8080 weight=2;  # Low-capacity server
}

# Nginx — canary deployment
upstream backend {
    server 10.0.1.10:8080 weight=9;  # Current version (v1)
    server 10.0.1.20:8080 weight=1;  # Canary (v2) — 10% traffic
}

# HAProxy — weighted round robin
backend api_backend
    balance roundrobin
    server api1 10.0.1.10:8080 weight 5
    server api2 10.0.1.11:8080 weight 3
    server api3 10.0.1.12:8080 weight 2

# AWS ALB — weighted target groups
# Target Group A (v1): weight=9
# Target Group B (v2): weight=1
# 10% of traffic goes to v2 for canary testing

Canary Deployments with Weighted Round Robin

One of the most practical uses of weighted round robin is canary deployments — gradually shifting traffic from an old version to a new version.

Canary deployment progression: Phase 1 (Day 1): Phase 2 (Day 2): V1: weight=95% ████████ V1: weight=80% ████████ V2: weight=5% ██ V2: weight=20% ████ Phase 3 (Day 3): Phase 4 (Day 4): V1: weight=50% ████████ V1: weight=0% (complete) V2: weight=50% ████████ V2: weight=100% ████████ At each phase, monitor error rates and latency. If V2 shows problems, shift weight back to V1 instantly.
Pitfall: Weights are static unless you use dynamic configuration. In a canary deployment, you need automation (CI/CD pipeline, feature flag system) to gradually adjust weights based on health metrics. Manual weight adjustment is error-prone and slow.

Common Interview Questions

  • "What is weighted round robin and why would you use it?"
    Like round robin, but each server has a weight proportional to its capacity. A server with weight 5 gets twice as much traffic as one with weight 2. Use it when servers have different hardware specs or during canary deployments to shift traffic gradually to a new version.
  • "How would you do a canary deployment with a load balancer?"
    Set the current version to a high weight (e.g., 95) and the new version to a low weight (e.g., 5). Monitor error rates, latency, and business metrics. Gradually shift weights toward the new version (95/5 → 80/20 → 50/50 → 100/0). If anything goes wrong, shift weight back to the old version instantly. This is much faster and safer than blue/green for gradual rollouts.
  • "What is smooth weighted round robin?"
    The naive approach sends all requests to the high-weight server in a burst (A A A A A B B). Smooth weighted round robin (used by Nginx) interleaves requests (A B A A B A A) so each server receives requests evenly over time rather than in bursts. The algorithm uses current_weight and effective_weight to select the best server while maintaining the correct proportion.
  • "How do you determine the right weights for each server?"
    Static: based on hardware specs (CPU cores, RAM, network bandwidth). Dynamic: auto-adjust based on real-time metrics like CPU usage, memory pressure, or request latency. In practice, start with static weights based on instance types and adjust based on observed performance metrics.

What to Skip

  • Smooth weighted round robin mathematical proof
  • Dynamic weight adjustment algorithms (least loaded, adaptive)
  • Multi-level weighted round robin
