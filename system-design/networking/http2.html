HTTP/2

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What HTTP/2 solves that HTTP/1.1 couldn't
  • Multiplexing — the killer feature
  • Server push, header compression, and binary framing
  • Why HTTP/2 doesn't fully solve head-of-line blocking

Definition

HTTP/2 (2015, RFC 7540) is a major revision of HTTP that uses binary framing, multiplexing, and header compression to solve HTTP/1.1's performance limitations — while keeping the same HTTP semantics (methods, headers, status codes unchanged).

Multiplexing

HTTP/1.1: 6 TCP connections, 6 parallel requests ┌─────────────────────────────────────────────┐ │ Connection 1: [Req1 → ...........Resp1........] │ │ Connection 2: [Req2 → Resp2 ✓] │ │ Connection 3: [Req3 → Resp3 ✓] │ │ Connection 4: [Req4 → Resp4 ✓] │ │ Connection 5: [Req5 → Resp5 ✓] │ │ Connection 6: [Req6 → Resp6 ✓] │ └─────────────────────────────────────────────┘ HTTP/2: 1 TCP connection, unlimited concurrent streams ┌─────────────────────────────────────────────┐ │ Single Connection: │ │ Stream 1: [Req1 → ..............Resp1......] │ │ Stream 2: [Req2 → Resp2 ✓] │ │ Stream 3: [Req3 → Resp3 ✓] │ │ Stream 4: [Req4 → Resp4 ✓] │ │ Stream 5: [Req5 → Resp5 ✓] │ │ Stream 6: [Req6 → Resp6 ✓] │ │ Stream 7: [Req7 → Resp7 ✓] │ │ Stream 8: [Req8 → Resp8 ✓] │ └─────────────────────────────────────────────┘ Stream 1 is slow, but Streams 2-8 complete independently. No head-of-line blocking at the HTTP level!

Header Compression (HPACK)

HTTP/1.1 problem: Headers repeated in full for every request.
  GET /api/users HTTP/1.1
  Host: api.example.com
  Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
  Accept: application/json
  Accept-Encoding: gzip, br

  → Sent identically for every request (~500 bytes)

HTTP/2: HPACK compresses headers using a static + dynamic table.
  First request: full headers sent and cached
  Subsequent: only delta (changed values) sent
  → ~80-90% reduction in header overhead

  Example:
  Request 1: "Authorization: Bearer abc..." (cached)
  Request 2: Same auth → just index reference (4 bytes)
  Request 3: Same + different path → path + index refs

Server Push

The server can proactively send resources the client hasn't requested but will likely need.

Client requests: GET /index.html

Server responds with index.html AND pushes:
  - styles.css (linked in HTML)
  - app.js (linked in HTML)

Client gets all 3 in a single round trip instead of:
  1. Request index.html → receive
  2. Parse HTML, find styles.css → request → receive
  3. Parse HTML, find app.js → request → receive

Note: Server Push is being deprecated in favor of:
  - <link rel="preload"> hints (client-driven)
  - <link rel="prefetch"> for future navigation
  Reason: Hard to control, can cause cache duplication.

Remaining TCP-Level HOL Blocking

Important nuance: HTTP/2 solves HTTP-level HOL blocking but NOT TCP-level HOL blocking. If a TCP packet is lost, ALL streams on that connection are blocked until retransmission. This is the main motivation for HTTP/3 (QUIC), which runs over UDP.
TCP packet loss affects ALL HTTP/2 streams: Connection: ──────[✓✓✓✗✓✓]────── ↑ Packet lost ↓ ALL streams pause until retransmitted: Stream 1: [.....█......] ← blocked Stream 2: [..█.........] ← blocked (even though it was fine) Stream 3: [.........█..] ← blocked HTTP/3 (QUIC) solves this: independent streams on UDP, so lost packet only blocks one stream.

Common Interview Questions

  • "What's the main improvement in HTTP/2?"
    Multiplexing — unlimited concurrent request/response streams on a single TCP connection. Eliminates HTTP/1.1's head-of-line blocking and reduces connection overhead from 6 connections per domain to 1.
  • "Does HTTP/2 eliminate head-of-line blocking?"
    At the HTTP level — yes. At the TCP level — no. A lost TCP packet still blocks all streams until retransmitted. This is why HTTP/3 exists.
  • "What's HPACK?"
    HTTP/2's header compression format. Uses a static dictionary (common headers) and a dynamic table (previously sent headers). Reduces header overhead by ~80-90% by sending only the delta instead of full headers every time.
