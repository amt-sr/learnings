Definition

A session is a server-side record of user state that persists across multiple HTTP requests. Since HTTP is stateless, sessions provide continuity вҖ” the server knows "this is the same user who logged in 5 minutes ago."

Analogy: Getting a wristband at a club. The wristband (session ID) proves you already paid. You show it each time you want to enter a different room (make a request). The club keeps a record of your wristband number (session store).

How Sessions Work

1. Login: POST /login {email, password} вҶ’ Server authenticates user вҶ’ Creates session: {user_id: 123, role: "admin", created: ...} вҶ’ Stores in Redis: session_id вҶ’ session_data вҶ’ Returns cookie: Set-Cookie: sid=abc123; HttpOnly; Secure 2. Subsequent requests: GET /api/profile Cookie: sid=abc123 вҶ’ Server reads sid from cookie вҶ’ Looks up Redis: GET session:abc123 вҶ’ Finds user_id=123, returns user profile 3. Logout: DELETE /logout вҶ’ Server deletes session from Redis вҶ’ Cookie expires or is cleared

Session Storage Options

StorageLatencyScalabilityUse When
Redis~1msHorizontal (cluster)Most production systems
Memcached~1msHorizontalSimple session data (no persistence)
Database~5-50msShardingWhen durability matters (sessions survive cache crash)
In-memory~0msSingle server onlySingle-server, low-traffic apps
Redis is the standard choice: Sub-millisecond lookups, built-in TTL (auto-expire sessions), supports clustering for horizontal scaling, and persists to disk (doesn't lose sessions on restart like Memcached).

Session at Scale

Challenges:
  1. Session affinity (sticky sessions): If session stored in-memory,
     user must hit same server. Avoid вҖ” use shared Redis instead.
  2. Redis failover: Session data lost if Redis crashes without persistence.
     Use Redis AOF persistence + Redis Cluster for HA.
  3. Memory: 10M active sessions Г— 1KB = 10GB. Plan accordingly.
  4. TTL: Set reasonable expiry (30 min inactive, 7 days max).

  This is why sticky sessions are an anti-pattern for stateless architectures.

Common Interview Questions

  • "Where do you store sessions in a distributed system?"
    Redis cluster. Every app server reads/writes sessions to shared Redis, so any server can handle any request. Avoid in-memory sessions (breaks stateless architecture) and database sessions (too slow).
  • "Session vs JWT?"
    Sessions for browser apps (easy invalidation, HttpOnly cookies). JWT for APIs and mobile (stateless, no server storage). For high-security: sessions. For high-scale API-first: JWT.
