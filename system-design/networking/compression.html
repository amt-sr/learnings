Compression

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • Why compression matters for bandwidth and latency
  • gzip vs brotli — the two main formats
  • When to compress and when not to
  • How compression works in the HTTP request/response cycle

Definition

HTTP compression reduces the size of HTTP responses (and sometimes requests) before transmission, reducing bandwidth usage and transfer time.

How It Works

1. Client advertises support: Accept-Encoding: gzip, deflate, br 2. Server compresses response: Content-Encoding: br (response body compressed with brotli) 3. Client decompresses: Browser automatically decompresses based on Content-Encoding The client and server negotiate compression automatically. Application code doesn't change — it's handled at the HTTP layer.

Compression Formats

FormatRatioSpeedBrowser SupportUse When
gzip60-70% reductionFast compress/decompressAll browsersUniversal fallback
brotli (br)65-80% reductionSlower compress, faster decompressAll modern browsersBest compression (preferred)
deflate60-70%FastAll browsersLegacy (avoid, use gzip)
Default choice: Brotli for modern browsers, gzip as fallback. Both are enabled by default in Nginx, Cloudflare, and most CDNs. Enable brotli (br) — it's 15-20% better than gzip at similar decompression speed.

Rule of thumb: Compress text-based content (HTML, CSS, JS, JSON, XML). Don't compress binary content that's already compressed (images, video, PDFs). Compressing already-compressed data can actually make it larger and waste CPU.

Compression at Scale

Considerations:
  1. CPU cost: Compression uses CPU. At high QPS, this matters.
     Mitigation: Compress once, cache the compressed response at CDN.
  2. Minimum size threshold: Don't compress responses < 1KB.
     The compression overhead exceeds the savings for small responses.
  3. Dynamic content: Compress on-the-fly. CDN handles this automatically.
  4. Static content: Pre-compress (brotli) and serve the .br file directly.

  CDN handles most of this automatically:
  - Origin sends uncompressed
  - CDN compresses and caches the compressed version
  - Serves compressed version to clients that support it
  - Serves uncompressed to clients that don't

Common Interview Questions

  • "How would you reduce API response size?"
    Enable brotli/gzip compression (60-80% reduction for JSON). Minimize fields in response (exclude unnecessary data). Use field selection (GraphQL or sparse fieldsets). Paginate large lists.
  • "Should you compress images?"
    No — images (JPEG, PNG, WebP) are already compressed. HTTP compression on already-compressed binary data wastes CPU and can increase size. Instead, optimize at the image level: serve WebP instead of JPEG, resize for viewport, use responsive images.
  • "What's the trade-off with compression?"
    Reduced bandwidth and transfer time vs increased CPU usage. For text responses, the trade-off overwhelmingly favors compression. For small responses (< 1KB) or already-compressed content, skip it.

What to Skip

  • Compression algorithm internals (LZ77, Huffman coding)
  • zstd and other niche formats
  • Request body compression (rare in practice)
