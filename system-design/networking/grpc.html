Definition

gRPC (Google Remote Procedure Call) is a high-performance, open-source RPC framework using Protocol Buffers for serialization and HTTP/2 for transport. It provides code generation in multiple languages, strong typing via .proto files, and built-in support for streaming.

Protocol Buffers (Protobuf)

Protobuf is gRPC's IDL (Interface Definition Language) and serialization format. You define messages and services in .proto files, and gRPC generates client and server stubs automatically.

// user.proto
syntax = "proto3";

package users;

message GetUserRequest {
  string user_id = 1;
}

message User {
  string id = 1;
  string name = 2;
  string email = 3;
  repeated string roles = 4;
}

message ListUsersResponse {
  repeated User users = 1;
}

// Service definition
service UserService {
  rpc GetUser(GetUserRequest) returns (User);           // Unary
  rpc ListUsers(Empty) returns (stream User);           // Server streaming
  rpc CreateUser(stream User) returns (User);            // Client streaming
  rpc Chat(stream ChatMessage) returns (stream ChatMsg); // Bidirectional
}

Four Communication Patterns

1. UNARY (request → response) ┌────────┐ ┌────────┐ │ Client │─── Request ──────→│ Server │ │ │←── Response ──────│ │ └────────┘ └────────┘ Use: GetUser, CreateOrder, standard RPC calls 2. SERVER STREAMING (request → stream of responses) ┌────────┐ ┌────────┐ │ Client │─── Request ──────→│ Server │ │ │←── Response 1 ────│ │ │ │←── Response 2 ────│ │ │ │←── Response 3 ────│ │ └────────┘ └────────┘ Use: Subscribe to updates, log streaming, stock prices 3. CLIENT STREAMING (stream of requests → response) ┌────────┐ ┌────────┐ │ Client │─── Request 1 ────→│ Server │ │ │─── Request 2 ────→│ │ │ │─── Request 3 ────→│ │ │ │←── Response ──────│ │ └────────┘ └────────┘ Use: File upload, bulk data ingestion, telemetry 4. BIDIRECTIONAL STREAMING (both directions) ┌────────┐ ┌────────┐ │ Client │─── Request 1 ────→│ Server │ │ │←── Response 1 ────│ │ │ │─── Request 2 ────→│ │ │ │←── Response 2 ────│ │ │ │─── Request 3 ────→│ │ └────────┘ └────────┘ Use: Chat, real-time collaboration, live navigation

gRPC vs REST Performance

REST (JSON)gRPC (Protobuf)
TransportHTTP/1.1 (or HTTP/2)HTTP/2 (mandatory)
SerializationJSON (text)Protobuf (binary)
Payload size~2-5x largerCompact binary
Latency (microservice call)~2-5ms overhead~0.2-0.5ms overhead
ThroughputGood7-10x higher
MultiplexingHTTP/1.1: no (one req per TCP conn)HTTP/2: yes (streams over single conn)
StreamingNot native (requires SSE or WS)Built-in (4 patterns)
Code generationExternal (OpenAPI)Native (protoc)
Browser supportNativeRequires gRPC-Web proxy
CachingHTTP caching (Cache-Control, ETag)No HTTP caching semantics
ObservabilityStandard HTTP toolingRequires gRPC-specific tooling
Why the performance gap? Three reasons: (1) Protobuf binary serialization is 3-10x faster and smaller than JSON. (2) HTTP/2 multiplexing eliminates head-of-line blocking. (3) Persistent connections with binary framing reduce per-request overhead. For a service making thousands of internal calls per second, this compounds significantly.

Why gRPC Dominates Internal Microservices

  • Language-agnostic — .proto generates stubs for Go, Java, Python, C++, Rust, Node.js, etc. Teams use different languages but share the same contract.
  • Strongly typed contract — .proto files are the source of truth. Breaking changes are caught at build time, not runtime.
  • Streaming built-in — server streaming, client streaming, and bidirectional streaming without additional protocols.
  • HTTP/2 multiplexing — multiple concurrent RPCs over a single TCP connection, reducing connection overhead. See HTTP/2.
  • Deadlines and cancellation — propagate timeouts across service boundaries automatically.
  • Interceptors/middleware — logging, auth, metrics, retries as pluggable middleware.
  • Load balancing — client-side load balancing with gRPC's built-in support (lookaside LB via xDS).
Production pattern: Most large systems use gRPC for internal service-to-service calls, then expose REST or GraphQL externally. An API gateway handles the translation (gRPC-Gateway, Envoy). This gives you internal performance + external compatibility.

gRPC-Web for Browser Support

Browsers cannot make raw gRPC calls (binary framing over HTTP/2). gRPC-Web bridges this gap.

Standard gRPC (service-to-service): Client ──binary protobuf──HTTP/2──→ gRPC Server gRPC-Web (browser): Browser ──gRPC-Web (text+binary)──HTTP/2──→ Envoy Proxy ──gRPC──→ gRPC Server Envoy (or grpc-web proxy) translates between gRPC-Web and native gRPC.
Native gRPCgRPC-Web
ClientAny languageBrowser (JS/TS) or mobile

PerformanceFull binarySlightly slower (proxy overhead)

Common Interview Questions

  • "Why use gRPC instead of REST for microservices?"
    Binary serialization (Protobuf) is smaller and faster than JSON. HTTP/2 multiplexing enables concurrent streams on one connection. Built-in code generation from .proto ensures type safety. Streaming is native. For high-throughput internal calls, gRPC provides significantly lower latency and higher throughput. See REST.
  • "Explain how Protocol Buffers work."
    You define message types in .proto files with field numbers. The protoc compiler generates serialization code in your language. Fields are encoded by number (not name), making wire format compact. Unknown fields are preserved for forward/backward compatibility. Proto3 is the current version.
  • "What are the 4 gRPC communication patterns?"
    Unary (one request, one response), server streaming (one request, stream of responses), client streaming (stream of requests, one response), bidirectional streaming (both sides stream independently).
  • "How does gRPC handle failure across services?"
    gRPC status codes map to common error conditions (UNAVAILABLE, DEADLINE_EXCEEDED, INVALID_ARGUMENT). Deadlines propagate across service boundaries — if Service A has a 5s deadline and calls Service B with 3s remaining, Service B sees a 3s deadline. Combined with retries and circuit breakers, this provides resilient cross-service error handling.
  • "How do you version a gRPC API?"
    Add new fields to messages (field numbers are permanent — never reuse). Create new RPC methods for breaking changes. Use package naming (v1, v2) in .proto files. Backward compatibility is designed into Protobuf — old clients ignore new fields, new servers use defaults for missing fields.

What to Skip

  • Protobuf encoding internals (varint, zigzag, wire types) — know it's binary and compact, not the bit-level details
  • gRPC reflection API — internal debugging tool
  • xDS protocol details — know Envoy uses it for service discovery, not the protocol spec
  • Channelz and gRPC-specific observability internals
