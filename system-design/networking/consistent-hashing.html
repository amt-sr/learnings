Consistent Hashing

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What consistent hashing is and how it solves the redistribution problem
  • The hash ring data structure and how lookups work
  • Why consistent hashing minimizes key remapping when servers change
  • Virtual nodes and how they improve distribution uniformity
  • Applications in load balancing, caching, and distributed databases

The Problem with Modular Hashing

With naive modular hashing, adding or removing a server causes massive redistribution of keys:

Modular hashing: server = hash(key) % N 3 servers (N=3): key "alice" → hash=7 % 3 = 1 → Server B key "bob" → hash=14 % 3 = 2 → Server C key "carol" → hash=21 % 3 = 0 → Server A key "dave" → hash=35 % 3 = 2 → Server C key "eve" → hash=42 % 3 = 0 → Server A Now add Server D (N=4) — EVERY key is rehashed: key "alice" → hash=7 % 4 = 3 → Server D ← MOVED! key "bob" → hash=14 % 4 = 2 → Server C ← stays key "carol" → hash=21 % 4 = 1 → Server B ← MOVED! key "dave" → hash=35 % 4 = 3 → Server D ← MOVED! key "eve" → hash=42 % 4 = 2 → Server C ← MOVED! 4 out of 5 keys moved — 80% redistribution! For a million keys, this means 800K cache misses and 800K database queries at once → thundering herd.

Consistent Hashing Solution

Consistent hashing maps both keys and servers onto the same hash ring (a circular hash space, typically 0 to 2^32-1). A key is assigned to the server that is the first server clockwise from the key's hash position on the ring.

Consistent Hashing — The Hash Ring: 0 ╭───────╮ C / \ key=bob │ │ D │ │ B │ │ A \ / key=alice ╰───────╯ 2^32 Servers on the ring (hashed by server name/IP): Server A → position 1200 Server B → position 3600 Server C → position 6800 Server D → position 9200 Key lookup (key → hash → walk clockwise to first server): key "alice" → hash=1500 → walk CW → Server B (3600) key "bob" → hash=2500 → walk CW → Server B (3600) key "carol" → hash=7200 → walk CW → Server D (9200) key "dave" → hash=300 → walk CW → Server A (1200)

Minimal Redistribution

The key property: when you add or remove a server, only keys in the affected range are remapped. All other keys stay on their current server.

Adding Server D to existing ring (A, B, C): Before (3 servers): 0 ──▶ A ──▶ B ──▶ C ──▶ (wrap to A) Keys: alice→B, bob→B, carol→C, dave→A After adding D between B and C: 0 ──▶ A ──▶ B ──▶ D ──▶ C ──▶ (wrap to A) Keys: alice→B (unchanged), bob→B (unchanged), carol→D (MOVED from C), dave→A (unchanged) Only 1 out of 4 keys moved — 25% redistribution (expected: 1/N = 1/4 = 25%) Removing Server D: carol moves to C (next clockwise) All other keys unchanged. Only 1 key affected again.
Key insight: When a server is added or removed, consistent hashing only redistributes keys that were assigned to (or would be assigned to) that server. The expected fraction of keys that move is 1/N where N is the number of servers. With modular hashing, nearly all keys move.

Virtual Nodes (VNodes)

With few servers, consistent hashing can produce uneven distribution — one server might "own" a large portion of the ring by chance. Virtual nodes solve this by placing each physical server at multiple positions on the ring.

Without virtual nodes (uneven): Ring: A──────B──────────────────C────── A owns 20%, B owns 55%, C owns 25% ← B overloaded! With virtual nodes (100 per server): Ring: A a A a A b B b B b b B c C c C c A a A b B c C ... A owns ~33%, B owns ~33%, C owns ~33% ← Even! Each physical server maps to 100-200 virtual positions. Key "alice" → hash → finds VNode "A-47" → physical Server A More virtual nodes = more uniform distribution Typical: 100-200 virtual nodes per physical server Trade-off: more vnodes = more memory for the ring

Use Cases

Load Balancing

Consistent hashing for load balancing routes a specific key (user ID, session ID) to the same server every time. This provides a form of session affinity without cookies, and is useful when the server caches user-specific data.

# Example: Route user to same server based on user ID
# hash(user_id) → position on ring → nearest server

user_id=12345 → hash → Server A (this user always goes to A)
user_id=67890 → hash → Server B (this user always goes to B)

# Benefits:
# - Server A can cache user 12345's data locally
# - No need for shared session store
# - Adding/removing servers only affects 1/N of users

Distributed Caching

Memcached, DynamoDB, Cassandra all use consistent hashing to determine which node stores a particular key. This minimizes data movement when nodes are added or removed.

Distributed Databases

Cassandra uses consistent hashing for data partitioning. Each node owns a range of tokens on the ring. Data is partitioned by partition key hash.

Implementation

# Python — consistent hashing with virtual nodes
import hashlib
import bisect

class ConsistentHash:
    def __init__(self, servers, vnodes=100):
        self.ring = {}           # position → server
        self.sorted_keys = []    # sorted positions
        for server in servers:
            for i in range(vnodes):
                key = f"{server}:{i}"
                pos = self._hash(key)
                self.ring[pos] = server
                bisect.insort(self.sorted_keys, pos)

    def _hash(self, key):
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def get_server(self, key):
        pos = self._hash(key)
        # Binary search for first position >= pos
        idx = bisect.bisect_left(self.sorted_keys, pos)
        if idx == len(self.sorted_keys):
            idx = 0  # wrap around
        return self.ring[self.sorted_keys[idx]]

# Usage
ch = ConsistentHash(["A", "B", "C"], vnodes=100)
print(ch.get_server("user:123"))  # Always returns same server
print(ch.get_server("user:123"))  # Same server, regardless of pool changes

Drawbacks

  • Uneven distribution without virtual nodes — A server might own a disproportionately large or small range of the ring
  • Slower lookups than round robin — O(log n) binary search vs O(1) counter increment (still very fast in practice)
  • Less dynamic — Doesn't consider server load; a hot key can overload one server even if others are idle
  • Server failures — When a server fails, all its keys are redistributed to the next clockwise server, which can temporarily double that server's load
Load balancing vs data partitioning: In interviews, consistent hashing comes up in both contexts. For load balancing, it's used to route specific clients to specific servers (e.g., user-based affinity). For data partitioning (caching, databases), it's used to determine where to store each key. Understand both use cases — they are the same underlying mechanism applied to different problems.

Common Interview Questions

  • "What is consistent hashing?"
    A hashing scheme where both keys and servers are mapped onto a circular hash ring. A key is assigned to the first server encountered walking clockwise from the key's position. The key benefit: adding or removing a server only redistributes 1/N of keys (where N = total servers), compared to nearly all keys with modular hashing.
  • "Why is consistent hashing better than modular hashing?"
    Modular hashing (hash(key) % N) redistributes nearly all keys when N changes. This causes cache stampedes and thundering herd problems. Consistent hashing only moves keys in the affected range — roughly 1/N keys per server change. This makes it practical for dynamic clusters where servers join and leave.
  • "What are virtual nodes and why do you need them?"
    Virtual nodes (vnodes) place each physical server at multiple positions on the hash ring. Without vnodes, a server might by chance own a very large or very small portion of the ring, causing uneven load. With 100-200 vnodes per server, distribution becomes nearly uniform. Each vnode is a copy of the same physical server at a different ring position.
  • "What happens when a server fails in consistent hashing?"
    All keys that were assigned to the failed server are remapped to the next server clockwise on the ring. That next server temporarily receives double traffic. This can be problematic if the ring has few servers. Solutions: use more virtual nodes, use replicas (store each key on multiple servers), or have the LB temporarily spread the failed server's load across multiple healthy servers.
  • "Where is consistent hashing used in real systems?"
    DynamoDB and Cassandra use it for data partitioning. Memcached clients use it for cache key distribution. Nginx and Envoy use it for upstream load balancing with session affinity. CDN edge nodes use it for content caching. Discord, Telegram, and Slack use it for mapping users to connection servers.
  • "How would you implement consistent hashing?"
    Hash each server name (with vnode suffix) using MD5 or SHA-256, map to a 32-bit or 64-bit space. Store positions in a sorted array. For lookup, hash the key and binary search for the nearest server position clockwise. O(log n) lookup, O(n) ring construction. Use 100-200 vnodes per physical server for uniform distribution.

What to Skip

  • Specific hash function internals (MD5 vs SHA-256 for ring positions)
  • Bounded-load consistent hashing (academic variant)
  • Rendezvous hashing (HRW — different algorithm, sometimes confused with consistent hashing)
