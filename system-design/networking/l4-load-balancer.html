L4 Load Balancer

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What Layer 4 load balancing means in the OSI model
  • How it operates on TCP/UDP connections without inspecting HTTP
  • Performance characteristics and trade-offs vs L7
  • Real-world use cases and examples: AWS NLB, HAProxy, IPVS

Definition

A Layer 4 load balancer operates at the transport layer of the OSI model, distributing connections based on IP address, port, and protocol (TCP/UDP). It makes routing decisions using packet headers without inspecting application-level content. This makes it extremely fast but unaware of HTTP paths, headers, or cookies.

OSI Model — Where L4 Load Balancing Happens: ┌───────────────────────────────────────────┐ │ Layer 7: Application (HTTP, gRPC) │ ← L7 LB operates here ├───────────────────────────────────────────┤ │ Layer 6: Presentation (TLS, SSL) │ ├───────────────────────────────────────────┤ │ Layer 5: Session │ ├───────────────────────────────────────────┤ │ Layer 4: Transport (TCP, UDP) │ ← L4 LB operates here ├───────────────────────────────────────────┤ │ Layer 3: Network (IP) │ ├───────────────────────────────────────────┤ │ Layer 2: Data Link (Ethernet) │ ├───────────────────────────────────────────┤ │ Layer 1: Physical (Cables, WiFi) │ └───────────────────────────────────────────┘

How It Works

  1. Client establishes a TCP connection to the load balancer's IP and port
  2. L4 LB selects a backend server based on the destination IP:port
  3. L4 LB opens a new TCP connection (or tunnels) to the selected backend
  4. Data is forwarded between client and backend with minimal packet modification
L4 Load Balancing Flow: Client L4 Load Balancer Backends │ │ │ │── SYN ────────────────▶│ │ │ │── SYN ──────────────▶│ Server A │ │ │ │◀── SYN-ACK ───────────│◀── SYN-ACK ─────────│ │ │ │ │── ACK ────────────────▶│── ACK ──────────────▶│ │ │ │ │══ DATA ═════════════▶│══ DATA ════════════▶│ │ │ │ The LB decides based on: - Source IP + Source Port (4-tuple hash) - Destination port (e.g., 443 → all backends on 8443) - Protocol (TCP or UDP)

Interview tip: DSR (Direct Server Return) is a high-performance technique where the load balancer only handles incoming traffic. The backend server responds directly to the client, bypassing the LB on the return path. This doubles throughput capacity. AWS Network Load Balancer supports this pattern.

Real-World Examples

AWS Network Load Balancer (NLB)

  • Operates at Layer 4, handles TCP, UDP, and TLS pass-through
  • Can handle millions of connections per second
  • Ultra-low latency — adds less than 1 ms
  • Static IP per AZ, preserves source IP
  • Supports DSR via cross-zone load balancing

HAProxy (TCP Mode)

# HAProxy L4 configuration
frontend tcp_front
    bind *:3306          # MySQL
    bind *:6379          # Redis
    default_backend db_back

backend db_back
    balance roundrobin
    server db1 10.0.1.10:3306 check
    server db2 10.0.1.11:3306 check
    server db3 10.0.1.12:3306 check

IPVS (Linux Kernel)

IPVS is built into the Linux kernel and is the technology behind kube-proxy in IPVS mode for Kubernetes. It uses a hash table for O(1) lookup and supports multiple scheduling algorithms including round-robin, least connections, and source hashing.

When to Use L4 vs L7

Use L4 when:
✓ Traffic is non-HTTP (TCP, UDP, gRPC, WebSocket raw frames)
✓ Maximum throughput and minimum latency are required
✓ You don't need URL-based routing
✓ Backend services are internal microservices
✓ You're load balancing databases, caches, or message brokers

Use L7 when:
✓ You need URL path routing (/api/users → user-service)
✓ You need header-based routing (Host, Authorization, API version)
✓ TLS termination is required at the LB
✓ Rate limiting, WAF, or request transformation is needed
✓ Traffic is public-facing HTTP/HTTPS
Common architecture pattern: Use an L7 load balancer (ALB) at the edge for public traffic with TLS termination, URL routing, and rate limiting. Use L4 load balancers (NLB) internally for microservice-to-microservice communication where raw throughput matters more than HTTP-aware routing.

Common Interview Questions

  • "What is the difference between L4 and L7 load balancing?"
    L4 operates at the transport layer — it routes based on IP, port, and protocol. L7 operates at the application layer — it routes based on HTTP path, headers, cookies. L4 is faster but less flexible. L7 is smarter but adds more latency. See: L7 Load Balancer
  • "When would you use a Layer 4 load balancer?"
    For non-HTTP protocols (database connections, game servers, IoT), internal microservice traffic where latency matters, or when you need to pass TLS through without termination. AWS NLB is the go-to example.
  • "How does a Layer 4 load balancer handle TLS?"
    It passes TLS through — it does not terminate the connection. The client establishes a TLS session directly with the backend server. This is called "TLS pass-through" and is useful when you want end-to-end encryption, but it means the LB cannot inspect or modify the traffic.
  • "What is DSR and why does it matter?"
    Direct Server Return lets the backend respond directly to the client, bypassing the LB on the return path. This reduces LB load by half and significantly improves throughput. It's commonly used for high-traffic UDP workloads (gaming, streaming) and CDNs.
