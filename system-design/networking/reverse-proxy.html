Reverse Proxy

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What a reverse proxy is and how it sits between clients and servers
  • How it differs from a forward proxy
  • TLS termination, rate limiting, and security at the reverse proxy layer
  • Real-world examples: Nginx, HAProxy, AWS ALB

Definition

A reverse proxy is a server that sits in front of backend servers and forwards client requests to them. The client never talks directly to the backend — it only knows about the reverse proxy. The reverse proxy decides which backend server handles each request, providing a single entry point for all traffic.

Client Request Flow: ┌────────┐ ┌──────────────┐ ┌─────────────┐ │ Client │──────▶│ Reverse Proxy│──────▶│ Server A │ └────────┘ │ (Nginx/ALB) │──────▶│ Server B │ └──────────────┘──────▶│ Server C │ │ └─────────────┘ │ Client never sees Server A, B, or C directly

Key Responsibilities

FunctionDescription
Load DistributionDistributes requests across multiple backend servers
TLS TerminationHandles SSL/TLS decryption so backends serve plain HTTP
Rate LimitingThrottles requests per client IP or API key
SSL OffloadingFrees backend servers from expensive encryption work
CachingServes cached responses for frequently requested resources
CompressionCompresses responses (gzip/brotli) before sending to clients
SecurityHides backend server IPs, blocks malicious traffic, WAF integration

TLS Termination

TLS termination means the reverse proxy decrypts incoming HTTPS traffic and forwards plain HTTP to backend servers. This is critical because TLS handshakes are CPU-intensive — offloading this to the proxy frees backend servers to focus on application logic.

Without TLS Termination: Client ──[TLS]──▶ LB ──[TLS]──▶ Server A Client ──[TLS]──▶ LB ──[TLS]──▶ Server B (Every server handles TLS decryption) With TLS Termination: Client ──[TLS]──▶ Reverse Proxy ──[HTTP]──▶ Server A Client ──[TLS]──▶ Reverse Proxy ──[HTTP]──▶ Server B (Only the proxy handles TLS decryption)
Security consideration: When using TLS termination, traffic between the reverse proxy and backends is unencrypted. In production, use TLS between the proxy and backends as well (end-to-end TLS), or place everything within a trusted VPC where network-level encryption is unnecessary.

Rate Limiting at the Reverse Proxy

A reverse proxy can enforce rate limits before requests reach backend servers, protecting them from abuse and traffic spikes.

# Nginx rate limiting example
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

server {
    location /api/ {
        limit_req zone=api burst=20 nodelay;
        proxy_pass http://backend;
    }
}

# Common rate limiting strategies:
# - Per IP address:        rate=10r/s
# - Per API key:           rate=100r/s  (map $arg_api_key)
# - Per user (from JWT):   rate=50r/s   (map $jwt_claim_sub)

Reverse Proxy vs Forward Proxy

Reverse ProxyForward Proxy
PositionIn front of serversIn front of clients
Who it servesServers (protects backends)Clients (protects users)
VisibilityClient knows proxy's address onlyServer doesn't know real client
Use caseLoad balancing, TLS terminationPrivacy, content filtering, caching
ControlServer admin controls the proxyClient admin controls the proxy
ExamplesNginx, AWS ALB, HAProxySquid, corporate proxy, VPN
Memory trick: A reverse proxy is the reverse of what you'd expect — it sits on the server side, not the client side. A forward proxy is the "normal" direction — it's on the client's behalf.

Real-World Examples

Nginx

# Nginx as reverse proxy with load balancing
upstream backend {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}

server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate /etc/ssl/certs/api.pem;
    ssl_certificate_key /etc/ssl/private/api.key;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

HAProxy

# HAProxy configuration
frontend http_front
    bind *:443 ssl crt /etc/ssl/api.pem
    default_backend http_back

backend http_back
    balance roundrobin
    server web1 10.0.1.10:8080 check
    server web2 10.0.1.11:8080 check
    server web3 10.0.1.12:8080 check

AWS Application Load Balancer (ALB)

AWS ALB is a Layer 7 reverse proxy managed by AWS. It supports path-based routing, TLS termination, and integrates with WAF. You pay per hour + per LCU (Load Balancer Capacity Unit).

# ALB path-based routing rules:
# /api/users/*     → Target Group: user-service
# /api/orders/*    → Target Group: order-service
# /api/payments/*  → Target Group: payment-service
# /*               → Target Group: web-frontend

Common Interview Questions

  • "What is a reverse proxy and why do you need one?"
    Sits in front of backend servers, forwarding client requests to them. Needed for load distribution, TLS termination, security (hides backend IPs), rate limiting, and caching. It provides a single entry point for all traffic.
  • "What is TLS termination and why is it important?"
    The reverse proxy decrypts HTTPS traffic so backend servers receive plain HTTP. This centralizes certificate management and offloads CPU-intensive crypto work from app servers. Trade-off: traffic between proxy and backend must be secured separately (VPC or mutual TLS).
  • "Reverse proxy vs load balancer — are they the same?"
    Every reverse proxy can do basic load balancing, but a dedicated load balancer offers advanced algorithms (least connections, consistent hashing), health checks, and often operates at both L4 and L7. A reverse proxy is typically more focused on request transformation and routing.
  • "Where would you put rate limiting — at the reverse proxy or the application?"
    At the reverse proxy for defense-in-depth. Rate limiting at the proxy prevents malicious traffic from reaching the app at all. But the app should also enforce limits for cases where traffic bypasses the proxy (internal services, direct access).
