Definition

TLS (Transport Layer Security) is a cryptographic protocol that provides encryption, authentication, and integrity for data in transit. It sits between the transport layer (TCP) and the application layer (HTTP).

Analogy: TLS is like sending a sealed, tamper-evident letter via registered mail. The recipient can verify the sender's identity (authentication), the contents can't be read in transit (encryption), and any tampering is detected (integrity).

What TLS Provides

PropertyMechanismWhy It Matters
EncryptionSymmetric cipher (AES-256-GCM)Prevents eavesdropping (man-in-the-middle)
AuthenticationDigital certificates (X.509)Verifies server identity (you're talking to google.com, not an imposter)
IntegrityMAC (Message Authentication Code)Detects any data tampering in transit

How TLS Works

Client Server ────── ────── 1. ClientHello ─────────────→ Supported cipher suites, TLS version, random 2. ←───────────── ServerHello Chosen cipher, random Certificate (server's public key) 3. Client verifies certificate (signed by trusted Certificate Authority) 4. Key Exchange ────────────→ Client generates pre-master secret Encrypts with server's public key Both derive session key (symmetric) 5. Finished ←──────────────→ Both confirm handshake integrity 6. Encrypted data transfer begins Uses symmetric encryption (fast!) Session key = shared secret
Key insight — Asymmetric then symmetric: TLS uses slow asymmetric encryption (RSA/ECDHE) only during handshake to exchange a shared secret. All subsequent data uses fast symmetric encryption (AES). This gives you the security of public-key crypto with the speed of symmetric crypto.

TLS 1.2 vs TLS 1.3

TLS 1.2TLS 1.3
Handshake RTTs2 RTTs1 RTT (0-RTT with resumption)
Supported ciphersMany (some weak)Only AEAD ciphers (AES-GCM, ChaCha20)
RSA key exchangeYesRemoved (only ECDHE)
Forward secrecyOptionalMandatory
Legacy algorithmsSHA-1, 3DES, RC4All removed
PerformanceGoodFaster (fewer round trips)
Forward secrecy: Even if the server's private key is compromised later, past sessions can't be decrypted. ECDHE generates a new ephemeral key pair for each session. TLS 1.3 makes this mandatory.

Certificate Chain

Root CA (self-signed, pre-installed in browser/OS) └── Intermediate CA (signed by Root CA) └── Server Certificate (signed by Intermediate CA) Verification chain: Server cert → signed by Intermediate? → signed by Root? Root in trust store? → TRUSTED Examples of Root CAs: - Let's Encrypt (free, automated) - DigiCert, Comodo, GlobalSign (commercial)

TLS Termination

In large systems, TLS is terminated at the load balancer or CDN edge, not at each backend server.

Client ──TLS──→ [CDN/Load Balancer] ──HTTP──→ [App Servers] │ TLS terminates here Decrypts, verifies certificate Forwards plain HTTP to backends Benefits: ✓ Certificates managed in one place ✓ Backend servers don't need certs ✓ CDN can cache encrypted content Internal traffic: - Public cloud: TLS between all internal services (mTLS) - Some teams use plain HTTP internally (simpler, faster) - Trade-off: security vs operational overhead

Common Interview Questions

  • "What does TLS do?"
    Encryption (confidentiality), authentication (server identity), integrity (tamper detection). It sits between TCP and the application layer.
  • "How does the TLS handshake work?"
    Client sends supported ciphers → Server picks cipher, sends certificate → Client verifies cert → Both do key exchange → Derive shared session key → Switch to symmetric encryption. TLS 1.3 does this in 1 RTT.
  • "Symmetric vs asymmetric encryption in TLS?"
    Asymmetric (RSA/ECDHE) is used only during handshake to exchange a secret. Symmetric (AES-GCM) is used for all data transfer because it's 100-1000x faster.
  • "What's forward secrecy?"
    Each session uses a unique ephemeral key. Compromising the server's long-term private key doesn't allow decryption of past sessions. TLS 1.3 mandates forward secrecy.
  • "Where do you terminate TLS in a large system?"
    At the load balancer or CDN edge. Backends communicate over internal HTTP. For zero-trust, use mTLS everywhere (service mesh).

What to Skip

  • Cipher suite internals (CBC, GCM mode details)
  • Certificate Signing Request (CSR) generation process
  • OCSP and CRL revocation mechanisms
  • DTLS (Datagram TLS) internals
