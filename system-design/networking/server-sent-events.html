Server-Sent Events (SSE)

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What SSE is and how one-way server push works
  • The text/event-stream protocol format
  • SSE vs WebSocket — when to use each
  • Built-in auto-reconnection and event IDs
  • Practical use cases at scale

Definition

Server-Sent Events (SSE) is a standard allowing servers to push real-time updates to clients over a persistent HTTP connection. The client opens a standard HTTP connection, and the server sends a stream of events as text. The connection stays open indefinitely, with the server pushing new events as they occur.

The Protocol

SSE uses the text/event-stream MIME type over a standard HTTP response. Events are sent as simple text with a structured format.

Server response (Content-Type: text/event-stream):

data: {"user": "alice", "action": "joined"}

data: {"user": "bob", "action": "liked", "post": 456}

event: notification
data: {"message": "You have 3 new followers"}

event: price-update
id: price-12345
retry: 3000
data: {"symbol": "AAPL", "price": 185.42}

event: price-update
id: price-12346
data: {"symbol": "AAPL", "price": 185.47}

Event Fields

FieldRequiredDescription
data:YesThe message payload (can span multiple lines)
event:NoEvent type — client listens with addEventListener
id:NoEvent ID — used for reconnection (Last-Event-ID header)
retry:NoReconnection interval in milliseconds
(empty line)YesBlank line separates events

Client API

// Open connection
const source = new EventSource('/api/notifications');

// Listen for all events
source.onmessage = (event) => {
  console.log(event.data);
};

// Listen for named event types
source.addEventListener('price-update', (event) => {
  const price = JSON.parse(event.data);
  renderPrice(price);
});

// Listen for custom events
source.addEventListener('notification', (event) => {
  showNotification(event.data);
});

// Connection state
source.readyState  // 0=CONNECTING, 1=OPEN, 2=CLOSED

// Close connection
source.close();

SSE vs WebSocket

SSEWebSocket
DirectionServer → Client onlyBoth directions
ProtocolStandard HTTPWebSocket (upgraded from HTTP)
Data formatText only (UTF-8)Text + binary
Auto-reconnectBuilt-in (browser handles it)Manual implementation
Last event replayBuilt-in (Last-Event-ID header)Manual
Event typesNamed events (event: field)Application-level message routing
Proxy/firewallWorks everywhere (standard HTTP)May be blocked
InfrastructureStandard HTTP servers, CDNsRequires WebSocket support
OverheadMinimal (text stream over HTTP)Minimal (binary frames)
Max connections6 per domain (HTTP/1.1 limit)No hard limit
ComplexityVery simple (standard HTTP)More complex (custom protocol)
Best forNotifications, feeds, live scoresChat, gaming, collaborative editing
The 6-connection limit: HTTP/1.1 browsers limit 6 concurrent connections per domain. Since SSE uses standard HTTP, each SSE connection counts. For multiple event streams, use domain sharding (sub1.api.com, sub2.api.com) or upgrade to HTTP/2 where the limit is much higher. Alternatively, multiplex multiple event types on a single SSE connection using the event: field.

Use Cases

Push Notifications

┌────────┐ EventSource('/notifications') ┌──────────┐ │Browser │◄══════ text/event-stream ════════════►│ API Srv │ └────────┘ └────┬─────┘ │ ┌─────▼─────┐ │ Redis │ │ Pub/Sub │ └───────────┘ Events: new-message, new-follower, like, mention

Live Feeds and Activity Streams

Twitter/X home feed, Facebook notifications, GitHub activity. Server pushes new items as they appear. Client renders incrementally without polling.

Stock Prices and Financial Data

Real-time price updates pushed every few seconds. SSE's id: field lets clients reconnect and resume from the last event they received — no data gap after a brief network hiccup.

AI / LLM Streaming Responses

ChatGPT, Claude, and similar LLM frontends use SSE to stream token-by-token responses. The server sends each generated token as a separate SSE event, allowing the UI to render text incrementally.

Auto-Reconnection

SSE's biggest advantage over raw WebSockets is built-in reconnection.

Normal flow: Client ──GET /stream──► Server Client ◄──data: event1── Server Client ◄──data: event2── Server Client ◄──data: event3── Server Connection drops (network glitch): Client ◄──data: event3── Server (last event id: "evt-3") ✕ connection lost ... browser auto-reconnects ... Client ──GET /stream──► Server (Last-Event-ID: evt-3) ← browser sends this automatically Client ◄──data: event4── Server ← server resumes from evt-3 Server implementation: const lastEventId = req.headers['last-event-id']; if (lastEventId) { // Replay missed events from event store const missed = await store.getEventsSince(lastEventId); missed.forEach(event => res.write(`id: ${event.id}\ndata: ${event.data}\n\n`)); }

Scaling SSE

SSE connections are long-lived HTTP responses. Each connection ties up a server thread (or async slot).

ChallengeSolution
Connection countUse async/non-blocking servers (Node.js, Go, Java WebFlux). Thread-per-connection models (Django, Rails) do NOT scale for SSE.
Load balancer timeoutsSet long idle timeouts (5+ minutes). Configure health checks on a separate endpoint, not the SSE stream.
Proxy timeoutsNginx: proxy_read_timeout, proxy_buffering off. CloudFlare: Enterprise plan for >100s timeouts.
6-connection limitMultiplex event types on one connection. Use HTTP/2. Use domain sharding.
Cross-server eventsRedis Pub/Sub to broadcast events to all SSE server instances.
Production consideration: SSE works through most CDNs and proxies because it's standard HTTP. However, many proxies buffer responses. Set Cache-Control: no-cache and X-Accel-Buffering: no (Nginx) to disable buffering. Without this, events are delayed until the buffer fills.

Common Interview Questions

  • "What are Server-Sent Events and when would you use them?"
    SSE is a one-way server-to-client push protocol over standard HTTP. Use it when the server needs to push updates to the client but the client only sends standard HTTP requests (not real-time bidirectional). Examples: notifications, live scores, AI streaming responses, stock prices. See WebSockets for the bidirectional alternative.
  • "SSE vs WebSocket — how do you decide?"
    SSE when: (1) only server-to-client push is needed, (2) you want simpler implementation, (3) auto-reconnect matters, (4) proxy compatibility matters, (5) text data is sufficient. WebSocket when: (1) bidirectional communication needed (chat, gaming), (2) binary data needed, (3) high-frequency message exchange, (4) more than 6 concurrent streams per domain.
  • "How does SSE auto-reconnect work?"
    When the connection drops, the browser automatically reconnects. If the server sent event IDs (via the id: field), the browser sends a Last-Event-ID header on reconnect. The server uses this to replay missed events. The server can set retry: to control reconnection interval.
  • "How do you scale SSE to thousands of connections?"
    Use non-blocking I/O (Node.js, Go). Set X-Accel-Buffering: no for Nginx. Use Redis Pub/Sub to broadcast events across server instances. Keep SSE connections stateless — each server can serve any client. Use caching headers to prevent buffering.
  • "What's the HTTP/1.1 connection limit problem?"
    Browsers limit 6 concurrent HTTP connections per domain. Each SSE stream uses one. For multiple event types, multiplex on one connection using the event: field. Otherwise, use domain sharding or HTTP/2 (which supports many concurrent streams over one TCP connection).
