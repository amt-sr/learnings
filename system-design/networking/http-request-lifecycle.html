HTTP Request Lifecycle

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • Every step from typing a URL to receiving the response
  • Where time is spent in each step
  • How to optimize each step in a system design
  • What happens at each layer of the network stack

Definition

The HTTP request lifecycle is the complete journey of a request from the browser to the server and back, encompassing DNS resolution, TCP connection, TLS handshake, request sending, server processing, and response delivery.

Complete Lifecycle

Step 1: URL Parsing Browser parses: https://api.example.com/users/123 → scheme: https, host: api.example.com, path: /users/123 Step 2: DNS Resolution api.example.com → DNS lookup → 142.250.80.46 Cached: ~0ms. Uncached: 1-100ms. See: DNS Caching Step 3: TCP Handshake Client → SYN → Server Client ← SYN-ACK ← Server Client → ACK → Server Cost: 1 RTT (~1ms same DC, ~50ms cross-region) See: TCP Step 4: TLS Handshake ClientHello → ServerHello + Certificate → Key Exchange Cost: 1 RTT (TLS 1.3) or 2 RTTs (TLS 1.2) See: TLS Step 5: Send HTTP Request GET /users/123 HTTP/1.1 Host: api.example.com Authorization: Bearer token... Headers + Body sent Step 6: Server Processing API Gateway → Route → Auth → Rate Limit → App Server → Business Logic → Database Query → Serialize response Cost: 5-500ms (varies wildly) Step 7: Receive HTTP Response HTTP/1.1 200 OK Headers + Body (JSON, HTML, etc.) See: Status Codes Step 8: Connection Handling Close (HTTP/1.0) or Keep-Alive (HTTP/1.1+) See: Keep Alive Total: ~10ms (cached, same DC) to ~500ms+ (uncached, cross-region)

Optimization Points

1. DNS: Use DNS prefetch <link rel="dns-prefetch" href="//api.example.com">
   Reduces DNS time from ~50ms to ~0ms

2. Connection: Use HTTP keep-alive (default in HTTP/1.1)
   Reuse TCP connections instead of handshake per request

3. TLS: Enable TLS 1.3 with session resumption
   0-RTT for repeat connections

4. Request: Minimize headers, use compression
   Smaller requests = faster transfer

5. Server: Add Redis cache, read replicas
   Reduce server processing time from 100ms to 1ms

6. Response: Use gzip/brotli compression
   Typical 60-80% reduction in response size
Interview tip: When asked "how would you speed up this API?", walk through the lifecycle. "First, check DNS caching. Then, are we reusing TCP connections? Is TLS 1.3 enabled? Is the response cached? Are we compressing?" This structured approach shows methodical thinking.

Microservice Lifecycle

Client → [CDN] → [LB] → [Service A] → [Service B] → [DB] Each arrow adds: - Network latency (~0.5-1ms same DC per hop) - Processing time (varies) - Potential serialization/deserialization 5 internal hops × 1ms = 5ms network overhead alone Plus TCP connection if not pooled, plus processing. This is why microservice latency adds up and why connection pooling, keep-alive, and HTTP/2 matter.

Common Interview Questions

  • "What happens when you type a URL in the browser?"
    Parse URL → DNS lookup → TCP handshake → TLS handshake → Send HTTP request → Server processes → Response → Browser renders. Walk through each step and its latency cost.
  • "Where is most time spent?"
    Server processing (typically 5-500ms) dominates. Network is 1-200ms depending on location. DNS is 0-100ms. Optimize server first (caching, indexes), then network (CDN, keep-alive), then DNS (caching).
  • "How would you reduce API latency?"
    Cache at CDN → Cache at application layer (Redis) → Use read replicas → Add indexes → Enable compression → Reuse connections → Move closer to user (CDN/edge).

What to Skip

  • Browser rendering pipeline (DOM, CSSOM, layout, paint)
  • Service Worker processing
  • HTTP cache validation internals (If-None-Match, If-Modified-Since — see Caching Headers)
