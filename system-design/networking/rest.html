REST

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What REST is and its core constraints
  • How to design a proper REST API
  • REST vs GraphQL vs gRPC
  • Common REST API design mistakes

Definition

REST (Representational State Transfer) is an architectural style for APIs that uses standard HTTP methods, URIs, and status codes. Resources are identified by URLs, and operations are performed using standard HTTP verbs.

Core Constraints

ConstraintMeaning
Client-ServerSeparation of concerns — UI and data storage are independent
StatelessEach request contains all info needed — no server-side session context
CacheableResponses must define themselves as cacheable or non-cacheable
Uniform InterfaceStandard HTTP methods, URIs, and status codes
Layered SystemClient can't tell if it's connected directly to the server or to an intermediary
Code on DemandOptional — server can extend client functionality (JavaScript)

REST API Design

Resources are nouns. Operations are HTTP verbs. GET /users → List users (200) GET /users/123 → Get user 123 (200 or 404) POST /users → Create user (201, Location: /users/456) PUT /users/123 → Replace user 123 (200) PATCH /users/123 → Update user 123 partially (200) DELETE /users/123 → Delete user 123 (204) Nested resources: GET /users/123/posts → Posts by user 123 POST /users/123/posts → Create post for user 123 GET /posts/456/comments → Comments on post 456 Pagination: GET /users?page=2&limit=20&sort=created_at:desc Filtering: GET /users?role=admin&status=active Versioning: GET /v2/users (preferred) or Accept: application/vnd.api+json;version=2

When to use each: REST for most public-facing APIs and CRUD operations. GraphQL for frontend-heavy apps with complex data requirements. gRPC for internal microservice communication where performance matters most. Many systems use all three: gRPC internally, REST for public API, GraphQL for frontend.

Common Mistakes

✗ Using verbs in URLs:
  POST /createUser          → POST /users
  GET /getUser/123           → GET /users/123
  POST /deleteUser/123      → DELETE /users/123

✗ Not using proper status codes:
  200 {"error": "Not found"} → 404 Not Found
  200 {"error": "Unauthorized"} → 401 Unauthorized

✗ Inconsistent naming:
  /users, /User, /getUsers → pick one (lowercase plural)

✗ No pagination:
  GET /users → returns 1M users → timeout, OOM
  → GET /users?page=1&limit=50

✗ No versioning:
  Changing API breaks existing clients
  → Use /v1/, /v2/ versioning

Common Interview Questions

  • "What makes an API RESTful?"
    Uses standard HTTP methods for CRUD (GET/POST/PUT/PATCH/DELETE), resource-based URLs (nouns, not verbs), proper status codes, stateless requests, and standard JSON format.
  • "REST vs GraphQL — when would you choose?"
    REST for simple CRUD, public APIs, when HTTP caching is important. GraphQL when the frontend needs flexible data access to complex nested resources, reducing over/under-fetching. See: GraphQL
  • "How would you version your API?"
    URL path versioning (/v1/users) is simplest and most common. Header-based (Accept: application/vnd.api.v2+json) is cleaner but harder to test. Query parameter (?version=2) is not recommended.
