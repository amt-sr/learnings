What is a Recursive Resolver?

A recursive resolver (also called a DNS recursor or recursive DNS server) is the server that receives DNS queries from client applications and does the work of tracking down the answer. It iteratively queries root, TLD, and authoritative servers until it obtains the final IP address, then returns that answer to the client.

Key insight: The recursive resolver is the client's advocate. The client makes one simple query ("what is the IP for api.example.com?") and the resolver handles the entire multi-step resolution process. This is why it is called "recursive" — the resolver recursively follows referrals until it gets a definitive answer.

Where the Resolver Sits

┌──────────┐ ┌──────────────────────────────────┐ │ │ Query: api.example.com? │ │ │ Client │──────────────────────────→│ Recursive Resolver │ │ (Browser│ │ (ISP resolver or public DNS) │ │ / App) │←──────────────────────────│ │ │ │ Answer: 203.0.113.42 │ │ └──────────┘ └──────────┬───────────────────────┘ │ ┌──────────────────────────┼──────────────────────────┐ │ │ │ ▼ ▼ ▼ ┌────────────────┐ ┌────────────────┐ ┌──────────────────┐ │ Root Servers │ │ TLD Servers │ │ Authoritative │ │ (13 clusters) │ │ (.com, .org..) │ │ Name Servers │ └────────────────┘ └────────────────┘ └──────────────────┘ The resolver iterates through these on behalf of the client. The client only ever talks to the resolver.

How Resolution Works (Resolver's Perspective)

StepResolver ActionDetails
1Check local cacheIf cached and TTL not expired, return immediately. No upstream queries needed.
2Query root serverAsk: "Where can I find .com?" Root returns NS records for the .com TLD.
3Query TLD serverAsk: "Where can I find example.com?" TLD returns NS records for example.com.
4Query authoritative serverAsk: "What is the IP for api.example.com?" Authoritative returns the A/AAAA record.
5Cache and respondStore result in cache with TTL. Return IP to client.
Iterative, not recursive at each step: The resolver itself makes iterative queries to root, TLD, and authoritative servers. Each server does NOT recurse on the resolver's behalf — it only returns a referral to the next level. Only the resolver's relationship with the client is recursive.

Resolver-Level Caching

The resolver maintains a cache of previously resolved domain names. When any client queries for a cached domain whose TTL has not expired, the resolver returns the answer immediately without contacting upstream servers.

Resolver Cache Behavior: First query (cold): Client → Resolver [cache miss] → Root → TLD → Authoritative → 203.0.113.42 Resolver stores: api.example.com → 203.0.113.42 (TTL: 300s) Latency: ~30-100ms Second query (warm, within TTL): Client → Resolver [cache hit] → 203.0.113.42 Latency: ~1-5ms (single RTT to resolver) Query after TTL expires: Client → Resolver [cache expired] → Root → TLD → Authoritative → 203.0.113.42 Latency: ~30-100ms (full resolution again) Cache invalidation triggers: - TTL expiry (most common) - Manual cache flush - Resolver restart
Why public resolvers have high cache hit rates: Google's 8.8.8.8 handles over 1 trillion queries per day. For any popular domain (Google, Facebook, Amazon), the resolver almost always has a fresh cached answer. This means the "100ms worst case" is rare in practice for well-known services.

EDNS Client Subnet (ECS)

ECS allows the recursive resolver to pass a truncated version of the client's IP address to the authoritative server. This enables geoDNS — returning different IPs based on the client's location — which is critical for CDN routing.

Without ECS:
  Client (NYC) → Resolver (8.8.8.8 in Iowa) → Authoritative
  Authoritative sees: resolver's IP in Iowa
  Returns: closest server to Iowa (not NYC)

With ECS:
  Client (NYC) → Resolver (8.8.8.8) → Authoritative
  Resolver appends: edns-client-subnet: 203.0.113.0/24 (NYC)
  Authoritative returns: closest server to NYC

Common Interview Questions

  • "What does a recursive resolver do?"
    Receives DNS queries from clients and iteratively walks the DNS hierarchy (root → TLD → authoritative) to resolve domain names to IPs. Caches results based on TTL. Acts as the client's single point of contact for DNS.
  • "Why would you use 8.8.8.8 instead of your ISP's DNS?"
    Public resolvers like 8.8.8.8 have larger caches, higher reliability, better security (DNSSEC validation, malware blocking), and no incentive for NXDOMAIN hijacking. Trade-off: slightly higher latency vs. ISP resolver, and the public resolver sees all your queries.
  • "How does the resolver decide when to re-query?"
    Each DNS record has a TTL (time-to-live) set by the authoritative server. The resolver caches the record and does not re-query until TTL expires. Some resolvers proactively refresh records near expiry to avoid cold misses.
  • "What happens if the resolver goes down?"
    Clients with cached results continue to work until TTL expires. Clients can be configured with fallback resolvers. Most OS configs allow specifying multiple resolvers (primary + secondary) for redundancy.
