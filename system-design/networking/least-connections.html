Least Connections

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • How the least connections algorithm dynamically selects the server with fewest active connections
  • Why it is superior to round robin when request durations vary
  • Weighted least connections for heterogeneous server pools
  • Trade-offs and real-world configuration

Definition

Least connections is a dynamic load balancing algorithm that routes each new request to the backend server with the fewest active connections at that moment. Unlike round robin (which assigns sequentially regardless of server load), least connections adapts in real time to how busy each server is.

Problem with Round Robin when requests vary in duration: Round Robin (naive): Req 1 ──▶ A (fast, 10ms) → A finishes quickly Req 2 ──▶ B (slow, 5000ms) → B is still processing Req 3 ──▶ C (slow, 5000ms) → C is still processing Req 4 ──▶ A (slow, 5000ms) → A now busy too Req 5 ──▶ B (fast, 10ms) → B still busy with Req 2! After 10 requests: A: 4 connections (some done, some pending) B: 3 connections (some done, some pending) C: 3 connections → Load accumulates unevenly because duration varies Least Connections (adaptive): Req 1 ──▶ A (fast, 10ms) → A finishes quickly, 0 active Req 2 ──▶ B (slow, 5000ms) → B busy, 1 active Req 3 ──▶ C (slow, 5000ms) → C busy, 1 active Req 4 ──▶ A (A has 0 active connections!) → A, 1 active Req 5 ──▶ A (A finished Req 4!) → A, 1 active Req 6 ──▶ A (A finished Req 5!) → A, 1 active Req 7 ──▶ A (B/C still busy!) → A, 1 active → The fast server handles more traffic naturally, while slow servers aren't overloaded.

How It Works

  1. For each incoming request, the LB checks the active connection count of every backend
  2. It selects the server with the lowest count
  3. If there is a tie, it typically uses round robin among tied servers
  4. The connection count is incremented when a request starts and decremented when it completes
# Pseudocode
def select_server(servers):
    # Find server with minimum active connections
    best = None
    min_connections = infinity
    for server in servers:
        if server.active_connections < min_connections:
            min_connections = server.active_connections
            best = server
    return best

# Time complexity: O(n) — must scan all servers
# Space complexity: O(n) — track connection count per server

Weighted Least Connections

For heterogeneous server pools, weighted least connections normalizes the connection count by dividing it by the server's weight. A server with weight 2 can handle twice as many connections as weight 1.

Weighted Least Connections formula: selected = server with lowest (active_connections / weight) Example: Server A: weight=4, connections=8 → 8/4 = 2.0 Server B: weight=2, connections=3 → 3/2 = 1.5 ← Selected! Server C: weight=1, connections=1 → 1/1 = 1.0 ← Selected! Server C gets the request — it has the lowest ratio. Even though A has more total capacity, C is more "available" relative to what it can handle.

Configuration Examples

# Nginx — least connections
upstream backend {
    least_conn;
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}

# Nginx — weighted least connections
upstream backend {
    least_conn;
    server 10.0.1.10:8080 weight=5;  # Higher capacity
    server 10.0.1.11:8080 weight=3;
    server 10.0.1.12:8080 weight=2;
}

# HAProxy — least connections
backend api_backend
    balance leastconn
    server api1 10.0.1.10:8080 weight 5
    server api2 10.0.1.11:8080 weight 3
    server api3 10.0.1.12:8080 weight 2

# (similar to least connections, counts pending requests)
AWS ALB note: AWS ALB uses least outstanding requests by default, not round robin. This means it routes to the target with the fewest in-flight requests. This is especially important because ALB targets can have very different response times (Lambda cold starts, varying EC2 load).

When Least Connections Shines

  • APIs with variable response times — Some endpoints return in 10ms, others in 5s. Round robin would overload slow servers.
  • WebSocket connections — Long-lived connections that accumulate. New connections should go to the server with fewer open sockets.
  • File uploads/downloads — Requests that take seconds to minutes. A server handling a large upload should not receive new requests.
  • Microservices with varying workloads — Different services behind the same LB have different processing times.

Edge Cases and Pitfalls

Slow server problem: If a server is slow (e.g., hardware issue, resource contention), it finishes requests slowly and keeps a low connection count. The LB sees few active connections and sends more requests to it, creating a death spiral. Mitigate this with health checks, circuit breakers, and weighted least connections (reduce the slow server's weight).
Connection counting accuracy: The LB must accurately track when connections open and close. With keep-alive connections, a connection may be idle but still counted as active. Use idle timeouts or track active requests (not just TCP connections) for better accuracy.

Common Interview Questions

  • "What is the least connections algorithm?"
    Routes each request to the server with the fewest active connections. It's a dynamic algorithm — O(n) per request because it must scan all servers. Best suited for variable-duration requests where round robin would cause uneven load. A fast server that finishes requests quickly naturally receives more traffic.
  • "When would you use least connections instead of round robin?"
    When requests have variable processing times. Example: an API where GET /users takes 10ms but POST /report takes 5 seconds. With round robin, a server could accumulate many slow requests. Least connections sends new requests to whichever server is least busy, adapting to real-time load.
  • "What is weighted least connections?"
    Like least connections, but divides each server's connection count by its weight. This allows servers with higher capacity to handle proportionally more connections. The selected server has the lowest ratio of (active_connections / weight).
  • "What are the weaknesses of least connections?"
    O(n) scan per request (vs O(1) for round robin), requires tracking state per server, and can suffer from the "slow server problem" where a struggling server gets more requests because it has fewer active connections (it processes them slowly). Mitigate with health checks and circuit breakers.
  • "What algorithm does AWS ALB use by default?"
    Round robin is NOT the default. AWS ALB uses least outstanding requests, which is functionally similar to least connections — it counts in-flight requests rather than TCP connections. This is better because it accounts for requests in the LB's own queue, not just connections to the backend.

What to Skip

  • Optimized least connections data structures (min-heap implementations)
  • Least time (fastest response) algorithm — niche variant
  • Connection draining during server removal details
