Cache Hit

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What a cache hit is and how the CDN decides to serve from cache
  • Cache hit ratios and what drives them higher or lower
  • Cache-Control headers and how they control cacheability
  • Typical hit ratios for different content types (70-95%)

Definition

A cache hit occurs when the CDN edge server finds the requested resource in its local cache and serves it directly without contacting the origin server. The client receives the response entirely from the edge, resulting in the lowest possible latency and zero load on the origin.

Cache Hit Flow: ┌───────┐ ┌────────────────┐ ┌────────┐ │Client │──GET──► │ Edge Server │ │ Origin │ │ │ │ │ │ │ │ │◄─200─── │ Found in │ (never │ │ │ │ ~10ms │ local cache! │ called) │ │ └───────┘ └────────────────┘ └────────┘ Cache Miss Flow (for comparison): ┌───────┐ ┌────────────────┐ ┌────────┐ │Client │──GET──► │ Edge Server │──GET──► │ Origin │ │ │ │ │◄─200─── │ │ │ │◄─200─── │ Store in │ │ │ │ │ ~200ms │ cache, serve │ │ │ └───────┘ └────────────────┘ └────────┘

How the CDN Decides: Cache Lookup

When a request arrives at the edge, the CDN follows a deterministic lookup process:

Request arrives at edge: GET /images/logo.png Step 1: Build cache key cache_key = hash(method + host + path + query_string + vary_headers) example: sha256("GET cdn.example.com /images/logo.png") Step 2: Look up cache key in local storage ├── Found → CHECK FRESHNESS │ ├── Fresh (not expired) → SERVE FROM CACHE (cache hit) │ │ Response headers: │ │ X-Cache: HIT │ │ Age: 3600 (seconds since cached) │ │ CF-Cache-Status: HIT (Cloudflare) │ │ X-Amz-Cf-Pop: NRT57 (CloudFront) │ │ │ └── Stale (expired but still stored) │ ├── Serve stale + revalidate in background │ │ (stale-while-revalidate) │ └── Or revalidate with origin first │ If-None-Match / If-Modified-Since │ ├── 304 Not Modified → SERVE CACHED (revalidated hit) │ └── 200 OK (new content) → UPDATE CACHE, serve new │ └── Not found → CACHE MISS (fetch from origin)

Cache-Control Headers

The origin server controls cacheability through HTTP response headers. These headers tell the CDN what to cache and for how long.

HeaderExampleMeaning
Cache-Control: max-agemax-age=3600Cache for 3600 seconds (1 hour)
Cache-Control: s-maxages-maxage=86400Shared cache (CDN) TTL: 1 day; overrides max-age for CDNs
Cache-Control: no-storeno-storeNever cache (user data, personalized pages)
Cache-Control: no-cacheno-cacheCache but always revalidate before serving
Cache-Control: publicpublicMay be cached by any cache (CDN, browser, proxy)
Cache-Control: privateprivateOnly browser may cache, CDN must not
Cache-Control: must-revalidatemust-revalidateAfter expiry, must verify with origin before using stale copy
Cache-Control: immutableimmutableNever revalidate (content never changes, e.g., hashed filenames)

Stale-While-Revalidate and Stale-If-Error

Cache-Control: max-age=3600, stale-while-revalidate=86400, stale-if-error=604800

  Time    State       Behavior
  ─────────────────────────────────────────────────────────
  0s      Fresh       Serve from cache, no validation
  3600s   Stale       Serve stale + trigger background revalidation
                      (user gets instant response, cache updates async)
  3600s   Error       If origin returns 5xx, serve stale for up to 7 days
  90000s  Expired     Must fetch from origin, nothing to serve if error

Validation Headers

When the CDN needs to check if a cached resource is still current, it sends validation headers to the origin:

MechanismRequest HeaderResponseResult
ETagIf-None-Match: "abc123"304 Not ModifiedServe cached copy (saves bandwidth)
Last-ModifiedIf-Modified-Since: Wed, 21 Oct 2025304 Not ModifiedServe cached copy
ETag changedIf-None-Match: "abc123"200 OK (new body)Update cache, serve new
304 vs full response: A 304 response has an empty body (just headers). This saves significant bandwidth for large assets like images or videos. The CDN updates its cached copy and serves it. This is a "revalidated hit" — the CDN had the resource but confirmed it is still valid.

Cache Hit Ratio

Cache hit ratio = cache hits / (cache hits + cache misses). This is the single most important CDN metric. Higher is better.

Content TypeTypical Hit RatioWhy
Static assets (JS, CSS, images with hashed URLs)90-95%Filenames change when content changes; can cache forever
Images, fonts (without hashing)80-90%Popular images requested repeatedly; long TTLs
Video streams (HLS/DASH segments)85-95%Same segments served to many viewers
API responses (short TTL)50-70%Frequent invalidation, varied query params
Dynamic/user-specific content0-10%Not cacheable (Cache-Control: private/no-store)
Overall site average70-95%Depends heavily on cache strategy

Impact of Hit Ratio on Origin Load

Scenario: 1 million requests per hour, origin can handle 200K requests/hour

  Hit Ratio: 95%   → 50K requests to origin   ✓ well within capacity
  Hit Ratio: 80%   → 200K requests to origin   ⚠ at capacity limit
  Hit Ratio: 60%   → 400K requests to origin   ✗ origin overflows

  A 15% drop in hit ratio doubled the origin load!

  Cost impact (assuming origin costs 10x more per request than CDN):
  Hit Ratio 95%:  50K origin × $0.01 + 950K CDN × $0.001  = $1,450/hr
  Hit Ratio 80%: 200K origin × $0.01 + 800K CDN × $0.001  = $2,800/hr
  Hit Ratio 60%: 400K origin × $0.01 + 600K CDN × $0.001  = $4,600/hr
Interview tip: When designing a system, always discuss cache hit ratio. A well-designed CDN strategy should target 90%+ hit ratios for static assets. Mention content hashing (putting a hash in the filename like app.a1b2c3.js) as the primary technique for achieving this, combined with Cache-Control: immutable.

What Drives Hit Ratio Down

  • URL proliferation — unique query parameters (e.g., /image.jpg?timestamp=123) create separate cache entries. Each unique URL is a potential miss.
  • Low TTLs — aggressive max-age values cause frequent revalidation, increasing miss-like behavior.
  • Long-tail content — millions of rarely-accessed URLs (e.g., user profile images for inactive users) each get fetched once and then sit idle.
  • Small cache capacity — if the edge server's cache fills up, it evicts entries (usually LRU), turning what would be hits into misses.
  • Authentication tokens in URLs — /api/user?token=abc is never a cache hit because every user has a unique token.
  • Incorrect Vary headers — Vary: User-Agent creates separate cache entries for every browser version, fragmenting the cache.

Vary Header and Cache Key Complexity

The Vary header tells the CDN which request headers affect the response.
Each unique combination creates a separate cache entry.

  Vary: Accept-Encoding
    → 2 entries: gzip version + br (Brotli) version
    → Normal and expected

  Vary: User-Agent
    → Thousands of entries: one per unique User-Agent string
    → Cache fragmentation! Hit ratio drops dramatically

  Vary: Cookie
    → Potentially one entry per user
    → Effectively disables caching for that resource

  Best practice: Only Vary on headers that genuinely change the response
  For most static content: Vary: Accept-Encoding (for compression)
  For localized content:   Vary: Accept-Language

Common Interview Questions

  • "What is a cache hit ratio and what's a good one?"
    Cache hit ratio = hits / (hits + misses). For static assets with content hashing, 90-95% is achievable. For a mixed-traffic site, 70-80% is good. Below 50% usually indicates a caching configuration problem (unique query params, low TTLs, auth tokens in URLs).
  • "How does the CDN decide whether to serve from cache?"
    The CDN builds a cache key from the URL (and Vary headers), looks it up in local storage. If found and not expired (per Cache-Control max-age), it serves directly. If expired, it may serve stale while revalidating. If not found, it triggers a cache miss and fetches from origin.
  • "How would you improve a low cache hit ratio?"
    (1) Normalize query parameters (strip tracking params like utm_source), (2) use content hashing in filenames, (3) increase TTLs with stale-while-revalidate, (4) remove unnecessary Vary headers, (5) move auth from query params to request headers, (6) increase edge cache capacity.

What to Skip

  • Browser cache vs CDN cache internals — focus on CDN behavior in system design
  • Detailed Vary: * edge cases — just know it effectively disables caching
  • Legacy Expires header — Cache-Control superseded it; know Expires exists but don't deep-dive
