Keep Alive

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What keep-alive does and why it's the default in HTTP/1.1
  • Performance impact: reusing connections vs opening new ones
  • Connection pooling as the practical implementation
  • Keep-alive's role in microservice latency

Definition

HTTP Keep-Alive allows a single TCP connection to be reused for multiple HTTP requests/responses, eliminating the overhead of establishing a new connection for each request.

Analogy: Without keep-alive, every request is like making a new phone call (dial, wait, hang up). With keep-alive, it's like keeping the line open and having a continuous conversation — much faster.

Without vs With Keep-Alive

WITHOUT Keep-Alive (HTTP/1.0 behavior): Request 1: DNS + TCP handshake + TLS + HTTP + TCP close Request 2: DNS + TCP handshake + TLS + HTTP + TCP close Request 3: DNS + TCP handshake + TLS + HTTP + TCP close WITH Keep-Alive (HTTP/1.1 default): DNS + TCP handshake + TLS (once) ├── HTTP Request 1 → Response 1 ├── HTTP Request 2 → Response 2 └── HTTP Request 3 → Response 3 TCP close (after idle timeout) Savings: Eliminates 2 TCP handshakes + 2 TLS handshakes Per connection saved: ~2-6 RTTs (~2-600ms depending on location)

Connection Pooling

Keep-alive is the protocol feature. Connection pooling is the practical implementation — maintaining a pool of reusable connections.

How connection pooling works:

  Client maintains N persistent connections to server
  Incoming request → pick idle connection from pool → send request
  Response received → return connection to pool

  Benefits:
  ✓ No TCP/TLS setup per request
  ✓ Bounded resource usage (max connections in pool)
  ✓ Better throughput (parallel requests on different connections)

  Configurable parameters:
  max_connections: 50-100 (tune per QPS)
  max_idle_time: 30s (close unused connections)
  max_lifetime: 5min (prevent stale connections)
System design impact: Connection pooling is one of the highest-impact optimizations. A service making 10K QPS without pooling opens 10K TCP connections/sec. With pooling of 100 connections, it reuses them — eliminating 99% of connection overhead.

Keep-Alive in Microservices

Without connection pooling: Service A → Service B: new TCP connection per request At 5K QPS: 5,000 TCP connections/sec opened and closed → Port exhaustion, TIME_WAIT accumulation, high latency With connection pooling: Service A → Service B: pool of 50 connections At 5K QPS: same 50 connections reused → Stable, fast, efficient This is why HTTP clients (OkHttp, http2, Go's http.Client) enable connection pooling by default.

Common Interview Questions

  • "What is HTTP keep-alive?"
    Reusing a single TCP connection for multiple HTTP requests. Default in HTTP/1.1. Eliminates TCP handshake and TLS negotiation overhead per request.
  • "What's the performance benefit?"
    Saves 1 RTT (TCP) + 1-2 RTTs (TLS) per reused connection. Same datacenter: ~3ms. Cross-region: ~150ms. For a page with 20 requests, that's 60-3000ms saved.
  • "What's connection pooling?"
    Maintaining a pool of persistent connections instead of opening/closing per request. Eliminates TCP handshake overhead and bounds resource usage. Essential for high-QPS services.
  • "What happens without connection pooling?"
    Port exhaustion, TIME_WAIT accumulation, high CPU from connection setup, high latency from repeated handshakes, potential connection refusal from server.
