Status Codes

Difficulty: ⭐☆☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • The 5 status code categories
  • The most common codes and when to use each one
  • Common API mistakes with status codes
  • How status codes relate to circuit breakers and retry logic

Server Error Codes (5xx)

CodeNameMeaningShould Retry?
500Internal Server ErrorUnexpected server failureYes (with backoff)
502Bad GatewayUpstream server returned invalid responseYes
503Service UnavailableServer overloaded or in maintenanceYes (after Retry-After)
504Gateway TimeoutUpstream server didn't respond in timeYes
Retry strategy: Only retry 429 and 5xx errors (with exponential backoff). Never retry 4xx (except 429) — the problem is the request, not the server. 409 might be retried with an idempotency key. See: Exponential Backoff

Common Mistakes

✗ Using 200 for errors
  {"status": 200, "error": "User not found"}
  → Always use proper error codes. 404 for not found.

✗ Using 500 for all errors
  500 for "invalid email format"
  → That's a 400 Bad Request, not a server error.

✗ Using 404 for "no search results"
  GET /users?q=nonexistent → 404
  → The endpoint exists. Return 200 with empty array [].

✗ Using 403 and 401 interchangeably
  401 = "Who are you?" (no auth or bad token)
  403 = "I know who you are, but you can't do this" (no permission)

✗ Not returning Location header on 201
  POST /users → 201 Created
  → Should include: Location: /users/456

Status Codes and System Design

Load Balancer sees 5xx → marks server unhealthy → stops routing Client sees 429 → backs off → retries after Retry-After Client sees 401 → redirects to login page Monitoring counts 5xx → triggers alert → on-call paged Status codes are signals that drive automated behavior. Use them correctly — your infrastructure depends on it.

Common Interview Questions

  • "What's the difference between 401 and 403?"
    401 Unauthorized = authentication failed (missing or invalid credentials). 403 Forbidden = authentication succeeded but no permission for this action.
  • "When would you return 201 vs 200?"
    201 when a new resource is created (POST). 200 for successful reads (GET) and updates (PUT/PATCH). 201 should include a Location header pointing to the new resource.
  • "Should you retry a 500 error?"
    Yes, with exponential backoff. 500 indicates a transient server error. But add a max retry count (typically 3-5) and circuit breaker logic to prevent retry storms.
  • "What status code for rate limiting?"
    429 Too Many Requests. Include a Retry-After header telling the client when to retry. See: Rate Limiting.
