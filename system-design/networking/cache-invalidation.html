Cache Invalidation

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand:

  • Three strategies for CDN cache invalidation: time-based (TTL), purge, and versioned URLs
  • Why "there are only two hard things in Computer Science: cache invalidation and naming things"
  • Best practices: content hashing in URLs as the gold standard
  • Trade-offs between each invalidation approach

Definition

Cache invalidation is the process of removing or updating cached content so that users receive fresh content instead of stale data. It is notoriously difficult because cached copies exist across hundreds of edge PoPs globally, and stale content can persist for hours or days if invalidation is not handled correctly.

"There are only two hard things in Computer Science: cache invalidation and naming things." — Phil Karlton. This quote comes up frequently in interviews. The core problem: once a resource is cached at 300 edge PoPs, how do you ensure every single one serves the updated version, without causing a thundering herd back to the origin?

Three Invalidation Strategies

┌─────────────────────────────────────────────────────────────┐ │ │ │ Strategy 1: Time-based (TTL) │ │ "Let it expire naturally" │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ Resource cached at T=0 with max-age=3600 │ │ │ │ T=0 to T=3599 → Serve from cache (may be stale!) │ │ │ │ T=3600 → Expired, revalidate with origin │ │ │ └─────────────────────────────────────────────────────┘ │ │ Pro: Zero operational effort │ │ Con: Up to TTL seconds of stale content │ │ │ │ Strategy 2: Purge (explicit invalidation) │ │ "Tell the CDN to delete cached copies now" │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ Content updated at origin │ │ │ │ API call: PURGE /api/products │ │ │ │ CDN removes from all PoPs → next request = MISS │ │ │ │ Next fetch gets fresh content from origin │ │ │ └─────────────────────────────────────────────────────┘ │ │ Pro: Immediate invalidation │ │ Con: Causes cache misses (thundering herd risk) │ │ │ │ Strategy 3: Versioned URLs (content hashing) │ │ "Never invalidate — deploy new URLs instead" │ │ ┌─────────────────────────────────────────────────────┐ │ │ │ v1: /css/app.a1b2c3.css (cached forever) │ │ │ │ v2: /css/app.d4e5f6.css (new hash = new URL) │ │ │ │ v1 stays in cache but is never requested again │ │ │ │ v2 is fetched, cached, served │ │ │ └─────────────────────────────────────────────────────┘ │ │ Pro: No invalidation needed, no stale content, no misses │ │ Con: Requires build system to generate hashed filenames │ │ │ └─────────────────────────────────────────────────────────────┘

Strategy 1: Time-Based (TTL)

The simplest approach. Set a Cache-Control: max-age header, and the CDN automatically evicts the resource after that duration. No active invalidation needed.

TTL ValueUse CaseStaleness Risk
10-60 secondsAPI responses, live data10-60s stale
5-60 minutesNews articles, leaderboards5-60 min stale
1-24 hoursProduct pages, user profiles1-24 hours stale
7-365 daysStatic assets with hashed URLsN/A (never stale)

Problems with TTL-Only

Scenario: Breaking news story published at 12:00
  - Homepage has max-age=300 (5 minutes)
  - CDN edge servers serve the old homepage until 12:05
  - Users see stale (pre-breaking-news) homepage for 5 minutes
  - Each edge PoP independently expires at different times
    (if cached at 12:01, expires at 12:06)
  - No way to force an immediate update

  With stale-while-revalidate:
  - max-age=300, stale-while-revalidate=3600
  - At T=300s, edge serves stale + fetches fresh in background
  - At T=301s, edge has fresh version for next request
  - But T=300s user still sees the stale version
TTL is passive invalidation. You accept some staleness in exchange for zero operational complexity. It works for content where eventual consistency is acceptable. It does NOT work when you need immediate updates.

Strategy 2: Purge

Active invalidation: you explicitly tell the CDN to remove specific cached entries. This is done via API calls or the CDN's dashboard.

Types of Purge

Purge TypeScopeSpeedRisk
URL purgeSingle URL: /images/hero.jpgFast (targeted)Low
Tag purgeAll URLs with a cache tag: product-123Fast (batched by tag)Low
Directory purgeAll URLs under a path: /images/*ModerateMay invalidate too much
Wildcard/host purgeEntire site or hostname: *.example.comSlow (every PoP)Massive miss storm

Purge API Example

Cloudflare Purge API:
  POST /zones/{zone_id}/purge_cache
  {
    "files": [
      "https://example.com/images/hero.jpg",
      "https://example.com/css/style.css"
    ],
    "tags": ["product-123", "homepage"]
  }

Fastly Purge API (instant soft purge):
  POST /purge/https://example.com/images/hero.jpg
  → Marks as stale, does not delete
  → Next request revalidates with origin
  → No thundering herd (stale copy available as fallback)

Amazon CloudFront invalidation:
  POST /2020-05-31/distribution/{dist_id}/invalidation
  {
    "InvalidationBatch": {
      "Paths": {
        "Quantity": 2,
        "Items": ["/images/*", "/css/style.css"]
      },
      "CallerReference": "deploy-20260628-001"
    }
  }
  → Note: AWS limits to 3,000 paths per request
  → Propagation can take 60-300 seconds
The purge problem: After a purge, every edge PoP needs to re-fetch from origin on the next request. For a popular resource with 300 PoPs, that means 300 cache misses in rapid succession. This is the thundering herd problem. Mitigations: use "soft purge" (mark stale but keep as fallback), origin shield to deduplicate, and pre-warm after purging.

Propagation Delay

Purge is NOT instant. CDN must propagate the invalidation to all PoPs.

  Cloudflare:  ~5-30 seconds  (fastest)
  Fastly:      ~1-3 seconds   (designed for real-time purging)
  CloudFront:  60-300 seconds (can be slow)
  Akamai:      ~30-120 seconds

  During propagation window:
  - Some PoPs still serve stale content
  - Some PoPs have already purged and need to re-fetch
  - Users see inconsistent versions depending on their PoP

  This is why purge alone is insufficient for consistency-critical content.

Strategy 3: Versioned URLs (Content Hashing)

The gold standard for cache invalidation. Instead of invalidating cached content, you deploy a new URL that embeds a content hash. The old URL stays cached (harmlessly) while the new URL is a cache miss that gets populated.

Content Hashing Strategy: Build Process: ┌──────────────────────────────────────────────────────────┐ │ 1. Compute hash of file contents: │ │ style.css (245 KB) → sha256 → "d4e5f6..." │ │ │ │ 2. Rename file to include hash: │ │ style.css → style.d4e5f6a7b8.css │ │ │ │ 3. Generate HTML reference: │ │ <link href="/css/style.d4e5f6a7b8.css"> │ │ │ │ 4. When content changes: │ │ style.css (246 KB, updated) → sha256 → "f7a8b9..." │ │ style.css → style.f7a8b9c0d1.css (NEW URL) │ │ │ │ Result: │ │ - Old URL (d4e5f6) stays in CDN cache forever │ │ - No one requests it anymore │ │ - New URL (f7a8b9) is a cache miss, fetched once │ │ - Zero invalidation needed │ └──────────────────────────────────────────────────────────┘ Timeline: Deploy v1 → /css/app.a1b2c3.css → CDN caches it (max-age=31536000) User visits → browser loads /css/app.a1b2c3.css → HIT Deploy v2 → /css/app.d4e5f6.css → CDN caches it (max-age=31536000) User visits → browser loads /css/app.d4e5f6.css → MISS (then HIT) Old URL → /css/app.a1b2c3.css → still cached, never requested

How Major Frameworks Handle Content Hashing

Vite / Webpack:
  Output:  app.a1b2c3.js, chunk.d4e5f6.js, style.f7a8b9.css
  HTML:    <script src="/assets/app.a1b2c3.js"></script>
  Headers: Cache-Control: max-age=31536000, immutable

  HTML file itself:
  Headers: Cache-Control: max-age=0, must-revalidate
  (HTML is NOT hashed — it references the hashed assets)

Next.js:
  Output:  _next/static/chunks/app.a1b2c3.js
  Headers: Cache-Control: public, max-age=31536000, immutable

Ruby on Rails:
  Output:  /assets/application-a1b2c3.css
  Helper:  stylesheet_link_tag "application"  → auto-adds hash

Django (whitenoise / django-storages):
  Output:  /static/css/app.a1b2c3.css
  Collectstatic + manifest.json maps original → hashed
This is the canonical solution to cache invalidation. In any system design interview, when asked about cache invalidation, lead with content hashing. It eliminates the invalidation problem entirely: old versions are never invalidated (they just become irrelevant), new versions are naturally fetched once and cached. The only "cost" is a build system that generates hashed filenames.

Real-World Strategy: Combine All Three

Production systems typically use all three strategies together, each for different content types:

Typical web application:

  ┌─────────────────────────────────────────────────────────┐
  │ Layer 1: Versioned URLs (content hashing)               │
  │   Content:  JS bundles, CSS, images, fonts             │
  │   TTL:      1 year (max-age=31536000, immutable)       │
  │   When updated: New hash → new URL → no invalidation   │
  ├─────────────────────────────────────────────────────────┤
  │ Layer 2: Short TTL + stale-while-revalidate            │
  │   Content:  API responses (GET /products, /news)       │
  │   TTL:      60s + stale-while-revalidate=600s          │
  │   When updated: Stale served for ≤60s, background refresh│
  ├─────────────────────────────────────────────────────────┤
  │ Layer 3: Active purge                                  │
  │   Content:  Homepage, dynamic pages with user content  │
  │   TTL:      300s                                       │
  │   When updated: Tag-based purge ("homepage", "user-42")│
  │   Fallback:   stale-while-revalidate for safety        │
  ├─────────────────────────────────────────────────────────┤
  │ Layer 4: No caching                                    │
  │   Content:  User dashboard, authenticated API calls    │
  │   Cache-Control: no-store, private                     │
  │   Always served from origin                            │
  └─────────────────────────────────────────────────────────┘

Cache Tags for Bulk Invalidation

Cache tags allow you to invalidate multiple URLs with a single API call, even if those URLs are different paths:

Setting cache tags (via origin response header):
  GET /products/123  →  Cache-Tag: product-123, products-list
  GET /products/123/reviews  →  Cache-Tag: product-123, reviews
  GET /categories/electronics  →  Cache-Tag: category-electronics

When product 123 is updated:
  Purge tag: "product-123"
  → Invalidates BOTH /products/123 AND /products/123/reviews

When a product review is updated:
  Purge tag: "product-123"
  → Invalidates both the product page and its reviews page

Cloudflare:  Cache-Tag header
Fastly:      Surrogate-Key header (same concept)
Akamai:      Edge-Cache-Tag header

Common Interview Questions

  • "How would you handle cache invalidation in a system design?"
    Three strategies in order of preference: (1) Content hashing in URLs for static assets — eliminates invalidation entirely, can cache forever. (2) Short TTLs with stale-while-revalidate for semi-dynamic content — acceptable staleness with automatic refresh. (3) Active purge (tag-based) for content that must update immediately — purge by tag to batch invalidations, use soft purge to avoid thundering herd.
  • "Why is cache invalidation hard?"
    Three reasons: (1) propagation delay — it takes time to reach all 300+ PoPs globally. (2) thundering herd — a purge creates simultaneous misses across all PoPs, potentially overwhelming origin. (3) consistency — during the propagation window, different users see different versions of the same content. Content hashing solves all three by making invalidation unnecessary.
  • "A user updates their profile picture but still sees the old one. How do you fix this?"
    The issue is CDN caching. Solutions: (1) if using content hashing, the new profile picture has a new URL, so the old cached version is irrelevant. (2) If the URL is fixed (e.g., /avatar.jpg), purge the URL (or a tag like user-42) from all CDN PoPs. (3) Add a cache-busting query parameter as a fallback. (4) Set a shorter TTL for user-specific assets. Long-term fix: always use content-hashed URLs.
  • "You deploy a new version of your CSS but users report seeing a broken site. What happened?"
    The CDN is still serving the old CSS file. The HTML references /css/style.css, which is cached at the edge with a long TTL. The new CSS was deployed to origin, but the CDN has not expired or purged the old version. Fix: use content hashing so the new CSS has a new URL (/css/style.a1b2c3.css), or reduce the TTL, or purge the old URL after deploy.

What to Skip

  • CDN vendor-specific purge API syntax (know the concept, not the exact JSON format)
  • Cache hierarchy internals (L1/L2/L3 cache layers in a single PoP)
  • Deep dive into stale-if-error vs stale-while-revalidate edge cases
