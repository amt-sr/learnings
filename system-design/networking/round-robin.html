Round Robin

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • How the simple round-robin load balancing algorithm works
  • Its strengths: simplicity, even distribution, O(1) selection
  • Its weaknesses: ignores server capacity, assumes uniform request cost
  • When to use it vs more sophisticated algorithms

Definition

Round robin is the simplest load balancing algorithm — it distributes requests to backend servers in sequential order, cycling through the list and wrapping back to the first server after reaching the last. Each server gets an equal share of requests.

Round Robin Algorithm — 3 servers, 7 requests: Request 1 ──▶ Server A Request 2 ──▶ Server B Request 3 ──▶ Server C Request 4 ──▶ Server A ← Wraps back to A Request 5 ──▶ Server B Request 6 ──▶ Server C Request 7 ──▶ Server A Result: A=3, B=2, C=2 (difference of at most 1) Implementation (pseudocode): ┌─────────────────────────────────┐ │ servers = [A, B, C] │ │ index = 0 │ │ │ │ def select_server(): │ │ server = servers[index] │ │ index = (index + 1) % n │ │ return server │ └─────────────────────────────────┘

Properties

PropertyValue
Time complexityO(1) per request (increment + modulo)
Space complexityO(1) (only a counter)
DistributionPerfectly even (difference ≤ 1)
State requiredMinimal (just an integer counter)
Server awarenessNone (treats all servers equally)

Strengths

  • Simple to implement and reason about — no complex logic or configuration
  • Even distribution — every server gets the same number of requests (within 1)
  • No state per server — just a counter, easy to distribute across multiple LB instances
  • Works well with homogeneous servers — when all backends have the same capacity
  • Predictable — easy to debug, easy to reason about traffic distribution

Weaknesses

Ignores Server Capacity

Problem: Unequal server hardware Server A: 8 CPU cores, 32 GB RAM (2x capacity) Server B: 4 CPU cores, 16 GB RAM (1x capacity) Server C: 4 CPU cores, 16 GB RAM (1x capacity) Round robin distribution: A: ████████████ (33%) ← Underutilized B: ████████████ (33%) ← Overloaded C: ████████████ (33%) ← Overloaded Better: use Weighted Round Robin A: ████████████████████ (50%) B: ████████████ (25%) C: ████████████ (25%)

Assumes Uniform Request Cost

Round robin treats every request as equal. If some requests are expensive (heavy computation, large DB queries) and others are cheap (static lookups), round robin can overload a server with consecutive heavy requests.

No Connection Awareness

Round robin doesn't consider how many active connections each server has. A server handling long-running requests may accumulate more connections than others, leading to uneven load. See: Least Connections

New Server Cold Start

When a new server joins the pool, round robin immediately sends it a full share of traffic. If the server is still warming up (JVM JIT, cache loading), this can cause timeouts and errors.

Configuration Examples

# HAProxy — round robin (default)
backend api_backend
    balance roundrobin
    server api1 10.0.1.10:8080
    server api2 10.0.1.11:8080
    server api3 10.0.1.12:8080

# Nginx — round robin (default, no directive needed)
upstream backend {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    server 10.0.1.12:8080;
}

# AWS ALB — round robin within target group (default)

Common Interview Questions

  • "Explain the round-robin load balancing algorithm."
    Cycle through servers in order. Request 1 goes to Server A, request 2 to Server B, request 3 to Server C, then back to A. Simple, O(1), even distribution. The main downside is it ignores server capacity — a powerful server gets the same traffic as a weak one. For heterogeneous servers, use weighted round robin.
  • "When would round robin NOT be a good choice?"
    When servers have different hardware specs (some bigger than others), when requests have highly variable processing times (some take 10ms, some take 10s), when you need session affinity, or when you need cache locality (consistent hashing is better).
  • "What's the time complexity of round robin?"
    O(1) per request. The LB maintains a single counter, increments it on each request, and uses modulo to wrap around. No complex data structures needed. This makes it the fastest algorithm for basic load distribution.
  • "How do you handle a new server joining the pool with round robin?"
    The new server immediately starts receiving traffic. If the server needs warmup time, you'd use slow start mode (gradually increase traffic) or weighted round robin starting with a low weight that increases over time. AWS ALB supports slow start mode.

What to Skip

  • Implementation details of concurrent counter increments
  • Distributed counter consistency across multiple LB nodes
  • Least time (response time weighted) algorithm — rarely asked
