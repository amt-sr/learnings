Edge Servers

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What edge servers are and how CDN architecture works
  • How content is served from the geographically nearest edge location
  • Why edge servers dramatically reduce latency (200ms to 10ms)
  • TLS termination at the edge and its benefits
  • Major CDN providers and their edge network footprints

Definition

An edge server is a server placed at the logical extreme (edge) of a network, geographically close to end users. In a CDN (Content Delivery Network), thousands of edge servers are distributed across global Points of Presence (PoPs) to serve cached content with minimal round-trip latency.

Instead of every user's request traveling across continents to a single origin data center, the request is intercepted by the nearest edge server, which serves the response locally. This is the foundational concept behind every CDN.

Latency Reduction

Edge servers solve the fundamental physics problem of the speed of light. A request from Tokyo to Virginia travels roughly 13,000 km one-way over fiber, introducing unavoidable propagation delay.

ScenarioDistanceOne-way latencyRTT
User to same-city edge~50 km~0.5 ms~5-10 ms
User to same-region edge~500 km~5 ms~20-40 ms
User to cross-region origin~5,000-13,000 km~50-100 ms~100-200 ms
Key insight: An edge server in the same city as the user reduces latency from ~200ms (cross-region round trip) to ~10ms (local). That is a 20x improvement. For a page loading 50+ assets, this compounds into seconds of saved load time.

Latency Breakdown

Total latency = DNS + TCP handshake + TLS handshake + TTFB + Content transfer

Without CDN (Tokyo → Virginia):
  DNS lookup           ~30 ms
  TCP handshake        ~100 ms
  TLS handshake        ~100 ms
  Time to first byte   ~50 ms
  Content transfer     ~80 ms  (image, 500 KB)
                       ─────────
  Total                ~360 ms per asset

With CDN (Tokyo → Tokyo edge):
  DNS lookup           ~1 ms
  TCP handshake        ~5 ms
  TLS handshake        ~5 ms
  Time to first byte   ~1 ms  (cache hit)
  Content transfer     ~2 ms
                       ─────────
  Total                ~14 ms per asset

For a page with 30 assets: 10.8s → 0.4s

TLS Termination at the Edge

TLS termination means the edge server decrypts incoming HTTPS traffic and handles the TLS handshake on behalf of the origin. The connection from edge to origin can use either re-encrypted TLS or plain HTTP over a private backbone.

Client ──TLS──► Edge Server ──HTTP/2──► Origin │ ├── TLS termination here ├── Client sees edge's certificate ├── Edge ↔ Origin over private backbone (encrypted or not) └── Origin never deals with client TLS handshakes Benefits: ┌─────────────────────────────────────────────────────┐ │ 1. Offload TLS computation from origin │ │ - TLS handshake = 2 RTTs + CPU-heavy crypto │ │ - Edge handles this; origin serves content │ │ │ │ 2. Faster TLS negotiation │ │ - Edge is closer → fewer RTTs for handshake │ │ - TLS 1.3 + edge = 1 RTT (0-RTT with resumption)│ │ │ │ 3. Automatic certificate management │ │ - CDNs auto-renew Let's Encrypt or custom certs │ │ - Origin doesn't need cert infrastructure │ │ │ │ 4. Origin can use simpler deployment │ │ - No need to manage TLS libraries on app server │ │ - Focus on application logic │ └─────────────────────────────────────────────────────┘
Interview detail: When an interviewer asks "how does TLS termination work in a CDN?", explain that the edge holds the public certificate, decrypts the request, optionally re-encrypts for the origin-to-edge hop (often with a different cert or mutual TLS), then forwards the decrypted request to origin. The key benefit is that expensive asymmetric crypto (RSA/ECDSA) happens at the edge, not at your application servers.

How CDNs Route Users to Nearest Edge

Method 1: DNS-based routing (most common)
  1. Client queries DNS for cdn.example.com
  2. CDN's authoritative DNS uses Anycast
  3. BGP Anycast advertises same IP from multiple PoPs
  4. BGP routing naturally directs to nearest PoP
     (based on AS path length, latency, cost)

Method 2: HTTP redirect (less common, adds latency)
  1. Client hits cdn.example.com (single entry point)
  2. CDN returns 302 → cdn-tokyo.example.com/asset.jpg
  3. Client follows redirect to nearest edge

Method 3: AnyCast (what Cloudflare, Akamai use)
  All PoPs advertise the same IP prefix via BGP
  Internet routers automatically route to the topologically nearest PoP
  No DNS tricks or redirects needed
  Failover is automatic (if a PoP goes down, BGP re-routes)

Edge Server Responsibilities

An edge server does far more than just serve static files. In modern CDNs, edge servers handle:

  • TLS termination — decrypt HTTPS, manage certificates
  • Static content caching — serve cached JS/CSS/images
  • Dynamic content caching — cache API responses with short TTLs
  • Request routing — forward uncached requests to origin
  • Load balancing — distribute traffic across multiple origins
  • DDoS mitigation — absorb and filter malicious traffic
  • Web Application Firewall (WAF) — block SQL injection, XSS at the edge
  • Edge compute (Workers/Functions) — run code at the edge (redirects, A/B testing, auth checks)
  • Image optimization — resize, compress, convert format on-the-fly
  • Rate limiting — throttle abusive clients before they hit origin

Common Interview Questions

  • "How does a CDN reduce latency?"
    CDN deploys edge servers in PoPs worldwide. DNS/Anycast routes the user to the nearest edge. If the content is cached (cache hit), the edge serves it directly (5-10ms). If not (cache miss), the edge fetches from origin, caches it, and serves it. TLS termination at the edge also removes 2 RTTs of handshake latency from the origin path.
  • "What is TLS termination and why do it at the edge?"
    TLS termination means decrypting HTTPS at the edge server rather than the origin. Benefits: offloads CPU-intensive crypto from app servers, reduces TLS handshake latency (edge is closer), and simplifies certificate management. The edge-to-origin hop can be re-encrypted or use the CDN's private backbone.
  • "How does Anycast routing work in a CDN?"
    All CDN PoPs advertise the same IP prefix via BGP. The internet's routing protocols (BGP) naturally direct each user's traffic to the topologically nearest PoP based on AS-path length and metrics. This provides automatic failover too — if a PoP goes down, BGP converges and routes traffic to the next nearest.
  • "Design a CDN from scratch. What components do you need?"
    Global PoPs with edge servers, Anycast IP routing, a cache layer (local storage at each PoP), an origin fetch mechanism, DNS-based routing for user direction, a control plane for cache invalidation and configuration, TLS termination, monitoring and metrics (hit rate, latency per PoP), and a purge API for cache invalidation.
