HTTP Methods

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What each HTTP method does and its semantics
  • Idempotency and safety — the two properties that matter most
  • How methods map to CRUD operations
  • Common API design mistakes with methods

Core Methods

MethodPurposeIdempotentSafeHas BodyHas Response Body
GETRead a resourceYesYesNoYes
POSTCreate a resourceNoNoYesYes
PUTReplace a resource entirelyYesNoYesOptional
PATCHPartial updateMaybeNoYesOptional
DELETEDelete a resourceYesNoOptionalOptional
Safe: Does not modify server state (read-only).
Idempotent: Calling it once has the same effect as calling it N times. GET /users/1 always returns the same user. DELETE /users/1 deletes the user — calling it again returns 404, but the effect (user is deleted) is the same.

Method Details

GET

Rules: Must be safe and idempotent. Never has a request body.
Use for: Fetching resources, search queries.

GET /users                     → List users (with pagination)
GET /users/123                 → Get user 123
GET /users?role=admin          → Filter users
GET /users?sort=name&limit=50 → Sorted, paginated list

Common mistake: Using GET for actions that modify state.
  ✗ GET /users/123/delete     → NEVER do this
  ✓ DELETE /users/123         → Use DELETE instead

POST

Rules: NOT idempotent. Creates new resources or triggers actions.
Use for: Creating resources, complex queries, actions.

POST /users
  Body: {"name": "Alice", "email": "alice@example.com"}
  Response: 201 Created, Location: /users/456

POST /users/search
  Body: {"filters": {"role": "admin"}, "sort": "name"}
  Use POST when query is too complex for URL params.

POST /orders/123/checkout  → Action endpoint (not CRUD)

PUT vs PATCH

PUT: Full replacement. Send the entire resource.
  PUT /users/123
  Body: {"name": "Alice", "email": "alice@new.com", "role": "admin"}
  → Missing fields will be reset to defaults or cleared

PATCH: Partial update. Send only changed fields.
  PATCH /users/123
  Body: {"email": "alice@new.com"}
  → Only email changes, other fields preserved

  PUT is idempotent (send same full state → same result).
  PATCH may not be (depends on the operation).
    Example: PATCH {"op": "increment", "field": "view_count"} is NOT idempotent.

DELETE

Rules: Idempotent. Returns 204 No Content or 200 OK.
  DELETE /users/123 → 204 (success, no body)
  DELETE /users/123 → 404 (already deleted — idempotent: same effect)

  Batch delete:
  DELETE /users?ids=1,2,3 → Delete multiple users

Common Interview Questions

  • "What's the difference between PUT and PATCH?"
    PUT replaces the entire resource — all fields must be sent, missing fields are reset. PATCH updates only specified fields — other fields are preserved. PUT is always idempotent; PATCH depends on the operation.
  • "What does idempotent mean?"
    Calling the operation N times produces the same effect as calling it once. GET is idempotent (reading doesn't change state). POST is not (creates a new resource each time). PUT and DELETE are idempotent.
  • "Is POST idempotent?"
    No. POST /users creates a new user each time. Calling it 3 times creates 3 users. If you need idempotent creation, use PUT with a client-generated ID or check for duplicates.
  • "Why not use GET for everything?"
    GET must be safe (no side effects). It shouldn't modify state. GET URLs are logged, cached, bookmarked, and pre-fetched by browsers — you don't want any of those to trigger mutations.

What to Skip

  • CONNECT method (used for tunnels/proxies)
  • WebDAV methods (PROPFIND, LOCK, UNLOCK, etc.)
