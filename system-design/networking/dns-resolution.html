DNS Resolution

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • The complete DNS resolution flow from browser to authoritative server
  • Roles of the recursive resolver, root server, TLD server, and authoritative server
  • Latency characteristics of DNS resolution (~100ms uncached, 0ms cached)
  • Why DNS is a critical bottleneck and single point of failure in system design

What is DNS Resolution?

DNS resolution is the process of translating a human-readable domain name (e.g., api.example.com) into an IP address (e.g., 203.0.113.42). It involves a chain of queries across multiple servers, each responsible for a different level of the DNS hierarchy.

Key insight: DNS resolution happens before any HTTP request is made. If DNS is slow or fails, the entire request fails. This is why DNS is one of the most critical infrastructure components in system design interviews.

Latency Budget

ScenarioLatencyNotes
Browser cache hit~0msDNS already resolved; no network call
OS cache hit~0msOS-level DNS cache (nscd, systemd-resolved)
Resolver cache hit1–5msSingle RTT to resolver (8.8.8.8)
Resolver cache miss (cold)30–100ms3 sequential round trips: root → TLD → authoritative
DNS timeout / failure1–5sClient retries, falls back to secondary resolver
System design impact: For a service handling 1M requests/second with cold DNS, that is 100ms of latency added to every first request. If your DNS provider goes down, your entire service is unreachable — even if all your application servers are healthy. This is why multi-provider DNS and health checks are essential.

Common Interview Questions

  • "What happens when you type a URL in the browser?"
    Browser checks cache → OS checks cache → query goes to recursive resolver → root → TLD → authoritative → IP returned → TCP handshake → TLS handshake → HTTP request. DNS resolution is the first 3 hops of this chain.
  • "Why can DNS be a bottleneck?"
    Cold resolution adds 30–100ms. DNS provider outage takes down your entire service regardless of server health. DNSSEC validation adds CPU overhead. High query volume can overwhelm resolvers during traffic spikes.
  • "How many root servers exist?"
    13 logical root server names (A through M), but each is deployed via anycast to 1000+ physical instances worldwide. Any single root server going down has minimal impact due to anycast routing.
  • "What is DNS hijacking and how do you prevent it?"
    Attacker redirects DNS queries to malicious IPs. Prevention: DNSSEC (cryptographic signatures), DNS over HTTPS (DoH), DNS over TLS (DoT), and using trusted recursive resolvers.
  • "Design a DNS system that can handle 10M queries per second."
    Anycast deployment across multiple regions, caching at multiple levels (browser, OS, resolver, CDN edge), sharding zones across authoritative servers, health checks and automatic failover, DNSSEC for integrity, rate limiting and DDoS protection.
  • "What is EDNS0 and why does it matter?"
    Extension Mechanism for DNS version 0. Increases UDP message size beyond 512 bytes (up to 4096) and adds optional fields (client subnet, flags). Critical for DNSSEC responses and large record sets that would otherwise fall back to TCP.

What to Skip

  • Deep dive into DNS packet format (header, question, answer, authority, additional sections)
  • DNSSEC cryptographic details (RSASHA256, ECDSAP256SHA256 specifics)
  • Historical DNS protocols (NIS, WINS)
  • Zone file syntax details
