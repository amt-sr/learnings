Kubernetes (K8s)

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Understand Kubernetes as a container orchestration platform, its architecture (control plane, worker nodes), core resource types (pods, services, deployments), and how it enables self-healing, scaling, and declarative infrastructure management.

Definition

Kubernetes (K8s) is an open-source container orchestration platform that automates deployment, scaling, and management of containerized applications. It provides a declarative API where you describe the desired state of your system, and K8s continuously works to make the actual state match the desired state.

Kubernetes solves the operational problems that emerge when you have hundreds of containers across dozens of machines: scheduling, service discovery, load balancing, self-healing, rolling updates, and auto-scaling.

Declarative Model: Desired vs Actual State

Kubernetes is declarative: You describe WHAT you want (3 replicas of nginx), not HOW to achieve it. The controller loop continuously reconciles the desired state (in etcd) with the actual state (running pods). If a pod crashes, the controller notices the mismatch and creates a new one.
You write a YAML manifest: ┌──────────────────────────┐ │ apiVersion: apps/v1 │ │ kind: Deployment │ │ spec: │ │ replicas: 3 │ ← Desired state │ template: ... │ └──────────┬───────────────┘ │ kubectl apply ▼ ┌──────────────────────────┐ │ etcd (stores desired │ │ state) │ └──────────┬───────────────┘ │ Controller notices discrepancy ▼ ┌──────────────────────────┐ │ Controller creates │ │ pods to match desired │ │ state (3 pods running) │ ← Actual state converges └──────────────────────────┘ Self-healing: If pod dies → controller detects mismatch → creates replacement Scaling: Change replicas: 3 → 5 → controller creates 2 more pods

When to Use (and Not Use) Kubernetes

Use K8s when: You have 10+ services, multiple environments, need auto-scaling, have a dedicated platform team, or run on multi-cloud/hybrid infrastructure.
Do not use K8s when: You have 1-3 services, a small team, simple deployment needs. A managed container service (ECS, App Runner, Cloud Run) or even PaaS (Heroku, Render) is simpler and cheaper.

Common Interview Questions

  1. What is Kubernetes and what problem does it solve?
  2. Explain the Kubernetes architecture (control plane and worker nodes).
  3. What is the declarative model and how does reconciliation work?
  4. What is the difference between a pod and a deployment?
  5. What is the role of etcd in Kubernetes?
  6. How does Kubernetes handle a node failure?
  7. What are the main resource types in Kubernetes?
  8. When would you NOT use Kubernetes?

What to Skip

Avoid diving into: YAML syntax for specific resources, kubectl command flags, Helm chart templates, or Kubernetes admission webhooks. Focus on the architecture, the declarative model, and the key resource types and their relationships.
