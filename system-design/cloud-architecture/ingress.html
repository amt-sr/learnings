Ingress

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★

Goal

Understand Kubernetes Ingress as an L7 HTTP(S) router, how it provides path-based routing, TLS termination, and why it is more efficient than exposing every service via a LoadBalancer.

Definition

Ingress is a Kubernetes resource that defines HTTP/S routing rules for external access to services within a cluster. It acts as an L7 (application layer) reverse proxy that routes traffic based on hostnames and paths, handles TLS termination, and provides a single entry point for multiple services.

Ingress solves the problem of needing a separate LoadBalancer per service. Instead of exposing 20 services each with their own cloud load balancer (expensive), one Ingress controller with one LoadBalancer handles all HTTP routing.

Path-Based and Host-Based Routing

# Ingress rules:
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: main-ingress
spec:
  rules:
    - host: api.example.com
      http:
        paths:
          - path: /orders
            pathType: Prefix
            backend:
              service:
                name: order-service
                port:
                  number: 80
          - path: /users
            pathType: Prefix
            backend:
              service:
                name: user-service
                port:
                  number: 80
    - host: www.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend-service
                port:
                  number: 80

TLS Termination

Client ──HTTPS──▶ Ingress Controller ──HTTP──▶ Service 1. Client connects to https://api.example.com (port 443) 2. Ingress Controller handles TLS handshake 3. Decrypts traffic, inspects HTTP host + path 4. Routes to appropriate backend service over HTTP (ClusterIP) 5. Backend services do not need TLS certificates TLS cert stored in K8s Secret: ┌────────────────────────────┐ │ apiVersion: v1 │ │ kind: Secret │ │ type: kubernetes.io/tls │ │ data: │ │ tls.crt: (base64 cert) │ │ tls.key: (base64 key) │ └────────────────────────────┘
Security note: If you need encryption within the cluster (defense in depth), use TLS between Ingress and Services as well. Many compliance frameworks (PCI, HIPAA) require this. Also, mTLS via a service mesh provides end-to-end encryption.

Ingress vs LoadBalancer Service

DimensionIngressLoadBalancer Service
LayerL7 (HTTP/HTTPS routing)L4 (TCP/UDP forwarding)
RoutingPath-based, host-basedPort-based only
Cost1 LB for N servicesN LBs for N services
TLSTerminates TLS, manages certsPasses through (or terminates one cert)
ProtocolsHTTP/HTTPS onlyAny TCP/UDP protocol
Best forWeb services, APIsNon-HTTP (gRPC native, TCP, WebSocket)

Interview tip: When asked how external traffic reaches services in your system, explain: "I would use an Ingress Controller for HTTP routing with TLS termination. This gives me one entry point with path-based routing to all backend services, avoiding the cost of a LoadBalancer per service."

Common Interview Questions

  1. What is Kubernetes Ingress and how does it work?
  2. Compare Ingress with LoadBalancer Service.
  3. How does TLS termination work with Ingress?
  4. Why not expose every service as a LoadBalancer?
  5. What is an Ingress Controller and which ones exist?
  6. How would you route different paths to different services?
  7. How does Ingress relate to the API Gateway pattern?

What to Skip

Avoid diving into: Specific Ingress controller configuration (NGINX annotations, Traefik dynamic config), Ingress class details, or pathType matching rules (Exact vs Prefix vs ImplementationSpecific). Focus on the L7 routing concept and why it is more efficient than multiple LoadBalancers.
