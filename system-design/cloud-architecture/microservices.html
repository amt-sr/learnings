Microservices Architecture

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Understand microservices architecture, how to define service boundaries, the pros and cons relative to monoliths, and when microservices are the right choice. This is one of the most frequently discussed architectural decisions in system design interviews.

Definition

Microservices is an architectural style where a system is decomposed into small, independent services, each owning its data and communicating over well-defined APIs (typically HTTP/REST or gRPC). Each service is independently deployable, scalable, and developed by a small team.

The key properties: independent deployment, independent data ownership, independent scaling, and technology heterogeneity. Services communicate over the network (never shared memory or shared database).

Service Boundaries

Defining service boundaries is the hardest part of microservices. Bad boundaries create distributed monoliths — worse than the original monolith.

Guiding Principles

PrincipleDescription
Single responsibilityEach service owns one bounded context (Domain-Driven Design)
Data ownershipService owns its data exclusively; other services access it via API
Loose couplingServices should not depend on each other's internal implementation
High cohesionRelated functionality belongs in the same service
Team alignmentService boundary matches team boundary (Conway's Law)
Rule of thumb: A service should be small enough that a single team (2-8 engineers) can own it, build it, deploy it, and operate it independently. If you need two teams for one service, it is too big. If one team owns 10 services, they are too small.

Pros and Cons

AdvantageWhy It Matters
Independent deploymentShip features faster without coordinating with other teams
Independent scalingScale the payment service 10x without scaling the auth service
Technology diversityUse Python for ML, Go for high-throughput API, Java for CRUD
Team autonomyTeams own their services end-to-end (build, deploy, operate)
Fault isolationOne service crashing does not bring down the entire system
Organizational scalingMore teams can work in parallel without merge conflicts
DisadvantageWhy It Matters
Distributed system complexityNetwork latency, partial failures, retries, timeouts
Data consistencyNo ACID across services — eventual consistency, sagas
Operational overheadMonitoring, logging, tracing for 50+ services
Deployment complexityNeed orchestration (K8s), CI/CD pipelines per service
Testing complexityIntegration testing across services is harder
Network overheadInter-service calls are 10-1000x slower than in-process calls
Contract managementAPI versioning, backward compatibility, breaking changes

Common Anti-Patterns

  • Distributed monolith: Services are tightly coupled through synchronous calls, shared databases, or circular dependencies. You get the complexity of microservices with none of the benefits.
  • Nanoservices: Services that are too small (one endpoint each), creating excessive inter-service communication overhead.
  • Shared-nothing denial: Every service reimplements common concerns (auth, logging, config) instead of using shared libraries or a service mesh.
Interview tip: When asked to design a system, start by justifying your architectural choice. Say "For this problem, I would choose [monolith/microservices] because [specific reason related to team size, scaling needs, or domain complexity]." Then explain what would make you switch to the other approach.

Common Interview Questions

  1. When would you choose microservices over a monolith?
  2. How do you define service boundaries?
  3. What is a distributed monolith and how do you avoid it?
  4. How do microservices handle data consistency?
  5. What is the impact of microservices on team structure?
  6. How do services communicate with each other?
  7. What is the downside of microservices that people underestimate?
  8. Compare REST vs gRPC vs events for inter-service communication.

What to Skip

Avoid diving into: Specific frameworks (Spring Cloud, Istio configuration), detailed sagas pattern implementation, or service mesh internals unless asked. Focus on the trade-off reasoning and boundary definition — these are what interviewers evaluate.
