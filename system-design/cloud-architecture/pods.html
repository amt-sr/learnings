Pods

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★

Goal

Understand the pod as the smallest deployable unit in Kubernetes, how containers within a pod share resources, lifecycle management, and the implications for system design.

Definition

Pod is the smallest and simplest deployable unit in Kubernetes. A pod encapsulates one or more containers that share the same network namespace (IP, ports), storage volumes, and lifecycle. Containers within a pod can communicate via localhost.

Pods exist because Kubernetes schedules at the pod level, not the container level. The pod provides the shared context (network, storage) that tightly-coupled containers need to work together.

Shared Resources in a Pod

ResourceShared?Details
Network namespaceYesAll containers share one IP address and port space. Container A talks to Container B via localhost.
Storage volumesYesVolumes mounted at the pod level are accessible to all containers in the pod.
IPC namespaceYesContainers can share System V semaphores and POSIX message queues.
Process namespaceConfigurableBy default separate. Set shareProcessNamespace: true to see all processes.
Resource limitsNoEach container has its own CPU/memory limits.

Multi-Container Pod Patterns

PatternDescriptionExample
SidecarExtends the main container's functionality without changing itLog forwarder (Fluentd), proxy (Envoy), health checker
AmbassadorProxy that handles external communication for the main containerDB connection proxy, SSL terminator
AdapterTransforms output/input to match external expectationsMetrics exporter (Prometheus format), log format normalizer
Most common pattern: The sidecar pattern is by far the most used. A web server container + a sidecar that forwards logs to ELK, or a main container + Envoy sidecar for service mesh. Mention this in interviews to show practical K8s knowledge.

Why Pods Are Ephemeral

Key insight: Pods are intentionally ephemeral. If a node fails, the pod dies and is recreated elsewhere. Never store persistent data inside a container filesystem — use PersistentVolumes. Never rely on a pod's IP address — use Services with stable DNS names.

Anti-Patterns

  • Multiple independent apps in one pod: If containers do not need to share network or storage, they should be in separate pods. Put nginx and a web app together (sidecar), but do not put a web app and a cron job together.
  • Storing state in container filesystem: Container filesystem is ephemeral. When a pod is rescheduled, the container is recreated and data is lost.
  • Pinpointing pod IPs: Pods get new IPs on recreation. Always use Services for stable networking.

Common Interview Questions

  1. What is a pod and why does Kubernetes use pods instead of running containers directly?
  2. What resources do containers share within a pod?
  3. Explain the sidecar pattern with an example.
  4. Why are pods ephemeral and what are the implications?
  5. What is the pause container and why does it exist?
  6. What happens when a container in a pod crashes?
  7. When would you use multiple containers in a single pod?

What to Skip

Avoid diving into: Pod spec YAML details (resource requests/limits syntax, volume mount configuration), init container specifics, or pod disruption budgets (covered in deployments). Focus on the conceptual model: what a pod is, why it exists, and the sharing semantics.
