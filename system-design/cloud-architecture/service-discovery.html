Service Discovery

Difficulty: вӯҗвӯҗвӯҗ Interview Frequency: вҳ…вҳ…вҳ…

Goal

Understand the service discovery problem in dynamic environments where service instances are created and destroyed frequently, the difference between client-side and server-side discovery, and tools like Consul, Eureka, and Kubernetes DNS.

Definition

Service discovery is the mechanism by which services in a distributed system find each other. In dynamic environments (auto-scaling, container orchestration), service instances have ephemeral IP addresses, so a mechanism is needed to map logical service names to actual network locations.

In a monolith, services call each other via function calls. In microservices, Service A needs to call Service B, but Service B might have 5 instances running at different IPs. How does Service A know where to send the request? That is the service discovery problem.

The client sends requests to a single load balancer/router, which handles registry lookup and routing. AWS ALB, NGINX, and Kubernetes Services use this approach.

Client-Side vs Server-Side

DimensionClient-SideServer-Side
Client complexityHigher (needs discovery SDK)Lower (calls a single URL)
Network hopDirect (client вҶ’ service)Extra hop (client вҶ’ LB вҶ’ service)
Load balancingIn client libraryIn centralized LB
LatencyLower (no extra hop)Higher (one extra hop)
Language couplingSDK per languageLanguage-agnostic
LB bottleneck riskNoneLB can become bottleneck
In practice: Kubernetes uses server-side discovery (Services + kube-proxy). If you are on Kubernetes, you typically do not need a separate service discovery tool. Consul and Eureka are more relevant for non-Kubernetes environments (VMs, bare metal, AWS EC2).

Service Registration

Services must register themselves with the registry. Two approaches:

ApproachHow It WorksExample
Self-registrationService registers itself on startup, deregisters on shutdownEureka, Consul
Third-party registrationAn external agent (registrar) monitors and registers instancesKubernetes, Marathon

Health Checking

# Consul health check config
{
  "check": {
    "http": "http://localhost:8080/health",
    "interval": "10s",
    "timeout": "3s",
    "deregister_critical_service_after": "30s"
  }
}

# If health check fails for 30s, instance is removed from registry
# Prevents traffic being routed to dead instances
