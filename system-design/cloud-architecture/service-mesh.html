Service Mesh

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand the service mesh pattern, how sidecar proxies (Envoy) handle cross-cutting networking concerns, mTLS for inter-service encryption, and traffic management capabilities like canary deployments and circuit breaking.

Definition

Service mesh is an infrastructure layer that handles service-to-service communication (east-west traffic) transparently. It uses a sidecar proxy (typically Envoy) injected into each pod that intercepts all network traffic, adding mTLS, retries, circuit breaking, observability, and traffic management without changing application code.

A service mesh decouples networking concerns from application code. Developers write business logic; the mesh handles the networking infrastructure: encryption, load balancing, retries, timeouts, rate limiting, and telemetry collection.

Architecture: Sidecar Pattern

What a Service Mesh Provides

CapabilityWhat It DoesWithout Mesh
mTLSAutomatic encryption for all inter-service trafficImplement TLS in every service (error-prone)
Traffic managementCanary deployments, A/B testing, traffic splittingCustom header-based routing or separate infrastructure
Retries and timeoutsAutomatic retry with configurable backoffLibrary per language or manual implementation
Circuit breakingStop sending traffic to unhealthy servicesLibrary per language (Hystrix, Resilience4j)
ObservabilityAutomatic metrics, distributed traces, access logsInstrument every service individually
Load balancingClient-side LB with multiple algorithmsServer-side LB or library
Rate limitingPer-service rate limitsCustom middleware
Fault injectionInject delays/errors for chaos testingCustom test infrastructure

mTLS (Mutual TLS)

mTLS requires both the client and server to present certificates. The service mesh control plane (Istiod) automatically issues and rotates short-lived certificates for every sidecar. All inter-service traffic is encrypted without application code changes.
Without mTLS: App A ──plaintext──▶ App B ← anyone on the network can read this With mTLS: App A → Envoy (encrypt) ──TLS──▶ Envoy (decrypt) → App B Both Envoy sides verify each other's certificates Certificate lifecycle (Istio/Envoy): 1. Envoy sidecar starts → requests cert from Istiod 2. Istiod issues short-lived cert (24h, auto-rotated) 3. Envoy presents cert on every outbound connection 4. Receiving Envoy validates cert against Istiod CA 5. Traffic flows encrypted between sidecars Security modes: - STRICT: mTLS required for all traffic (reject non-mTLS) - PERMISSIVE: Accept both mTLS and plaintext (migration mode)

When to Use (and Not Use) a Service Mesh

Use a service mesh when: You have 20+ services, need mTLS for compliance, want zero-trust networking, need traffic splitting for canary deployments, or have polyglot services needing consistent networking behavior.
Do not use a service mesh when: You have fewer than 10 services, all in one language (use a library instead), or your team cannot handle the operational complexity. A service mesh adds latency (sidecar hop), CPU overhead (~2-5%), and significant debugging complexity.

Trade-offs

AdvantageDisadvantage
Zero-code networking (language-agnostic)Adds ~2-5ms latency per hop (sidecar proxy)
Automatic mTLSCPU/memory overhead per sidecar (~50MB RAM, ~50m CPU each)
Rich traffic management (canary, mirroring)Complex to debug (app error vs Envoy error?)
Consistent observability across all servicesOperational complexity (control plane management)
No per-language networking librariesAnother moving part that can fail
Interview tip: "For a system with 50+ microservices, I would use a service mesh (Istio/Envoy) for automatic mTLS and traffic management. This gives me zero-trust networking and canary deployments without modifying application code. For a smaller system, I would use language-specific libraries for retries and circuit breaking instead."

Common Interview Questions

  1. What is a service mesh and what problem does it solve?
  2. How does the sidecar proxy pattern work?
  3. What is mTLS and why does a service mesh make it easier?
  4. Compare service mesh with API gateway — what is the difference?
  5. What are the trade-offs of using a service mesh?
  6. When would you NOT use a service mesh?
  7. How does Istio handle traffic management for canary deployments?
  8. What is the latency overhead of a service mesh?

What to Skip

Avoid diving into: Istio configuration YAML (VirtualService, DestinationRule), Envoy filter chain configuration, or specific mesh implementations (Linkerd, Consul Connect). Focus on the architectural concept: sidecar proxy, control plane vs data plane, and the capabilities it provides.
