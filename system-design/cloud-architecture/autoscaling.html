Autoscaling

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand the three levels of Kubernetes autoscaling (HPA, VPA, Cluster Autoscaler), their triggers, how they interact, and how to design an effective autoscaling strategy for production systems.

Definition

Autoscaling is the ability to automatically adjust the number of compute resources based on current load. In Kubernetes, autoscaling operates at three levels: pod replicas (HPA), pod resource requests (VPA), and cluster nodes (Cluster Autoscaler).

Three Levels of Autoscaling

Level 3: Cluster Autoscaler ┌──────────────────────────────────────────────────────────┐ │ Adds/removes worker NODES based on pending pods │ │ Trigger: pods cannot be scheduled (insufficient resources) │ │ Action: add new EC2 instance / GCE instance │ └──────────────────────────────────────────────────────────┘ │ Level 2: VPA (Vertical Pod Autoscaler) ┌──────────────────────────────────────────────────────────┐ │ Adjusts CPU/memory REQUESTS for pods │ │ Trigger: resource utilization too high or too low │ │ Action: restart pod with updated resource limits │ └──────────────────────────────────────────────────────────┘ │ Level 1: HPA (Horizontal Pod Autoscaler) ┌──────────────────────────────────────────────────────────┐ │ Scales NUMBER OF PODS based on metrics │ │ Trigger: CPU % > 70%, memory > 80%, custom metrics │ │ Action: increase/decrease replicas in deployment │ └──────────────────────────────────────────────────────────┘

HPA (Horizontal Pod Autoscaler)

HPA adjusts the number of pod replicas based on observed metrics. It is the most commonly used autoscaler and should be your default scaling mechanism.
Metric SourceMetrics AvailableUse Case
Resource metricsCPU utilization, memory utilization (per pod)General-purpose scaling
Custom metricsQueue depth, request rate, concurrent connections (via Prometheus adapter)Application-specific triggers
External metricsCloudWatch, Datadog, external systemsScale based on external events
HPA Example: Scale based on CPU Target: CPU average < 70% Current: 3 replicas, CPU at 85% Desired replicas = ceil(current_replicas × current_cpu / target_cpu) Desired replicas = ceil(3 × 85 / 70) = ceil(3.64) = 4 HPA creates 1 more replica → 4 pods → CPU drops below 70% Traffic drops: CPU at 30% → HPA scales down to 2 replicas (min: 2) Config: ┌───────────────────────────────────┐ │ minReplicas: 2 │ │ maxReplicas: 50 │ │ targetCPUUtilizationPercentage: 70│ │ scaleDown: │ │ stabilizationWindowSeconds: 300 │ ← wait 5 min before scaling down └───────────────────────────────────┘
Stabilization window: Always configure a scale-down stabilization window (5-10 minutes). Without it, brief traffic dips will cause rapid scale-down, followed by scale-up when traffic returns — oscillation that wastes resources and increases latency.

VPA (Vertical Pod Autoscaler)

VPA adjusts the CPU and memory resource requests for pods. When a pod needs more resources, VPA kills the existing pod and recreates it with higher requests.
HPA and VPA conflict: Do not run HPA (CPU/memory-based) and VPA on the same workload. HPA scales horizontally (more pods), VPA scales vertically (bigger pods). They fight each other. Use HPA for scaling; use VPA only for right-sizing recommendations or on non-HPA workloads.

Cluster Autoscaler

Cluster Autoscaler adds new nodes when pods cannot be scheduled due to insufficient resources, and removes nodes when utilization is low.
Cluster Autoscaler Flow: 1. Pod pending (no node has enough resources) └──▶ Cluster Autoscaler detects pending pods └──▶ Adds new node (EC2 instance, GCE VM) └──▶ Scheduler places pending pod on new node 2. Node underutilized (< 50% capacity) └──▶ Cluster Autoscaler checks if pods can fit on other nodes └──▶ Evicts pods (respecting PDB limits) └──▶ Removes node Requirements: - Pods must have resource requests set (CPU, memory) - Pod Disruption Budgets to prevent critical pod eviction - Node groups with min/max bounds

Interview tip: When designing a system, specify your autoscaling strategy. "I would use HPA with a CPU target of 70% and min 3/max 50 replicas. For event-driven workloads, I would use KEDA to scale based on Kafka queue depth. The cluster autoscaler handles node scaling with a 5-minute scale-down stabilization window."

Common Interview Questions

  2. How does HPA decide to scale up or down?
  3. Why should you not run HPA and VPA together on the same workload?
  4. What is the cluster autoscaler and when does it trigger?
  5. What are good scaling triggers beyond CPU utilization?
  6. How do you prevent scaling oscillation?
  7. What is a Pod Disruption Budget and why does it matter for autoscaling?

What to Skip

Avoid diving into: HPA YAML configuration, Kubernetes metrics server setup, or cloud-specific autoscaler implementations (EKS, GKE node pools). Focus on the three levels, their triggers, and the interaction between them.
