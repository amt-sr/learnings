Deployments

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand Kubernetes Deployments as the mechanism for declarative pod management, including rolling updates, rollbacks, replica management, and health probes (readiness and liveness).

Definition

Deployment is a Kubernetes resource that manages a ReplicaSet and provides declarative updates for pods. It defines the desired state (number of replicas, container image version, resource limits) and the controller continuously reconciles actual state to match.

Deployments are the primary way to deploy stateless applications in Kubernetes. They add rolling update and rollback capabilities on top of ReplicaSets.

Rolling Update Flow

Initial state: replicas: 3, image: v1 ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Pod v1 │ │ Pod v1 │ │ Pod v1 │ └──────────┘ └──────────┘ └──────────┘ Rolling update to v2 (maxSurge: 1, maxUnavailable: 0): Step 1: Create 1 new pod (v2) ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Pod v1 │ │ Pod v1 │ │ Pod v1 │ │ Pod v2 │ ← new └──────────┘ └──────────┘ └──────────┘ └──────────┘ Step 2: Delete 1 old pod (v1), create 1 new pod (v2) ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Pod v1 │ │ Pod v1 │ │ Pod v2 │ │ Pod v2 │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ Step 3: Delete 1 old pod, create 1 new pod ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Pod v1 │ │ Pod v2 │ │ Pod v2 │ └──────────┘ └──────────┘ └──────────┘ Step 4: Final old pod replaced ┌──────────┐ ┌──────────┐ ┌──────────┐ │ Pod v2 │ │ Pod v2 │ │ Pod v2 │ └──────────┘ └──────────┘ └──────────┘ Key: At every step, enough pods are serving traffic. maxUnavailable: 0 = never reduce serving capacity below desired maxSurge: 1 = allow 1 extra pod during rollout

Rollback

Every deployment change creates a new ReplicaSet revision. Rollback reverts to a previous ReplicaSet.

# Rollback is instant because old ReplicaSet still exists
# New pods are created from the old image, old pods are terminated

Revision 3 (current):  image: v3  ← broken
Revision 2 (rollback): image: v2  ← target
Revision 1:            image: v1

# kubectl rollout undo deployment/my-app --to-revision=2

Readiness vs Liveness Probes

ProbeQuestionFailure ActionUse Case
Readiness"Is the pod ready to serve traffic?"Remove from Service endpoint (no traffic routed)Wait for app to warm up, load config, connect to DB
Liveness"Is the pod still running correctly?"Restart the container (kill and recreate)Detect deadlocks, infinite loops, unresponsive processes
Startup"Has the app started?" (disables other probes during startup)Failure counts toward livenessSlow-starting apps (Java, ML models) where liveness would fire prematurely
Common mistake: Using a liveness probe that is too aggressive. If the probe checks an endpoint that depends on an external service (DB, API) and that service is slow, the liveness probe will kill the pod repeatedly (CrashLoopBackOff), making the situation worse. Liveness probes should check local health only. 
readinessProbe:
  httpGet:
    path: /health/ready
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 10

livenessProbe:
  httpGet:
    path: /health/live
    port: 8080
  initialDelaySeconds: 15
  periodSeconds: 20
  failureThreshold: 3    # 3 failures before restart

Replica Management

The Deployment controller ensures the desired number of replicas are always running. If a node fails, pods on that node are recreated on healthy nodes.

Deployment: replicas: 3 Desired: 3 pods running and healthy Actual: (continuously monitored by controller) ┌──────────────────────────────────────────────────┐ │ Deployment Controller │ │ │ │ 1. Compare desired replicas vs actual running │ │ 2. If gap: create/delete pods via ReplicaSet │ │ 3. Wait for readiness probe to pass │ │ 4. Pod is "available" when ready │ └──────────────────────────────────────────────────┘ Node 1 fails: Before: [Pod A (node1), Pod B (node2), Pod C (node3)] After: [Pod B (node2), Pod C (node3)] ← Pod A gone Controller detects: 2/3 running → creates Pod D on node2 or node3

Deployment Strategies Comparison

Rolling UpdateNoneLow (maxSurge pods)LowDefault for most apps
RecreateYes (all down, then all up)NoneHighMajor DB schema changes that need all instances updated at once
Blue/GreenNone2x (full duplicate)Low (instant rollback)Critical services needing instant rollback
CanaryNoneLowMediumProgressive rollout with validation
Interview tip: When asked about deployment strategies, mention canary deployments: "I would roll out to 1% of traffic, monitor error rates and latency for 10 minutes, then progressively increase to 25%, 50%, 100%. If metrics degrade, automatically rollback." This demonstrates operational maturity.

What to Skip

Avoid diving into: Deployment YAML syntax, strategy-specific YAML fields, pod disruption budgets (PDB), or kubectl rollout commands. Focus on the concepts: rolling update flow, probe semantics, and strategy trade-offs.
