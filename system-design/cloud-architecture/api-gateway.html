API Gateway

Difficulty: вӯҗвӯҗвӯҗ Interview Frequency: вҳ…вҳ…вҳ…вҳ…вҳ…

Goal

Understand the API gateway pattern, its responsibilities (routing, authentication, rate limiting, SSL termination, request aggregation), trade-offs, and when to use it versus when it adds unnecessary complexity.

Definition

API Gateway is a single entry point for all client requests into a microservices architecture. It acts as a reverse proxy that routes requests to the appropriate service, handles cross-cutting concerns (auth, rate limiting, SSL, logging), and can aggregate responses from multiple services.

The API gateway decouples clients from services. Clients talk to one endpoint; the gateway routes to many services. This simplifies client code and centralizes infrastructure concerns.

Core Responsibilities

ConcernDescriptionImplementation
Request RoutingMap incoming URLs to downstream servicesPath-based, header-based, query-param routing
AuthenticationVerify JWT tokens, API keys, OAuth tokensReject unauthorized requests before they reach services
Rate LimitingLimit requests per client/IP/tierToken bucket, sliding window, per-service limits
SSL TerminationHandle TLS at the gateway; internal traffic is HTTPReduces cert management on every service
Load BalancingDistribute requests across service instancesRound-robin, least-connections, or delegate to service mesh
Request/Response TransformNormalize protocols (REST вҶ” gRPC), strip/add headersProtocol translation for polyglot services
CachingCache frequently requested responsesRedis-backed cache for GET requests
Circuit BreakingFail fast when downstream services are unhealthyPrevent cascading failures

Rate Limiting Strategies

# Token Bucket: burst-friendly, allows short bursts up to capacity
# Leaky Bucket: steady rate, smooth output
# Sliding Window: precise, memory-intensive
# Fixed Window: simple, allows burst at window boundary

# API Gateway rate limiting config:
routes:
  - path: /api/search/*
    rate_limit:
      requests_per_second: 100
      burst: 50
      key: api_key        # per-API-key limit
  - path: /api/checkout/*
    rate_limit:
      requests_per_second: 10
      burst: 5
      key: user_id         # per-user limit
  default:
    rate_limit:
      requests_per_second: 50
      key: ip_address       # per-IP limit

API Gateway Trade-offs

AdvantageDisadvantage
Single entry point for all clientsSingle point of failure (requires HA deployment)
Centralizes cross-cutting concernsBecomes a bottleneck if overloaded
Decouples clients from service topologyAdds latency (extra hop)
Simplifies client-side codeCoupling: every routing change requires gateway update
Enables request aggregation (BFF)Can become a "smart pipe" with too much business logic
Deployment: The API gateway MUST be deployed in high-availability mode (minimum 2 instances behind a load balancer or using a managed service like AWS API Gateway, Kong, or Envoy). A single gateway instance is a critical reliability risk.

Gateway vs Ingress vs Service Mesh

ComponentLayerTrafficScope
API GatewayL7 (application)North-south (client to service)Business-facing: routing, auth, rate limiting
IngressL7 (Kubernetes)North-south (external to cluster)Kubernetes-specific entry point
Service MeshL7 (sidecar)East-west (service to service)Inter-service: mTLS, retries, circuit breaking
Interview tip: When designing a system, include an API gateway early and justify it. "I would use an API gateway to centralize auth and rate limiting, preventing unauthenticated traffic from reaching backend services." Then mention that it must be HA-deployed to avoid a SPOF.

What to Skip

Avoid diving into: Specific gateway product configurations (Kong plugins, AWS API Gateway YAML), detailed TLS handshake protocol, or WAF (Web Application Firewall) rules. Focus on the architectural role and trade-offs.
