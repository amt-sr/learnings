Docker Containers

Difficulty: ⭐⭐ Interview Frequency: ★★★★

Goal

Understand Docker containers as a packaging and isolation mechanism, how images are layered, and how containers compare to virtual machines. This knowledge underpins all container orchestration topics (Kubernetes, pods, deployments).

Definition

Docker container is a lightweight, standalone, executable package that includes everything needed to run a piece of software: code, runtime, system tools, libraries, and settings. Containers are isolated from each other and the host system using OS-level virtualization (namespaces and cgroups), sharing the host kernel.

Containers solve the "it works on my machine" problem by packaging the application with its entire dependency tree into an immutable artifact. The same container image runs identically on a developer's laptop, a CI server, and production.

Image Layers

Docker Image (read-only layers): ┌─────────────────────────────────┐ │ Layer 4: Application code │ ← Changes most often │ COPY app/ /app/ │ ├─────────────────────────────────┤ │ Layer 3: Dependencies │ │ RUN pip install -r requirements │ ├─────────────────────────────────┤ │ Layer 2: Runtime │ │ FROM python:3.11-slim │ ├─────────────────────────────────┤ │ Layer 1: Base OS │ │ debian:bookworm-slim │ └─────────────────────────────────┘ Container = Image layers (read-only) + Writable container layer (thin) ┌─────────────────────────────────┐ │ Container Layer (writable) │ ← Runtime state, logs, temp files ├─────────────────────────────────┤ │ Image Layers (read-only) │ ← Shared across all containers │ (same as above) │ from this image └─────────────────────────────────┘
Caching insight: Docker caches each layer. If Layer 3 does not change, Docker reuses the cached layer. This is why you should order Dockerfile instructions from least to most frequently changing. Put OS and dependency installs before application code to maximize cache hits.

Optimized Dockerfile

# BAD: rebuilds on every code change
FROM node:20
COPY . /app          # Invalidates ALL layers below
RUN npm install

# GOOD: dependencies cached separately
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./     # Only changes when package.json changes
RUN npm ci --production   # Cached if package.json unchanged
COPY . ./                 # Only this layer rebuilt on code change
EXPOSE 3000
CMD ["node", "server.js"]

Dockerfile Best Practices

  • Use multi-stage builds: Build stage with full toolchain, runtime stage with only binaries. Reduces image from 1.2GB to 50MB.
  • Use minimal base images: alpine, slim, or distroless instead of full OS images.
  • Run as non-root: Add a non-root user to reduce attack surface.
  • Pin versions: Never use latest tag in production. Pin to specific digest or version.
  • Minimize layers: Combine RUN commands with && to reduce layer count.

Interview tip: When asked about containerization in a system design interview, focus on the "why": "Containers provide immutable, reproducible deployments that eliminate environment drift. Combined with orchestration (Kubernetes), they enable declarative infrastructure and self-healing systems."

Common Interview Questions

  1. What is a Docker container and how is it different from a VM?
  2. How do Docker image layers work and why do they matter?
  3. How would you optimize a Dockerfile for fast builds?
  4. What is container isolation and how does it work?
  5. Why are containers important for microservices?
  6. What is a multi-stage build and when would you use it?
  7. How would you keep Docker images small and secure?

What to Skip

Avoid diving into: Docker CLI command syntax, Docker Compose YAML, or container networking details (bridge, host, overlay networks). Focus on the conceptual model: images, layers, isolation, and why containers matter for deployment architecture.
