Monolith Architecture

Difficulty: ⭐⭐ Interview Frequency: ★★★

Goal

Understand the monolith architecture, its advantages and disadvantages, when it is the correct architectural choice, and how to evaluate it against microservices in an interview trade-off discussion.

Definition

Monolith is an architectural style where all application functionality (UI, business logic, data access) is deployed as a single deployable unit. The codebase may be well-structured internally with clear module boundaries, but it runs as one process on one server (or replicated behind a load balancer).

A monolith is not necessarily a "big ball of mud." A well-structured monolith has clean internal boundaries, modular code, and a layered architecture. The key characteristic is single-unit deployment.

When Monolith Is the Right Choice

Start with a monolith until the pain of not having microservices exceeds the pain of having them. This is the consensus advice from experienced architects.
  • Early-stage startup: Team is small (1-10 engineers), product is finding product-market fit, speed matters more than perfect architecture.
  • Moderate traffic: System handles under 10K RPS. A monolith behind a load balancer can handle this easily.
  • Tight domain coupling: When modules share a lot of data and business logic is deeply intertwined (e.g., checkout that touches inventory, payments, shipping).
  • Small team: If you cannot assign 2+ engineers per service, you do not have enough people to own microservices.
  • Regulatory/compliance: Single database simplifies audit trails and data governance.
  • Performance-sensitive: In-process function calls are orders of magnitude faster than network calls.

Modular Monolith

The best of both worlds: maintain clean module boundaries within a single deployable unit. This makes future extraction to microservices possible.

┌──────────────────────────────────────────────┐ │ Modular Monolith │ │ │ │ ┌────────────┐ ┌────────────┐ │ │ │ Auth Bounded│ │ Order Bounded│ │ │ │ Context │ │ Context │ │ │ │ (own models, │ │ (own models, │ │ │ │ own API) │ │ own API) │ │ │ └──────┬──────┘ └──────┬──────┘ │ │ │ │ │ │ ┌──────┴────────────────┴──────┐ │ │ │ Public API / Event Bus │ │ │ │ (modules communicate here) │ │ │ └───────────────────────────────┘ │ └──────────────────────────────────────────────┘ Key: Modules have STRICT internal boundaries. Communication via well-defined APIs, not direct DB access. Each module owns its data and exposes only interfaces.
Interview tip: When asked "microservices or monolith?", do not give a blanket answer. Frame it as a trade-off: "I would start with a well-structured modular monolith and extract services when I have clear evidence of a specific module needing independent scaling, deployment, or technology." This demonstrates judgment over dogma.

Common Interview Questions

  1. When would you choose a monolith over microservices?
  2. What are the advantages of a monolith architecture?
  3. How do you know when it is time to break a monolith into microservices?
  4. What is a modular monolith and how is it different from a traditional monolith?
  5. How do you scale a monolith?
  6. What is the "monolith first" philosophy and do you agree with it?
  7. How do you prevent a monolith from becoming a big ball of mud?

What to Skip

Avoid diving into: Specific monolithic frameworks (Spring Boot, Django, Rails), code-level module patterns, or build tool configuration. Focus on the architectural trade-offs and decision criteria.
