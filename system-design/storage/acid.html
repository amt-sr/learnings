3.4.1 ACID

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

  • Define all four ACID properties precisely, with concrete examples of what each guarantees
  • Explain how the four properties interact and the trade-offs between them
  • Identify real-world scenarios where ACID guarantees are violated and the consequences
  • Understand how ACID applies (or breaks down) in distributed systems
  • Articulate when BASE is a viable alternative and the trade-offs involved

Definition

ACID is an acronym for four properties that guarantee reliable processing of database transactions: Atomicity, Consistency, Isolation, and Durability. These properties ensure that database operations are processed reliably -- either all changes within a transaction are applied, or none are. The concept was formalized in the early 1980s by Theo Harder and Andreas Reuter, building on Jim Gray's foundational work on transaction processing.

Critical distinction: ACID is not a monolith. Each property is independently enforced through different mechanisms. A database can provide strong atomicity and durability (via WAL) while offering weak isolation (read committed). Understanding each property independently is essential for system design interviews.

Atomicity -- All or Nothing

A transaction is an indivisible unit of work. Either all its operations complete successfully, or none of them take effect. There is no intermediate state visible to other transactions.

ViolationExampleConsequence
Partial writeDebit account A succeeds, credit account B failsMoney disappears from the system
No rollbackMulti-step operation fails halfway, earlier steps persistCorrupt application state

Implementation: Write-Ahead Log (WAL) with undo information. On commit, the transaction is marked as committed in the WAL. On rollback (or crash before commit), all changes are reversed using the undo log.

Consistency -- Valid State Transitions

A transaction moves the database from one valid state to another. Validity is defined by the application's rules: constraints, triggers, cascading rules, and any declared invariants. This is the only ACID property that is partially the application's responsibility -- the database enforces declared constraints, but the correctness of business logic is up to the developer.

Constraint TypeExampleEnforced By
Primary keyuser_id must be unique and non-nullDatabase
Foreign keyorder.user_id must reference user.idDatabase
Check constraintaccount_balance >= 0Database
Business invariantTotal debits = total creditsApplication
Common confusion: ACID Consistency is NOT the same as "distributed consistency" (reads returning the latest write, strong consistency in CAP theorem). ACID consistency is about maintaining database invariants within a single transaction. Distributed consistency is about read-write ordering across replicas. These are fundamentally different concepts that are frequently conflated in interviews.

Isolation -- Concurrent Correctness

Concurrent transactions execute as if they were executed serially, one after the other. The degree of isolation determines how much interference between concurrent transactions is allowed. This is the most nuanced ACID property because it exists on a spectrum with a well-known taxonomy of isolation levels.

Isolation LevelDirty ReadNon-Repeatable ReadPhantom
Read UncommittedPossiblePossiblePossible
Read CommittedPreventedPossiblePossible
Repeatable ReadPreventedPreventedPossible
SerializablePreventedPreventedPrevented

Each higher isolation level incurs a performance cost. Most production workloads use Read Committed (PostgreSQL default, MySQL default for non-InnoDB) or Repeatable Read (MySQL InnoDB default) as a practical compromise.

Durability -- Surviving Failures

Once a transaction commits, its effects are permanent even in the face of crashes, power failures, and hardware errors. The data will survive regardless of what happens after the commit acknowledgment is sent.

Implementation: WAL records are forced to stable storage (fsync) before the commit is acknowledged. Additionally, replication to other nodes provides durability against total node failure.

ACID Violations in Practice

Real-world failure patterns:
  • Network partition during commit: Client receives timeout but transaction actually committed on the server. Client retries, causing double-execution. Mitigation: idempotency keys.
  • Replication lag after commit: Transaction is acknowledged as committed (durable on leader) but a read from a follower returns stale data. The data is "durable" but not "consistent" from the client's perspective.
  • Schema evolution breaks consistency: Adding a NOT NULL column without a default value violates consistency for existing rows. Many ORMs generate invalid intermediate states during migrations.

ACID in Distributed Systems

ACID was designed for single-node databases. In distributed systems, providing all four properties simultaneously becomes significantly harder:

  • Atomicity: Achieved via Two-Phase Commit (2PC) or consensus protocols (Raft, Paxos), but with higher latency and availability cost (coordinator failure blocks progress).
  • Consistency: Distributed constraints (foreign keys across shards) are extremely expensive to enforce. Most sharded systems push this to the application layer.
  • Isolation: Distributed locks and timestamp ordering (Spanner TrueTime) can provide serializable isolation, but at significant latency cost. Most systems settle for weaker guarantees.
  • Durability: Achieved via quorum-based replication, but the definition of "committed" becomes nuanced (committed on leader? replicated to quorum? all replicas?).
Interview tip: Google Spanner is the gold standard example of "ACID in distributed systems." It provides external consistency (linearizable) using TrueTime (synchronized atomic clocks + GPS). The trade-off is geographic latency bound by clock uncertainty (~10ms across datacenters).

BASE: The Alternative

BASE (Basically Available, Soft state, Eventual consistency) is an explicit design philosophy that trades ACID guarantees for availability and partition tolerance. It is not a formal specification but a design principle.

PropertyACIDBASE
AvailabilityMay block during failuresAlways available (accepts writes)
ConsistencyStrong (immediate)Eventual (converges over time)
LatencyHigher (synchronization cost)Lower (no cross-node coordination)
Use caseFinancial transactions, inventorySocial feeds, caching, analytics
ExamplesPostgreSQL, MySQL, SpannerCassandra, DynamoDB (eventual), Redis
Design principle: In interviews, avoid framing ACID vs. BASE as a binary choice. Modern systems blend both: DynamoDB offers both eventual and strong consistency modes. CockroachDB provides serializable isolation. Spanner offers external consistency with high availability. The right choice depends on the specific consistency requirements of each operation.

Common Interview Questions

  1. What are the four ACID properties? Give me a real example for each.
  2. Why is consistency considered the application's responsibility? What does the database enforce?
  3. Explain the difference between ACID consistency and distributed consistency (CAP theorem).
  4. What isolation level does your production system use? Why? What anomalies can still occur?
  5. Is it possible to have ACID in a distributed database? What are the trade-offs?
  6. What happens to ACID guarantees during a network partition?
  7. Compare ACID and BASE. When would you choose one over the other?
  8. How does a system like Spanner provide ACID across multiple datacenters?
  9. Can you give me an example where relaxing isolation improves throughput by 10x?
  10. Design a payment system. Which ACID properties are non-negotiable? Which can you relax?

What to Skip

In an interview setting, skip:
  • The full history of transaction processing research (Jim Gray's 1970s papers, etc.)
  • Debating whether ACID is "outdated" -- it remains the gold standard for transactional correctness
  • Reciting isolation level definitions without understanding their performance implications
  • Deep dives into specific database's constraint syntax -- focus on concepts and trade-offs
  • Over-emphasizing BASE when the question is about ACID -- understand both but answer the question asked
