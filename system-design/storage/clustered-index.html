Clustered Index

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand that a clustered index merges the index and the table into a single structure where leaf nodes are the data pages, the implications for range scan performance, and the design trade-off between clustered and non-clustered storage.

Definition

A clustered index is an index where the leaf nodes contain the actual row data rather than pointers to the data. The index key determines the physical ordering of rows on disk. There can be only one clustered index per table, because the data can only be sorted in one physical order. InnoDB (MySQL) uses the primary key as the clustered index by default.

Structure

Sequential range scans are fast (sequential I/O).

Key Characteristics

Property Detail
Leaf nodes = data pages No separate heap structure. The B+ tree leaves hold the full row.
One per table Data can only be physically ordered one way. Only one clustered index allowed.
Physical ordering Rows on disk are sorted by the clustered index key. Adjacent keys are co-located.
Range scan performance Excellent -- sequential I/O for range queries on the clustered key.
Point lookup Excellent -- single B+ tree traversal, no bookmark hop needed.
Secondary index bookmark Secondary indexes store the clustered key as the bookmark, not a physical pointer.

Secondary Indexes in a Clustered Table

A critical consequence of the clustered architecture: secondary indexes cannot point directly to the physical row location (because the row may move due to page splits). Instead, they store the clustered key value as the bookmark.

Secondary index returns clustered key = order_id 2. Use order_id to look up full row in clustered index 3. Total: 2 B+ tree traversals
Wide clustered keys increase secondary index size: If the clustered key is a VARCHAR(255) email column, every secondary index leaf must store that 255-byte value as a bookmark. This inflates secondary index size and degrades cache efficiency. Keep clustered keys narrow (e.g., 4-byte INT or 8-byte BIGINT).

Range Scan Benefits

Clustered indexes excel at range queries on the indexed key because adjacent rows are physically stored together. This turns random I/O into sequential I/O.

-- Time-series query: excellent with clustered index on timestamp-based key
SELECT * FROM sensor_readings
WHERE sensor_id = 42
  AND reading_time BETWEEN '2025-01-01' AND '2025-01-31'
ORDER BY reading_time;

-- If clustered on (sensor_id, reading_time):
-- All matching rows are on contiguous (or nearly contiguous) data pages.
-- Sequential read: orders of magnitude faster than random page fetches.

Drawbacks

  • Page fragmentation: Random inserts (e.g., UUID primary keys) cause frequent page splits, leaving pages partially empty and wasting space.
  • Slower non-key inserts: When inserting in non-key order, the engine must find the correct position in the B+ tree, potentially splitting pages along the path.
  • Secondary index cost: Every secondary index lookup requires a bookmark lookup into the clustered index (double traversal).
  • Rebuild cost: Rebuilding (reorganizing) a clustered index rewrites the entire table.

Clustered Index by Engine

Engine Default Behavior Configurable?
InnoDB (MySQL) Always clustered on primary key (or hidden 6-byte row ID) Not configurable -- always clustered
SQL Server Heap by default; clustered index optional Yes -- can choose clustering key or use heap
PostgreSQL Heap table with separate indexes No native clustered index (CLUSTER rewrites but doesn't maintain order)
Oracle Heap table with separate indexes (Index-Organized Table as alternative) IOT available for specific use cases

  • What bookmark do secondary indexes use in a clustered table?
  • Why is a large clustered key bad for secondary index performance?
  • What is page fragmentation and how does it relate to clustered indexes?
  • How does PostgreSQL differ from InnoDB in terms of clustered storage?

What to Skip

Depth to avoid in interviews:
  • InnoDB page compression internals within clustered index leaf pages.
  • Online DDL internals for rebuilding a clustered index (ALGORITHM=INPLACE details).
  • Adaptive hash index interaction with clustered index lookups.
  • Oracle IOT overflow segment internals.
