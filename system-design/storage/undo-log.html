Undo Log

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What the undo log records — old values (before-images) for rollback
  • Rollback: reversing uncommitted changes using undo records
  • Crash recovery order: REDO first, then UNDO
  • MVCC: how undo logs provide consistent snapshots for readers
  • Undo log segments and space management

Definition

The undo log (also called the rollback log or before-image log) stores the previous values of data before it was modified. It serves two critical purposes: enabling transaction rollback (undoing uncommitted changes) and providing consistent snapshots for MVCC (allowing readers to see the state of data as it existed before a concurrent writer modified it).

Undo vs Redo

UPDATE accounts SET balance = 900 WHERE id = 42; (Previous balance was 1000) Redo Log (forward — for recovery): ┌────────────────────────────────────────────┐ │ LSN 1005 │ Page 42, offset 120: │ │ Write 0x384 (900) at offset 120 │ │ This replays the change after a crash │ └────────────────────────────────────────────┘ Undo Log (backward — for rollback): ┌────────────────────────────────────────────┐ │ TXN 501 │ Page 42, offset 120: │ │ Previous value was 0x3E8 (1000) │ │ This reverses the change if rolled back │ └────────────────────────────────────────────┘ Together: Redo replays forward, Undo reverses backward. Both are needed for full recovery guarantees.

Rollback with Undo

When a transaction issues ROLLBACK (or is terminated), the database follows the undo chain to reverse each modification:

Transaction T1: UPDATE accounts SET balance = 900 WHERE id = 42; UPDATE accounts SET balance = 800 WHERE id = 43; ROLLBACK; ← user cancels Undo Chain for T1: ┌──────────┐ ┌──────────┐ │ Undo #2 │ ←── │ Undo #1 │ ← T1's undo pointer │ Page 43 │ │ Page 42 │ │ bal: 1000│ │ bal: 2000│ │ (prev) │ │ (prev) │ └──────────┘ └──────────┘ Rollback process: 1. Follow undo chain from newest to oldest 2. Undo #2: Write balance=1000 back to page 43 3. Undo #1: Write balance=2000 back to page 42 4. T1 is fully rolled back — all changes reversed Important: The rollback itself generates redo records (undo operations are also logged for crash safety)

Crash Recovery: REDO First, Then UNDO

The order matters. During crash recovery, the database must:

  1. REDO phase (forward): Replay all redo records from the last checkpoint to reconstruct the state at crash time. This replays both committed and uncommitted changes.
  2. UNDO phase (backward): Identify uncommitted transactions (no COMMIT record in the WAL) and use the undo log to reverse their changes.
Why REDO before UNDO: After a crash, the buffer pool is empty — all dirty pages are lost. Before we can undo uncommitted transactions, we must first restore the pages to their crash-time state. The redo log contains all changes (committed and uncommitted). Once redo reconstructs the pages, the undo log reverses the uncommitted changes.
Crash Recovery Order: Step 1: REDO (replay all changes from checkpoint LSN) ┌─────────────────────────────────────────────┐ │ Replay LSN 1001 → page 30 (committed TXN) │ │ Replay LSN 1002 → page 42 (uncommitted TXN) │ │ Replay LSN 1003 → page 30 (committed TXN) │ │ Replay LSN 1004 → COMMIT TXN 500 │ │ Replay LSN 1005 → page 42 (uncommitted TXN) │ │ │ │ Pages now reflect state at crash time │ └─────────────────────────────────────────────┘ Step 2: UNDO (reverse uncommitted transactions) ┌─────────────────────────────────────────────┐ │ TXN 501 has no COMMIT record │ │ → Follow TXN 501's undo chain │ │ → Undo LSN 1005: restore page 42 old value │ │ → Undo LSN 1002: restore page 42 old value │ │ │ │ Uncommitted changes are reversed │ └─────────────────────────────────────────────┘ Result: Database is consistent with only committed data.

Undo and MVCC

Multi-Version Concurrency Control (MVCC) allows readers to see a consistent snapshot without blocking writers. The undo log provides the old row versions needed for snapshots:

MVCC Snapshot Example: Timeline: T1: BEGIN; T2: BEGIN; T1: UPDATE accounts SET balance = 900 WHERE id = 42; COMMIT; T2: SELECT balance FROM accounts WHERE id = 42; → What does T2 see? If T2 uses READ COMMITTED: → Sees T1's committed change → balance = 900 If T2 uses REPEATABLE READ: → T2's snapshot was taken before T1 committed → Needs the OLD value → balance = 1000 → Obtained from undo log! ┌─────────────────────────────────────────┐ │ Current page: balance = 900 (T1 commit)│ │ │ │ Undo chain for row 42: │ │ Current: balance = 900 (T1) │ │ Undo #1: balance = 1000 (before T1) │ └─────────────────────────────────────────┘ T2's REPEATABLE READ query follows the undo chain to find the version visible at T2's snapshot timestamp. This provides isolation without locks!

Undo in Different Databases

DatabaseUndo StorageMVCC Implementation
MySQL InnoDBSeparate undo tablespace (undo_001, undo_002)Undo log provides old versions; data page has current value + pointer to undo chain
OracleSeparate undo tablespace (undo segments)Same approach — undo provides old row versions
PostgreSQLNo separate undo log — old row versions stored in-place on data pagesMulti-version tuples directly in the table; VACUUM reclaims old versions
SQL ServerSeparate version store in tempdbOld row versions in tempdb (since SQL Server 2005)
PostgreSQL's approach is different: Instead of maintaining a separate undo log, PostgreSQL keeps old row versions directly in the main table's data pages (as new tuples with xmin/xmax headers). Each tuple header records which transaction created it (xmin) and which deleted it (xmax). To find an older version, follow the tuple chain within the data pages. The VACUUM process cleans up old tuples no longer visible to any snapshot.

Undo Log Segments

Undo logs are organized into segments for space management:

  • Undo segment: A logical container for undo records from multiple transactions. MySQL InnoDB uses up to 128 rollback segments.
  • Undo page: Physical pages within the undo tablespace that store undo records. Filled sequentially.
  • Purge thread: A background process that asynchronously removes undo records for committed transactions that are no longer needed by any active snapshot.
  • Undo retention: Undo data must be kept as long as any active transaction (or long-running query) might need it for its MVCC snapshot.
Common pitfall: Long-running queries (e.g., a 30-minute analytics query) prevent the purge thread from recycling undo space. This causes the undo tablespace to grow and can eventually cause ORA-01555 "snapshot too old" errors (Oracle) or undo tablespace exhaustion (MySQL). Solution: avoid long-running queries alongside write-heavy workloads.

Common Interview Questions

  • "What is the undo log and what are its two purposes?"
    The undo log stores before-images of modified data. Purpose 1: enables transaction rollback by providing the old values to reverse uncommitted changes. Purpose 2: enables MVCC by providing old row versions that concurrent readers can see without waiting for writers to release locks. See: Redo Log
  • "During crash recovery, why is REDO done before UNDO?"
    After a crash, the buffer pool is empty — all dirty pages in memory are lost. Before the database can undo uncommitted transactions, it must first reconstruct the pages to their crash-time state. The redo log replays all changes (committed and uncommitted) to rebuild the pages. Then the undo log reverses only the uncommitted changes. Without redo first, undo would have no pages to operate on.
  • "How does the undo log enable MVCC?"
    When a reader starts a transaction, it gets a snapshot timestamp. If it reads a row that was modified by a later transaction, the database follows the undo chain to find the version of the row as it existed at the reader's snapshot timestamp. This provides repeatable reads without any locking — readers never block writers and writers never block readers.
  • "What happens if the undo log runs out of space?"
    If undo data is needed by an active transaction but has been overwritten or recycled, the transaction cannot maintain its snapshot. In Oracle, this causes ORA-01555 "snapshot too old." In MySQL, the purge thread is delayed and the undo tablespace grows. Solution: size undo appropriately, avoid long-running queries alongside short transactions, and tune undo retention parameters.
