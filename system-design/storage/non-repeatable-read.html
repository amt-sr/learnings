Non Repeatable Read

Difficulty ⭐⭐⭐☆☆ Interview Frequency ★★★★☆

Goal

Understand what a non-repeatable read is, how it differs from a dirty read, why it is more common than dirty reads, and which isolation levels prevent it.

Definition

A non-repeatable read occurs when a transaction reads the same row twice within the same transaction and gets different values. This happens because another transaction modified and committed a change to that row between the two reads.

Key distinction from dirty read: In a dirty read, you see uncommitted data that may be rolled back. In a non-repeatable read, you see committed data that changed between reads. The data is real -- the issue is inconsistency within your transaction's view.

Anatomy of a Non-Repeatable Read

Non-Repeatable Read Timeline ───────────────────────────────────────────────────────────────── Transaction A (Checkout) Transaction B (Price Update) ──────────────────────────── ──────────────────────────── BEGIN; SELECT price FROM products WHERE id = 42; -- returns $100.00 UPDATE products SET price = 120.00 WHERE id = 42; COMMIT; SELECT price FROM products WHERE id = 42; -- returns $120.00 (CHANGED!) -- Transaction A calculated total based on $100 -- But by the time it charged, price was $120 -- Customer charged $100, product worth $120 COMMIT;

Non-Repeatable Read vs. Dirty Read vs. Phantom Read

Aspect Dirty Read Non-Repeatable Read Phantom Read
Data state Uncommitted (may be rolled back) Committed (permanent) Committed (new/deleted rows)
What changes Value of an existing row Value of an existing row Set of rows matching a query
Caused by Reading in-progress modifications UPDATE committed between reads INSERT or DELETE committed between reads
Query type Point query (single row) Point query (same row) Range query (WHERE clause)
Prevented by Read Committed Repeatable Read Serializable (or gap locks)

Why Non-Repeatable Reads Are Common

Non-repeatable reads are far more common than dirty reads because Read Committed is the default isolation level in both PostgreSQL and Oracle. At this level, each statement gets a fresh snapshot of committed data, so any committed change between two statements in your transaction is visible.

Read Committed: fresh snapshot per statement Transaction A (Read Committed): ────────────────────────────────── SELECT price FROM products WHERE id = 42; -- Snapshot at t=1: price = $100 -- Transaction B commits price = $120 at t=2 SELECT price FROM products WHERE id = 42; -- NEW snapshot at t=3: price = $120 -- Non-repeatable read occurs!
Repeatable Read: frozen snapshot for entire transaction Transaction A (Repeatable Read): ────────────────────────────────── SELECT price FROM products WHERE id = 42; -- Snapshot at t=1: price = $100 (frozen) -- Transaction B commits price = $120 at t=2 -- A's snapshot is FROZEN at t=1 SELECT price FROM products WHERE id = 42; -- SAME snapshot: price = $100 -- Non-repeatable read prevented!

Which Isolation Levels Prevent Non-Repeatable Reads?

Isolation Level Non-Repeatable Read Mechanism
Read Uncommitted Possible No protection
Read Committed Possible Fresh snapshot per statement
Repeatable Read Prevented Frozen snapshot for entire transaction
Snapshot Isolation Prevented Frozen snapshot at transaction start
Serializable Prevented Full serialization

Mitigation Strategies Without Changing Isolation

-- 1. Explicit locking (pessimistic)
SELECT price FROM products WHERE id = 42 FOR UPDATE;
-- Locks the row; other transactions must wait

-- 2. Optimistic concurrency control
SELECT price, version FROM products WHERE id = 42;
-- Application stores version
UPDATE products SET price = @new_price, version = version + 1
  WHERE id = 42 AND version = @stored_version;
-- If version mismatch → concurrent modification detected

-- 3. Accept and handle at application level
-- Read once, make a decision, and commit atomically
BEGIN;
  SELECT price FROM products WHERE id = 42;  -- read once
  INSERT INTO orders (product_id, price) VALUES (42, 100);
  UPDATE products SET stock = stock - 1 WHERE id = 42;
COMMIT;  -- all or nothing
Practical advice: For most applications, non-repeatable reads at Read Committed are benign. The key is to read a value and act on it within a single statement or a short transaction. The problem arises only when you read, perform application logic, then read again and make a different decision based on the changed value.

Common Interview Questions

  1. What is a non-repeatable read and how is it different from a dirty read?
  2. How does PostgreSQL's Read Committed handle non-repeatable reads?
  3. At which isolation levels are non-repeatable reads prevented?
  4. Is a non-repeatable read always a bug? When is it acceptable?
  5. How do you prevent non-repeatable reads without escalating to Repeatable Read?
  6. How does MySQL's InnoDB prevent non-repeatable reads at its default isolation level?

What to Skip

Depth to avoid in interviews:
  • The difference between row-level non-repeatable reads and block-level non-repeatable reads
  • Oracle-specific consistent read implementation details
  • Deep internals of PostgreSQL's snapshot creation per statement
