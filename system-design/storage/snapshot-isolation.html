Snapshot Isolation

Difficulty ⭐⭐⭐⭐☆ Interview Frequency ★★★★☆

Goal

Understand snapshot isolation as a practical concurrency model: how it provides consistent reads without locking, where it fits in the isolation spectrum, and what anomalies it still allows.

Definition

Snapshot Isolation (SI) is a concurrency control model where every transaction reads from a consistent snapshot of the database taken at the transaction's start time. The transaction sees only the effects of transactions that committed before it began. Concurrent writes from other transactions are invisible until they commit, at which point the commit protocol detects conflicts and aborts one of the conflicting transactions.

Key guarantee: A transaction under snapshot isolation always sees a consistent state as of a single point in time. No dirty reads, no non-repeatable reads, and no phantom reads -- from that transaction's perspective, the database is frozen.

How MVCC Implements Snapshot Isolation

Snapshot isolation is implemented via Multiversion Concurrency Control (MVCC). Each row is stored with multiple versions tagged by transaction IDs. When a transaction begins, it captures a snapshot of all in-progress and committed transaction IDs. Every subsequent read consults the version chain and returns only the version that was committed before the snapshot time.

What Snapshot Isolation Prevents

Anomaly Prevented by SI? Why
Dirty Read Yes Only committed versions are visible in the snapshot
Non-Repeatable Read Yes Snapshot is frozen at transaction start; re-reading yields same data
Phantom Read Yes Snapshot consistency applies to entire predicate; new rows are invisible
Write Skew No Two transactions read overlapping data, write to disjoint rows -- no conflict detected

Write Skew: The Anomaly SI Does Not Prevent

Write skew is the primary anomaly that snapshot isolation does not prevent. It occurs when two transactions read the same data, make decisions based on that data, and then write to different rows. Because the writes target disjoint rows, the first-committer-wins check does not flag a conflict.

Write Skew Example: Hospital On-Call Doctors ───────────────────────────────────────────────────────────────── Business rule: At least one doctor must be on call at all times. doctors table: ┌────┬────────┬─────────┐ │ id │ name │ on_call │ ├────┼────────┼─────────┤ │ 1 │ Alice │ true │ │ 2 │ Bob │ true │ └────┴────────┴─────────┘ Time ──────────────────────────────────────────────────────────► Tx A (Alice): Tx B (Bob): BEGIN BEGIN SELECT * FROM doctors SELECT * FROM doctors → sees both on_call=true → sees both on_call=true "Bob is on call, safe to go off" "Alice is on call, safe to go off" UPDATE doctors SET on_call=false UPDATE doctors SET on_call=false WHERE id=1 WHERE id=2 COMMIT (succeeds) COMMIT (succeeds) Result: BOTH doctors off call -- invariant violated! SI did NOT detect a conflict because Tx A wrote row 1 and Tx B wrote row 2 (disjoint writes).
Interview tip: Write skew is a favorite follow-up question. Know how to prevent it: serializable isolation, SELECT FOR UPDATE on the rows read, or an explicit constraint (e.g., a CHECK constraint with a partial index).

Conflict Detection: First-Committer-Wins

Snapshot isolation uses an optimistic approach to conflict detection. When a transaction attempts to commit, the database checks whether any row it modified was also modified (and committed) by another transaction that started after our snapshot. If so, our transaction is aborted with a serialization failure.

-- PostgreSQL conflict abort
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 42;   -- reads 500
-- Meanwhile: another tx commits UPDATE accounts SET balance=600 WHERE id=42
UPDATE accounts SET balance = 500 - 100 WHERE id = 42;
-- ERROR: could not serialize access due to concurrent update
COMMIT;
-- Application must retry the entire transaction

Snapshot Isolation vs. Repeatable Read

Aspect Snapshot Isolation Repeatable Read (SQL Standard)
Dirty Read Prevented Prevented
Non-Repeatable Read Prevented Prevented
Phantom Read Prevented Allowed (SQL standard)
Write Skew Allowed Allowed
Implementation MVCC (version chains) Locking (shared locks held to end of tx) or MVCC
Blocking behavior Readers never block writers Readers may block writers (in locking impl)
Note: PostgreSQL's REPEATABLE READ is actually implemented as snapshot isolation internally. In PostgreSQL, Repeatable Read and Snapshot Isolation are effectively the same thing, both preventing phantom reads. MySQL's InnoDB REPEATABLE READ is also implemented with MVCC but uses next-key locking to prevent phantoms -- so MySQL's implementation is strictly stronger than standard SI for range queries.

PostgreSQL and Snapshot Isolation

PostgreSQL uses snapshot isolation as its default concurrency control for REPEATABLE READ and READ COMMITTED (the latter takes a new snapshot for each statement, not each transaction). The internal implementation:

  • Each transaction gets an xid (transaction ID) at BEGIN
  • Each row version stores xmin (creating transaction) and xmax (deleting transaction)
  • A snapshot records the xmin horizon -- the set of in-progress and committed transactions at snapshot time
  • Visibility test: a row version is visible if its xmin is committed and <= the snapshot's xmin horizon, and its xmax is either not set or refers to an uncommitted/in-snapshot transaction
-- PostgreSQL: inspecting transaction IDs
SELECT xmin, xmax, cmin, cmax, * FROM accounts WHERE id = 42;

-- Force REPEATABLE READ (snapshot isolation in Postgres)
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE id = 42;  -- consistent snapshot
-- other concurrent commits are invisible here
COMMIT;
Why SI is popular: Readers never block writers and writers never block readers. This provides excellent throughput for read-heavy workloads (the norm in most applications) while still giving strong consistency guarantees for single-entity operations.

Common Interview Questions

  1. What is snapshot isolation and how does it differ from serializable isolation?
  2. Explain write skew and why snapshot isolation does not prevent it. How would you fix it?
  3. How does MVCC implement snapshot isolation? Describe the version visibility check.
  4. What happens when two transactions under snapshot isolation try to update the same row?
  5. Why does PostgreSQL use snapshot isolation as the foundation for its isolation levels?
  6. Compare snapshot isolation to two-phase locking for read-heavy workloads. What are the tradeoffs?
  7. How does a retry loop work when a snapshot isolation transaction is aborted?
  8. Is PostgreSQL's REPEATABLE READ the same as the SQL standard's REPEATABLE READ?

What to Skip

Depth to avoid in interviews:
  • Internal details of PostgreSQL's CLOG (commit log) and subtrans/slru structures
  • The exact predicate locking implementation in SSI (Serializable Snapshot Isolation internals)
  • Comparison with Oracle's SCN-based snapshot implementation specifics
  • Free space map and vacuum interactions unless directly relevant to long-running snapshots
