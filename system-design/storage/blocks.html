Blocks

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What a block is — the smallest addressable unit of storage I/O
  • Block devices vs character devices and why the distinction matters
  • How the OS manages blocks vs how the CPU/MMU manages pages
  • Filesystem blocks vs disk sectors vs database pages
  • Block alignment and its impact on I/O performance

Definition

A block (also called a filesystem block or disk block) is the smallest unit of storage that the operating system reads from or writes to a block device. On modern Linux systems, this is typically 4 KB. The OS filesystem organizes all files into blocks — even a 1-byte file occupies a full 4 KB block on disk.

Block Devices vs Character Devices

Block DeviceCharacter Device
Access patternFixed-size blocks, random accessByte stream, sequential access
CachingCached in OS page cacheNo caching (typically)
ExamplesHDD, SSD, loop devices, LVM volumesTerminal (/dev/tty), serial port, /dev/null
I/O modelread()/write() with seek (lseek)read()/write() as stream
Relevance to databasesPrimary storage interfaceMinimal (used for logging to terminals)
Why this matters: Databases store their data files on block devices. The block device interface determines the I/O semantics: the OS always reads and writes in whole-block multiples, and it caches these blocks in the page cache. When a database uses O_DIRECT (direct I/O), it bypasses this cache and issues block I/O directly to the device.

OS Blocks vs CPU Pages

These are related but distinct concepts that are often confused:

AspectOS Block (I/O)CPU Page (Memory)
Managed byOS kernel / filesystemCPU MMU / kernel VM subsystem
Size4 KB (filesystem)4 KB (typical, can be 2 MB / 1 GB huge pages)
PurposeUnit of disk I/OUnit of virtual memory mapping
Cached inPage cache (in-kernel)Physical RAM frames
ErrorsEIO (disk I/O error)Page fault (virtual → physical)
RelationshipWhen an OS block is read from disk, it is stored in RAM as a CPU page in the page cache
Interview distinction: When someone says "page fault," they are talking about CPU virtual memory (a process accessing unmapped memory). When someone says "block I/O," they are talking about reading/writing storage. The page cache is where these two worlds meet — disk blocks are cached as CPU pages.

Block Alignment

Modern disks use 4 KB physical sectors internally but expose a 512-byte logical interface for backward compatibility. If data is misaligned — starting at an offset that does not match the 4 KB physical boundary — every write requires a read-modify-write cycle:

Correct Alignment: Physical: [| Sector 0 (4 KB) | Sector 1 (4 KB) |] Logical: [| Block A (4 KB) | Block B (4 KB) |] offset 0 offset 4096 ✓ Aligned ✓ Aligned Misaligned (partition starts at 512-byte offset): Physical: [| Sector 0 | Sector 1 | Sector 2 |] Logical: [| Block A | Block B |] offset 512 offset 4608 ✗ Straddles sectors 0-1 ✗ Straddles sectors 1-2 Misaligned write = READ sector + MODIFY bytes + WRITE sector back = 2x I/O amplification for every write
Practical note: Modern Linux partitioning tools (parted, fdisk with GPT) automatically align partitions to 1 MB boundaries. This eliminates misalignment for typical filesystem block sizes. However, databases using raw devices or custom storage managers must handle alignment explicitly.

Block I/O and the Buffer Pool

The relationship between OS blocks and the database buffer pool has performance implications:

  • Without O_DIRECT: Database writes go to the buffer pool (dirty), then to the OS page cache, then to disk. Data is cached twice — wasteful but simple. PostgreSQL default.
  • With O_DIRECT: Database writes go to the buffer pool (dirty), then directly to disk, bypassing the OS page cache. Single copy in memory. MySQL InnoDB default.
  • Block size matching: If the database page size is a multiple of the OS block size (e.g., 16 KB DB page / 4 KB OS block), I/O is efficient. Non-aligned sizes cause cross-block reads.

Common Interview Questions

  • "What is the difference between a database page, an OS block, and a CPU page?"
    An OS block (4 KB) is the smallest unit of filesystem I/O. A CPU page (4 KB, or 2 MB/1 GB for huge pages) is the unit of virtual memory mapping. A database page (4–16 KB) is the database engine's I/O unit. When the OS reads a block from disk, it stores it as a CPU page in the page cache. The database manages its own pages in the buffer pool, which is a separate memory area from the page cache.
  • "Why can't you read half a block from disk?"
    The disk hardware and filesystem are designed around fixed-size blocks. The disk controller reads entire sectors (512 B or 4 KB) from the physical media. The OS filesystem layer manages space allocation in whole blocks. There is no mechanism to address or transfer partial blocks — it would require firmware-level changes to every storage device and filesystem.
  • "What is a block device and how is it different from a character device?"
    A block device (/dev/sda, /dev/nvme0n1) supports random access to fixed-size blocks and is cached by the OS. A character device (/dev/tty, /dev/null) provides a byte-stream interface with no caching. Databases use block devices exclusively for data storage.
