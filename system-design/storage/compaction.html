Compaction

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand:

  • Why compaction is necessary in LSM trees (read amplification, space amplification)
  • Size-tiered vs leveled compaction strategies
  • The write amplification tradeoff of compaction
  • How compaction handles tombstones and过期数据

Definition

Compaction is the background process of merging multiple SSTables into fewer, larger SSTables. As the LSM tree accumulates SSTables from memtable flushes, the number of files grows unchecked. Without compaction, reads would need to check hundreds or thousands of files. Compaction merges overlapping SSTables, resolves duplicates (keeping the newest version), removes tombstones, and frees disk space.

Why compaction is needed — the problem: Without compaction, 10,000 writes produce: Level 0: [sst_001] [sst_002] [sst_003] ... [sst_500] (500 files!) GET("user:42") → check memtable → check 500 SSTables on disk → Read amplification: 500x (catastrophic) With compaction: Level 0: [sst_498] [sst_499] [sst_500] (3 files) Level 1: [sst_101] [sst_102] (non-overlapping) Level 2: [sst_050] Level 3: [sst_001] GET("user:42") → check memtable → 3 L0 files → 1 L1 file → 1 L2 file → Read amplification: ~5x (manageable)

Compaction Strategies

1. Size-Tiered Compaction (STCS)

Cassandra default. Merge SSTables of similar size. Level 0 after many flushes: [sst_001: 100MB] [sst_002: 100MB] [sst_003: 100MB] [sst_004: 100MB] When 4 SSTables reach similar size → merge them into one: [sst_005: 400MB] Next 4 SSTables of ~400MB → merge into 1.6 GB: [sst_006: 1.6GB] Tier pattern: each tier is 4x (or configurable) larger than the previous. Pros: Good for write-heavy workloads, fast merges (few large files) Cons: High read amplification (may check all tiers), high space amplification

2. Leveled Compaction

RocksDB default. Each level has a size limit, non-overlapping key ranges. L0 (up to 4 MB): [sst_a] [sst_b] (may overlap) L1 (up to 40 MB): [sst_c] [sst_d] [sst_e] (non-overlapping) L2 (up to 400 MB): [sst_f] [sst_g] ... (non-overlapping) L3 (up to 4 GB): [sst_h] ... L4 (up to 40 GB): ... L5 (up to 400 GB): ... L6 (up to 4 TB): ... Compaction merges ONE file from Ln into overlapping range in Ln+1. Example: L1 [sst_c: keys a-m] merges with L2 [sst_f: keys c-k] Result: new L2 files replace old ones, non-overlapping. Pros: Low read amplification (at most 1 file per level) Cons: Higher write amplification (more data rewritten per level)

3. FIFO Compaction

Simplest strategy. Drop oldest SSTables when size limit reached. [sst_001: oldest] [sst_002] [sst_003] [sst_004: newest] │ ▼ When total size > limit → delete sst_001 Use case: Time-series data with natural TTL Data naturally expires (no need to merge, just drop old files) Pros: Zero write amplification, minimal CPU Cons: No deduplication, no tombstone removal

The Amplification Tradeoff

The fundamental LSM tree tradeoff: There is no free lunch. You must pick two out of low write amplification, low read amplification, and low space amplification. Compaction strategy determines which you optimize for. 
  Amplification explained:

  Write amplification:
    Each key-value pair is rewritten how many times total?
    Leveled:  WAL(1) + flush(1) + L0→L1(1) + L1→L2(1) + ... = ~10-30x
    This means writing 1 GB of data actually writes 10-30 GB to disk.

  Read amplification:
    How many disk seeks per read in worst case?
    Leveled: check memtable + L0 + L1 + L2 + ... = ~num_levels + L0_files
    With 7 levels and 4 L0 files: ~11 disk seeks worst case

  Space amplification:
    How much extra disk space is used vs actual data size?
    Before compaction runs: old SSTables + new SSTables coexist temporarily
    Leveled steady-state: ~2x (compaction in progress adds ~1x overhead)

Compaction and Tombstones

Compaction is when tombstones actually delete data. When merging SSTables, if a tombstone is the newest version of a key, and the tombstone has been present longer than a grace period (garbage collection window), the key and tombstone are both discarded during compaction.

SSTable 1: user:42 = "Alice" SSTable 2: user:42 = TOMBSTONE ← delete happened later During compaction: Merge SSTable 1 + SSTable 2: user:42 = TOMBSTONE (tombstone is newer, wins) If tombstone is older than gc_grace_seconds (e.g., 10 days): user:42 is completely removed (not written to new SSTable) → disk space reclaimed, tombstone cleaned up

Compaction Tuning in Practice

  RocksDB tuning (common interview scenario):

  write_buffer_size:       64 MB     (memtable size)
  max_write_buffer_number: 3         (max pending memtables)
  level0_file_num_compaction_trigger: 4  (L0 files before compaction)
  max_bytes_for_level_base:   256 MB  (L1 size limit)
  max_bytes_for_level_multiplier: 10   (each level 10x the previous)

  Cassandra tuning:
  compaction_throughput_mb_per_sec:  16  (throttle I/O)
  tombstone_threshold:               0.2  (20% tombstones trigger compaction)
  min_sstable_size:                   50 MB (STCS minimum)
Compaction storms: If writes are too fast, compaction can't keep up. SSTables accumulate in L0, reads get very slow, and disk I/O spikes. This is a production crisis. Mitigation: throttle write rate, add more nodes, increase compaction throughput, or switch to FIFO compaction for TTL data.

Common Interview Questions

  • "What is compaction in an LSM tree?"
    Compaction is a background process that merges multiple SSTables into fewer, larger SSTables. It resolves duplicates (keeping the newest), removes tombstones, reduces the total number of files, and frees disk space. Without compaction, the number of SSTables grows without bound and read performance degrades catastrophically.
  • "Compare size-tiered vs leveled compaction."
    Size-tiered: merges SSTables of similar size into larger ones. Better write throughput, higher read amplification. Used by Cassandra. Leveled: each level has a size limit, non-overlapping ranges within a level. Better read performance, higher write amplification. Used by RocksDB and HBase.
  • "What is write amplification and why does compaction cause it?"
    Write amplification is how many times a piece of data is rewritten. In an LSM tree, each key is written to the WAL, flushed to L0, then rewritten at each compaction level. With 7 levels, a single write may cause ~10-30 total writes. Compaction is the primary source of write amplification — it rewrites existing data to merge and reorganize SSTables.
  • "How would you handle compaction falling behind?"
    SSTables pile up in L0, causing read amplification to explode. Solutions: throttle incoming writes (backpressure), increase compaction thread count, use a larger buffer so writes happen faster, or add nodes to distribute load. In Cassandra, increase compaction_throughput_mb_per_sec.
