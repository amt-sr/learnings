Tombstones

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • Why immutable SSTables cannot support in-place deletes
  • How tombstones (delete markers) solve this problem
  • The lifecycle of a tombstone: creation, propagation, and eventual garbage collection
  • The "zombie data" problem and its impact on read performance

Definition

A tombstone is a special delete marker written to an SSTable to indicate that a key has been deleted. Since SSTables in an LSM tree are immutable (they cannot be modified in-place), actual deletion is impossible. Instead, a tombstone is appended as a new record. During reads, the tombstone masks any older versions of the key. During compaction, tombstones eventually allow the physical removal of both the tombstone and the deleted data from disk.

Tombstone lifecycle in an LSM tree TIME →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→ 1. INSERT: PUT("user:42", "Alice") WAL → Memtable → SSTable_001: [user:42 = "Alice"] 2. DELETE: DELETE("user:42") WAL → Memtable → SSTable_002: [user:42 = TOMBSTONE] 3. READ: GET("user:42") Check SSTable_002 (newest): user:42 = TOMBSTONE → return "not found" (SSTable_001 still has "Alice", but SSTable_002's tombstone takes precedence) 4. COMPACTION: Merge SSTable_001 + SSTable_002 user:42: newest is TOMBSTONE → If within gc_grace_seconds: keep tombstone → If past gc_grace_seconds: discard both data and tombstone → SSTable_003: [empty for user:42] → disk space freed

Why Tombstones Exist

The fundamental reason is immutability of SSTables:

ApproachWhy It Doesn't Work
In-place deleteSSTables are immutable — no in-place modifications allowed
Remove the SSTableOne SSTable contains thousands of keys; deleting the whole file loses all other data
Append a tombstoneWrite a delete marker as a new entry. Newest tombstone wins over old data. Cleaned up later by compaction.

How Tombstones Work During Reads

LSM tree reads check from newest to oldest data source. The first version found wins:

  GET("user:42")

  Check memtable:              not found
  Check SSTable L0 (newest):   user:42 = TOMBSTONE → STOP, return "not found"
                                (older SSTables with user:42 = "Alice" are never read)

  Why stop at tombstone? Because the tombstone is newer than any older version.
  The key is definitively deleted — no need to check older levels.

Types of Tombstones

TypeScopeUsed By
Row tombstoneDeletes a single key (one row)RocksDB, Cassandra, HBase
Range tombstoneDeletes a range of keys [start, end)RocksDB, MyRocks
Column tombstoneDeletes a specific column within a rowCassandra, HBase
Partition tombstoneDeletes an entire partition (wide row)Cassandra

The Zombie Data Problem

Deleted data is not physically removed until compaction runs. Between the delete and the next compaction, the old data still exists on disk. This has several consequences:
The zombie data problem: SSTable_001 (100 MB): [a=1, b=2, c=3, d=4, e=5, ... thousands more] SSTable_002 (50 KB): [c=TOMBSTONE] Problem 1: Space waste "c=3" still occupies space in SSTable_001 (zombie data) Won't be freed until compaction merges 001 and 002 Problem 2: Read performance Bloom filter for SSTable_001 returns YES for key "c" → read SSTable_001 to find c=3 → then read SSTable_002 to find c=TOMBSTONE → extra disk I/O for zombie data Problem 3: Accumulation Heavy delete workloads → many tombstones → many zombie SSTables → compaction can't keep up → disk fills up → reads slow down significantly

Garbage Collection (GC) Grace Period

Tombstones are not removed immediately during compaction. They must persist for a configurable grace period to ensure that replicas that were temporarily unavailable have a chance to receive the tombstone.

  Cassandra example:

  gc_grace_seconds: 864000  (default: 10 days)

  Timeline:
    Day 0:   DELETE key → tombstone written
    Day 1:   Compaction runs → tombstone is only 1 day old → KEPT
    Day 5:   Node goes down, misses the tombstone
    Day 6:   Node comes back up, repair runs → receives tombstone from other replicas
    Day 11:  Compaction runs → tombstone is 11 days old (past gc_grace) → REMOVED
             Both zombie data and tombstone are finally purged

  If gc_grace_seconds were too short (e.g., 60 seconds):
    Day 0:   DELETE key → tombstone
    Day 0:   Compaction → removes tombstone immediately
    Day 1:   Repair → node never learns about the delete → zombie data reappears!

  Rule: gc_grace_seconds must exceed your maximum repair interval.

Tombstone-Related Failures

FailureCauseSymptom
Tombstone heap exhaustionToo many tombstones during read repairOutOfMemoryError, node crash
Read timeoutToo many tombstones in a partition → single partition read touches many SSTablesRead latency spikes, timeouts
Disk fullCompaction can't keep up with delete workloadDisk usage grows despite deletes
Zombie resurrectiongc_grace too short + failed repairDeleted data reappears
Interview tip: If asked about a Cassandra cluster where disk is filling up despite deletes, the answer is tombstone accumulation. The fix: run compaction, tune gc_grace_seconds, and ensure regular repairs.

Best Practices

  • Never delete data you don't need to. Use TTL instead of explicit deletes when possible.
  • Run regular repairs. Ensures all replicas receive tombstones before gc_grace expires.
  • Monitor tombstone counts. Cassandra: nodetool tablestats shows tombstone ratio per table.
  • Tune gc_grace_seconds. Must be larger than maximum expected repair interval.
  • Avoid batch deletes of large partitions. A single partition with 100K tombstones will cause read timeouts.

Common Interview Questions

  • "How does deletion work in an LSM tree?"
    Since SSTables are immutable, you can't delete in-place. Instead, a tombstone (delete marker) is appended. During reads, the tombstone is the newest version and masks any older data. During compaction, old data and the tombstone are physically removed from merged SSTables.
  • "What is the zombie data problem?"
    Deleted data physically remains on disk in older SSTables until compaction merges those SSTables with the tombstone. This wastes space and degrades read performance (bloom filters still say YES for zombie keys). In extreme cases, heavy deletes without compaction can fill the disk.
  • "Why does Cassandra have a gc_grace_seconds setting?"
    In distributed systems, replicas may be temporarily down during a delete. The tombstone must persist long enough for all replicas to receive it via repair. If tombstones are removed before all replicas get them, deleted data can "resurrect" on the lagging replica. gc_grace_seconds (default 10 days) sets how long tombstones survive before compaction removes them.
  • "How would you troubleshoot a cluster where reads are slow after a mass delete?"
    Mass deletes create many tombstones. Reads must traverse SSTables containing both zombie data and tombstones, causing high read latency. Solution: trigger major compaction, ensure repairs run, check tombstone counts with nodetool tablestats, and consider using TTL instead of explicit deletes for future workloads.
