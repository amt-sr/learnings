3.4.5 Durability

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

  • Define durability as the guarantee that committed data survives crashes, power failures, and hardware errors
  • Explain the mechanisms that provide durability: WAL, fsync, and replication
  • Understand battery-backed write cache and its role in enterprise storage
  • Analyze the durability vs. performance trade-off, including the cost of fsync()
  • Explain group commit and how it amortizes the cost of durable writes

Definition

Durability guarantees that once a transaction has been committed, its effects will persist even in the face of system failures -- including process crashes, power outages, kernel panics, and hardware failures. The commit acknowledgment sent to the client is a promise: this data is permanent and will be recoverable after any failure.

The durability contract: When the database says "committed," the client must be able to trust that the data will survive. Breaking this contract -- acknowledging a commit that is later lost -- is one of the most severe failures a database can have. It is the reason why fsync costs are accepted as the price of trustworthiness.

Durability Mechanisms

DURABILITY LAYERS (defense in depth) ──────────────────────────────────── Layer 1: Write-Ahead Log (WAL) ┌─────────────────────────────────────────────────────────────┐ │ All changes logged BEFORE being applied to data pages │ │ Log records contain before/after images for recovery │ │ Committed records fsync'd to stable storage on commit │ └─────────────────────────────────────────────────────────────┘ │ ▼ Layer 2: Force to Stable Storage (fsync) ┌─────────────────────────────────────────────────────────────┐ │ fsync() ensures data is written past OS page cache │ │ to disk platters / SSD persistent media │ │ Bypasses volatile memory: survives process AND power failure│ └─────────────────────────────────────────────────────────────┘ │ ▼ Layer 3: Replication ┌─────────────────────────────────────────────────────────────┐ │ Synchronous replication: commit waits until replica ACKs │ │ Survives total node failure (disk corruption, machine death) │ │ Quorum writes: data survives failure of N/2 - 1 nodes │ └─────────────────────────────────────────────────────────────┘ │ ▼ Layer 4: Backups / Snapshots ┌─────────────────────────────────────────────────────────────┐ │ Point-in-time recovery for disaster scenarios │ │ Survives data corruption, accidental deletion, region failure│ └─────────────────────────────────────────────────────────────┘

Write-Ahead Log (WAL)

The WAL is the primary durability mechanism. Before any data page is modified, the corresponding log record is written to the WAL. On commit, the WAL is flushed to stable storage (fsync) before the client receives the commit acknowledgment. If the system crashes after commit, the WAL contains all the information needed to redo any changes that were not yet flushed to data pages.

Key property: The WAL is append-only, sequential I/O. Sequential writes are 5-10x faster than random writes on both HDDs and SSDs. By writing sequentially to the WAL and lazily flushing data pages, the database converts random writes into sequential writes, dramatically improving commit performance.

fsync() -- Forcing Data to Stable Storage

The operating system maintains a page cache in RAM for performance. When a process writes data, it goes to the page cache first. The OS periodically flushes dirty pages to disk in the background. This means that a simple write() call does NOT guarantee durability -- a power failure can lose data still in the page cache.

fsync() is the system call that forces dirty pages for a specific file descriptor to be written to the physical storage media. It blocks until the device reports the write is complete. This is the actual durability guarantee at the hardware level.

The fsync() cost: On HDDs, fsync involves a physical disk seek (~10ms). This is why single-threaded database commit throughput is capped at ~100 TPS on spinning disks (1 commit per fsync). On SSDs, fsync is faster (~0.1-2ms), allowing ~500-1000 TPS per thread. This is the fundamental reason databases use group commit.

Battery-Backed Write Cache (BBWC)

Enterprise storage arrays and RAID controllers often include a Battery-Backed Write Cache (BBWC) or NVRAM cache. This is a small amount of battery-backed or flash-backed RAM on the storage controller that absorbs writes and acknowledges them immediately.

WITHOUT BBWC: WITH BBWC: ─────────────── ────────────── DB → OS page cache → disk DB → OS page cache → BBWC → ACK │ (battery-backed) │ │ ▼ ▼ (flushed to disk lazily) fsync() waits for disk Fast ACK commit latency: ~1-15ms commit latency: ~0.1-0.5ms

The BBWC acts as a trusted intermediary: the database calls fsync, the controller ACKs immediately (data is in battery-backed RAM), and the controller flushes to disk asynchronously. If power fails, the battery keeps the cache alive long enough to flush to disk.

At FAANG scale, this can mean millions of lost records. Always understand what you are trading away.

Group Commit

Group commit amortizes the cost of fsync across multiple transactions. Instead of calling fsync after every single commit, the database collects multiple transactions' log records and flushes them all with a single fsync call.

WITHOUT GROUP COMMIT: WITH GROUP COMMIT: ───────────────────── ────────────────── TXN A commit ──► fsync ──► ACK TXN A commit ──┐ (1-15ms wait) TXN B commit ──┼──► fsync ──► ACK all TXN C commit ──┘ (1-15ms wait, TXN B commit ──► fsync ──► ACK shared by 3) (1-15ms wait) TXN C commit ──► fsync ──► ACK Throughput: ~3x higher (1-15ms wait) Per-TXN latency: similar Total: 3 fsyncs, 3 commits Throughput: limited by fsync

Group commit is effective because the WAL is append-only. The database can batch multiple commit records into a single sequential write + fsync. This is the primary reason why modern databases can achieve tens of thousands of TPS even with full durability on SSDs.

  • PostgreSQL uses group commit via its commit_delay and commit_siblings parameters, though the default is no delay (immediate fsync).
  • MySQL InnoDB groups commits automatically when using innodb_flush_log_at_trx_commit = 1.
  • RocksDB uses group commit extensively with its write batch API.
Interview tip: When asked how to improve write throughput while maintaining durability, group commit should be your first answer. It is the most practical optimization because it provides identical durability guarantees with significantly higher throughput. The trade-off is slightly higher per-transaction latency (waiting for the group to fill), which is usually negligible.

Replication as Durability

Replication provides a second layer of durability beyond a single node's WAL + fsync:

Replication ModeDurability GuaranteeLatency Cost
Synchronous (quorum write)Data on N/2+1 nodes before ACK2x+ network RTT
Semi-synchronousData on leader + at least 1 replica ACK1x network RTT
AsynchronousData on leader only at commit timeNone

In cloud-native systems, replication is often the primary durability mechanism. The rationale: a single node's disk can fail (hardware corruption), but data replicated across 3 nodes in different availability zones is effectively durable. Systems like CockroachDB, Spanner, and Cassandra rely heavily on replication for durability.

Nuance: Synchronous replication does not eliminate the need for WAL + fsync on each individual node. Each node must still guarantee local durability to protect against its own crashes. Replication protects against node-level failures; WAL protects against crash-level failures. Both are needed for comprehensive durability.

Durability Failure Modes

Defense in depth: At FAANG scale, durability is never a single mechanism. It is a layered strategy: WAL on the primary, synchronous replication to replicas, nightly full backups, continuous WAL archiving, and delayed replicas for human-error recovery. Each layer protects against a different failure mode.

Common Interview Questions

  1. What does durability mean? How does the database ensure a committed transaction survives a crash?
  2. Explain the role of fsync() in durability. Why not just use write()?
  3. What is group commit? How does it improve throughput while maintaining durability?
  4. Describe the durability vs. performance trade-off. When would you relax durability?
  5. How does a battery-backed write cache improve commit performance?
  6. What happens to durability when you run MySQL with innodb_flush_log_at_trx_commit = 2?
  7. How does replication contribute to durability? Is it a replacement for WAL?
  8. What is the "fsync lies" problem? How do you detect it?
  9. Design a system that must achieve 100,000 durable writes per second on a single node. What trade-offs do you make?
  10. What durability guarantees does your current production database provide? How would you verify?

What to Skip

In an interview setting, skip:
  • Detailed OS page cache internals (dirty page writeback threads, pdflush, etc.)
  • Specific hardware RAID controller configurations and vendor-specific BBWC implementations
  • Filesystem journaling internals (JFS, XFS internals) -- focus on the database-level mechanisms
  • Detailed benchmarks of fsync latency on specific hardware -- discuss the concept and orders of magnitude
  • Backup and restore procedures -- this is operations, not system design
