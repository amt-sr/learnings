SSTable

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What a Sorted String Table (SSTable) is and how it's structured on disk
  • The internal format: data blocks, index blocks, bloom filter, footer
  • How SSTables enable efficient reads via sparse indexing
  • How SSTables accumulate in LSM tree levels

Definition

A SSTable (Sorted String Table) is an immutable, sorted file on disk that stores an ordered sequence of key-value pairs. It is the persistent storage format produced when a memtable is flushed from memory in an LSM tree. Once written, an SSTable is never modified — it is only read, merged with other SSTables during compaction, or deleted.

SSTable Read Path

  GET(key = "user:42")

  1. Read footer (last bytes of file) → get bloom filter offset
  2. Read bloom filter → "Does this SSTable contain user:42?"
     - NO  → skip this file entirely (saved a disk read!)
     - YES → continue
  3. Read index block → binary search for the block containing "user:42"
     - Find: Block 7 contains keys from "user:40" to "user:49"
  4. Read Block 7 from disk → scan for "user:42"
  5. Return value (or "not found")

  Disk reads per SSTable check:
  - Bloom filter miss: 1 read (footer + bloom, often cached in memory)
  - Bloom filter hit:  2-3 reads (footer + bloom + index + data block)

Key Properties

PropertyDetail
ImmutableOnce written, never modified. New versions of keys go into newer SSTables. Old SSTables are cleaned up by compaction.
SortedKeys are in sorted order within each SSTable. Enables binary search, range scans, and efficient merge operations.
Append-only creationCreated by a single sequential write when a memtable flushes. Very efficient disk I/O.
Self-containedEach SSTable has its own index and bloom filter. No external metadata needed for reads.
CompressibleSorted data compresses extremely well (high redundancy in keys with shared prefixes). Compression ratios of 5-10x are common.

SSTable Levels in LSM Tree

SSTables accumulate in tiers (levels). Each level is 10x larger. Level 0: SSTables from memtable flushes (not merge-compacted) May have overlapping key ranges Size: ~100 MB, up to 4-10 files Files: [sst_001] [sst_002] [sst_003] Level 1: Result of compacting Level 0 files Non-overlapping key ranges within this level Size: ~1 GB (10x L0) Files: [sst_010] [sst_011] [sst_012] Level 2: Result of compacting Level 1 files Non-overlapping key ranges Size: ~10 GB Files: [sst_020] [sst_021] ... [sst_030] Level 3: Size: ~100 GB Level 4: Size: ~1 TB Level 5: Size: ~10 TB (typical max for RocksDB) Level 6: Size: ~100 TB (Cassandra, HBase) Key property: Within L1+, SSTables have non-overlapping key ranges. A key exists in at most one SSTable per level (L1+).

Compression in SSTables

Compression is a major SSTable advantage over B+ tree pages. Since SSTables are sorted and immutable, they compress extremely well. B+ tree pages are updated in-place and must support random access, limiting compression options.

Common Interview Questions

  • "What is an SSTable?"
    An immutable, sorted file on disk that stores key-value pairs. It's the persistent storage format in LSM tree engines, created when a memtable is flushed from memory. Once written, it's never modified — only read, merged during compaction, or deleted.
  • "How does a read work against an SSTable?"
    First check the bloom filter (in memory, cached) — if the key might exist, binary search the index block to find the right data block, then read that block from disk and scan for the key. Most lookups touch only 1-2 disk blocks.
  • "Why are SSTables immutable?"
    Immutability eliminates in-place updates, avoiding random I/O and write-ahead logging for data files. It simplifies concurrency (no locks needed for reads), enables efficient compression, and makes file management simple (create, read, delete — no complex update logic).
  • "What's the difference between Level 0 and Level 1+ SSTables?"
    Level 0 SSTables come directly from memtable flushes and may have overlapping key ranges. Reads against L0 may need to check multiple files. Level 1+ SSTables are the result of compaction and have non-overlapping key ranges within each level — so a read against L1+ checks at most one SSTable per level.
