Serializable

Difficulty: ★★★★☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What Serializable isolation guarantees
  • The two implementation approaches: pessimistic (locking) vs. optimistic (SSI)
  • When serializable isolation is necessary
  • The performance cost and retry logic

Definition

Serializable is the strictest SQL isolation level. It guarantees that the effect of concurrently executing transactions is equivalent to running them one after another in some serial order. There exists at least one serial ordering of the transactions that produces the same result.

What Serializable Prevents

AnomalyStatus
Dirty ReadPrevented
Non-Repeatable ReadPrevented
Phantom ReadPrevented
Write SkewPrevented
Strongest guarantee: Serializable is the only isolation level that prevents all concurrency anomalies, including write skew. If your business logic requires that certain invariants hold across multiple rows, serializable isolation is the simplest way to enforce them.

Implementation Approaches

1. Pessimistic: Two-Phase Locking (2PL)

Traditional approach used by SQL Server and older systems. All reads acquire shared locks, all writes acquire exclusive locks, and locks are held until transaction end.

Pessimistic Serializable (Two-Phase Locking) Transaction A: READ → shared lock (held until commit) Transaction A: WRITE → exclusive lock (held until commit) Transaction A: COMMIT → release all locks Transaction B: tries to READ same row → BLOCKED Transaction B: tries to WRITE same row → BLOCKED Problems: - High lock contention → low throughput - Deadlocks (circular wait) → requires deadlock detection + rollback - Performance degrades linearly with concurrency

2. Optimistic: Serializable Snapshot Isolation (SSI)

Modern approach used by PostgreSQL (since 9.1), CockroachDB, and FoundationDB. Transactions proceed without blocking but are checked for serializability violations before commit.

Optimistic Serializable (SSI) Transaction A: READ row 1, WRITE row 2 Transaction B: READ row 2, WRITE row 1 Both proceed without blocking (snapshot reads) At commit time, SSI checks: - Did A read something that B wrote? (rw-dependency) - Did B read something that A wrote? (rw-dependency) - Is there a cycle in the dependency graph? If cycle exists → abort one transaction ( serialization failure) If no cycle → both commit successfully Performance: high concurrency, occasional retries
SSI is a breakthrough: Before SSI, serializable meant terrible performance due to locking. SSI uses snapshot reads (no blocking) with lightweight tracking of read-write dependencies. Only transactions that actually create a conflict cycle are aborted. In practice, the abort rate is low for most workloads.

How SSI Works: Step by Step

1. Each transaction reads from a snapshot (like Repeatable Read)
2. The database tracks which rows each transaction read
3. When a transaction writes, it checks if any concurrent transaction
   read the old value of that row (rw-conflict)
4. If Transaction A read X, and Transaction B wrote X, and
   Transaction B read Y, and Transaction A wrote Y:
   → Cycle detected! Abort one of them.
5. The aborted transaction must be retried by the application

Write Skew Example: Caught by Serializable

-- The on-call doctors example that Repeatable Read cannot prevent
-- Rule: at least one doctor must be on call

BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  SELECT on_call FROM doctors WHERE id = 1;  -- TRUE
  -- Concurrent transaction sets doctor 2 to off-call
  UPDATE doctors SET on_call = FALSE WHERE id = 1;
  -- SSI detects: you read a row that was written by someone else
COMMIT;
-- ERROR: could not serialize access due to concurrent update

The application must catch this error and retry:

-- Application-level retry logic (pseudocode)
MAX_RETRIES = 5
for attempt in 1..MAX_RETRIES:
    try:
        execute_transaction()
        break  -- success
    catch SerializationFailure:
        if attempt == MAX_RETRIES:
            raise  -- give up after max retries
        sleep(random_backoff())  -- avoid thundering herd

When to Use Serializable

  • Financial systems: Account transfers, balance constraints, double-spend prevention
  • Inventory management: Preventing overselling when stock checks span multiple rows
  • Scheduling: Preventing double-booking when the constraint spans multiple resources
  • Any multi-row invariant: Business rules like "at least one admin must exist" or "total budget cannot exceed limit"
Alternative approach: Instead of serializable isolation, you can enforce invariants with explicit locking (SELECT ... FOR UPDATE) or application-level locks. This is often more targeted and has lower overhead than full serializable isolation.

Common Interview Questions

  • "How does PostgreSQL implement serializable isolation?"
    PostgreSQL uses Serializable Snapshot Isolation (SSI). Transactions read from snapshots (no blocking), but the database tracks read-write dependencies. At commit time, if a cycle is detected in the dependency graph, one transaction is aborted with a serialization error. The application must retry.
  • "What is the performance cost of serializable vs. read committed?"
    With optimistic SSI (PostgreSQL), the cost is primarily in the rare case of aborts. For low-contention workloads, the overhead is minimal. With pessimistic 2PL (SQL Server), the cost is continuous: readers block writers, throughput drops, and deadlocks increase. The cost difference between the two implementations is enormous.
  • "How do you handle serialization failures in application code?"
    Wrap the transaction in a retry loop with exponential backoff. Catch the serialization error, wait briefly (with jitter to avoid thundering herd), and retry. Set a maximum retry count and handle the case where retries are exhausted.

What to Skip

  • Precise SSI algorithm details (rw-conflict tracking internals)
  • Formal definitions of serializability (conflict serializability, view serializability)
  • Concurrency control theory (strict 2PL, conservative 2PL, etc.)
