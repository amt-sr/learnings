Secondary Index

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand how secondary indexes provide alternative access paths to data on non-primary key columns, the bookmark lookup pattern they require in clustered tables, and the write amplification cost of maintaining them on every mutation.

Definition

A secondary index is any index that is not the primary index. It is built on one or more non-primary key columns to speed up queries that filter, sort, or join on those columns. A table can have zero or many secondary indexes. Each secondary index is a separate B+ tree whose leaf nodes contain the indexed key plus a bookmark (a pointer to the primary key value or the physical row location).

Structure

Primary Index (clustered, on user_id) Secondary Index (on email) [50|100] [m@z|s@z] / \ / \ [10|30|40] [60|80|90] [a@c|j@b] [t@b|w@a] | | | | | | | | | | Row Row Row Row Row Row (5) (42) (67) (95) PK=5 PK=42 PK=67 \ | / | ^ bookmark = primary key value | (user_id) +--- These two indexes point to the SAME row data

Bookmark Lookup

In a clustered table, the secondary index leaf stores the primary key value as the bookmark. To retrieve the full row, the engine must perform a second B+ tree traversal on the primary index.

-- Query using a secondary index on email
SELECT * FROM users WHERE email = 'jane@example.com';

-- Step 1: Secondary index lookup
--   Traverse B+ tree on email -> find leaf with email='jane@example.com'
--   Leaf contains: (email='jane@example.com', PK bookmark = user_id=42)

-- Step 2: Bookmark lookup (primary index lookup)
--   Traverse primary index B+ tree on user_id=42 -> read full row from leaf

-- Total cost: 2 B+ tree traversals (O(log N) + O(log N))
Double traversal cost: Each row returned from a secondary index requires an additional primary index lookup. For queries returning many rows, this can dominate query latency. This is the core motivation for covering indexes (3.8.6) and index-only scans.

Write Amplification

Every INSERT, DELETE, and UPDATE that touches an indexed column must update all secondary indexes on that table, plus the primary index. This is the fundamental write-cost trade-off of secondary indexes.

Operation Indexes Affected Cost per Secondary Index
INSERT Primary + ALL secondary indexes One B+ tree insert per index
DELETE Primary + ALL secondary indexes One B+ tree delete per index
UPDATE (indexed column) Primary (if PK changed) + secondary index on changed column Delete + insert in affected indexes
UPDATE (non-indexed column) None (in non-clustered) / Primary leaf (clustered) Minimal
Real-world impact: A table with 8 secondary indexes means every INSERT writes to 9 B+ trees (1 primary + 8 secondary). In write-heavy workloads (e.g., metrics ingestion, event logging), this can reduce effective write throughput by 5-10x compared to a table with no secondary indexes.

Non-Unique vs Unique Secondary Indexes

  • Unique secondary index: Enforces uniqueness on the indexed column. Each key maps to exactly one primary key bookmark.
  • Non-unique secondary index: Multiple rows can share the same key. Internally, the engine appends the primary key to the secondary key to guarantee uniqueness in the B+ tree.
-- Unique secondary index
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- Non-unique secondary index
CREATE INDEX idx_users_status ON users(status);

-- Internally, non-unique index stores: (status, user_id) to ensure uniqueness

Secondary Indexes Across Storage Engines

Engine Bookmark Type Lookup Pattern
InnoDB Primary key value Secondary tree -> PK value -> Primary tree -> row
MyISAM Physical row offset (ROWID) Secondary tree -> offset -> direct seek to row
PostgreSQL CTID (tuple ID: page + offset) Secondary tree -> CTID -> direct fetch from heap
SQL Server (non-clustered) RID (heap) or PK (clustered table) Depends on whether table has a clustered index

When to Add a Secondary Index

  • High selectivity: The column has many distinct values relative to the table size (e.g., email, username). Low-selectivity columns (e.g., boolean flags, country codes) benefit less.
  • Frequent query patterns: The column appears in WHERE, JOIN, ORDER BY, or GROUP BY clauses of hot queries.
  • Read-heavy workload: The read speedup justifies the write amplification.
  • Can be a covering index: The index can satisfy the query without a bookmark lookup.

Common Interview Questions

  • What is a secondary index and how does it differ from a primary index?
  • Explain the bookmark lookup process in InnoDB.
  • Why does adding too many secondary indexes hurt write performance?
  • How does InnoDB store the bookmark in a secondary index versus MyISAM?
  • A table has 5 secondary indexes -- how many B+ tree writes does a single INSERT require?
  • What is index selectivity and why does it matter for secondary indexes?
  • When would you choose to drop a secondary index?
  • How do non-unique secondary indexes handle duplicate key values internally?

What to Skip

Depth to avoid in interviews:
  • Internal lock/latch protocols for concurrent B+ tree modifications in secondary indexes.
  • Change buffer internals for secondary index maintenance in InnoDB.
  • Exact format of CTID or RID structures in PostgreSQL/SQL Server.
  • Purge thread internals for cleaning up secondary index entries after row deletion.
