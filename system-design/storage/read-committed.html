Read Committed

Difficulty: ★★★☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What Read Committed guarantees and what it does not
  • How it prevents dirty reads but allows non-repeatable reads
  • Two implementation strategies: locking vs. MVCC
  • Why it is the default in most databases

Definition

Read Committed is the most common isolation level and the default in PostgreSQL, Oracle, and SQL Server. It guarantees that a transaction only reads data that has been committed by other transactions. However, within the same transaction, re-reading the same row may return a different value if another transaction committed a change in between.

Dirty Read Is Prevented

Transaction A Transaction B ──────────────────────────────────────────────────────── BEGIN; UPDATE accounts SET balance = balance - 500 WHERE id = 1; BEGIN; SELECT balance FROM accounts WHERE id = 1; -- BLOCKED! (waiting for A to -- commit or rollback) COMMIT; -- Now reads 500 (committed value) COMMIT;
Guarantee: Transaction B will only ever see the pre-update value (1000) or the post-commit value (500). It can never see an uncommitted intermediate state.

Non-Repeatable Read Is Still Possible

Transaction A Transaction B ──────────────────────────────────────────────────────── BEGIN; BEGIN; SELECT name FROM users WHERE id = 1; -- returns "Alice" UPDATE users SET name = "Bob" WHERE id = 1; COMMIT; SELECT name FROM users WHERE id = 1; -- returns "Bob" (changed!) COMMIT;

The same query within Transaction A returned different results within the same transaction. This is a non-repeatable read — allowed at Read Committed level.

Two Implementation Strategies

Strategy 1: Row-Level Locking (Pessimistic)

Used by older systems and SQL Server (without RCSI). When a transaction writes a row, it holds an exclusive lock until commit. Other transactions that want to read must wait for the lock to be released.

Writer holds exclusive lock on row Reader waits until writer commits or rolls back Reader acquires shared lock (released immediately after read) Transaction A: WRITE lock on row 1 ──── COMMIT ──→ release Transaction B: waits... ──── READ lock on row 1 ──→ release after read ↑ sees only committed data but lock is released after read so next read might see something new

Strategy 2: MVCC (Optimistic)

Used by PostgreSQL, MySQL/InnoDB, and Oracle. Each query within a transaction sees a snapshot of committed data as of the start of that query. No blocking between readers and writers.

Each statement gets a fresh snapshot of committed data Readers never block writers, writers never block readers Transaction A: UPDATE row 1 → creates new version Transaction B: SELECT row 1 → sees old committed version (no blocking) After A commits: Transaction B: SELECT row 1 → sees new committed version (fresh snapshot) Key difference from locking: no waiting, but same anomaly behavior
Interview distinction: Both strategies prevent dirty reads and allow non-repeatable reads. The difference is in how they achieve it: locking uses waits, MVCC uses versioning. MVCC is strictly better for read-heavy workloads because readers never block writers.

Anomaly Summary

AnomalyStatusExplanation
Dirty ReadPreventedOnly committed data is visible
Non-Repeatable ReadPossibleSame row may change between reads
Phantom ReadPossibleNew rows may appear in range query
Lost UpdatePossibleLast write wins without detection

When Read Committed Is Sufficient

  • Most OLTP workloads: The majority of web applications operate correctly at Read Committed
  • Read-mostly systems: When reads greatly outnumber writes, the occasional non-repeatable read is acceptable
  • Single-row operations: When transactions touch only one row at a time, non-repeatable reads are less likely
  • Idempotent operations: When the result of reading stale data has no harmful side effect

When Read Committed Is NOT Enough

  • Financial transfers: Reading an account balance, making a decision, then acting on stale data could cause double-spending
  • Inventory management: Checking stock levels then decrementing requires a consistent view
  • Batch processing: Long-running transactions that need a stable view of the data across multiple queries
-- Escalate to Repeatable Read when needed
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- Or use explicit locking for specific rows
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;

-- Or use serializable for full guarantees
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
Default levels matter: PostgreSQL defaults to Read Committed because it uses MVCC, making it fast and safe. MySQL defaults to Repeatable Read because InnoDB's MVCC implementation makes the upgrade cheap. Both are reasonable defaults — the key is knowing when to change them.

Common Interview Questions

  • "What is the default isolation level in PostgreSQL and what does it prevent?"
    PostgreSQL defaults to Read Committed. It prevents dirty reads (you cannot see uncommitted data from other transactions) but allows non-repeatable reads (the same row may change between reads within your transaction) and phantom reads (new rows may appear in subsequent range queries).
  • "How does MVCC implement Read Committed differently from locking?"
    With MVCC (PostgreSQL, Oracle), each statement sees a snapshot of committed data as of the statement's start. No shared locks are acquired, so readers never block writers. With locking (SQL Server default), readers acquire shared locks and wait for writers to release exclusive locks. Both achieve the same isolation guarantees.
  • "Can two transactions update the same row at Read Committed?"
    Yes, but the second transaction must wait for the first to commit. After the first commits, the second proceeds with the new value. However, if the second transaction read the row before updating, it might make decisions based on stale data — the lost update problem. Use SELECT ... FOR UPDATE or a higher isolation level to prevent this.
