Visibility Rules

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★☆☆

Goal

Understand the exact rules MVCC uses to determine which version of a row is visible to a given transaction, how the snapshot filters version chains, and how different isolation levels modify these visibility rules.

Definition

Visibility rules are the predicate logic that MVCC applies when walking a version chain to decide whether a specific tuple version should be returned to the querying transaction. For each tuple version, the rules examine the tuple's xmin (creator) and xmax (deleter) against the transaction's snapshot to determine visibility. These rules are what make snapshot isolation work correctly.

Key insight: Visibility rules are the bridge between version chains and snapshots. The version chain provides the candidate versions; the snapshot defines which transaction IDs are considered committed; the visibility rules combine both to select the single correct version.

The Visibility Check Algorithm

For each tuple version encountered while traversing the chain, PostgreSQL applies the following checks in order:

Function HeapTupleSatisfiesMVCC(tuple, snapshot): ┌─ Check 1: Is the inserter visible? │ if tuple.xmin >= snapshot.xmax: │ → INSERT happened after our snapshot → INVISIBLE │ if tuple.xmin is in snapshot.xip[]: │ → INSERT was in-progress at snapshot time → INVISIBLE │ if tuple.xmin < snapshot.xmin and not frozen: │ → Must check CLOG (commit log) for committed/aborted │ if xmin is aborted: │ → Creator rolled back → INVISIBLE │ → Creator is committed ✓ │ ├─ Check 2: Is the deleter invisible? │ if tuple.xmax == 0: │ → No delete marker → this version is still LIVE → VISIBLE ✓ │ if tuple.xmax >= snapshot.xmax: │ → DELETE happened after our snapshot → VISIBLE ✓ │ (delete hasn't happened yet from our perspective) │ if tuple.xmax is in snapshot.xip[]: │ → DELETE was in-progress at snapshot time → VISIBLE ✓ │ (we can't see uncommitted deletes) │ if tuple.xmax is committed: │ → DELETE is visible to us → INVISIBLE │ → DELETE was rolled back → VISIBLE ✓ │ └─ Result: VISIBLE or INVISIBLE for this tuple version The reader walks the chain from newest to oldest. The FIRST visible version found is returned. If no version is visible, the row does not exist from this snapshot.

Step-by-Step Walkthrough

Version chain: V3 (xmin=300, xmax=0) → V2 (xmin=200, xmax=300) → V1 (xmin=100, xmax=200) Snapshot: xmin=150, xmax=350, xip[]=[280, 320] Checking V3 (newest): xmin=300: 150 ≤ 300 < 350, 300 is NOT in xip[], check CLOG → committed ✓ xmax=0: no deleter → LIVE → VISIBLE ✓ → Return V3 ───────────────────────────────────────────────────────────── Scenario: T1 (xmin=100) committed, T2 (xmin=200) aborted, T3 (xmin=300) committed Version chain: V3 (xmin=300, xmax=0) → V2 (xmin=200, xmax=300) → V1 (xmin=100, xmax=200) Snapshot: xmin=150, xmax=350, xip[]=[] Checking V3: xmin=300: committed ✓, xmax=0 → VISIBLE ✓ → Return V3 ───────────────────────────────────────────────────────────── Scenario: Snapshot taken BEFORE T3 committed (T3 in xip) Snapshot: xmin=150, xmax=350, xip[]=[300] Checking V3: xmin=300: in xip[] → INVISIBLE ✗ Checking V2: xmin=200: not in xip[], check CLOG → aborted ✗ → INVISIBLE ✗ Checking V1: xmin=100: 100 < xmin(150), check CLOG → committed ✓ xmax=200: 150 ≤ 200 < 350, not in xip[], check CLOG → aborted → VISIBLE ✓ → Return V1 (the pre-T2, pre-T3 version!)

Isolation Level Variations

The core visibility rules above describe PostgreSQL's behavior under Read Committed and Repeatable Read. The difference lies in when the snapshot is taken, not in the rules themselves:

Isolation Level Snapshot Timing Effect on Visibility
Read Committed New snapshot at the start of each SQL statement A row updated by a concurrent transaction between statements will be visible in the next statement (reader sees latest committed)
Repeatable Read Single snapshot at the first statement of the transaction The reader never sees changes from concurrent transactions, even those committed mid-transaction
Serializable Single snapshot (same as RR) Same visibility as RR, but additional SIREAD locks detect serialization anomalies (throws serialization_failure on conflict)
PostgreSQL's Repeatable Read is actually Snapshot Isolation: Strictly speaking, PostgreSQL's REPEATABLE READ prevents dirty reads, non-repeatable reads, and phantom reads, but it allows write skew (a serialization anomaly). True ANSI SQL REPEATABLE READ would prevent write skew. PostgreSQL reserves the name SERIALIZABLE for the stronger guarantee that requires predicate locking.

The Commit Log (CLOG)

The visibility rules frequently need to check whether a transaction committed or aborted. Rather than scanning the WAL or transaction table, PostgreSQL maintains the CLOG (commit log)—a small, fixed-size structure with two bits per transaction:

CLOG Status Meaning
00 In-progress
01 Committed
10 Aborted
11 Sub-committed (internal two-phase commit state)
Performance implication: The CLOG is memory-mapped and frequently cached. Visibility checks for transactions within the active window typically hit memory. For very old transactions (before the active window), the system checks pg_xact (the on-disk CLOG). This is why long-running transactions degrade performance—they expand the window of CLOG pages that must remain accessible.

Self-Referencing Visibility

A special case arises when the querying transaction itself created or modified a row. If transaction T1 (ID=200) updates a row and then reads it within the same transaction, the visibility rules must account for the fact that T1's own modifications are visible to T1 even though T1 hasn't committed yet:

  • If tuple.xmin == current_txid, the tuple is visible (self-inserted).
  • If tuple.xmax == current_txid, the tuple is invisible (self-deleted or self-updated).
  • These checks happen before the snapshot-based checks.
Why this matters: Without self-referencing visibility, a transaction could not read its own writes. A UPDATE ... WHERE that modifies 100 rows must be able to see its own previous modifications within the same statement.

Common Interview Questions

  • Walk through the visibility check algorithm for a specific tuple version given a snapshot.
  • How does MVCC determine which version of a row is visible when three transactions have modified the same row?
  • What is the CLOG and why is it necessary for visibility checks?
  • How do the visibility rules differ between Read Committed and Repeatable Read?
  • What happens when a transaction reads a row it modified itself? Explain self-referencing visibility.
  • Why does PostgreSQL's Repeatable Read not prevent write skew? How does Serializable address this?
  • Design a visibility check function. What are the inputs and outputs? Walk through edge cases.

What to Skip

  • PostgreSQL's HeapTupleSatisfies* C function implementation details and optimization paths

  • Subtransaction visibility (sub-xid tracking, SubTransGetTopmostTransaction)
  • PostgreSQL's hint bits (tuple-level caching of commit status) at the memory-barrier level
