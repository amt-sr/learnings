Bloom Filter

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What a bloom filter is: a space-efficient probabilistic data structure
  • How it works: bit array + multiple hash functions
  • False positives vs true negatives (the critical tradeoff)
  • Why bloom filters are essential for SSTable reads in LSM trees

Definition

A bloom filter is a space-efficient probabilistic data structure that tests whether an element is a member of a set. It can definitively say "NO" (element is not in the set) but can only say "MAYBE" (element might be in the set). The tradeoff is a configurable false positive rate in exchange for dramatically reduced memory usage compared to storing the full set.

The Two Answers

ResultMeaningGuarantee
"Definitely not in set"At least one bit position is 0100% accurate (no false negatives)
"Possibly in set"All bit positions are 1False positive possible (all bits set by other elements)
Key insight: Bloom filters never produce false negatives. If it says an element is not present, it is definitively not there. If it says an element might be present, you need a secondary check (e.g., reading the actual SSTable) to confirm.

ParameterTypical ValueEffect
n (number of elements)Varies by datasetMore elements → larger bit array needed
m (bit array size)~10 bits per elementLarger → lower false positive rate, more memory
k (number of hash functions)~7 hash functionsOptimal k = (m/n) * ln(2). Too few → more false positives. Too many → slower, fills array faster.
p (false positive rate)1% (default) or 0.1%Lower → more memory and more hash functions
  Space efficiency:
  - Storing 1 million URLs as strings:  ~50 MB

  That's a 40x memory savings for 1% false positive rate.

  For databases:
  - SSTable with 100K keys: bloom filter ≈ 120 KB
  - Kept in memory, saves millions of disk reads

Role in LSM Trees

Bloom filters are the critical optimization that makes LSM tree reads practical. Without them, every read would need to check every SSTable at every level on disk. With bloom filters, most SSTables are eliminated from consideration before any disk I/O occurs.

Limitations

Cannot delete elements. Setting a bit back to 0 could falsely mark other elements as absent. Solutions:
  • Counting Bloom Filter — use counters (4-bit integers) instead of bits. Decrement on delete. More memory (4x).
  • Rebuild the filter — regenerate from source data periodically.
  • Scalable Bloom Filter — add new bloom filters for new elements, expire old ones.

Common Interview Questions

  • "What is a bloom filter?"
    A space-efficient probabilistic data structure for membership testing. It uses a bit array and k hash functions. Can definitively say an element is NOT in the set (no false negatives) but may incorrectly say an element IS in the set (false positives, rate is configurable, typically 1%).
  • "Why does Cassandra use bloom filters?"
    Cassandra stores data in SSTables organized in levels. Without bloom filters, a read would need to check every SSTable on disk. Bloom filters are kept in memory and can quickly eliminate SSTables that definitely don't contain the key, reducing disk I/O by 80-95%.
  • "How do you size a bloom filter?"
    Given n elements and desired false positive rate p: m = -(n * ln(p)) / (ln(2))^2 bits. For n=1M and p=1%: m ≈ 9.6 million bits ≈ 1.2 MB. Optimal k = (m/n) * ln(2) ≈ 7 hash functions. In practice, use 10 bits per element for 1% FPR.
  • "Can you remove elements from a bloom filter?"
    Standard bloom filters don't support deletion because multiple elements may share the same bit positions. Setting a bit to 0 could invalidate other elements. Solutions: use a counting bloom filter (counters instead of bits, 4x memory), or rebuild the filter periodically.
  • "Bloom filter vs hash set?"
    Hash set: exact membership testing, but memory proportional to element size. Bloom filter: approximate, but memory is constant per element (~10 bits) regardless of element size. For 1 million 100-byte URLs: hash set needs ~100 MB, bloom filter needs ~1.2 MB. The tradeoff is a 1% false positive rate.
