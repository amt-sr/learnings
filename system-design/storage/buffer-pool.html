Buffer Pool

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • Why databases use an in-memory buffer pool instead of reading directly from disk
  • Page replacement algorithms: LRU, LRU-K, clock sweep, and why naive LRU fails
  • Dirty pages: modified pages not yet flushed, and how they are managed
  • Checkpointing dirty pages to disk — fuzzy vs sharp
  • Concurrency control: pinning, shared vs exclusive latches

Definition

The buffer pool is the database's in-memory cache of frequently and recently accessed disk pages. It sits between the query executor and the storage engine: every page read passes through the buffer pool first, and every page modification is written to the buffer pool before eventually being flushed to disk. It is typically the largest consumer of RAM on a database server.

Why Not Read Directly from Disk?

The core problem: Disk I/O is orders of magnitude slower than memory access. A single 8 KB page read from SSD takes ~100 microseconds; from RAM it takes ~100 nanoseconds — a 1000x gap. Without a buffer pool, every query would hit disk for every page, limiting throughput to the disk's IOPS capacity.
Query requests page #42 — two paths: WITHOUT buffer pool (every query hits disk): ┌──────────┐ ┌──────────┐ │ Query │───────────────────→│ Disk │ ~100 μs per page │ Executor │←──────────────────│ (SSD) │ └──────────┘ └──────────┘ 1000 queries/sec * 3 pages each = 3000 disk reads/sec WITH buffer pool (hot pages cached): ┌──────────┐ ┌──────────────┐ ┌──────────┐ │ Query │────→│ Buffer Pool │────→│ Disk │ cold path │ Executor │←────│ (RAM) │←────│ (SSD) │ └──────────┘ └──────────────┘ └──────────┘ ↑ │ └─────────────────┘ hot path: ~100 ns (99%+ hit rate) With 99% hit rate: only 30 disk reads/sec = 100x reduction in disk I/O

Key Data Structures

  • Page frames: Fixed-size slots in RAM, each holding one database page. The buffer pool has N frames (determined by configuration, e.g., 16 GB / 16 KB = 1M frames).
  • Hash map: Maps page_id to frame address for O(1) lookup.
  • Free list: Linked list of unoccupied frames, used when loading new pages from disk.
  • LRU list: Ordered list of all occupied frames, used for eviction decisions.
  • Dirty list: Ordered list of modified frames, used by the background flush thread.

Page Replacement Algorithms

Simple LRU (Least Recently Used)

  • Maintain a linked list ordered by last access time; evict the tail.
  • Fatal flaw: A full table scan loads every page into the buffer pool, evicting all hot pages. After the scan, the buffer pool is cold and every subsequent OLTP query hits disk.

LRU-K (MySQL InnoDB, PostgreSQL)

Track the last K access timestamps to distinguish between pages accessed once (likely a scan) and pages accessed multiple times (hot data):

MySQL InnoDB Midpoint Insertion LRU: Old (hot) sublist ←────── Midpoint ──────→ Young (cold) sublist ┌───────┬───────┬───────┬──────┬───────┬───────┬───────┐ │ Pg 42 │ Pg 87 │ Pg 15 │ MP │ Pg300 │ Pg301 │ Pg302 │ │ 63/8 │ 45/8 │ 20/8 │ │ 1/8 │ 1/8 │ 1/8 │ └───────┴───────┴───────┴──────┴───────┴───────┴───────┘ MRU end New pages enter here Default ratio: 3/8 old, 5/8 young (innodb_old_blocks_pct=37) Behavior: - New page loaded → insert at midpoint (young sublist tail) - Page accessed again while in young sublist → promote to old sublist (MRU) - Table scan loads 10,000 pages → all go to young sublist → they do NOT evict hot pages in old sublist → they get evicted first when space is needed - Hot pages (Pg 42, Pg 87) sit safely in old sublist

Clock Sweep (PostgreSQL)

A circular buffer with a "hand" pointer. Each page frame has a reference bit:

  • When a page is accessed, set its reference bit to 1.
  • When eviction is needed, the clock hand sweeps through frames. If reference bit = 1, clear it and skip. If reference bit = 0, evict.
  • Pages accessed multiple times get multiple "second chances" before eviction.
  • Effectively approximates LRU without maintaining an expensive sorted list.

Dirty Pages

A page is "dirty" when it has been modified in the buffer pool but not yet written to the data file on disk:

Write Path (UPDATE users SET name = 'Bob' WHERE id = 42): 1. Request page #42 (read from pool or disk) 2. Modify page #42 in the buffer pool 3. Mark page #42 as DIRTY 4. Write the change to the WAL log buffer 5. fsync WAL to disk (BEFORE acknowledging commit) 6. Return success to client Background Thread (async, non-blocking): ┌─────────────────────────────────────────┐ │ Dirty Page #42 in Buffer Pool │ │ (modified, not yet on disk) │ │ │ │ │ ↓ (background flush thread) │ │ Data File on Disk │ │ (page #42 eventually written here) │ └─────────────────────────────────────────┘ Key: WAL flush is synchronous (blocks commit). Data page flush is asynchronous (never blocks queries). If crash happens before flush → WAL replay recovers changes.
Core principle: The WAL is the source of truth for crash recovery. Dirty pages in the buffer pool are a performance optimization — they avoid reading from disk on subsequent access. If the database crashes, dirty pages are lost from memory, but the WAL contains all the changes needed to reconstruct them.

Checkpointing Dirty Pages

A checkpoint forces all dirty pages modified before a specific LSN to be written to disk:

StrategyDescriptionUsed By
Sharp checkpointFlush all dirty pages quickly, possibly blocking writesPostgreSQL
Fuzzy checkpointGradually flush dirty pages in the background over timeMySQL InnoDB

Pinning and Latches

The buffer pool uses latches (lightweight locks) to protect page frames from concurrent access:

OperationLock TypePurpose
Reading a pageShared latch (S)Multiple readers can read the same page concurrently
Modifying a pageExclusive latch (X)Only one writer; blocks all readers
Preventing evictionPin countIncremented when a thread is actively using a page; decremented when done. A page with pin > 0 cannot be evicted.
Interview tip: When discussing buffer pool concurrency, mention that pinning prevents the background eviction thread from evicting a page that a query thread is actively reading. Without pinning, the eviction thread could evict a page between the time a query reads it from the hash map and the time it copies the data.

Buffer Pool Sizing

Rule of thumb: The buffer pool should be 1.25–1.5x the hot working set size. It should never exceed ~80% of total RAM (leaving room for the OS, connection buffers, and other memory).
DatabaseSettingDefaultRecommendation
MySQL InnoDBinnodb_buffer_pool_size128 MB50–75% of total RAM
PostgreSQLshared_buffers128 MB25% of total RAM
SQL Servermax server memory2 TB70–80% of total RAM
Why PostgreSQL recommends less: PostgreSQL relies on the OS page cache for additional buffering (no O_DIRECT by default), so shared_buffers is only part of the total memory picture. MySQL InnoDB uses O_DIRECT, bypassing the OS cache, so the buffer pool must be larger.

Common Interview Questions

  • "What is the buffer pool and why is it critical?"
    The buffer pool is an in-memory cache of database pages that avoids disk I/O for hot data. Since disk access is ~1000x slower than memory, the buffer pool is the single most important performance knob. A well-sized buffer pool with >99% hit ratio means almost all queries read from memory. If the hit ratio drops to 90%, every 10th query hits disk, causing tail latency spikes.
  • "How does eviction work, and why does naive LRU fail?"
    Naive LRU evicts the least-recently-used page. But a full table scan loads every page, evicting hot data. MySQL InnoDB uses LRU with midpoint insertion: new pages enter at the midpoint of the LRU list and are only promoted to the hot end if accessed again. PostgreSQL uses a clock sweep algorithm that gives pages multiple "second chances" before eviction.
  • "What happens to dirty pages during a crash?"
    Dirty pages in memory are lost. However, all modifications were written to the WAL before the transaction was committed. During crash recovery, the database replays the WAL (redo log) to reconstruct the dirty pages and apply any committed changes that were not flushed.
  • "What is page pinning?"
    Pinning increments a reference counter on a buffer pool frame while a thread is actively reading or modifying the page. A pinned page (pin count > 0) cannot be evicted by the replacement algorithm. This prevents the eviction thread from removing a page between the time a query locates the frame and the time it finishes accessing the data.
