B+ Tree

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand:

  • How a B+ tree is structured (internal nodes vs leaf nodes)
  • Why B+ trees are the default index structure in relational databases
  • How balanced height guarantees O(log n) lookups
  • Sequential I/O optimization via linked leaf nodes

Definition

A B+ tree is a self-balancing tree data structure that keeps data sorted and allows searches, insertions, deletions, and sequential access in O(log n) time. Internal nodes store only keys for routing; all actual data (or pointers to data) reside in the leaf nodes, which are linked together in a doubly-linked list.

B+ Tree (order = 3, max 3 keys per node) Level 0 (Root): [ 30 | 60 ] / | \ Level 1 (Internal): [10|20] [40|50] [70|80] / | \ / | \ / | \ Level 2 (Leaves): [1][5][15] [25][35] [45][55][65] [75][85][95] ——————————————————————————————————————————————→ linked list Properties: - Root: 2..3 keys (flexible, fewer on low fill) - Internal: 2..3 keys (must be at least half full) - Leaf: 2..3 keys + pointer to next leaf - All leaves at same depth → tree is always balanced

Key Properties

PropertyDetail
BalancedAll leaves at the same depth — guaranteed by the tree invariant. No degenerate O(n) paths.
Wide nodesEach node holds 100s of keys (tuned to a disk page, typically 4 KB or 16 KB). This makes the tree very shallow.
Data in leaves onlyInternal nodes are routing tables. This keeps internal nodes small and cache-friendly.
Linked leaf nodesLeaves form a doubly-linked list — enables fast range scans and sequential access.
OrderedKeys are sorted within every node and across the tree — supports efficient range queries.

Why B+ Trees Dominate Database Indexing

1. Disk-Friendly Design

Database nodes are sized to match a disk page (4-16 KB). A single disk read fetches an entire node with hundreds of keys. This means a B+ tree with 1 billion rows needs only 3-4 levels:

Assumptions:
  - Page size:        8 KB
  - Key size:         12 bytes
  - Page capacity:   ~500 keys per internal node
  - Fan-out:          500

  Levels for 1 billion rows:
    Level 0 (root):         1 page
    Level 1 (internal):     1 page  (500 keys → 500 children)
    Level 2 (internal):     ~2,000 pages (1M keys → 1M children)
    Level 3 (leaf):         ~2,000,000 pages

  Lookup = 4 disk seeks → predictable, fast

2. Sequential I/O for Range Scans

Leaf nodes are linked, so a range query like WHERE age BETWEEN 20 AND 30 reads contiguous leaf pages — perfect for disk prefetching and OS read-ahead.

3. Predictable Performance

OperationTime ComplexityDisk I/O
Point lookup (by key)O(log n)3-4 page reads
Range scanO(log n + k)3-4 reads + k leaf pages
InsertO(log n)3-4 reads + 1 write (no split) or more (with split)
DeleteO(log n)3-4 reads + 1 write (no merge) or more (with merge)

Insertion and Splits

Insert key=42 into this B+ tree: Before: After (split at leaf): [ 30 | 60 ] [ 30 | 45 | 60 ] ← promoted 45 / | \ / | \ [...] [40|50] [...] [...] [40|42] [50] [...] ← split leaf Split process: 1. Find target leaf 2. Insert key in sorted position 3. If leaf overflows (too many keys): a. Split leaf into two halves b. Copy middle key up to parent c. If parent overflows → recurse up (rare) In practice, bulk loading (bottom-up construction) is preferred for initial index creation — O(n) vs O(n log n) for individual inserts.

Key insight: B+ trees are superior for databases because higher fan-out means shallower trees, and linked leaves enable fast range scans — the most common query pattern in analytical workloads.

When B+ Trees Struggle

Write-heavy workloads: Random insertions cause page splits and fragmentation. Each write requires a read-modify-write cycle (read leaf → modify → write back). This is why write-intensive systems use LSM trees instead.
  • Page splits cause write amplification (write 1 row → rewrite multiple pages)
  • Fragmentation over time reduces page fill factor (typically drops to 60-70%)
  • Random I/O on inserts — poor for SSDs under high write load

Common Interview Questions

  • "Why are B+ trees used instead of hash tables for database indexes?"
    Hash tables support only point lookups (O(1)), not range queries. B+ trees support both point and range queries in O(log n), and linked leaves make range scans efficient. Most database queries involve ranges (WHERE, ORDER BY, BETWEEN).
  • "How many disk reads to look up a key in a B+ tree with 10 million rows?"
    With 4 KB pages and 8-byte keys, fan-out is ~400. log₄₀₀(10M) ≈ 3.4 → 4 disk reads. With buffer pool caching, often 0-1 actual disk reads.
  • "What happens during a B+ tree node split?"
    When a node exceeds its capacity, it splits into two. The median key is promoted (copied, not moved) to the parent. If the parent also overflows, the split propagates upward — potentially creating a new root level (tree grows by 1 level). This is rare and amortized O(1) per insert.
  • "B+ tree vs LSM tree for a write-heavy workload?"
    B+ tree: in-place updates, read-optimized, writes cause page splits. LSM tree: append-only writes, write-optimized, reads may check multiple levels. See: LSM Tree
