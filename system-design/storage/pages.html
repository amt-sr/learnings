Database Pages

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • Why databases organize data into fixed-size pages
  • Common page sizes: 4KB, 8KB, 16KB — and the tradeoffs
  • Page as the fundamental unit of I/O in the buffer pool
  • Internal page structure: header, slot array, records, free space
  • How page size choice affects B-tree height and scan performance

Definition

A database page (also called a block or data page) is the fundamental unit of storage and I/O for a database engine. All reads and writes between disk and the buffer pool happen in whole pages. A page typically contains multiple records along with metadata for managing those records.

Internal Page Structure

Typical Database Page Layout (e.g., PostgreSQL 8 KB): ┌──────────────────────────────────────────┐ │ Page Header (24 bytes) │ │ ├─ Page type (data, index, overflow...) │ │ ├─ Free space offset │ │ ├─ Number of tuples (records) │ │ └─ Special space pointer │ ├──────────────────────────────────────────┤ │ │ │ Item Pointers (Slot Array) │ │ ┌─────┐ ┌─────┐ ┌─────┐ │ │ │ #1 │ │ #2 │ │ #3 │ ... │ │ │off,len│off,len│off,len│ │ │ └─────┘ └─────┘ └─────┘ │ │ │ │ Free Space (grows/shrinks) │ │ ████████████████████████████████ │ │ │ │ Tuples (Records) — grow from bottom up │ │ ┌──────────────────────────────┐ │ │ │ Record #3 │ │ │ ├──────────────────────────────┤ │ │ │ Record #2 │ │ │ ├──────────────────────────────┤ │ │ │ Record #1 │ │ │ └──────────────────────────────┘ │ └──────────────────────────────────────────┘ Records grow upward from the bottom. Slot array grows downward from the top. They meet in the middle when the page is full.

Page Size Tradeoffs

FactorSmaller Page (4 KB)Larger Page (16 KB)
B-tree fanoutLower (more levels)Higher (fewer levels)
Index depthDeeper tree (more I/O for lookup)Shallower tree (fewer I/O)
Wasted space (fragmentation)Less waste per pageMore waste per page
Point query (single record)Less data read from diskMore data read (full page)
Range scan (sequential)More page reads for same dataFewer page reads (more records/page)
Concurrent updatesLess contention per pageMore contention (more rows share a page)
Fit with OS block size1:1 match with 4 KB OS blocksSpans multiple OS blocks (4x 4 KB blocks)
Choosing page size: OLTP workloads (point queries, high concurrency) favor smaller pages — less contention, less wasted I/O per lookup. OLAP workloads (range scans, batch processing) favor larger pages — fewer I/O operations for scanning large amounts of data. This is why MySQL InnoDB defaults to 16 KB (targeting mixed workloads) while PostgreSQL defaults to 8 KB.

Page as the I/O Unit

Even if you need a single 100-byte record, the database reads the entire page containing it:

Read Request Flow: Query: SELECT * FROM users WHERE id = 42 1. B-tree index lookup finds: Record is in Page #1,234 2. Check buffer pool: Is Page #1,234 in RAM? ├─ YES → Read directly from memory (~100 ns) └─ NO → Read entire 8 KB page from disk (~100 μs SSD) 3. Find record within page using slot array offset 4. Return the single record Key: You always read a full page, never partial pages. This is why the buffer pool caches pages, not individual records.

Page Types

A database file contains multiple types of pages:

Page TypePurpose
Data PageStores actual table rows (records)
Index PageStores B-tree or B+tree index entries
Overflow PageStores large values that do not fit on the main page (TOAST in PostgreSQL)
Free Space MapTracks which pages have available space for new records
Visibility MapTracks which pages contain only committed-visible rows (helps vacuum/Index-only scans)
WAL PageContains write-ahead log records

Common Interview Questions

  • "Why do databases use pages instead of reading individual rows?"
    Pages align with the OS block size and SSD erase/write units. Reading aligned, fixed-size chunks is more efficient than variable-size reads. Pages also provide a natural unit for the buffer pool to cache, manage, and evict.
  • "How does page size affect B-tree performance?"
    Larger pages = more keys per page = higher fanout = shallower B-tree = fewer disk reads for point lookups. A B-tree with 16 KB pages might need 3 levels to index 1 billion rows, while 4 KB pages might need 5 levels. However, each level reads more data (16 KB vs 4 KB). The tradeoff is tree height vs page read size.
  • "How many records fit in a page?"
    Depends on record size. An 8 KB page with 200-byte records holds roughly ~35 records (8192 / 200, minus header and slot array overhead). Wider records mean fewer per page, more pages to scan, and more disk I/O. This is why selecting only needed columns (avoiding SELECT *) can reduce I/O in some engines through covering indexes.
