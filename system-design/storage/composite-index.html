Composite Index

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand how composite (multi-column) indexes work, why column order matters via the leftmost prefix rule, how to choose the optimal column ordering, and when a composite index outperforms multiple single-column indexes.

Definition

A composite index (also called a multi-column or compound index) is an index built on two or more columns. The index key is formed by concatenating the column values in the defined order. This order is critical: the index can only be used for queries that reference columns starting from the leftmost column in sequence.

Leftmost Prefix Rule

The most important concept for composite indexes. An index on (A, B, C) can satisfy queries that filter on:

  • (A) -- yes
  • (A, B) -- yes
  • (A, B, C) -- yes
  • (B) -- no (skips A)
  • (B, C) -- no (skips A)
  • (A, C) -- partial -- uses A for index seek, C is not used (skips B)
Index on (last_name, first_name, birth_date) Tree is sorted by: last_name -> first_name -> birth_date [King]----[Smith]----[Williams] / | \ [King,A..] [Smith,J..] [Williams,M..] [King,B..] [Smith,M..] [Williams,R..] Query: WHERE last_name = 'Smith' AND first_name = 'John' -> Index seek on (Smith, John): FULLY USED Query: WHERE last_name = 'Smith' -> Index seek on (Smith): FULLY USED (leftmost prefix) Query: WHERE first_name = 'John' -> Cannot use index (skips leftmost column last_name) Query: WHERE last_name = 'Smith' AND birth_date = '1990-01-01' -> Only last_name='Smith' is used; birth_date is skipped (first_name gap)

Column Ordering Strategy

Equality Columns Before Range Columns

When a composite index contains both equality and range conditions, place equality columns first. Once a range condition is encountered, columns to its right cannot be used for index seek.

-- Given: index on (status, created_at)

-- GOOD: equality on status, range on created_at
SELECT * FROM orders WHERE status = 'shipped' AND created_at > '2025-01-01';
-- Index seeks on status='shipped', then range scan on created_at. Both used.

-- BAD: if index were on (created_at, status)
SELECT * FROM orders WHERE status = 'shipped' AND created_at > '2025-01-01';
-- Range on created_at first -> status cannot be used. Only partial index utilization.

High Cardinality First?

A common rule of thumb is to put the highest-cardinality (most selective) column first. However, this is secondary to the equality-before-range rule. The optimal ordering depends on the query mix.

Cardinality vs access pattern: If column A has 2 values (boolean) and column B has 10,000 values, but queries always filter on A=BETWEEN, putting B first means the index can filter to a small set before scanning the range. Always evaluate based on actual query patterns, not just cardinality alone.

When a Composite Index Works vs Does Not

Query Index: (last_name, first_name) Used? Explanation
WHERE last_name = 'Smith' Leftmost prefix match Yes Single column prefix
WHERE last_name = 'Smith' AND first_name = 'John' Full key match Yes Both columns used for seek
WHERE first_name = 'John' Skips leftmost No Cannot skip last_name
WHERE last_name = 'Smith' AND age = 30 Partial match Partial last_name used, age not in index
ORDER BY last_name, first_name Index order matches Yes Avoids filesort
ORDER BY first_name, last_name Reverse order No Index is (last, first), not (first, last)

Covering Index Optimization

A composite index can serve as a covering index when it includes all columns referenced by the query (SELECT, WHERE, ORDER BY, GROUP BY). This eliminates the bookmark lookup entirely.

-- Index on (user_id, created_at, amount)
-- Query:
SELECT created_at, amount FROM orders WHERE user_id = 42 ORDER BY created_at;

-- All columns (user_id, created_at, amount) are in the index.
-- No bookmark lookup needed. Index-only scan.

This ties directly into the next topic: 3.8.6 Covering Index, where the INCLUDE clause provides additional flexibility for non-key columns.

Composite Index vs Multiple Single-Column Indexes

Aspect Composite Index (A, B) Two Single Indexes (A) + (B)
Disk space 1 index (slightly wider rows) 2 indexes (duplicate B+ tree overhead)
Write amplification 1 index to update on INSERT 2 indexes to update on INSERT
WHERE A AND B Single index seek (optimal) Index merge or intersection (slower)
WHERE A only Leftmost prefix works Direct index use
WHERE B only Cannot use (skips A) Direct index use
ORDER BY A, B Avoids filesort Cannot avoid filesort
Interview tip: When asked "how many indexes should this table have?", discuss composite indexes as a way to serve multiple query patterns with fewer total indexes, reducing write amplification. For example, an index on (tenant_id, user_id, status) can serve queries filtering on tenant_id alone, (tenant_id, user_id), and (tenant_id, user_id, status).

Common Interview Questions

  • What is a composite index and why does column order matter?
  • Explain the leftmost prefix rule with an example.
  • Given a query with equality and range filters, how would you order the columns in a composite index?
  • Can a composite index on (A, B, C) be used for a query that filters only on (B, C)?
  • What are the trade-offs between one composite index and multiple single-column indexes?
  • How would you design composite indexes for a multi-tenant SaaS application?
  • Can ORDER BY benefit from a composite index? Under what conditions?
  • How do you decide between high-cardinality-first and equality-before-range column ordering?

What to Skip

Depth to avoid in interviews:
  • Index merge intersection/union algorithms (MySQL's index_merge optimization details).
  • Skip scan index access path internals (Oracle/MySQL 8.0 feature).
  • Collation-specific sort order in composite index keys.
  • Bitmap index merge internals in warehouse engines.
