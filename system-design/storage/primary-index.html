Primary Index

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the primary index as the foundational data structure that enforces uniqueness on the primary key, organizes data as a B+ tree, and in many engines determines the physical storage order of rows.

Definition

A primary index is an index built on a table's primary key column(s). It guarantees uniqueness and a NOT NULL constraint on the indexed column(s). Internally it is organized as a B+ tree. In clustered storage engines (e.g., InnoDB), the primary index is the table -- leaf nodes hold the actual row data and the key order determines physical layout on disk. Every table has exactly one primary index (or one implicit one if none is declared).

Structure

B+ Tree (Primary Index on user_id) [50|100] / \ [10|30|40] [60|80|90] [110|120] / | \ / | \ / \ [5] [15] [35] [65] [85] [95] [115] [125] | | | | | | | | Row Row Row Row Row Row Row Row (PK=5) (PK=35) (PK=125) Leaf nodes contain the full row data (clustered). All keys are unique and NOT NULL.

Key Properties

Property Detail
Uniqueness Every key value appears at most once; duplicates are rejected on INSERT.
NOT NULL Primary key columns cannot contain NULL. The engine enforces this.
One per table A table can have only one primary key. This is a fundamental constraint.
B+ tree organization Internal nodes guide search; leaf nodes hold sorted key-value pairs (or full rows in clustered engines).
Physical ordering In clustered engines, row order on disk follows primary key order. Range scans are sequential I/O.
Implicit primary key If no primary key is defined, InnoDB creates a hidden 6-byte auto-increment row ID internally.

How It Works in Practice

Point Lookup

A lookup by primary key traverses from the B+ tree root down to a leaf in O(log N) page reads. In a clustered table the leaf is the data page, so no extra hop is needed.

SELECT * FROM users WHERE user_id = 42;

-- Execution path: root -> internal node -> leaf (data page)
-- Cost: ~3-4 page reads for a 10M-row table (tree depth ~3-4)

Range Scan

Because leaf nodes are linked in sorted order and physically co-located in clustered tables, a range query on the primary key benefits from sequential I/O -- one of the most important performance characteristics for time-series or append-only workloads.

SELECT * FROM orders WHERE order_id BETWEEN 1000 AND 2000;

-- Leaf nodes are sequentially linked; minimal random I/O.

Insert Path

Inserting a row requires finding the correct leaf position in the B+ tree (log N reads), writing the row into the leaf page, and potentially splitting the page if it is full. Page splits incur an extra random write.

Random I/O on inserts: If the primary key is a random UUID, each insert targets a different leaf page, causing heavy random I/O and poor buffer pool utilization. Use auto-increment or time-ordered keys for append-heavy workloads.

Clustered vs Heap Storage

Engine Primary Index Behavior Physical Order
InnoDB (MySQL) Clustered -- leaf nodes ARE data pages Rows stored in PK order
MyISAM (MySQL) Non-clustered -- leaf nodes point to heap rows Insertion order
PostgreSQL Heap table with separate B-tree index Insertion order
SQL Server Configurable -- clustered or non-clustered Depends on design

Primary Key Selection Considerations

  • Query pattern: Choose a key that appears frequently in WHERE clauses and JOINs.
  • Insert pattern: Monotonically increasing keys (auto-increment, snowflake IDs) avoid random I/O and page splits.
  • Size: Smaller keys (4-byte INT vs 36-byte UUID) mean more keys fit per B+ tree page, reducing tree depth and I/O.
  • Stability: The primary key should never change. Updating a primary key requires deleting and re-inserting the row across all indexes.
Interview tip: When asked "how would you design the primary key for a messaging table?", discuss trade-offs between auto-increment (sequential writes, single point of contention), UUID v4 (globally unique, terrible insert locality), UUID v7 / ULID (globally unique, time-ordered, best of both worlds), and snowflake IDs (distributed generation, time-ordered).

Common Interview Questions

  • What is a primary index and how does it differ from a unique index?
  • Why can a table have only one primary key?
  • How does InnoDB organize data using the primary index?
  • What happens when you insert a row with a duplicate primary key?
  • Why is a random UUID a bad choice for a primary key in InnoDB?
  • What is the hidden row ID InnoDB creates when no primary key is defined?
  • How does primary key choice affect insert throughput and page fragmentation?
  • Explain the difference between clustered and non-clustered primary indexes.

What to Skip

Depth to avoid in interviews:
  • Internal page split/merge algorithms (e.g., how sibling pointers are updated during a split).
  • Buffer pool LRU details specific to index page eviction.
  • Undo log structures for MVCC on primary index modifications.
  • Exact byte-level header format of InnoDB index pages.
