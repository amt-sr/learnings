Wide-Column Database

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What a wide-column database is: partition key + sort key + column families
  • How it differs from document and relational databases
  • Why Cassandra achieves massive write throughput using LSM trees
  • Data modeling with partition keys and clustering keys

Definition

A wide-column database stores data in tables with rows identified by a partition key and optionally a clustering key, where each row can have a variable number of columns. Unlike relational databases where rows have a fixed set of columns, wide-column databases allow rows to have different columns. Data is organized by partition keys (which determine data distribution across nodes) and clustering keys (which determine sort order within a partition).

Data Model: Keys

Key TypePurposeExample
Partition KeyDetermines which node(s) store the data. All rows with the same partition key are co-located.user_id — all orders for a user on the same node
Clustering Key (Sort Key)Determines sort order within a partition. Enables efficient range queries.order_date — orders sorted by date within a user's partition
Composite Partition KeyMultiple columns form the partition key for finer-grained distribution.(tenant_id, user_id) — distribute across more nodes
Critical rule: In Cassandra, you can only query by partition key (fast, single-node) or by partition key + clustering key range (fast, sorted scan). You CANNOT query by an arbitrary column unless you create a secondary index or a materialized view. This means data modeling is query-driven — you model your tables around how data will be accessed.

Query-Driven Data Modeling

Unlike relational databases where you model entities and write queries to join them, in Cassandra you model each query pattern as a separate table:

Relational approach: Cassandra approach: One "orders" table: Multiple query-specific tables: SELECT * FROM orders Table: orders_by_user WHERE user_id = 42; (PK: user_id, CK: order_date) SELECT * FROM orders Table: orders_by_date WHERE order_date > '2024-01-01' (PK: order_date, CK: order_id) SELECT * FROM orders Table: orders_by_product WHERE product_id = 7; (PK: product_id, CK: order_date) 1 table, 3 queries (using indexes) 3 tables, 3 queries (no indexes needed) In Cassandra: denormalize and duplicate data to make each query pattern a simple primary key lookup. Writes are cheap (append-only LSM tree), reads are optimized.

Why High Write Throughput

Wide-column databases achieve massive write throughput because:

  1. LSM tree storage engine — Writes are sequential appends (see LSM Tree). No in-place updates, no page splits.
  2. Append-only architecture — No read-before-write. Each write is a single sequential disk operation.
  3. No cross-partition transactions — Writes go to a single partition (single node). No distributed locking needed.
  4. Tunable consistency — Write acknowledged after W replicas (configurable). Lower W = faster writes.
  5. Log-structured replication — Replication via commit log shipping, efficient for high-throughput.
  Write throughput comparison:

  Cassandra:  100K-500K writes/sec per node (LSM tree, sequential I/O)
  PostgreSQL: 10K-50K   writes/sec per node (B+ tree, random I/O)
  MySQL:      20K-100K   writes/sec per node (InnoDB, random I/O)

  At cluster level with 100 nodes:
  Cassandra:  10M-50M writes/sec
  PostgreSQL: ~500K writes/sec (limited by replication overhead)

Common Interview Questions

  • "What is a wide-column database?"
    A database that stores data in tables where rows are identified by a partition key (determines node placement) and a clustering key (determines sort order within a partition). Each row can have different columns. Cassandra is the most popular example. Designed for massive write throughput using LSM tree storage and masterless, ring-based architecture.
  • "How does Cassandra data modeling differ from relational?"
    Relational: model entities, normalize, use JOINs. Cassandra: model queries — create one table per query pattern, denormalize data, use partition key for distribution and clustering key for sorting. You can only efficiently query by partition key or partition key + clustering key range. No ad-hoc queries without secondary indexes (which are expensive).
  • "Why can Cassandra handle more writes than MySQL?"
    LSM tree storage converts random writes to sequential appends (no page reads before writes, no page splits). Masterless architecture means no single write bottleneck. No cross-node transactions means writes go to a single partition on a single node. Tunable consistency means you can write to fewer replicas for speed.
  • "When would you choose Cassandra?"
    High write throughput (time-series, IoT telemetry, event logging, messaging), large datasets that don't fit a single node, need for linear horizontal scaling, and when your query patterns are known and predictable. Avoid when you need complex ad-hoc queries, strong ACID transactions, or multi-table JOINs.
