Write Ahead Log (WAL)

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand:

  • The WAL rule: write the log entry BEFORE writing the data page
  • How WAL enables atomicity and crash recovery
  • Why the WAL is append-only and why sequential writes are critical
  • The relationship between WAL and the redo log
  • fsync considerations: when must the WAL hit physical disk?
  • How InnoDB and PostgreSQL implement WAL differently

Definition

The Write-Ahead Log (WAL, also called the transaction log) is an append-only log file that records every modification made to database pages before those modifications are written to the actual data files. It is the fundamental mechanism that enables crash recovery: if the database crashes before dirty pages are flushed from the buffer pool, the WAL can replay the modifications to restore the database to a consistent state.

The WAL Rule

The Write-Ahead Logging Protocol: Before a modified data page is written to the data file on disk, the log record describing that modification MUST be written to the WAL on disk first. This is sometimes stated as: "WAL records must precede data page writes."
The WAL Rule in Action: Transaction: UPDATE users SET balance = balance - 100 WHERE id = 42 Step 1: Record the intention in the WAL ┌───────────────────────────────────────────┐ │ WAL Log Record (appended sequentially) │ │ LSN: 1024 │ │ Type: UPDATE │ │ Table: users │ │ Page: #42, Offset: 156 │ │ Before: balance = 500 │ │ After: balance = 400 │ │ Transaction ID: TXN-789 │ └───────────────────────────────────────────┘ │ ↓ fsync (MUST hit disk BEFORE commit) Step 2: Modify the data page in the buffer pool ┌───────────────────────────────────────────┐ │ Buffer Pool: Page #42 │ │ Record: (id=42, balance=400) ← modified │ │ Status: DIRTY │ └───────────────────────────────────────────┘ │ ↓ (later, by background thread — NOT blocking) Step 3: Flush dirty page to data file ┌───────────────────────────────────────────┐ │ Data File: Page #42 │ │ Record: (id=42, balance=400) ← persisted │ └───────────────────────────────────────────┘

Why WAL Works

Atomicity

A transaction either fully completes or not at all. Before acknowledging commit to the client, the database ensures the transaction's WAL records are durably on disk (via fsync). If the system crashes after the WAL is written but before data pages are flushed, the recovery process replays the WAL to complete the transaction.

Crash Recovery

After a crash, the database reads the WAL from the last checkpoint and replays any committed changes that were not yet flushed to data files:

  • Redo (forward): Replay committed WAL records to reconstruct dirty pages
  • Undo (backward): Roll back any uncommitted transactions using the undo log

Performance

The WAL is append-only, meaning writes are always sequential. Sequential writes are fast even on HDDs (100–200 MB/s) and extremely fast on SSDs. By contrast, updating data pages requires random I/O (seeking to the correct page on disk). WAL converts many random writes into a single sequential append.

Without WAL (random writes for every transaction): TXN 1: UPDATE page_42 → random write to page 42 TXN 2: UPDATE page_87 → random write to page 87 TXN 3: UPDATE page_15 → random write to page 15 TXN 4: UPDATE page_201 → random write to page 201 = 4 random disk seeks per 4 transactions With WAL (sequential append + deferred page writes): TXN 1: APPEND WAL record → sequential write TXN 2: APPEND WAL record → sequential write TXN 3: APPEND WAL record → sequential write TXN 4: APPEND WAL record → sequential write = 1 sequential disk write for all 4 transactions Page writes happen later, in batch, by background thread (and can be reordered for efficiency)

WAL and Redo Log Relationship

The WAL and redo log are closely related concepts:

ConceptDescription
WAL (protocol)The write-ahead logging rule: log before data. This is a protocol/constraint.
Redo log (data structure)The physical log file that implements the WAL protocol. Contains the actual byte-level changes to replay during recovery.

In practice, these terms are often used interchangeably. In MySQL InnoDB, the redo log IS the WAL implementation. In PostgreSQL, the WAL is the actual log file. The redo log section covers the physical log records in detail.

fsync Considerations

Durability requires that the WAL record is physically on disk before the transaction is committed. The fsync() system call forces data from the OS page cache to the physical disk:

Durability SettingBehaviorPerformanceRisk
fsync=ONfsync WAL on every commitSlowest (~1 fsync per commit)Fully durable
fsync_group_commitBatch fsyncs from multiple commitsFast (amortizes fsync cost)Fully durable
fsync=OFFLet OS flush WAL to disk eventuallyFastestRisk of data loss on OS crash
Interview tip: fsync costs ~1–10 ms depending on the storage device. At 10 ms per fsync with no group commit, you get ~100 TPS maximum. This is why group commit and battery-backed write caches (BBWC) are critical for high-transaction throughput. Modern SSDs with power-loss protection (PLP) can complete fsync in ~0.1–0.5 ms, enabling thousands of TPS.

PostgreSQL WAL Details

  • WAL files are 16 MB each, stored in pg_wal/
  • Each WAL record contains: LSN, transaction ID, modified page ID, before/after images (delta)
  • WAL is fsync'd based on wal_sync_method (fdatasync by default on Linux)
  • WAL recycling: old WAL files are reused once all checkpoints have moved past them

MySQL InnoDB WAL Details

  • Redo log files are circular (ib_logfile0, ib_logfile1, typically 48 MB–1 GB each)
  • Written to the redo log buffer in memory, flushed to log files on commit or every second
  • innodb_flush_log_at_trx_commit: 0=flush every second (risk), 1=flush on commit (safe), 2=flush to OS cache on commit (risk on OS crash)
  • Uses LSN (Log Sequence Number) to track position in the log

Common Interview Questions

  • "What is WAL and why is it important?"
    WAL (Write-Ahead Logging) is a protocol where every modification to a data page is first recorded in an append-only log file before the data page is written to disk. This ensures atomicity (transactions are all-or-nothing) and durability (committed data survives crashes). The WAL also improves performance by converting random page writes into sequential log appends.
  • "Why is WAL append-only?"
    Appending to the end of a file is a sequential write — the disk head (on HDD) does not seek, and the write completes at the disk's sequential throughput rate. On SSDs, sequential writes avoid write amplification from garbage collection. If the log were randomly written, it would be no faster than writing data pages directly.
  • "What happens if the database crashes after writing to the WAL but before flushing the data page?"
    During crash recovery, the database reads the WAL from the last checkpoint and replays (redoes) all committed changes that were not flushed to data files. The WAL contains the complete record of what each page should look like. The redo log is the data structure used for this replay.
  • "What is group commit and why does it matter?"
    Group commit batches multiple transactions' WAL records into a single fsync. Instead of one fsync per transaction (~10 ms each, limiting to 100 TPS), multiple transactions that commit within the same time window share one fsync, potentially achieving 10,000+ TPS. Both PostgreSQL and MySQL InnoDB implement group commit.
  • "What is the difference between WAL and redo log?"
    WAL is the protocol (write log before data). The redo log is the physical implementation — the file containing the actual log records used to replay changes during recovery. In MySQL, the redo log IS the WAL. In PostgreSQL, the WAL files serve as the redo log. The terms overlap significantly.
