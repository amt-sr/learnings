Covering Index

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand how a covering index satisfies a query entirely from the index B+ tree without requiring a bookmark lookup back to the data pages, how to identify covering index opportunities via EXPLAIN, and the trade-off between index size and read performance.

Definition

A covering index is an index that contains all columns required by a query -- including columns in the SELECT list, WHERE clause, ORDER BY, and GROUP BY. Because the index alone satisfies the query, the database engine does not need to perform a bookmark lookup (also called a key lookup or table fetch) to retrieve the full row. This turns an otherwise two-step lookup into a single B+ tree traversal.

Creating Covering Indexes

Using a Composite Index

The simplest way to create a covering index is to include all needed columns as part of the index key.

-- Query:
SELECT user_id, status, created_at
FROM orders
WHERE user_id = 42 AND status = 'active';

-- Covering composite index:
CREATE INDEX idx_orders_user_status_created
  ON orders(user_id, status, created_at);

-- All three columns are in the index key.
-- Query satisfied entirely from the index.

Using INCLUDE (Non-Key Columns)

Most modern databases support an INCLUDE clause that adds columns to the index leaf without making them part of the key. Included columns are not used for index ordering or seek but are stored in the leaf for covering purposes.

-- SQL Server / PostgreSQL 11+
CREATE INDEX idx_orders_user_status
  ON orders(user_id, status)
  INCLUDE (created_at, amount);

-- Key columns: user_id, status (used for seek + ordering)
-- Included columns: created_at, amount (stored in leaf, not in key)

-- Benefits of INCLUDE:
-- 1. Smaller key -> more entries per page -> shallower tree
-- 2. Included columns don't affect sort order or uniqueness
-- 3. Can include wide columns (VARCHAR, TEXT) without bloating the key
Engine INCLUDE Clause Support Notes
SQL Server Yes Original inventor of INCLUDE syntax. Widely used.
PostgreSQL 11+ Yes Supported for B-tree indexes. Uses "covering indexes" terminology.
MySQL / InnoDB No Must include all columns in the key (or use a full composite index)
SQLite Yes Supported since 3.9.0.

Identifying Covering Index Opportunities

Using EXPLAIN

-- MySQL
EXPLAIN SELECT name, email FROM users WHERE email = 'jane@example.com';

-- Look for: Extra: Using index
-- "Using index" means the query was satisfied entirely from an index (covering).

-- PostgreSQL
EXPLAIN ANALYZE SELECT name, email FROM users WHERE email = 'jane@example.com';

-- Look for: "Index Only Scan using idx_users_email on users"
-- vs "Index Scan using idx_users_email on users" (non-covering, needs heap fetch)
-- vs "Bitmap Heap Scan" (requires table access)
EXPLAIN Output MySQL PostgreSQL Meaning
Covering (index-only) Using index Index Only Scan Query satisfied from index alone
Non-covering (blank) or Using where Index Scan Requires table/heap access (bookmark lookup)
Full table scan ALL Seq Scan No index used at all

Practical guidance: Covering indexes are most impactful when (a) the query is hot (executed frequently), (b) the table is large (bookmark lookup is expensive), and (c) the additional columns are small (index bloat is minimal). For write-heavy tables with many indexes already, adding more columns to an index may worsen write performance more than it improves reads.

Common Patterns

Paginated Listing Queries

-- Common API pattern: paginated user list with sorting
SELECT user_id, name, email, created_at
FROM users
WHERE status = 'active'
ORDER BY created_at DESC
LIMIT 50 OFFSET 0;

-- Covering index:
CREATE INDEX idx_users_status_created
  ON users(status, created_at DESC)
  INCLUDE (name, email);

-- Satisfies WHERE, ORDER BY, and SELECT entirely from the index.
-- No bookmark lookup for any of the 50 rows.

Aggregate Queries

-- Dashboard query: count active users per day
SELECT DATE(created_at) AS day, COUNT(*)
FROM users
WHERE status = 'active'
GROUP BY DATE(created_at);

-- Covering index:
CREATE INDEX idx_users_status_created
  ON users(status, created_at);

-- WHERE status + GROUP BY created_at are both in the index.
-- No heap access needed.
