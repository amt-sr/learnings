Read Uncommitted

Difficulty: ★★★☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What Read Uncommitted isolation allows and what it prevents
  • The dirty read anomaly in detail
  • When Read Uncommitted might be useful (rarely)
  • Why no major production database recommends it

Definition

Read Uncommitted is the lowest SQL isolation level. A transaction at this level can read changes made by other transactions that have not yet been committed. This means a transaction may observe data that could later be rolled back, leading to a dirty read.

The Dirty Read Problem

Transaction A Transaction B ──────────────────────────────────────────────────────── BEGIN; UPDATE accounts SET balance = balance - 500 WHERE id = 1; BEGIN; -- balance is now 500 (was 1000) SELECT balance FROM accounts WHERE id = 1; -- reads 500 (UNCOMMITTED!) ROLLBACK; -- balance reverts to 1000 -- Transaction B used value 500 -- which never actually existed! COMMIT;
Danger: Transaction B read a value (500) that was never committed to the database. Transaction A rolled back, reverting the balance to 1000. Transaction B made business decisions based on phantom data that never existed.

When Is Read Uncommitted Useful?

In practice, almost never. The few legitimate use cases include:

  • Approximate aggregations: When you need a rough count or sum and do not care about exact numbers (e.g., analytics dashboards with tolerance for inaccuracy)
  • Long-running reports: When you want to avoid holding locks for the duration of a report query
  • Checking if a row exists: If the existence check itself has no business consequence
-- Read Uncommitted is rarely set explicitly
-- Most databases do not even support it or warn against it
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- PostgreSQL ignores this; it treats it as Read Committed
-- SQL Server supports it via WITH (NOLOCK) hint
SELECT COUNT(*) FROM orders WITH (NOLOCK);

-- MySQL supports it
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

NOLOCK Hint in SQL Server

The most common real-world encounter with Read Uncommitted is the WITH (NOLOCK) hint in SQL Server, which is equivalent to Read Uncommitted for that specific query:

-- SQL Server: NOLOCK = Read Uncommitted for this query
SELECT * FROM products WITH (NOLOCK);

-- This avoids blocking but returns dirty data
-- The query can read:
--   - Rows being modified but not yet committed
--   - Rows that may be rolled back
--   - In rare cases, rows twice or not at all (page splits)
Anti-pattern: Using WITH (NOLOCK) as a default for all queries to "improve performance" is a common mistake. It introduces data corruption risk that is extremely hard to debug. The performance gain from avoiding read locks is almost never worth the correctness risk.

Why Databases Discourage It

Practical takeaway: If you need better performance and can tolerate some inaccuracy, use Read Committed with a shorter transaction duration, or use application-level caching. Read Uncommitted is almost never the right answer.

Common Interview Questions

  • "What is a dirty read?"
    A dirty read occurs when a transaction reads data written by another uncommitted transaction. If that transaction rolls back, the data never existed, but the reading transaction already used it. See: Dirty Read
  • "When would you use Read Uncommitted?"
    Almost never in production. The only valid case is for approximate queries where you can tolerate seeing uncommitted changes, such as showing a live counter on a dashboard where perfect accuracy is not required. Even then, most teams prefer Read Committed with shorter transactions.
  • "Does PostgreSQL support Read Uncommitted?"
    No. PostgreSQL silently upgrades it to Read Committed. The PostgreSQL developers considered Read Uncommitted too dangerous to support, and since they use MVCC for all reads, the performance justification does not apply.
