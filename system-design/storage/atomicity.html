3.4.2 Atomicity

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

  • Define atomicity precisely as "all or nothing" semantics for transaction processing
  • Explain how atomicity is implemented via Write-Ahead Logging and undo logs
  • Differentiate between commit (all changes persist) and rollback (all changes reverted)
  • Understand atomicity in distributed systems through Two-Phase Commit (2PC)
  • Identify atomicity failure modes and their consequences in production systems

Definition

Atomicity guarantees that a transaction is treated as a single, indivisible unit of work. Either all operations within the transaction are applied to the database (commit), or none of them are (rollback). There is no intermediate state where some operations succeed and others fail. This prevents the database from entering an inconsistent state due to partial updates.

Think of it this way: Atomicity is the database's guarantee that your money transfer either fully completes (debit A + credit B) or does not happen at all. You will never be in a state where money was debited from account A but never credited to account B.

How Atomicity Works: The Mechanism

Transaction: Transfer $100 from Account A to Account B ──────────────────────────────────────────────────────── Step 1: Write BEGIN record to WAL WAL: [LSN 001: BEGIN TXN-42] Step 2: Debit Account A (write to buffer pool + WAL) WAL: [LSN 002: TXN-42 | UPDATE accounts SET balance=900 WHERE id='A' | before: 1000, after: 900] Step 3: Credit Account B (write to buffer pool + WAL) WAL: [LSN 003: TXN-42 | UPDATE accounts SET balance=1100 WHERE id='B' | before: 1000, after: 1100] Step 4a: COMMIT -- write commit record + fsync WAL WAL: [LSN 004: COMMIT TXN-42] → ACK to client. All changes are permanent. OR Step 4b: ROLLBACK -- reverse all changes using before-images Read WAL backward from LSN 003: LSN 003: revert B's balance to 1000 (before-image) LSN 002: revert A's balance to 1000 (before-image) WAL: [LSN 005: ROLLBACK TXN-42] → ACK to client. Database unchanged from before TXN.

The Write-Ahead Log as the Atomicity Engine

The WAL is the fundamental mechanism that enables atomicity. Before any modification is written to the actual data pages, the corresponding log record (containing both before-image and after-image) is written to the WAL. This creates a complete audit trail that can be used for both redo (replaying after-images) and undo (reverting before-images).

ComponentRole in Atomicity
Before-image in WALStores old values needed to reverse changes during rollback or crash recovery
After-image in WALStores new values needed to replay changes during crash recovery
Commit recordMarks the decision point: before this record, transaction can be undone; after, it is permanent
Prev-LSN chainLinks all records of a transaction for sequential undo during rollback
fsync on commitEnsures the commit decision is durable before acknowledging the client

Commit vs. Rollback

┌────────────────────────────────────────────────────────────────────┐ │ TRANSACTION LIFECYCLE │ ├────────────────────────────────────────────────────────────────────┤ │ │ │ BEGIN ──── Execute Operations ──── Decision Point │ │ │ │ │ ┌─────────┴─────────┐ │ │ │ │ │ │ ▼ ▼ │ │ COMMIT ROLLBACK │ │ │ │ │ │ ▼ ▼ │ │ fsync WAL to disk Reverse all changes │ │ Write commit record using before-images │ │ ACK to client Write rollback record │ │ ACK to client │ │ │ │ │ │ ▼ ▼ │ │ All changes PERSIST All changes DISCARDED │ │ │ └────────────────────────────────────────────────────────────────────┘ CRASH SCENARIOS: ──────────────── Crash before commit → Recovery finds TXN in Active TXN list → UNDO all changes Crash during commit → If commit record is NOT on disk → UNDO all changes Crash during commit → If commit record IS on disk → REDO any unflushed changes Crash after commit → REDO any changes not yet flushed to data pages

Implicit vs. Explicit Rollback

Rollback can occur in several ways:

  • Explicit rollback: Application calls ROLLBACK due to business logic error
  • Constraint violation: Database detects a rule violation (unique key, FK, check constraint) and automatically rolls back the statement or transaction
  • Deadlock detection: Database selects one transaction as the deadlock victim and rolls it back
  • Timeout: Transaction exceeds its configured timeout (e.g., statement_timeout in PostgreSQL) and is automatically rolled back
  • Crash recovery: After restart, uncommitted transactions are rolled back using WAL undo
Performance note: Rollback is expensive. Long-running transactions that roll back have to undo potentially thousands of changes by reading the WAL backward and applying before-images. This is why it is critical to keep transactions short and avoid nested operations that increase the rollback window.

Atomicity in Distributed Systems: Two-Phase Commit (2PC)

When a transaction spans multiple nodes or databases, atomicity requires coordination. Two-Phase Commit (2PC) is the standard protocol:

Phase 1: PREPARE (Voting Phase) ────────────────────────────────── Coordinator Participants │ │ │──── PREPARE TXN-42 ─────►│ Node A: Write to WAL, vote YES │──── PREPARE TXN-42 ─────►│ Node B: Write to WAL, vote YES │ │ │◄─── VOTE_COMMIT (A) ─────│ │◄─── VOTE_COMMIT (B) ─────│ │ │ Phase 2: COMMIT (Decision Phase) ───────────────────────────────── │ │ │ All votes = COMMIT │ │ │ │──── COMMIT TXN-42 ──────►│ Node A: Write commit, fsync, ACK │──── COMMIT TXN-42 ──────►│ Node B: Write commit, fsync, ACK │ │ │◄─── ACK (A) ─────────────│ │◄─── ACK (B) ─────────────│ │ │ IF ANY VOTE = ABORT: ───────────────────── │──── ROLLBACK TXN-42 ────►│ Node A: Undo changes │──── ROLLBACK TXN-42 ────►│ Node B: Undo changes

2PC Failure Scenarios

FailureWhat HappensRecovery
Participant crashes before votingCoordinator times out, aborts transactionParticipant recovers, finds no outcome, queries coordinator
Participant crashes after voting YESCoordinator may or may not have decidedParticipant must block until coordinator outcome is known
Coordinator crashes after Phase 1Participants have voted but no decision was sentParticipants block indefinitely until coordinator recovers (blocking problem)
Network partition during Phase 2Some participants commit, others do notCoordinator retries until all participants ACK
The 2PC blocking problem: A participant that voted YES but does not receive the coordinator's decision must block, holding locks and resources, until the coordinator recovers. This is a fundamental limitation of 2PC. Systems like Spanner use Paxos groups to eliminate this blocking, replacing 2PC with a consensus-based approach where any replica can act as coordinator.

Alternatives to 2PC in Distributed Atomicity

ApproachMechanismTrade-off
Saga patternSequence of local transactions with compensating actionsNo distributed locks, but provides only semantic atomicity (requires compensation logic)
Outbox patternWrite to local table + outbox table atomically, then publish asynchronouslyEventual atomicity across services, requires a poller or CDC stream
Paxos/Raft-based commitConsensus protocol replaces coordinator, eliminating single point of failureHigher latency (multiple round trips), but no blocking problem
Best-effort 1PCOptimistic single-phase commit with conflict detectionVery fast when conflicts are rare, but requires full rollback on conflict
Interview tip for L6+: At Staff level, expect to discuss the Saga pattern and Outbox pattern as pragmatic alternatives to 2PC in microservices architectures. Be able to explain when semantic atomicity (sagas) is sufficient vs. when true atomicity (2PC or consensus) is required (e.g., financial transfers, inventory management).

Atomicity Failure Modes in Production

Common production issues:
  • Orphaned transactions: Application crashes without sending COMMIT or ROLLBACK. Database holds locks indefinitely. Mitigation: transaction timeouts, connection pool cleanup.
  • Commit but no ACK: Transaction commits successfully but network failure prevents ACK from reaching client. Client retries, potentially causing duplicate operations. Mitigation: idempotency keys.
  • Partial 2PC commit: Coordinator sends COMMIT to some participants but crashes before reaching all. System is in an inconsistent state until coordinator recovers. Mitigation: use consensus-based coordination (Raft).

Common Interview Questions

  1. What does atomicity mean? Give me a concrete example where atomicity prevents data corruption.
  2. How does the database guarantee atomicity? Walk me through the mechanism.
  3. What is the role of the before-image in the WAL for atomicity?
  4. What happens to in-flight transactions when the database crashes?
  5. Explain Two-Phase Commit. What is the blocking problem?
  6. How would you implement atomicity across two different microservices that use separate databases?
  7. What is the Saga pattern? When is it appropriate vs. 2PC?
  8. Why is rollback expensive? How do you minimize rollback cost in a high-throughput system?
  9. Design a money transfer system between two banks. How do you ensure atomicity?
  10. What happens if the coordinator crashes in 2PC? How do modern systems solve this?

What to Skip

In an interview setting, skip:
  • Exact WAL record byte layouts for specific databases
  • Three-Phase Commit (3PC) -- it is theoretically interesting but never used in production
  • Savepoints and nested transaction syntax -- rarely used in production systems
  • Low-level undo log implementation details beyond the concept of before-images
  • Specific ORM transaction API details (e.g., SQLAlchemy session.commit())
