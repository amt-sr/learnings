Document Database

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What a document database is: stores self-contained documents (typically JSON/BSON)
  • Flexible/dynamic schema vs rigid relational tables
  • Embedding vs referencing: when to nest data vs normalize
  • When document databases are the right choice

Definition

A document database stores data as documents — typically JSON, BSON, or XML objects — in collections rather than tables. Each document is self-contained, can have a different structure (flexible schema), and can contain nested objects and arrays. Documents are accessed by a unique document ID, but secondary indexes enable querying by any field.

Key Characteristics

CharacteristicDetail
Flexible schemaDocuments in the same collection can have different fields. New fields can be added to individual documents without altering a schema.
Nested documentsDocuments can contain arrays and sub-documents. Related data is often embedded in a single document, eliminating JOINs.
Document IDEach document has a unique _id (typically auto-generated ObjectId or user-specified). Primary access path.
Secondary indexesIndexes on any field enable querying by non-ID fields. Unlike pure key-value stores.
Rich query languageFilter, sort, project, aggregate — more powerful than KV but less powerful than SQL.

Embedding vs Referencing

This is the most important design decision in document databases. When to embed related data in a single document vs reference other documents.
Embedding (denormalize)Referencing (normalize)
PatternNested objects/arrays within documentStore ID reference to another document
ReadsFast — single document readSlower — multiple reads or JOIN ($lookup)
WritesSlower if updating embedded data (must update parent)Fast — update only the referenced document
Data consistencyStrong — atomic single-document updateWeaker — multi-document updates need transactions
Document sizeCan grow large (16 MB MongoDB limit)Documents stay small
Use when1:1 or 1:few relationship, data always read together1:many (unbounded), data updated independently
Embedding (good): { "user_id": 42, "name": "Alice", "addresses": [ {"type": "home", "street": "123 Main", "city": "NYC"}, {"type": "work", "street": "456 Oak", "city": "SF"} ] } → User always read with addresses. 1:2 relationship. Perfect for embedding. Referencing (good): posts collection: { "_id": 1, "author_id": 42, "title": "Hello", "body": "..." } { "_id": 2, "author_id": 42, "title": "World", "body": "..." } ... thousands of posts per user → 1:N (unbounded) relationship. Cannot embed (16 MB limit). → Reference author_id, query separately for user info.

MongoDB

MongoDB is the dominant document database. Key features for interviews:

  Basic CRUD:
  db.users.insertOne({ name: "Alice", email: "a@b.com" })
  db.users.findOne({ name: "Alice" })
  db.users.updateOne({ name: "Alice" }, { $set: { age: 30 } })
  db.users.deleteOne({ name: "Alice" })

  Querying (richer than KV):
  db.users.find({ age: { $gte: 25 }, city: "NYC" }).sort({ name: 1 }).limit(10)

  Aggregation (like SQL GROUP BY):
  db.orders.aggregate([
    { $match: { status: "completed" } },
    { $group: { _id: "$user_id", total: { $sum: "$amount" } } },
    { $sort: { total: -1 } },
    { $limit: 10 }
  ])

  Indexes:
  db.users.createIndex({ email: 1 })          // secondary index
  db.users.createIndex({ name: 1, age: -1 })   // compound index

Common Interview Questions

  • "When would you use a document database?"
    When your data is naturally document-shaped (user profiles with preferences, product catalogs with attributes), when the schema evolves frequently, when reads are mostly by ID or a few indexed fields, and when related data is usually accessed together (good for embedding). Avoid when you have complex many-to-many relationships or need multi-table JOINs.
  • "Explain embedding vs referencing in MongoDB."
    Embedding nests related data in a single document (good for 1:1 and 1:few, atomic updates, fast reads). Referencing stores IDs pointing to other documents (needed for 1:many unbounded relationships, independent updates, smaller documents). The key tradeoff: embedding avoids JOINs but risks document size limits and update anomalies.
  • "What is the 16 MB document limit in MongoDB?"
    BSON documents cannot exceed 16 MB. This limits how much you can embed. If a user has 10,000 comments, you cannot embed them all. You must reference (separate comments collection) or paginate. This is a fundamental constraint that drives the embedding vs referencing decision.
  • "How does indexing work in MongoDB?"
    MongoDB uses B+ trees for indexes (same as relational databases). You can create indexes on any field, including nested fields and array fields. Compound indexes support queries on multiple fields. The query planner selects the optimal index. Missing indexes lead to collection scans (COLLSCAN) — always check with explain().
