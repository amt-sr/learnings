Relational Database

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What a relational database is: tables, rows, columns, SQL, schemas
  • ACID properties and why they matter
  • Normalization and joins
  • When relational databases are the right (or wrong) choice

Definition

A relational database organizes data into tables (relations) with defined relationships between them, accessed via SQL (Structured Query Language). Each table has a fixed schema (columns and types), rows represent individual records, and keys (primary, foreign) enforce relationships and integrity constraints.

ACID Properties

ACID is the fundamental guarantee of relational databases. It ensures that transactions are processed reliably and consistently, even in the face of crashes, power failures, and concurrent access.
PropertyMeaningExample
AtomicityA transaction is all-or-nothing. Either all operations succeed, or none do.Transfer $100: debit account A AND credit account B. If credit fails, debit is rolled back.
ConsistencyTransaction moves the database from one valid state to another. All constraints (FK, unique, check) are enforced.Cannot create an order for a user that doesn't exist (FK constraint).
IsolationConcurrent transactions don't interfere with each other. Each transaction sees a consistent snapshot.Two users buying the last item: only one succeeds (row-level lock).
DurabilityOnce a transaction commits, it persists even through crashes. Data is written to the WAL before acknowledging commit.After COMMIT, data survives server reboot.

Normalization

Normalization reduces data redundancy by splitting data into multiple related tables. Key normal forms:

Normal FormRuleProblem Solved
1NFEach cell holds atomic values. No repeating groups."phone: 555-1234, 555-5678" → separate rows or separate phone table
2NF1NF + all non-key columns depend on the entire primary key.Order(item_id, item_name, qty): item_name depends on item_id alone, not the full key
3NF2NF + no transitive dependencies. Non-key columns depend only on the primary key.User(zip, city): city depends on zip, not user_id → move to a zip_codes table
In practice: Most production databases use 3NF. However, analytical systems (data warehouses) intentionally denormalize for faster reads — see SQL vs NoSQL.

Joins

-- INNER JOIN: users who have placed orders
SELECT u.username, o.total
FROM users u
INNER JOIN orders o ON u.id = o.user_id;

-- LEFT JOIN: all users, with their orders (NULL if no orders)
SELECT u.username, o.total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;

-- Self-JOIN: manager-employee relationship
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- JOIN performance: indexed join columns are critical
-- Without index on orders.user_id: full table scan O(n)
-- With index on orders.user_id: index lookup O(log n) per user

Common Interview Questions

  • "Explain ACID properties."
    Atomicity: transactions are all-or-nothing. Consistency: database moves between valid states. Isolation: concurrent transactions don't interfere. Durability: committed data survives crashes. All four are guaranteed by the database engine using WAL, locking, and MVCC.
  • "When would you NOT use a relational database?"
    When your data doesn't fit a fixed schema (rapidly evolving attributes), when you need extremely high write throughput (time-series, IoT), when you need horizontal scaling beyond a single node, or when your access patterns are simple key-value lookups (caching). See: SQL vs NoSQL
  • "How does a JOIN work under the hood?"
    Nested Loop Join: for each row in table A, scan matching rows in table B. O(n*m). Hash Join: build hash table on smaller table, probe with larger. O(n+m). Merge Join: both tables sorted by join key. O(n+m). The optimizer chooses based on table sizes, indexes, and statistics.
  • "What is the difference between a clustered and non-clustered index?"
    Clustered index: determines physical row order. The table IS the index. One per table (usually the primary key). Non-clustered index: separate structure pointing to rows. Many per table allowed. Covering indexes (non-clustered with all needed columns) avoid the row lookup entirely.

What to Skip

  • SQL syntax details (the interviewer expects you to know basic SQL)
  • BCNF, 4NF, 5NF (3NF is enough)
  • Stored procedures, triggers (implementation details)
