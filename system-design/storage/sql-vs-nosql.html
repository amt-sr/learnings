SQL vs NoSQL

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • Comprehensive comparison of SQL (relational) and NoSQL databases
  • The NoSQL landscape: key-value, document, wide-column, graph
  • When to choose SQL vs NoSQL (the most common interview question)
  • Why the answer is almost always "both" (polyglot persistence)

Overview

SQL databases (relational databases) organize data into tables with fixed schemas and use SQL for querying. NoSQL is an umbrella term for non-relational databases, including key-value, document, wide-column, and graph databases. The choice depends on your data model, access patterns, scalability requirements, and consistency needs.

When to Choose SQL

Choose SQL when: Data has complex relationships (many-to-many), you need ACID guarantees (financial data, orders), your queries are complex (multi-table JOINs, aggregations), your team knows SQL well, and your data fits on a single node or a few read replicas.
  • Financial systems — banking, payments, accounting (ACID is non-negotiable)
  • E-commerce orders — orders, products, inventory (complex relationships, ACID)
  • ERP / CRM — structured business data with many relationships
  • Analytics / reporting — complex ad-hoc SQL queries, aggregations, JOINs
  • SaaS backends — when data model is well-understood and stable

When to Choose NoSQL

Choose NoSQL when: You need horizontal scaling beyond a single node, write throughput is very high, the schema is evolving rapidly, data is naturally suited to a specific NoSQL model (documents, time-series, graphs), or you need tunable consistency for availability.
  • Real-time analytics — clickstream, event tracking (Cassandra, high write volume)
  • Content platforms — articles, user profiles with variable attributes (MongoDB)
  • IoT / sensor data — millions of data points per second (InfluxDB, Cassandra)
  • Caching layer — session storage, query caching (Redis, Memcached)
  • Social graphs — friend relationships, recommendations (Neo4j)
  • Rapid prototyping — schema not yet defined, iterating quickly (MongoDB)

The Real Answer: Polyglot Persistence

In production systems, you almost always use BOTH SQL and NoSQL. Most large systems use a polyglot persistence approach: SQL for transactional/core data, NoSQL for specialized workloads.

Common Mistakes

Mistake 1: "NoSQL means no SQL" — NoSQL means "Not Only SQL", not "never use SQL". Most NoSQL databases have query languages inspired by SQL.
Mistake 2: Choosing NoSQL to "avoid schema design" — NoSQL still requires data modeling. In fact, Cassandra data modeling is arguably harder than relational because you must model queries, not entities.
Mistake 3: "NoSQL doesn't have consistency" — Many NoSQL databases offer strong consistency. MongoDB supports multi-document ACID transactions. DynamoDB offers strong read consistency. The tradeoff is tunable, not binary.
Mistake 4: Using NoSQL for everything — avoid the hammer-and-nail trap. Use the right tool for the job. SQL for transactional core, NoSQL for specialized workloads.

Common Interview Questions

  • "SQL vs NoSQL — when would you choose each?"
    SQL for relational data with complex relationships, ACID transactions, complex ad-hoc queries, and when the team knows SQL. NoSQL for horizontal scaling, high write throughput, flexible schemas, and when the access pattern maps to a specific NoSQL model (KV cache, document store, time-series, etc.). In practice, most systems use both — SQL for core transactional data, NoSQL for caching, search, time-series, or other specialized workloads.
  • "What is NoSQL?"
    NoSQL ("Not Only SQL") is an umbrella term for non-relational databases that trade some relational features (ACID, joins, SQL) for benefits like horizontal scalability, flexible schemas, and high write throughput. Includes key-value stores (Redis), document databases (MongoDB), wide-column stores (Cassandra), and graph databases (Neo4j).
  • "Design Twitter/X — which database would you use?"
    Polyglot: PostgreSQL for user accounts and relationships (ACID), Redis for timeline caching (home feed), Cassandra for tweets and activity events (massive write throughput, time-ordered), Elasticsearch for search. Each workload uses the best-fit database.
  • "Can NoSQL databases handle transactions?"
    It depends. Single-entity (single document, single key) operations are atomic in most NoSQL databases. MongoDB 4.0+ supports multi-document ACID transactions. Cassandra offers lightweight transactions (Paxos-based) for per-partition operations. But cross-partition, cross-database distributed transactions are still limited compared to SQL databases.
  • "What is the CAP theorem and how does it relate to SQL vs NoSQL?"
    CAP: a distributed system can guarantee at most 2 of 3: Consistency, Availability, Partition tolerance. SQL databases typically choose consistency (CP). Many NoSQL databases choose availability (AP) with eventual consistency (Cassandra, DynamoDB). But this is a simplification — modern systems often offer tunable consistency levels. See: system design references on CAP theorem.

What to Skip

  • Multi-model databases (ArangoDB, OrientDB) — niche
  • Base vs BASE semantics details (beyond eventual vs strong)
