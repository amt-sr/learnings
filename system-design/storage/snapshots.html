Snapshots

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand what a snapshot is in the context of MVCC, how PostgreSQL constructs and assigns snapshots, why read-only transactions benefit from consistent snapshots, and the dangers of long-running transactions holding snapshots open.

Definition

A snapshot in MVCC is a logical point-in-time view of the database that answers the question: "which transactions are visible to me?" When a transaction (or individual statement, depending on isolation level) begins, the database captures a snapshot that records which transactions had committed at that moment. Every subsequent read by this transaction uses the snapshot to filter row versions—seeing only versions created by committed transactions and ignoring versions from transactions that were still in-progress or had not yet started.

Key insight: A snapshot is not a physical copy of the database. It is a small metadata structure that describes a visibility horizon—a set of transaction IDs that the transaction considers committed. The actual data is read from the version chains; the snapshot merely determines which version in each chain is selected.

PostgreSQL Snapshot Structure

PostgreSQL represents a snapshot as three values:

Component Description
xmin The lowest (oldest) active transaction ID at snapshot time. All transactions with IDs less than xmin are guaranteed to be either committed or aborted—their status is known.
xmax The first transaction ID that is not yet started at snapshot time. All transactions with IDs greater than or equal to xmax are invisible (they started after the snapshot).
xip[] array A sorted array of transaction IDs that were in-progress when the snapshot was taken. These fall in the range [xmin, xmax) and their outcome (commit/abort) was unknown at snapshot time.

Consistent Snapshot for Read-Only Transactions

In REPEATABLE READ and SERIALIZABLE isolation levels, PostgreSQL takes a single snapshot at the start of the transaction's first query. All subsequent statements within that transaction use the same snapshot, guaranteeing a consistent view of the database throughout the transaction.

  • Read Committed: A new snapshot is taken at the start of each statement. This means a transaction can see changes committed by other transactions between its statements.
  • Repeatable Read: A single snapshot at first statement. The transaction sees a frozen view of the database, never seeing changes from concurrent commits.
  • Serializable: Same snapshot mechanism as Repeatable Read, plus additional predicate locking to detect serialization anomalies.
Why this matters for read-heavy workloads: A long-running analytics query under Repeatable Read takes a snapshot once and then reads a consistent dataset, even while thousands of concurrent writes are streaming in. The snapshot ensures the query sees no partial updates.

Long-Running Transactions and Snapshot Too Old

A snapshot pins old row versions in the database: as long as any active snapshot might need to see a particular version, that version cannot be garbage-collected. This creates a direct conflict between long-running transactions and storage reclamation.

Timeline of danger: T1 (xmin=100): BEGIN; SELECT * FROM large_table; │ (query runs for 45 minutes) │ T2 (xmin=110): UPDATE users SET status='active'; T3 (xmin=111): UPDATE users SET status='active'; ... T99 (xmin=199): UPDATE users SET status='active'; │ │ VACUUM cannot reclaim any old versions of "users" rows │ because T1's snapshot (xmin=100) might still need them. │ ▼ Table bloats. Undo log grows. Performance degrades. In PostgreSQL: no "snapshot too old" error by default, but VACUUM is starved, table grows unbounded. In Oracle: ORA-01555: snapshot too old (undo overwritten) because undo segments are finite and get recycled.
Operational impact: A single forgotten BEGIN without COMMIT or ROLLBACK can prevent vacuum across the entire database. In production systems, set idle_in_transaction_session_timeout and monitor pg_stat_activity for long-held snapshots.

Snapshot Across Different Systems

System Snapshot Scope Retention Mechanism Failure Mode
PostgreSQL Per-statement (RC) or per-transaction (RR/S) Active snapshots prevent VACUUM from reclaiming dead tuples Table bloat, transaction ID wraparound
MySQL InnoDB Per-transaction in RC/RR Undo log segments; innodb_undo_log_truncate ORA-01555 equivalent (rollback segment overrun)
Oracle Per-statement (RC) or per-transaction (SERIALIZABLE) Undo tablespace with undo_retention setting ORA-01555: snapshot too old
CockroachDB Per-statement (RC) or per-transaction (SERIALIZABLE) GC threshold based on oldest active timestamp Similar bloat if GC is stalled

Common Interview Questions

  • What is a snapshot in MVCC? Is it a physical copy of the data?
  • Explain the three components of a PostgreSQL snapshot (xmin, xmax, xip).
  • Why does Read Committed take a new snapshot per statement while Repeatable Read takes one per transaction? What are the trade-offs?
  • What happens to VACUUM when a long-running transaction holds a snapshot open?
  • How do you detect and prevent snapshot-holding transactions in production?
  • What is ORA-01555 "snapshot too old"? Why doesn't PostgreSQL throw this error by default?
  • Design a system to take consistent snapshots across a distributed database. What are the challenges?
