Garbage Collection

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand why MVCC garbage collection is necessary, how PostgreSQL's VACUUM determines which row versions are reclaimable, the mechanics of autovacuum, the concept of dead tuple accumulation and table bloat, and the impact on query and write performance.

Definition

MVCC garbage collection is the process of reclaiming storage occupied by old row versions that are no longer visible to any active transaction. In PostgreSQL, this process is called VACUUM. Because MVCC never overwrites rows in place, every UPDATE and DELETE leaves behind a dead tuple—an old version that may still be needed by some concurrent transaction. Once no transaction can possibly need a given version, vacuum removes it and marks the space as reusable for future inserts.

Key insight: MVCC trades write amplification for read concurrency. The cost of never overwriting in place is accumulated garbage. Without effective garbage collection, every database table grows without bound regardless of the logical data size.

Why Old Versions Need Cleanup

Every UPDATE in PostgreSQL is semantically a DELETE + INSERT:

  • The old tuple version gets xmax set to the updating transaction's ID.
  • A new tuple version is created with the new data and a fresh xmin.
  • The old version remains on the page as a dead tuple.

Over time, tables accumulate dead tuples proportional to their write rate. Without vacuum, these dead tuples:

  • Consume disk space (table bloat).
  • Slow down sequential scans (reading past dead tuples).
  • Waste buffer pool space (caching dead tuples instead of live data).
  • Prevent reuse of space for HOT (Heap-Only Tuple) updates.
  • Delay transaction ID wraparound prevention.

How Vacuum Determines Reclaimable Versions

VACUUM uses the visibility horizon—the minimum of all active snapshot xmin values across the cluster. Any tuple version whose xmax (deleting transaction) is older than this horizon is guaranteed invisible to all current and future transactions:

Active snapshots: T_oldest: xmin = 150 (long-running analytics query) T_current: xmin = 300 T_recent: xmin = 310 Visibility horizon = min(150, 300, 310) = 150 Dead tuples (safe to vacuum): Tuple A: xmax = 120 (< 150, no active snapshot can see it) → RECLAIMABLE Tuple B: xmax = 140 (< 150) → RECLAIMABLE Tuple C: xmax = 160 (>= 150, T_oldest might still need it) → NOT YET SAFE The long-running T_oldest is BLOCKING vacuum of Tuple C. Once T_oldest completes: New horizon = min(300, 310) = 300 Tuple C: xmax = 160 (< 300) → now RECLAIMABLE
Critical dependency: Vacuum's effectiveness is entirely determined by the oldest active snapshot. A single long-running transaction prevents vacuum from reclaiming any dead tuples across the entire cluster, even on tables that transaction never touched.

Autovacuum

PostgreSQL runs autovacuum—a background daemon process that periodically scans tables and triggers VACUUM (and optionally ANALYZE) when thresholds are exceeded:

Parameter Default Behavior
autovacuum_vacuum_threshold 50 Minimum number of dead tuples before considering vacuum
autovacuum_vacuum_scale_factor 0.2 (20%) Fraction of table rows that triggers vacuum
autovacuum_vacuum_cost_delay 2ms Sleep between cost-limited vacuum operations (throttling)
autovacuum_max_workers 3 Maximum parallel autovacuum processes

Vacuum is triggered for a table when:

dead_tuples > autovacuum_vacuum_threshold
           + (autovacuum_vacuum_scale_factor * live_tuples)
Tuning for high-write workloads: For tables with millions of rows, the default 20% scale factor means vacuum won't run until 200K+ dead tuples accumulate. In write-heavy OLTP systems, reduce autovacuum_vacuum_scale_factor to 0.05 (5%) or 0.01 (1%) and lower autovacuum_vacuum_cost_delay to 1ms or less for more aggressive cleanup.

Dead Tuple Accumulation and Bloat

Table growth over time (1M row table, 10% daily churn): Day 0: 1.0M live + 0 dead = 1.0M total (compact) Day 1: 1.0M live + 100K dead = 1.1M total (vacuum reclaims → back to 1.0M) Day 2: 1.0M live + 100K dead = 1.1M total (vacuum reclaims → back to 1.0M) WITHOUT vacuum: Day 1: 1.0M live + 100K dead = 1.1M total Day 2: 1.0M live + 200K dead = 1.2M total Day 10: 1.0M live + 1.0M dead = 2.0M total (100% bloat!) Day 30: 1.0M live + 3.0M dead = 4.0M total (300% bloat!) Indexes also bloat: - Each dead tuple has index entries pointing to it - Regular VACUUM marks index entries as dead but doesn't remove them - VACUUM FULL or REINDEX is needed to actually compact indexes

Impact on Performance

Impact Area Mechanism Severity
Sequential scan speed Must skip over dead tuples on every page High (linear degradation with bloat)
Buffer pool efficiency Dead tuples consume cache space, evicting live data High (cascading effect on all queries)
Index scan performance Index entries point to dead tuples (extra heap fetches) High (random IO waste)
Write amplification Larger pages means more WAL for page modifications Medium
Transaction ID wraparound Unvacuumed tables block pg_xact recycling Critical (can force emergency shutdown)

VACUUM vs. VACUUM FULL

Operation Locking Space Reclamation Use Case
VACUUM Shared (allows reads and writes) Marks space reusable within the table; does NOT return space to OS Regular maintenance (autovacuum)
VACUUM FULL Exclusive (blocks all access) Rewrites the table, compacts pages, returns space to OS Emergency bloat removal (offline maintenance)
pg_repack None (triggers-based, online) Creates new table, swaps, drops old Online bloat removal without downtime
Production warning: Never run VACUUM FULL on a production table without planning. It takes an ACCESS EXCLUSIVE lock for the duration of the rewrite. For large tables, this can mean hours of downtime. Use pg_repack or pg_squeeze for online alternatives.

Monitoring Vacuum Health

Key metrics to watch in production systems:

  • pg_stat_user_tables.dead_tuples — count of unreclaimed dead tuples per table
  • pg_stat_user_tables.last_vacuum — timestamp of last vacuum
  • pg_stat_user_tables.last_autovacuum — timestamp of last autovacuum
  • pg_stat_user_tables.n_dead_tup / n_live_tup — bloat ratio
  • pg_stat_activity — long-running transactions blocking vacuum
  • pg_stat_database.age(xid) — transaction ID age (wraparound proximity)
Interview tip: Be ready to discuss how you would diagnose vacuum issues in production. Mention pg_stat_user_tables, explain the bloat ratio metric, and describe how you would identify the blocking transaction using pg_stat_activity.
