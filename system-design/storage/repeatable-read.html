Repeatable Read

Difficulty: ★★★★☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What Repeatable Read guarantees beyond Read Committed
  • How it prevents non-repeatable reads but still allows phantom reads
  • MySQL/InnoDB's implementation using MVCC and gap locks
  • The critical difference between MySQL's Repeatable Read and PostgreSQL's Repeatable Read

Definition

Repeatable Read guarantees that if a transaction reads a row, it will see the same value if it reads that row again later in the same transaction. Unlike Read Committed, which takes a fresh snapshot for each statement, Repeatable Read takes a single snapshot at the start of the transaction and uses it for all reads.

Non-Repeatable Read Is Prevented

Transaction A Transaction B ──────────────────────────────────────────────────────── BEGIN; SELECT name FROM users WHERE id = 1; -- returns "Alice" (snapshot taken) UPDATE users SET name = "Bob" WHERE id = 1; COMMIT; SELECT name FROM users WHERE id = 1; -- STILL returns "Alice" (same snapshot!) COMMIT;
Guarantee: Within a single transaction, every read of the same row returns the same value. The snapshot is fixed at transaction start, not at statement start.

Phantom Read Is Still Possible (SQL Standard)

Under the SQL standard, Repeatable Read prevents non-repeatable reads but not phantom reads. A phantom read occurs when a new row appears in a range query that was not there before:

Transaction A Transaction B ──────────────────────────────────────────────────────── BEGIN; SELECT * FROM users WHERE age BETWEEN 20 AND 30; -- returns 5 rows (snapshot) INSERT INTO users (name, age) VALUES ("Carol", 25); COMMIT; SELECT * FROM users WHERE age BETWEEN 20 AND 30; -- STILL returns 5 rows in standard SQL -- (but MySQL/InnoDB returns 5 rows due to gap locking) -- A true phantom read would show 6 rows here -- at the SQL standard Repeatable Read level

MySQL/InnoDB Goes Beyond the Standard

MySQL's InnoDB implementation of Repeatable Read uses MVCC for reads and next-key locking (gap locks + record locks) for writes, which actually prevents phantom reads:

How MySQL Prevents Phantoms at Repeatable Read

InnoDB uses "next-key locks" that lock both: 1. The record itself (record lock) 2. The gap before the record (gap lock) SELECT * FROM users WHERE age = 25 FOR UPDATE; Locks: → gap (20, 25) → row 25 → gap (25, 30) → No other transaction can INSERT into these gaps This prevents phantom rows from appearing This is why MySQL's Repeatable Read actually behaves closer to Serializable than the SQL standard
Critical interview point: MySQL/InnoDB's Repeatable Read prevents phantom reads due to next-key locking, even though the SQL standard says phantoms are possible at this level. This is one of the most common gotchas in database interviews. Always clarify which database you are discussing.

Write Skew: The Remaining Anomaly

Both MySQL and PostgreSQL at Repeatable Read are vulnerable to write skew, an anomaly not explicitly named in the SQL standard:

-- Write skew example: two doctors checking on-call rules
-- Rule: at least one doctor must be on call at all times

-- Doctor A's transaction:                 -- Doctor B's transaction:
BEGIN;                                     BEGIN;
SELECT on_call FROM doctors               SELECT on_call FROM doctors
  WHERE id = 1;                              WHERE id = 2;
-- returns TRUE                             -- returns TRUE

-- "The other doctor is on call,            -- "The other doctor is on call,
--  so I can go off duty"                   --  so I can go off duty"
UPDATE doctors SET on_call = FALSE           UPDATE doctors SET on_call = FALSE
  WHERE id = 1;                               WHERE id = 2;
COMMIT;                                     COMMIT;

-- Result: BOTH doctors are off call!
-- Neither Repeatable Read nor Snapshot Isolation prevents this
Write skew requires Serializable isolation to prevent. This is the most common real-world anomaly that teams encounter when using Repeatable Read, and it is the primary reason to escalate to Serializable.

Common Interview Questions

  • "What is the difference between Read Committed and Repeatable Read?"
    At Read Committed, each statement gets a fresh snapshot — the same row may return different values if another transaction committed a change between reads. At Repeatable Read, the snapshot is taken once at transaction start and reused for all reads — the same row always returns the same value within the transaction.
  • "Does MySQL's Repeatable Read prevent phantom reads?"
    Yes, unlike the SQL standard. MySQL/InnoDB uses next-key locks (gap locks + record locks) to prevent other transactions from inserting rows into the gaps of a query's range. This is a significant departure from the standard and is one of MySQL's distinctive features.
  • "What is write skew and how do you prevent it?"
    Write skew occurs when two transactions read overlapping data, each make a decision based on what the other has, and both update different rows. The result violates a business invariant even though no single row was corrupted. It is prevented only by Serializable isolation. See: Lost Update
