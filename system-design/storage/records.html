Records

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What a record (row/tuple) is and how it maps to database pages
  • Row-oriented vs column-oriented storage and their tradeoffs
  • Fixed-length vs variable-length record formats
  • Record headers: what metadata is stored per record
  • Slotted pages: how variable-length records are packed efficiently

Definition

A record (also called a row or tuple) is a single logical entry in a table, containing values for all columns of that row. Records are the atomic unit of read/write for queries, but they are not the unit of I/O — multiple records are packed into a single database page, and I/O happens at page granularity.

Row-Oriented vs Column-Oriented Storage

Row-Oriented (OLTP)Column-Oriented (OLAP)
Storage layoutAll columns of a row stored together on the same pageAll values of each column stored together
Point query (SELECT by id)Fast — one page read returns the full rowSlow — must read from multiple column files
Range scan (full table)Reads all columns, including unneeded onesReads only needed columns (1/10th the I/O)
Aggregation (AVG, SUM)Must decompress full rows to access one columnOperates on compressed column data directly
CompressionLow — mixed data types per blockHigh — homogeneous data enables run-length, dictionary encoding
INSERT / UPDATEEfficient — append or update in-placeExpensive — must update every column file
ExamplesPostgreSQL, MySQL, OracleClickHouse, Amazon Redshift, Snowflake
Best forOLTP (transactional, point lookups)OLAP (analytics, aggregations, scans)
Row-Oriented: Page: [Row1: id|name|age|email] [Row2: id|name|age|email] [Row3: ...] All columns together. Good for "SELECT * WHERE id = 42" Column-Oriented: id_col: [1][2][3][4][5]... ← sorted, compressed name_col: ["Alice"]["Bob"]["C"]... ← dictionary-encoded age_col: [25][30][28][35]... ← run-length encoded email_col: ["a@"]["b@"]["c@"]... ← prefix compressed SELECT AVG(age) WHERE age > 25 → scan age_col ONLY (1/4 of the data, highly compressed) → never touches id, name, or email columns

Slotted Pages

Most real databases (PostgreSQL, MySQL InnoDB, SQLite) use slotted pages to handle variable-length records efficiently:

Slotted Page Layout: Low Address (start of page) ┌────────────────────────────────────┐ │ Page Header (24 bytes) │ │ ├─ LSN, checksum, flags │ │ ├─ pd_lower (end of slot array) │ │ └─ pd_upper (start of record area) │ ├────────────────────────────────────┤ │ Slot Array (Item Pointers) │ │ Grows downward ↓ │ │ ┌──────┐ ┌──────┐ ┌──────┐ │ │ │ #0 │ │ #1 │ │ #2 │ ... │ │ │offset│ │offset│ │offset│ │ │ │len │ │len │ │len │ │ │ └──────┘ └──────┘ └──────┘ │ │ │ │ ← Free Space ← │ │ pd_lower ... pd_upper │ │ │ │ Record Area │ │ Grows upward ↑ │ │ ┌────────────────────┐ │ │ │ Record #2 (var) │ │ │ ├────────────────────┤ │ │ │ Record #1 (var) │ │ │ ├────────────────────┤ │ │ │ Record #0 (var) │ │ │ └────────────────────┘ │ └────────────────────────────────────┘ High Address (end of page) Slot array points to each record by (offset, length). Records grow from bottom; slots grow from top. They meet when the page is full.

How Slotted Pages Handle Operations

  • INSERT: Allocate space for the new record at pd_upper (growing upward). Add a new slot entry at pd_lower (growing downward). If free space is insufficient, the page is full.
  • DELETE: Set the slot's offset to 0 (or mark as dead). The record space becomes reclaimable only when the page is compacted. This avoids shifting other records.
  • UPDATE: In PostgreSQL (MVCC), create a new version of the record. In MySQL InnoDB (in-place), overwrite if the new record fits; otherwise, store the overflow and redirect.
  • Page full?: Allocate a new page from the table's file. The B-tree index is updated to point to the new page.

Record Headers

Every record carries a header with metadata needed by the storage engine:

Header FieldPurpose
Transaction ID (xmin)The transaction that created this record version (for MVCC)
Transaction ID (xmax)The transaction that deleted/updated this record (or 0 if current)
Command ID (cid)Statement ordering within the transaction (for visibility)
NULL bitmap / infomaskWhich columns are NULL (avoids storing NULL data)
Record lengthTotal byte length of the record (for slot array)
Header overhead matters: A PostgreSQL tuple header is 23 bytes. A MySQL InnoDB record header is roughly 5–7 bytes plus per-column NULL flags. For a table with many small columns (e.g., 10 INT columns = 40 bytes of data), the header can be 30–50% of the total record size. This is why wide tables with many nullable columns waste significant space.

Common Interview Questions

  • "How does a database find a specific record?"
    The B-tree index maps the search key to a (page_id, slot_number) tuple. The database reads the page from the buffer pool (or disk if not cached), then uses the slot array to locate the record within the page at the correct byte offset. The slot array provides O(1) lookup within the page.
  • "Why are slotted pages used instead of fixed offsets?"
    Records are variable-length (VARCHAR, TEXT, NULL columns). Fixed offsets would require reserving maximum space for every field, wasting enormous amounts of disk. Slotted pages allow variable-length records to be packed tightly — the slot array maps logical slot numbers to physical byte offsets, supporting both efficient lookup and space-efficient storage.
  • "Why do OLAP databases use column-oriented storage?"
    Analytics queries typically aggregate a few columns across millions of rows (e.g., SELECT AVG(revenue) BY region). Column storage reads only the needed columns (potentially 1/10th the I/O), and homogeneous data types enable much better compression (10–100x with run-length encoding, dictionary encoding). Row storage would read all columns for every row, wasting I/O on data the query never uses.
  • "What happens when a record is deleted?"
    In a slotted page, the slot entry is marked as dead (offset set to 0), but the physical space is not immediately reclaimed. The space becomes reusable only when the page is compacted (all live records are rewritten contiguously). This avoids the cost of shifting other records on every delete.

What to Skip

  • Exact byte-level format of PostgreSQL HeapTupleHeaderData or MySQL InnoDB record format
  • TOAST (The Oversized-Attribute Storage Technique) compression algorithms
  • PAX (Partition Attributes Across) hybrid storage format
