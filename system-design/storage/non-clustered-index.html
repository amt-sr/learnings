Non Clustered Index

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the non-clustered index as a separate B+ tree whose leaf nodes contain pointers (bookmarks) to row data rather than the data itself, how lookup works differently on heap tables versus clustered tables, and when this design is preferable to a clustered index.

Definition

A non-clustered index is an index stored separately from the table data. Its leaf nodes contain the indexed key plus a bookmark (a pointer to the actual row) rather than the full row data. A table can have many non-clustered indexes. The physical order of data pages is independent of the non-clustered index key order.

Structure

Non-Clustered Index (on last_name) Heap Table (no clustered index) [M|T] +------+ +------+ +------+ / \ | Row1 | | Row2 | | Row3 | [A|J|L] [N|S|W] | id=7 | | id=3 | | id=1 | / | | \ | | | \ +------+ +------+ +------+ .. .. .. .. .. .. .. +------+ +------+ +------+ | | | | | Row4 | | Row5 | | Row6 | RID RID RID RID | id=5 | | id=9 | | id=2 | (offset) +------+ +------+ +------+ Leaf nodes contain: (last_name, RID) RID = physical row offset in the heap. Rows are NOT sorted by last_name on disk.

Heap Table vs Clustered Table Lookup

The bookmark lookup behavior depends on whether the underlying table uses a heap or a clustered index.

Heap Table Lookup

Non-Clustered Index --> Heap Table 1. Traverse B+ tree on index key 2. Leaf contains: (key, RID) where RID = (page_number, slot_offset) 3. Direct seek to page_number + slot_offset in heap 4. Cost: 1 B+ tree traversal + 1 direct page fetch (no second tree walk) Advantage: bookmark is a physical pointer, not a key that requires its own B+ tree traversal.

Clustered Table Lookup

Non-Clustered Index --> Clustered Index (= data) 1. Traverse B+ tree on secondary key 2. Leaf contains: (secondary_key, clustered_key) -- NOT a physical pointer 3. Traverse clustered index B+ tree using clustered_key 4. Cost: 2 B+ tree traversals (secondary + clustered) Disadvantage: extra tree traversal adds latency, especially when returning many rows.

Comparison

Aspect Non-Clustered Index (Heap) Non-Clustered Index (Clustered)
Bookmark type Physical RID (page + offset) Clustered key value
Lookup cost 1 B+ tree + 1 direct fetch 2 B+ tree traversals
Row movement RID changes on heap reorganization Clustered key is stable
Index maintenance Indexes must be updated if heap compacts Secondary indexes unaffected by page splits
Full index rebuild Needed if heap reorganizes (e.g., VACUUM FULL in PG) Not needed on clustered page splits
Sequential range scan on index key Random I/O -- each RID points to a different page Random I/O -- each bookmark requires clustered lookup

Engines That Use Heap Tables

Engine Default Storage Model Non-Clustered Index Bookmark
PostgreSQL Heap table CTID (tuple identifier: block number + item offset)
MyISAM (MySQL) Heap table Physical byte offset in .MYD file
SQL Server (no clustered index) Heap table RID (page:file:slot)
MongoDB (WiredTiger) Heap-like RecordId (internal storage location)

Advantages of Non-Clustered Indexes

  • Multiple per table: You can create as many non-clustered indexes as needed. Each provides an alternative access path.
  • Independent ordering: The data can be physically organized by one key (clustered) while offering fast lookups on other keys via non-clustered indexes.
  • Heap table insert speed: In heap tables, inserts always go to the last page -- no B+ tree traversal needed for data insertion (only for index updates).
  • Faster full table scans: Heap tables can be scanned sequentially in physical order without navigating a B+ tree.

Disadvantages

  • Bookmark lookup overhead: Every row requires an additional fetch (RID lookup or clustered key lookup) beyond the index traversal.
  • Stale RIDs in heap tables: When PostgreSQL runs VACUUM FULL or SQL Server rebuilds the heap, row locations change and all non-clustered indexes with RID bookmarks must be rebuilt.
  • Write amplification: Like all secondary indexes, every INSERT/DELETE/UPDATE touching indexed columns updates the non-clustered B+ tree.

When to Prefer Non-Clustered (Heap) Over Clustered

Heap table is better when:
  • Insert order is unpredictable and you want fast appends without page splits.
  • The table is small enough that range scan performance is not critical.
  • You need multiple equally important access paths (no single dominant query key).
  • Workload is OLTP with many single-row lookups via indexes, not range scans.
Clustered is better when:
  • Queries frequently scan ranges on a specific key (e.g., time-series data, ordered feeds).
  • You need index-only scans and the clustered key is always included.
  • Data has a natural sort order that aligns with access patterns.

Common Interview Questions

  • What is a non-clustered index and how does it differ from a clustered index?
  • Explain the difference between non-clustered index lookup on a heap table versus a clustered table.
  • What is a RID bookmark and when does it become stale?
  • Why does PostgreSQL use heap tables instead of clustered indexes?
  • What happens to non-clustered indexes when you run VACUUM FULL in PostgreSQL?
  • When would you choose a heap table design over a clustered table design?
  • What is the cost of a bookmark lookup in a non-clustered index?

What to Skip

Depth to avoid in interviews:
  • Internal format of CTID in PostgreSQL (block/item identifier bit layout).
  • SQL Server forwarding pointers for heap row updates.
  • MyISAM .MYI index file header and key compression internals.
  • Bitmap index internals (which are a non-clustered variant in some engines).
