RAM (Random Access Memory)

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • RAM's role in the storage hierarchy — nanosecond access, volatile
  • DRAM vs SRAM and why databases use DRAM
  • How RAM is used for buffer pools, caches, and in-memory databases
  • Memory hierarchy tradeoffs: capacity vs speed vs cost
  • The relationship between RAM, buffer pools, and OS page cache

Definition

RAM (Random Access Memory) is volatile, byte-addressable primary storage with nanosecond-level access times. It provides the fastest tier of storage available to the CPU for working data, but loses all contents on power loss. In database systems, RAM is the critical resource that determines how much data can be served without touching SSD or HDD.

Key point: Database servers use DRAM for their main memory. The CPU cache (SRAM) is managed by hardware and is invisible to the database. When interviewers ask about "in-memory" databases or caching, they are talking about DRAM.

RAM in Database Systems

RAM is used in several critical ways within a database server:

1. Buffer Pool (dedicated topic)

The largest consumer of RAM. The buffer pool caches frequently accessed database pages in memory to avoid disk I/O. Typical sizing: 50–80% of available RAM.

2. Query Execution Memory

Sort operations, hash joins, and intermediate result sets. If the working set exceeds available RAM, the database spills to temporary files on disk, causing dramatic performance degradation.

3. Log Buffer

The WAL log buffer holds log records in memory before flushing to disk. Sized to absorb bursts of write activity.

4. Connection Overhead

Each database connection has a thread/process with its own stack and buffers. 10,000 connections can consume 2–10 GB of RAM just for connection state.

Memory Bandwidth vs Latency

Two distinct performance dimensions matter for databases:

LatencyBandwidth
What it measuresTime for a single accessTotal data throughput per second
DDR4/DDR5 value~75–100 ns~25–50 GB/s (per channel)
Why it mattersBuffer pool page lookups (pointer chasing in B-trees)Table scans, batch processing, analytics
BottleneckPointer-chasing patterns (random access)Vectorized processing, columnar scans
Interview insight: Memory bandwidth is often the real bottleneck for analytical queries, not latency. A full table scan reading 10 GB of data at 50 GB/s bandwidth takes 200 ms — fast. But 10 GB of random 4KB page reads at 100 ns each through pointer chasing takes ~100 ms per GB = 1 second. This is why in-memory databases like Redis use optimized data structures (compact, cache-line friendly) rather than raw hash tables.

NUMA Considerations

Modern multi-socket servers use NUMA (Non-Uniform Memory Access). Memory attached to one CPU socket has lower latency and higher bandwidth for that socket's cores:

  • Local access: ~75 ns latency, full bandwidth
  • Remote access: ~120–150 ns latency, reduced bandwidth, crosses QPI/UPI interconnect

Database buffer pools must be NUMA-aware — binding memory to sockets avoids the ~2x latency penalty for remote memory access. PostgreSQL, MySQL, and Oracle all have NUMA configuration options.

Common Interview Questions

  • "How much RAM does a database server need?"
    It depends on the working set. The golden rule: buffer pool RAM should cover your hot dataset. If your active dataset is 100 GB, you need at least 120–150 GB of RAM (buffer pool + overhead). If the hot data does not fit in RAM, you will see excessive disk reads and latency spikes. See: Buffer Pool
  • "Why not just use an in-memory database for everything?"
    Cost (RAM is 100x more expensive than SSD per GB), volatility (power loss = data loss without replication), and data size (many datasets exceed single-server RAM). In-memory databases like Redis and Memcached complement persistent databases rather than replacing them.
  • "What happens when a query needs more memory than available?"
    The database spills to disk (temporary files). Sorts, hash joins, and aggregations that exceed the memory limit degrade from memory-speed (~100 ns) to disk-speed (~100 μs for SSD), a 1000x slowdown. This is why work_mem (PostgreSQL) or sort_buffer_size (MySQL) tuning is critical for query performance.
