Phantom Read

Difficulty ⭐⭐⭐☆☆ Interview Frequency ★★★★☆

Goal

Understand what a phantom read is, how it differs from a non-repeatable read, why it is harder to prevent, and how gap locks solve it in MySQL/InnoDB.

Definition

A phantom read occurs when a transaction executes the same range query twice and gets a different number of rows. The difference is not that existing rows changed (that is a non-repeatable read), but that new rows appeared or existing rows disappeared from the result set.

Key distinction: A non-repeatable read changes an existing row's value. A phantom read changes the set of rows matching a query. Phantoms involve INSERTs and DELETEs, not UPDATEs.

Anatomy of a Phantom Read

Phantom Read Timeline ───────────────────────────────────────────────────────────────── Transaction A (Salary Report) Transaction B (New Employee) ──────────────────────────── ────────────────────────────── BEGIN; SELECT name, salary FROM employees WHERE department = 'Engineering' AND salary > 100000; -- Returns 5 rows INSERT INTO employees (name, department, salary) VALUES ('Dave', 'Engineering', 150000); COMMIT; SELECT name, salary FROM employees WHERE department = 'Engineering' AND salary > 100000; -- Returns 6 rows! (Dave appeared) -- Report total salary is inconsistent: -- First query: sum of 5 rows -- Second query: sum of 6 rows COMMIT;

Phantom Read vs. Non-Repeatable Read

Aspect Non-Repeatable Read Phantom Read
What changes Existing row's value Set of rows (new row appears or disappears)
Caused by UPDATE on existing row INSERT or DELETE
Query type Point query (single row) Range query (WHERE clause)
Example Price changed from $100 to $120 New employee row appeared in department query
Prevented by Repeatable Read Serializable (or gap locks / snapshot isolation)

Why Phantoms Are Harder to Prevent

Preventing non-repeatable reads is straightforward: lock the row. But preventing phantoms requires locking the gap between rows -- the abstract space where a new row could be inserted. This is fundamentally harder because:

  • Gaps are not physical objects: You cannot lock a row that does not exist yet.
  • Range overlap: Two range queries may partially overlap in complex ways.
  • Index dependency: Gap locking behavior depends on which index is used for the query.

Solution: Gap Locks

MySQL/InnoDB uses next-key locks, which combine a record lock on the existing row with a gap lock on the space before it. This prevents any other transaction from inserting a row that would fall into that gap.

InnoDB Next-Key Locking ───────────────────────────────────────────────────────────────── Index on salary (sorted): ┌───────┬───────┬───────┬───────┬───────┬───────┐ │ 50000 │ 75000 │ 90000 │ 120000│ 150000│ 200000│ └───────┴───────┴───────┴───────┴───────┴───────┘ ↑ ↑ ↑ ↑ ↑ ↑ gap gap gap gap gap gap Query: SELECT * FROM employees WHERE salary > 100000 FOR UPDATE Next-key locks acquired: → gap(90000, 120000) + record(120000) → gap(120000, 150000) + record(150000) → gap(150000, 200000) + record(200000) → gap(200000, +∞) No INSERT with salary in these ranges is allowed! → Phantom read prevented at Repeatable Read.
MySQL advantage: MySQL/InnoDB's next-key locking means its Repeatable Read level actually prevents phantom reads, exceeding the SQL standard's requirements. PostgreSQL's Repeatable Read (implemented as Snapshot Isolation) also prevents phantoms via frozen snapshots, but through a different mechanism.

Which Isolation Levels Prevent Phantoms?

Isolation Level Phantom Read Notes
Read Uncommitted Possible No protection
Read Committed Possible Fresh snapshot per statement
Repeatable Read (SQL standard) Possible Standard allows phantoms here
Repeatable Read (MySQL/InnoDB) Prevented Gap locks prevent phantoms
Snapshot Isolation Prevented Frozen snapshot -- new rows invisible
Serializable Prevented Full serialization guarantee

Real-World Example: Double-Booking

-- Booking system: check for conflicts, then insert
BEGIN;
SELECT COUNT(*) FROM bookings
  WHERE room_id = 42
    AND start_time < '2025-01-15 14:00'
    AND end_time > '2025-01-15 13:00';
-- Returns 0 (no conflict)

-- Concurrent booking inserts for the same room/time (phantom row!)

INSERT INTO bookings (room_id, start_time, end_time)
  VALUES (42, '2025-01-15 13:00', '2025-01-15 14:00');
-- Double booking! Phantom read allowed the conflict
COMMIT;
Fix: Use Serializable isolation, gap locks (MySQL), or a materialized constraint such as a UNIQUE index on (room_id, start_time) to prevent conflicting inserts at the database level.

Common Interview Questions

  1. What is a phantom read and how does it differ from a non-repeatable read?
  2. Does MySQL's Repeatable Read prevent phantom reads? Why or why not?
  3. How do gap locks work? What is a next-key lock?
  4. How does snapshot isolation prevent phantom reads without locking?
  5. How do you prevent phantom reads without escalating to Serializable isolation?
  6. Why are phantoms harder to prevent than non-repeatable reads?
