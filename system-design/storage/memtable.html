Memtable

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What a memtable is and its role in the LSM tree write path
  • Why memtables must be sorted data structures
  • Skip lists vs red-black trees vs other sorted structures
  • The memtable flush cycle and memory management

Definition

A memtable is an in-memory, sorted data structure that serves as the write buffer in an LSM tree. All incoming writes are first inserted into the memtable. When it reaches a configurable size threshold, the memtable is flushed to disk as an immutable SSTable, and a new empty memtable takes its place.

Memtable Lifecycle in an LSM Tree Client Writes: PUT("a", 1) PUT("c", 3) PUT("b", 2) PUT("d", 4) │ │ │ │ ▼ ▼ ▼ ▼ ┌─────────────────────────────────────────────────────────────────┐ │ WAL (disk) │ │ [put a=1] [put c=3] [put b=2] [put d=4] (sequential append) │ └─────────────────────────────┬───────────────────────────────────┘ │ replay on crash recovery ┌─────────────────────────────▼───────────────────────────────────┐ │ ACTIVE Memtable (memory) │ │ Skip List / Red-Black Tree / AVL Tree │ │ │ │ "a" → 1 "b" → 2 "c" → 3 "d" → 4 (sorted order) │ │ │ │ Size: 42 MB / 128 MB threshold │ └─────────────────────────────┬───────────────────────────────────┘ │ threshold reached ▼ ┌─────────────────────────────────────────────────────────────────┐ │ Flush to SSTable (disk) │ │ data_000004.db: a=1, b=2, c=3, d=4 (sorted, immutable) │ └─────────────────────────────────────────────────────────────────┘ New empty memtable takes over. Old memtable becomes "immutable memtable" until flush completes, then is discarded.

Why Memtables Must Be Sorted

The memtable is sorted by key so that when it's flushed to disk, the resulting SSTable is already sorted. Sorted SSTables are essential because:

  • Binary search — sorted keys enable O(log n) lookup within an SSTable using a sparse index
  • Merge during compaction — merging sorted files is O(n), much faster than sorting unsorted files O(n log n)
  • Bloom filter efficiency — sorted data allows building compact bloom filters
  • Range queries — sorted data supports efficient range scans

Why Skip Lists Dominate

Skip lists are the most popular choice (used by LevelDB, RocksDB, Cassandra) because they offer the same asymptotic performance as balanced BSTs but are simpler to implement, faster in practice due to lower constant factors, and support concurrent operations more easily.
Skip List Example — searching for key "d": Level 3: [a] ————————————————————————————————→ [h] ———→ NIL Level 2: [a] ——————→ [c] ————————————→ [g] ———→ [h] ———→ NIL Level 1: [a] ——→ [b] ——→ [c] ——→ [d] ——→ [g] ——→ [h] → NIL Level 0: [a] → [b] → [c] → [d] → [e] → [f] → [g] → [h] → NIL Search path: a → c → d (found in 3 steps) Time: O(log n) expected, same as balanced BST Advantages over red-black tree: - Simpler implementation (no rotations, no rebalancing) - Better cache locality (pointers go forward, not up/down) - Easy concurrent insertion (lock individual nodes, not whole tree) - Lower constant factor in practice

Active vs Immutable Memtable

Dual Memtable Pattern (used by RocksDB, Cassandra) ┌────────────────────────────┐ ┌────────────────────────────┐ │ ACTIVE Memtable │ │ IMMUTABLE Memtable │ │ (receiving writes) │ │ (being flushed to disk) │ │ │ │ │ │ Accepts: PUT, GET, DEL │ │ Read-only │ │ Size: growing... │ │ Sorted, complete │ └────────────────────────────┘ └────────────┬───────────────┘ │ ▼ ┌────────────────────────────┐ │ SSTable on disk │ │ (immutable file) │ └────────────────────────────┘ Flow: 1. Writes go to active memtable 2. When active reaches size limit: a. Active becomes immutable b. New empty active memtable is created c. Background thread flushes immutable memtable to SSTable 3. Reads check: active memtable → immutable memtable → SSTables Why dual? — No write blocking during flush. Clients continue writing to the new active memtable while the old one flushes.

Memory Sizing

Typical memtable sizes:

  RocksDB:     64 MB - 256 MB    (write_buffer_size)
  Cassandra:   determined by memtable_allocation_type
               - on_heap:    limited by JVM heap
               - off_heap:   256 MB - 1 GB (recommended for large datasets)
  LevelDB:     4 MB (hardcoded, small)

  Tuning considerations:
  - Larger memtable  → fewer flushes → fewer SSTables → less compaction
  - Larger memtable  → more memory used → higher recovery time (WAL replay)
  - Smaller memtable → more flushes → more L0 SSTables → read amplification

  Production recommendation: 128 MB is a good starting point.
  Adjust based on write throughput vs memory budget.
Crash recovery: If the process crashes, the memtable is lost (it's in-memory). The WAL is replayed to reconstruct the memtable state. This is why the WAL exists — it provides durability for data that hasn't been flushed to an SSTable yet.

Common Interview Questions

  • "What is a memtable and why is it needed?"
    A memtable is the in-memory write buffer in an LSM tree. It absorbs writes at memory speed, keeping them sorted so they can be efficiently flushed to disk as sorted SSTables. Without it, every write would need to go directly to disk, killing throughput.
  • "Why use a skip list instead of a red-black tree for the memtable?"
    Skip lists have the same O(log n) complexity but are simpler to implement, have better cache locality (forward pointers), and support concurrent operations more easily (lock individual nodes during insert). In practice, skip lists are often faster due to lower constant factors.
  • "What happens when the memtable fills up?"
    The active memtable becomes immutable (read-only). A new empty active memtable is created to continue accepting writes. A background thread writes the immutable memtable to disk as a sorted SSTable (sequential write). Once flush completes, the immutable memtable is discarded. During this time, reads must check both the active memtable and the immutable memtable before checking SSTables.
  • "How does crash recovery work with memtables?"
    The WAL (Write-Ahead Log) records all writes in order on disk. On crash, the memtable is empty (it's in-memory). The WAL is replayed: each write operation is re-applied to rebuild the memtable state. This is why writes are first appended to the WAL before modifying the memtable.
