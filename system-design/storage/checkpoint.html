Checkpoint

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What checkpointing is — flushing dirty pages from the buffer pool to data files
  • Why checkpointing dramatically speeds up crash recovery
  • Fuzzy vs sharp checkpoints and their tradeoffs
  • How often to checkpoint — balancing recovery time vs write performance
  • Impact of checkpointing on write throughput and latency

Definition

A checkpoint is a synchronization point where the database ensures all dirty pages modified before a specific LSN are flushed from the buffer pool to the data files on disk. The checkpoint records this LSN, establishing a known recovery starting point. During crash recovery, the database only needs to replay redo records after the last checkpoint LSN — everything before it is already on disk.

Why Checkpointing Matters

Without Checkpoint: Crash at time T: → Must replay ENTIRE WAL from the beginning → If database ran for 30 days, replaying 30 days of changes → Recovery takes hours With Checkpoint every 5 minutes: Checkpoint 1 (T=5min) Checkpoint 2 (T=10min) Checkpoint 3 (T=15min) ──────┼─────────────────────┼─────────────────────┼──────────→ WAL │ │ │ All dirty pages All dirty pages All dirty pages before LSN 500 before LSN 1000 before LSN 1500 flushed to disk flushed to disk flushed to disk Crash at T=12min (between checkpoint 2 and 3): → Only replay WAL from LSN 1000 to LSN 1200 → Recovery takes seconds, not hours!
Key insight: Checkpoints limit the amount of WAL that must be replayed during crash recovery. Without checkpoints, recovery would replay the entire transaction history from database creation. With frequent checkpoints, recovery only replays changes since the last checkpoint.

Checkpoint Mechanics

Checkpoint Process: 1. Record checkpoint LSN (current position in WAL) 2. Flush all dirty pages with LSN <= checkpoint LSN 3. Write checkpoint record to data file header 4. Sharp Checkpoint (PostgreSQL)

PostgreSQL forces a sharp checkpoint: it flushes all dirty pages to disk, writes the checkpoint record, and resumes. This causes a brief I/O spike.

  • Can be mitigated with checkpoint_completion_target (spread the flush over a longer window)
  • Default: 0.9 (use 90% of the checkpoint interval for flushing)
  • checkpoint_timeout controls maximum time between checkpoints (default 5 min)

Fuzzy Checkpoint (MySQL InnoDB)

MySQL InnoDB performs fuzzy checkpointing: a background thread gradually flushes dirty pages, spread over time. The checkpoint position advances as pages are flushed.

  • No single point where all writes stop
  • Page flushing is driven by multiple triggers: dirty page ratio, redo log space pressure, and age of dirty pages
  • innodb_max_dirty_pages_pct: controls when aggressive flushing begins (default 75%)
  • innodb_io_capacity: controls how many pages per second the flush thread can write

How Often to Checkpoint

The tradeoff: More frequent checkpoints = faster recovery but more I/O overhead. Less frequent checkpoints = less I/O overhead but slower recovery. The sweet spot depends on your recovery time objective (RTO) and write throughput tolerance.
Checkpoint FrequencyRecovery TimeI/O OverheadBest For
Every 1–5 minutesSecondsHigher (more flushing)OLTP with strict RTO requirements
Every 5–15 minutesSeconds to minutesModerateGeneral-purpose OLTP
Every 15–60 minutesMinutesLower (less flushing)Write-heavy workloads with relaxed RTO

Checkpoint Triggers

  • Time-based: checkpoint_timeout (PostgreSQL) or age-based flushing (MySQL InnoDB — flush pages older than a threshold)
  • WAL size-based: When WAL reaches max_wal_size (PostgreSQL) or redo log nears capacity (MySQL)
  • Dirty page ratio: When dirty pages exceed a threshold (e.g., 75% of buffer pool)
  • Manual: CHECKPOINT command (PostgreSQL), FLUSH TABLES WITH READ LOCK (MySQL)

Impact on Performance

Checkpointing generates a burst of disk writes (for sharp) or sustained background writes (for fuzzy). This impacts performance:

  • Write throughput: Checkpoint I/O competes with normal write queries. On HDDs, this can saturate disk bandwidth.
  • Latency spikes: Sharp checkpoints can cause brief stalls while dirty pages are flushed. Queries may wait for page flushing to complete.
  • Buffer pool hit ratio: After a checkpoint, recently flushed pages are still in the buffer pool (they are not evicted, just marked clean). No immediate impact.
Best practice: Monitor checkpoint frequency and duration. If checkpoints are too frequent, increase max_wal_size (PostgreSQL) or innodb_log_file_size (MySQL). If recovery is too slow, decrease the checkpoint interval. Use a monitoring tool to track WAL generation rate (MB/s) and checkpoint duration.

Common Interview Questions

  • "What is a checkpoint and why is it important?"
    A checkpoint flushes all dirty pages from the buffer pool to the data files and records the corresponding LSN. This limits the amount of redo log that must be replayed during crash recovery. Without checkpoints, recovery would replay the entire WAL from database creation. With checkpoints every 5 minutes, recovery only replays the last 5 minutes of WAL.
  • "What is the difference between a fuzzy and a sharp checkpoint?"
    A sharp checkpoint flushes all dirty pages at once (PostgreSQL), causing a brief I/O spike but providing an exact recovery starting point. A fuzzy checkpoint gradually flushes dirty pages over time (MySQL InnoDB), spreading I/O but providing an approximate recovery point. Fuzzy checkpoints are better for write throughput; sharp checkpoints provide more precise recovery boundaries.
  • "How do checkpoints affect performance?"
    Checkpoints generate disk I/O to flush dirty pages. Sharp checkpoints cause latency spikes when they fire (a burst of writes). Fuzzy checkpoints generate sustained background I/O. Both compete with application queries for disk bandwidth. If checkpoints are too frequent, they waste I/O on pages that haven't been re-dirtied. If too infrequent, recovery takes too long and the redo log may fill up.
  • "What happens if the redo log fills up before a checkpoint completes?"
    The redo log is circular — once the write position catches up to the checkpoint position, no more redo records can be written. Write operations block until the checkpoint advances the checkpoint LSN far enough to free space. This is called "log file full" or "checkpoint not complete" and is a serious performance problem that indicates the redo log is too small or checkpoints are not keeping up.
