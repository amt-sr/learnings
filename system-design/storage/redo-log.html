Redo Log

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What the redo log records — physical byte-level changes to pages
  • REDO phase during crash recovery: replaying committed changes
  • Relationship to the WAL protocol
  • Redo log buffer: in-memory staging area before writing to log files
  • Log Sequence Numbers (LSN): global ordering of all redo records
  • Circular redo log files and how they are managed

Definition

The redo log is the physical log file that records byte-level changes to database pages. It is the concrete implementation of the WAL protocol. During crash recovery, redo records are replayed in order to restore the database to the state it was in at the time of the crash. The redo log contains the exact byte changes needed to reconstruct dirty pages that were lost from the buffer pool.

Terminology note: In MySQL InnoDB, "redo log" and "WAL" refer to the same thing. In PostgreSQL, the WAL files serve as the redo log. In Oracle, the redo log is distinct from the undo log. Regardless of naming, the redo log always records forward changes for replay.

What Redo Records

The redo log records physical changes to database pages — the exact bytes that changed, where they changed, and what the new values are:

Redo Log Record Structure: ┌──────────────┬───────────┬──────────────┬─────────────────┐ │ LSN │ TXN ID │ Page ID │ Change Data │ │ (sequence) │ (who) │ (which page) │ (what changed) │ └──────────────┴───────────┴──────────────┴─────────────────┘ Example: UPDATE accounts SET balance = 900 WHERE id = 42 (Previous balance was 1000, stored at byte offset 120 of page 42) Physical Redo Record (MySQL InnoDB style): ┌──────────┬──────────┬──────────────┬────────────────────┐ │ LSN 1005 │ TXN 500 │ Page 42 │ Offset 120: │ │ │ │ │ Before: 0x000003E8 │ │ │ │ │ After: 0x00000384 │ │ │ │ │ (1000 → 900) │ └──────────┴──────────┴──────────────┴────────────────────┘ "At byte offset 120 of page 42, change the value from 0x3E8 (1000) to 0x384 (900)"

Physical vs Logical Redo

RecordsExact byte-level changes (page, offset, before, after)High-level operations (INSERT row, UPDATE column)
ReplayIdempotent — safe to replay any number of timesNot naturally idempotent — needs care

Idempotency is critical: During crash recovery, the database may crash again before completing. On the next recovery attempt, redo records are replayed again from the last checkpoint. Physical redo is naturally idempotent — writing the same byte value to the same offset multiple times produces the same result. Logical redo needs additional logic to handle replay safely.

Redo Log Buffer

Before redo records are written to the log files on disk, they are staged in the redo log buffer — a contiguous block of memory:

Transaction commit (if sync commit) │ 2. Buffer is 1/2 full │ 3. Every 1 second (background) ↓ ┌──────────────────────┐ │ Redo Log Files │ ← On disk (circular files) │ ib_logfile0 / ib_ │ │ logfile1 (InnoDB) │ └──────────────────────┘
  • Space check: If the redo log buffer is too full, the writing thread must wait for the flush thread to write to disk — this is a performance bottleneck.
  • Sizing: MySQL innodb_log_buffer_size (default 16 MB). Large transactions (bulk INSERT, LOAD DATA) benefit from a larger buffer.

Log Sequence Numbers (LSN)

Every redo record is assigned a monotonically increasing Log Sequence Number (LSN):

  • Global ordering: LSNs order all modifications across all transactions
  • Page-level tracking: Each data page in the buffer pool stores the LSN of the last redo record applied to it
  • Checkpoint coordination: A checkpoint records the LSN up to which all dirty pages have been flushed to data files
  • Replay starting point: After a crash, recovery starts replaying from the checkpoint LSN
LSN Timeline: Checkpoint LSN Current LSN │ │ ▼ ▼ ─────┼────────────────────┼──────→ WAL File │ │ LSN 1000 LSN 1500 All changes before LSN 1000 are guaranteed to be in the data files (flushed at checkpoint). Changes LSN 1001–1500 may be in the buffer pool only (dirty pages not yet flushed). If crash happens now → replay LSN 1001–1500 from WAL

Circular Redo Log Files

The redo log files are circular — once the write position reaches the end, it wraps around to the beginning:

Circular Redo Log (2 files, 1 GB each): File 0: [████████████████████████████████| ] File 1: [ | | ] Write position moves forward →→→ Checkpoint position follows →→→ (trailing behind) ┌──────────────────────────────────────────────────┐ │ File 0 (1 GB) │ │ [written][written][written][free ][free ] │ │ ↑ ↑ │ │ Checkpoint LSN Write LSN │ │ (safe to overwrite) (active) │ │ │ │ File 1 (1 GB) │ │ [free ][free ][free ][free ][free ] │ └──────────────────────────────────────────────────┘ When Write LSN catches up to Checkpoint LSN: → "Log file full" → must checkpoint before writing more → If checkpoint is slow → writes are BLOCKED → This is why redo log sizing matters!
Interview tip: If the redo log is too small, the write position catches up to the checkpoint position, forcing a "sharp" checkpoint. During this forced checkpoint, write operations block. If the redo log is too large, crash recovery takes longer because more WAL records must be replayed. The right size balances these tradeoffs.

REDO During Crash Recovery

Crash Recovery — REDO Phase: 1. Find last checkpoint LSN (from checkpoint file) 2. Read WAL from checkpoint LSN to end of WAL 3. For each redo record: a. Check if the target page's LSN >= redo record LSN b. If YES → page already has this change (was flushed) → skip c. If NO → apply the byte change to the page → mark dirty 4. After all redo records processed: - Buffer pool now reflects state at crash time - All committed changes are present Then UNDO phase: roll back uncommitted transactions (using undo log)

Common Interview Questions

  • "What does the redo log record?"
    The redo log records physical byte-level changes to database pages — the exact offset within a page, the before-value, and the after-value. During crash recovery, these changes are replayed to reconstruct dirty pages that were lost from the buffer pool when the database crashed.
  • "What is an LSN?"
    A Log Sequence Number is a monotonically increasing identifier assigned to every redo record. It provides a global total order of all modifications across all transactions. The LSN coordinates between the redo log, the buffer pool, and checkpoints: each data page stores the LSN of the last redo applied to it, and checkpoints record the LSN up to which all changes are flushed.
  • "Why must redo be idempotent?"
    During crash recovery, the database may crash again before completing. On the second recovery attempt, redo records from the first attempt are replayed again. If redo were not idempotent, replaying the same change twice would corrupt the data. Physical redo (byte-level) is naturally idempotent; logical redo needs careful design.
  • "What happens when the redo log fills up?"
    The redo log files are circular. When the write position catches up to the checkpoint position, no more redo can be written. The database must perform a checkpoint to flush dirty pages and advance the checkpoint position. If the checkpoint is slow, all write operations block. This is why sizing the redo log appropriately is critical — it must hold enough redo records for the busiest write period.
