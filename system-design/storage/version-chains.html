Version Chains

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★☆☆

Goal

Understand how MVCC maintains multiple versions of each row as a linked chain, how transaction IDs annotate those versions, and how readers traverse the chain to find the correct visible version without blocking writers.

Definition

A version chain is the core data structure in MVCC systems. Instead of overwriting a row in place when it is updated, the database creates a new version of that row and links it to the previous version via a pointer. Each version is annotated with the transaction ID of the transaction that created it and, optionally, the transaction ID that deleted or superseded it. This chain allows concurrent readers and writers to operate on different versions of the same row without mutual blocking.

Key insight: In-place updates would require readers to lock rows, blocking concurrent writers. Version chains eliminate this by making every write produce a new version, so readers always see a consistent (though possibly stale) snapshot of the data.

Version Chain Anatomy

In PostgreSQL, each row version (called a tuple) stores two critical transaction ID fields:

Field Meaning
xmin Transaction ID of the transaction that created (inserted) this tuple version.
xmax Transaction ID of the transaction that deleted or expired this tuple version. Zero means the version is still live.
ctid Physical pointer (block number + offset) to the next newer version of the same row in the heap.
Not just PostgreSQL: MySQL InnoDB stores version chain links in the undo log rather than inline on the heap page. The principle is the same—each version points to its predecessor—but the storage mechanism differs. Oracle uses a similar undo-segment-based approach.

Version Chain Evolution

Step 1: INSERT by T1 (xmin=100) ┌──────────────────────────────┐ │ Tuple A │ name="Alice" │ │ │ xmin=100 xmax=0 │ ← live version │ │ ctid=NULL │ ← chain end (newest) └──────────────────────────────┘ Step 2: UPDATE by T2 (xmin=200) ┌──────────────────────────────┐ │ Tuple A │ name="Alice" │ │ │ xmin=100 xmax=200 │ ← expired by T2 │ │ ctid ──────────────────→ newer version └──────────────────────────────┘ ┌──────────────────────────────┐ │ Tuple B │ name="Bob" │ │ │ xmin=200 xmax=0 │ ← live version │ │ ctid=NULL │ ← chain end (newest) └──────────────────────────────┘ Step 3: UPDATE by T3 (xmin=300) ┌──────────────────────────────┐ │ Tuple A │ name="Alice" │ │ │ xmin=100 xmax=200 │ │ │ ctid ─────────────────────→ Tuple B └──────────────────────────────┘ ┌──────────────────────────────┐ │ Tuple B │ name="Bob" │ │ │ xmin=200 xmax=300 │ │ │ ctid ─────────────────────→ Tuple C └──────────────────────────────┘ ┌──────────────────────────────┐ │ Tuple C │ name="Charlie" │ │ │ xmin=300 xmax=0 │ ← current live version │ │ ctid=NULL │ └──────────────────────────────┘ Full chain: A (xmin=100) ──→ B (xmin=200) ──→ C (xmin=300, live)

How Readers Traverse the Chain

When a query needs to find the visible version of a row, it starts at the newest version (found via the index) and walks backward through the chain:

  1. Start at the most recent tuple pointed to by the index entry.
  2. Check if this version is visible to the current transaction (using snapshot-based visibility rules).
  3. If visible, return it. If not, follow the ctid pointer to the previous version.
  4. Repeat until a visible version is found or the chain is exhausted.
Readers never block writers: A long-running SELECT follows old tuple versions in the chain while concurrent UPDATEs keep appending new versions at the head. The reader is completely unaware of the newer versions.

Undo Log as Version Storage

In systems like MySQL InnoDB and Oracle, old tuple versions are stored in the undo log rather than remaining on the heap page. The undo log is a dedicated, append-only storage area:

  • Insert undo records: Stored when a row is first inserted; needed only for rollback, not for MVCC reads.
  • Update undo records: Store the before-image of modified rows; these form the version chain that MVCC readers traverse.
Approach Example Version Location Chain Traversal
Heap-based PostgreSQL All versions live on heap pages Follow ctid pointers on the page
Undo-based MySQL InnoDB, Oracle Only latest version on page; old versions in undo log Follow undo log pointers (roll_ptr)
Trade-off: Heap-based versioning keeps all versions together (simpler traversal) but bloats the table. Undo-based versioning keeps the table lean but requires an additional IO to the undo log for historical reads. Undo segments are also subject to retention policies and can be recycled.

Transaction ID Wraparound

PostgreSQL uses a 32-bit transaction ID that wraps around after ~4 billion transactions. To avoid ambiguity, the system divides the ID space into "past," "now," and "future" epochs. When the oldest version in any chain approaches the wraparound point, VACUUM FREEZE must rewrite tuples to set their xmin to a special frozen value. Failure to freeze leads to data loss—the system will shut down to protect itself.

Operational risk: Transaction ID wraparound is a silent killer. Long-running transactions prevent VACUUM from advancing the freeze horizon. Monitor pg_stat_activity for long-lived transactions and set autovacuum_freeze_max_age conservatively.

Common Interview Questions

  • How does MVCC allow readers to not block writers? Walk through the version chain mechanism.
  • What are xmin and xmax? How do they encode the lifecycle of a tuple?
  • How does a SELECT find the correct version of a row when multiple concurrent updates have occurred?
  • What is the difference between PostgreSQL's heap-based versioning and MySQL InnoDB's undo-log-based versioning?
  • What happens if a version chain gets very long? How does this affect read performance?
  • What is transaction ID wraparound and how does VACUUM FREEZE prevent data corruption?
  • Design a version chain data structure. What fields does each node need? How do you link them?

What to Skip

For interviews, skip:
  • The exact on-disk format of PostgreSQL heap tuple headers (Infomask bits, combination CID, etc.)
  • Undo log segment internals in InnoDB (purge thread details, undo tablespace layout)
  • Oracle's ORA-01555 snapshot too old internals beyond the conceptual level
  • Multi-version concurrency control in non-relational systems (Cassandra's tombstone-based approach is a different concept)
