Lost Update

Difficulty ⭐⭐⭐☆☆ Interview Frequency ★★★★☆

Goal

Understand the lost update problem, why it occurs in read-then-write patterns, and the spectrum of solutions from atomic operations to optimistic and pessimistic concurrency control.

Definition

A lost update occurs when two transactions read the same data, then both update it based on what they read. The second writer overwrites the first writer's change, and the first update is lost -- as if it never happened.

Core pattern: Read-then-write without any protection. Both transactions see the same initial value, compute a new value independently, and the second commit silently discards the first commit's work.

Anatomy of a Lost Update

Lost Update Timeline ───────────────────────────────────────────────────────────────── Transaction A (User 1 likes post) Transaction B (User 2 likes post) ──────────────────────────── ──────────────────────────────── BEGIN; SELECT likes FROM posts WHERE id = 42; -- reads likes = 100 BEGIN; SELECT likes FROM posts WHERE id = 42; -- reads likes = 100 UPDATE posts SET likes = 100 + 1 WHERE id = 42; -- writes likes = 101 UPDATE posts SET likes = 100 + 1 WHERE id = 42; -- writes likes = 101 (based on stale read!) COMMIT; COMMIT; Final value: likes = 101 Expected: likes = 102 User 1's like was LOST!

When Lost Updates Occur

Lost updates are not tied to a single isolation level -- they can occur even at Repeatable Read. The problem is structural: the application performs a read-modify-write cycle without ensuring atomicity of the entire cycle.

Isolation Level Lost Update Possible? Why
Read Uncommitted Yes No protection at all
Read Committed Yes Both reads see same committed value, no write conflict detection
Repeatable Read (SQL standard) Yes Snapshot frozen, but no conflict detection on same-row writes
Snapshot Isolation No First-committer-wins aborts second writer on same row
Serializable No Full conflict detection
Important nuance: At standard SQL Repeatable Read (without snapshot isolation), lost updates can still occur because the standard only guarantees that re-reading yields the same value -- it does not detect write-write conflicts on the same row. PostgreSQL's implementation of Repeatable Read uses snapshot isolation internally, so it does prevent lost updates. MySQL/InnoDB's Repeatable Read also prevents them via next-key locks.

Solutions

1. Atomic Increment

The simplest fix: avoid the read entirely. Use an atomic operation that increments the value in a single statement.

-- WRONG: read-then-write (lost update)
SELECT likes FROM posts WHERE id = 42;  -- read
UPDATE posts SET likes = 100 + 1 WHERE id = 42;  -- write (stale!)

-- CORRECT: atomic increment
UPDATE posts SET likes = likes + 1 WHERE id = 42;
-- The database handles the increment atomically.
-- No read needed in application code.

2. SELECT FOR UPDATE (Pessimistic Locking)

Explicitly lock the row when reading, preventing other transactions from modifying it until your transaction completes.

-- Transaction A:
BEGIN;
SELECT likes FROM posts WHERE id = 42 FOR UPDATE;
-- Row is now locked. Transaction B must wait.
UPDATE posts SET likes = 100 + 1 WHERE id = 42;
COMMIT;  -- lock released

-- Transaction B (was waiting, now proceeds):
BEGIN;
SELECT likes FROM posts WHERE id = 42 FOR UPDATE;
-- Now reads likes = 101 (A's commit is visible)
UPDATE posts SET likes = 101 + 1 WHERE id = 42;
COMMIT;

-- Final: likes = 102

3. Optimistic Concurrency Control (Compare-and-Set)

Read the value and a version column. When writing, include a WHERE clause that checks the version has not changed. If the update affects zero rows, the application retries.

-- posts table has a version column:
-- id | likes | version
-- 42 | 100   | 5

-- Transaction A:
BEGIN;
SELECT likes, version FROM posts WHERE id = 42;
-- reads likes=100, version=5
UPDATE posts SET likes = 101, version = version + 1
  WHERE id = 42 AND version = 5;
-- Affects 1 row -- success
COMMIT;

-- Transaction B:
BEGIN;
SELECT likes, version FROM posts WHERE id = 42;
-- reads likes=101, version=6 (A already committed)
UPDATE posts SET likes = 102, version = version + 1
  WHERE id = 42 AND version = 5;
-- Affects 0 rows -- version mismatch! Concurrent modification.
ROLLBACK;
-- Application retries the transaction.

Lost Update vs. Write Skew

Aspect Lost Update Write Skew
Pattern Two txs read same row, write same row Two txs read overlapping rows, write different rows
What is lost One transaction's update to the same row An invariant across multiple rows
Detected by SI? Yes (same-row conflict) No (disjoint writes)
Example Counter: 100 to 101, then 100 to 101 = lost one Two doctors both go off-call (different rows, broken invariant)

Common Interview Questions

  1. What is a lost update and how does it occur?
  2. How do you prevent lost updates? Name at least three approaches.
  3. When would you use SELECT FOR UPDATE vs. optimistic concurrency control?
  4. Can a lost update occur at Repeatable Read isolation?
  5. What is the difference between a lost update and write skew?
  6. How does snapshot isolation prevent lost updates on the same row?
  7. Design a rate limiter -- how do you prevent lost updates on the counter?
  8. What happens when optimistic concurrency control fails? How does the retry loop work?

What to Skip

Depth to avoid in interviews:
  • Deep internals of InnoDB's lock wait timeout and deadlock detection
  • Predicate locking for multi-row lost update scenarios
  • Distributed lost update prevention (e.g., etcd transactions, DynamoDB conditional writes) unless the question specifically targets distributed systems
