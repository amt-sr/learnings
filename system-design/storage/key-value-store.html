Key-Value Store

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What a key-value store is: simple get/put/delete API on opaque values
  • Why key-value stores offer extreme simplicity and performance
  • Common use cases: caching, session storage, configuration, leaderboards
  • Tradeoffs: no joins, no queries by non-key columns, limited consistency models

Definition

A key-value store is the simplest database model — each piece of data is stored as a value indexed by a unique key. The API is minimal: PUT(key, value), GET(key), DELETE(key). The database does not understand or query the structure of the value — it's treated as an opaque blob. This simplicity enables extreme performance and horizontal scalability.

Key-Value Store — Simple API ┌───────────────────────────────────────────────────────┐ │ Key-Value Store │ │ │ │ PUT("session:abc123", '{"user_id": 42, "token": ...}')│ │ PUT("user:42:profile", '{"name":"Alice","age":30}') │ │ PUT("cache:homepage:html", '...') │ │ PUT("counter:page_views", "1584293") │ │ PUT("config:feature_flags", '{"dark_mode": true}') │ │ │ │ GET("session:abc123") → '{"user_id": 42, ...}' │ │ GET("user:42:profile") → '{"name":"Alice", ...}' │ │ DELETE("session:abc123") │ │ │ │ Rules: │ │ - Keys are unique (PUT overwrites existing key) │ │ - Values are opaque (DB doesn't inspect content) │ │ - No joins, no SQL, no schema │ │ - Lookup ONLY by exact key │ └───────────────────────────────────────────────────────┘

Key Characteristics

CharacteristicDetail
Schema-freeNo schema definition. Any key maps to any value. Values can be strings, JSON, binary blobs, numbers.
Single-key accessData is accessed only by its primary key. No secondary indexes (in pure KV stores). No queries by non-key fields.
Opaque valuesThe database does not parse or understand values. No server-side joins, no filtering, no aggregation.
Horizontal scalingKey-based partitioning (hash or range) distributes data across nodes. Simple and effective.
Low latencySingle key lookup = single disk read (or memory hit). Sub-millisecond common.

Common Use Cases

Key-value stores excel when access patterns are simple: you always know the key and need to get/put a single value. They fail when you need complex queries, joins, or filtering.

Under the Hood

  Key-Value Store Architecture:

  Client
    │
    ▼
  Consistent Hash Ring (partitioning)
    │
    ├── Node A (shard 1):  keys hash(key) ∈ [0, 42)
    ├── Node B (shard 2):  keys hash(key) ∈ [42, 85)
    └── Node C (shard 3):  keys hash(key) ∈ [85, 128)

  GET("user:42"):
    hash("user:42") = 67 → Node B → lookup → return value

  Storage per node:
    Redis:     in-memory hash table + RDB/AOF persistence
    DynamoDB:  LSM tree (like RocksDB) + multi-AZ replication
    Memcached: slab allocation + LRU eviction in memory

Limitations

Key limitations: No secondary queries, no joins, no aggregation, no server-side filtering (in basic KV). To query by non-key fields, you must either maintain secondary indexes yourself (e.g., composite keys) or use a database with built-in secondary indexing (DynamoDB GSI).
  • No ad-hoc queries — cannot SELECT * WHERE age > 25. You need the key.
  • No joins — related data must be denormalized into a single value or fetched with multiple GETs.
  • Value size limits — DynamoDB: 400 KB item max. Memcached: 1 MB.
  • Eventual consistency (in some systems) — reads may return stale data briefly after writes.

Common Interview Questions

  • "When would you use a key-value store instead of a relational database?"
    When access patterns are simple get/put by key: session management, caching, configuration, rate limiting. When you need sub-millisecond latency. When you need effortless horizontal scaling. When data has no relationships or those relationships are denormalized into the value.
  • "How would you implement a shopping cart using Redis?"
    Key = cart:{user_id}, value = Redis Hash with product_id fields and quantity values. Operations: HSET cart:42 product:1 3, HINCRBY cart:42 product:1 1, HGETALL cart:42. TTL for cart expiry. Atomic operations prevent race conditions.
  • "Redis vs Memcached for a caching layer?"
    Redis: data structures, persistence, pub/sub, Lua scripting, more memory overhead. Memcached: simpler, slightly faster for pure string gets, multithreaded, no persistence. Choose Redis unless you need pure string caching with maximum memory efficiency.
  • "How does DynamoDB partition data?"
    DynamoDB uses a hash of the partition key to determine which physical partition stores the item. Each partition is on a specific node. If you also have a sort key, items with the same partition key are stored together (ordered by sort key). This enables efficient queries within a partition key.
