Time-Series Database

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What a time-series database is: optimized for timestamped data with high write throughput
  • Data model: metric name + timestamp + value + tags/labels
  • Downsampling, retention policies, and rollups
  • Why general-purpose databases struggle with time-series workloads

Definition

A time-series database (TSDB) is optimized for storing and querying time-stamped data points, where data is always appended in chronological order and queries typically analyze ranges of time. Time-series data has unique characteristics: extremely high write volume (millions of data points per second), reads are almost always range-based (last hour, last day), old data is either aggregated (downsampled) or deleted (TTL), and recent data is hot while old data is cold.

Time-Series Data Model Each data point: ┌─────────────────────────────────────────────────────────────┐ │ Metric: cpu_usage │ │ Timestamp: 2024-01-15T10:00:00Z │ │ Value: 75.2 │ │ Tags: host=server-1, region=us-east, dc=us-east-1 │ └─────────────────────────────────────────────────────────────┘ Thousands of data points per second, per metric, per host: cpu_usage {host=server-1, region=us-east} 10:00:00 → 75.2 10:00:01 → 76.1 10:00:02 → 74.8 10:00:03 → 77.3 ... (1 point per second, per host) If you have 10,000 hosts × 100 metrics × 1 point/sec: = 1,000,000 data points/second = 86.4 billion points/day Query patterns: - "Average CPU over the last 5 minutes for host=server-1" - "P95 latency across all hosts in us-east for the last hour" - "Alert if memory_usage > 90% for any host in last 10 minutes"

Why Relational Databases Struggle

Cardinality managementIndex bloat with many tag combinationsOptimized label indexing (inverted index)





Downsampling and Retention



  Downsampling is the most important TSDB optimization. Raw data at 1-second granularity is only useful for a short window (hours/days). For longer time ranges, data is aggregated (averaged, min'd, max'd) into coarser granularity. This dramatically reduces storage while preserving analytical value.




  Data Lifecycle with Downsampling

  Raw data (1-second intervals):
  ┌───────────────────────────────────────────────────────┐
  │ cpu_usage {host=server-1}:                            │
  │   10:00:00 → 75.2, 10:00:01 → 76.1, 10:00:02 → 74.8 │
  │   ... (86,400 points per day per host per metric)    │
  │   Retention: 7 days                                   │
  │   Storage: ~2 bytes/point × 86,400 = 173 KB/day/host  │
  └───────────────────────────────────────────────────────┘
          │ after 7 days, downsample to 1-minute
          ▼
  1-minute rollups:
  ┌───────────────────────────────────────────────────────┐
  │ cpu_usage {host=server-1}:                            │
  │   10:00 → avg=75.3, min=74.1, max=76.8                │
  │   ... (1,440 points per day per host per metric)       │
  │   Retention: 30 days                                   │
  │   Storage: 73% reduction                              │
  └───────────────────────────────────────────────────────┘
          │ after 30 days, downsample to 1-hour
          ▼
  1-hour rollups:
  ┌───────────────────────────────────────────────────────┐
  │ cpu_usage {host=server-1}:                            │
  │   2024-01-15T10 → avg=74.8, min=71.2, max=78.1       │
  │   ... (24 points per day)                              │
  │   Retention: 1 year                                    │
  │   Storage: 98% reduction vs raw                        │
  └───────────────────────────────────────────────────────┘

  Total storage savings: ~95% vs keeping all raw data forever



Time-Series Compression


TSDBs use specialized compression that exploits the nature of time-series data:




  
TechniqueHow It WorksCompression Ratio

  
Delta-of-delta encodingStore the difference between consecutive timestamp deltas~2-4x on timestamps

  
XOR-based float compressionGorilla paper: store XOR of consecutive float values~5-10x on values

  
Columnar storageStore timestamps separately from values (better compression of similar data)~2-3x

  
Dictionary encodingMap repeated string tags to integers~10-100x on tags






  The Gorilla paper: Facebook's 2015 paper "Gorilla: A Fast, Scalable, In-Memory Time Series Database" introduced delta-of-delta timestamp encoding and XOR float compression. These techniques are now used by most TSDBs (InfluxDB, Prometheus, TimescaleDB). Know this paper's name for interviews.

Common Interview Questions


    
  
  • "What is a time-series database and why not just use PostgreSQL?"
    TSDBs are optimized for time-stamped data with high write volume, range-based reads, automatic data expiry, and downsampling. PostgreSQL uses B+ trees (random I/O for inserts), requires manual cleanup for old data (no automatic TTL), and lacks built-in downsampling. A TSDB handles millions of points/second with built-in rollups and compression that reduces storage by 95%.
    • 
  
  • "What is downsampling?"
    Reducing the granularity of data over time. Raw 1-second data is kept for a short window (hours/days), then aggregated into 1-minute averages for a longer window (weeks/months), then into 1-hour averages for years. This preserves analytical value while reducing storage by 95%+. Every TSDB provides this as a built-in feature.
    • 
  
  • "How would you design a metrics collection system?"
    Collect metrics via push (agents) or pull (Prometheus scraping), write to a TSDB with retention policies (raw data: 7 days, 1-minute rollups: 30 days, 1-hour rollups: 1 year), query with range queries, visualize with Grafana, alert with Prometheus Alertmanager or equivalent. Consider cardinality limits to prevent explosion of time series from high-cardinality tags.
    • 
  
  • "What is the cardinality problem?"
    Each unique combination of metric + tags creates a separate time series. High cardinality (e.g., tagging by user_id where millions of users exist) creates millions of time series, consuming massive memory and CPU for index maintenance. TSDBs can be brought down by cardinality explosions. Solution: limit tag values (use buckets instead of exact values), enforce cardinality limits, or move high-cardinality data to OLAP systems.
