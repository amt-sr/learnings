Page Cache

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What the OS page cache is — the kernel's built-in file data cache
  • OS page cache vs database buffer pool — and the double buffering problem
  • Direct I/O (O_DIRECT) to bypass the page cache
  • When to rely on the OS page cache vs the database buffer pool
  • Trade-offs between the two caching strategies

Definition

The OS page cache (also called the buffer cache or file cache) is the Linux kernel's in-memory cache of file data blocks. Whenever the kernel reads a file, it keeps the data in RAM as CPU pages. Subsequent reads of the same file data are served directly from memory, avoiding disk I/O. The OS manages this cache automatically using reclaimed memory — no application configuration required.

The Double Buffering Problem

When a database reads a file through the standard OS read()/write() syscalls, the data passes through the OS page cache AND the database buffer pool. The same page exists in memory twice:

Copies in memory2 (buffer pool + page cache)1 (buffer pool only)

Double buffering is not always bad: PostgreSQL intentionally uses double buffering because it trusts the OS page cache's LRU algorithm, which benefits from seeing access patterns across the entire system. The OS page cache provides additional caching beyond shared_buffers, effectively giving PostgreSQL two layers of cache. The tradeoff is less memory efficiency.

Direct I/O (O_DIRECT)

O_DIRECT is a Linux flag (passed to open()) that tells the kernel to bypass the page cache for file I/O:

  • Reads go directly from disk to the application's buffer (the buffer pool page frames)
  • Writes go directly from the application's buffer to disk
  • The data never enters the OS page cache
  • Requirements: buffer address must be page-aligned, file offset must be block-aligned, size must be multiple of block size
Important nuance: O_DIRECT on Linux is a hint, not a guarantee. The kernel may still cache data internally in some cases. For strict bypass, combine O_DIRECT with O_SYNC or use fdatasync(). On macOS, the equivalent is F_NOCACHE. On Windows, it is FILE_FLAG_NO_BUFFERING.

Page Cache Internals

The Linux page cache is organized as a set of page structures in kernel memory:

  • Address space (address_space): Each open file has an address_space object that tracks all cached pages for that file.
  • Radix tree / xarray: Maps file offsets to cached page frames. O(log N) lookup.
  • LRU lists: Active list (recently accessed) and inactive list (candidates for eviction). Similar to the buffer pool's two-list approach.
  • Dirty writeback: The kernel periodically flushes dirty pages via pdflush/flush threads (controlled by vm.dirty_ratio and vm.dirty_background_ratio).
sysadmin tip: You can see page cache usage with free -h (the "buff/cache" column), clear it with echo 3 > /proc/sys/vm/drop_caches (for testing), and tune dirty page flushing thresholds with vm.dirty_ratio and vm.dirty_background_ratio.

Common Interview Questions

  • "What is the OS page cache?"
    The OS page cache is the kernel's built-in file data cache. When any process reads a file, the kernel stores the data in memory (as CPU pages). Subsequent reads of the same file data are served from memory. The kernel manages this automatically using a modified LRU algorithm with active/inactive lists.
  • "What is double buffering and why is it a problem?"
    When a database reads file data through normal OS syscalls (without O_DIRECT), the data is stored in both the database buffer pool and the OS page cache. This wastes memory — the same page exists twice in RAM. On a server with 64 GB RAM, if the buffer pool uses 32 GB and the OS page cache also caches 32 GB of the same data, effectively only 32 GB of unique data is cached, not 64 GB.
  • "What is O_DIRECT and when should you use it?"
    O_DIRECT bypasses the OS page cache for file I/O. The database reads and writes directly to/from disk without the kernel caching the data. This eliminates double buffering, making memory usage more efficient. MySQL InnoDB uses O_DIRECT by default. Use it when the database has a large dedicated buffer pool and the server runs no other significant workloads.
  • "Why does PostgreSQL NOT use O_DIRECT?"
    PostgreSQL relies on the OS page cache as an extension of its buffer pool. The OS cache is dynamically shared — when the DB needs more memory, the OS shrinks the page cache. When other processes need memory, the OS shrinks the page cache too. This provides flexible memory management on shared servers. The cost is double buffering (same data cached twice).
