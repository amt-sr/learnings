3.3.5 Crash Recovery

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

  • Understand the ARIES recovery algorithm (Analysis, Redo, Undo) and why it is the industry standard
  • Explain WAL replay mechanics and how the database reconstructs a consistent state after a crash
  • Describe the full crash recovery pipeline step by step, including checkpoint interaction
  • Evaluate recovery time trade-offs, fuzzy checkpoints, and their implications on system availability
  • Reason about recovery in the context of FAANG-scale systems handling millions of TPS

Definition

Crash recovery is the process by which a database restores itself to a consistent state after an unplanned shutdown (power failure, kernel panic, process crash). It uses the Write-Ahead Log (WAL) to replay committed operations, undo uncommitted changes, and bring the database back to the state it was in at the moment of the crash. The dominant algorithm is ARIES (Algorithm for Recovery and Isolation Exploiting Semantics), used by IBM DB2, SQL Server, PostgreSQL (via WAL), and many others.

Why Crash Recovery Matters at Scale

At FAANG scale, databases serve millions of QPS across thousands of nodes. Crashes are not rare events -- they are routine. Pods get OOM-killed, hardware fails, kernel panics occur, and rolling deployments restart processes constantly. The key metric is Mean Time to Recovery (MTTR): how fast can a node come back online and serve traffic? Fast crash recovery directly impacts SLA targets and the blast radius of any single node failure.

Key insight: Recovery time is a function of (1) how much WAL must be scanned and (2) how many dirty pages must be flushed. Checkpointing reduces both, but introduces write amplification. This is a fundamental trade-off in write-heavy workloads.

The ARIES Recovery Algorithm

ARIES is a WAL-based recovery algorithm with three phases. It achieves steal/no-force buffer management, meaning dirty pages from uncommitted transactions can be flushed to disk (steal), and committed transactions need not force all dirty pages to disk before commit (no-force). This maximizes I/O efficiency at the cost of more complex recovery.

Phase 1: Analysis

The analysis phase scans the WAL starting from the last checkpoint record to reconstruct two critical data structures:

  • Active Transaction Table (ATT): All transactions that were in progress at the time of the crash. Any transaction with a begin record but no commit record is considered active.
  • Dirty Page Table (DPT): All pages that were dirty (modified in buffer pool but potentially not flushed to disk) at crash time. Built by tracking page-level LSNs in log records.

This phase is purely a scan -- no data pages are read or written. It is fast because it only reads the sequential WAL.

Optimization: The checkpoint record contains a snapshot of the ATT and DPT at checkpoint time, so analysis only needs to scan WAL records written after the checkpoint, not the entire WAL history.

Phase 2: Redo

The redo phase replays all logged operations to bring the database to the exact physical state it was in at the moment of the crash. It processes log records in LSN order, starting from the earliest dirty page's LSN (the redo LSN, which is the minimum LSN across all pages in the DPT).

For each log record, ARIES checks the page's on-disk LSN (stored in the page header). If the page's on-disk LSN is already >= the log record's LSN, that modification was already persisted and can be skipped. This LSN-based filtering makes redo idempotent and avoids unnecessary I/O.

Important: Redo replays ALL logged changes, including those from uncommitted transactions. This is because dirty pages from uncommitted transactions may have been flushed to disk before the crash (steal policy). The undo phase will clean these up.

Phase 3: Undo

The undo phase reverses all changes made by transactions that were active (uncommitted) at the time of the crash. It processes the active transaction list from the ATT in reverse order of their last LSN, following the prev-LSN chain backward through the WAL.

Each undo operation generates a Compensation Log Record (CLR) that is written to the WAL. CLRs record the undo action and contain an Undo-Next-LSN pointer to the next record to undo for that transaction. This ensures that:

  • Undo is idempotent -- if recovery crashes again during undo, the CLRs allow it to pick up where it left off
  • Undone operations are themselves logged, maintaining the write-ahead invariant

Fuzzy Checkpoints and Recovery Implications

A fuzzy checkpoint does not flush all dirty pages to disk or stop the world. Instead, it:

  • Writes a checkpoint record to the WAL containing the current ATT and DPT
  • Asynchronously flushes dirty pages in the background
  • Does not guarantee all pages are clean at checkpoint completion

This is critical for write-heavy workloads because a sharp (consistent) checkpoint that flushes all dirty pages would stall writes for seconds to minutes on a large database.

Recovery time formula: Recovery time is approximately proportional to (WAL_size_since_earliest_dirty_page / WAL_scan_throughput) + (dirty_page_count * average_flush_time). In practice, checkpoint interval is tuned so recovery completes within the SLA's downtime budget (typically 30-60 seconds for single-node databases).

Recovery Time at FAANG Scale

For systems handling millions of TPS, WAL generation can be tens of GB per minute. Recovery strategies diverge:

  • PostgreSQL: Uses fuzzy checkpoints with a configurable checkpoint_completion_target. Recovery scans WAL from the last checkpoint. For very large databases, checkpoint tuning is critical -- too-frequent checkpoints cause write amplification; too-infrequent checkpoints cause long recovery.
  • MySQL/InnoDB: Similar fuzzy checkpoint approach. InnoDB's innodb_log_file_size and innodb_flush_log_at_trx_commit directly affect recovery scope.
  • RocksDB: Uses MANIFEST file (a checkpoint-like structure) plus WAL replay. Recovery is fast because the MANIFEST captures the consistent state and only a small tail of the WAL needs replay.
  • CockroachDB / Spanner: Use Raft replication for durability. A node crash is recovered by replaying the Raft log, not a traditional ARIES-style WAL. Recovery time depends on the Raft log segment size.

Design Considerations for Interviews

When designing a storage system:
  • Always use WAL-based recovery -- it is the only proven approach for crash safety
  • Choose checkpoint intervals based on your recovery time SLA and write amplification budget
  • Consider WAL pre-allocation to avoid filesystem fragmentation during high-throughput writes
  • Group commit multiple transactions' log records to amortize fsync cost
  • In distributed systems, Raft log replay replaces ARIES for many use cases
Pitfall: Never assume that a flushed data page means the transaction is durable. Pages can be flushed at any time (steal policy). Only the WAL commit record with a subsequent fsync guarantees durability. Data pages without corresponding WAL records are considered corrupt.

Common Interview Questions

  1. Walk me through what happens when a database process crashes mid-transaction.
  2. Explain the ARIES recovery algorithm. Why are there three phases?
  3. What is the difference between redo and undo? Can you skip redo if you have all before-images?
  4. Why does ARIES replay changes from uncommitted transactions during redo?
  5. What is a fuzzy checkpoint and how does it affect recovery time?
  6. How would you reduce recovery time for a database generating 10 GB of WAL per minute?
  7. What happens if the system crashes again during the undo phase?
  8. How does crash recovery interact with replication? Does a replica need to run ARIES?
  9. Compare crash recovery strategies in PostgreSQL vs. a log-structured engine like RocksDB.
  10. Design a system where crash recovery must complete within 5 seconds. What trade-offs do you make?

What to Skip

In an interview setting, skip:
  • Exact byte-level WAL record formats for specific databases -- focus on the concepts
  • Implementation details of shadow paging recovery (ARIES is the dominant approach)
  • Filesystem-level journaling internals (ext4 journal, ZFS TXG) unless designing storage infrastructure
  • Specific PostgreSQL configuration parameters -- talk about the concepts they control instead
  • Historical recovery algorithms (No-Undo/Redo, No-Redo/Undo variants) that are not ARIES
