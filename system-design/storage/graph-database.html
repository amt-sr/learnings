Graph Database

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand:

  • What a graph database is: nodes, edges/relationships, properties
  • Why graph databases excel at relationship-heavy queries
  • How relationship traversal performance differs from JOIN-based approaches
  • Common use cases: social networks, fraud detection, recommendation engines

Definition

A graph database stores data as nodes (entities) connected by edges (relationships), where both nodes and edges can have properties. Unlike relational databases that model relationships through foreign keys and JOINs, graph databases store relationships as first-class citizens, enabling efficient traversal of connected data.

Graph Database — Social Network Example [Alice]────KNOWS────►[Bob]────KNOWS────►[Carol] │ │ │ WORKS_AT WORKS_AT WORKS_AT │ │ │ ▼ ▼ ▼ [Company A] [Company B] [Company A] │ LIKES │ ▼ [Post:42] Nodes (entities): Alice, Bob, Carol, Company A, Company B, Post:42 Edges (relationships): KNOWS, WORKS_AT, LIKES Properties on nodes: Alice: {name, age, email} Properties on edges: KNOWS: {since: 2020, strength: 0.8} Query: "Find all people Alice knows who work at Company A" → Start at Alice → follow KNOWS edges → filter by WORKS_AT = Company A → Result: Carol (2 hops from Alice) In a relational database: → SELECT p.* FROM people p JOIN knows k ON p.id = k.friend_id JOIN employment e ON p.id = e.person_id WHERE k.person_id = 42 AND e.company_id = (SELECT id FROM companies WHERE name = 'Company A') With indexes: O(log n) per join. With 6-table joins: expensive. In Neo4j: O(1) index-free adjacency — traverse pointer directly.

Index-Free Adjacency

This is the key performance advantage. Each node stores direct pointers to its adjacent nodes (edges are pointers). Traversing a relationship is O(1) — you don't need an index lookup. In a relational database, traversing a relationship requires a JOIN, which requires an index lookup: O(log n) per hop. 
  Performance comparison for multi-hop queries:

  Query: "Find all friends of friends of Alice" (3-hop traversal)

  Relational database:
    SELECT DISTINCT f2.friend_id
    FROM friends f1
    JOIN friends f2 ON f1.friend_id = f2.person_id
    WHERE f1.person_id = 42

    Each JOIN = index lookup O(log n)
    2 hops = 2 index lookups
    But with friends-of-friends explosion: could scan millions of rows
    Time: O(n * log n) where n = average friends per person

  Graph database (Neo4j):
    MATCH (alice:Person {name: "Alice"})-[:KNOWS]->()-[:KNOWS]->(fof)
    RETURN fof

    Each hop = pointer dereference O(1)
    Time: O(n) where n = actual friends traversed
    No index needed for relationship traversal

  For 5-hop queries, the gap widens dramatically.
  For 10+ hops, relational databases practically cannot do it efficiently.

Cypher Query Language

Neo4j's Cypher is the most common graph query language:

-- Create nodes and relationships
CREATE (alice:Person {name: "Alice", age: 30})
CREATE (bob:Person {name: "Bob", age: 25})
CREATE (company:Company {name: "Acme Corp"})
CREATE (alice)-[:WORKS_AT {since: 2022}]->(company)
CREATE (alice)-[:KNOWS {since: 2020}]->(bob)

-- Find coworkers of Alice
MATCH (alice:Person {name: "Alice"})-[:WORKS_AT]->(c:Company)
      <-[:WORKS_AT]-(coworker:Person)
RETURN coworker.name

-- Find mutual friends (people Alice and Bob both know)
MATCH (alice:Person {name: "Alice"})-[:KNOWS]->(mutual:Person)
      <-[:KNOWS]-(bob:Person {name: "Bob"})
RETURN mutual.name

-- Shortest path between two people
MATCH path = shortestPath(
  (a:Person {name: "Alice"})-[*]-(b:Person {name: "Frank"})
)
RETURN path

In practice: Most systems don't need a dedicated graph database. Graph-like queries can often be handled in a relational database with recursive CTEs, or in a document database with graph-like features. Dedicated graph databases are worth it when relationship traversal is the core access pattern and queries involve many hops.

Common Interview Questions

  • "What is a graph database?"
    A database that stores data as nodes connected by directed, typed relationships. Both nodes and relationships can have properties. Relationships are first-class citizens — stored as direct pointers (index-free adjacency), making multi-hop traversals O(1) per hop instead of O(log n) per JOIN.
  • "When would you use a graph database instead of a relational database?"
    When your core access pattern is traversing relationships between entities: social networks (friends-of-friends), fraud detection (shared attributes across accounts), recommendation engines (collaborative filtering), dependency graphs, or any scenario where queries involve many-hop traversals. Relational databases can handle 2-3 JOINs, but graph databases handle 5-10+ hops efficiently.
  • "What is index-free adjacency?"
    Each node in a graph database stores direct pointers to its adjacent nodes. Traversing a relationship means following a pointer — O(1) — no index lookup needed. In a relational database, traversing a relationship requires a JOIN, which requires an index lookup — O(log n). For multi-hop queries, the gap compounds: k hops = k index lookups in SQL vs k pointer dereferences in a graph DB.
  • "How would you implement friend recommendations with a graph database?"
    Store people as nodes, friendships as :KNOWS edges. For recommendations: find friends-of-friends who are not already direct friends. Cypher: MATCH (me:Person)-[:KNOWS]->(friend)-[:KNOWS]->(fof) WHERE NOT (me)-[:KNOWS]->(fof) RETURN fof.name, COUNT(*) AS mutual_friends ORDER BY mutual_friends DESC. This is a 2-hop traversal, O(1) per hop.
