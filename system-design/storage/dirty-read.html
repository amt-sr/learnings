Dirty Read

Difficulty ⭐⭐☆☆☆ Interview Frequency ★★★★☆

Goal

Understand what a dirty read is, why it is the most dangerous concurrency anomaly, and how every isolation level above Read Uncommitted prevents it.

Definition

A dirty read occurs when a transaction reads data that has been modified by another transaction but not yet committed. If the modifying transaction rolls back, the reading transaction has seen data that was never permanently stored in the database -- data that represents a transient, possibly invalid state.

Core problem: The value you read may never have existed in the committed state of the database. You are making decisions on phantom data.

Anatomy of a Dirty Read

Dirty Read Timeline ───────────────────────────────────────────────────────────────── Transaction A (Transfer) Transaction B (Report) ──────────────────────────── ────────────────────────── BEGIN; UPDATE accounts SET balance = 500 WHERE id = 1; -- balance changed from 1000 to 500 -- NOT yet committed -- in-memory only, not yet written to WAL as committed BEGIN; SELECT balance FROM accounts WHERE id = 1; -- reads 500 (DIRTY READ!) -- This value may never be committed ROLLBACK; -- Transfer failed, balance reverts to 1000 -- The value 500 never existed in the committed database -- Transaction B already used -- the value 500 for its report COMMIT;

What Happens on Rollback

When the uncommitted transaction rolls back, the dirty value disappears. The reading transaction has already acted on data that never became permanent:

  • Phantom data: The value (500) was never committed. It existed only transiently in the modifying transaction's private workspace.
  • Non-reproducible bugs: The anomaly occurs only when Transaction A rolls back at exactly the right moment. Under normal conditions (commit), the value would be correct. This makes the bug extremely difficult to reproduce and debug.
  • Cascading corruption: If Transaction B makes decisions based on the dirty value and writes new data, the corruption spreads to other transactions and tables.
  • Integrity violations: Business invariants can be violated because the system operated on data that was never real.

Real-World Impact

Banking: Incorrect Balance Display

-- Transaction A: Transfer fails mid-way
BEGIN;
UPDATE accounts SET balance = 0 WHERE id = 1;  -- deduct all
-- Error: insufficient funds in source account
ROLLBACK;

-- Transaction B: User sees balance = 0 (dirty read)
-- User panics, thinks their money was stolen
-- In reality, balance was never changed

Inventory: Overselling

-- Transaction A: Adding items fails
BEGIN;
INSERT INTO inventory (product_id, qty)
  VALUES (42, 100);    -- adding stock
-- FK constraint violation → ROLLBACK

-- Transaction B: Reads the uncommitted qty = 100
-- Allows customer to order 100 items
-- But stock was never actually added!

Authentication: False Login

-- Transaction A: Admin activating account
BEGIN;
UPDATE users SET active = TRUE WHERE id = 123;
-- Something goes wrong → ROLLBACK

-- Transaction B: Reads active = TRUE (dirty read)
-- Allows user to log in with unactivated account

Which Isolation Levels Prevent Dirty Reads?

Isolation Level Dirty Read
Read Uncommitted Possible
Read Committed Prevented
Repeatable Read Prevented
Snapshot Isolation Prevented
Serializable Prevented
The fix: Read Committed is sufficient to prevent dirty reads. This is why Read Committed is the minimum acceptable isolation level for virtually all production systems. Only Read Uncommitted allows dirty reads, and it is almost never used in production.

How MVCC Prevents Dirty Reads

MVCC databases prevent dirty reads without any locking. Each row version carries a "created by transaction" marker. Readers simply skip versions from uncommitted transactions.

MVCC approach: every read sees only committed versions Row "accounts id=1": ┌──────────────────────────────────────────────┐ │ Version 1: balance=1000 (committed by txn 99)│ │ Version 2: balance=500 (created by txn 101, │ │ NOT committed yet) │ └──────────────────────────────────────────────┘ Transaction B reads this row: → Sees Version 1 (balance=1000) -- it's committed → Ignores Version 2 -- it's not committed → No dirty read, no blocking, no waiting If txn 101 commits → Version 2 becomes visible to new transactions If txn 101 rolls back → Version 2 is discarded

Common Interview Questions

  1. What is a dirty read and how do you prevent it?
  2. Can you give a concrete example of a dirty read in a real system?
  3. Why is dirty read considered the most dangerous concurrency anomaly?
  4. How does MVCC prevent dirty reads without locking?
  5. Why is Read Uncommitted almost never used in production?
  6. What happens to a transaction that performed a dirty read when the writer rolls back?

What to Skip

Depth to avoid in interviews:
  • Low-level details of how specific databases implement dirty read prevention beyond the MVCC overview
  • Read Uncommitted internals (since no production database recommends it)
  • Lock-based implementations since virtually all modern databases use MVCC
