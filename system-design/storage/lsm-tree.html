Definition

A Log-Structured Merge tree is a storage engine that converts all writes into sequential appends, batching them in memory and periodically flushing sorted immutable files to disk. Instead of updating data in-place (like a B+ tree), the LSM tree writes new data to an in-memory structure, then flushes sorted runs to disk, merging them in the background.

The Write Path (Why LSM Trees Are Fast)

Core idea: All writes are sequential appends вҖ” no in-place updates, no page reads before writes, no page splits. This makes writes extremely fast, especially on both HDDs and SSDs.
  1. Append to WAL вҖ” Write-ahead log provides durability. Sequential write to disk.
  2. Insert into Memtable вҖ” In-memory sorted structure. Fast O(log n) or O(1).
  3. Flush to SSTable вҖ” When memtable fills up, write entire sorted contents to disk as an immutable SSTable. Sequential write.
  Write latency comparison:
  B+ tree (random update):  read page вҶ’ modify вҶ’ write page вүҲ 2-3 I/Os
  LSM tree (append):        append to WAL + insert into memtable вүҲ 1 sequential I/O

  Throughput: LSM tree can be 10-100x faster for write-heavy workloads.

The Read Path (Why LSM Trees Can Be Slow)

Reading requires checking multiple levels, from newest to oldest:

  GET(key = "user:42")

  1. Check memtable          вҶ’ not found (or found вҶ’ return latest)
  2. Check SSTable Level 0   вҶ’ not found (multiple files, newest first)
  3. Check SSTable Level 1   вҶ’ not found
  4. Check SSTable Level 2   вҶ’ found! return value

  Worst case: check every level вҶ’ read amplification
Read amplification: In the worst case, every read must check memtable + all SSTable levels. With 7 levels, that's up to 7 disk seeks. B+ trees need only 3-4. This tradeoff вҖ” fast writes, potentially slower reads вҖ” is the fundamental LSM tree design decision.

Choosing between them: Use B+ trees when reads dominate or you need strong range query performance. Use LSM trees when writes dominate (time-series, logging, chat messages, IoT telemetry) or when you need high write throughput on commodity hardware.

Common Interview Questions

  • "Why would you choose an LSM tree over a B+ tree?"
    LSM trees convert random writes to sequential writes, which are 10-100x faster on both SSDs and HDDs. Ideal for write-heavy workloads: event logging, IoT data ingestion, chat history, metrics collection. The tradeoff is higher read latency and write amplification from background compaction.
  • "What is write amplification in an LSM tree?"
    Each key-value pair is written multiple times: once to the WAL, once when the memtable flushes to Level 0, and once per compaction level it passes through. With size ratio 10 and 7 levels, a single write gets rewritten ~7-10 times total. This is the LSM tree's main cost.
  • "How does an LSM tree handle reads efficiently?"
    Bloom filters (see Bloom Filter) are critical вҖ” they sit in front of each SSTable and quickly eliminate SSTables that definitely don't contain the key, avoiding expensive disk reads. With good bloom filters, most reads hit the memtable or only 1-2 SSTable levels.
  • "Explain the role of the WAL in an LSM tree."
    The Write-Ahead Log (WAL) is an append-only file on disk that records every write before it's applied to the in-memory memtable. If the process crashes, the WAL is replayed to reconstruct the memtable. The WAL is the durability guarantee вҖ” data is safe once it's in the WAL, even if the memtable hasn't been flushed.
