Isolation Levels

Difficulty: ★★★★☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What isolation is and why it is the "I" in ACID
  • The four standard SQL isolation levels and their trade-offs
  • Which concurrency anomalies each level prevents
  • Default isolation levels in popular databases
  • Why stronger isolation is not always better

Definition

Isolation determines how and when the changes made by one transaction become visible to other concurrently running transactions. It is the mechanism that prevents dirty reads, non-repeatable reads, and phantom reads.

Isolation is a spectrum: higher isolation means fewer anomalies but lower concurrency (and thus lower throughput). Every system must choose where to land on this spectrum based on its consistency requirements and performance targets.

Why Not Always Use Serializable?

Serializable isolation guarantees correctness but at a steep cost:

  • Locking overhead: Traditional implementations acquire locks on entire ranges of data, blocking all concurrent access
  • Deadlocks: More locks mean more potential for deadlocks, requiring retry logic
  • Throughput reduction: Serializing access means transactions run as if sequentially, negating concurrency benefits
  • False positives: Even optimistic serializable implementations may abort transactions that were actually safe
-- Example: Serializable might abort this legitimately safe transaction
-- because it detected a potential conflict
BEGIN TRANSACTION ISOLATION LEVEL SERIALIZABLE;
  SELECT balance FROM accounts WHERE id = 1;   -- reads 1000
  SELECT balance FROM accounts WHERE id = 2;   -- reads 2000
  -- another transaction modifies account 3 here
  -- this transaction gets aborted due to serialization failure
COMMIT;

Isolation vs. the Real World

In practice, most applications rely on Read Committed or Repeatable Read and handle edge cases at the application level:

  • Read Committed: Safeguards against reading garbage (uncommitted data) while allowing reasonable concurrency
  • Repeatable Read: Ensures a consistent view during a transaction, preventing surprises from concurrent updates
  • Serializable: Reserved for financial systems, inventory management, or any domain where anomalies are unacceptable
Design principle: Start with the database default. Escalate to a stronger isolation level only when you can demonstrate a specific anomaly that the default does not prevent. Premature use of serializable isolation is a common source of performance problems.

Common Interview Questions

  • "What are the ACID properties and what does isolation guarantee?"
    Isolation ensures that concurrently executing transactions do not interfere with each other. It prevents three classes of anomalies: dirty reads, non-repeatable reads, and phantom reads. The SQL standard defines four levels with increasing strictness.
  • "Why doesn't everyone just use Serializable isolation?"
    Serializable has the highest performance cost. It either requires locking entire data ranges (pessimistic) or aborting and retrying transactions that might conflict (optimistic). For most workloads, the anomaly risk at Read Committed or Repeatable Read is acceptable and the performance gain is significant.
  • "What is the default isolation level in MySQL and PostgreSQL, and why are they different?"
    MySQL/InnoDB defaults to Repeatable Read because it was designed with MVCC from the start, making the cost of repeatable reads low. PostgreSQL defaults to Read Committed because it prioritizes throughput and uses explicit locking or serializable for cases that need it.
  • "How does Snapshot Isolation relate to the SQL isolation levels?"
    Snapshot Isolation (SI) is not one of the four SQL standard levels. It falls between Repeatable Read and Serializable: it prevents dirty reads, non-repeatable reads, and phantom reads, but can still exhibit write skew. PostgreSQL's Repeatable Read is actually implemented as SI. See: Snapshot Isolation
