Tracing

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand how distributed tracing tracks a single request as it flows through multiple services, the span model, context propagation between services, and how OpenTelemetry standardizes instrumentation.

Definition

Distributed tracing tracks the journey of a single request across multiple services, recording each operation as a "span" with timing, metadata, and causality. A trace is the complete tree of spans for one request; a span represents a single unit of work within that trace.

Tracing fills the gap between metrics (aggregated) and logs (per-event) by showing the end-to-end path and timing of an individual request. It is the primary tool for identifying latency bottlenecks, understanding service dependencies, and debugging cascading failures.

Context Propagation

When Service A calls Service B, the trace context (trace ID, span ID, flags) must be propagated via wire protocols. This is called context propagation.

# HTTP headers (W3C Trace Context standard)
traceparent: 00-abc123def456789012345678901234-5678901234567890-01
                │  │                         │             │
                │  trace-id (32 hex)         span-id (16)  │
                version (00)                               trace-flags

# Also carried in:
# - gRPC metadata
# - Kafka message headers
# - AMQP properties

Propagation is the hardest part: Every internal call (HTTP, gRPC, message queue, database client) must propagate the trace context. A single missed propagation point creates a "gap" in the trace that hides latency and errors. This is why auto-instrumentation libraries (OpenTelemetry) are critical.

OpenTelemetry Components

  • API: Language-agnostic interfaces for traces, metrics, logs. Defines types like Tracer, Span, Meter.
  • SDK: Actual implementation that handles context propagation, sampling, batching, and exporting.
  • Collector: Vendor-neutral intermediary. Receives telemetry in OTLP format, processes (filter, enrich, transform), and exports to one or more backends.
  • Auto-instrumentation: Zero-code libraries that automatically create spans for HTTP, gRPC, database calls, Redis, Kafka, etc.
  • Semantic Conventions: Standard attribute names (http.method, db.system, messaging.system) for consistency across services.

Sampling Strategies

StrategyDescriptionProsCons
Always OnRecord every traceComplete visibilityUnsustainable at scale
Always OffRecord nothingZero overheadNo observability
ProbabilitySample N% of traces (e.g., 1%)Predictable volumeMay miss rare errors
Rate LimitingSample up to N traces per secondBounded costProportional to volume, not errors
Tail-basedBuffer all spans, decide after seeing full traceKeep error traces alwaysComplex, resource-intensive
AdaptiveAdjust rate based on traffic and error rateBest of both worldsMost complex to implement
Interview tip: When asked about observability, always mention the three pillars: logs (what happened), metrics (how much/often), traces (where it happened and why). Then explain how OpenTelemetry unifies them.

Common Interview Questions

  1. What is distributed tracing and why is it needed?
  2. Explain the relationship between a trace and a span.
  3. How does context propagation work across service boundaries?
  4. What is OpenTelemetry and why does it matter?
  5. How would you decide on a sampling strategy?
  6. How do you identify a latency bottleneck using traces?
  7. What is the difference between head-based and tail-based sampling?
  8. How do traces relate to logs and metrics?

What to Skip

Avoid diving into: OpenTelemetry SDK configuration code, specific backend setup (Jaeger deployment YAML), or trace storage format details. Focus on the conceptual model (trace/span/propagation) and architectural decisions (sampling, auto-instrumentation vs manual).
