System Design / Observability / 11.1 Logging

Structured Logging

Difficulty: ⭐⭐ Interview Frequency: ★★★★

Goal

Design a logging architecture that provides structured, searchable, and correlated log data across a distributed system. Understand how logs fit into the three pillars of observability (logs, metrics, traces) and how to make logs actionable for incident response.

Definition

Structured logging emits log events as machine-parsable data (typically JSON) rather than unstructured text strings. Each log entry is a structured record with predefined fields: timestamp, level, service name, trace ID, message, and arbitrary key-value pairs for context.

Structured logs enable programmatic querying, filtering, and aggregation across thousands of services. In distributed systems, unstructured logs are nearly useless at scale because grep-style searching cannot efficiently correlate events across services or filter by semantic fields.

Log Levels

LevelSemanticsTypical Use
TRACEFinest-grained debuggingEntry/exit of every function
DEBUGDevelopment-time diagnosticsVariable state, decision branches
INFONormal operational eventsService started, request completed
WARNPotential problems, degraded stateRetry triggered, circuit breaker open
ERRORFailures requiring attentionUnhandled exception, DB connection lost
FATALCatastrophic failure, process exitingCannot bind to port, OOM kill
Production guidance: Set default level to INFO. Enable DEBUG/TRACE only for specific services or request IDs. ERROR-level logs should trigger alerts. WARN logs signal degradation before it becomes an outage.

Structured vs Unstructured

// Unstructured (bad for production at scale)
logger.error("User 12345 failed to checkout: payment timeout after 30000ms")

// Structured (queryable, filterable, aggregatable)
logger.error({
  "event": "checkout_failed",
  "user_id": 12345,
  "reason": "payment_timeout",
  "timeout_ms": 30000,
  "trace_id": "abc123def456",
  "service": "order-service",
  "timestamp": "2025-06-28T14:32:01.003Z"
})

Log Aggregation: ELK Stack

Architecture

Services ──▶ Filebeat / Fluentd ──▶ Logstash ──▶ Elasticsearch ──▶ Kibana (collect) (parse/ (index + (visualize/ enrich) store) query) │ ▼ (optional) Kafka / buffer queue (decouple spikes)

Components

ComponentRoleWhy It Matters
ElasticsearchDistributed search engine, stores indexed logsSupports full-text search, structured queries, aggregations
LogstashPipeline: ingest, parse, enrich, transformAdds geo-IP, parses unstructured logs into fields, drops noise
KibanaVisualization dashboardQuery builder, charts, saved searches, alerting
Filebeat / FluentdLog shipper on each host/containerLightweight agent, tails log files, sends to Logstash or ES directly
Modern alternative: Many teams replace ELK with Grafana Loki + Grafana. Loki indexes only metadata (labels), not full log text, making it significantly cheaper at scale. Query using LogQL. Trade-off: full-text search is slower without full indexing.

Log Correlation

Logs become exponentially more valuable when correlated across services. Three correlation mechanisms:

  1. Trace ID propagation: Inject trace_id into every log line. When a request spans 5 services, all logs share the same trace_id, enabling a unified timeline view.
  2. Correlation ID (X-Request-ID): Header-based ID propagated through the entire request chain. Often embedded in the trace context.
  3. Span ID: Identifies a specific operation within a trace. Maps to a single service's processing of the request.
Request arrives with trace_id=abc123 API Gateway ──trace_id: abc123──▶ User Service ──trace_id: abc123──▶ Order Service [span: s1] [span: s2] [span: s3] log: "auth OK" log: "user found" log: "order created" {trace_id: abc123, span: s1} {trace_id: abc123, span: s2} {trace_id: abc123, span: s3} Kibana query: trace_id:"abc123" → Returns all 3 log entries in timeline order across all services

Log Sampling and Cost Control

At scale, logging volume is a top-3 infrastructure cost. Strategies:

  • Head-based sampling: Sample a fixed percentage (e.g., 10%) of all requests. Simple but loses rare error patterns.
  • Tail-based sampling: Buffer all logs for a trace, keep the full trace if any span is an error. Complex but preserves error context.
  • Dynamic level adjustment: Use a configuration service to dynamically raise/lower log levels per-service without redeployment.
  • Log retention tiers: Hot storage (7 days, full search) → Warm (30 days, metadata only) → Cold (1 year, archive only).
Interview tip: When asked "how would you debug a production issue?", mention structured logging with trace correlation as the first step. Being able to say "I'd search Kibana for the trace_id" demonstrates operational maturity.

Common Interview Questions

  1. What is structured logging and why is it better than unstructured logging?
  2. How would you aggregate logs from 100 microservices?
  3. How do you correlate logs across services in a distributed system?
  4. What are the log levels and when would you use each?
  5. How would you control log costs in a high-throughput system?
  6. What is the ELK stack and how do its components interact?
  7. How would you handle log aggregation during traffic spikes (10x normal)?

What to Skip

Avoid diving into: Specific logging library internals (log4j configuration, Python logging module details). These are implementation concerns. Focus on the architecture: how logs are collected, transported, stored, queried, and correlated. Also skip Syslog/rsyslog unless asked specifically about legacy infrastructure.
