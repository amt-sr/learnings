System Design / Observability / 11.4 Distributed Tracing

Distributed Tracing

Difficulty: вӯҗвӯҗвӯҗ Interview Frequency: вҳ…вҳ…вҳ…вҳ…

Goal

Understand how to trace a request across multiple services, the tools used (Jaeger, Zipkin), how to read and interpret waterfall diagrams, and the operational challenges of running distributed tracing at production scale.

Definition

Distributed tracing is the practice of instrumenting a distributed system to record the path and timing of individual requests as they traverse multiple services, databases, caches, and message queues. Each trace produces a directed acyclic graph (DAG) of spans that can be visualized as a waterfall diagram.

While the previous topic (11.3) covered tracing fundamentals (span model, context propagation, OpenTelemetry), this topic focuses on the operational side: how traces are stored, queried, and visualized in production.

Reading a Waterfall

  • Total width of the root span = end-to-end request latency
  • Nested spans show synchronous sub-operations (child waits for parent)
  • Adjacent spans at the same level show parallel operations
  • Red/error spans indicate failures that contributed to the trace outcome
  • Gaps between child spans indicate time spent in application code (not I/O)
  • The critical path is the longest chain of sequential spans вҖ” optimizing this chain reduces total latency

Storage and Retention

Cost reality: At 10,000 requests/sec with 8 spans each, you generate 80,000 spans/sec. At ~1KB per span, that is 80MB/sec or 6.9TB/day. Storage strategy is critical.
  • Hot tier: 7 days, indexed for fast query by trace_id and tags. Elasticsearch or Cassandra.
  • Warm tier: 30 days, searchable but slower. Compressed in object storage.
  • Cold tier: 90-365 days, archived. Only accessible via batch jobs. S3/GCS with Parquet format.
  • Sampling to reduce volume: Keep 100% of error traces, 1% of success traces.

Trace-Based Alerting

Beyond dashboards, traces enable automated analysis:

  • Slower than baseline: Alert when p99 for an endpoint exceeds 2x its rolling 7-day average.
  • Dependency anomalies: Detect when Service C starts appearing in traces for endpoint A but did not last week (regression).
  • Error pattern detection: Spike in specific error codes on a downstream dependency.
  • Critical path changes: The dominant latency source shifted from DB to cache (potential cache outage).
Interview tip: When designing a system, say "I would add distributed tracing from day one using OpenTelemetry with a Jaeger backend." Then describe what you would look for in the waterfall diagram to identify bottlenecks. This demonstrates you understand observability as a first-class architectural concern.

What to Skip

Avoid diving into: Backend-specific deployment configurations (Jaeger Helm chart values, Cassandra schema for spans), specific trace wire protocols beyond W3C Trace Context, or UI navigation details. Focus on architecture decisions: backend selection, storage strategy, sampling, and how to use traces for debugging.
