Graceful Degradation

Difficulty ⭐⭐⭐ Interview Frequency ★★★★
Goal: Understand how to design systems that continue functioning with reduced capability when components fail, rather than failing entirely. Master fallback strategies, feature prioritization, and load-shedding techniques.

Definition

Graceful degradation is the practice of designing a system so that when a non-essential component fails or becomes unavailable, the system continues to operate with reduced functionality rather than failing completely. The core value proposition remains available, while peripheral features are disabled or served from fallbacks.

Core Principle: A partially working system with reduced features is almost always better than a completely down system. Users can tolerate missing recommendations, stale data, or degraded images — they cannot tolerate a blank error page.

Degradation Levels

Level 0: Full Functionality ├── All features operational ├── All data fresh └── All integrations active Level 1: Stale Data / Cached Responses ├── Core functionality works ├── Data may be minutes/hours old ├── Non-critical integrations degraded └── Example: show cached product prices instead of live ones Level 2: Feature Disabled ├── Core functionality works ├── Non-essential features disabled ├── Static fallbacks for dynamic content └── Example: recommendations hidden, search limited to exact match Level 3: Emergency Mode (Read-Only) ├── Only reads work ├── All writes deferred/queued ├── Basic pages served └── Example: social media shows feed but cannot post Level 4: Maintenance Page ├── All functionality suspended ├── Scheduled downtime only └── Last resort — rarely needed with proper degradation

Implementation Patterns

1. Fallback Chain

function getProductPrice(productId) {
    try {
        // Level 0: Live price from pricing service
        return pricingService.getPrice(productId);
    } catch (ServiceUnavailable) {
        try {
            // Level 1: Cached price (may be stale)
            return cache.get("price:" + productId);
        } catch (CacheMiss) {
            // Level 2: Default/fallback price from database
            return db.query("SELECT base_price FROM products WHERE id = ?", productId);
        }
    }
}

2. Feature Flags for Degradation

// Global degradation level (can be set via config push in seconds)
DEGRADATION_LEVEL = config.get("degradation.level", 0);

function renderPage(user) {
    if (DEGRADATION_LEVEL >= 2) {
        // Hide non-essential features
        return renderBasicPage(user, includeRecommendations: false);
    }
    return renderFullPage(user, includeRecommendations: true);
}

3. Load Shedding

Actively reject lower-priority requests when the system is overloaded:

SignalAction
CPU > 80%Start rejecting P3 requests
CPU > 90%Reject P3 and P2 requests
Queue depth > 1000Reject non-critical async jobs
Error rate > 5%Enable all fallbacks; return cached data
Memory > 90%Stop accepting new connections; drain existing

Graceful Degradation vs Circuit Breaker

Circuit breakers enable graceful degradation. The circuit breaker detects the failure; graceful degradation defines what to do about it.

Service Call → Circuit Breaker detects failure → OPEN │ ┌──────────┼──────────┐ ▼ ▼ ▼ Return Queue Feature cached write disabled data for later in UI

Common Interview Questions

  • "How would you ensure your system stays partially available when a dependency fails?" — Graceful degradation: classify features by priority, define fallbacks for each, use circuit breakers to detect failures and trigger fallbacks.
  • "Give me an example of graceful degradation." — Netflix: if the recommendation service is down, show trending/popular content instead. If the search service is down, allow browsing by category. Core (streaming) never degrades.
  • "How would you decide what to degrade first?" — Feature priority matrix: P0 never degrades, P3 degrades first. Business impact analysis: what does the revenue impact of each feature being down?
  • "How do you implement graceful degradation?" — Fallback chains (live → cache → database), feature flags to toggle degradation levels, circuit breakers to trigger fallbacks, load shedding to reject low-priority traffic under pressure.
  • "What is load shedding?" — Actively rejecting lower-priority requests when the system is near capacity. This is different from graceful degradation: load shedding prevents overload; graceful degradation handles failures.
