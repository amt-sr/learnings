Backup Strategies

Difficulty ⭐⭐⭐ Interview Frequency ★★★★
Goal: Understand backup strategies (full, incremental, differential), snapshot vs continuous backup, the 3-2-1 backup rule, and how to design backup systems that enable recovery within RPO/RTO targets.

Definition

A backup is a copy of data stored separately from the original, intended for recovery in case of data loss, corruption, hardware failure, or disaster. Backup strategy determines what to back up, how often, how long to retain backups, and how to verify they are recoverable. A backup that cannot be restored is not a backup — it is an illusion.

Backup Types

Full Backup

Complete copy of all data. Simple but slow and storage-intensive.

Full Backup Timeline: Day 1: ████████████████████ Full (100 GB) Day 2: ████████████████████ Full (102 GB) Day 3: ████████████████████ Full (104 GB) Day 4: ████████████████████ Full (107 GB) Storage: 4 × ~100GB = ~400GB for 4 days Backup time: ~4 hours each Restore: Fast — single file to restore from

Incremental Backup

Only backs up data that has changed since the last backup of any type. Most storage-efficient but restore requires the full backup + all subsequent incrementals.

Incremental Backup Timeline: Day 1: ████████████████████ Full (100 GB) Day 2: ██ Incremental (2 GB changed) Day 3: █ Incremental (1 GB changed) Day 4: ███ Incremental (3 GB changed) Storage: 100 + 2 + 1 + 3 = 106 GB (much less!) Backup time: ~10 min each day Restore: SLOW — need Full + Day2 + Day3 + Day4 To restore Day 4: 1. Restore Full (Day 1) 2. Apply Incremental (Day 2) 3. Apply Incremental (Day 3) 4. Apply Incremental (Day 4) If ANY incremental is corrupted → data loss from that point forward

Differential Backup

Backs up all data that has changed since the last full backup. A middle ground: faster restore than incremental, less storage than full.

Differential Backup Timeline: Day 1: ████████████████████ Full (100 GB) Day 2: ████ Differential (2 GB + 1 GB prior = 3 GB) Day 3: █████ Differential (2 GB + 1 GB + 1 GB = 4 GB) Day 4: ████████ Differential (2+1+1+3 = 7 GB) Storage: 100 + 3 + 4 + 7 = 114 GB Restore: Fast — Full + latest differential only To restore Day 4: 1. Restore Full (Day 1) 2. Apply Differential (Day 4) ← only 2 steps!

Snapshot vs Continuous Backup

Storage Snapshots

Point-in-time, immutable captures of the entire storage volume. Near-instant creation (copy-on-write).

Key Advantage: Snapshots are nearly instantaneous because they use copy-on-write. Only changed blocks are stored after the snapshot. However, snapshots are tied to the storage system — if the storage system fails, snapshots may be lost too.
AspectSnapshotContinuous / Replication
Creation timeNear-instantOngoing (always running)
RPOLast snapshot pointNear-zero (seconds)
Storage locationSame storage system (risk!)Separate system / region
Use caseQuick rollback, testingDisaster recovery

The 3-2-1 Backup Rule

3-2-1 Rule: Keep at least 3 copies of your data, on at least 2 different types of storage media, with at least 1 copy off-site (different region/cloud provider).
Production Data │ ├── Copy 1: Primary replication (same region, different AZ) │ ├── Copy 2: Cross-region replication (different region, same cloud) │ └── Copy 3: Off-site backup (different cloud provider or cold storage) └── S3 Glacier, Google Coldline, or physical tape

Backup Verification

Critical Rule: An untested backup is not a backup. You must regularly restore backups to a staging environment and verify data integrity. Google's SRE team reports that untested backups fail to restore in ~30% of cases.

Common Interview Questions

  • "How would you design a backup strategy for a production database?" — Full backup weekly, incremental daily, WAL/log shipping for near-real-time recovery, cross-region replication for disaster recovery. Verify restores weekly.
  • "What is the 3-2-1 backup rule?" — 3 copies, 2 media types, 1 off-site. Prevents single-point-of-failure in backup storage.
  • "Full vs incremental vs differential — which would you choose?" — Incremental for storage efficiency + weekly full for restore reliability. Most production systems use a combination.
  • "What is RPO and how does backup strategy affect it?" — RPO is the maximum acceptable data loss. Continuous replication → RPO near zero. Daily backups → RPO of 24 hours. Choose based on data criticality.
  • "Why is testing backups important?" — 30%+ of untested backups fail to restore. Corrupted backups, format changes, missing dependencies, and incorrect procedures all cause restore failures.
  • "What about cloud-managed database backups?" — AWS RDS, Cloud SQL, Atlas all provide automated snapshots. But you still need cross-region copies and off-site backups (different cloud provider) for true disaster recovery.
