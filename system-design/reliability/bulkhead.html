Bulkhead

Difficulty вӯҗвӯҗвӯҗ Interview Frequency вҳ…вҳ…вҳ…вҳ…
Goal: Understand the bulkhead pattern (thread pool and process isolation), how it prevents one slow dependency from consuming all resources, and how it complements circuit breakers.

Definition

The Bulkhead pattern isolates different parts of a system into separate resource pools so that a failure in one part cannot cascade to others. Named after ship bulkheads вҖ” watertight compartments that prevent the entire ship from sinking if one section is breached. In software, bulkheads typically isolate threads, connections, or memory per dependency.

Bulkhead Strategies

1. Thread Pool Isolation

Each dependency gets its own dedicated thread pool with a fixed number of threads. When the pool is exhausted, new requests are immediately rejected.

// Each dependency gets its own thread pool
ThreadPoolConfig paymentPool    = new ThreadPool(50);   // max 50 concurrent calls to Payment
ThreadPoolConfig inventoryPool  = new ThreadPool(30);   // max 30 concurrent calls to Inventory
ThreadPoolConfig notificationPool = new ThreadPool(20); // max 20 concurrent calls to Notifications

// If Notification service is slow:
// - Only notificationPool fills up (20 threads)
// - paymentPool and inventoryPool are unaffected
// - New notification requests get RejectedExecutionException вҶ’ handled by fallback
StrategyProsCons
Thread Pool IsolationFine-grained control; one slow dependency only affects its own poolThread overhead; need to size pools correctly; thread contention on CPU
Process IsolationStrongest isolation; crash of one process does not affect othersHeavy resource overhead; inter-process communication cost; deployment complexity
Connection Pool IsolationPrevents one dependency from exhausting connection poolLess granular than thread pool; connection overhead

3. Connection Pool Bulkhead

Limit the number of connections available to each dependency:

// HikariCP with per-service connection pools
HikariDataSource paymentDB = new HikariDataSource();
paymentDB.setMaximumPoolSize(20);

HikariDataSource analyticsDB = new HikariDataSource();
analyticsDB.setMaximumPoolSize(10);

// If analyticsDB is slow:
// - Only 10 connections are blocked
// - paymentDB connections are completely independent

Sizing Bulkhead Pools

Pool sizing is critical. Too small: legitimate traffic is rejected. Too large: defeats the purpose of isolation.

Approach: Start with observed peak concurrent requests + 20% headroom. Monitor rejection rates and adjust. The key metric is: "What percentage of time is the pool exhausted?" Target: <0.1% rejection rate during peak.
Pool TypeSizing MethodExample
Thread pool (outbound)Peak concurrent calls to dependency Г— 1.2Peak 40 calls вҶ’ 48 threads
Connection pool (database)Formula: cores Г— 2 + effective_spindle_count16 cores, 1 SSD вҶ’ 33 connections
Semaphore (lightweight)Max concurrent + 20% bufferPeak 100 вҶ’ 120 permits

Bulkhead vs Circuit Breaker

They solve different problems: Circuit breakers detect and stop calling a failing service. Bulkheads limit the resources one dependency can consume. They are complementary вҖ” use both.
AspectCircuit BreakerBulkhead
Question"Is the service healthy?""How many resources can this dependency use?"
TriggerFailure rate exceeds thresholdResource pool exhausted
ActionStop all calls (open circuit)Reject excess calls (pool full)
RecoveryHalf-open probe after timeoutInstant вҖ” pool drains naturally
Protects againstSustained failuresSlow responses consuming resources

Common Interview Questions

  • "What is a bulkhead and how does it work?" вҖ” Resource isolation per dependency. Each downstream service gets its own thread/connection pool so one slow service cannot consume all resources.
  • "How is it different from a circuit breaker?" вҖ” Circuit breaker detects failures and stops calls. Bulkhead limits resources. They complement each other.
  • "How would you size a thread pool bulkhead?" вҖ” Based on observed peak concurrent calls to that dependency plus 20% headroom. Monitor and adjust.
  • "What happens when a bulkhead pool is full?" вҖ” New requests are immediately rejected (fast fail) with a fallback response. This preserves threads for other dependencies.
  • "Give me a real-world example of needing bulkheads." вҖ” Payment service goes slow; without bulkheads, all 200 server threads get stuck waiting for payment responses, making the entire app unresponsive even for non-payment requests.
