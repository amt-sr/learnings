System Design / Reliability / 10.4.2 Restore

Restore

Difficulty ⭐⭐⭐ Interview Frequency ★★★
Goal: Understand the restore process, how RPO and RTO drive restore strategy, the restore vs recovery distinction, testing backups regularly, and restoring at different granularities.

Definition

Restore is the process of recovering data from a backup and bringing a system back to an operational state after a failure. A restore strategy must be RPO/RTO-driven: how much data loss is acceptable (RPO), and how fast must service resume (RTO). A backup is only as good as its restore process — many organizations discover during a real disaster that their backups are incomplete, corrupted, or the restore procedure is poorly documented.

Restore vs Recovery

Key Distinction: Restore is copying data from backup to a target system. Recovery is the full process of restoring data AND bringing the application back to a functional state. Recovery includes restore + reconfiguring + re-deploying + verifying + redirecting traffic.

AssessmentDetermine scope of failure, choose correct backup, notify stakeholdersMinutes

Restore Strategies by Failure Type

Failure Type Strategy ─────────────────────────────────────────────────────── Accidental data deletion → PITR to before deletion, replay logs (human error) Duration: minutes to hours Data corruption → Restore last known-good backup (software bug) Then replay clean logs after corruption time Single server failure → Failover to replica (no restore needed) (hardware) Restore is passive — replica IS the backup Region-wide failure → Restore from cross-region backup (natural disaster) Provision new infrastructure in another region Ransomware / attack → Restore from air-gapped offline backup (security) Offline backups cannot be encrypted by attacker

RPO/RTO-Driven Restore Design

Design Principle: Start with RPO and RTO requirements, then choose the backup and restore strategy that meets them. Never design a backup system and then figure out what RPO/RTO you get — work backwards from business requirements.
RPO RequirementRTO RequirementStrategy
< 1 minute< 1 minuteActive-active multi-region with sync replication
< 5 minutes< 5 minutesActive-passive with async replication + automated failover
< 1 hour< 1 hourContinuous replication + point-in-time restore
< 24 hours< 4 hoursDaily full backup + hourly incrementals
< 7 days< 24 hoursWeekly full + daily incrementals + off-site storage

Restore Testing

The #1 Rule of Backups: Test your restore process regularly. An untested backup is a hope, not a plan. Google's SRE teams conduct regular "disaster recovery drills" where they simulate a region failure and time the full recovery process.

Testing Pyramid

Restore Testing Pyramid: ▲ ╱ ╲ Full DR Drill (quarterly) ╱ ╲ Simulate complete region failure ╱─────╲ Time the full recovery + measure actual RTO ╱ ╲ ╱ Sample ╲ Application Restore Test (monthly) ╱ Restore ╲ Restore to staging, run integration tests ╱───────────────╲ ╱ Database Restore ╲ DB Restore Test (weekly) ╱ to Staging ╲ Restore latest backup, verify row counts ╱──────────────────────╲ ╱ Automated Integrity ╲ Checksum + Metadata Validation (daily) ╱ Checks (every backup) ╲ Verify backup completion, sizes, checksums

Common Restore Pitfalls

  • Missing dependencies: Restored database works but the app cannot connect because config/secrets are missing.
  • DNS caching: Traffic still points to the failed instance for hours because of DNS TTL.
  • Schema drift: Backup was from a month ago; schema has changed since. Restore fails on incompatible schema.
  • Orphan data: Restored data references objects in other services that have moved on (e.g., order data referencing a user account created after the backup).
  • License/encryption: Backup is encrypted with a key stored in the failed region. Cannot decrypt without the key.

Common Interview Questions

  • "How would you restore a database after accidental data deletion?" — PITR to before the deletion event using WAL/binlog replay. Verify integrity, cutover traffic.
  • "What is the difference between restore and recovery?" — Restore is copying data back. Recovery is the entire process: assess → provision → restore → replay → verify → cutover.
  • "How often should you test backups?" — Daily automated integrity checks, weekly database restores to staging, monthly full application restores, quarterly full DR drills.
  • "How long does it take to restore a 1TB database?" — Depends on the method: snapshot restore ~30-60 min, cross-region full backup 4-8 hours, plus replay time for incrementals.
  • "What could go wrong during a restore?" — Schema drift, missing config/secrets, DNS caching, encryption key unavailability, orphan data references, insufficient capacity in the restore target.
