System Design / Reliability / 10.2.3 Timeout

Timeout

Difficulty ⭐⭐ Interview Frequency ★★★★
Goal: Understand timeout budgets across connection/read/write phases, cascading timeout chains, deadline propagation, and how to set appropriate timeout values.

Definition

A timeout is a maximum duration allowed for an operation to complete before it is forcibly terminated and considered failed. Timeouts are the most basic fault-tolerance mechanism — without them, a single slow or dead dependency can hang all upstream callers indefinitely, exhausting threads and connections.

Types of Timeouts

Timeout Budget

When a request chains through multiple services, the total timeout budget must be allocated across each hop. This is called deadline propagation — the caller's remaining time is passed downstream.

Client Request (total budget: 5 seconds) │ ├── API Gateway Budget: 5000ms │ Remaining: 5000ms → calls AuthService with deadline = now + 4500ms │ ├── AuthService Budget: 4500ms │ Connection: 500ms │ Processing: 1000ms │ Remaining: 3000ms → calls UserDB with deadline = now + 2500ms │ ├── UserDB Query Budget: 2500ms │ Connection: 200ms │ Read: 1000ms │ Remaining: 1300ms → success │ └── Total elapsed: 2700ms (within 5000ms budget)
Key Rule: Each service must reserve some of its budget for its own processing and for returning the response upstream. If a service uses the entire remaining budget, the upstream caller cannot receive the response before its own timeout.

Cascading Timeouts

The most common timeout mistake is setting equal or larger timeouts for downstream services than upstream ones. This creates impossible situations where a downstream service "succeeds" but the response is discarded because the upstream timeout already fired.

WRONG (cascading failure): ┌──────────┐ 3s timeout ┌──────────┐ 5s timeout ┌──────────┐ │ Service A │ ────────────────▶ │ Service B │ ────────────────▶ │ Service C │ └──────────┘ └──────────┘ └──────────┘ │ │ A times out at 3s C responds at 4s │ │ └── B is still waiting for C (hasn't timed out yet) └── B returns at 4s to A, but A is already gone └── Wasted work, wasted resources CORRECT (deadline propagation): ┌──────────┐ 3s deadline ┌──────────┐ 2.5s deadline ┌──────────┐ │ Service A │ ────────────────▶ │ Service B │ ────────────────▶ │ Service C │ │ 5000ms │ (reserves 500ms│ 2500ms │ (reserves 500ms│ 2000ms │ │ total │ for response) │ remaining │ for response) │ remaining │ └──────────┘ └──────────┘ └──────────┘

Setting Timeout Values

Timeout values should be derived from observed latency data, not arbitrary numbers.

// Derive timeouts from SLO targets and latency data
// Use p99 latency as the baseline, then add safety margin

p50 latency = 50ms
p95 latency = 120ms
p99 latency = 350ms
p99.9 latency = 800ms

// Read timeout = p99.9 * 2 = 1600ms (generous safety margin)
// Connection timeout = 500ms
// Total timeout = connection + read = 2100ms

Timeout + Retry + Circuit Breaker

Timeouts should always be combined with retries and circuit breakers:

Request Flow: ┌──────────────────────────────────────────────────────────────┐ │ 1. Set overall deadline (e.g., 5s) │ │ 2. Try request with 2s timeout │ │ ├── Success → return │ │ └── Timeout → check remaining budget │ │ 3. Retry with 2s timeout (remaining budget: ~3s) │ │ ├── Success → return │ │ └── Timeout → check circuit breaker state │ │ 4. Circuit breaker says OPEN → return fallback immediately │ └──────────────────────────────────────────────────────────────┘

Deadline Propagation (gRPC/HTTP)

In distributed systems, deadlines must propagate across service boundaries:

// gRPC: deadline is propagated automatically via metadata
// HTTP: use custom header
Deadline-Propagation: remaining_ms=3200

// Each service reads the header and sets its own timeout accordingly:
downstream_timeout = min(local_default_timeout, received_remaining_ms)

Common Interview Questions

  • "How would you set timeouts for a microservices call chain?" — Calculate from observed p99 latency, propagate deadlines downstream, each hop reserves time for response.
  • "What happens if you don't set timeouts?" — Threads/connections accumulate, eventually exhausting all resources, bringing down the entire service.
  • "What is a cascading timeout?" — When upstream timeouts are shorter than downstream timeouts, the downstream succeeds but the response is wasted because upstream already gave up.
  • "How do you propagate timeouts across services?" — Deadline propagation via headers (HTTP) or metadata (gRPC), each service uses remaining budget minus safety margin.
  • "What is a good timeout for a Redis call?" — Very short (50-100ms). Redis should respond in sub-millisecond; a longer timeout masks a real problem.
