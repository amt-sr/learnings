Fixed Window Counter

Difficulty ⭐⭐ Interview Frequency ★★★★
Goal: Understand the fixed window counter rate limiting algorithm, its simplicity, the critical boundary spike problem, and the rolling window variant that mitigates it.

Definition

The Fixed Window Counter algorithm divides time into fixed-size windows (e.g., 1 minute, 1 hour) and maintains a simple counter for each window. Each request increments the counter. If the counter exceeds the limit, the request is rejected. When the window resets, the counter resets to zero.

How It Works

Time → ┃ 00:00 00:30 01:00 01:30 02:00 02:30 03:00 ┃────────────────────────────────────────────────── ┃ Window 1 (limit=100) │ Window 2 (limit=100) ┃ Counter: 80 │ Counter: 40 ┃ ← 80 requests allowed → │ ← 40 requests so far → ┃ │ ┃ At 00:59:99 → counter=80 → ALLOWED ┃ At 01:00:00 → new window → counter resets to 0 ┃ At 01:00:01 → counter=1 → ALLOWED (fresh window)

Implementation

class FixedWindowCounter {
    int limit;           // max requests per window
    int windowSeconds;   // window size in seconds
    Map<String, int> counters;  // key → count
    Map<String, long> windowStart; // key → window start time

    synchronized boolean allowRequest(String key) {
        long now = System.currentTimeMillis();
        long windowStart = (now / (windowSeconds * 1000)) * (windowSeconds * 1000);

        // New window? Reset counter
        if (windowStart != windowStart.getOrDefault(key, 0L)) {
            counters.put(key, 0);
            windowStart.put(key, windowStart);
        }

        int count = counters.getOrDefault(key, 0);
        if (count < limit) {
            counters.put(key, count + 1);
            return true;   // allowed
        }
        return false;      // limit exceeded
    }
}

Distributed Implementation (Redis)

-- Fixed window counter in Redis (atomic)
-- Key: rate_limit:{user_id}:{window}
-- Window = current_minute

local key = "rate_limit:" .. KEYS[1] .. ":" .. math.floor(now / 60)
local count = redis.call("INCR", key)
if count == 1 then
    redis.call("EXPIRE", key, 120)  -- auto-cleanup after 2 minutes
end
if count > tonumber(ARGV[1]) then
    return 0  -- rejected
else
    return 1  -- allowed
end

The Boundary Spike Problem

The critical flaw of fixed window counters. At the boundary between two windows, a client can send up to 2x the limit in a short burst.

Limit: 100 requests per minute Window 1 (00:00 - 00:59) Window 2 (01:00 - 01:59) ┃ ┃ ┃ ...requests... 99 so far ┃ 60 requests arrive at 01:00 ┃ 99 ┃ counter = 60 ┃ ┃ ┃ At 00:59: send 1 more (count=100)│ (window just reset) ┃ At 00:59: count = 100 → allowed │ ┃ ┃ ┃ Wait 1 second... │ ┃ At 01:00: new window (count=0) │ ┃ Send 100 more → count = 100 │ Total: 100 + 100 = 200 ┃ │ in 2 seconds! ┃ ████████████████████████████████ ┃ ████████████████████████ ┃ ┃ ┃ ← 100 requests in last second ┃ ← 100 requests in first second ┃ of window 1 ┃ of window 2 ┃ ┃ ┗━━━━━━━━━━━━ boundary spike ━━━━━━━━┛ 200 requests in 2 seconds when limit should be 100/min
Impact: This is a real problem in production. A malicious user can send 2x the allowed rate by timing requests at window boundaries. The solution is either rolling windows, sliding windows, or token buckets.

Rolling Window (Improved Variant)

Mitigates the boundary spike by averaging counts across the current and previous window weighted by time overlap:

// Weighted rolling window calculation
// currentWindow: window that contains the current time
// previousWindow: the window before current

long now = System.currentTimeMillis();
long elapsedInCurrentWindow = now % windowDuration;
double weight = 1.0 - (double) elapsedInCurrentWindow / windowDuration;

int currentCount = getCurrentWindowCount(key);
int previousCount = getPreviousWindowCount(key);

// Estimated count = previous weighted + current full
double estimatedCount = (previousCount * weight) + currentCount;
return estimatedCount <= limit;
Rolling Window Estimation: ┃ ←──── previous window (weight: 10%) ────→│←── current (90%) ──→ ┃ ┃███████████████████████████████████████┃┃████████████████████┃ ┃ ┃ count: 80 ┃┃ count: 30 ┃ ┃ │ ┃ estimated = 80 × 0.10 + 30 × 1.0 = 8 + 30 = 38 ┃ if limit = 100 → ALLOWED ┃ ┃ As time passes, previous window weight decreases: ┃ At 50%: estimated = 80 × 0.50 + 30 × 1.0 = 70 ┃ At 10%: estimated = 80 × 0.10 + 30 × 1.0 = 38 ┃ At 0%: estimated = 80 × 0.00 + 30 × 1.0 = 30

Common Interview Questions

  • "How does a fixed window counter work?" — Divide time into windows; increment counter per window; reject if counter exceeds limit; reset at window boundary.
  • "What is the boundary spike problem?" — At the transition between two windows, a client can send up to 2x the limit because the counter resets. Mitigate with rolling windows or token buckets.
  • "How would you implement rate limiting in Redis?" — INCR on a windowed key with EXPIRE for cleanup. Atomic, simple, scalable.
  • "How does the rolling window fix the boundary spike?" — Weighted average of current and previous window counts, where the weight of the previous window decreases linearly to zero.
  • "What is the advantage of fixed window over sliding window?" — Much simpler to implement; single counter per window; O(1) memory; easy to distribute with Redis INCR.
