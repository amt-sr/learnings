Service Level Indicator (SLI)

Difficulty ⭐⭐ Interview Frequency ★★
Goal: Understand what to measure, how to choose good SLIs, the three canonical SLI categories (availability, latency, throughput), and how SLIs feed into SLO targets.

Definition

A Service Level Indicator (SLI) is a quantitative measure of some aspect of the level of service provided. SLIs are the raw measurements that SLOs are evaluated against. A good SLI is: measurable, understandable, actionable (the team can influence it), and aligned with user experience.

The Three Canonical SLIs

┌──────────────────────────────────────────────────┐ │ SLI Categories │ ├──────────────────┬───────────────────────────────┤ │ Availability │ Ratio of successful requests │ │ "Is it working?"│ to total requests │ │ │ success_count / total_count │ ├──────────────────┼───────────────────────────────┤ │ Latency │ Ratio of requests completing │ │ "Is it fast?" │ within a threshold │ │ │ fast_count / total_count │ ├──────────────────┼───────────────────────────────┤ │ Throughput │ Ratio of achieved vs target │ │ "Can it handle │ capacity │ │ the load?" │ actual_qps / target_qps │ └──────────────────┴───────────────────────────────┘

Availability SLI

The most common SLI. Measures the ratio of successful responses.

Availability = successful_requests / total_requests

// "Successful" must be carefully defined:
// - HTTP: 2xx/3xx responses (not 4xx client errors from bad input)
// - gRPC: OK status codes
// - Exclude health check / synthetic traffic
// - Include timeout errors as failures
Gotcha: A service can be "available" (returning 200) but serving wrong data. Availability SLIs do not capture correctness. For financial systems, correctness SLIs (e.g., double-entry balance validation) are equally important.

Latency SLI

Measures how fast the service responds, typically at specific percentiles.

PercentileUse CaseWhat It Captures
p50 (median)Typical user experienceHalf of requests; too optimistic for reliability
p90Good overall feelMarginal cases, caching misses
p99SLO targetTail latency; outliers, GC pauses, queueing
p99.9Critical servicesExtreme tail; hot loops, lock contention
Latency SLI = count(requests ≤ 200ms) / total_requests

// Use a threshold that maps to user-perceived "fast"
// Not the average — averages hide tail latency

Throughput SLI

Measures whether the system can handle the expected load.

Throughput SLI = actual_requests_served / target_capacity

// Example: if the system is designed for 10k QPS
// and is serving 9.5k QPS successfully → SLI = 95%

Choosing Good SLIs

Google's SRE team identifies these criteria for useful SLIs:

Good SLI Checklist: ✓ Measures user-facing behavior, not internal metrics ✓ The team can actually affect the number ✓ It's measurable with existing or easily-added instrumentation ✓ It's sensitive enough to detect regressions ✓ It's not so noisy that it triggers false alarms ✗ Does NOT measure things users don't care about ✗ Does NOT include synthetic/health-check traffic

Where to Instrument

  • Client-side (gold standard): Captures the full request path including network, DNS, and TLS. Hardest to instrument but most accurate.
  • Gateway / Load Balancer: Captures upstream latency. Good balance of effort vs accuracy. Most teams use this.
  • Service-side: Captures processing time only. Misses network latency. Can be misleading.
  • Server-side: Captures internal processing. Useful for debugging but not for user-facing SLIs.

Common Pitfalls

Averaging: Never use mean latency for SLIs. A single outlier request does not degrade user experience as much as many slow requests. Use percentiles.
Over-filtering: Some teams exclude "client errors" (4xx) from availability. This can mask real problems — a misconfigured client sending bad requests is still a problem the service owner should see.
Too many SLIs: Having 20 SLIs makes it impossible to focus. Start with 2-3 per service and add more only when a specific failure mode justifies it.

Common Interview Questions

  • "How would you measure availability for your designed system?" — Define success criteria, instrumentation point, time window, and exclusions.
  • "What is the difference between p50 and p99 latency?" — p50 is the median (half of requests); p99 captures the tail where most user complaints come from.
  • "Why shouldn't you use average latency as an SLI?" — Averages hide outliers. A few extremely slow requests are invisible in the mean but devastating to users.
  • "Where would you instrument SLI measurements?" — Ideally at the edge/client side; pragmatically at the load balancer/gateway layer.
  • "How many SLIs should a service have?" — Start with 2-3: availability and latency at minimum. Add only when justified by failure modes.

What to Skip

Depth: Specific Prometheus/OpenTelemetry metric export configurations.
Depth: Histogram bucket sizing strategies — interviewers want the concept, not the instrumentation details.
Depth: Red-metric vs blue-metric (RED/USE) framework details unless asked.
