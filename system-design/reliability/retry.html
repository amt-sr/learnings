Retry with Backoff

Difficulty ⭐⭐⭐ Interview Frequency ★★★★★
Goal: Understand retry strategies (fixed, exponential, jittered), idempotency requirements, retry storms, and when NOT to retry.

Definition

Retry with backoff is a fault-tolerance technique where a failed operation is automatically re-attempted after a delay. The delay typically increases with each successive retry (exponential backoff) to give the failing service breathing room to recover. Without backoff, retries can amplify failures into retry storms that take down both the client and the server.

Exponential Backoff with Jitter

This is the gold standard. Jitter randomizes the delay to prevent all clients from retrying simultaneously.

// Exponential backoff with full jitter
delay = min(base_delay * 2^attempt + random(0, jitter_range), max_delay)

// Example: base=100ms, jitter_range=100ms, max=30s
// Attempt 0: random(100, 200)ms   → ~137ms
// Attempt 1: random(200, 400)ms   → ~287ms
// Attempt 2: random(400, 800)ms   → ~612ms
// Attempt 3: random(800, 1600)ms  → ~1.2s
// Attempt 4: random(1600, 3200)ms → ~2.4s
// Attempt 5: min(random(3200, 6400), 30000) → ~4.1s
Critical: Without jitter, 10,000 clients all retrying at exactly 1s, 2s, 4s create synchronized waves that can be worse than the original failure. Always add jitter.

Retry Storm: The Danger of Naive Retries

Normal load: ┃ ▃▅▇█▇▅▃ ┃ Server handles 1000 QPS fine Failure occurs: ┃ ▃▅▇█▇▅▃ ┃ Server responds slowly First retry: ┃ ▃▅▇███▇▃ ┃ 1000 original + 1000 retries = 2000 Second retry: ┃ ▃▅██████▃ ┃ 4000 requests now Third retry: ┃ █████████ ┃ Server overwhelmed, ALL fail ↑ Retry storm: retries amplify the problem by 3-5x the original load

Idempotency: The Prerequisite

Rule: You MUST ensure idempotency before implementing retries for non-read operations. A retry of a POST that creates a resource will create duplicates. A retry of a payment will charge twice.

Retry + Circuit Breaker Combination

Best practice: Combine both patterns Request → Circuit Breaker → [CLOSED: forward to service] │ ├── Service succeeds → return response │ └── Service fails → RETRY (with backoff) │ ├── Retry succeeds → return response └── All retries fail → return error (and increment CB failure count) If Circuit Breaker is OPEN: Request → Circuit Breaker → immediately return fallback/error (no retries, no downstream calls)

Common Interview Questions

  • "When would you use retries vs circuit breakers?" — Retries handle transient failures; circuit breakers handle sustained failures. Use both together.
  • "Why is jitter important in exponential backoff?" — Prevents thousands of clients from retrying simultaneously, causing a thundering herd/retry storm.
  • "How do you ensure retries don't cause duplicate charges?" — Idempotency keys, conditional writes, state machine guards, or database unique constraints.
  • "What errors should you NOT retry?" — 4xx client errors (400, 401, 404, 409). These won't succeed on retry.
  • "What happens if 10k clients all retry at once?" — Retry storm. The retry load is 10x the original, making recovery impossible. This is why you need jitter + circuit breaker.
  • "How many retries is too many?" — More than 3-5 total attempts for synchronous user-facing calls. For async/background jobs, 10-25 with longer backoff is acceptable.
