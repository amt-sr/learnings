Circuit Breaker

Difficulty ⭐⭐⭐ Interview Frequency ★★★★★
Goal: Understand the circuit breaker pattern, its three states, how it prevents cascading failures, configuration parameters, and when to use it in a system design.

Definition

The Circuit Breaker pattern monitors calls to an external service and "trips" (opens) the circuit when failures exceed a threshold. While open, requests are immediately rejected (or served from a fallback) without making the actual call. This prevents a failing downstream service from consuming upstream resources (threads, connections, memory), which would cause a cascading failure across the entire system.

Named by analogy to electrical circuit breakers that trip to prevent fires when a short circuit occurs.

State Details

StateBehaviorTransition Out
ClosedNormal operation. Requests pass through to the downstream service. A sliding window tracks success/failure counts.Opens when failure rate exceeds threshold (e.g., >50% failures in last 20 requests)
OpenCircuit is tripped. All requests are immediately rejected or served from a fallback. No actual calls to the downstream service are made.Transitions to Half-Open after a wait/delay timer expires
Half-OpenAllows a limited number of probe requests through to test if the downstream service has recovered.Closes if probes succeed; re-opens if probes fail

Fallback Strategies

When the circuit is open, the caller needs a strategy for handling the rejected request:

Implementation: Netflix Hystrix / Resilience4j

Netflix open-sourced Hystrix in 2012 (now in maintenance mode). Resilience4j is the modern, lightweight alternative.

// Resilience4j circuit breaker configuration
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)               // 50% failures → open
    .slidingWindowType(SlidingWindowType.COUNT_BASED)
    .slidingWindowSize(20)                  // last 20 requests
    .waitDurationInOpenState(Duration.ofSeconds(30))  // wait 30s
    .permittedNumberOfCallsInHalfOpenState(5)         // 5 probes
    .slowCallDurationThreshold(Duration.ofSeconds(5)) // 5s = slow
    .slowCallRateThreshold(80)             // 80% slow = failure
    .build();

Where to Place Circuit Breakers

Guideline: Place circuit breakers at every outbound dependency boundary. Each service should have its own circuit breaker — do not share circuit breakers across different downstream services.
Order Service ├── CB → Payment Service (critical — fail fast) ├── CB → Inventory Service (critical — cache fallback) ├── CB → Notification Service (non-critical — queue fallback) ├── CB → Recommendation Service (non-critical — default fallback) └── CB → Analytics Service (non-critical — skip/drop)

Common Interview Questions

  • "What is a circuit breaker and how does it work?" — Three states (closed/open/half-open), prevents cascading failures by fast-failing when downstream is unhealthy.
  • "How does it differ from a retry?" — Retries try the same call again; circuit breakers stop making the call entirely. They complement each other.
  • "What happens when the circuit is open?" — Requests are immediately rejected or served from a fallback. The downstream service gets no traffic, giving it time to recover.
  • "How do you choose the failure threshold?" — Too low: trips on transient errors. Too high: never trips. Start at 50% over last 20 requests and tune based on production data.
  • "Can a circuit breaker cause cascading failures?" — Yes, if the fallback itself is slow or resource-intensive. Fallbacks must be fast and lightweight.
  • "Where would you place circuit breakers in your designed system?" — At every outbound dependency boundary, each with its own circuit breaker and appropriate fallback strategy.
