Token Bucket Algorithm

Difficulty ⭐⭐⭐ Interview Frequency ★★★★★
Goal: Understand the token bucket algorithm for rate limiting, its parameters (capacity, refill rate), how it allows burst traffic while enforcing average rate, and implementation considerations.

Definition

The Token Bucket algorithm is a rate limiting mechanism that controls the rate of requests by maintaining a bucket of tokens. Each request consumes one token. Tokens are added to the bucket at a fixed rate (refill rate). If the bucket is empty when a request arrives, the request is rejected. The bucket has a maximum capacity, allowing burst traffic up to the bucket size.

How It Works

Token Bucket: ┌─────────────────────────────┐ │ Bucket │ │ capacity = 10 tokens │ │ │ │ ┃████████████┃ ← tokens │ │ ┃░░░░░░░░░░░┃ ← empty │ │ └─────────────┘ │ └─────────────────────────────┘ ▲ │ │ ▼ Refill rate: Each request 2 tokens/sec consumes 1 token (constant) (if available) Timeline: t=0s: Bucket has 10 tokens (full) t=0s: 8 requests arrive → consume 8 → 2 remaining t=0s: 2 more arrive → consume 2 → 0 remaining t=0s: 1 more arrives → REJECTED (no tokens) t=0.5s: 1 token refilled (0.5 × 2) t=1.0s: 2 tokens refilled (1.0 × 2) t=2.0s: 4 tokens refilled (capped at capacity of 10) Key insight: 10 requests can fire instantly (burst), then sustained at 2/sec average.

Parameters

ParameterDescriptionExample
Capacity (Burst Size)Maximum tokens the bucket can hold10 tokens
Refill RateTokens added per unit time2 tokens/second
Token CostTokens consumed per request (usually 1)1 token/request
Design Tradeoff: Capacity controls burst tolerance (how many requests can fire at once). Refill rate controls sustained throughput (average rate over time). High capacity = more burst tolerance but less protection. Low refill rate = stricter limiting.

Token Bucket Properties

  • Burst-friendly: Allows short bursts up to bucket capacity, making it suitable for real-world traffic patterns.
  • Memory efficient: Only need to store: bucket size, current tokens, last refill time. O(1) space per rate limiter.
  • O(1) per request: Each request just checks tokens and does arithmetic. No sliding windows or logs.
  • Average rate enforcement: Over time, the average rate is bounded by the refill rate.

Algorithm Implementation

class TokenBucket {
    double capacity;      // max tokens
    double refillRate;    // tokens per second
    double tokens;         // current tokens
    long lastRefillTime;   // last refill timestamp

    synchronized boolean allowRequest() {
        long now = System.currentTimeMillis();
        double elapsed = (now - lastRefillTime) / 1000.0;

        // Refill tokens (capped at capacity)
        tokens = Math.min(capacity, tokens + elapsed * refillRate);
        lastRefillTime = now;

        if (tokens >= 1.0) {
            tokens -= 1.0;
            return true;   // request allowed
        }
        return false;      // request rejected (rate limited)
    }
}

Implementation in Distributed Systems

Single-node token buckets are easy. Distributed token buckets are harder due to race conditions:

Single Node: Distributed (Redis + Lua): ┌──────────┐ ┌──────────┐ │ Service │ │ Service A│──┐ │ Instance │ │ │ │ │ [bucket] │ └──────────┘ │ └──────────┘ │ ▼ Works for single server. ┌──────────┐ ┌──────────┐ For multiple instances, each │ Redis │ │ Service B│ instance has its own bucket. │ [atomic │◀────│ │ → 3 instances = 3x the limit! │ bucket] │ │ │ └──────────┘ └──────────┘ Solution: Centralized token bucket in Redis using Lua script for atomic check-and-decrement 
-- Redis Lua script for distributed token bucket
-- KEYS[1] = rate_limit:{user_id}
-- ARGV[1] = capacity, ARGV[2] = refill_rate, ARGV[3] = now

local rate_limit_key = KEYS[1]
local capacity = tonumber(ARGV[1])
local refill_rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

local bucket = redis.call("HMGET", rate_limit_key, "tokens", "last_refill")
local tokens = tonumber(bucket[1]) or capacity
local last_refill = tonumber(bucket[2]) or now

-- Refill tokens
local elapsed = (now - last_refill) / 1000.0
tokens = math.min(capacity, tokens + elapsed * refill_rate)

if tokens >= 1.0 then
    tokens = tokens - 1.0
    redis.call("HMSET", rate_limit_key, "tokens", tokens, "last_refill", now)
    redis.call("EXPIRE", rate_limit_key, capacity / refill_rate * 2)
    return 1   -- allowed
else
    redis.call("HMSET", rate_limit_key, "tokens", tokens, "last_refill", now)
    return 0   -- rejected
end

Common Interview Questions

  • "How does the token bucket algorithm work?" — Tokens in a bucket; requests consume tokens; tokens refill at a constant rate; burst up to bucket capacity allowed.
  • "What is the difference between token bucket and leaky bucket?" — Token bucket allows bursting; leaky bucket enforces a constant output rate. Token bucket is burst-friendly, leaky bucket is smoother.
  • "How would you implement a distributed rate limiter?" — Centralized token bucket in Redis using Lua scripts for atomic operations. Consistent hashing for multi-Redis sharding at scale.
  • "What happens at the boundary of a time window?" — No boundary spike problem with token buckets (unlike fixed-window counters). The bucket state carries over continuously.
  • "How would you design a rate limiter for an API?" — Token bucket with per-user keys in Redis, Lua for atomicity, 429 Too Many Requests response with Retry-After header.
