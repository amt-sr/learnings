Sliding Window Rate Limiting

Difficulty ⭐⭐⭐ Interview Frequency ★★★★
Goal: Understand the sliding window log and sliding window counter algorithms for rate limiting, how they solve the boundary spike problem of fixed windows, their memory and performance tradeoffs, and when to choose each variant.

Definition

Sliding window rate limiting maintains a continuous time window rather than discrete fixed windows. For each incoming request, the algorithm counts all requests within the last N seconds (the window size). This eliminates the boundary spike problem entirely because the window is always the "last N seconds" regardless of alignment to discrete time boundaries.

Variant 1: Sliding Window Log

Store a timestamp for every request. To check the limit, count how many timestamps fall within the last N seconds.

Sliding Window Log (limit=5 per minute): Time → ┃──0s──10s──20s──30s──40s──50s──60s── ┃ ● ● ● ● ● ● ┃ ↑ ↑ ↑ ↑ ↑ ↑ ┃ req1 req2 req3 req4 req5 ┃ ┃ At t=55s: check last 60s → 5 requests → ALLOWED (≤5) ┃ At t=55s: new request arrives → 6 requests → REJECTED (>5) ┃ ┃ At t=65s: req1 falls outside window (older than 60s) ┃ Count = 5 (req2-req6) → ALLOWED again ┃ ┃ Memory: O(n) where n = requests in window ┃ If limit=1000/min → up to 1000 log entries per key

Implementation

class SlidingWindowLog {
    int limit;              // max requests per window
    int windowSeconds;      // window size
    Queue<Long> log;        // timestamps of recent requests

    synchronized boolean allowRequest() {
        long now = System.currentTimeMillis();
        long windowStart = now - windowSeconds * 1000;

        // Remove expired entries
        while (!log.isEmpty() && log.peek() < windowStart) {
            log.remove();
        }

        if (log.size() < limit) {
            log.add(now);
            return true;   // allowed
        }
        return false;      // rejected
    }
}

Distributed Implementation (Redis Sorted Set)

-- Sliding window log using Redis sorted set
-- Key: rate_limit:{user_id}
-- Score: request timestamp
-- Member: unique request ID

local key = "rate_limit:" .. KEYS[1]
local now = tonumber(ARGV[1])
local window = tonumber(ARGV[2])
local limit = tonumber(ARGV[3])
local requestId = ARGV[4]

-- Remove entries outside the window
redis.call("ZREMRANGEBYSCORE", key, 0, now - window)

-- Count remaining entries
local count = redis.call("ZCARD", key)

if count < limit then
    redis.call("ZADD", key, now, requestId)
    redis.call("EXPIRE", key, math.ceil(window / 1000) + 1)
    return 1   -- allowed
else
    return 0   -- rejected
end

Variant 2: Sliding Window Counter

A memory-efficient approximation that combines the simplicity of fixed windows with smoother rate limiting. Uses weighted counts from the current and previous fixed window.

Sliding Window Counter: ┃ ←── previous window ──→│←── current window ──→ ┃ ┃█████████████████████┃┃████████████████████┃ ┃ ┃ count = 70 ┃┃ count = 30 ┃ ┃ ┃ elapsed = 30% into current window ┃ weight_of_previous = 1 - 0.30 = 0.70 ┃ ┃ estimated_rate = (70 × 0.70) + (30 × 1.0) = 49 + 30 = 79 ┃ if limit = 100 → ALLOWED ┃ ┃ At 90%: estimated = (70 × 0.10) + (30 × 1.0) = 7 + 30 = 37 ┃ At 100%: estimated = (70 × 0.00) + (30 × 1.0) = 0 + 30 = 30

Tradeoffs Summary

Decision Framework:
  • Need exact accuracy + few requests per window? → Sliding Window Log
  • Need memory efficiency + smooth rate? → Sliding Window Counter or Token Bucket
  • Need burst support? → Token Bucket (always the best choice)
  • Need maximum simplicity? → Fixed Window (accept boundary spike)
  • Need traffic shaping? → Leaky Bucket

Redis Sorted Set Memory Concerns

The sliding window log with sorted sets can be memory-intensive at high limits:

// Memory per sorted set entry: ~64 bytes (timestamp + member)
// If limit = 10,000 requests/minute and 1M users:
// 10,000 × 1M × 64 bytes = 640 GB (impractical)

// Solution: For high limits, use sliding window counter instead

Common Interview Questions

  • "What is the difference between sliding window and fixed window rate limiting?" — Sliding window looks at the "last N seconds" continuously; fixed window uses discrete time boundaries. Sliding window eliminates the boundary spike problem.
  • "How would you implement sliding window rate limiting?" — Sliding window log: store timestamps in a sorted set (Redis ZSET), remove expired entries, count remaining. Sliding window counter: weighted average of current and previous window.
  • "What are the memory tradeoffs of sliding window log?" — O(n) memory where n is the number of requests in the window. At high limits (e.g., 10k/min), this becomes expensive. Sliding window counter is O(1) memory as an approximation.
  • "Compare all rate limiting algorithms." — Token bucket (burst-friendly, O(1)), fixed window (simple, O(1), boundary spike), sliding window log (exact, O(n)), sliding window counter (smooth, O(1)), leaky bucket (constant rate).
  • "How would you rate limit a distributed system?" — Centralized counter in Redis using Lua for atomicity. Choose algorithm based on requirements: token bucket for bursty traffic, sliding window for smooth limiting.
