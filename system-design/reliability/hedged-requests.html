Hedged Requests

Difficulty ⭐⭐⭐⭐ Interview Frequency ★★★
Goal: Understand the hedged request technique — sending duplicate requests to multiple replicas and returning the first response — when to use it, the cost tradeoffs, and Google's use case.

Definition

Hedged requests (also called "request hedging" or "speculative execution") is a latency optimization technique where the client sends a request to one server, and if no response is received within a timeout, sends a duplicate request to another server. The client returns whichever response arrives first and cancels the other. This is particularly effective for reducing tail latency (p99/p99.9).

The technique was popularized by Google Search, where Dean and Barroso described it in their paper "The Tail at Scale" (2013).

How It Works

Timeline: t=0ms t=50ms t=95ms t=200ms ┃ ┃ ┃ ┃ ┃ Send ┃ No response ┃ Response ┃ Cancel ┃ to A ┃ (trigger ┃ from B ┃ A (if still ┃ ┃ hedge) ┃ arrives ┃ pending) ┃ ┃ ┃ ┃ Client ──────▶ Server A (no response within 50ms) │ └──────────▶ Server B (response at 95ms) │ └──── response to client Client sees total latency: 95ms (vs 200ms+ if waiting only for Server A) Cost: 2 requests served, but only 1 response used

Two Variants

VariantBehaviorUse Case
Timeout-based HedgeSend second request after timeout on firstHigh-value reads where latency matters more than cost (search, recommendations)
Parallel HedgeSend to N replicas simultaneously, return firstCritical-path reads where cost is less important than latency (leader election, config fetch)

"The Tail at Scale" — Why It Matters

Google's key insight: even a tiny fraction of slow requests becomes a huge problem at scale.

// At 1 million QPS:
// If p99 latency = 1s and p50 = 100ms
// → 10,000 requests per second experience 1s latency
// → 10,000 users per second see a "slow" page

// With hedging:
// p99.9 of the FIRST response drops dramatically
// because the second request usually succeeds fast
Google's Finding: In their paper, Google showed that sending a single hedged request reduced p99 latency by up to 10x for certain operations. The cost was a modest increase in total resource usage (typically 2-5%).

Cost Analysis

Tradeoff: Hedged requests trade server resources for better latency. At Google's scale, a 1% increase in total request load can reduce p99 latency by 10x. But you must ensure your system has capacity for the extra load.

// Extra load = 5% of total traffic (one extra request each)

Implementation Sketch

Response hedgedGet(Request req, Duration hedgeTimeout) {
    Future<Response> first = sendAsync(replica_A, req);

    try {
        return first.get(hedgeTimeout);          // Wait for first replica
    } catch (TimeoutException e) {
        Future<Response> second = sendAsync(replica_B, req);

        // Return whichever completes first, cancel the other
        return CompletableFence.anyOf(first, second)
                                .get(overallTimeout);
    } finally {
        first.cancel(true);   // Cancel pending requests
        second.cancel(true);
    }
}

Common Interview Questions

  • "What is a hedged request?" — Send a duplicate request to another replica if the first one is slow, return whichever responds first.
  • "How does Google Search use hedged requests?" — To reduce p99 latency. The paper "The Tail at Scale" showed that a small fraction of slow requests at high QPS means many users affected, and hedging dramatically reduces tail latency.
  • "What is the cost of hedging?" — Extra server load (typically 1-5% overhead). Each hedge is a duplicate request that consumes CPU, network, and database capacity.
  • "Can you hedge write operations?" — Generally no, unless writes are idempotent. Hedging a non-idempotent write causes duplicates (double charges, double posts).
  • "How is hedging different from retrying?" — Retries are sequential (wait for failure, try again). Hedging sends a second request in parallel, and returns whichever is faster.
  • "How would you decide the hedge timeout?" — Set it at the p95 latency threshold. This means only the slowest 5% of requests trigger a hedge.
