Leaky Bucket Algorithm

Difficulty ⭐⭐⭐ Interview Frequency ★★★★
Goal: Understand the leaky bucket algorithm for rate limiting, how it enforces a constant output rate, its two variants (queue-based vs metered), and when to choose it over token bucket.

Definition

The Leaky Bucket algorithm controls the rate of outgoing requests by processing them at a constant fixed rate, like water leaking from a bucket with a hole. Incoming requests are added to a queue (the bucket). If the bucket overflows (queue is full), excess requests are rejected. The bucket "leaks" requests at a constant rate regardless of the input rate.

Two Variants

Variant 1: Queue-Based (Classic Leaky Bucket)

Requests enter a FIFO queue. The queue is drained at a fixed rate. If the queue is full, new requests are rejected.

Queue-Based: Request → [Queue] → (fixed rate processor) → Server Properties: - Constant output rate (smoothing effect) - Requests may experience queuing delay - Bursty input → steady output - If queue full → reject immediately

Variant 2: Metered (Leaky Bucket as Counter)

Similar to token bucket but inverts the logic. The "water level" rises with each request and drains at a fixed rate. If the water level exceeds capacity, the request is rejected.

Metered Variant: Request → check water level → if below capacity: allow + add water → if at capacity: reject Water drains at fixed rate: level = max(0, level - elapsed_time × drain_rate) Properties: - No queue; requests are immediately allowed or rejected - No queuing delay for accepted requests - Essentially equivalent to token bucket (mathematically)

Interview Tip: Most interviewers asking about "leaky bucket" actually want you to compare it with token bucket. Know the key difference: leaky bucket smooths traffic (constant output), token bucket allows bursts (variable output). Token bucket is generally preferred for API rate limiting because bursts are common in real traffic.

Implementation

class LeakyBucket {
    int capacity;        // max queue size
    double drainRate;    // requests per second
    Queue<Request> queue;
    double waterLevel;   // current level (metered variant)
    long lastDrainTime;

    synchronized boolean allowRequest(Request req) {
        long now = System.currentTimeMillis();
        double elapsed = (now - lastDrainTime) / 1000.0;

        // Drain the bucket
        waterLevel = Math.max(0, waterLevel - elapsed * drainRate);
        lastDrainTime = now;

        if (waterLevel < capacity) {
            waterLevel += 1.0;
            return true;   // request allowed
        }
        return false;      // bucket overflow — reject
    }
}

Common Interview Questions

  • "How does the leaky bucket algorithm work?" — Requests enter a queue; the queue drains at a constant rate; overflow causes rejection. It smooths bursty input into steady output.
  • "What is the difference between leaky bucket and token bucket?" — Leaky bucket has constant output rate (no burst). Token bucket allows bursts up to bucket capacity. Token bucket is generally better for API rate limiting.
  • "When would you choose leaky bucket over token bucket?" — When you need traffic shaping (smooth output) rather than rate limiting (burst protection). E.g., controlling egress traffic to a downstream service.
  • "Does the leaky bucket cause queuing delay?" — Yes, in the queue-based variant. Requests wait in the queue until they are processed. The metered variant does not queue.
  • "How do you handle the boundary spike problem?" — Leaky bucket does not have the boundary spike problem because there are no time windows. The bucket state is continuous.
