Recovery Point Objective (RPO)

Difficulty ⭐⭐⭐ Interview Frequency ★★★★
Goal: Understand RPO as a business-driven metric for acceptable data loss, how it drives backup frequency and replication strategy, and the relationship between RPO and RTO.

Definition

Recovery Point Objective (RPO) is the maximum acceptable amount of data loss, measured in time. It defines how far back in time you must be able to restore data after a failure. If RPO = 1 hour, the system must be able to restore data from no more than 1 hour before the failure. Any data written in the last hour may be permanently lost.

RPO answers the question: "How much data can we afford to lose?" — Not "How fast can we recover?" (that is RTO). RPO is about data freshness at the point of recovery.

How RPO Drives Architecture Decisions

RPO Requirement → Architecture Strategy RPO = 0 (zero data loss) │ → Synchronous multi-region replication │ → Distributed consensus (Raft/Paxos) │ → Cost: Highest (2x write latency at minimum) │ RPO < 1 minute │ → Semi-synchronous replication │ → WAL shipping with very short delay │ → Cost: High │ RPO < 15 minutes │ → Async replication with streaming │ → Continuous backup (e.g., DynamoDB PITR) │ → Cost: Moderate │ RPO < 1 hour │ → Frequent incremental backups (every 15-30 min) │ → Async replication │ → Cost: Low │ RPO < 24 hours → Daily full backup → Async replication → Cost: Lowest

RPO vs RTO

They measure different things: RPO = how much data you can lose (backward in time). RTO = how fast you can recover (forward in time). Both are business requirements that drive different architecture decisions.

Calculating Cost of Data Loss

RPO should be determined by the cost of losing data, not arbitrary:

// Cost calculation framework:
//
// Data loss cost = (number of affected records) × (cost per record)
//
// Example: Payment service
// - 1000 transactions per minute
// - Average transaction value: $50
// - RPO = 5 minutes → potential loss: 5000 × $50 = $250,000
//
// Cost of achieving RPO < 5 min (sync replication):
// - $50,000/year in infrastructure
//
// Trade-off: $50K/year infrastructure vs $250K potential loss per incident
// → Sync replication is justified

// Example: Analytics pipeline
// - 1M events per minute
// - Each event: ~100 bytes (essentially free)
// - RPO = 1 hour → 60M events lost (recreatable from raw logs)
// → Async replication is sufficient

Common Interview Questions

  • "What is RPO?" — The maximum acceptable amount of data loss, measured in time. RPO = 5 minutes means you can lose up to 5 minutes of data.
  • "How does RPO differ from RTO?" — RPO is about data loss (backward). RTO is about downtime (forward). They drive different architecture decisions.
  • "How would you achieve RPO = 0?" — Synchronous multi-region replication where a write only succeeds after all replicas acknowledge. This adds write latency but guarantees no data loss.
  • "How does replication lag affect RPO?" — Replication lag IS the RPO for replica-based recovery. If lag is 5 seconds, RPO is 5 seconds when failing over to the replica.
  • "How would you determine the right RPO for a service?" — Business impact analysis: what is the cost per lost record? Multiply by expected loss window. Compare with infrastructure cost to achieve a lower RPO.
  • "What RPO would you set for a payment processing system?" — RPO = 0. Every financial transaction must be durable before acknowledging to the customer. Use synchronous replication or a distributed consensus log.
