Virtual Nodes (VNodes)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆
Goal: Understand how virtual nodes solve the data skew problem of basic hash rings by mapping each physical shard to multiple positions on the ring, enabling uniform distribution and granular rebalancing.

Definition

Virtual nodes (VNodes) are multiple logical representations of a single physical shard on the hash ring. Instead of placing each physical shard at one position, you place it at many positions (typically 100-256). Each vnode is assigned a small random range of the hash space, and all vnodes belonging to the same physical shard store data on the same physical machine. VNodes are also called "tokens" in Cassandra terminology.

Why Basic Hash Rings Need VNodes

With a naive hash ring (one position per physical shard), distribution depends entirely on how the hash function maps shard IDs to ring positions. With few shards, it is common to get severe skew: one shard might own 70% of the key space while another owns only 5%. VNodes fix this by statistical averaging -- with 256 random positions per physical node, the law of large numbers ensures near-uniform distribution.

How VNodes Work

Assignment

  1. Choose the number of vnodes per physical shard (e.g., 256 for Cassandra).
  2. For each physical shard, generate N vnode identifiers: shard_A_vnode_1, shard_A_vnode_2, ..., shard_A_vnode_256.
  3. Hash each vnode identifier to get its position on the ring.
  4. Assign each data key to the nearest vnode clockwise, exactly as in a basic hash ring.
  5. The physical shard owning that vnode handles the data.

Lookup

// Pseudocode for key lookup with VNodes
function findShard(key, ring) {
    keyHash = consistentHash(key)
    vnode = ring.clockwiseNext(keyHash)  // find nearest vnode
    return vnode.physicalShard            // return the physical machine
}

Distribution Characteristics

VNodes per Shard Expected Max Skew Use Case
1 (no vnodes) Can be 5-10x off Small clusters, testing
10-50 ~20% off ideal Medium clusters
100 ~10% off ideal Riak default (160)
256 ~5-7% off ideal Cassandra default
Tradeoff: More vnodes means better distribution but higher metadata overhead. Each vnode adds an entry to the ring metadata that all clients/nodes must maintain. For Cassandra with 256 vnodes per node across 100 nodes, that is 25,600 vnode entries.

Rebalancing with VNodes

When adding or removing a physical shard, only its vnodes are affected. The rebalancing granularity is per-vnode rather than per-physical-node:

  • Adding a shard: Its 256 vnodes are inserted at random ring positions. Each vnode takes over a small slice from the existing vnode that previously owned it. Data movement is spread evenly across all existing shards.
  • Removing a shard: Its 256 vnodes are removed. Each vnode's data is redistributed to the nearest remaining vnodes. No single shard absorbs a disproportionate load.
Key Benefit: Without vnodes, removing one shard means the next shard clockwise inherits all of its data -- a massive spike. With vnodes, the load of 256 small slices is distributed across multiple physical nodes.

What to Skip

Avoid going deep into:
  • Cassandra token allocation algorithms (vnode placement strategies like random vs. rack-aware) unless the interview context is specifically Cassandra.
  • Weighted vnodes or heterogeneous hardware scenarios -- rarely discussed in interviews.
  • Specific vnode gossip protocol details.
