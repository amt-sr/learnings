Hot Keys

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★
Goal: Understand why hot keys are a fundamental weakness of hash-based partitioning, recognize real-world hot key scenarios, and apply multiple mitigation strategies from local caching to key splitting.

Definition

A hot key is a single data key that receives disproportionately high read or write traffic compared to other keys. Because hash-based partitioning maps a single key to exactly one shard, all traffic for that key lands on a single machine. No amount of additional shards or vnodes helps -- the hot key's shard becomes a bottleneck.

Core Problem: Consistent hashing distributes keys uniformly across shards. But it cannot distribute access patterns uniformly. If 10% of your traffic targets one key, 10% of all traffic hits one shard, regardless of how many shards you have.

Real-World Hot Key Examples

Scenario Hot Key Traffic Pattern

Gaming leaderboard Top-10 global ranking key Every player reads this key

Shard B: 33% traffic Shard B: 15% traffic Shard C: 33% traffic Shard C: 15% traffic Even with 100 shards: Still 70% on Shard A! Shard A: 70% Shard B: 0.3% Shard C: 0.3% ... (98 more shards at ~0.3%) The hot key ALWAYS goes to ONE shard. Hash(user_key) is deterministic -- same key = same shard.

Why Consistent Hashing Does Not Help

Consistent hashing ensures that each key maps to exactly one shard. This is a feature for data consistency but a limitation for hot keys. Adding more shards spreads the other keys more thinly, but the hot key remains concentrated on its single shard. The problem is at the key level, not the shard level.

Solutions

Solution 1: Split Hot Key (Key Splitting)

Divide the hot key into multiple sub-keys, each stored on a different shard. Combine results at read time.

// Instead of:
key = "tweet:12345"           // all reads hit one shard

// Split into N sub-keys:
for i in 0..N:
    sub_key = "tweet:12345:part_{i}"

// Write: fan out to all sub-keys
// Read: read from random sub-key OR merge all

// Example: split a counter
key = "video:view_count:42"
split into: "video:view_count:42:shard_0", "video:view_count:42:shard_1", ...
Read: sum all parts = total view count
Tradeoff: Key splitting requires changes to the read path (merge/aggregate results). For counters, summing is trivial. For complex data, merging may be expensive or impossible.

Solution 2: Local Caching

Cache the hot key's value in memory on the application server or in a local in-process cache (e.g., Guava cache, LRU cache). Subsequent reads hit the local cache and never reach the shard.

Application Server
  |
  +-- Local Cache (in-memory, ~1ms)
  |     key="tweet:12345" => value (cached)
  |
  +-- Cache Miss? -> Fetch from shard (only once)
  |
  Shard (only gets ~1 request per cache expiry)
Best for: Read-heavy hot keys where eventual consistency is acceptable. A local cache with a 10-second TTL reduces shard traffic by 99%+ for a key getting 1000 QPS.

Solution 3: Caching Layer

Place a dedicated caching layer (Redis, Memcached) in front of the data store. The cache absorbs the read traffic before it reaches the shards.

Client | v [Redis / Memcached Cluster] <-- Cache layer absorbs hot reads | | | (miss) | v v [Shard A] [Shard B] <-- Rarely hit for hot keys | v [Persistent Storage]

For write-heavy hot keys, the cache helps less (writes still go to the shard), but read-heavy keys benefit enormously.

Solution 4: Read Replicas for Hot Partition

Create additional read replicas specifically for the shard holding the hot key. Distribute read traffic across replicas while writes go to the primary.

Write | v [Shard A - Primary] Holds hot key / | \ v v v [R1] [R2] [R3] Read replicas for hot key Reads distributed across replicas (primary handles writes only) Replica lag is acceptable for hot reads (eventual consistency)

Comparing Solutions

Solution Best For Reads Writes Consistency
Key Splitting Counters, aggregatable data Good Good Strong
Local Caching Read-heavy, tolerable staleness Excellent No help Eventual
Caching Layer General read-heavy keys Excellent No help Eventual
Read Replicas Hot reads, strong write needs Good Still on primary Replica lag

Detection

Hot keys are detected by monitoring per-key or per-shard request rates:

  • Shard-level metrics: If one shard has 10x the QPS of others, it likely holds a hot key.
  • Key-level sampling: Sample a fraction of requests and count per-key frequency. Keys appearing far above the median are hot.
  • Latency anomaly detection: A shard with elevated latency may be overwhelmed by a hot key.

Common Interview Questions

  1. What is a hot key and why is it a problem for hash-based partitioning?
  2. How would you handle a celebrity tweet that gets millions of reads per second?
  3. Why doesn't adding more shards solve the hot key problem?
  4. Explain how key splitting works. What are its limitations?
  5. Compare local caching vs. a caching layer for hot key mitigation.
  6. How would you detect hot keys in a production system?
  7. Design a counter system that handles millions of writes per second on a single logical key.
  8. How do read replicas help with hot keys? What are their limitations?

What to Skip

Avoid going deep into:
  • Cache invalidation strategies for hot keys -- this is a caching topic, not a partitioning topic.
  • Specific Redis cluster anti-affinity or key migration internals.
  • Write-ahead log or write splitting for write-heavy hot keys beyond the counter pattern, unless the interviewer asks for it.
