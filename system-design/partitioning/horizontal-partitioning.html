Horizontal Partitioning (Sharding)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand the most common data partitioning strategy used at scale — splitting rows across multiple servers to distribute load, increase storage capacity, and maintain query performance as data grows beyond what a single machine can handle.

Definition

Horizontal partitioning (sharding) splits a logically single table into multiple smaller tables — each containing the same schema but a different subset of rows. Each partition (shard) lives on its own physical server. The application or a middleware layer routes queries to the correct shard using a partition key (shard key).

Why Horizontal Partitioning?

Scalability Beyond a Single Machine

No single server can hold infinite data. A single MySQL instance might handle 1-5 TB comfortably; what happens at 50 TB? Sharding distributes data across N servers, where total capacity scales linearly with N.

Smaller Indexes, Faster Queries

Each shard holds only a fraction of the data, so B-tree indexes are shallower. A B+tree with 100M rows might be 4 levels deep, but with 10M rows per shard it drops to 3 levels — reducing disk seeks per lookup.

Distribute Write Load

A single database server bottleneck on writes (single-writer replication, lock contention). With sharding, writes to different shards can happen concurrently across separate machines, multiplying aggregate write throughput.

Benefit How It Helps at Scale
Storage capacity Linear scaling — add more shards, store more data
Query latency Smaller datasets per shard = fewer disk pages, shallower indexes
Write throughput N shards handle N independent write streams concurrently
Availability Shard failure affects only 1/N of users; others unaffected
Cache efficiency Per-shard caches hold a larger fraction of shard data in memory

Cross-Shard Queries

This is the hardest problem in sharding. When a query needs data from multiple shards, the application must:

  • Parallel fan-out: Send the query to all shards, collect results, merge in the application layer.
  • Secondary index lookup: If querying by a non-shard-key column, you need a global secondary index (GSI).
Interview tip: When designing a sharded system, always choose the shard key carefully. The shard key determines which queries can be served by a single shard (efficient) and which require fan-out (expensive). If 90% of queries filter by user_id, then user_id is the right shard key.

Global Secondary Indexes

A global secondary index (GSI) is a separate, non-sharded or differently-sharded index that maps a non-shard-key column back to the shard key. Example: if data is sharded by user_id but you need to query by email, maintain a separate lookup table mapping email → user_id, then route to the correct shard.

Query: SELECT * FROM users WHERE email = 'bob@x.com' 1. Lookup GSI (email_index table): ┌──────────────┬─────────┐ │ email │ user_id │ │ bob@x.com │ 42 │ ← Found └──────────────┴─────────┘ 2. Route to shard containing user_id=42: ┌──────────────────────────────────────┐ │ Shard N (user_id % 16 == 10) │ │ id=42, name=Bob, email=bob@x.com │ ← Direct lookup └──────────────────────────────────────┘

Choosing a Shard Key

Good shard keys have three properties: high cardinality (many unique values), uniform distribution (even data spread), and query alignment (most queries filter on this key).
Shard Key Cardinality Distribution Verdict
user_id High (billions) Uniform Excellent
country_code Low (~200) Skewed (US/India dominate) Poor
created_at (day) Medium Uneven (recent data skews hot) Risky
order_id (UUID) Very high Uniform Excellent (if queries align)

Common Interview Questions

  • How would you shard a user database for a social network? What shard key would you choose?
  • Explain cross-shard queries. How do you handle JOINs across shards?
  • What happens when a shard becomes too large — how do you reshard?
  • How do you maintain consistency when a single transaction spans multiple shards?
  • What is a global secondary index and when would you use one?
  • How does sharding differ from replication? Can you use both together?
  • Design Instagram's photo storage — how would you partition the data?

What to Skip

Do not deep-dive into specific sharding middleware (Vitess, Citus, Mongo sharding internals) unless asked. In an interview, focus on the concepts and tradeoffs, not implementation details of a particular tool. Also skip theoretical discussions about CAP theorem unless the question explicitly connects sharding to consistency.
