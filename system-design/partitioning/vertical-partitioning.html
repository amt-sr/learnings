Vertical Partitioning

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand how splitting columns (rather than rows) into separate tables or servers can improve performance, reduce I/O for common queries, and enable independent scaling of different data access patterns.

Definition

Vertical partitioning splits a table by columns rather than rows. Each partition contains the same rows but different columns. A single logical entity is decomposed into multiple physical tables, typically joined back together by a shared primary key. In its most aggressive form, different column groups can be placed on different servers.

Wide Rows vs Narrow Rows

As a table accumulates columns, each row becomes "wide." A single query that only needs 3 columns still reads entire rows (including unused columns) if the table is not vertically partitioned. This wastes I/O bandwidth and buffer pool memory.

Key insight: In most databases, the smallest unit of I/O is a page (typically 4KB-16KB). If a user profile row is 8KB and spans 50 columns, but your query only needs id, name, and email (totaling 200 bytes), you still read the full 8KB page. Vertical partitioning means the "core" table fits more rows per page, improving cache hit rates.

When to Use Vertical Partitioning

Separate Hot and Cold Columns

  • Hot: Columns accessed on every request (user_id, username, email). Keep on fast SSD storage, possibly in-memory.
  • Cold: Columns rarely accessed (biography, profile picture blob, activity logs). Move to cheaper storage or a separate database.

Different Access Patterns

  • Authentication queries need email and password_hash — these belong in a "credentials" table.
  • Profile page queries need bio, avatar_url, location — these belong in a "profile" table.
  • Admin/analytics queries need all columns — join across partitions.

Row-Level Security and Compliance

  • PII columns (SSN, date of birth) can be isolated on a separate server with stricter access controls and encryption.
  • GDPR/right-to-deletion is simpler when personal data is in its own table rather than scattered across wide rows.

Vertical vs Horizontal Partitioning

Dimension Vertical Partitioning Horizontal Partitioning
What is split Columns Rows
Each partition has Same rows, fewer columns Same columns, fewer rows
Primary benefit Reduce I/O per query Distribute storage and write load
Joins required Yes (recombine columns) No (single shard is self-contained)
Complexity Low (schema change) High (routing, resharding)
Scale limit Each partition still holds all rows True linear scaling across machines
Use together? Yes — vertically partition columns, then horizontally partition each column-group by rows
Interview tip: Vertical partitioning is a useful first step before horizontal sharding. Many teams vertically partition first (cheaper, lower risk) and only introduce horizontal sharding when a single column-group table grows too large. In an interview, mention both and explain the sequencing.

Common Interview Questions

  • What is the difference between vertical and horizontal partitioning?
  • When would you choose vertical partitioning over horizontal sharding?
  • How does vertical partitioning affect JOIN performance?
  • Design a user table that separates authentication data from profile data — explain the tradeoffs.
  • How would vertical partitioning help with GDPR compliance?

What to Skip

Do not over-complicate this topic. Vertical partitioning is conceptually simple. Do not get into debates about normalization vs denormalization — vertical partitioning is orthogonal to both. Skip discussions of column-oriented databases (like BigQuery or Redshift) unless the interviewer specifically asks about OLAP vs OLTP storage engines.
