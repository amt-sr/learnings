Hash Sharding

Difficulty: вӯҗвӯҗвӯҗвҳҶвҳҶ Interview Frequency: вҳ…вҳ…вҳ…вҳ…вҳ…

Goal

Understand the most widely used sharding strategy in distributed databases вҖ” using a hash function to distribute keys uniformly across shards. Learn why it solves the hot range problem, what breaks when you change the number of shards, and how consistent hashing addresses the resharding challenge.

Definition

Hash sharding applies a hash function to the shard key and uses the result modulo the number of shards to determine placement. The formula is simple: shard_id = hash(key) % N. This ensures even distribution of keys regardless of the key's natural distribution, eliminating hot spots.

Uniform Distribution

A good hash function (MD5, SHA-256, MurmurHash) maps keys to a large output space with near-uniform distribution. The modulo operation then distributes keys evenly across N shards. With a good hash function and sufficient cardinality in the key space, each shard will have approximately total_records / N records.

Key advantage over range sharding: Hash sharding eliminates hot ranges. Even if 80% of users have last names starting with "S", hash(user_id) % N distributes them uniformly. No single shard becomes a hotspot due to data skew.

The Resharding Problem

When you change N (add or remove a shard), the modulo changes and almost every key maps to a different shard. Going from N=4 to N=5 means hash(key) % 5 produces different results than hash(key) % 4 for nearly all keys.

Before (N=4): After (N=5): hash("alice") % 4 = 3 вҶ’ Shard 3 hash("alice") % 5 = 0 вҶ’ Shard 0 вҶҗ MOVED hash("bob") % 4 = 1 вҶ’ Shard 1 hash("bob") % 5 = 3 вҶ’ Shard 3 вҶҗ MOVED hash("carol") % 4 = 2 вҶ’ Shard 2 hash("carol") % 5 = 4 вҶ’ Shard 4 вҶҗ MOVED hash("dave") % 4 = 0 вҶ’ Shard 0 hash("dave") % 5 = 2 вҶ’ Shard 2 вҶҗ MOVED Adding 1 shard causes ~75% of all keys to relocate. For N вҶ’ N+1, approximately N/(N+1) keys must move.
The resharding nightmare: In production, moving 75%+ of your data between shards means massive network I/O, downtime or degraded performance, and risk of data loss if the process fails. This is why simple hash modulo sharding is rarely used directly in systems that need to scale dynamically.

Relationship to Consistent Hashing

Consistent hashing solves the resharding problem. Instead of hash(key) % N, both keys and shards are hashed onto a ring (0 to 2^32 or 2^160). Each key is assigned to the nearest shard in the clockwise direction on the ring. When a shard is added or removed, only 1/N of keys need to move (the keys in the affected arc of the ring).

Hash Sharding in Practice

Most production systems use a variation of hash sharding:

  • DynamoDB: Hash + range composite keys. Partition key hashed, sort key provides ordering within a partition.
  • Cassandra: Murmur3 hash onto a token ring (consistent hashing). Virtual nodes (vnodes) improve uniformity.
  • MongoDB: Hashed shard keys using MD5. Supports hashed indexes for even distribution.
  • Redis Cluster: CRC16 hash slot mapping. 16,384 hash slots distributed across nodes.
Interview tip: When asked "how would you shard X?", default to hash sharding unless the workload has heavy range queries. Then discuss the tradeoff: hash gives even distribution but kills range queries; range preserves query locality but risks hot shards. The strongest answer proposes a hybrid: hash on the primary dimension for distribution, with secondary indexes or a separate range-partitioned store for range queries.

Common Interview Questions

  • Explain hash sharding. Why is it the most common sharding strategy?
  • What happens when you add a shard to a hash-sharded system? How do you solve this?
  • What is consistent hashing and how does it solve the resharding problem?
  • How does DynamoDB use hash sharding? Explain partition keys and sort keys.
  • Design a URL shortener вҖ” how would you shard the storage layer?
  • Hash sharding destroys range queries вҖ” how would you handle a workload that needs both uniform distribution and range scans?
  • Explain virtual nodes in consistent hashing. Why are they needed?

What to Skip

Do not implement a hash function from scratch or debate hash function properties (avalanche effect, collision resistance) unless the interviewer specifically asks. In interviews, assume a good hash function exists and focus on the system-level implications of hash sharding: distribution, resharding, and the tradeoff with range queries. Skip deep dives into specific hash function implementations (MurmurHash internals, etc.).
