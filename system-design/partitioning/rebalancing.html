Rebalancing

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆
Goal: Understand how consistent hashing enables efficient rebalancing when shards are added or removed, how it compares to modular hashing, and the mechanics of streaming data during rebalance operations.

Definition

Rebalancing is the process of redistributing data across shards when the cluster topology changes -- either by adding new shards (scaling out) or removing existing ones (scaling in, failures, or maintenance). The key question rebalancing answers is: how much data must move, and where does it go?

What Happens When Adding/Removing Shards

BEFORE: 3 Shards Key Range on Ring: [0 -------- A -------- B -------- C -------- 2^32] ADD Shard D at position between A and B: [0 -------- A -------- D -------- B -------- C -------- 2^32] Data Movement: - Keys between A and D move from B --> D (only 1/N of B's data) - All other keys stay in place Data that does NOT move: ~2/3 of all keys Data that moves: ~1/3 of all keys (specifically, ~1/N = ~1/4)

Adding a Shard

  1. New shard is hashed onto the ring at a specific position.
  2. The shard that previously owned the range clockwise of the new position splits its data.
  3. Keys falling between the new shard and the next clockwise shard are reassigned.
  4. Only ~1/N of the total data needs to move (where N is the new total number of shards).

Removing a Shard

  1. Shard is removed from the ring (failure or planned removal).
  2. All keys owned by the removed shard are reassigned to the next shard clockwise.
  3. Again, only ~1/N of total data needs to move.

Consistent Hashing vs. Modular Hashing Rebalancing

Scenario Modular Hashing (N=3 to N=4) Consistent Hashing (add 1 shard to 3)
Data movement fraction ~75% (3/4 keys change slot) ~25% (1/4 keys move)
Keys that stay ~25% ~75%
Mechanism hash(key) % 4 != hash(key) % 3 Only keys in affected ring range move
System impact Massive data shuffle Targeted stream between 2 shards
The 1/N Rule: With consistent hashing, adding or removing one shard from a cluster of N shards requires moving approximately 1/N of the total data. This is the fundamental advantage -- it scales gracefully with cluster size.

Streaming Data During Rebalance

Rebalancing is not instantaneous. Data must be physically transferred between shards, which involves a streaming phase:

Rebalance Flow (Adding Shard D): Phase 1: Join Shard D joins the cluster, announces its ring position Cluster updates ring metadata (gossiped to all nodes) Phase 2: Stream Source shard (B) streams affected key ranges to Shard D +---[B]---stream range A-->D---+ | | +---[B] continues serving reads-+ Phase 3: Handoff Reads for affected keys redirect from B to D Writes dual-written during transition (optional) Phase 4: Cleanup B deletes transferred key ranges Cluster stabilizes with new topology

Key Properties of Streaming

  • Incremental: Data flows in batches or streams, not as a bulk copy. This limits impact on serving traffic.
  • Concurrent: Multiple shards can stream to multiple targets simultaneously (with backpressure).
  • Resumable: If a node fails during streaming, it can resume from the last checkpoint.
  • Read-your-writes: During streaming, writes must be replicated to both the source and destination to avoid data loss.

Design Goals for Rebalancing

Goal Description How Consistent Hashing Helps
Minimal data movement Only move what is necessary Only 1/N of data moves per topology change
Uniform distribution After rebalance, data is evenly spread Combined with VNodes, distribution stays uniform
Availability System stays readable during rebalance Only affected ranges have brief handoff periods
Scalability Cost of rebalance decreases as cluster grows 1/N shrinks as N grows

Common Interview Questions

  1. What happens to data when you add a new shard to a consistent hashing ring?
  2. How much data moves when you remove one shard from a 10-shard cluster using consistent hashing?
  3. Compare the rebalancing cost of modular hashing vs. consistent hashing.
  4. How do you handle reads and writes during a rebalance operation?
  5. What is streaming in the context of cluster rebalancing?
  6. Can a rebalance cause data unavailability? How do you prevent that?
  7. How does the rebalancing process work with virtual nodes?

What to Skip

Avoid going deep into:
  • Specific streaming protocol implementations (e.g., Cassandra's streaming internals or DynamoDB's internal rebalancing).
  • Rack-aware or zone-aware rebalancing strategies unless explicitly asked.
  • Consensus-based membership protocols (SWIM, gossip) -- these are cluster membership topics, not rebalancing per se.
