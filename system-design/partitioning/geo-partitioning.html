Geo Partitioning

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand how to partition data by geographic region to comply with data residency laws, reduce latency for users in different regions, and achieve fault isolation across continents. Master the consistency challenges that arise when the same logical data lives in multiple geographic partitions.

Definition

Geo partitioning distributes data across database instances in different geographic regions based on the user's location. EU users' data resides in EU data centers, US users' data in US data centers, and so on. Each region operates semi-independently with its own complete copy of relevant data.

Why Geo Partition?

Compliance and Data Residency

GDPR (EU), CCPA (California), LGPD (Brazil), and similar regulations require that personal data of residents be stored within the jurisdiction. Geo partitioning is often the simplest way to achieve compliance — data never leaves the region.

GDPR Article 44: Transfer of personal data to a third country may only take place if the conditions laid down in the GDPR are met. Geo partitioning sidesteps this entirely by ensuring data never crosses borders for processing.

Latency Reduction

The speed of light limits round-trip time. A request from Frankfurt to Virginia is ~100ms; within Frankfurt, it is <5ms. Geo partitioning ensures users talk to a nearby data center.

Route Typical RTT Impact on P99
Same region 1-5 ms Excellent
EU to US East 80-120 ms Degraded
EU to APAC 180-250 ms Unacceptable for interactive

Fault Isolation by Region

If the US East region goes down, EU and APAC users are unaffected. Geo partitioning provides natural blast-radius containment at the continental level.

Cross-Region Queries

The fundamental challenge: what happens when a user in one region needs data from another?

Scenario: Global Leaderboard

A gaming app needs a worldwide leaderboard. With geo partitioning, each region has its own leaderboard. The solution is an aggregation layer that asynchronously merges regional data into a global view.

Scenario: User Travels Abroad

A EU user travels to the US. Their data is in the EU partition. Options:

  • Reroute: API requests go back to EU (adds latency but ensures compliance)
  • Replicate: Cache a read-only copy in the visiting region (requires careful cache invalidation)
  • Session migration: Temporarily move the user's session to the visiting region's DB (requires legal review)
Interview tip: When designing a globally distributed system, always address the "traveling user" problem. Interviewers love this edge case because it forces you to reconcile compliance requirements with user experience.

Consistency Challenges Across Regions

With geo partitioning, you face the hardest problems in distributed systems: maintaining consistency across regions with 100+ ms network latency between them.

Replication Topology

  • Multi-leader: Each region can accept writes. Conflicts resolved with CRDTs, last-write-wins, or application-level merge logic. Used by Cassandra, CockroachDB.
  • Single-leader with regional followers: All writes go to one region (e.g., US), followers replicate asynchronously. Simple consistency model but poor write latency for non-leader regions.
Consistency tradeoff: With multi-leader geo partitioning, you cannot have strong consistency without sacrificing availability (CAP theorem). If EU and US can both write simultaneously, you must accept eventual consistency or design conflict-free data types (CRDTs).

Conflict Resolution Strategies

Strategy Mechanism When to Use
Last Write Wins (LWW) Timestamp-based, discard older value Simple, acceptable data loss for non-critical fields
CRDTs Mathematically merge-able data structures Counters, sets, registers with commutative operations
Application-level merge Custom logic (e.g., merge shopping carts) Domain-specific conflict handling
Escalation to human Flag conflicts for manual resolution Financial transactions, critical data

Common Interview Questions

  • How would you design a globally distributed system that complies with GDPR?
  • A EU user travels to the US — how do you handle their data access?
  • How do you maintain a global leaderboard when data is geo-partitioned?
  • Explain the consistency challenges in multi-region deployments.
  • Design a system where each region must be fully self-contained — what are the tradeoffs?
  • How does CockroachDB / Google Spanner handle geo-distributed transactions?
  • What is the CAP theorem implication of geo-partitioned multi-leader systems?

What to Skip

Do not deep-dive into specific cloud provider regions or pricing. The interviewer cares about the architectural decisions, not whether you pick eu-west-1 vs eu-central-1. Also skip implementation details of consensus protocols (Raft, Paxos) unless the conversation naturally leads there. The focus should be on why geo partitioning, not how to configure Terraform for multi-region deployment.
