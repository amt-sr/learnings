Hot Partitions

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆
Goal: Understand how a range of keys (not just a single key) can cause disproportionate load on one shard, identify common causes of hot partitions, and apply solutions including salting, composite keys, and range splitting.

Definition

A hot partition is a shard or partition that receives disproportionately high traffic compared to others, not because of a single hot key, but because an entire range of keys happens to be accessed far more frequently. Unlike hot keys (one key overwhelming one shard), hot partitions involve many keys within a contiguous or correlated range that collectively overwhelm the shard owning that range. This is particularly problematic for range-based partitioning but can also affect hash-based partitioning when key patterns correlate with access patterns.

Hot Keys vs. Hot Partitions

Property Hot Keys (6.3.5) Hot Partitions
Scope Single key Range or group of keys
Cause One key is extremely popular Key access patterns are correlated
Hash-based partitioning Cannot help (single key = single shard) Can help (keys hash to different shards)
Range-based partitioning Vulnerable (as with hash-based) Extremely vulnerable
Mitigation Key splitting, caching, replicas Salting, composite keys, split ranges

Common Causes

Time-Based Keys

Keys that include a timestamp or date component often create hot partitions because recent data is accessed far more frequently than old data. In range-based partitioning, the "latest" partition receives nearly all traffic.

// Problematic key design:
key = "user_activity:2026-06-28"     // Today's partition gets ALL writes
key = "logs:2026-06-28:hour=14"       // Current hour gets all writes
key = "events:timestamp=1751089200"   // Current time range is hot
Classic Example: In a logging system where keys are timestamps, the current day's partition receives 100% of writes. Yesterday's partition may receive 10% of reads. All other partitions are nearly idle.

Geographic Skew

When keys are based on geographic identifiers (country, region, city), populations are unevenly distributed. A shard holding data for a populous region (e.g., California, India) will receive far more traffic than one holding data for a small region.

Sequential Key Generation

Auto-incrementing IDs or monotonically increasing keys (e.g., UUID v1) cluster writes into a narrow range, causing the shard owning that range to become a hotspot.

// Auto-increment creates sequential hot range:
insert: id=1001 -> shard A
insert: id=1002 -> shard A    (same shard as 1001)
insert: id=1003 -> shard A    (same shard again)
// All writes hit shard A until the range crosses to shard B
Range-Based Partitioning with Time-Based Keys: Partition 1: [2026-01 to 2026-03] Cold: ~2% of reads Partition 2: [2026-04 to 2026-05] Cold: ~5% of reads Partition 3: [2026-06 to 2026-06] HOT: ~80% of reads + 100% of writes Partition 4: [2026-07 to 2026-12] Cold: ~0% (future) The "current" partition is always the hotspot.

Solutions

Solution 1: Salting

Prefix keys with a random or deterministic salt value to distribute them across multiple partitions. Each logical key is spread across N physical keys.

// Original: all writes go to one partition
key = "events:2026-06-28"

// With salting: writes distributed across N partitions
salt = hash(timestamp + event_id) % N
key = "events:salt_{salt}:2026-06-28"

// Reads must query all N salted partitions:
for i in 0..N-1:
    query("events:salt_{i}:2026-06-28")
merge results
Tradeoff: Salting distributes write load but increases read complexity (must query N partitions and merge). Works best for write-heavy workloads where reads target a specific key (not a range scan).

Solution 2: Composite Keys

Redesign the key to include additional dimensions that break up the natural clustering. Instead of partitioning by time alone, partition by (time, user_id_hash) or (time, region).

// Instead of:
key = "metric:2026-06-28"              // One partition

// Use composite:
key = "metric:2026-06-28:shard_{user_id % N}"  // Spread across N partitions

// Each partition holds data for 1/N of users for that day

Solution 3: Split Ranges

For range-based partitioning, proactively split hot ranges into smaller sub-partitions. This is a form of dynamic partitioning where the system detects a hot range and automatically splits it.

BEFORE (hot partition): [2026-06-01 ... 2026-06-30] 1 partition, ALL traffic AFTER (split range): [2026-06-01 ... 2026-06-07] Split 1: cold [2026-06-08 ... 2026-06-14] Split 2: cold [2026-06-15 ... 2026-06-21] Split 3: cold [2026-06-22 ... 2026-06-28] Split 4: HOT, but 1/4 of original load [2026-06-29 ... 2026-06-30] Split 5: current Or even finer granularity (daily partitions): [2026-06-28] Today's partition: still hot, but isolated [2026-06-27] Yesterday: moderate ...

Solution 4: Reverse Keys or Hashed Prefixes

Reverse the key or hash the prefix to break sequential patterns. For example, instead of storing keys in order 0001, 0002, 0003, store them as 1000, 2000, 3000 (reversed) or hash the first portion.

// Original (sequential):        // Reversed:
key = "10000001"                  key = "10000001" (reversed: "10000001")
key = "10000002"                  key = "20000001" (reversed: "20000001")
key = "10000003"                  key = "30000001" (reversed: "30000001")

// Original keys cluster in one range; reversed keys are spread across the key space.

Monitoring and Detection

Detecting hot partitions requires monitoring at the partition level:

Metric Alert Condition Indicates
QPS per partition > 3x average across partitions Hot partition (read or write)
CPU utilization per partition > 80% while cluster average is 30% Likely hot partition
Disk I/O per partition > 5x average Hot partition (often write-heavy)
Latency per partition > P99 is 3x higher than other partitions Hot partition causing queueing

Comparing Solutions

Solution Best For Read Impact Write Impact
Salting Write-heavy, point reads Must query N partitions Spread across N partitions
Composite Keys Multi-dimensional data May require multi-partition queries Spread by secondary dimension
Split Ranges Range-partitioned data No change if querying specific range Spread across sub-partitions
Reverse/Hash Prefix Sequential keys Point reads unaffected Distributed across key space
