Range Sharding

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the simplest sharding strategy — assigning contiguous ranges of keys to shards. Learn why range sharding is efficient for range queries but dangerous when data distribution is uneven, and how to mitigate the "hot range" problem.

Definition

Range sharding assigns each shard a contiguous range of the shard key. A routing table maps key ranges to shard identifiers. For example, keys A-F go to Shard 1, G-N to Shard 2, O-Z to Shard 3. The router determines which shard a key belongs to by checking which range it falls in.

Shard Key: last_name (alphabetical) ┌────────────────────────────────────────────────────┐ │ Routing Table │ │ │ │ [A - F] ───────▶ Shard 0 (Server A) │ │ [G - N] ───────▶ Shard 1 (Server B) │ │ [O - Z] ───────▶ Shard 2 (Server C) │ └────────────────────────────────────────────────────┘ Query: SELECT * FROM users WHERE last_name = 'Kumar' → 'Kumar' in [G-N] → Route to Shard 1 Query: SELECT * FROM users WHERE last_name BETWEEN 'J' AND 'M' → Range [J-M] entirely in [G-N] → Route to Shard 1 only (efficient!)

Range Query Efficiency

Range sharding's killer feature: range queries can be served by a single shard (or a small number of shards) instead of fan-out to all shards. If your workload involves frequent range scans (e.g., "all orders from January," "all users with names starting with J"), range sharding preserves query locality.

SELECT * FROM orders WHERE order_date BETWEEN '2024-01-01' AND '2024-01-31' Date-based range sharding: ┌──────────────────────┬─────────────┐ │ Jan 2024 (Shard 0) │ 1.2M orders │ ← Query hits only this shard ├──────────────────────┼─────────────┤ │ Feb 2024 (Shard 1) │ 1.1M orders │ ├──────────────────────┼─────────────┤ │ Mar 2024 (Shard 2) │ 0.9M orders │ └──────────────────────┴─────────────┘ Result: Single-shard lookup instead of fan-out to all shards.

Examples of Range Sharding

Shard Key Range Type Common Use Case Advantage
Timestamp / Date Monthly or weekly ranges Time-series data, logs Natural temporal locality; old shards become read-only and can be archived
Alphabetical (last_name) Letter ranges User directories Intuitive, easy to understand
Numeric ID ranges 0-999, 1000-1999, etc. Auto-increment IDs Simple to implement; sequential scan efficient
Geographic (postal code) 10000-19999, 20000-29999 Location-based services Co-locates nearby users

The Hot Range Problem

Range sharding's fatal flaw: data is rarely uniformly distributed. If you shard by last_name, the "S" shard (Smith, Singh, Silva, Sanchez) will be massively larger than the "Q" shard. One shard becomes a hotspot while others sit idle.

The hot range problem in production: If you shard orders by date, the current month's shard receives ALL writes while historical shards receive none. The "hot shard" becomes the bottleneck you were trying to eliminate by sharding in the first place. This is why most large-scale systems avoid pure date-based sharding for write-heavy workloads.

Mitigation Strategies

  • Dynamic range splitting: Monitor shard sizes and split hot ranges into smaller sub-ranges (similar to B-tree node splitting).
  • Compound keys: Use (date, user_id) as the shard key to spread writes across users within a date range.
  • Hybrid approach: Combine range sharding (for query locality) with hash-based distribution within each range.
  • Manual rebalancing: Periodically analyze shard sizes and adjust range boundaries.
Interview tip: Always mention the hot range problem when discussing range sharding. The interviewer wants to see that you understand the real-world failure mode, not just the textbook definition. Propose a mitigation strategy to demonstrate depth.

Uneven Distribution

Even without a single hot range, gradual skew accumulates. Over months, some shards grow faster than others. Without active management, you end up with:

6 months after initial deployment: Shard 0 [A-D]: 2.1 GB (underutilized) Shard 1 [E-J]: 8.7 GB (overloaded!) Shard 2 [K-P]: 5.3 GB Shard 3 [Q-T]: 3.2 GB Shard 4 [U-Z]: 6.8 GB → Need to rebalance: split Shard 1 into [E-G] and [H-J]

Common Interview Questions

  • Explain range sharding and give an example where it works well.
  • What is the hot range problem and how would you solve it?
  • How does range sharding compare to hash sharding for range queries?
  • Design a sharding scheme for a time-series database — which strategy would you choose and why?
  • How would you handle rebalancing when one shard becomes too large?
  • What happens to range queries when you switch from range sharding to hash sharding?
  • How does MongoDB's range-based sharding (tag-aware sharding) work?

What to Skip

Do not confuse range sharding with range partitioning in a single database. Range partitioning (MySQL, PostgreSQL) splits a single table into partitions on one server. Range sharding distributes partitions across multiple servers. Also skip detailed discussions of B-tree internals — the interviewer wants the sharding concept, not a database internals lecture.
