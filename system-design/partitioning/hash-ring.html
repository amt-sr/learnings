Hash Ring

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆
Goal: Understand how a hash ring distributes data keys across shards using consistent hashing, and why it is preferred over modular hashing for partitioned data stores.

Definition

A hash ring is a circular address space formed by mapping both shard identifiers and data keys onto the same hash range (typically 0 to 232-1 or 0 to 2160-1). Each key is assigned to the first shard encountered when moving clockwise around the ring from the key's hash position. This is the same underlying mechanism used in consistent hashing for load balancing, but applied to data partitioning rather than request distribution.

Hash Ring (0 .. 2^32 - 1) Shard A / / 0 ------ | | | | | | ------ \ \ Shard C Keys are mapped clockwise to the next shard: key1 (hash=42) --> Shard A (hash=100) key2 (hash=200) --> Shard B (hash=300) key3 (hash=950) --> Shard C (hash=1000, wraps to 0) Ring positions (clockwise): 0 --------> Shard A --------> Shard B --------> Shard C --------> back to 0

How Keys Are Assigned to Shard Positions

Step-by-Step Lookup

  1. Hash the key: Apply a consistent hash function (e.g., SHA-1, MD5, or MurmurHash) to the data key to produce a position on the ring.
  2. Hash each shard: Apply the same hash function to each shard's identifier to determine its position on the ring.
  3. Walk clockwise: From the key's hash position, move clockwise around the ring until you reach the first shard. That shard owns the key.
  4. Wrap-around: If no shard is found between the key's position and the end of the ring, wrap around to the first shard at position 0.
Key Insight: Both data keys and shard IDs live in the same hash space. The ring is just a sorted list of positions. Lookup is an O(log N) binary search or an O(1) lookup with a bounded hash table.

Hash Ring vs. Modular Hashing

Property Modular Hashing Hash Ring (Consistent)
Key formula shard = hash(key) % N shard = clockwise_next(hash(key))
Adding a shard Remaps ~N/N = almost all keys Remaps ~1/N keys
Removing a shard Remaps almost all keys Remaps ~1/N keys
Hotspot risk Lower (uniform by default) Higher (see VNodes section)
Implementation complexity Simple Moderate

Relationship to Load Balancing

The hash ring concept originates from consistent hashing in load balancing, where server nodes are placed on a ring and client requests are routed to the next server clockwise from the request's hash. The data partitioning application is identical in mechanism but differs in intent:

  • Load balancing: Requests (or sessions) are mapped to backend servers to distribute traffic.
  • Data partitioning: Data keys are mapped to storage shards to distribute data and load across machines.

In both cases, adding or removing a node only affects 1/N of the mappings, which is the core benefit over modular hashing.

Real-World Usage

  • DynamoDB: Uses consistent hashing to partition data across physical storage nodes.
  • Cassandra: Hash ring with virtual nodes for token-aware routing.
  • Riak: Consistent hashing ring (default 160 virtual nodes per physical node).
  • Memcached (Ketama): Consistent hashing for cache key distribution.

Common Interview Questions

  1. How does a hash ring work for data partitioning?
  2. What happens when you add a new shard to a hash ring?
  3. Why is consistent hashing better than hash(key) % N for partitioning?

What to Skip

Avoid going deep into:
  • Specific hash function implementations (e.g., MurmurHash internals) -- just name them.
  • Jump consistent hashing and Rendezvous hashing -- these are alternatives rarely asked about at this level.
  • Low-level ring data structures like skip lists or balanced BSTs unless the interviewer explicitly asks.
