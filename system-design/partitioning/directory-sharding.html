Directory Sharding

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the most flexible sharding strategy — using a lookup table (directory) to map keys to shards. Learn when directory-based sharding is the right choice, how it compares to range and hash sharding, and why the directory itself becomes a critical reliability concern.

Definition

Directory sharding uses a central lookup table (directory) that maps each key to its assigned shard. Instead of computing the shard via a formula (range or hash), the application consults the directory to find which shard holds the data. The directory can assign keys to shards arbitrarily, enabling fine-grained control over placement.

Flexible Assignment

The core advantage: the directory can place keys wherever it wants. Unlike hash sharding (deterministic formula) or range sharding (contiguous ranges), directory sharding allows arbitrary mapping.

Use Cases for Flexibility

  • Manual load balancing: Move specific high-traffic keys to less-loaded shards.
  • Tenant isolation: Ensure enterprise customer A's data is on a dedicated shard with premium resources, while small customers share a shard.
  • Gradual migration: Move keys one at a time from old shards to new shards without a massive resharding event.
  • Colocation: Place related entities (a user and their orders) on the same shard to avoid cross-shard joins.
Resharding advantage: With directory sharding, adding a shard requires only updating the directory. Move specific keys to the new shard by updating their directory entry. No formula change, no massive data movement. Compare this to hash modulo where adding a shard moves nearly all keys.

Single Point of Failure

The directory itself is the critical vulnerability. If the directory is unavailable, no data can be routed — every query requires a directory lookup first. The directory becomes the most important service in your infrastructure.

Query flow with directory sharding: Client → [1] Query Directory: "Where is user:1001?" │ ▼ ┌─────────────────────┐ │ Directory Svc │ ← SPOF if not replicated │ user:1001 → S2 │ └──────────┬──────────┘ │ ▼ [2] Route to Shard 2: "SELECT * FROM users WHERE id = 1001" │ ▼ ┌─────────────────────┐ │ Shard 2 (S2) │ │ user:1001 data │ └─────────────────────┘ Extra latency: +1 network round-trip for directory lookup

Mitigating the SPOF

  • Cache the directory: Client-side caching (with TTL) avoids hitting the directory on every query. Invalidation on key moves propagates asynchronously.
  • Replicate the directory: Run the directory on a highly available cluster (Raft consensus, multi-AZ). It is read-heavy so replication is straightforward.
  • Embed routing in application: Keep a local copy of the directory in each application instance, refreshed periodically.
Performance consideration: Every query now requires two network hops: one to the directory, one to the shard. With caching, this is reduced to one hop for repeated queries. But the first query for a key always incurs the extra latency. At high QPS, the directory must handle reads at the same rate as the data shards — it needs its own horizontal scaling strategy.

Interview tip: Directory sharding is rarely the first answer to "how would you shard this?" Start with hash sharding as the default, then mention directory sharding as an option when you need fine-grained control over key placement (e.g., multi-tenant SaaS with per-tenant isolation). Showing that you know all three strategies and their tradeoffs demonstrates senior-level understanding.

Directory Sharding in Production

Real-world systems often combine directory sharding with hash or range sharding:

  • Composite: Use hash sharding for most data, but maintain a directory for exceptional cases (VIP tenants, hot keys that need manual placement).
  • Two-level: First-level directory maps tenant → shard group; second-level hash sharding distributes within the group.
  • Service mesh routing: Modern service meshes (Envoy, Istio) use xDS API as a form of directory for routing requests to backend shards.

Common Interview Questions

  • What is directory sharding and when would you use it?
  • How do you handle the single point of failure in a directory-based sharding system?
  • Compare and contrast range, hash, and directory sharding.
  • Design a multi-tenant SaaS database — how would you ensure tenant isolation while sharing infrastructure?
  • How would you reshard a directory-based system? What happens to in-flight requests during a key move?
  • What is the latency impact of directory sharding compared to hash sharding?

What to Skip

Do not over-engineer the directory implementation. The interviewer does not need to see a schema design for the directory table or a consensus protocol for directory replication. Focus on the architectural tradeoffs (flexibility vs complexity vs latency). Skip discussions of ZooKeeper, etcd, or Consul internals unless the conversation specifically turns to directory implementation.
