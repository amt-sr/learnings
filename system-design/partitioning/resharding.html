Resharding

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆
Goal: Understand when and why you need to reshard a data store, the different approaches (double-write, online resharding), and the operational challenges of resharding at scale.

Definition

Resharding is the process of changing the number of shards or the partitioning scheme of an existing data store. Unlike rebalancing (which redistributes data within the same partitioning scheme), resharding often involves a fundamental change to how keys are mapped -- for example, moving from a 10-shard to a 20-shard configuration, or switching from range-based to hash-based partitioning. Resharding is a planned, often complex operation distinct from the incremental rebalancing that consistent hashing handles automatically.

Why Reshard?

Data Growth

As data volume grows beyond the capacity of existing shards, you need more shards to maintain performance. A shard that held 100GB may need to be split into two 50GB shards, or a 10-shard cluster may need to become a 20-shard cluster.

Uneven Load

Even with consistent hashing, certain key ranges may become hot due to access patterns rather than data size. Resharding can redistribute load by changing the partitioning scheme or adding capacity to handle skewed distributions.

Performance Requirements

Adding shards reduces the amount of data and traffic per shard, improving latency and throughput per partition. This is critical when approaching per-shard limits (IOPS, storage, CPU, network).

Resharding vs. Rebalancing: Rebalancing is an ongoing, incremental process handled by the partitioning system (consistent hashing does this natively). Resharding is typically a larger, planned operation that may involve changing the partition count, the hashing scheme, or the partition strategy itself.

Approaches to Resharding

Approach 1: Double-Write During Migration

Phase 1: Dual Write +--- Client ---+ | | v v [Old Shard] [New Shard] Both receive writes ^ ^ | | +--- Router --+ Reads from old shard Phase 2: Backfill [Old Shard] --copy data--> [New Shard] Historical data migrated Phase 3: Cutover +--- Client ---+ | v [New Shard] Old shard decommissioned Phase 4: Cleanup [Old Shard] deleted
  1. Dual-write phase: Route all writes to both old and new shard schemes. Reads continue from the old shard.
  2. Backfill phase: Copy all existing data from old shards to new shards. This may take hours or days for large datasets.
  3. Cutover phase: Switch reads to new shards. Validate data consistency between old and new.
  4. Cleanup phase: Decommission old shards after confirming correctness.

Approach 2: Online Resharding (Background)

Similar to double-write but without the explicit cutover. The system gradually moves data in the background while serving traffic from both old and new locations. A routing layer directs each request to the correct shard based on a progress map.

Key Distinction: Double-write is simpler to reason about (writes go everywhere), while online resharding is more efficient (writes go only to the correct shard, data is migrated in the background). Online resharding is what DynamoDB and Cassandra do under the hood.

Challenges

Zero-Downtime

Production systems cannot afford downtime for resharding. The migration must happen while the system serves traffic, which means handling concurrent reads, writes, and data movement without inconsistencies.

Data Verification

After migration, you must verify that every key in the old shards exists in the correct new shard. This typically involves cross-checking counts, checksums, or running sample validation queries. Inconsistencies can arise from writes that occurred during migration.

Rollback

If something goes wrong during resharding, you need a rollback plan. Double-write simplifies rollback (the old shards still have all data). Online resharding may require reverse-migration, which is expensive and error-prone.

Operational Complexity

Challenge Risk Mitigation
Increased write load Double-write doubles write amplification Throttle migration speed; batch writes
Routing consistency Stale routing causes reads from wrong shard Versioned routing metadata with TTL
Partial failure System in inconsistent mid-migration state Idempotent migration steps; checkpoint progress
Data loss Writes during migration lost if not dual-written Write-ahead log; dual-write or background with WAL replay
Latency spikes Backfill competes with serving traffic for I/O Rate-limit backfill; dedicated bandwidth

Real-World Examples

  • DynamoDB: Automatic sharding; when a partition reaches 10GB or throughput limits, it splits into two partitions transparently. No customer involvement needed.
  • Cassandra: Add nodes to the cluster; vnodes automatically redistribute token ranges. Resharding is effectively continuous.
  • YouTube (historical): Migrated from a sharded MySQL setup to Vitess, requiring a coordinated resharding effort with double-write.
  • Uber: Migrated from a single-shard Schemaless store to multi-shard, using online migration with background data copying.

Common Interview Questions

  1. How would you reshard a database with 100TB of data?
  2. Explain the double-write approach to resharding. What are its pros and cons?
  3. How do you ensure zero data loss during a resharding operation?
  4. What happens if resharding fails halfway through? How do you roll back?
  5. How is resharding different from rebalancing in a consistent hashing system?
  6. Design a system to reshards from 10 shards to 20 shards with zero downtime.
  7. How would you verify data integrity after resharding?

What to Skip

Avoid going deep into:
  • Vitess-specific resharding internals (vttablet, VReplication) unless the interview is about Vitess.
  • Specific database vendor resharding tools (e.g., MongoDB chunk migration protocol details).
  • Backup-and-restore resharding (shut down, copy, restart) -- not relevant for zero-downtime discussion.
