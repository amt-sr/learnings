Trade-offs

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • Why trade-offs are the #1 skill in senior system design interviews
  • The CAP Theorem and its practical implications
  • The consistency spectrum and when to pick each level
  • Key trade-off pairs and how to reason about them
  • The 5-step interview framework for discussing trade-offs

Definition

A trade-off is the result of choosing one alternative over another when both cannot be achieved simultaneously. In system design, every decision has a cost.

Every "yes" to one property is a "no" (or partial "no") to another. The only question is: which cost are you willing to pay?
The iron triangle: Fast, cheap, good — pick two. In distributed systems: consistency, availability, partition tolerance — pick two (CAP).

Why Trade-offs Are the #1 Interview Skill

At senior/staff level, interviewers test whether you can reason about trade-offs, not whether you know patterns.

The questions that matter:
  • "Why did you choose X over Y?"
  • "What's the cost of that decision?"
  • "When would this be the wrong choice?"
  • "What would you give up to get that?"

A junior says: "We'll use Redis caching." A senior says: "We'll use Redis caching, which gives us sub-millisecond reads, but we trade strong consistency for latency, and need a reliability strategy for when Redis goes down."

CAP Theorem

The most famous trade-off in distributed systems. During a network partition, you must choose between consistency and availability.

CAP Theorem C (Consistency) / \ / \ A P (Availability) (Partition Tolerance) During a partition: CP: Reject writes or errors → data is correct but system may be unreachable AP: Serve stale data → system stays up but data may be inconsistent Outside of a partition: most well-designed systems are both C and A.
CategorySystemsDuring Partition
CPZooKeeper, etcd, HBase, MongoDBErrors to preserve correctness
APCassandra, DynamoDB, CouchDB, RiakServes stale data to stay up
TunablePostgreSQL, MySQL, DynamoDBConfigurable per request
Nuance: CAP doesn't mean you always choose C or A. Most real systems are CP or AP with tuning knobs. PostgreSQL can do sync or async replication. DynamoDB offers strong reads per request.

Consistency Spectrum

LevelGuaranteeCost
StrongEvery read returns latest writeHighest latency, sync replication
CausalCausally related ops in orderModerate, causal tracking
SessionConsistent within a sessionModerate, per-session routing
Read-your-writesYour reads reflect your writesLow-moderate, version tracking
EventualConverges eventually if no new writesLowest latency, async replication
When to use each: Bank transfers (strong), social feeds (eventual), shopping carts (read-your-writes). The answer is always "it depends on the use case."

Latency vs Consistency

Strong ConsistencyEventual Consistency
ReplicationSynchronous (wait for replicas)Asynchronous (local ack)
LatencyHigher (bounded by slowest replica)Lower (local write)

Throughput vs Latency

Optimize ThroughputOptimize Latency
BatchingHigher (amortized overhead)Higher (wait for batch)

Storage vs Computation

Write latencySlower (update all views)Fast (write raw data only)

The 5-Step Interview Framework

  1. State the trade-off explicitly — Name both options
  2. Explain both sides — What each gives you, what each costs
  3. Choose one and justify — Link to the specific use case
  4. State when the other choice would be better — This shows maturity
  5. Show awareness of the cost — What did you sacrifice?
Example: "SQL or NoSQL?"

1. STATE: "There's a trade-off — SQL gives us ACID transactions and rich queries,
   while NoSQL gives us horizontal scalability and flexible schema."

2. EXPLAIN: "SQL is better for relational data with complex queries.
   NoSQL is better when write volume exceeds single-node capacity
   or data is unstructured."

3. CHOOSE: "For this messaging system with ordering guarantees,
   I'd choose SQL (PostgreSQL) because message ordering and delivery
   guarantees are critical."

4. REVERSE: "If we were building a real-time analytics pipeline with
   millions of events/sec, I'd choose NoSQL (Cassandra)."

5. COST: "The cost is that we'll need to handle sharding ourselves
   if write volume grows beyond what a single primary can handle."
Interview tip: Never present a solution without addressing the alternative. Say "I'd use X" and immediately address "but if Y were needed..." This alone puts you in the top 20% of candidates.

Common Interview Questions

"When would you choose SQL over NoSQL?"

Choose SQL when: Data has relationships, ACID transactions needed, complex queries, team has SQL expertise
Choose NoSQL when: Write volume exceeds single node, data is unstructured, schema evolves rapidly
Cost of SQL: Vertical scaling ceiling, schema migrations. Cost of NoSQL: No ACID, no joins.

"Cache-aside vs write-through?"

Cache-aside: Check cache first, miss → DB → populate cache. Good for read-heavy.
Write-through: Write to cache AND DB simultaneously. Reads always fresh. Better for write-heavy or freshness-critical.

Default to cache-aside for most systems because reads are more common.
Use write-through when stale reads are unacceptable (user settings, configuration).

"Synchronous vs asynchronous replication?"

Sync: Primary waits for replica ack → zero data loss, higher latency
Async: Primary acks immediately → potential data loss, lower latency

For financial: synchronous. For logging/analytics: asynchronous.
Most databases offer both as a tuning knob.

"Monolith vs microservices?"

Monolith: Simple to develop and deploy, shared DB, lower ops overhead.
  Cost: Coupling grows, single point of failure, scaling is all-or-nothing.
Microservices: Independent scaling and deployment, tech diversity per service.
  Cost: Network latency, distributed transactions, operational complexity, serial
  dependency chain reduces availability.

Rule: Start monolith. Extract services when you have clear boundaries and
independent scaling needs.

What to Skip

  • PACELC theorem (academic extension of CAP)
  • Formal consistency model proofs
  • Academic consensus algorithm proofs (Paxos, Raft)

Focus on practical reasoning, not formal proofs.
