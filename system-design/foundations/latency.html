Latency

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • The difference between latency, response time, and processing time
  • Latency numbers every engineer should memorize
  • Why percentiles matter more than averages
  • How latency compounds in distributed systems
  • Techniques to reduce latency

Definition

Latency is the time elapsed between initiating a request and receiving a response.

Key distinction — Latency vs Response Time: Latency = time for a single operation (network round-trip).
Response Time = total time from user's perspective (includes queuing, processing, rendering).
A single request may have 50ms latency but 200ms response time due to a queue.

Intuition: Memory is ~10,000x faster than disk. Network within a datacenter is ~100x faster than cross-region. Every network hop within a datacenter adds ~0.5ms. Every cross-region hop adds ~50-200ms. This is why capacity planning and data locality matter.

Percentiles (P50, P90, P99, P99.9)

Averages lie. Always use percentiles for latency.

Service A:  P50 = 50ms, P99 = 200ms, P99.9 = 5000ms
Service B:  P50 = 60ms, P99 = 70ms, P99.9 = 80ms

Average (both):  ~75ms

Service A looks better on average, but 0.1% of requests take 5 seconds.
Service B is consistently fast. Service B is the better system for users.

At 1M requests/day, 0.1% = 1,000 users getting 5-second responses per day.
That's a terrible user experience hidden by the average.
Interview tip: When asked about latency targets, always specify a percentile. "P99 latency under 200ms" is a proper SLO. "Average latency under 200ms" is not.

Tail Latency

Tail latency is the latency experienced by the slowest requests. In distributed systems, tail latency compounds.

If each of N services has 99% chance of being fast: Overall P99 = 1 - 0.99^N 10 services in series: 0.99^10 = 0.90 → Only 90% of requests are fast overall 
Example: 10 microservices, each P99 = 50ms
Individual P99: 50ms (each service is fast)
End-to-end P99: much worse because the tail compounds —
  one slow service can make the entire chain slow

This is why "microservices aren't always faster" is a valid critique.

Sources of Latency

Client request lifecycle: DNS Lookup: 1-100ms (cached ~0ms) TCP Handshake: 1 RTT (~1ms same DC, ~50ms cross-region) TLS Handshake: 1-2 RTTs Server Process: 1-500ms (varies wildly) Database Query: 1ms-10s (cached vs disk) Response: same path in reverse Total: 50-200ms (typical) to several seconds (problematic)

Latency in Distributed Systems

Every hop in a distributed system adds latency.

Client → [API Gateway] → [Service A] → [Service B] → [Database] Each → adds ~1-5ms network + processing time Total: single request can touch 20-100ms just in network hops Cross-region calls make this dramatically worse: Client (US) → Service A (US) → Service B (EU) → DB (EU) Added latency: ~80-120ms per cross-region hop

HTTP Request Lifecycle Breakdown

DNS (cached):         0-1ms
TCP handshake:           1 RTT  (~1ms same DC)
TLS handshake:           1-2 RTTs
Server processing:       5-500ms
Response:                Same path back

Total for cached DNS, same DC:
  ~3-7ms network + 5-500ms processing = ~10-500ms total

HTTP/2 eliminates the head-of-line blocking problem and supports multiplexing,
reducing the connection overhead for multiple requests.

Common Interview Questions

  • "What latency numbers should I know?"
    L1:1ns, Memory:100ns, SSD:100μs, HDD:10ms, Same DC:0.5-1ms, Cross-region:50-200ms. This table.
  • "Why are percentiles better than averages for latency?"
    Averages hide tail latency. P99 shows the experience of the worst 1% of users. At scale, those users represent thousands of real people.
  • "How does adding microservices affect latency?"
    Each hop adds network + processing time. Serial service calls compound latency. Cross-region calls add 50-200ms. Trade scalability benefits for latency costs.
  • "How would you reduce latency for a slow API?"
    Cache responses → Add read replicas → Check for N+1 queries → Add Redis cache → Use HTTP/2 → Consider denormalization.
  • "What's the difference between latency and throughput?"
    Latency: time for one operation. Throughput: operations per unit time. High latency doesn't always mean low throughput (batching). See: Throughput.

What to Skip

  • Hardware timing diagrams and CPU pipeline details
  • Kernel-level latency tuning (sysctl parameters)
  • HTTP/3 QUIC protocol internals (unless specifically asked)
