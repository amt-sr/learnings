Scalability

Definition

Scalability is the ability of a system to continue meeting its performance objectives as workload increases by adding resources in a predictable and cost-effective manner.

A scalable system should be able to support growth in:

  • Users
  • Requests
  • Data
  • Geographic regions
  • Services

without requiring a complete redesign.

Scalability does not mean the system is fast today.

It means the system continues to perform well as demand grows.

Performance vs Scalability

This is one of the most misunderstood concepts.

Suppose two systems process 1,000 requests/sec.

System A

1 Server
1000 RPS
CPU = 95%

Cannot grow. Needs complete redesign.

System B

5 Servers
1000 RPS
CPU = 20%

Just add 45 more servers. Performance stayed almost identical.

Performance is how fast you are today.

Scalability is how easily you can handle tomorrow.

Why Scalability Matters

Every successful product eventually experiences growth.

Netflix

2008: Few million users
Today: Hundreds of millions

YouTube

Early: Few TB
Today: Many Exabytes
WhatsApp: Thousands → Billions of messages/day

If the architecture cannot grow gradually, every growth milestone becomes a rewrite.

Types of Scaling

Vertical Scaling (Scale Up)

Increase the capacity of one machine.

8 GB RAM  →  32 GB RAM  →  64 GB RAM
4 CPU     →  16 CPU     →  64 CPU

Advantages:

  • Very simple
  • No distributed systems complexity
  • No replication, no sharding

Disadvantages:

Eventually: Largest machine reached. No further scaling possible. Single point of failure. Cost increases exponentially.

Horizontal Scaling (Scale Out)

Instead of buying a larger machine, buy more machines.

Server A

  ↓

A  B  C  D

Each handles 250 RPS (was 1000 total)

Need 8000 RPS? Simply add more:

A  B  C  D  E  F  G  H

Advantages:

  • Almost unlimited scaling
  • Better fault tolerance
  • Supports global deployments

Disadvantages:

  • Much harder — now you must solve: Load balancing, Distributed cache, Replication, Consistency, Failures, Consensus

This is why distributed systems exist.

Horizontal Scaling Requires Stateless Services

User logs in → Request goes to Server A → Session stored in memory

Next request → Load Balancer → Server B

Server B has no session. User appears logged out.

Solution: Store session outside the application.

  • Redis
  • Database
  • JWT

Now every server behaves identically. Stateless services are one of the biggest enablers of horizontal scaling.

Common Bottlenecks

Every scalable system eventually becomes limited by something. Usually one of:

  • CPU
  • Memory
  • Disk
  • Network
  • Database

You optimize your API → CPU drops → Database becomes bottleneck → Add replicas → Network becomes bottleneck. Optimization is iterative.

Real Example

Today:    100 schools  → 500 RPS   → Single server (works)

Next year: 10000 schools → 40000 RPS → One server cannot handle

Option A (Vertical):
  Bigger EC2 → Bigger EC2 → Bigger EC2 → Eventually impossible

Option B (Horizontal):
  Load Balancer → App1 App2 App3 App4 → Each serves ~10000 RPS

Database also becomes bottleneck:
  Read Replicas → Partitioning → Sharding → Caching

Each scalability technique solves the next bottleneck.

Trade-offs

Vertical ScalingHorizontal Scaling
SimplerMore Complex
Limited GrowthNearly Unlimited Growth
Single FailureBetter Fault Tolerance
Expensive Large MachinesCheaper Commodity Machines
Easier DebuggingDistributed Systems Complexity

Interview Expectations

When an interviewer asks "How would you scale this?", they expect you to think in layers:

  1. Remove obvious bottlenecks
  2. Add caching
  3. Add load balancers
  4. Make services stateless
  5. Scale reads (read replicas)
  6. Scale writes
  7. Partition data
  8. Scale globally

Not "Let's add Kubernetes." Kubernetes doesn't solve scalability by itself. It only makes deployment easier.
