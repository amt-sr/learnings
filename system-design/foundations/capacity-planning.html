Capacity Planning

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • The step-by-step estimation framework used in interviews
  • How to estimate QPS, storage, bandwidth, and server count
  • How to reason about memory and cache sizing
  • Worked examples for Twitter, YouTube, and WhatsApp

Definition

Capacity planning is the process of determining the resources needed to meet current and future workload demands.

Analogy: Planning how many checkout lanes a supermarket needs for Black Friday. Too few = long queues. Too many = wasted money. The goal is the right number with headroom for spikes.

Why It Matters in Interviews

Almost every system design interview requires back-of-envelope estimation. "How much storage for Google Photos?" "How many servers for Twitter?" This is where you demonstrate quantitative reasoning.

The Estimation Framework

Step 1: Clarify scale DAU, MAU, total users, growth rate Step 2: Requests per user per day From similar systems or user behavior Step 3: Total requests per day DAU × requests/user/day Step 4: Peak QPS Average QPS × peak factor (2-3x) Some apps use 5-10x for bursty traffic Step 5: Read/write split Apply read:write ratio (10:1 to 100:1 for social) Step 6: Storage estimation Data size per item × items/day × retention × replication factor Step 7: Bandwidth Total bytes/sec in and out Step Step 8: Memory and cache Working set size × cache overhead

Worked Example: Twitter

Given:  DAU = 100M, 2 tweets/day/user, read:write = 100:1, avg tweet = 200 bytes

Writes:
  Total writes/day: 100M × 2 = 200M
  Peak write QPS: 200M / 86400 × 3 (peak factor) = 6,944

Reads:
  Peak read QPS: 6,944 × 100 = 694,400
  Assuming 95% cache hit rate: DB read QPS = 34,720

Storage (per year):
  200M × 365 = 73 billion tweets/year
  73B × 200 bytes = 14.6 TB/year raw data
  × 1.5 (metadata overhead) = 21.9 TB
  × 3 (replication factor) = 65.7 TB

Servers (assuming 5,000 RPS/server):
  Writes: 6,944 / 5,000 = 2 servers (×2 for redundancy) = 4
  Reads (DB only): 34,720 / 5,000 = 7 servers (×2 for redundancy) = 14
  App servers (if needed): 5-10

Network bandwidth (peak writes):
  6,944 × 200 bytes = 1.39 MB/sec (1.1 GB/min)

Worked Example: YouTube

Given: DAU = 5M, 500K video uploads/day, avg video = 500MB, 1K views/video

Uploads:
  Daily upload: 500K × 500MB = 250 TB/day
  Monthly: 250TB × 30 = 7.5 PB/month
  ×3 replication = 22.5 PB/month

Views (read):
  Daily views: 500K × 1000 = 500M views/day
  Peak QPS: 500M / 86400 × 3 = 17,361 QPS

Storage (yearly, metadata only in DB):
  500K videos/day × 365 = 182.5M videos
  Per video metadata: ~1KB (title, description, creator, views, timestamps)
  182.5M × 1KB = 182.5 GB/year
  × 3 replication = 547.5 GB

CDN bandwidth (daily):
  Assume 50MB avg video, 1K views:
  500M × 50MB = 25 PB/day through CDN (impossible)
  → This is why YouTube uses adaptive bitrate streaming
  → CDN serves multiple quality levels, not the full file every time


Worked Example: WhatsApp


Given: DAU = 1B, 30 messages/day/user, avg message = 100 bytes

Peak QPS:
  1B × 30 / 86400 × 2 = 694,444 QPS

Storage (per year):
  1B × 30 × 365 = 10.95 trillion messages
  10.95T × 100 bytes = 1.095 PB/year
  × 3 replication = 3.29 PB/year

This is why WhatsApp rewrote their server in Erlang —
they needed to handle millions of concurrent connections,
not millions of QPS.


Server Sizing



  
Server TypeTypical CapacityUse For

  
Small (4-core, 8GB RAM)1,000-3,000 RPSAPI gateways, lightweight services

  
Medium (8-core, 32GB RAM)3,000-10,000 RPSApplication servers, workers

  
Large (16-core, 64GB RAM)10,000-20,000 RPSDatabase replicas, compute-heavy services





Example: Need 50,000 peak QPS
  Using medium servers (5,000 RPS each):
  50,000 / 5,000 = 10 servers
  With 50% headroom: 15 servers
  With redundancy (2×): 15 × 2 = 30 servers



Cache Sizing


Working set = data accessed frequently × size per item

Example: 10M active users, each with a profile (~5KB cached data)
  Working set = 10M × 5KB = 50GB
  Plus overhead (key size, metadata): ~60-70GB
  Need Redis cluster with ~64GB memory per node

Hot data rule of thumb: Cache the data that 20% of users access 80% of the time.



Capacity Planning Lifecycle


  Monitor  →  Analyze  →  Forecast  →  Provision  →  Verify
  (current)    (trends)    (growth)    (acquire)    (validate)



  Key principle: Capacity planning is not a one-time activity. It's a continuous cycle. The moment you stop monitoring, your capacity assumptions become stale.




Common Interview Questions


    
  
  • "Estimate storage for a photo-sharing app with 100M users, each uploading 3 photos/day."
    100M × 3 × 2MB × 365 = 219 TB/year raw. With 3x replication: 657 TB.
    • 
  
  • "How many servers for a messaging system with 1B DAU?"
    Peak QPS estimation → server capacity → add headroom → multiply for redundancy.
    • 
  
  • "How do you plan capacity for a system growing 10% month?"
    Project current usage × 1.1^12 for next year. Plan infrastructure 3-6 months ahead.
