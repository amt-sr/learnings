Fault Tolerance

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • The difference between faults, errors, and failures
  • Types of faults and how to handle each
  • Why distributed systems make fault tolerance hard
  • Specific techniques: circuit breakers, bulkheads, retries, timeouts, idempotency
  • Failure detection and why perfect detection is impossible

Definition

Fault tolerance is the ability of a system to continue operating correctly in the presence of component failures, degrading gracefully rather than crashing.

Analogy: A car with a spare tire is fault-tolerant to a flat tire. It doesn't prevent the flat, but it keeps you moving. The spare tire is redundancy. Knowing how to change it is the recovery plan.

Fault vs Error vs Failure

This distinction is critical. Interviewers expect you to use these terms precisely.

FaultErrorFailure
WhatA defect in a componentDeviation from correct stateSystem no longer delivers service
ExampleBuggy line of code, memory leakWrong calculation, corrupted stateService returns 500 errors
CauseDesign flaw, wear and tearFault activated at runtimeError propagated to user
The cascade: Fault → Error → Failure. Not every fault causes an error, and not every error causes a failure. Fault tolerance aims to stop the cascade at the fault stage or contain the error before it becomes a failure.

Why Fault Tolerance is Hard in Distributed Systems

The fundamental challenge: In a distributed system, partial failures are normal. You cannot know with certainty whether a remote node is dead, slow, or the network is partitioned. The FLP impossibility theorem proves that in asynchronous networks, there is no algorithm that can reliably distinguish a crashed node from a slow one. 
The CAP theorem captures this: during a network partition, you must choose between
consistency and availability. You cannot have both. Every distributed system design
is an implicit bet on which failures are more likely.

Fault Tolerance Techniques

1. Redundancy and Replication

Active-Active: [Server A] ←→ [Server B] Both serve traffic. Either can fail. Active-Passive: [Server A (serving)] → [Server B (standby)] B takes over only when A fails.

2. Circuit Breakers

Stop cascading failures by failing fast when a dependency is unhealthy.

CLOSED (normal) ──threshold exceeded──→ OPEN (fail fast) ↑ │ after timeout └──single success──────────────→ HALF-OPEN (probe)
CLOSED:    Normal requests flow to dependency
OPEN:       All requests return immediately (fallback/error)
HALF-OPEN:  One test request; success→CLOSED, fail→OPEN

Configurable: failure count threshold, timeout before retry, half-open success count

See: Circuit Breaker

3. Bulkheads

Isolate failures so one dependency cannot exhaust shared resources.

Without:  Shared thread pool → Slow Service B → all threads blocked → app dead
With:     Pool A → Service A (isolated)
           Pool B → Service B (slow, but only Pool B affected)
           Pool C → Service C (unaffected)

See: Bulkhead

4. Retries with Exponential Backoff

Request fails → Wait 1s + jitter → Retry → fails
  → Wait 2s + jitter → Retry → fails
  → Wait 4s + jitter → Retry → success

Formula:  delay = base * 2^attempt + random(0, jitter_ms)
Max: 3-5 retries, cap at ~60s
ALWAYS add jitter to prevent thundering herd.
Retries MUST be paired with idempotency. Use idempotency keys for non-idempotent operations (payments, orders).

5. Timeouts

Every remote call MUST have an explicit timeout. Without timeouts, a hung connection blocks a thread forever.

Connection timeout: 5s  (how long to establish connection)
Read timeout:    10s  (how long to wait for response)
Overall timeout:  15s  (total time budget)

Rule: Set timeouts based on the slowest NORMAL response time, not the fastest.
If p99 latency is 200ms, set read timeout to 2-5 seconds (10-25x).

6. Idempotency

An operation is idempotent if executing it multiple times produces the same result as once.

Non-idempotent: POST /charge { amount: 50 }  → Retry → double charge!
Idempotent:     POST /charge { idempotency_key: "req_abc" }  → Retry → cached result

Design: Use PUT instead of POST (PUT is idempotent by definition). Use idempotency keys for POST.

7. Graceful Degradation

Normal:      Full search with filters, sorting, recommendations
Degraded:     Basic search (recommendations service down)
Further:      Cached popular results (search service down)
Unavailable: "Try again later" (entire backend down)

See: Graceful Degradation

8. Rate Limiting

Prevent a single client from overwhelming the system, which could cause failures.

  • Token Bucket — burst-friendly, most common
  • Leaky Bucket — steady rate
  • Sliding Window — smoother limits

Failure Detection

How do you know something has failed?

  • Heartbeats: Periodic "I'm alive" messages. Missing N heartbeats = failure suspected.
  • Phi Accrual Detector: (Cassandra/Akka) Adaptive — models heartbeat arrival times as a Gaussian distribution. Phi value represents suspicion level. Better than fixed thresholds because it adapts to network conditions.
  • Gossip protocols: Nodes exchange failure information. All nodes eventually learn about failures. Used by Cassandra, Consul.
FLP Impossibility (1985): In an asynchronous network, there is no deterministic algorithm that can solve consensus if even one process can crash. This is why systems like ZooKeeper and etcd use timeouts (which assume a synchronous window) rather than pure async detection.

Byzantine Fault Tolerance

Most systems only handle crash faults (the node stops). Byzantine faults are when a node sends contradictory messages to different peers — like a general sending "attack north" to half the army and "attack south" to the other half.

Why most systems don't need BFT: BFT is expensive (requires 3f+1 nodes for f Byzantine faults). In practice, the risk of a node behaving arbitrarily is much lower than the risk of it crashing. ZooKeeper, etcd, and most production systems handle crash faults only.

BFT is needed in: aerospace, financial exchanges, blockchain (where participants are adversarial).

Real World Examples

  • Netflix: Hystrix (now Resilience4j) for circuit breakers. Thread pool isolation (bulkheads). Chaos Monkey for testing.
  • Google: Borg preempts tasks for hardware failures. Service-level auto-restart within seconds.
  • AWS: Multi-AZ deployment. Automated instance recovery. Route 53 health checks trigger failover.

Common Interview Questions

  • "How do you prevent cascading failures?"
    Circuit breakers (stop calling failing service), bulkheads (isolate thread pools), timeouts (never block forever), graceful degradation.
  • "Why do retries need jitter?"
    Without jitter, all clients retry at the same intervals after a shared failure → synchronized waves of load that crash the recovering service. Random jitter spreads retries over time.
  • "What happens when a database fails?"
    Circuit breaker opens → requests go to fallback → Read from replicas (stale reads) → Retry with backoff → Failover to standby → Graceful degradation.
  • "How do you design idempotent retries?"
    Idempotency key on every request. Server stores result per key. Retries with same key return cached result without re-executing.
  • "What's the difference between circuit breaker and bulkhead?"
    Circuit breaker: Protects the caller from a failing dependency. Bulkhead: Prevents one dependency from consuming all resources.
  • "What is graceful degradation?"
    When a non-critical component fails, reduce functionality instead of going down entirely. Serve cached content, show simplified UI, disable features.

What to Skip

  • PBFT, Raft, Paxos protocol details (unless specifically asked)
  • FLP impossibility proof details
  • Specific resilience library implementations
