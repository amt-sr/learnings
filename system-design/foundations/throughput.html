Throughput

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What throughput is and how it differs from latency
  • Little's Law and how to apply it
  • Amdahl's Law and why adding servers has diminishing returns
  • How to estimate QPS in interviews
  • Factors that limit throughput

Definition

Throughput is the number of operations a system processes per unit of time.

Throughput = Operations / Time Common units: RPS (Requests Per Second), QPS (Queries Per Second), TPS (Transactions Per Second), IOPS (I/O Operations Per Second)
Analogy: Latency is the speed limit on a highway. Throughput is how many cars pass through per hour. A narrow lane (low latency) can still have high throughput if cars are bumper-to-bumper.

Throughput vs Latency

Optimize LatencyOptimize Throughput
StrategySmaller requests, parallelism, cachingBatching, larger payloads, pipelining
BatchingHigher latency (wait for batch)Higher throughput (amortized overhead)
Connection poolingFewer connection setupsFewer connection setups
ConcurrencyFewer connections, less contentionMore concurrent connections
Use whenUser-facing APIs, real-time systemsETL, log processing, bulk imports
Key insight: High latency does NOT always mean low throughput. A message queue processing batches of 1000 messages has high per-message latency but very high throughput. This is why latency and throughput are independent metrics that must be specified separately.

Little's Law

The most useful queueing theory result for system design.

L = λ × W L = number of items in the system (queue + processing)
λ = arrival rate (items/time)
W = average time an item spends in the system 
Example 1 — Concurrent users:
   DAU = 10M, average time on site = 5 min = 300s
  Arrival rate = 10M / 86400 = ~116 requests/sec
  Concurrent users L = 116 × 300 = 34,700

Example 2 — Server capacity:
  Server handles 5ms per request, want max latency of 50ms
  W = 50ms = 0.05s
  Max throughput = L/W = 1 / 0.05 = 20 RPS per server

Example 3 — How many servers?
  Need to handle 10,000 RPS, each server handles 2,000 RPS
  Servers needed = 10,000 / 2,000 = 5 servers
  With redundancy: 5 + 2 = 7 servers

Amdahl's Law

Adding more processors has diminishing returns because only the parallelizable portion benefits.

Speedup(n) = 1 / ((1 - p) + p / n) p = fraction of work that is parallelizable
n = number of processors 
Example: 8 servers, p = 0.5 (50% parallelizable)
Speedup = 1 / (0.5 + 0.5/8) = 1 / 0.5625 = 1.78x

You added 8x the hardware, got only 1.78x the speedup.
The serial portion is the bottleneck.

If p = 0.95 (95% parallelizable):
Speedup = 1 / (0.05 + 0.95/8) = 1 / 0.16875 = 5.93x

Better, but still not linear. This is why
"just add more servers" doesn't always work.

Universal Scalability Law (Gunther)

Extends Amdahl by accounting for contention and coherency overhead.

Throughput(n) = n / (1 + σ(n-1) + δn(n-1)) σ = contention coefficient (waiting for shared resources)
δ = coherency coefficient (overhead of keeping state consistent)
Real-world implication: 10 servers might only deliver 6x throughput, not 10x. Adding more servers beyond a point can actually decrease throughput due to contention/coherency overhead. This is why microservices with many inter-service calls can perform worse than monoliths at the same request rate.

QPS Estimation in Interviews

Step 1: Clarify scale "How many DAU?" → Estimate MAU if needed Step 2: Requests per user per day Typical: 10-50 for social apps, 5-20 for productivity Step 3: Total requests per day DAU × requests per user per day Step 4: Average QPS Total requests per day / 86400 Step 5: Peak QPS (use peak factor) Peak QPS = Average QPS × 2 to 3 (some use 5x for bursty apps) Step 6: Read/write split Apply read:write ratio (typically 10:1 to 100:1 for social apps) 
Examples:

Twitter:   DAU=100M, 2 tweets/day per user, read:write=100:1
  Total writes/day: 200M
  Peak write QPS: 200M / 86400 × 3 = 6,944
  Peak read QPS: 6,944 × 100 = 694,400
  Cache 95% reads: DB read QPS = 34,720

WhatsApp: DAU=1B, 30 messages/day per user
  Peak QPS: 1B × 30 / 86400 × 2 = 694,444

URL Shortener: DAU=1M, 5 shortens/day
  Peak write QPS: 1M × 5 / 86400 × 3 = 174

YouTube: DAU=5M, 30 views/day
  Peak QPS: 5M × 30 / 86400 × 3 = 5,208

Common Interview Questions

  • "How do you estimate QPS for a system?"
    DAU → requests/day per user → total/day → average QPS → peak QPS (×2-3) → split reads/writes.
  • "What limits throughput more — CPU or database?"
    Depends on the workload. Read-heavy: DB. Compute-heavy: CPU. Write-heavy: DB writes. Most real systems hit DB first because adding app servers is easier than scaling databases.
  • "Why doesn't adding servers always increase throughput linearly?"
    Amdahl's Law (serial portion limits parallelism), Universal Scalability Law (contention/coherency overhead), shared resource bottlenecks.
  • "How does Little's Law help with capacity planning?"
    Calculate concurrent users (L) from arrival rate (λ) and time on site (W). L = λ × W. This tells you how many connections your servers need.
