Bottleneck Analysis

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What a bottleneck is and why every system has one
  • The typical order in which bottlenecks appear as you scale
  • How to identify and eliminate each type
  • How to estimate bottlenecks without running the system

Definition

A bottleneck is the component that limits overall system throughput or performance. Like the narrowest section of a pipe — regardless of how wide the rest is, flow is limited at that point.

The fundamental truth: Optimization is iterative. You fix one bottleneck, another emerges. This is the normal engineering cycle. Understanding this pattern is what separates experienced architects from novices.

Common Bottlenecks in Web Systems

CPU100% utilization, high load avg, GC pausesHorizontal scaling, caching, optimize hot paths

Disk I/OHigh iowait, slow queries, seek latencySSD, cache, read replicas, connection pooling

DatabaseSlow queries, lock contention, N+1 queriesIndexes, sharding, caching, query optimization

The Bottleneck Hierarchy

In a typical web application, bottlenecks appear in a predictable order as you scale.

Scale Stage 1: Single server — everything is the bottleneck Scale Stage 2: CPU/memory optimized → Database becomes the bottleneck Scale Stage 3: Read replicas added → Database writes become the bottleneck Scale Stage 4: Write sharding added → Network becomes the bottleneck Scale Stage 5: Multi-region → Cross-region latency is the bottleneck Scale Stage 6: Extreme scale → Cost becomes the bottleneck
Interview tip: When asked "what's the bottleneck?", start from the top. Most systems hit the database first. The specific bottleneck depends on read/write ratio, query patterns, and data size.

Database Bottlenecks (Most Common)

Too many reads

Symptoms: High database CPU, slow queries even with indexes.

Mitigation:  Add read replicas
           Add Redis/Memcached cache layer
           Use covering indexes

Too many writes

Symptoms: Write latency increasing, lock contention, replication lag growing.

Mitigation: Shard the data by key
           Use write buffering / message queue (Kafka)
           Batch writes where possible
           Use optimistic locking instead of pessimistic

Slow queries

Symptoms: Specific queries taking seconds.

Mitigation: EXPLAIN ANALYZE to find slow queries
           Add appropriate indexes
           Rewrite queries to avoid full table scans
           Consider denormalization for frequently-joined data

Connection exhaustion

Symptoms: "Too many connections" errors, connection queue buildup.

Mitigation: Increase max_connections in database config
           Use connection pooling (HikariCP, PgBouncer)
           Reduce connection holding time (faster queries)

Network Bottlenecks

  • DNS latency: Uncached DNS adds 1-100ms per hop. Use cached DNS or mDNS.
  • TCP handshake overhead: Each new connection costs 1 RTT. Use connection pooling and HTTP/2.
  • TLS overhead: 1-2 extra RTTs. Connection reuse and session resumption reduce this.
  • Bandwidth saturation: Large payloads or high QPS exceed available bandwidth. Use compression, CDN.

The "Fix One, Find Next" Principle

Step 1: CPU at 95% → Optimize hot code Step 2: CPU drops to 30% → Database becomes bottleneck Step 3: Add read replicas → DB reads solved, writes bottlenecked Step 4: Shard writes → Network bandwidth becomes bottleneck Step 5: Add multi-AZ → Cross-region latency is the bottleneck This never ends — you just move to harder problems. Which is the whole point of system design.

Back-of-Envelope Bottleneck Estimation

Example: "100K QPS, each request does 1 DB read (5ms)"

  Concurrent DB connections needed: 100,000 × 0.005 = 500 connections
  If max_connections = 100 → bottleneck!

  Solution: Add connection pool or reduce per-request DB calls.
  Cache the DB reads → 500 connections × (5ms / 100ms with cache) = 25 needed.

Real World Examples

  • Stack Overflow: Started on a single SQL Server. Moved to CQRS architecture with separate read/write stores. Used Redis for caching. Now handles billions of requests/month.
  • WhatsApp: Rewrote server in Erlang to handle 2M+ concurrent connections per server. Each connection uses ~300KB → 2M × 300KB = 600GB RAM per server.
  • Discord: Elixir + Rust backend for real-time chat. Moved from 100% Go to hybrid. Cron jobs offloaded to Rust workers.

Common Interview Questions

  • "What's the bottleneck in your proposed design?"
    Walk through the hierarchy: "App servers can handle 50K QPS, but the database is maxed at 10K writes/sec, so the write path is the bottleneck. We'd shard by user ID."
  • "How do you identify bottlenecks?"
    Monitor CPU, memory, disk I/O, network I/O, queue lengths. Use APM tools (Datadog, New Relic). Look for resource saturation and queue buildup.
  • "What happens when the database becomes the bottleneck?"
    Add read replicas → Cache frequently-accessed data → Shard writes → Offload heavy queries to a separate analytical DB.
  • "How do you handle a bottleneck you can't eliminate?"
    Accept the limit and design around it. Rate limit to stay within capacity. Prioritize critical paths. Use graceful degradation.

What to Skip

  • Profiler tool specifics (perf, Flame Graphs, dtrace)
  • Kernel-level tuning (sysctl, /proc settings)
  • Hardware performance counters
