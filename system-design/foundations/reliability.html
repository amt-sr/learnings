Reliability

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What reliability means and how it differs from availability and durability
  • How to measure reliability quantitatively (MTBF, MTTR, failure rate, R(t))
  • Architectural patterns that make systems reliable
  • How failures are classified and how to design for each type
  • How to reason about reliability in system design interviews

Definition

Reliability is the probability that a system performs its intended function correctly over a specified period of time under stated conditions.

Reliability = Probability(system works correctly at time t) Often measured as R(t), the reliability function
Intuitive definition: If I use this system for a year, what is the chance it will give me the correct answer every time? A system can be 99.99% available (always reachable) but unreliable (returns wrong data).
Real-world analogy: A car that always starts when you turn the key (available) but sometimes takes you to the wrong address (unreliable). Availability is about reachability. Reliability is about correctness.

Why Reliability Matters

Without reliability, users cannot trust the system. A banking system that occasionally shows the wrong balance is worse than one that is occasionally down.

The cost of unreliability: In 2023, Stripe lost transaction history due to corrupted backups. In 2017, GitLab accidentally deleted a production database and lost 6 hours of data. Reliability failures directly translate to financial loss, reputational damage, and user trust erosion.

Reliability vs Availability vs Durability

These three concepts are constantly confused in interviews. Know the differences cold.

Interview trap: A system can be available but not reliable (always up, returns wrong data). A system can be reliable but not available (correct when up, but down frequently). Most interview questions require both.

Core Metrics

MTBF — Mean Time Between Failures

The average time the system operates correctly between two consecutive failures.

MTBF = Total Uptime / Number of Failures Higher is better. A system with MTBF of 30 days fails on average once per month.

MTTR — Mean Time To Repair

The average time it takes to restore the system to working order after a failure.

MTTR = Total Downtime / Number of Failures Lower is better. If MTTR is 5 minutes, the system is back quickly.

Reliability Function R(t)

The probability that the system operates correctly from time 0 to time t without failure.

R(t) = e^(-λt) (for constant failure rate λ) λ = 1 / MTBF. This is the exponential distribution model. 
Example: MTBF = 720 hours (30 days)
λ = 1/720 = 0.00139 failures/hour

R(24h) = e^(-0.00139 * 24) = e^(-0.0333) = 0.967 = 96.7%
  → 96.7% chance the system runs correctly for a full day

R(720h) = e^(-0.00139 * 720) = e^(-1) = 0.368 = 36.8%
  → Only 36.8% chance it runs for 30 days without failure

Adding redundancy in parallel increases R(t) significantly.

Failure Rate

Failure Rate = 1 / MTBF = λ Measured in failures per unit time. Common unit: FIT (failures per 10^9 hours) for hardware.
Key relationship: Reliability improves when MTBF increases (fail less often) or MTTR decreases (recover faster). Most engineering effort goes into increasing MTBF through redundancy, replication, and fault tolerance. SRE effort goes into decreasing MTTR through automated recovery, monitoring, and runbooks.

How to Measure Reliability

SLIs for Reliability

  • Request success rate: Ratio of successful (2xx/3xx) responses to total requests
  • Error budget: 100% - SLO. If SLO is 99.95%, error budget is 0.05%
  • Data consistency lag: Time between a write and it being visible on all replicas
  • Stale read rate: Percentage of reads returning data older than the latest write
  • Retry rate: Percentage of requests requiring retries — high rates signal underlying problems
  • Circuit breaker open rate: How often circuit breakers trip
Google's rule: Set your SLO so you are occasionally in danger of missing it. If you never come close to violating your SLO, you are over-investing in reliability. The error budget tells you how much risk you can take.

Reliability Patterns

1. Redundancy

Run multiple copies of every critical component. A single server is a single point of failure.

Without: Client → [Server] → [DB] With: Client → [LB] → [Server A] → [Server B] → [Server C] ↓ [DB Primary] [DB Replica]

2. Replication

Copy data across nodes so a single node failure does not cause data loss.

Synchronous: Client → [Primary] ──ack after──→ [Replica] Asynchronous: Client → [Primary] → ack immediately └──async──→ [Replica]

3. Failover

Automatically switch from a failed component to a healthy one.

Primary fails → Health check detects (seconds) → Replica promoted → Traffic rerouted
Hot standby:    Always ready, 0s cold start, highest cost
Warm standby:   Running idle, ~10-60s recovery
Cold standby:  Off, ~minutes to start, cheapest

4. Circuit Breakers

Prevent cascading failures by failing fast when a dependency is unhealthy.

CLOSED ──failures exceed threshold──→ OPEN (fail fast) ↑ │ after timeout └───────request succeeds────── HALF-OPEN (test one)

5. Bulkheads

Isolate failures so one failing dependency cannot take down the entire system.

Without Bulkhead:  Shared thread pool → Service B slow → all threads blocked → app down
With Bulkhead:     Pool A → Service A (isolated)
                   Pool B → Service B (isolated, slow but contained)
                   Pool C → Service C (unaffected)

6. Retries with Exponential Backoff

Request fails → Wait 1s + jitter → Retry → fails → Wait 2s + jitter → Retry → fails → Wait 4s + jitter
Formula: delay = base * 2^attempt + random(0, jitter_ms)
Max retries: 3-5. MUST have jitter to avoid thundering herd.
Retries require idempotency. If a payment is retried, you must not charge twice. Use idempotency keys.

7. Health Checks

Continuous monitoring to detect failures and trigger automated recovery. Liveness (alive?), readiness (can serve?), startup (initialized?).

8. Graceful Degradation

Normal:   Full search with filters, autocomplete, recommendations
Level 1:  Basic search (autocomplete down)
Level 2:  Cached results only (search service down)
Level 3:  "Search unavailable" + popular results

Real World Examples

  • Google (Borg): Every job runs multiple replicas across failure domains. Scheduler anti-affines tasks so a single machine/rack failure never takes down a service. Automated restart within seconds.
  • Netflix (Chaos Engineering): Chaos Monkey randomly kills production instances. Hystrix for circuit breakers. Bulkheads via thread pool isolation. Serve cached content when backends fail.
  • Amazon (Cell-based): Each service runs in isolated cells. A failure in one cell does not propagate. "Failure as a normal operational event."

Common Interview Questions

  • "How would you make this system reliable?"
    Identify failure modes → Add redundancy → Replicate data → Automated failover → Circuit breakers + bulkheads → Idempotent operations → Graceful degradation → Set SLIs/SLOs.
  • "Reliability vs availability — which is more important?"
    Depends on the system. Banking: reliability (correct amounts). CDN: availability (content served). Always ask the interviewer what matters for their specific system.
  • "Third-party API is unreliable. How do you handle it?"
    Circuit breaker → Cache previous responses → Graceful degradation → Tight timeouts → Retry with backoff → Monitor error rates → Fallback provider.
  • "How do retries avoid thundering herd?"
    Exponential backoff with jitter: delay = min(base * 2^n + random(0, jitter_ms), max_delay). Jitter spreads retries randomly.
  • "What is an error budget?"
    100% - SLO. If SLO is 99.95%, budget is 0.05%. Spent on incidents AND new releases. When budget is exhausted, slow down deploys, focus on reliability.
  • "System returns wrong data intermittently. How to diagnose?"
    Check race conditions → Replication lag → Cache staleness → Clock skew → Add request tracing → Route critical reads to primary.

What to Skip

  • ITIL processes and frameworks
  • FMEA / FTA (manufacturing reliability engineering)
  • Weibull distribution, hardware-specific MTBF calculations
  • Specific SRE tooling (Prometheus/Grafana configs)

Focus on architectural patterns and quantitative reasoning.
