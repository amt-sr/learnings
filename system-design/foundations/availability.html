Availability

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand:

  • What availability means and how it's measured
  • The nines scale and what each level costs
  • How availability differs from reliability
  • Techniques to achieve high availability
  • How to calculate and reason about availability in interviews

Definition

Availability is the proportion of time a system is operational and able to serve requests over a given period.

Availability = Uptime / (Uptime + Downtime) Expressed as a percentage or in "nines" (e.g., 99.99% = four nines)
Key insight: Availability does NOT guarantee performance. A system returning error pages in 5ms is "available" in the sense that it's responding — but it's not functionally available. Always clarify what "available" means in the context of the system.

The Nines Scale

This is the single most important table to memorize for interviews. Every interviewer expects you to know these numbers.

Availability Nines Downtime / Year Downtime / Month Downtime / Week
99% Two 3.65 days 7.3 hours 1.68 hours
99.9% Three 8.76 hours 43.8 min 10.1 min
99.99% Four 52.6 min 4.38 min 1.01 min
99.999% Five 5.26 min 26.3 sec 6.05 sec
99.9999% Six 31.5 sec 2.63 sec 0.6 sec
Interview tip: When given a design problem, the interviewer expects you to ask about the availability target. A good opening: "What availability are we targeting — 99.9%, 99.99%?" This immediately signals that you think in terms of SLAs.

What Each Level Means in Practice

NinesTypical SystemWhat It Takes
99% (two)Internal tools, blogsSingle server, basic monitoring
99.9% (three)SaaS products, most APIsRedundancy, load balancing, automated restarts
99.99% (four)Payment systems, cloud servicesMulti-AZ deployment, failover, circuit breakers
99.999% (five)Core infra (DNS, auth)Multi-region, zero-downtime deploys, chaos engineering
99.9999% (six)Global financial exchanges, 911Everything above + extreme redundancy, custom hardware

Availability vs Reliability

Interviewers test whether you understand the difference. They are related but not the same.

AvailabilityReliability
MeasuresIs the system up?Does the system work correctly?
Question"Can I reach it?""Does it give the right answer?"
Failure modeSystem is unreachableSystem returns wrong data
MetricUptime percentageMTBF / MTTR
Example: A system can be 99.99% available but return stale data 50% of the time. It's up, but it's not reliable. Conversely, a system can be correct every time it responds but goes down frequently — reliable but not available.

Serial vs Parallel Availability

When a request flows through multiple components, overall availability depends on whether they are in series or parallel.

Serial (All must be up)

Client → [Load Balancer] → [App Server] → [Database] → [Cache] If any component fails, the request fails. All components are in series.
Atotal = A1 × A2 × A3 × ... × An Serial availability is always LESS than any individual component. 
Example:  LB=99.99%  App=99.99%  DB=99.99%  Cache=99.99%

A_total = 0.9999 × 0.9999 × 0.9999 × 0.9999
        = 0.9999⁴
        = 99.96%

Four nines components → three nines system.
Adding more components always reduces availability in series.
This is why microservices can reduce overall availability. Each extra service in the request path multiplies the failure probability. This is the "microservices tax."

Parallel (Any one is enough)

Client → [Server A] → [Server B] (any one can serve) → [Server C]
Atotal = 1 - (1 - A1) × (1 - A2) × ... × (1 - An) Parallel availability is always MORE than any individual component. 
Example:  Each server = 99%

A_total = 1 - (1 - 0.99)³
        = 1 - 0.000001
        = 99.9999%

Three 99% servers → six nines.
Redundancy is the main tool for achieving high availability.

How to Achieve High Availability

1. Redundancy

Remove single points of failure by having backups for every component.

ComponentRedundancy Strategy
App ServersRun multiple instances behind a load balancer
DatabasePrimary + replicas
CacheRedis Cluster with replicas
API GatewayDeploy across multiple AZs
Data CenterMulti-region deployment
DNSMultiple DNS providers, low TTL

2. Failover

When a component fails, another takes over automatically.

Primary fails
    ↓
Detected by health check (seconds)
    ↓
Replica promoted to primary
    ↓
Traffic rerouted
    ↓
Total downtime: seconds, not hours

Types:

  • Hot standby: Replica is always running and ready. Zero cold start. Higher cost.
  • Warm standby: Replica is running but at reduced capacity. Takes seconds to scale up.
  • Cold standby: Replica is off. Takes minutes to start. Cheapest, highest downtime.

3. Load Balancing

Distributes traffic across healthy instances. Detects and removes failed nodes.

┌→ Server A (healthy) Client → LB ─┼→ Server B (healthy) └→ Server C (failed ✕) ← LB stops sending traffic here

Key behaviors:

  • Health checks detect failures
  • Round Robin or Least Connections distribute load
  • Sticky sessions for stateful services (though stateless is preferred)

4. Multi-Region / Multi-AZ

A single data center is a single point of failure. Deploy across Availability Zones and regions.

Region: US-East ├── AZ-1: App × 3, DB Primary └── AZ-2: App × 3, DB Replica Region: US-West ├── AZ-1: App × 3, DB Replica (cross-region) └── AZ-2: App × 3
FailureSurvives?Strategy
Single server crashYesLoad balancer reroutes
Entire AZ goes downYesTraffic shifts to other AZ
Entire region goes downYesDNS routes to other region
Entire cloud provider failsYesMulti-cloud (rare, expensive)

5. Graceful Degradation

When parts of the system fail, degrade functionality instead of going down entirely.

Normal:    Full search with filters, sorting, pagination
Degraded:  Basic search, no filters (cache hit)
Further:   "Search is temporarily unavailable" + cached popular results

Examples in real systems:

  • Netflix serves cached content when the recommendation engine is down
  • Instagram shows feed without stories when stories service fails
  • Amazon shows "Buy" button but disables recommendations during outages

6. Circuit Breaker

Stop cascading failures by failing fast when a dependency is down.

Normal flow:   App → Service B → Database
               Service B starts failing
                  ↓
Circuit opens: App → Circuit Breaker → Immediate error/fallback
                  ↓
               No more requests hammering Service B
                  ↓
               Service B recovers
                  ↓
Circuit half-opens: Test one request
                  ↓
               If success → circuit closes, normal flow resumes

Related: Circuit Breaker, Bulkhead

7. Rate Limiting and Backpressure

Prevent the system from being overwhelmed, which causes cascading failures and downtime.

  • Token Bucket — burst-friendly, most common
  • Leaky Bucket — steady rate
  • Sliding Window — smoother limits

Backpressure: When downstream is slow, upstream slows down too — instead of everyone crashing.

8. Zero-Downtime Deployments

Deployments should not cause availability dips.

StrategyHow
Blue-GreenTwo identical environments. Switch traffic atomically.
CanaryRoute small % of traffic to new version. Monitor, then roll out.
RollingReplace instances one at a time. Old ones drain connections first.

SLA, SLO, SLI

These three terms come up constantly. Know them cold.

Term Stands For What It Is Example
SLI Service Level Indicator The metric you measure Request latency, error rate, throughput
SLO Service Level Objective The target you set for that metric P99 latency < 200ms, 99.9% availability
SLA Service Level Agreement The contract with consequences "If uptime < 99.9%, customer gets 10% credit"
SLI: "Our API returned 500 errors 0.05% of the time last month" ↓ SLO: "We target error rate below 0.1%" ↓ SLA: "If we exceed 0.1% error rate for a month, customers get service credits"
Google's rule of thumb: Set your SLO such that you're occasionally in danger of missing it. If you never come close to violating your SLO, it's set too loose and you're over-investing in reliability. The "error budget" (gap between SLO and current performance) tells you how much risk you can take with new releases.

Detailed breakdown:

  • SLA — contracts and consequences
  • SLO — targets and error budgets
  • SLI — metrics and measurement

Calculating System Availability

Interviewers sometimes ask you to estimate the availability of a proposed architecture.

Worked Example

System: Load Balancer → App Server → Database

Components:  LB: 99.99%,  App: 99.99%,  DB: 99.95%

Serial:  0.9999 × 0.9999 × 0.9995 = 0.9993 = 99.93%

If you add a redundant LB (parallel):
  LB pair:  1 - (1 - 0.9999)² = 99.999999%

Serial with redundant LB:
  0.99999999 × 0.9999 × 0.9995 = 0.999499 ≈ 99.95%

Impact of Replicas

Database primary:  99.95%  (downtime: ~4.4 hrs/year)
Add 1 replica:      Still 99.95%  (replica only helps reads)
Add failover:      ~99.99%  (primary fails → replica takes over in seconds)

Key point: Replicas alone don't improve write availability.
Failover mechanism + replicas does.

Common Interview Questions

  • "How many nines of availability should we target?"
    Ask follow-up: What's the cost of downtime? What's the revenue impact? Then recommend. Most systems: 99.9% to 99.99%.
  • "How would you design for 99.99% availability?"
    Walk through: redundancy, multi-AZ, automated failover, circuit breakers, monitoring, zero-downtime deploys.
  • "What's the difference between SLA and SLO?"
    SLA is the contract with consequences. SLO is the internal target. SLI is what you measure.
  • "How does adding more services affect availability?"
    Serial dependency: A_total = A₁ × A₂ × ... × Aₙ. More services = lower overall availability. This is the microservices trade-off.
  • "How do you handle a regional outage?"
    Multi-region deployment, active-active or active-passive, DNS failover, data replication with conflict resolution.
  • "What happens when your database goes down?"
    Read from replicas (stale reads acceptable?), circuit breaker on the app side, retry with backoff, failover to standby, graceful degradation.

Availability Killers

Be able to identify these in any design you propose.

KillerWhy It Kills AvailabilityMitigation
Single points of failureOne component = one failure takes everything downRedundancy everywhere
Cascading failuresOne slow service overloads everything upstreamCircuit breakers, timeouts, bulkheads
deploymentsBad release brings down productionCanary, blue-green, feature flags
Shared databasesOne team's query impacts everyoneDatabase per service, rate limiting
Hard dependenciesIf a non-critical service is down, everything failsGraceful degradation, fallbacks, cached responses
DNS failureNo one can reach your system at allMultiple DNS providers, low TTL, anycast
Certificate expiryTLS connections fail everywhereAutomated renewal (cert-manager, Let's Encrypt)
Resource exhaustionNo more connections, memory, diskMonitoring, autoscaling, rate limiting

What to Skip

  • ITIL availability management processes
  • Detailed uptime monitoring tools (Nagios, Zabbix configs)
  • Formal availability calculation methodologies (FTA, FMEA)
  • Hardware-specific redundancy (RAID, dual power supplies)

Focus on architectural patterns and interview reasoning, not ops tooling.
