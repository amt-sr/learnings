Durability

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What durability means and how it differs from reliability and availability
  • How databases guarantee durability through WAL and fsync
  • How replication and backups provide durability
  • RPO and RTO — how they connect to durability guarantees
  • How cloud systems achieve extreme durability (e.g., S3's 11 nines)

Definition

Durability is the property that guarantees data, once committed, will survive any subsequent failures.

Intuitive definition: Once the system tells you "write successful," that data is permanent. Even if the server crashes, the data center burns down, or the disk fails — the data is recoverable.
Analogy: Writing in pen (durable) vs pencil (not durable). A pencil mark can be erased by a sneeze. Pen ink survives until the paper itself is destroyed.

Why Durability Matters

Data loss is one of the most catastrophic failures a system can experience.

Real incidents:
  • GitLab (2017): Admin accidentally deleted production database. Lost 6 hours of data because backups were not properly maintained.
  • Stripe (2023): Corrupted database backup led to irrecoverable transaction history.
  • AWS us-east-1 (2011): EBS failure caused hours of data loss for Netflix and others.

How Databases Achieve Durability

Durability is not magic — it requires specific mechanisms at every stage of the write path.

Client sends write request ↓ Database receives and processes in memory ↓ Write to Write-Ahead Log (WAL) ← THIS is the durability step ↓ fsync() — force OS to flush WAL to disk ← Confirms durability ↓ Write acknowledged to client ↓ Later: Write data to main storage (lazy) ↓ WAL can be replayed on crash recovery

Write-Ahead Log (WAL)

The fundamental durability mechanism. The database writes a log entry before modifying the actual data pages.

Rule: Write the log FIRST, then write the data.

On crash recovery:
  1. Read the WAL from the last checkpoint
  2. Re-apply all committed transactions
  3. Undo any uncommitted transactions (using undo log)
  4. Database is now consistent
Why WAL works: Sequential appends to a log file are fast (no seek overhead). The log is small and always fits in the OS page cache. fsync forces it to disk. On crash, only un-fsynced data is lost — which is just the last few milliseconds.

See also: Write-Ahead Log, Redo Log, Undo Log

Replication for Durability

Having copies of data on other machines provides durability against machine-level failures.

SynchronousAsynchronous
DurabilityMaximum — data confirmed on replica before ackGood but not perfect — unreplicated data at risk
Write latencyHigher (waits for replica)Lower (local ack)
RPOZeroNon-zero (seconds to minutes of data at risk)
AvailabilityLower (replica failure blocks writes)Higher
Key insight: Synchronous replication trades write latency and availability for zero data loss. Asynchronous replication trades some data loss for lower latency. This is the fundamental durability trade-off.

Backup Strategies

TypeWhatSpeedStorage
Full backupComplete copy of all dataSlowHigh
IncrementalOnly data changed since last backupFastLow
DifferentialChanges since last FULL backupMediumMedium
Critical rule: A backup that has never been tested for restore is NOT a backup. It is a false sense of security. Regularly verify backups by restoring to a test environment.

RPO and RTO

MetricStands ForQuestionExample
RPORecovery Point Objective"How much data can we afford to lose?"5 minutes, 1 hour, 24 hours
RTORecovery Time Objective"How long can we be down?"30 seconds, 1 hour, 4 hours
RPO = 0 (zero data loss):
  → Synchronous replication + frequent WAL backups
  → Cost: Higher latency, lower write throughput

RPO = 5 minutes:
  → Asynchronous replication with 5-second lag + WAL
  → Acceptable for most systems

RPO = 24 hours:
  → Daily backups sufficient
  → Acceptable for analytics, logs, non-critical data

S3's 11 nines explained: With 11 nines of durability, if you store 10,000 objects, you would expect to lose one object every 10 million years. This is achieved through erasure coding: data is split into chunks with redundancy, stored across multiple facilities, and continuously verified.

Common Interview Questions

  • "How does a database guarantee durability on a write?"
    Write to WAL first, fsync to disk, then ack. If crash occurs before ack, the WAL is replayed to restore the committed transaction. Data pages are written lazily after the ack.
  • "What's the difference between sync and async replication?"
    Sync: primary waits for replica ack before confirming write (zero data loss, higher latency). Async: primary acks immediately, replica catches up later (potential data loss, lower latency).
  • "What is RPO and how do you achieve RPO=0?"
    RPO is how much data you can afford to lose. RPO=0 requires synchronous replication — every write is confirmed on at least one replica before acknowledging to the client.
  • "How does S3 achieve 11 nines of durability?"
    Erasure coding (data split into chunks with parity across multiple AZs), continuous integrity verification, automatic repair of degraded chunks.
  • "What's the durability of Redis?"
    By default, Redis is in-memory with no durability — data lost on restart. AOF (Append Only File) provides durability with some latency cost. RDB snapshots provide point-in-time recovery.
