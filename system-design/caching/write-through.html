7.2.3 Write Through

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆
Goal: Understand synchronous write-through caching — how it guarantees strong consistency between cache and database, the latency cost it imposes, and when it is the right choice over write-around or write-back strategies.

Definition

Write Through is a caching pattern where every write operation updates both the cache and the database synchronously before returning success to the caller. The application (or cache provider) writes to the cache first, then immediately writes to the database. The write is considered successful only after both operations complete. This ensures the cache and database are always consistent — any subsequent read will always return the latest written value.

How It Works

# Pseudocode — Write Through
function updateUser(userId, newData):
    cacheKey = "user:" + userId

    # Step 1: Write to cache synchronously
    cache.set(cacheKey, newData)

    # Step 2: Write to database synchronously
    db.execute("UPDATE users SET name = ? WHERE id = ?", newData.name, userId)

    # Step 3: Only return success after BOTH complete
    return success

# Any subsequent read gets the fresh data from cache
function getUser(userId):
    return cache.get("user:" + userId)   # Always consistent

Latency Characteristics

Write Through adds latency to every write operation because the client must wait for both the cache write and the database write to complete:

Write latency = max(cache_write_time, db_write_time)
               ≈ cache_write_time + db_write_time  (if sequential)

Example:
  Cache write:  ~1 ms (Redis SET)
  DB write:     ~5 ms (MySQL UPDATE with index)
  Total:        ~6 ms per write (vs ~5 ms without cache)

  For reads: latency drops from ~5 ms (DB) to ~1 ms (cache hit)
When is the latency cost worth it? Write Through trades write latency for read latency and strong consistency. This is a good tradeoff when your system is read-heavy (high read-to-write ratio), requires strong consistency, and the write volume is low enough that the added latency is acceptable.

Failure Handling

A critical question for Write Through: what happens when one of the two writes fails?

Cache write succeeds, DB write fails

  • The cache now contains data that does not exist (or is different) in the database.
  • Subsequent reads will serve stale/incorrect data from cache.
  • Mitigation: Roll back the cache write on DB failure (transactional approach), or use a short TTL as a safety net.

DB write succeeds, cache write fails

  • The database has the correct data, but the cache does not.
  • Subsequent reads will get a cache miss and fetch from DB (safe but slower).
  • This is the safer failure mode — the system degrades to a cache miss rather than serving wrong data.
Interview trap: If you propose Write Through, the interviewer may ask about partial write failures. Always discuss the failure modes and how you would handle them (rollback, retry, TTL fallback).

Real-World Usage

  • Database buffer pools: Many databases internally use a write-through policy for their buffer pools (dirty pages are flushed to disk on write, though some use write-back).
  • Configuration stores: Systems where configuration changes must be immediately visible to all readers (e.g., feature flags, rate limit configurations).
  • Session stores: When session data must be immediately consistent between cache and persistent storage.
  • Financial systems: Where read-after-write consistency is a hard requirement (e.g., account balance updates).

Common Interview Questions

  • Explain the Write Through caching pattern. What guarantees does it provide?
  • What is the latency cost of Write Through? When is this cost acceptable?
  • What happens if the cache write succeeds but the database write fails?
  • How does Write Through compare to Cache Aside with invalidation for writes?
  • When would you choose Write Through over Write Around for a system?
  • Can Write Through cause write amplification? How?

What to Skip

For a system design interview, skip:
  • Specific implementations of two-phase commit between cache and DB (the pattern itself is usually sufficient without diving into distributed transaction protocols).
  • Write Through in the context of CPU caches or hardware (unless specifically asked about hardware architecture).
  • Database-specific internals of how WAL and buffer pool flushing relate to write-through behavior.
