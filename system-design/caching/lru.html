LRU (Least Recently Used)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand the LRU eviction policy well enough to reason about when it works, when it fails, and how to implement it in O(1) time. Expect to discuss trade-offs against LFU and FIFO at the whiteboard and to sketch the data-structure internals during coding rounds.

Definition

LRU (Least Recently Used) is a cache eviction policy that removes the item that has not been accessed for the longest time. Every read or write to a cache entry promotes it to the "most recently used" position. When the cache is full, the entry at the tail of the recency list is evicted.

Key insight: LRU approximates temporal locality -- it assumes that data accessed recently is likely to be accessed again soon. This works well for workloads with strong recency patterns (e.g., social-media feeds, session data).

How LRU Works

LRU maintains a total ordering of all cache entries by last access time. On every cache hit, the accessed entry is moved to the head of the list. On eviction, the entry at the tail is removed.

LRU Doubly-Linked List + HashMap (capacity = 3) Access order (MRU ... LRU): [D] ←→ [A] ←→ [C] HashMap: { D→NodeD, A→NodeA, C→NodeC } Step 1 — get(E) → cache miss, insert E: [E] ←→ [D] ←→ [A] ←→ [C] (E at head) HashMap: { E→NodeE, D→NodeD, A→NodeA, C→NodeC } Step 2 — Cache full (capacity=3), evict C (LRU tail): [E] ←→ [D] ←→ [A] HashMap: { E→NodeE, D→NodeD, A→NodeA } Step 3 — get(A) → cache hit, promote A to head: [A] ←→ [E] ←→ [D] HashMap: { A→NodeA, E→NodeE, D→NodeD }

O(1) Implementation

Achieving O(1) for both get and put requires two cooperating data structures:

Data StructureRoleOperations
HashMap (key → Node)O(1) lookupget(key), put(key, value)
Doubly-Linked ListOrdered access trackingMove to head, remove tail
  • get(key): HashMap lookup → if hit, detach node from list, prepend to head. O(1).
  • put(key, value): HashMap lookup → if exists, update value + move to head. If new and at capacity, detach tail node, remove from hashmap, then prepend new node. O(1).
class LRUCache:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.map = {}          # key → node
        self.head = Node()      # dummy head
        self.tail = Node()      # dummy tail
        self.head.next = self.tail
        self.tail.prev = self.head

    def get(self, key: int) -> int:
        if key not in self.map: return -1
        node = self.map[key]
        self._remove(node)
        self._add_to_head(node)
        return node.val

    def put(self, key: int, value: int) -> None:
        if key in self.map:
            node = self.map[key]
            node.val = value
            self._remove(node)
            self._add_to_head(node)
        else:
            if len(self.map) == self.cap:
                lru = self.tail.prev
                self._remove(lru)
                del self.map[lru.key]
            node = Node(key, value)
            self.map[key] = node
            self._add_to_head(node)

The One-Time Scan Problem

LRU weakness -- one-time scan: A sequential scan of a large dataset that exceeds cache capacity will evict every previously cached item, replacing it with data that is only accessed once. After the scan completes, the cache is cold and subsequent requests experience 100% misses.
Scenario: cache capacity = 3, keys A-D are hot Before scan: [C] ←→ [A] ←→ [B] (all hot keys cached) Sequential scan of X, Y, Z (each accessed once): [Z] ←→ [Y] ←→ [X] (A, B, C all evicted) After scan -- A requested → cache miss! [A] ←→ [Z] ←→ [Y] (rebuilding from scratch)
Mitigation: Use LRU-K (track last K accesses) or 2Q (two-queue: a small FIFO admission queue before the main LRU). These prevent one-time scans from polluting the main cache. Redis uses a variant of LRU with a sampling-based eviction to reduce this effect.

Common Interview Questions

  1. Design an LRU cache with O(1) get and put. (LeetCode 146 -- expect this as a coding exercise.)
  2. How would you implement LRU in a distributed cache with millions of entries?
  3. What happens when a sequential scan hits an LRU cache? How do you prevent cache pollution?
  4. Compare LRU vs LFU for a social-media feed vs a video recommendation system.
  5. How does Redis implement approximate LRU? Why not exact LRU?
  6. What is the time complexity of evicting N entries in a single LRU operation?
  7. How would you make LRU thread-safe in a high-concurrency environment?

What to Skip

Skip in interviews:
  • Detailed locking strategies for the doubly-linked list (mention thread-safety at a high level, don't descend into read-write lock internals unless asked).
  • Off-topic data structures like skip lists or splay trees -- stick to HashMap + DLL.
  • Optimizations like batching moves (lazy promotion) unless the interviewer signals deeper interest.
  • Full Redis source-code internals of the LRU eviction sampling algorithm -- state the concept, don't re-implement.
