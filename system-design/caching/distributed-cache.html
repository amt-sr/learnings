Distributed Cache

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand distributed caching as a concept -- multiple cache nodes acting as a single logical cache layer. Know the key architectural decisions: topology (client-side sharding vs consistent hashing), cache warming strategies, invalidation challenges at scale, and trade-offs against local in-process caching. This topic bridges individual cache fundamentals into the system-level caching architecture that appears in design interviews.

Definition

A distributed cache is a cache layer spread across multiple nodes (machines/VMs/containers) that presents a single logical cache to client applications. Data is partitioned across nodes using a hashing or sharding scheme. Distributed caches provide aggregate memory capacity that exceeds any single machine, enabling horizontal scaling for high-throughput, low-latency access patterns.

Key insight: A distributed cache is fundamentally different from a local cache (in-process cache like Guava/Caffeine in Java). Local caches are fast (nanosecond latency, no network) but per-process with limited capacity. Distributed caches add network latency (microseconds to low milliseconds) but share state across all application instances, enabling cache coherence and much larger capacity.

Cache Topologies

Client-Side Sharding

The application or client library knows about all cache nodes and hashes keys locally to determine the target node. No intermediate proxy is required.

  • Memcached model: Client library uses ketama consistent hashing to map keys to nodes. Add/remove a node causes minimal remapping (only 1/N keys move on average).
  • Redis Cluster model: Client library maintains a slot-to-node map and follows MOVED/ASK redirects.

Proxy-Based

A proxy layer sits between clients and cache nodes. Clients send all requests to the proxy; the proxy handles routing.

  • Twemproxy (nutcracker): Stateless proxy, supports Memcached and Redis. Single point of failure unless fronted by a load balancer. No auto-discovery.
  • Codis: Stateful proxy with Zookeeper/Etcd for coordination. Supports resharding. Largely superseded by Redis Cluster.
  • Predixy: Multi-threaded proxy for Redis with Lua scripting support.

Consistent Hashing

The standard algorithm for mapping keys to nodes in a distributed cache. Keys and nodes are both placed on a hash ring. Each key is assigned to the nearest node clockwise on the ring. Virtual nodes (vnodes) improve balance.

Consistent Hash Ring (with virtual nodes) Node A.1 ╱ Node B.2 ──●─── Node A.2 ╱ ╲ Node C.3 Node A.3 │ │ Node C.1 ──●─── Node B.1 ──●─── Node C.2 │ │ key1(key1 maps to key2(key2 maps to Node B.1) Node C.2) Adding Node D: Only keys in the range [D, next_node) remap ~1/N keys move (not all keys)

Cache Warming

A cold distributed cache can cause a thundering herd to the database. Warming strategies:

  • Lazy loading (cache-aside): Populate on first miss. Simple but causes cold-start storms.
  • Write-through: Write to cache and DB simultaneously. Cache is always warm for write-heavy data, but increases write latency.
  • Pre-warming (bootstrap): On deploy or restart, load hot keys from DB into cache before accepting traffic. Requires knowing which keys are hot.
  • Background refresh: Async workers refresh keys with approaching TTLs before they expire, preventing cache misses entirely for hot data.
Cache stampede: When a hot key expires, many concurrent requests simultaneously miss and all fetch from the database. Mitigations: request coalescing (only one request fetches, others wait), probabilistic early expiration (scatter TTL jitter so not all replicas expire at once), or mutex locks per key.

Cache Invalidation in Distributed Settings

Invalidation is the hardest part of distributed caching. The core challenge: when data changes in the source of truth, how do you ensure all cache nodes are updated or invalidated?

Golden rule: "There are only two hard things in Computer Science: cache invalidation and naming things." -- Phil Karlton. In a distributed cache, invalidation requires a coordination mechanism (pub/sub, message queue, or versioning). There is no free lunch -- you trade off between consistency, latency, and complexity.

ExamplesGuava, Caffeine, Ehcache (local)Redis, Memcached, Hazelcast
Best practice -- use both: Most production systems use a two-tier cache: L1 is a small local cache (Caffeine/Guava) for ultra-hot keys with nanosecond latency and short TTL; L2 is a distributed cache (Redis/Memcached) for broader capacity and cross-instance coherence. The local cache absorbs the majority of reads for hot keys, reducing load on the distributed cache.

Common Interview Questions

  1. How would you design a distributed caching layer for a social-media feed system?
  2. Explain consistent hashing. What happens when a node is added or removed?
  3. Compare client-side sharding vs proxy-based caching. When would you choose each?
  4. How do you handle cache invalidation across multiple cache nodes?
  5. What is cache stampede and how do you prevent it?
  6. Compare local cache vs distributed cache. When would you use both in a two-tier setup?
  7. How would you warm a distributed cache after a cold start or deployment?
  8. What happens to your distributed cache during a network partition? How do you maintain availability?
  9. How do you handle hot keys that overwhelm a single shard?
  10. Design a cache invalidation strategy for a system where data can be updated by multiple services.

What to Skip

Skip in interviews:
  • Hazecast/Ignite multi-tiered caching internals unless the design specifically calls for an embedded distributed data grid.
  • Memcached slab allocator rebalancing algorithms -- state the concept (slabs with per-slab LRU) and move on.
  • Consistent hashing jump-consistent-hash variant unless the interviewer specifically asks about alternatives to the ring-based approach.
  • Proxy implementations (Envoy filters, mcrouter) unless designing a custom proxy layer is part of the question.
