7.1 Why Caching

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★
Goal: Understand why caching is a foundational technique in distributed systems, the performance gains it delivers, where it sits in the request lifecycle, and when caching is the wrong approach.

Definition

Caching is the technique of storing a copy of frequently accessed data in a faster, closer storage layer so that subsequent requests can be served without hitting the origin data source. The cached data is typically a subset of the full dataset, chosen based on access patterns, size constraints, and consistency requirements.

Why Caching Matters

1. Latency Reduction

The primary motivation for caching is to serve data from a storage layer that is orders of magnitude faster than the origin. In a typical web stack, each layer adds latency:

Storage LayerTypical LatencyRelative Speed
L1 CPU Cache~1 nsFastest
L2 CPU Cache~4 nsVery fast
Main Memory (RAM)~100 nsFast
SSD~100 μsModerate
Network (same DC)~0.5 msSlow
Disk (HDD)~10 msSlow
DB Query (complex)~50–500 msVery slow

Caching in memory (Redis, Memcached) reduces latency from milliseconds to sub-milliseconds — a 100x to 1000x improvement for read-heavy workloads.

2. Database Load Reduction

Every cache hit is a request that never reaches the database. For systems where the read-to-write ratio is high (common ratios are 100:1 or 1000:1), caching can eliminate 90–99% of database reads. This directly translates to:

  • Lower database CPU and I/O utilization
  • Ability to defer or avoid database vertical/horizontal scaling
  • Reduced connection pool contention
  • Longer hardware lifespan for expensive database infrastructure

3. Throughput Increase

By serving reads from cache, the system's overall throughput (requests per second) increases significantly. A single Redis instance can handle 100K+ ops/sec on a single thread, while a typical relational database query might consume 5–50 ms of connection time. The math is straightforward:

Without cache:  DB handles 100K reads/sec → requires 100 DB connections (1ms per query, idealistic)
With 95% hit ratio: DB handles 5K reads/sec  → requires 5 DB connections
Throughput gain: ~20x with same DB infrastructure

Caching Hierarchy

Caching is not a single layer — modern systems employ multiple caching tiers, each at a different point in the request path:

Client Request │ ▼ ┌──────────┐ ┌──────────────────────────────────────────────┐ │ Browser │────│ Browser Cache (HTTP cache headers, │ │ │ │ localStorage, service workers) │ └──────────┘ └──────────────────────────────────────────────┘ │ ▼ ┌──────────┐ ┌──────────────────────────────────────────────┐ │ CDN │────│ Edge Cache (CloudFront, Cloudflare, │ │ │ │ Fastly) — static assets, pre-warmed pages │ └──────────┘ └──────────────────────────────────────────────┘ │ ▼ ┌──────────┐ ┌──────────────────────────────────────────────┐ │ API │────│ API / Application Cache (Redis/Memcached, │ │ Gateway │ │ response caching, rate limit counters) │ └──────────┘ └──────────────────────────────────────────────┘ │ ▼ ┌──────────┐ ┌──────────────────────────────────────────────┐ │ App │────│ Application-level cache (in-process caches, │ │ Server │ │ computed aggregations, session data) │ └──────────┘ └──────────────────────────────────────────────┘ │ ▼ ┌──────────┐ ┌──────────────────────────────────────────────┐ │ Database │────│ DB Buffer Pool / Query Cache │ │ │ │ (InnoDB buffer pool, MySQL query cache*) │ └──────────┘ └──────────────────────────────────────────────┘ * MySQL query cache was removed in 8.0; PostgreSQL has no query cache.
Key insight for interviews: Each layer has different characteristics — TTL policies, invalidation mechanisms, storage capacity, and consistency guarantees. A strong answer discusses the tradeoffs at each tier rather than treating "caching" as a single monolithic concept.

Cache Hit Ratio

The cache hit ratio is the fundamental metric for evaluating cache effectiveness:

Hit Ratio = Cache Hits / (Cache Hits + Cache Misses)

Example:
  10,000 requests served
  9,200 served from cache (hits)
  800  served from origin (misses)
  Hit Ratio = 9200 / 10000 = 92%

What constitutes a "good" hit ratio?

Hit RatioAssessmentImplication
> 95%ExcellentCache is well-tuned, data fits working set
85–95%GoodTypical for well-designed systems
70–85%AcceptableMay indicate cold cache or suboptimal key design
< 70%PoorCache may be too small, TTL too short, or wrong data cached
Interview tip: Hit ratio alone is misleading. A 99% hit ratio on 10 requests/day is less valuable than a 90% hit ratio on 10M requests/day. Always discuss hit ratio in the context of traffic volume and the cost of a miss (i.e., how expensive is the origin query?).

When NOT to Cache

Caching introduces complexity (consistency, invalidation, memory overhead). It is not always the right choice:

  • Write-heavy workloads: If reads are rare relative to writes, caching adds overhead without proportional benefit. The invalidation cost may exceed the read savings.
  • Frequently mutated data: Data that changes constantly (e.g., real-time stock prices, live counters) becomes stale immediately, making cache hit ratios low and consistency burdens high.
  • Large result sets: If individual responses are too large to fit in memory efficiently, caching them may evict many smaller, more useful entries. Consider caching computed aggregations instead.
  • Low-traffic endpoints: The operational complexity of maintaining cache invalidation logic may not be justified for endpoints with minimal traffic.
  • Strong consistency requirements: If your system cannot tolerate any stale reads (e.g., financial transaction balances), caching introduces a consistency window that may violate business invariants.
  • Data with unpredictable access patterns: Zipfian or uniform random access makes cache eviction aggressive and hit ratios poor. Caching works best with skewed (power-law) access distributions.
Common mistake: Adding a cache to every endpoint by default. Caching should be a deliberate decision driven by measured performance characteristics, not a reflexive architectural choice. Always profile before caching.

Common Interview Questions

  • Why would you add a cache to your system? What problems does it solve?
  • What is a cache hit ratio, and what is a good target?
  • Explain the caching hierarchy in a typical web application.
  • When would you choose NOT to use a cache?
  • How do you decide what data to cache and what to leave behind?
  • What are the tradeoffs between caching at the CDN layer vs. the application layer?
  • How does caching affect your database capacity planning?

What to Skip

For a system design interview, skip:
  • Deep dives into specific vendor implementations (e.g., Redis internals like SDS, ziplist encoding) unless the question specifically targets Redis.
  • Browser caching mechanics beyond mentioning it as a layer (no need for ETag/If-None-Match header details).
  • Cache hardware architecture (L1/L2/L3 CPU cache details) unless discussing low-latency trading systems.
  • The mathematical derivation of eviction policies (e.g., Belady's optimal algorithm proof).
