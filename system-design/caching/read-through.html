7.2.2 Read Through

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆
Goal: Understand the Read Through caching pattern, how it differs from Cache Aside, and when to choose it. Recognize that it simplifies application code at the cost of reduced flexibility.

Definition

Read Through is a caching pattern where the cache provider (or a cache library) sits between the application and the database. The application issues a read request only to the cache. If the data is present (cache hit), it is returned immediately. If the data is absent (cache miss), the cache itself is responsible for fetching it from the database, caching the result, and then returning it to the application. The application never interacts with the database for reads.

Key distinction from Cache Aside: In Cache Aside, the application manages cache population explicitly. In Read Through, the cache library/provider handles the database fallback transparently. The application is unaware of whether the data came from cache or database.

# Pseudocode — Read Through
# The application code is simpler — it only talks to the cache
function getUser(userId):
    # The cache library handles the miss logic internally
    return cache.getOrLoad("user:" + userId, () -> {
        return db.query("SELECT * FROM users WHERE id = ?", userId)
    })

# Application never sees the database.
# Cache library handles: check cache → miss → query DB → store in cache → return

When to Use Read Through

  • Simpler application code is a priority: When you want developers to focus on business logic rather than cache management. The cache becomes a transparent read layer.
  • Consistent cache key patterns: When all cached data follows the same access and invalidation patterns, a one-size-fits-all cache provider makes sense.
  • Using a cache library that supports it: Caffeine (Java), Guava Cache, Django's cache framework, and some ORM layers (e.g., Hibernate second-level cache) implement Read Through semantics.
  • Read-heavy workloads with uniform patterns: When you have a well-defined set of read queries that all benefit from the same caching strategy.

When Cache Aside Is Preferable

  • You need fine-grained control over which data gets cached and which does not.
  • Different keys require different TTLs, eviction policies, or population strategies.
  • You are using a simple key-value cache (Redis, Memcached) that does not provide Read Through abstractions.
  • You need to cache results from multiple heterogeneous data sources.
  • You want the application to handle cache misses differently based on context (e.g., fallback to a different source).

Combining Read Through with Write Strategies

Read Through only handles the read path. For writes, you must pair it with one of the write strategies:

CombinationBehaviorConsistency
Read Through + Write ThroughBoth reads and writes go through cacheStrong
Read Through + Write AroundReads via cache; writes go to DB, cache invalidatedEventual
Read Through + Write BackReads via cache; writes buffered in cacheWeak (data loss risk)
Interview insight: In practice, most production systems at FAANG-scale use Cache Aside rather than Read Through. Cache Aside gives teams the flexibility to handle heterogeneous data sources, custom TTLs per key, and complex invalidation logic that a Read Through abstraction would obscure. However, Read Through is a clean pattern for simpler applications or when using an ORM with built-in cache support.

Common Interview Questions

  • What is the Read Through caching pattern? How does it differ from Cache Aside?
  • What are the advantages of Read Through over Cache Aside?
  • How does the cache layer know how to fetch data from the database on a miss?
  • What write strategy would you pair with Read Through?
  • When would you choose Cache Aside over Read Through for a production system?

What to Skip

For a system design interview, skip:
  • Specific library configuration for Read Through (e.g., Caffeine's LoadingCache builder API in Java).
  • ORM-specific caching details (e.g., Hibernate second-level cache internals) unless the question is about designing a data access layer.
  • Deep comparisons of every cache library's Read Through implementation.
