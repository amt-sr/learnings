7.3.2 Cache Coherence

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆
Goal: Understand cache coherence — the problem of keeping multiple copies of the same data consistent across different caches. Distinguish it from cache invalidation, understand the origins in CPU architecture, and map the concept to distributed systems.

Definition

Cache coherence is the consistency model that ensures multiple caches (typically CPU caches in a multi-core system) maintain a consistent view of a single memory location. When one core writes to a memory address, all other cores that have cached that address must be notified so they can update or invalidate their local copy. Cache coherence is a hardware-level protocol that operates transparently below the operating system and application layers.

Cache Invalidation vs. Cache Coherence: These terms are related but distinct.
  • Cache Invalidation (7.3.1): A general technique for removing stale data from cache. It is a software concept that application developers manage.
  • Cache Coherence (this section): A hardware-level protocol that ensures consistency across multiple cache instances automatically. It is a systems architecture concept managed by the CPU/hardware.
In system design interviews, cache invalidation is the more practical and frequently tested concept. Cache coherence comes up when discussing distributed cache clusters or multi-core consistency.

Origin: CPU Multi-Core Caches

Cache coherence was originally designed for multi-core processors where each core has its own L1/L2 cache but shares the same main memory:

Core 0 Core 1 Core N ┌──────────────┐ ┌──────────────┐ ┌──────────────┐ │ L1 Cache │ │ L1 Cache │ │ L1 Cache │ │ L2 Cache │ │ L2 Cache │ │ L2 Cache │ └──────┬───────┘ └──────┬───────┘ └──────┬───────┘ │ │ │ └─────────────────────────┼─────────────────────────┘ │ ┌──────▼───────┐ │ Cache │ │ Coherence │ │ Protocol │ │ (MESI, MOESI)│ └──────┬───────┘ │ ┌──────▼───────┐ │ Main Memory │ │ (RAM) │ └──────────────┘

Without coherence protocols, Core 0 could modify address X in its cache while Core 1 continues reading the stale value from its own cache. The coherence protocol prevents this by ensuring all cores see the same value for any given memory location.

Coherence Protocols

MESI Protocol

The most common cache coherence protocol, named after its four states:

StateMeaningCan Read?Can Write?
ModifiedCache line modified — only this cache has it, memory is staleYesYes (local only)
ExclusiveCache line unmodified — only this cache has it, matches memoryYesYes (becomes Modified)
SharedCache line unmodified — multiple caches may have it, matches memoryYesNo (must request Exclusive)
InvalidCache line is invalid — data not available in this cacheNoNo
State Transitions (simplified): Invalid ──(read)──> Shared (fetch from memory, other caches may have it) Invalid ──(write)─> Modified (fetch exclusive, modify) Shared ──(write)─> Modified (broadcast invalidate to other caches) Modified──(read)──> Shared (write-back to memory, share) Modified──(evict)──> Invalid (write-back to memory, discard) Shared ──(evict)──> Invalid (simply discard) Exclusive─(write)──> Modified Exclusive─(evict)──> Invalid

MOESI Protocol

An extension of MESI that adds the Owner state, allowing a modified cache line to be shared (as dirty data) with other caches without writing back to memory first. Used by AMD processors.

Cache Coherence in Distributed Systems

While cache coherence is a hardware concept, the same problem arises in distributed systems with multiple cache nodes:

App Instance A App Instance B App Instance C │ │ │ ┌────▼────┐ ┌────▼────┐ ┌────▼────┐ │ Local │ │ Local │ │ Local │ │ Cache │ │ Cache │ │ Cache │ └────┬────┘ └────┬────┘ └────┬────┘ │ │ │ └────────────────────────┼─────────────────────────┘ │ ┌───────────▼───────────┐ │ Distributed Cache │ │ Cluster (Redis, │ │ Memcached) │ └───────────────────────┘ Problem: If App A writes user:123 to its local cache, App B's local cache may serve stale data.

Solutions for Distributed Cache Coherence

ApproachHow It WorksTradeoffs
Central cache (no local)All app instances read from a shared Redis/Memcached clusterNo coherence problem, but adds network hop on every cache access
Invalidate via pub/subOn write, publish invalidation; all local caches subscribe and evictLow staleness, but adds pub/sub infrastructure
Versioned keysInclude data version in cache key; stale versions naturally missSimple but requires version lookup on each read
Short TTLLocal caches have very short TTL; data refreshed from central cacheSimplicity at the cost of higher staleness
Multicast invalidationUse multicast/UDP to broadcast invalidations to all nodesFast but unreliable (at-most-once delivery)

Common Interview Questions

  • What is cache coherence? How does it differ from cache invalidation?
  • Explain the MESI protocol. What does each state represent?
  • How does cache coherence work in a multi-core CPU?
  • In a distributed system with local application caches, how do you keep them coherent?
  • When would you use a central cache vs. local caches with coherence?

What to Skip

For a system design interview, skip:
  • Detailed MESI state transition diagrams with every bus transaction type (BusRd, BusRdX, BusUpgr, etc.).
  • Cache coherence in NUMA architectures and inter-node coherence protocols.
  • MOESI vs. MESI performance comparison at the hardware level.
  • Snooping vs. directory-based coherence protocol internals.
