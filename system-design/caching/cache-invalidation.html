7.3.1 Cache Invalidation

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★
Goal: Master cache invalidation — the hardest problem in caching. Understand why invalidation is fundamentally difficult, the major strategies (time-based, event-based, version-based), the stale cache problem, and how to design invalidation for real distributed systems.

Definition

Cache invalidation is the process of removing or updating cached data when the underlying source data has changed, ensuring that stale (outdated) data is not served to users. The challenge is that caches are, by definition, copies of data that can become outdated the moment the source data is modified.

"There are only two hard things in Computer Science: cache invalidation and naming things."
— Phil Karlton (attributed)
Why it is hard: In a distributed system, there is always a window between when data changes in the source and when all cache copies are updated. The fundamental question is: how do you ensure that stale data is not served during this window, and how long can that window be before it violates business requirements?

The Stale Cache Problem

Stale data occurs when a cache entry no longer reflects the current state of the source data. This can happen due to:

  • Delayed invalidation: The invalidation signal has not yet reached all cache instances.
  • Race conditions: A read repopulates the cache with old data after an invalidation but before the DB reflects the update.
  • Failed invalidation: The invalidation message is lost (network partition, cache node down).
  • Missing invalidation: A code path modifies data but forgets to invalidate the corresponding cache entry.
Time ──────────────────────────────────────────────────────> Source DB: [value: A]──────────[write: B]──────────[value: B] Cache: [value: A]──────────[stale!]────[invalidate]──[value: B] ▲ Stale window: cache serves A but DB has B

Invalidation Strategies

1. Time-Based Invalidation (TTL)

The simplest strategy: set a time-to-live (TTL) on each cache entry. After the TTL expires, the entry is automatically removed or treated as stale.

cache.set("user:123", data, TTL=3600)   # Expires in 1 hour
ProsCons
Zero coordination overheadAlways serves stale data until TTL expires
Simple to implementCannot react to data changes in real time
Failsafe — guarantees eventual freshnessHard to choose optimal TTL
Best for: Data that changes infrequently, where brief staleness is acceptable, or as a safety net alongside active invalidation. See 7.3.3 TTL for deep coverage.

2. Event-Based Invalidation (Active Invalidation)

When the source data is modified, an explicit invalidation event is triggered to remove or update the corresponding cache entries. This can be implemented via:

a) Application-Level Invalidation

# After writing to the database
db.execute("UPDATE users SET name = ? WHERE id = ?", "Alice", 123)
cache.delete("user:123")        # Explicit invalidation
cache.delete("user:123:profile") # Invalidate related keys
cache.delete("users:popular")    # Invalidate computed aggregations

b) Message Queue-Based Invalidation

Write Service Message Queue Cache Instances │ │ │ │ 1. Update DB │ │ │ ───────────────────────────────────────> │ │ │ │ │ 2. Publish event: │ │ │ {type: INVALIDATE│ │ │ key: "user:123"}│ │ │ ──────────────────>│ │ │ │ │ │ │ 3. Deliver to all │ │ │ cache instances │ │ │ ─────────────────────>│ │ │ │ │ │ │ 4. Delete key │ │ │ from all │ │ │ instances

c) Database Trigger / CDC (Change Data Capture)

Use database triggers or CDC tools (Debezium, Kafka Connect) to automatically publish change events when data is modified. The cache layer subscribes to these events.

ProsCons
Low staleness — near real-timeMore complex infrastructure
Granular — invalidate only affected keysMust track all cache keys that depend on a data change
Decoupled — writer does not need to know about cacheEvent delivery not guaranteed (at-least-once requires idempotent invalidation)
Best for: Systems that require low staleness, where data changes are infrequent enough that the invalidation infrastructure cost is justified, and where the cache-key-to-data mapping is well-defined.

3. Version-Based Invalidation

Attach a version number to each piece of data. The cache stores the data along with its version. On a read, compare the cached version with the current version from the source. If they differ, re-fetch and update the cache.

# Version stored in DB
# users table: id, name, updated_at (or version column)

cache_key = "user:123:v" + db.getVersion(123)
cache.set(cache_key, data, TTL=3600)

# On read: if version changed, key won't exist in cache
data = cache.get("user:123:v" + db.getVersion(123))
if data is null:
    data = db.query("SELECT * FROM users WHERE id = 123")
    cache.set("user:123:v" + db.getVersion(123), data)
ProsCons
No explicit invalidation neededRequires a version lookup on every read (reduces cache hit ratio)
Automatic staleness detectionVersion lookup can become a bottleneck
Works across distributed cachesOld versions accumulate in cache until TTL expires
Best for: Scenarios where you need cache consistency but cannot rely on explicit invalidation (e.g., data is modified by external systems that cannot publish invalidation events).

Choosing the Right Strategy

ScenarioRecommended Strategy
Public CDN-cached contentTTL (long) + purge API for manual invalidation
User profile dataEvent-based invalidation + short TTL as safety net
Inventory / stock levelsVery short TTL or write-through (strong consistency)
Computed aggregations (leaderboards)Event-based: invalidate on any constituent data change
Data modified by external systemsVersion-based + short TTL
Configuration / feature flagsEvent-based (push invalidation) for immediate visibility

Challenges at Scale

Identifying All Dependent Cache Keys

A single data change can affect multiple cache keys. For example, updating a user's name invalidates:

  • user:123 — the user object itself
  • user:123:profile — a denormalized profile view
  • users:search:alice — search results containing the user
  • leaderboard:global — if the user appears in a leaderboard
  • feed:user:456 — if this user's posts appear in another user's feed

Tracking all dependent keys is extremely difficult at scale. Solutions include:

  • Tag-based invalidation: Assign tags to cache entries; invalidate by tag (e.g., invalidate all entries tagged user:123).
  • Key pattern invalidation: Use SCAN + DEL for wildcard patterns (Redis KEYS/SCAN).
  • Accept higher TTL: Let staleness expire naturally rather than tracking all dependencies.

Distributed Cache Invalidation

In a cluster of cache nodes, an invalidation must reach all nodes holding the relevant keys:

  • Pub/Sub: Publish invalidation events; all cache nodes subscribe and delete locally.
  • Consistent hashing: Key-based routing means invalidation goes to a specific node (but replicas need to be handled).
  • Client-side invalidation: Each application instance invalidates from its local cache.

Common Interview Questions

  • Why is cache invalidation considered one of the hardest problems in CS?
  • Explain three different cache invalidation strategies and when to use each.
  • How would you handle cache invalidation when a single data change affects multiple cache keys?
  • What is the stale cache problem? How do you minimize the staleness window?
  • How would you implement event-based cache invalidation using a message queue?
  • What happens when an invalidation message is lost in a distributed cache?
  • How do you design cache invalidation for a social media feed?

What to Skip

For a system design interview, skip:
  • CDN-specific purge APIs and invalidation headers (e.g., Cloudflare purge API details).
  • Low-level invalidation in CPU caches (MESI protocol details — that is cache coherence, covered in 7.3.2).
  • Implementation specifics of CDC tools (Debezium connector configuration).
  • Database trigger syntax for any specific RDBMS.
