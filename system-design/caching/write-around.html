7.2.4 Write Around

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆
Goal: Understand the Write Around pattern, why it is preferable for write-heavy workloads, and the read-miss penalty it introduces. Know when to choose it over Write Through or Write Back.

Definition

Write Around is a caching pattern where write operations bypass the cache entirely and go directly to the database. If the written key exists in the cache, it may optionally be invalidated; otherwise, the cache is simply not updated. The rationale is to avoid polluting the cache with data that may never be read again — a common scenario in write-heavy workloads where many writes are for data with low read recurrence.

    # Write goes directly to DB — cache is NOT updated

# The next read will repopulate the cache via Cache Aside or Read Through
function getUser(userId):
    data = cache.get("user:" + userId)
    if data is null:
        data = db.query("SELECT * FROM users WHERE id = ?", userId)
        cache.set("user:" + userId, data, TTL=3600)
    return data

Why Write Around Exists

Consider a workload where you write 1,000 records per second but only read 10 per second. With Write Through, every write would also update the cache, meaning you are filling the cache with 990 entries per second that will never be read. This cache pollution:

  • Evicts genuinely hot data from the cache (useful entries get pushed out by write-only entries).
  • Wastes cache memory on data with zero read value.
  • Adds unnecessary write latency (cache write on top of DB write).

Write Around solves this by keeping the cache focused on data that is actually read.

The Read-Miss Penalty

The downside of Write Around is the first read after a write always results in a cache miss (cold read). The tradeoff is explicit:

Write Through: Write is slower, but the first read after a write is always a cache hit.
Write Around: Write is fast (DB only), but the first read after a write is a cache miss.

Whether this penalty matters depends on the read-after-write pattern:

Access PatternImpactRecommendation
Write then immediately read same keyHigh — always a missWrite Through is better
Write then read much later (seconds/minutes)Low — miss is acceptableWrite Around is fine
Write-heavy, rarely read written dataNegligible — avoids pollutionWrite Around is ideal
User profile updatesMedium — user may re-read profileDepends on read frequency

Real-World Usage

  • Log/aggregation systems: Writing time-series or log data that is rarely re-read after initial write.
  • Social media posts: Writing a new post/tweet — the write is frequent but re-reads of any specific post are less common than reads of the feed.
  • Audit trails: Appending audit records that are written once and read only during investigations.
  • IoT sensor data: High-frequency writes of sensor readings where only recent data is cached (time-windowed).

Common Interview Questions

  • What is the Write Around pattern? Why would you choose it over Write Through?
  • What is "cache pollution," and how does Write Around avoid it?
  • What is the read-miss penalty with Write Around? When does it matter?
  • Describe a workload where Write Around is clearly better than Write Through.
  • How does Write Around differ from Cache Aside with invalidation?

What to Skip

For a system design interview, skip:
  • Implementation details of cache invalidation in Write Around — the concept is straightforward.
  • Hardware-level write-around policies in CPU caches (this is a systems architecture topic, not distributed systems).
  • Perf benchmarks comparing Write Around vs. Write Through in specific database engines.
