Redis Cluster

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand how Redis Cluster enables horizontal scaling of Redis across multiple nodes. Know the hash slot mechanism, master-replica topology, client-side routing, failover behavior, and the critical limitations around cross-slot operations. Be ready to discuss trade-offs against client-side sharding and proxy-based approaches.

Definition

Redis Cluster is Redis's native solution for horizontal scaling and high availability. It distributes data across multiple Redis nodes using a fixed set of 16384 hash slots. Each master node owns a subset of slots, and replicas provide failover. Clients receive redirections when accessing a key on the wrong node.

Key insight: Redis Cluster provides sharding and replication in a single deployment model. Unlike client-side sharding (which requires application logic to manage partitions) or proxy-based solutions (Twemproxy, Codis), Redis Cluster embeds routing metadata directly in the protocol -- clients learn the slot-to-node mapping and self-correct after redirects.

Hash Slots

Redis Cluster uses a fixed hash space of 16384 hash slots. Each key is mapped to a slot via: CRC16(key) % 16384. Each master node is assigned a contiguous range of slots.

Hash Tags

Keys enclosed in {} use only the content inside the braces for hashing. This allows related keys to be placed on the same slot, enabling multi-key operations.

# Both keys hash to the SAME slot (only "user1001" is hashed)
SET {user1001}:profile '{"name":"Alice"}'
SET {user1001}:sessions "abc123"

# MGET works because both keys are on the same slot
MGET {user1001}:profile {user1001}:sessions

Master-Replica Topology

Each master has one or more replicas. If a master fails, its replicas participate in an election (Raft-like consensus using gossip protocol). The replica with the most recent replication offset becomes the new master.

Failover Scenario: Before failure: Master B (healthy) ←── Replica B' (in sync) Clients route to Master B Master B crashes: 1. Replica B' detects master is down (gossip + PING timeout) 2. Replica B' requests election from majority of masters 3. Other masters vote → Replica B' promoted to Master 4. Clients receive MOVED redirects to new Master B' (former replica) After failover: Master B' (formerly Replica B') ←── (need new replica) Clients route to Master B' at new address

Client-Side Routing

Redis Cluster clients maintain a local mapping of hash slots to nodes. The routing flow:

  1. Client computes the slot for a key: CRC16(key) % 16384
  2. Client looks up the slot in its local map → sends command to the owning node
  3. If the mapping is stale, the node responds with MOVED <slot> <ip:port>
  4. Client updates its local map and retries the command to the correct node
Smart vs Dumb clients: A "smart" client (Jedis, redis-py-cluster) maintains the full slot map locally and proactively routes. A "dumb" client sends every command to a random node and follows MOVED redirects. Smart clients reduce round trips but must handle map refresh on resharding.

Cross-Slot Limitations

Critical limitation: Multi-key operations (MGET, MSET, pipeline, Lua scripts, transactions) only work when all keys map to the same hash slot. If keys are on different slots, Redis returns a CROSSSLOT error. This is a fundamental constraint of the hash-slot architecture.
  • MGET/MSET across slots: Must use hash tags to co-locate keys, or issue separate commands per slot.
  • Lua scripts: All keys accessed by a Lua script must be on the same slot (or use hash tags).
  • Transactions (MULTI/EXEC): Same constraint -- all keys must be on one slot.
  • Key enumeration: KEYS * and SCAN only iterate over one node's keys, not the entire cluster.

Resharding

Moving hash slots between nodes for capacity rebalancing. Redis Cluster supports live resharding -- data migrates incrementally without downtime:

  1. CLUSTER SETSLOT <slot> IMPORTING <node-id> -- destination node marks slot as importing
  2. CLUSTER SETSLOT <slot> MIGRATING <node-id> -- source node marks slot as migrating
  3. Source node returns ASK <slot> <ip:port> for keys still on source but assigned to destination
  4. Client follows ASK redirect (one-time, unlike MOVED which updates the permanent map)
  5. All keys migrated → slot ownership transferred permanently

Common Interview Questions

  1. How does Redis Cluster shard data? Explain the hash slot mechanism.
  2. What happens when a client sends a command to the wrong node?
  3. How does failover work in Redis Cluster? What triggers it?
  4. What are hash tags and why are they important?
  5. What is the limitation of cross-slot operations? How do you work around it?
  6. Compare Redis Cluster vs client-side sharding vs a proxy (Twemproxy/Codis).
  7. How does resharding work without downtime?
  8. What is the difference between MOVED and ASK redirects?

What to Skip

Skip in interviews:
  • Redis Cluster bus protocol internals (Gossip message format, PING/PONG packet structure).
  • Exact details of the gossip protocol's message types and failure detection tuning.
  • Redis Cluster proxy (redis-cluster-proxy) -- it is niche and rarely deployed at scale.
  • Configuration file details for cluster creation unless asked about operational specifics.
  • Redis on Flash (disk-backed keys with RAM index) -- mention it exists but don't explain internals.
