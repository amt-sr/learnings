7.2.1 Cache Aside

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★
Goal: Master the most widely used caching pattern in production systems. Understand the read and write flows, when to use it versus other patterns, and how to handle edge cases like race conditions during population.

Definition

Cache Aside (also known as Lazy Loading) is a caching pattern where the application is responsible for maintaining the cache. On a read, the application checks the cache first; if the data is not found (cache miss), it reads from the database and populates the cache with the result. On a write, the application updates the database and then invalidates (or updates) the corresponding cache entry. The cache never autonomously loads or writes data — the application orchestrates all cache interactions.

Why "aside"? The cache sits alongside the database rather than between the application and database. The application talks to both the cache and the database directly, choosing when and how to interact with each.

# Pseudocode — Cache Aside Read
function getUser(userId):
    cacheKey = "user:" + userId

    # Step 1: Check cache
    data = cache.get(cacheKey)
    if data is not null:
        return data                    # Cache HIT

    # Step 2: Cache miss — read from DB
    data = db.query("SELECT * FROM users WHERE id = ?", userId)

    if data is not null:
        cache.set(cacheKey, data, TTL=3600)   # Step 3: Populate cache

    return data

# Pseudocode — Cache Aside Write (invalidate)
function updateUser(userId, newData):
    # Step 1: Update the database FIRST
    db.execute("UPDATE users SET name = ? WHERE id = ?", newData.name, userId)

    # Step 2: Invalidate the cache entry
    cache.delete("user:" + userId)

    return success
Why invalidate instead of update? Invalidating is safer and simpler than updating the cache on write. If you update the cache, you must handle partial writes, complex data transformations, and race conditions with concurrent reads. Invalidating simply removes the entry — the next read will repopulate it with fresh data from the DB.

Advantages

  • Simplicity: The pattern is easy to understand, implement, and debug. Application code explicitly controls cache interactions.
  • Flexibility: The application decides what to cache, when to cache it, and what TTL to use. Different keys can have different eviction and invalidation strategies.
  • No stale data on write: Since writes invalidate the cache immediately (after DB commit), the next read always fetches fresh data.
  • Cache only contains hot data: Data is loaded lazily on demand, so cold data never occupies cache memory.
  • Fault isolation: If the cache goes down, the application falls back to reading from the database. There is no intermediate layer that must be functional for the system to operate.

Challenges and Edge Cases

Race Condition: Read-After-Invalidate

A classic concurrency issue occurs when a cache invalidation and a cache population race:

Thread A (Write) Thread B (Read) │ │ │ DB UPDATE user:123 │ │ ────────────────────> │ │ │ │ │ Cache MISS │ │ DB SELECT user:123 │ │ ────────────────────> │ │ <─── stale data (timing) │ │ │ Cache DELETE user:123 │ │ ────────────────────> │ │ │ │ │ Cache SET user:123 (STALE!) │ │ ────────────────────> │ │ Result: Cache now has STALE data!
Mitigation: Use a short TTL as a safety net (even if you invalidate on write), use distributed locks for critical sections, or accept brief inconsistency windows (often acceptable for most applications). For L6+ interviews, discuss this tradeoff explicitly.

Thundering Herd on Cold Cache

When a popular key expires, many concurrent requests may simultaneously find a cache miss and all hit the database. This is the Cache Stampede problem (covered in 7.4.1), and solutions include request coalescing (mutex/lock), probabilistic early expiration, and background refresh.

Common Interview Questions

  • Explain the Cache Aside pattern. How does it work for both reads and writes?
  • Why does Cache Aside invalidate on write instead of updating the cache?
  • What happens if the cache is unavailable? Is the system still functional?
  • Describe the race condition between invalidation and repopulation. How would you mitigate it?
  • How would you choose between Cache Aside and Read Through for a given system?
  • What happens to the cache when the database write fails?

What to Skip

For a system design interview, skip:
  • Implementation details of specific cache client libraries (e.g., Redis Jedis connection pooling configuration).
  • Low-level concurrency primitives (e.g., whether to use compare-and-swap vs. mutex for the race condition — just mention the existence of the problem and a high-level mitigation).
  • Benchmark numbers for specific cache operations unless you have them memorized and they add value to your argument.
