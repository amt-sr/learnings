7.4.2 Cache Penetration

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆
Goal: Understand cache penetration — when queries for nonexistent data bypass the cache and hit the database on every request. Learn the three main solutions: caching null values, Bloom filters, and request rate limiting.

Definition

Cache Penetration occurs when a client repeatedly queries for data that does not exist in the database. Since the data is not in the database, it is never cached, and every request results in a cache miss followed by a database query that returns nothing. The cache provides no protection — every request reaches the database, defeating the purpose of caching entirely.

Cache Penetration vs. Cache Stampede: Cache Stampede (7.4.1) is about many requests for existing data overwhelming the DB when a cache entry expires. Cache Penetration is about requests for nonexistent data always bypassing the cache. Both result in excessive DB load, but for different reasons.

The Problem Visualized

Attacker / Client sends 1000 requests for user:-999 (does not exist) Request 1: Cache MISS → DB query → NULL → (nothing to cache) Request 2: Cache MISS → DB query → NULL → (nothing to cache) Request 3: Cache MISS → DB query → NULL → (nothing to cache) ... Request 1000: Cache MISS → DB query → NULL → (nothing to cache) Cache hit ratio: 0% (cache is useless for these queries) Database: processes ALL 1000 queries that all return NULL

Common Causes

  • Malicious attacks: An attacker intentionally queries for random/nonexistent IDs to overwhelm the database (a form of DoS).
  • Invalid user input: Users search for products, users, or content that does not exist (e.g., searching for a deleted tweet).
  • Data lifecycle gaps: Queries for recently deleted or expired data that has been removed from the database but whose cache entries have already expired.
  • Enumeration attacks: Systematically querying sequential IDs (user:1, user:2, ..., user:1000000) to find valid entries.

Solutions

Solution 1: Cache Null Values

When a query returns null (no data found), cache the null result with a short TTL. Subsequent requests for the same key get a cache hit and return null without hitting the database.

# Pseudocode — Cache Null Values
function getUser(userId):
    cacheKey = "user:" + userId

    data = cache.get(cacheKey)
    if data is not null:
        if data == NULL_SENTINEL:
            return null    # Known to not exist
        return data         # Cached real data

    # Cache miss — query DB
    data = db.query("SELECT * FROM users WHERE id = ?", userId)

    if data is null:
        # Cache the null result with a SHORT TTL (e.g., 60 seconds)
        cache.set(cacheKey, NULL_SENTINEL, TTL=60)
    else:
        cache.set(cacheKey, data, TTL=3600)

    return data
ProsCons
Simple to implementWastes cache memory on null entries
Effective against repeated queries for same missing keyDoes not protect against random/unique missing keys
No additional infrastructureShort TTL means protection is temporary
Key consideration: Use a much shorter TTL for null values (e.g., 30–60 seconds) than for real data (e.g., 1 hour). This limits cache memory consumption while still providing short-term protection.

Solution 2: Bloom Filter

Place a Bloom filter in front of the cache as a fast pre-check. The Bloom filter contains a compact representation of all valid keys. Before querying the cache or database, check the Bloom filter. If it says the key does not exist, return immediately without touching cache or DB.

Request │ ▼ ┌─────────────┐ Key NOT in set? ┌─────────────┐ │ Bloom Filter │───────(NO)──────────────│ Return NULL │ │ │ │ immediately │ │ "Does user: │ └─────────────┘ │ 999 exist?" │ │ │ Key MAY be in set? └──────┬───────┘────────(YES)───────────────────────── │ ▼ ┌─────────────┐ HIT ┌─────────────┐ │ Cache │────────────────────────>│ Return data │ │ │ └─────────────┘ │ │ MISS └──────┬───────┘───────────────────────────────────── │ ▼ ┌─────────────┐ │ Database │─────── Found? ──── Cache result + Return │ │ │ │─────── Not found? ─ Return NULL └─────────────┘ 
# Pseudocode — Bloom Filter
bloomFilter = BloomFilter(expectedInsertions=10_000_000, falsePositiveRate=0.01)

# Populate Bloom filter with all valid user IDs
for user in db.query("SELECT id FROM users"):
    bloomFilter.put(user.id)

function getUser(userId):
    # Step 1: Check Bloom filter
    if not bloomFilter.mightContain(userId):
        return null    # Definitely does not exist — no DB query needed

    # Step 2: Check cache
    data = cache.get("user:" + userId)
    if data is not null:
        return data

    # Step 3: Query DB (may still be a false positive)
    data = db.query("SELECT * FROM users WHERE id = ?", userId)
    if data is not null:
        cache.set("user:" + userId, data, TTL=3600)

    return data
Bloom filter properties:
  • False positives possible: The filter may say a key exists when it does not (hit the DB to confirm).
  • False negatives impossible: If the filter says a key does not exist, it definitely does not exist.
  • Space efficient: A Bloom filter for 10 million keys at 1% false positive rate requires only ~12 MB.
  • Cannot delete entries: Standard Bloom filters do not support deletion (counting Bloom filters do, but with higher overhead).

Solution 3: Request Rate Limiting and Input Validation

Implement rate limiting and input validation at the API layer to prevent abusive or malformed queries from reaching the cache and database.

# Pseudocode — Request rate limiting
function getUser(userId):
    # Validate input
    if not isValidUserId(userId):
        raise InvalidInput("User ID must be a positive integer")

    # Rate limit per client/IP
    rateLimiter.checkLimit(clientIP, maxRequests=100, windowSec=60)

    # Proceed with cache → DB lookup
    ...
Best combined approach: In production, use a layered defense — Bloom filter to quickly reject known-nonexistent keys, cache null values for repeated misses that pass the Bloom filter, and rate limiting to prevent abuse. This is what FAANG-scale systems typically deploy.

Common Interview Questions

  • What is cache penetration? How does it differ from cache stampede?
  • Explain three solutions to cache penetration. Which do you prefer?
  • What is a Bloom filter? What are its properties (false positives, false negatives)?
  • How much memory does a Bloom filter for 100 million keys require at 1% false positive rate?
  • Why would you cache null values? What TTL would you use?
  • Design a system to prevent cache penetration for a URL shortener.
  • What happens when you add a new user to the database but forget to update the Bloom filter?

What to Skip

For a system design interview, skip:
  • Mathematical derivation of Bloom filter false positive probability formulas.
  • Optimal hash function selection for Bloom filters (MurmerHash vs. xxHash vs. FNV).
  • Counting Bloom filter or Cuckoo filter internals (just mention their existence if needed).
  • Rate limiting implementation details (token bucket vs. sliding window internals).
