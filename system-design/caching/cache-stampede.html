7.4.1 Cache Stampede (Dogpiling)

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★
Goal: Understand the cache stampede problem — when a popular cache entry expires and many concurrent requests simultaneously hit the database. Learn the three main solutions (mutex/lock, probabilistic early expiration, background refresh) and when to apply each.

Definition

A Cache Stampede (also called thundering herd or dogpiling) occurs when a highly popular cache key expires and a large number of concurrent requests all detect the cache miss simultaneously. Every request independently queries the database to repopulate the cache, causing a sudden and massive spike in database load that can overwhelm the origin system.

Impact

ImpactSeverity
Database CPU/IO spikeCritical — can cause cascading failures
Increased latency for all requestsHigh — DB under pressure, all queries slow down
Connection pool exhaustionHigh — too many concurrent DB connections
Wasted compute — identical queriesMedium — N identical queries instead of 1
Cache thrashing — repeated overwritesLow — same data written N times

Solutions

Solution 1: Mutex / Lock (Request Coalescing)

When a request finds a cache miss, it acquires a distributed lock before querying the database. Other requests that find the same cache miss either wait for the lock or return a stale value.

R1: Cache MISS → Acquire lock → DB query → SET cache → Release lock R2: Cache MISS → Lock taken → Wait (or return stale) → Cache HIT R3: Cache MISS → Lock taken → Wait (or return stale) → Cache HIT ... Only ONE request hits the database at a time. 
# Pseudocode — Mutex Lock approach
function getUser(userId):
    cacheKey = "user:" + userId
    lockKey = "lock:" + userId

    data = cache.get(cacheKey)
    if data is not null:
        return data

    # Try to acquire a lock (with short timeout)
    lock = cache.set(lockKey, "locked", NX=True, EX=3)
    if lock:
        try:
            # Double-check cache (another process may have populated it)
            data = cache.get(cacheKey)
            if data is null:
                data = db.query("SELECT * FROM users WHERE id = ?", userId)
                cache.set(cacheKey, data, TTL=3600)
            return data
        finally:
            cache.delete(lockKey)
    else:
        # Lock not acquired — another request is fetching.
        # Option A: Return stale data if available
        staleData = cache.get(cacheKey + ":stale")
        if staleData:
            return staleData

        # Option B: Wait and retry (short sleep + recheck)
        sleep(0.1)
        data = cache.get(cacheKey)
        if data:
            return data

        # Option C: Return error / fallback
        raise ServiceUnavailable("Cache is being refreshed")
Pros: Guarantees only one DB query. Cons: Adds latency for waiting requests, lock management complexity, risk of deadlock if lock holder crashes (mitigate with lock TTL).

Solution 2: Probabilistic Early Expiration

Add randomness to the TTL so that the cache entry expires before the official TTL, preventing simultaneous expiration. When a request finds an "early expired" entry, it triggers a background refresh while still serving the (slightly stale) data.

# Pseudocode — Probabilistic Early Expiration
function getUser(userId):
    cacheKey = "user:" + userId
    data, expiry = cache.getWithExpiry(cacheKey)

    if data is not null:
        # If within 10% of TTL expiry, probabilistically refresh
        timeRemaining = expiry - now()
        if timeRemaining < (originalTTL * 0.1):
            if random() < 0.1:    # 10% chance of triggering refresh
                asyncRefresh(userId)  # Background refresh
        return data               # Serve data (possibly slightly stale)

    # Full cache miss — must wait for DB
    data = db.query("SELECT * FROM users WHERE id = ?", userId)
    cache.set(cacheKey, data, TTL=3600)
    return data
Pros: No coordination overhead, no locks, simple to implement. Cons: Stale data served during refresh window, probabilistic (not deterministic — some stampedes may still occur).

Solution 3: Background Refresh (Asynchronous Revalidation)

Use a background process that proactively refreshes cache entries before they expire. The cache never actually expires from the user's perspective — the background job keeps it warm.

Background Worker: Application: │ │ │ Scan cache for entries │ │ expiring in next N minutes │ │ │ │ For each expiring entry: │ │ 1. Read from DB │ │ 2. Update cache with new data │ │ 3. Reset TTL │ │ │ │ (cache never expires from the │ │ application's point of view) │ │ │ │ │ All requests get │ │ cache HIT 
# Background refresh pseudocode
def refreshWorker():
    while true:
        # Find keys expiring in the next 5 minutes
        expiringKeys = cache.scanExpiringKeys(before=300)

        for key in expiringKeys:
            data = db.query(keyToQuery(key))
            cache.set(key, data, TTL=3600)

        sleep(30)  # Run every 30 seconds
Pros: Zero stampede risk — cache never expires for application requests. Cons: Background infrastructure, refreshes data even when it has not changed (wasted DB queries), must handle the full key space.

What to Skip

For a system design interview, skip:
  • Specific Redis SET NX EX implementation details (just explain the concept).
  • Distributed lock algorithms (Redlock debate) — just mention using a lock with TTL.
  • Kernel-level thundering herd problems (epoll vs. select) — different domain.
  • Exact probability calculations for early expiration (just explain the concept).
