Redis Basics

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Master the fundamentals of Redis as an in-memory data structure store. Know its core data types, persistence options, threading model, and common use cases inside a larger system design. Redis appears in nearly every FAANG system design interview as either the primary cache or a supporting component.

Definition

Redis (Remote Dictionary Server) is an open-source, in-memory data structure store. It supports key-value operations on multiple data types (strings, hashes, lists, sets, sorted sets), provides optional persistence to disk, and can serve as a database, cache, message broker, or session store. It operates as a single-threaded event loop processing commands sequentially over TCP.

Key insight: Redis is not just a cache -- it is a versatile in-memory data structure server. While commonly used as a cache (cache-aside, write-through), its rich data types make it suitable for leaderboards (sorted sets), pub/sub messaging, rate limiting, distributed locks, and more. The single-threaded model is a deliberate architectural choice, not a limitation.

Common Command Patterns

# Cache-aside pattern
GET user:1001                          # miss → fetch from DB
SET user:1001 '{"name":"Alice"}' EX 3600  # cache with 1h TTL

# Atomic counter
INCR page:views:article:42

# Distributed lock (SETNX)
SET lock:resource:1 "uuid" NX EX 10   # acquire (10s TTL)

# Sorted set leaderboard
ZADD leaderboard 9500 "player_1"
ZADD leaderboard 8700 "player_2"
ZREVRANGE leaderboard 0 9 WITHSCORES   # top 10

# Hash for object caching
HSET user:1001 name "Alice" age 30
HGET user:1001 name

Single-Threaded Model

Redis processes all commands on a single main thread using an I/O multiplexing event loop (epoll/kqueue). This design eliminates concurrency bugs (no locks needed) and avoids context-switching overhead.

Redis Single-Threaded Architecture Client 1 ──┐ Client 2 ──┤ ┌──────────────────────────┐ Client 3 ──┼── TCP ──│ I/O Multiplexer (epoll) │ Client N ──┘ │ │ │ │ Event Loop (single) │ │ │ │ │ ┌──────▼──────┐ │ │ │ Command │ │ │ │ Processor │ │ │ └──────┬──────┘ │ │ │ │ │ ┌──────▼──────┐ │ │ │ In-Memory │ │ │ │ Data Store │ │ │ └─────────────┘ │ └──────────────────────────┘
Why single-threaded? Redis operations are CPU-bound only on data-type-level operations (hashing, sorting). Memory access is the real bottleneck, and a single thread maximizes L1/L2 cache hit rates. Context switching between threads would degrade memory locality. For I/O, the event loop handles thousands of concurrent connections efficiently.
Background I/O threads: Redis 6.0+ uses separate threads for I/O (reading from sockets, writing to sockets) and for fsync / unlink (file deletion). The command execution itself remains single-threaded. This improves network throughput (up to 2x) without sacrificing simplicity.

Persistence

Redis offers two persistence mechanisms, often used together:

MethodRDB (Redis Database)AOF (Append Only File)
WhatPoint-in-time snapshot of entire datasetSequential log of every write command
FormatBinary, compactText (Redis protocol commands)
Performance impactFork-based, minimal disruptionContinuous writes to disk
Recovery speedFast (load binary dump)Slower (replay all commands)
Data safetyCan lose data between snapshotsConfigurable (always, every sec, never)
File sizeCompactGrows over time (rewrite compacts)
# RDB configuration
save 900 1          # snapshot after 900s if >= 1 write
save 300 10         # snapshot after 300s if >= 10 writes
save 60 10000       # snapshot after 60s if >= 10000 writes

# AOF configuration
appendonly yes
appendfsync everysec   # fsync once per second (default for AOF)

Common Use Cases

Caching (Cache-Aside)

Most common use case. Application checks Redis first, falls back to database on miss, and writes back to Redis. TTLs ensure eventual consistency.

Session Store

Store user session data with automatic TTL expiration. Hashes work well for multi-field session objects. On logout, simply delete the key.

Rate Limiting

Use INCR with TTL (sliding window via sorted sets or Lua scripts). Common patterns: fixed-window counters, sliding-window logs, token bucket.

# Simple fixed-window rate limiter (Lua for atomicity)
EVAL "
  local key = KEYS[1]
  local limit = tonumber(ARGV[1])
  local window = tonumber(ARGV[2])
  local current = redis.call('INCR', key)
  if current == 1 then
    redis.call('EXPIRE', key, window)
  end
  if current > limit then
    return 0  -- rejected
  end
  return 1  -- allowed
" 1 rate_limit:user:42 100 60

Leaderboards

Sorted sets with scores as the ranking metric. O(log N) insert/update, O(log N + M) range queries for top-M players.

Publish/Subscribe

Real-time messaging between services. Producers publish to channels; subscribers receive messages. Note: messages are not persisted -- if no subscriber is listening, the message is lost.

Common Interview Questions

  1. What data types does Redis support? When would you use a sorted set vs a list?
  2. Why is Redis single-threaded? What are the benefits and limitations?
  3. Explain the difference between RDB and AOF persistence. Which would you choose for a session store vs an analytics cache?
  4. How would you implement rate limiting with Redis? What are the edge cases?
  5. Design a leaderboard system using Redis. How does it scale?
  6. How does Redis handle eviction when memory is full? What are the different policies?
  7. What is the difference between Redis and Memcached? When would you choose one over the other?
  8. How would you implement a distributed lock with Redis? What happens if the lock holder crashes?

What to Skip

Skip in interviews:
  • Redis Modules internals (RediSearch, RedisJSON, RedisGraph) unless the design specifically calls for them. Mention their existence, don't explain internals.
  • Low-level details of the I/O multiplexer implementation (epoll vs kqueue specifics).
  • RDB file format binary layout or AOF rewrite internals.
  • Redis Sentinel deep internals -- state the concept (auto-failover) and move on unless asked.
  • Exact byte-level encoding of Redis protocol (RESP).
