FIFO (First In First Out)

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand FIFO as the simplest eviction policy, recognize its limitations, and articulate when it is an acceptable choice. Be prepared to compare it against LRU and LFU and explain why most production systems choose a more sophisticated alternative.

Definition

FIFO (First In First Out) is a cache eviction policy that removes the oldest entry in the cache -- the one that was inserted first. Unlike LRU, it does not track access order after insertion. An item remains in its insertion position regardless of how many times it is subsequently read.

Key insight: FIFO is the simplest possible eviction policy. It requires only a queue data structure with no per-access bookkeeping. This makes it extremely fast and easy to implement, but at the cost of ignoring all access patterns.

How FIFO Works

New entries are appended to the tail of a queue. When the cache is full, the entry at the head (oldest) is removed. Cache hits do not change the item's position in the queue.

FIFO Cache (capacity = 3) Queue (oldest → newest): [A] ← [B] ← [C] put(D): evict A (oldest) Queue: [B] ← [C] ← [D] get(B) → hit (B's position unchanged) Queue: [B] ← [C] ← [D] ← B is NOT promoted put(E): evict B (still oldest despite being just accessed) Queue: [C] ← [D] ← [E]
Core problem: FIFO evicts B immediately after it was accessed because it ignores recency. A frequently accessed item that happens to be old will be evicted over a rarely accessed but newly inserted item.

Second Chance (Clock) Algorithm

A practical improvement over naive FIFO is the Second Chance (or Clock) algorithm. Each entry gets a reference bit. On eviction, if the head entry's bit is set (it was accessed recently), clear the bit and move it to the tail instead of evicting. This gives frequently accessed items a second chance before eviction.

Clock Algorithm (capacity = 4, circular buffer): Head → [A:0] [B:1] [C:0] [D:1] ← Tail Eviction check: A has bit=0 → evict A Result: [B:1] [C:0] [D:1] [E:0] If A had bit=1: Clear bit, move to tail: [B:0] [C:0] [D:1] [A:0] Then check B → if B:0, evict B
Note: The Clock algorithm approximates LRU with O(1) amortized time and is widely used in OS page replacement. Linux's Active/Inactive LRU lists are conceptually derived from this idea.

When Is FIFO Acceptable?

FIFO can be a reasonable choice in specific scenarios:

  • Write-through caches with TTL: When all entries have a natural expiration (TTL) and the eviction policy is only a backstop for TTL expiration delays, FIFO's simplicity is sufficient.
  • Uniform access patterns: When all cached items have roughly equal access probability (no hot/cold skew), FIFO performs comparably to more complex policies.
  • Hardware/constrained environments: In embedded systems or hardware caches where per-access bookkeeping is too expensive, FIFO's minimal overhead is a real advantage.
  • Supplementary queue before LRU: FIFO is used as the "admission queue" in 2Q/LRU-K algorithms, filtering one-time accesses before they enter the main LRU cache.
Redis has volatile-fifo and allkeys-fifo policies available. While rarely chosen over LRU in production, they are useful for A/B testing or for workloads where simplicity is prioritized.

Common Interview Questions

  1. Explain FIFO eviction and why it is the simplest policy. What is its main weakness?
  2. How does the Second Chance (Clock) algorithm improve on FIFO?
  3. Compare FIFO vs LRU hit rates for a social-media feed workload.
  4. Under what circumstances would you choose FIFO over LRU?
  5. How is FIFO used as part of more complex eviction schemes (2Q, LRU-K)?
  6. What is the time complexity of FIFO eviction vs LRU eviction?

What to Skip

Skip in interviews:
  • Implementation details of enhanced FIFO variants like Clock-Pro or CAR (Clock with Adaptive Replacement) unless asked directly.
  • Theoretical cache hit-rate analysis using Belady's optimal algorithm -- mention it as the theoretical upper bound, don't derive it.
  • Detailed comparison of FIFO vs ARC vs 2Q unless the conversation naturally goes there.
  • Memcached slab allocator internals in the context of FIFO -- the slab allocator is orthogonal to eviction policy.
