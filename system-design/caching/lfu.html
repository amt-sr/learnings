LFU (Least Frequently Used)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand LFU eviction, its frequency-counting mechanics, the new-item starvation problem, decay mechanisms, and how Redis implements LFU in practice. Be ready to compare LFU against LRU and justify when each is the better choice.

Definition

LFU (Least Frequently Used) is a cache eviction policy that removes the item with the lowest access frequency count. Each cache entry maintains a counter that increments on every access (read or write). When the cache is full, the entry with the smallest counter is evicted.

Key insight: LFU approximates popularity -- it assumes that data accessed many times in the past will continue to be accessed often. This works well for workloads with stable popularity distributions (e.g., trending content, product catalogs, recommendation caches).

How LFU Works

Every cache entry tracks a frequency counter. On each access, the counter increments. During eviction, the entry with the lowest frequency is chosen. Ties are broken by recency (LRU within the same frequency bucket).

LFU Cache (capacity = 3) Frequency buckets (LRU within each bucket): freq=5: [D] freq=3: [B] ←→ [A] freq=1: [C] get(E) → miss, insert E with freq=1: freq=5: [D] freq=3: [B] ←→ [A] freq=1: [C] ←→ [E] (E added to freq=1 bucket) Evict lowest freq → remove C (LRU tail of freq=1): freq=5: [D] freq=3: [B] ←→ [A] freq=1: [E]

The New-Item Starvation Problem

LFU weakness -- new-item starvation: A newly inserted item starts with frequency = 1. In a cache full of items with high frequency counts (e.g., freq=1000), the new item will be evicted immediately on the next cache miss -- even if it will become popular. Old, once-popular items that are no longer accessed keep their high counts and persist indefinitely.
Without decay: freq=500: [X] (popular 2 months ago, never accessed since) freq=500: [Y] (popular 2 months ago, never accessed since) freq=500: [Z] (popular 2 months ago, never accessed since) New hot item [H] arrives → freq=1 → immediately evicted! X, Y, Z are "zombie" entries consuming cache slots.

Decay Mechanism

To prevent zombie entries, LFU implementations periodically decay (reduce) all frequency counters. This gives older, less-accessed entries a chance to be overtaken by newer, more active entries.

Redis LFU: Probabilistic Decay

Redis implements a practical LFU with a logarithmic counter and probabilistic decay:

  • Logarithmic counter: An 8-bit counter (0-255) that grows logarithmically. Each increment only has a probability of incrementing based on the current value. This means counter 10 increments more often than counter 200.
  • Probabilistic decay: Every N minutes, Redis probabilistically decrements each key's counter by 1. The probability is configured via lfu-decay-time (default: 1 minute). This is NOT a full scan -- counters are decayed lazily when accessed.
  • LFUttl: A new item starts with a configurable initial counter value (default: 5) to provide a "warm-up" period and reduce starvation.
# Redis LFU configuration
maxmemory-policy volatile-lfu    # evict LFU among keys with TTL
lfu-decay-time 1                 # decay factor: ~10% chance per minute
lfu-log-factor 10                # logarithmic counter increment probability
Why logarithmic? A linear counter would overflow an 8-bit field quickly in high-traffic systems. A logarithmic counter provides a wider dynamic range -- it can distinguish between items accessed 10 times vs 10,000 times without needing a 32-bit counter.

O(1) LFU Implementation

An O(1) LFU cache uses a HashMap for key lookup, a HashMap mapping frequency to a doubly-linked list (DLL) of keys at that frequency, and a minimum-frequency tracker:

class LFUCache:
    def __init__(self, capacity: int):
        self.cap = capacity
        self.key_to_val_freq = {}        # key → (value, freq)
        self.freq_to_keys = defaultdict(OrderedDict)  # freq → OrderedDict
        self.min_freq = 0

    def get(self, key: int) -> int:
        if key not in self.key_to_val_freq:
            return -1
        val, freq = self.key_to_val_freq[key]
        # Move key to freq+1 bucket
        del self.freq_to_keys[freq][key]
        if not self.freq_to_keys[freq]:
            if freq == self.min_freq:
                self.min_freq += 1
            del self.freq_to_keys[freq]
        self.freq_to_keys[freq + 1][key] = True
        self.key_to_val_freq[key] = (val, freq + 1)
        return val

    def put(self, key: int, value: int) -> None:
        if self.cap <= 0: return
        if key in self.key_to_val_freq:
            self.get(key)  # promote freq
            self.key_to_val_freq[key] = (value, self.key_to_val_freq[key][1])
            return
        if len(self.key_to_val_freq) >= self.cap:
            # Evict LRU key from min_freq bucket
            evict_key, _ = self.freq_to_keys[self.min_freq].popitem(last=False)
            del self.key_to_val_freq[evict_key]
        self.key_to_val_freq[key] = (value, 1)
        self.freq_to_keys[1][key] = True
        self.min_freq = 1

Common Interview Questions

  1. Design an LFU cache with O(1) get and put. (LeetCode 460.)
  2. How does LFU handle the new-item starvation problem? What is decay?
  3. Compare LRU vs LFU for a video streaming platform's content cache.
  4. How does Redis implement LFU? Why use a logarithmic counter?
  5. What happens if you use LFU for a workload with strong temporal locality but no stable popularity?

What to Skip

Skip in interviews:
  • The exact Redis logarithmic counter probability formula -- describe the concept, don't derive the math.
  • Fine-grained details of the Redis lfu-log-factor lookup table implementation.
  • Optimizations like TinyLFU (used in Caffeine) unless the interviewer explicitly brings up advanced eviction policies.
  • ARC (Adaptive Replacement Cache) internals unless prompted -- it is a hybrid LRU/LFU but rarely asked directly.
