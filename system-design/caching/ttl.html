7.3.3 TTL (Time To Live)

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆
Goal: Understand TTL as the simplest and most universally applicable cache invalidation strategy. Learn the difference between absolute and sliding TTL, how to choose TTL values, and advanced HTTP-style patterns like stale-while-revalidate and stale-if-error.

Definition

TTL (Time To Live) is an expiration mechanism that automatically invalidates a cache entry after a specified duration. Once the TTL elapses, the entry is either removed or treated as expired. TTL is the simplest form of cache invalidation — it requires no coordination, no event propagation, and no knowledge of when the underlying data actually changes.

Types of TTL

Absolute TTL

The entry expires at a fixed point in time, regardless of when it was last accessed. The countdown starts when the entry is created.

# Absolute TTL: expires 1 hour from creation, regardless of reads
cache.set("user:123", data, TTL=3600)

# Even if accessed 59 minutes later, it expires at the 60-minute mark

Sliding TTL (Rotating TTL)

The TTL is reset on every access. As long as the entry is being actively read, it never expires. The entry only expires after a period of inactivity.

# Sliding TTL: TTL resets to 1 hour on every read
cache.set("user:123", data, TTL=3600, sliding=True)

# If accessed at 59 minutes, TTL resets to 1 hour
# Expires only if not accessed for 60 consecutive minutes

Choosing TTL Values

Selecting the right TTL is a balancing act between freshness and performance. There is no universal formula — it depends on the data type, access pattern, and business requirements.

Guidelines by Data Type

Data TypeRecommended TTLRationale
User profiles5–60 minutesProfiles change infrequently; short TTL limits staleness
Feed / timeline1–5 minutesUsers expect near-real-time updates
Product catalog1–24 hoursChanges are batched; stale data is acceptable
Configuration / feature flags10–60 secondsShould propagate quickly but not overload DB
Search results5–15 minutesIndex updates are eventual anyway
Computed aggregations (counts, stats)1–60 minutesExactness is often not critical
Rate limit counters1–60 secondsWindow-aligned with the rate limit period
Sessions15–60 minutes (sliding)Inactivity-based expiration

The TTL Tuning Framework

Three factors to balance:
  1. Business staleness tolerance: How out-of-date can this data be without user complaints or business impact?
  2. DB load capacity: How many cache misses (DB reads) can your database handle per second?
  3. Data change frequency: How often does this data actually change? Set TTL close to the typical change interval.
Rule of thumb: TTL should be shorter than the business staleness tolerance but long enough that the database can handle the resulting miss rate.

Advanced TTL Patterns

Stale-While-Revalidate

When a cache entry expires, serve the stale data immediately while asynchronously refreshing it from the source. The user never waits for a cache miss — they get stale data instantly, and the next request gets fresh data.

Time ──────────────────────────────────────────────────────> Cache entry: [FRESH]─────────[EXPIRED]────────[FRESH again] │ ▲ 1. Serve stale data 3. Revalidation 2. Background refresh complete from source User experience: Always fast response (stale, then fresh) 
# HTTP header example (RFC 5861)
Cache-Control: max-age=3600, stale-while-revalidate=86400

# Application-level pseudocode
function getUser(userId):
    data, expired = cache.getWithExpiry("user:" + userId)
    if data is not null:
        if expired:
            asyncRefresh(userId)   # Fire-and-forget background refresh
        return data               # Serve stale data immediately
    else:
        data = db.query("SELECT * FROM users WHERE id = ?", userId)
        cache.set("user:" + userId, data, TTL=3600)
        return data

Stale-If-Error

If the origin server returns an error (5xx, timeout), serve the stale cached data instead of an error page. This dramatically improves resilience.

# HTTP header example (RFC 5861)
Cache-Control: max-age=3600, stale-if-error=86400

# Application-level pseudocode
function getUser(userId):
    data, expired = cache.getWithExpiry("user:" + userId)
    if data is not null:
        if expired:
            try:
                newData = db.query("SELECT * FROM users WHERE id = ?", userId)
                cache.set("user:" + userId, newData, TTL=3600)
                return newData
            catch (DBError):
                return data   # Serve stale data on DB failure
        return data
    else:
        # No data at all — cannot serve stale
        raise NotFoundError

Redis TTL Configuration

# Set key with TTL (seconds)
SET user:123 '{"name":"Alice"}' EX 3600

# Set key with TTL (milliseconds)
PSET user:123 '{"name":"Alice"}' PX 3600000

# Set key with expiration at a specific Unix timestamp
EXPIREAT user:123 1720000000

# Get remaining TTL
TTL user:123    # Returns seconds remaining (or -1 if no TTL, -2 if key missing)
PTTL user:123   # Returns milliseconds remaining

# Update TTL on access (sliding expiration) — must do manually
# No built-in sliding TTL in Redis; implement in application code
if exists("user:123"):
    EXPIRE user:123 3600   # Reset TTL on access

Common Pitfalls

  • TTL too long: Data becomes stale for extended periods. Users see outdated information (e.g., wrong price, old profile data).
  • TTL too short: Cache hit ratio drops, defeating the purpose of caching. Database load increases.
  • All keys with the same TTL: Causes Cache Avalanche (7.4.3) — massive simultaneous expiration.
  • No TTL at all: Memory leak if entries are never evicted. The cache grows unbounded until OOM.

Common Interview Questions

  • What is TTL? Why is it the simplest cache invalidation strategy?
  • How would you choose TTL values for different types of cached data?
  • What is the difference between absolute and sliding TTL? Give an example of when to use each.
  • What is stale-while-revalidate? How does it improve user experience?
  • What is stale-if-error? When is it useful?
  • What happens if all cache entries have the same TTL? What problem does this cause?

What to Skip

For a system design interview, skip:
  • Redis TTL implementation internals (how Redis uses a hashed timer wheel for expiry).
  • DNS TTL mechanics (a different use of TTL in a different system).
  • IP packet TTL (network layer, not cache layer).
  • Exact RFC 5861 header syntax and edge cases — just explain the concept.
