7.2.5 Write Back (Write Behind)

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆
Goal: Understand the Write Back caching pattern — how it achieves the lowest write latency by deferring database writes, the data loss risk it introduces, and where it is used in real systems (CPU caches, databases, message queues).

Definition

Write Back (also called Write Behind) is a caching pattern where write operations update the cache only, and the cache asynchronously flushes the data to the database at a later time. The application receives an immediate acknowledgment after the cache write — it does not wait for the database write. A background process (or the cache itself) is responsible for periodically writing dirty (modified) entries from the cache to the database.

Critical tradeoff: Write Back provides the lowest possible write latency but introduces a data loss window. If the cache fails or is restarted before dirty entries are flushed to the database, those writes are permanently lost.

The write latency improvement is dramatic — from milliseconds to sub-millisecond. But this comes at the cost of a consistency window and data loss risk.

The Dirty Bit and Flush Mechanics

Write Back implementations track which cache entries have been modified but not yet persisted:

  • Dirty bit: Each cache entry has a flag indicating whether it has been modified since the last DB flush.
  • Flush triggers: Background flush can be triggered by:
    • Time-based: Every N seconds (e.g., every 5 seconds)
    • Threshold-based: When dirty entries exceed a count (e.g., 1000 dirty entries)
    • Size-based: When dirty data exceeds a memory threshold
    • On eviction: When a dirty entry is evicted from cache, it must be flushed first
  • Write coalescing: Multiple writes to the same key can be coalesced into a single DB write. If key K is written 100 times in a flush interval, only the final value needs to be persisted.
Write coalescing benefit: In write-heavy workloads (e.g., incrementing a counter 10,000 times/sec), Write Back can reduce 10,000 DB writes to a single batched write per flush interval. This is a massive reduction in DB load.

Data Loss Risk

This is the critical tradeoff: If the cache server crashes or loses power before dirty entries are flushed, those writes are lost forever. The window of vulnerability is the time between the cache write and the successful flush to the database.

Mitigating Data Loss

  • Short flush intervals: Reduce the window of vulnerability (e.g., flush every 1–5 seconds).
  • Write-ahead log (WAL): Before acknowledging the write, log it to a durable WAL. If the cache crashes, the WAL can be replayed to recover dirty entries.
  • Replication: Replicate the cache to a standby node. If the primary fails, the standby has the dirty entries and can take over flushing.
  • Acceptable loss: Some applications can tolerate brief data loss (e.g., analytics counters, page view counts) and use Write Back without additional safeguards.

Real-World Usage

SystemHow It Uses Write Back
CPU Caches (L1/L2/L3)Writes go to cache line (marked dirty); flushed to main memory on eviction or cache line replacement. This is the original write-back pattern.
Database Buffer PoolsPostgreSQL, MySQL InnoDB buffer pools use write-back: dirty pages are written to disk asynchronously by a background writer.
Redis (with appendonly)Redis can be configured as a write-back cache using AOF fsync=everysec — writes are acknowledged immediately, flushed to disk every second.
KafkaProducers write to the page cache (memory); the OS asynchronously flushes to disk. Acknowledgment depends on the configured acks setting.
InfluxDBWrites go to an in-memory WAL; flushed to persistent storage in the background.

Common Interview Questions

  • Explain the Write Back caching pattern. What are its advantages and risks?

  • What is write coalescing? How does Write Back benefit from it?
  • Compare Write Back vs. Write Through. When would you choose each?

What to Skip

For a system design interview, skip:
  • Low-level CPU cache line replacement algorithms (LRU vs. pseudo-LRU vs. random) unless the question is about hardware architecture.
  • Database WAL implementation details (e.g., how InnoDB's redo log handles checkpointing) unless the question targets database internals.
  • Specific Linux kernel page cache mechanics (dirty_ratio, dirty_background_ratio tunables).
