7.4.3 Cache Avalanche

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆
Goal: Understand the cache avalanche problem — when a large portion of cached data expires simultaneously, overwhelming the database with misses. Learn the common causes and three main solutions: jittered TTL, multi-layer caching, and circuit breakers.

Definition

A Cache Avalanche occurs when a large number of cache entries expire at approximately the same time, causing a massive surge of cache misses that all hit the database simultaneously. Unlike cache stampede (which affects a single popular key), an avalanche affects many keys across the cache, potentially representing a significant portion of the cache's working set.

Avalanche vs. Stampede vs. Penetration:
  • Stampede (7.4.1): Many requests for one key that just expired.
  • Penetration (7.4.2): Requests for nonexistent keys that always miss.
  • Avalanche (this section): Many requests for many different keys that all expired at the same time.

Causes

Cause 1: Mass Expiration with Same TTL

The most common cause: a batch process or bulk import caches thousands of entries with the same TTL. When the TTL elapses, all entries expire simultaneously.

Time ──────────────────────────────────────────────────────> Batch process at T=0 caches 10,000 entries, all with TTL=3600s T=0 (cache warm): Key 0001: [CACHED] TTL=3600 Key 0002: [CACHED] TTL=3600 Key 0003: [CACHED] TTL=3600 ... Key 9999: [CACHED] TTL=3600 T=3600 (all expire simultaneously): Key 0001: [EXPIRED] ────> DB query Key 0002: [EXPIRED] ────> DB query Key 0003: [EXPIRED] ────> DB query ... Key 9999: [EXPIRED] ────> DB query Database: Suddenly handles 10,000 queries that were previously cached

Cause 2: Cache Server Restart / Failure

When a Redis or Memcached node restarts or crashes, its entire in-memory cache is lost. All requests that were previously cache hits now become cache misses and hit the database until the cache is repopulated.

Normal: Cache handles 95% of reads, DB handles 5% ┌──────────────┐ Requests ──> │ Cache (95%) │ ──> Return data │ │ │ DB (5%) │ ──> Return data └──────────────┘ After restart: Cache is empty, ALL requests hit DB ┌──────────────┐ Requests ──> │ Cache (0%) │ ──> MISS │ │ │ DB (100%) │ ──> OVERWHELMED! └──────────────┘ Recovery time depends on: - How quickly requests repopulate the cache - How many unique keys are accessed - Whether DB can handle the surge without timing out

Cause 3: Network Partition Between App and Cache

If the network connection between application servers and the cache cluster is interrupted, all cache reads fail, and requests fall through to the database. This is similar to a cache restart but transient (assuming the network recovers).

Solutions

Solution 1: Jittered TTL (Randomized Expiration)

Add random jitter to each cache entry's TTL so that entries do not expire simultaneously. Instead of all entries expiring at T+3600, they expire uniformly between T+3600 and T+4200 (with a 10-minute jitter).

# Pseudocode — Jittered TTL
import random

BASE_TTL = 3600          # 1 hour base TTL
JITTER   = 600           # 10 minutes of random jitter

def getJitteredTTL():
    return BASE_TTL + random.randint(0, JITTER)

# When caching data:
cache.set("user:123", data, TTL=getJitteredTTL())   # TTL: 3600-4200s
cache.set("user:456", data, TTL=getJitteredTTL())   # TTL: 3600-4200s
cache.set("user:789", data, TTL=getJitteredTTL())   # TTL: 3600-4200s

# Entries expire over a 10-minute window instead of all at once
Pros: Simple to implement, no additional infrastructure, effectively spreads expiration over a time window. Cons: Some entries expire earlier than necessary (slightly lower average TTL), does not protect against cache server restarts.

Solution 2: Multi-Layer Caching

Deploy caching at multiple levels so that if one layer fails or expires, the next layer provides protection. The layers act as fallbacks.

Layer 1: In-Process Cache (per app instance, LRU cache) │ TTL: 60 seconds │ Capacity: 1000 entries ▼ Layer 2: Distributed Cache (Redis cluster) │ TTL: 1 hour (jittered) │ Capacity: 50 GB ▼ Layer 3: Database Source of truth If Redis restarts: - Layer 1 still serves hot data for up to 60 seconds - DB handles the rest, but only Layer 1 misses (limited set) - Redis gradually repopulates from DB via Layer 1 misses
LayerTechnologyTTLRole
L1Guava Cache, Caffeine (in-process)Short (30–120s)Absorb brief cache outages, reduce Redis calls
L2Redis / Memcached (distributed)Medium (minutes to hours)Primary cache layer, shared across app instances
L3CDNLong (hours to days)Edge caching for static or semi-static content

Solution 3: Circuit Breaker

Implement a circuit breaker that detects when the database is under excessive load (due to cache avalanche or other causes) and temporarily rejects requests or serves degraded responses.

Normal state: Requests → Cache → (miss) → DB → Response DB under stress (circuit breaker opens): Requests → Cache → (miss) → Circuit Breaker → Fallback response (reject / stale data / queue for retry) 
# Pseudocode — Circuit Breaker
class CircuitBreaker:
    CLOSED:     # Normal — requests pass through
    OPEN:       # Failing — requests rejected
    HALF_OPEN:  # Testing — allow some requests to test DB health

def getUser(userId):
    data = cache.get("user:" + userId)
    if data is not null:
        return data

    # Cache miss — check circuit breaker
    if circuitBreaker.state == OPEN:
        # DB is under stress — serve fallback
        return getFallbackResponse(userId)

    try:
        data = db.query("SELECT * FROM users WHERE id = ?", userId)
        cache.set("user:" + userId, data, TTL=3600)
        circuitBreaker.recordSuccess()
        return data
    catch (DBTimeout):
        circuitBreaker.recordFailure()
        return getFallbackResponse(userId)
Pros: Protects database from cascading failure. Cons: Degrades user experience during avalanche (serving fallback data), requires tuning of failure thresholds and recovery timing.

Solution 4: Graceful Degradation with Stale Data

When the database is overwhelmed, serve stale cached data even if the TTL has expired. This is the stale-while-revalidate pattern (covered in 7.3.3 TTL) applied at the system level.

# Keep expired entries in cache (marked as stale)
# Serve stale data when DB is under pressure

def getUser(userId):
    data, expired = cache.getWithExpiry("user:" + userId)

Common Interview Questions

  • What is a cache avalanche? How does it differ from cache stampede?
  • What are the common causes of cache avalanche?
  • How does jittered TTL prevent cache avalanche?
  • How does multi-layer caching protect against cache avalanche?
  • What is a circuit breaker? How does it help during a cache avalanche?
  • How would you handle a Redis cluster restart in a production system?
  • Compare and contrast cache stampede, penetration, and avalanche.

What to Skip

For a system design interview, skip:
  • Circuit breaker library configuration (e.g., Hystrix, Resilience4j settings and internals).
  • Redis persistence mechanisms (RDB/AOF) as a recovery strategy for cache restarts — that is a different topic.
  • Mathematical analysis of TTL jitter distribution (uniform vs. Gaussian).
  • Specific CDN failover mechanisms (e.g., Cloudflare origin shield internals).
