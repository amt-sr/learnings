4.1 Why Replication

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★★

Goal

Understand why replication is a foundational primitive in distributed systems, what problems it solves, and the fundamental tradeoffs it introduces. This topic is the gateway to understanding every replication model, consistency guarantee, and partition tolerance decision that follows.

Definition

Replication means maintaining multiple copies of data across different nodes or data centers. Each copy is called a replica. The process of keeping these copies in sync is called replication.

Replication is not an optional feature in distributed systems -- it is a structural necessity. Any system that stores data on more than one machine (which is every system that scales) must decide how to keep those copies consistent. The choices made here cascade through every other aspect of system design.

Why Replicate Data?

1. High Availability

A single machine will eventually fail. Disk dies, network partitions, kernel panics, entire racks lose power. With replication, when one node fails, others continue serving requests.

Without Replication: ┌──────────┐ │ Client │──▶ ┌──────────┐ └──────────┘ │ Node A │ ◀── 💥 CRASH = SYSTEM DOWN │ (single) │ └──────────┘ With Replication: ┌──────────┐ │ Client │──▶ ┌──────────┐ └──────────┘ │ Node A │ ◀── 💥 CRASH └──────────┘ ┌──────────┐ │ Node B │ ◀── Continues serving requests └──────────┘
Interview Tip: When asked "how would you make this system highly available?", replication is almost always the first answer. Frame it as: "We replicate data across multiple nodes so that no single point of failure can bring down the system."

2. Read Scalability

Writes are hard to scale because they require coordination and ordering. Reads, however, are embarrassingly parallel. By replicating data, you can distribute read traffic across multiple nodes.

┌─────────┐ ┌──────────┐ │Client 1 │──▶ │ Node A │ ◀── Handles writes + reads └─────────┘ └──────────┘ ┌─────────┐ ┌──────────┐ │Client 2 │──▶ │ Node B │ ◀── Handles reads only └─────────┘ └──────────┘ ┌─────────┐ ┌──────────┐ │Client 3 │──▶ │ Node C │ ◀── Handles reads only └─────────┘ └──────────┘ Write: Node A only (1 node) Read: Node A + B + C (3 nodes) Read throughput: 3x improvement

This is why most web applications have read-heavy workloads (typically 80-95% reads). Replication lets you scale reads almost linearly by adding more replicas, while writes still go through the coordination path.

3. Geographic Locality

Speed of light is the ultimate latency floor. A request from Tokyo to a data center in Virginia takes roughly 150-200ms round trip. By placing replicas closer to users, you can reduce latency dramatically.

4. Disaster Recovery

Entire data centers can fail -- natural disasters, power grid failures, network provider outages. Cross-DC replication ensures data survives catastrophic events.

Failure ScenarioWithout ReplicationWith Cross-DC Replication

Rack failureMultiple nodes lostReplicas spread across racks
Entire DC failureCatastrophic data lossOther DCs take over
Region-wide disasterComplete outageRemote region survives

The Fundamental Tradeoff

Replication forces you to choose: Do you want strong consistency (all replicas always show the same data) or high availability (the system is always responsive)? You cannot perfectly have both in the presence of network partitions. This is the CAP theorem in action.
Strong Consistency ▲ │ │ │ Replication │ Tradeoff Space │ │ ┌──────────────────┼──────────────────┐ │ │ │ │ Eventual │ Causal │ │ Consistency │ Consistency │ │ │ │ │ ● DynamoDB │ ● Google │ │ ● Cassandra │ Spanner │ │ │ ● CockroachDB │ │ ● S3 │ │ │ │ │ └──────────────────┴──────────────────┘ │ │ ▼ High Availability

When you design a replicated system, you are implicitly choosing a point on this spectrum:

  • Strong consistency: Every read returns the most recent write. Achieved via synchronous replication, consensus protocols (Paxos, Raft). Tradeoff: higher latency, lower availability during partitions.
  • Eventual consistency: Replicas converge over time but may serve stale reads temporarily. Tradeoff: low latency, high availability, but clients may see outdated data.
  • Causal consistency: A middle ground. If operation B causally depends on operation A, every replica sees A before B. Good for collaborative systems.

Replication as Foundation of Distributed Systems

Replication is not just a storage concern. It touches every layer of a distributed system:

  • Load balancing: Replicas give you more backend targets to distribute traffic to.
  • Caching: Caches are essentially ephemeral, volatile replicas.
  • Sharding: Each shard is typically replicated for fault tolerance.
  • Consensus: Protocols like Raft and Paxos exist primarily to solve the replication ordering problem.
  • Message queues: Replicated logs (like Kafka) ensure no data loss on broker failure.
Key Insight: Almost every system design interview question implicitly involves replication. Designing a URL shortener? You need replication for the database. Designing a chat system? You need replication for message durability. Designing a notification system? You need replication for delivery guarantees. Understanding replication deeply is one of the highest-ROI investments for interview preparation.

Common Interview Questions

  1. "Why do you need replication in your design?"
    Answer: "Replication serves four purposes: (1) High availability -- no single node failure causes downtime. (2) Read scalability -- distribute read traffic across replicas. (3) Geographic locality -- serve users from nearby data centers. (4) Disaster recovery -- survive entire data center failures. For this system, the primary driver is [X], which is why I chose [Y] replication model."
  2. "What is the CAP theorem and how does it relate to replication?"
    Answer: "In the presence of a network partition, you must choose between consistency and availability. Replication inherently creates a consistency challenge because you have multiple copies of data. Synchronous replication favors consistency (C), asynchronous replication favors availability (A). For [this system], I'd choose [C or A] because [rationale]."
  3. "How would you decide between strong and eventual consistency?"
    Answer: "It depends on the use case. Financial transactions require strong consistency -- you cannot double-spend. Social media feeds can tolerate eventual consistency -- a slightly stale feed is acceptable. I would analyze each data type in the system and choose consistency per-key, not per-system."
  4. "What happens during a network partition with replicated data?"
    Answer: "During a partition, some replicas cannot communicate. You face a choice: stop serving writes to maintain consistency (CP) or continue serving writes and resolve conflicts later (AP). The resolution strategy depends on the replication model -- leader-follower typically stops non-leader writes, leaderless systems use quorum and conflict resolution."

What to Skip

Do not go deep into:
  • Specific replication protocols (Raft, Paxos, Viewstamped Replication) -- these are covered in the Consensus chapter. Know what they do, not how the internals work.
  • Multi-master conflict resolution details (CRDT internals, operational transforms) -- save for the Multi-Leader section.
  • Storage engine internals (LSM trees, B-trees in the context of replication) -- keep the focus on the replication logic itself.
  • Specific database configuration syntax (e.g., MySQL binlog settings) unless you have deep operational experience.
