Hinted Handoff

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand hinted handoff — a mechanism that provides high write availability even when replicas are temporarily down, by temporarily storing writes on behalf of unavailable nodes and replaying them when the nodes recover.

Definition

Hinted handoff is a mechanism in Dynamo-style distributed databases where, when a replica node is unavailable or slow, a neighboring node temporarily stores (hints) the writes intended for the unavailable node. When the downed node comes back online, the hinting node delivers (hands off) all the stored writes to it.

How Hinted Handoff Works

NORMAL STATE (3 replicas, N=3, W=2): Ring hash: key="user:42" -> Node-A (primary), Node-B (replica-1), Node-C (replica-2) Client writes {user:42, name: "Alice"} with W=2 Client -> Coordinator -> Node-A: WRITE [ACK] -> Node-B: WRITE [ACK] -> Node-C: (not needed, W=2 satisfied) Result: Write succeeded with W=2 acknowledgments. NODE-C GOES DOWN: Ring hash: key="user:42" -> Node-A (primary), Node-B (replica-1), Node-C (DOWN) Client writes {user:42, city: "NYC"} with W=2 Client -> Coordinator -> Node-A: WRITE [ACK] -> Node-B: WRITE [ACK] -> Node-C: WRITE [NO RESPONSE - down!] HINTED HANDOFF TRIGGERED: Coordinator stores hint on Node-A: Hint = { target: Node-C, key: "user:42", value: {city: "NYC"}, v=8 } (or Coordinator stores locally) W=2 still satisfied (A and B acknowledged). Client sees: Write succeeded. But Node-C is missing the write. NODE-C RECOVERS: Node-A detects Node-C is back (gossip protocol / heartbeat) HANDOFF: Node-A sends hint to Node-C: "Here are the writes you missed while you were down: { key: user:42, value: {city: "NYC"}, v=8 }" Node-C applies the write: Node-C: { user:42, name: "Alice", city: "NYC", v=8 } Hint deleted from Node-A. All replicas consistent again.

Key Properties

Property Details
Write availability Writes succeed even when some replicas are down. The system does not require all N nodes to be healthy to accept writes (as long as W nodes are reachable).
Temporary storage Hints are stored on a transient basis. They are not a permanent replacement for the target node. The hinting node is a surrogate, not the owner.
Eventual delivery Hints are delivered when the target node recovers. Delivery is asynchronous and not guaranteed within a specific timeframe.
No ordering guarantee Hints may be delivered out of order. The receiving node must handle this (e.g., using timestamps or vector clocks for conflict resolution).

Hint Storage and Delivery

Where Are Hints Stored?

  • On the coordinator node: The coordinator that receives the write request stores hints for unreachable replicas locally. Simple but puts extra load on the coordinator.
  • On the next node in the ring: In a hash ring topology, the node clockwise from the downed node stores hints. This is Cassandra's approach. The ring topology ensures a natural surrogate.
  • Dedicated hint storage: Some systems use a separate hint store (e.g., a local queue or directory) to isolate hint management from normal data operations.

How Are Hints Delivered?

  • Gossip-based detection: Nodes exchange membership information via gossip. When a hinting node detects the target is back (via gossip), it begins handoff.
  • Bulk delivery: Hints are replayed in batches to avoid overwhelming the recovering node.
  • Rate limiting: Handoff rate is throttled to prevent the recovering node from being overloaded with a flood of replayed writes.
Hint expiration: Hints have a TTL (typically 1-3 hours in Cassandra). If the target node does not recover within the TTL, hints are discarded. The data is recovered later via anti-entropy repair (Merkle tree comparison). This prevents hint storage from growing unbounded during prolonged outages.

Hints vs. Write-Ahead Log

Aspect Hints WAL / Commit Log
Purpose Store writes for a specific downed node Durability for the local node's writes
Scope Per-target-node, limited to missed writes All writes, node-wide
Delivery Sent to the specific node when it recovers Replayed during node recovery (crash recovery)
Ownership Data belongs to another node Data belongs to the local node

Use in Production Systems

  • Cassandra: Enabled by default. Each node can store hints for other nodes in the cluster. Hints stored in a separate directory (hints/) and replayed when the target node is detected via gossip. Configurable TTL (max_hint_window_in_ms, default 3 hours).
  • DynamoDB: Uses hinted handoff internally. Writes to unavailable replicas are buffered and delivered when the node recovers.
  • Riak: Supports hinted handoff as a configurable option. Hints stored on fallback nodes and replayed on recovery.
  • Voldemort: Uses hinted handoff as part of its Dynamo-inspired design.

Limitations

  • Hint overflow during long outages: If a node is down for hours, hint storage can grow large. Hint TTL prevents unbounded growth but means data is lost if the node is down too long.
  • Not a substitute for repair: Hints only capture writes that occurred while the node was down. They do not repair pre-existing inconsistencies. Anti-entropy repair is still needed.
  • Recovery bottleneck: When a node comes back, the handoff replay can overwhelm it. Rate limiting helps but extends recovery time.
  • Hints can be lost: If the hinting node itself fails before delivering hints, those writes are lost (for the target node). Anti-entropy repair recovers them later.

Common Interview Questions

  1. "How does Cassandra handle writes when a node is down?" — Coordinator writes to available replicas (satisfying W). For the downed node, the coordinator stores a hint on the next node in the ring. When the node recovers (detected via gossip), hints are replayed. If the node is down longer than the hint TTL, anti-entropy repair covers the gap.
  2. "What is hinted handoff and why is it useful?" — It temporarily stores writes for unavailable replicas, allowing the system to maintain write availability during node failures. Without it, the coordinator would have to wait for all replicas (slowing writes) or lose the write for the downed node.
  3. "What happens if hints are lost (hinting node crashes)?" — Those writes for the target node are not delivered. The gap is eventually filled by anti-entropy repair (Merkle tree comparison), which compares all replicas and syncs missing data.
  4. "How does hinted handoff differ from read repair?" &mdash> Hinted handoff is write-path: it ensures writes for downed nodes are eventually delivered. Read repair is read-path: it fixes inconsistencies discovered during reads. They are complementary mechanisms for eventual consistency.

What to Skip

Avoid Cassandra hint storage internals: The specific file format, directory structure, and disk layout of Cassandra hints are implementation details. Focus on the concept and behavior.

Avoid gossip protocol details: SWIM or gossip membership protocols are covered in separate membership topics. Mention gossip briefly but do not dive into protocol mechanics.

Avoid multi-DC hinted handoff: Cross-datacenter hinted handoff adds complexity (bandwidth, latency). Focus on single-DC hinted handoff unless multi-DC is discussed.
