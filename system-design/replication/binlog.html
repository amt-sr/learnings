Binlog (Binary Log)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the MySQL binary log as the foundational mechanism for replication and change data capture. Know the three replication formats, how followers use the binlog to replay changes, and the broader use cases including CDC and point-in-time recovery.

Definition

The binary log (binlog) is a sequential log file that records all data modification events in a MySQL server. Every INSERT, UPDATE, DELETE, DDL operation, and (depending on format) the actual SQL statements are written to the binlog. The binlog is the single source of truth for replication: followers connect to the leader, request binlog events from a specific position, and replay them locally to maintain an identical copy of the data.

Key idea: The binlog decouples write execution from replication. The leader writes to its storage engine and to the binlog. Followers read the binlog independently and apply changes asynchronously. This is the foundation of MySQL's leader-follower replication.

Replication Formats

Format What's Logged Pros Cons
STATEMENT Original SQL statement (e.g., UPDATE users SET age = 25 WHERE id = 7) Compact, human-readable, logs less data for batch operations Non-deterministic functions (NOW(), UUID()) produce different results on follower. Requires row-level locking for safety.
ROW Changed row data (e.g., before/after images of affected rows) Deterministic by design. Handles non-deterministic functions correctly. Safe for all operations. Verbose for bulk operations (e.g., DELETE FROM users WHERE active = 0 logs every deleted row).
MIXED Statement by default, switches to row for non-deterministic statements Best of both worlds: compact for safe statements, safe for non-deterministic ones Behavior can be surprising. Difficult to predict which format a given statement uses.
Production recommendation: ROW format is the default in MySQL 8.0 and is recommended for all production use. The determinism guarantee eliminates entire classes of replication bugs. Disk space for binlog is cheap; data inconsistency is not.

Binlog Position and GTID

Binlog Position (Classic)

Each binlog event is identified by a pair: (binlog file, offset). For example, mysql-bin.000142:4521 means position 4521 in file 142. Followers track their position and resume from there after reconnection.

GTID (Global Transaction Identifier)

Introduced in MySQL 5.6, GTIDs provide a globally unique identifier for every transaction: server_uuid:sequence_number (e.g., 3E11FA47-71CA-11E1-9E33-C80AA9429562:23).

  • Simplified failover: When promoting a follower to leader, the new leader already knows which GTIDs have been applied. No need to calculate binlog file:position offsets.
  • Multi-source replication: Each source server's GTIDs are tracked separately, making it easy to replicate from multiple sources.
  • Circular replication safety: GTIDs make it trivial to detect and skip transactions that originated from the current server.

How Followers Use the Binlog

  1. Connect: Follower opens a connection to the leader and sends a COM_BINLOG_DUMP (or GTID-based) request.
  2. Receive: Leader streams binlog events to the follower in order.
  3. Relay Log: Follower writes received events to its local relay log (a local copy of the leader's binlog).
  4. Apply: A separate SQL thread reads the relay log and applies each event to the follower's data files.
  5. Track Position: Follower records the last applied position/GTID. On restart, replication resumes from this point.
Two threads on the follower: The IO thread manages the network connection and writes to the relay log. The SQL thread reads the relay log and applies changes. This separation allows the follower to receive events faster than it can apply them (decoupling network I/O from apply throughput).

Use Cases Beyond Replication

Change Data Capture (CDC)

Tools like Debezium, Maxwell, and Canal read the binlog and publish row-level changes to Kafka or other streaming systems. This enables downstream consumers (analytics pipelines, search indexes, caches) to react to data changes in real time.

Point-in-Time Recovery (PITR)

By replaying binlog events from a full backup to a specific timestamp, you can restore the database to any point in time. This is critical for recovering from accidental data deletion or corruption.

Audit and Compliance

The binlog provides a complete, ordered record of all data changes. It can be used for forensic analysis, compliance auditing, and understanding what changed and when.

Binlog Lifecycle

  • Rotation: When a binlog file reaches max_binlog_size (default 1GB), MySQL creates a new file and increments the sequence number.
  • Purging: Old binlog files are automatically purged based on expire_logs_days (MySQL 5.7) or binlog_expire_logs_seconds (MySQL 8.0). Default is 30 days.
  • Retention: For CDC or PITR, you may want longer retention. Set expiration accordingly, but monitor disk usage.

Common Interview Questions

  • "What is the MySQL binlog?" Answer: A sequential log that records all data modification events. Followers connect to the leader, request binlog events, and replay them locally to maintain a replica.
  • "What are the three binlog formats?" Answer: STATEMENT (logs SQL), ROW (logs changed row data), MIXED (statement by default, row for non-deterministic statements). ROW is the recommended format for production.
  • "What is GTID and why is it important?" Answer: Global Transaction Identifier provides a unique ID for every transaction. It simplifies failover, enables multi-source replication tracking, and prevents circular replication loops.
  • "How would you build a real-time data pipeline from MySQL?" Answer: Use CDC tools like Debezium to read the binlog, publish change events to Kafka, and consume them in downstream services (search index updates, cache invalidation, analytics).
  • "What is the difference between binlog and redo log?" Answer: The redo log (InnoDB) is for crash recovery -- it ensures durability of committed transactions. The binlog is for replication and PITR -- it allows other servers to replay the same changes. Both are written during a commit (two-phase commit).

What to Skip

Avoid going deep into:
  • Binlog file format internals (magic number, checksum algorithm, event header byte layout). The interviewer wants the concept, not the binary specification.
  • mysqlbinlog command-line options and output format. Know that binlogs can be inspected with this tool; skip the flags.
  • Two-phase commit internals between InnoDB redo log and binlog. Mention the concept briefly (both must be consistent); skip the prepare/commit protocol details.
