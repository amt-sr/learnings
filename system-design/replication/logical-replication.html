Logical Replication

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand logical replication as a higher-level replication approach that copies data changes at the row/table level rather than the byte level. Know the publisher/subscriber model, CDC use cases, database version flexibility, and the limitations compared to physical replication.

Definition

Logical replication replicates data changes at the logical level -- individual row insertions, updates, and deletions are decoded from the source's write-ahead log and published as discrete change events. Unlike physical replication (which copies raw disk page modifications), logical replication works with the meaning of the data rather than its physical storage representation. This allows replication between different database versions, architectures, and even between different database systems.

Key idea: Physical replication copies bytes. Logical replication copies changes. Physical produces an exact disk-level clone. Logical produces a logically equivalent copy that may have different physical layout.

How It Works

  1. Write on publisher: A client performs INSERT, UPDATE, or DELETE on the publisher.
  2. WAL generation: PostgreSQL writes the change to the WAL (physical log) as usual.
  3. Logical decoding: A logical decoding plugin (e.g., pgoutput, wal2json, test_decoding) reads the WAL and converts physical changes into logical change events.
  4. Publication: The publisher makes logical change events available to subscribers. A publication defines which tables and which operations (INSERT, UPDATE, DELETE) are replicated.
  5. Subscription: The subscriber connects to the publisher, requests changes for the subscribed publications, and applies the row-level changes to its local tables.
  6. Apply: The subscriber applies each change event as a normal INSERT/UPDATE/DELETE on its local tables.

Advantages

  • Version flexibility: Can replicate between different PostgreSQL major versions (e.g., PG 14 to PG 16). Physical replication requires exact same major version.
  • Partial replication: Replicate specific tables or columns, not the entire database. Publications allow fine-grained control over what is replicated.
  • Cross-platform potential: Logical decoding outputs can be consumed by non-PostgreSQL systems (Kafka, Elasticsearch, other databases via CDC tools like Debezium).
  • Publisher/subscriber model: One publisher can serve multiple subscribers with different subscriptions. Each subscriber chooses which publications to consume.
  • Schema differences allowed: The subscriber can have a different physical layout (different indexes, different table structure) as long as the logical schema is compatible.
  • Read replicas with different workload: The subscriber can have different indexes optimized for analytics while the publisher has indexes optimized for OLTP.

Limitations

  • Slower than physical replication: Logical decoding adds overhead (parsing WAL, constructing row events). Physical replication copies bytes directly, which is faster.
  • No DDL replication: Logical replication replicates data changes (DML) but not schema changes (DDL). Schema changes must be applied manually on the subscriber.
  • No sequences replicated: Sequence values are not replicated. Auto-increment/serial columns can diverge between publisher and subscriber.
  • No unlogged tables: Unlogged tables are not written to WAL, so they cannot be logically replicated.
  • Trigger behavior: Triggers on the subscriber fire for replicated changes, which can cause unintended side effects unless handled explicitly.
  • Conflict resolution limited: If a subscriber has local writes on the same data, conflicts arise. Logical replication provides limited conflict detection compared to multi-master systems.
  • Large transactions: Very large bulk operations can overwhelm the logical replication stream.

CDC with Logical Replication

Logical decoding is the foundation of Change Data Capture (CDC) in PostgreSQL:

  • Debezium: Connects to PostgreSQL using the logical replication protocol, decodes WAL into change events, and publishes them to Kafka.
  • wal2json: Output plugin that converts WAL changes to JSON format, consumed by custom CDC pipelines.
  • pgoutput: Built-in plugin (PostgreSQL 10+) that produces logical replication protocol messages. Used by native logical replication and Debezium.
CDC vs Replication: Logical replication is a specific use case of CDC where the subscriber is another PostgreSQL database. CDC is the broader concept of capturing and streaming data changes to any consumer (Kafka, search indexes, caches, analytics).

Publication and Subscription Configuration

Publications

  • A publication defines a set of tables from which changes are replicated.
  • Can filter by operation type: INSERT only, UPDATE only, DELETE only, or all.
  • Multiple publications can exist on one database.

Subscriptions

  • A subscription connects to a publisher and requests changes from one or more publications.
  • The subscriber needs an initial snapshot of the published tables (can use existing data or copy from publisher).
  • Subscriptions are live -- changes are applied in real time as they arrive.

Common Interview Questions

  • "What is logical replication?" Answer: Replication at the row/table level where individual INSERT, UPDATE, DELETE events are decoded from the WAL and published to subscribers. It is database-version-agnostic and allows partial replication of specific tables.
  • "How does logical replication differ from physical replication?" Answer: Physical replication copies disk page bytes (exact clone, same version required). Logical replication copies row changes (logical equivalence, allows version differences and partial replication). Physical is faster; logical is more flexible.
  • "What is CDC and how does it relate to logical replication?" Answer: CDC (Change Data Capture) is the broader concept of streaming data changes to consumers. Logical replication is a specific implementation where the consumer is another database. Tools like Debezium use logical decoding to implement CDC pipelines.
  • "What are the limitations of logical replication?" Answer: No DDL replication (schema changes must be manual), no sequence replication, slower than physical replication, limited conflict resolution for multi-writer scenarios.
  • "When would you choose logical over physical replication?" Answer: When you need to replicate between different PostgreSQL versions, replicate only specific tables, or send changes to non-database consumers (Kafka, search indexes). Physical is better when you need an exact clone with maximum performance.

What to Skip

Avoid going deep into:
  • CREATE PUBLICATION and CREATE SUBSCRIPTION SQL syntax. Understand the concepts of publications and subscriptions; skip the exact DDL.
  • Logical decoding plugin implementation details (how pgoutput parses WAL internally). Know that plugins exist; skip how they work internally.
  • Replication slot internals for logical replication. Know the concept; skip the implementation.
