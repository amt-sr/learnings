Divergence

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand when and why replicas in a distributed system end up holding different data, how to detect divergence, and strategies for resolving it.

Definition

Replication divergence occurs when two replicas that should hold the same data end up with different versions. Unlike replica lag (where the follower is simply behind but will eventually catch up), divergence implies the replicas have taken different paths and cannot converge without explicit conflict resolution.

Causes of Divergence

1. Split Brain (Concurrent Leaders)

The most dangerous cause. After a network partition, if two nodes both act as leader, they independently accept writes for the same keys. When the partition heals, their states are irreconcilably different.

2. Multi-Leader Conflicts

In multi-leader or active-active setups, two leaders may receive writes to the same key concurrently. Without conflict resolution, each leader applies its own write and ignores the other's, causing permanent divergence.

3. Asynchronous Replication + Leader Failover

If a leader fails and a follower is promoted before receiving all writes, the old leader's unreplicated writes are lost. If the old leader comes back online (e.g., after a brief network glitch), it has data that the new leader does not. Reconciling this requires manual intervention.

Timeline showing divergence from async failover: t0 Leader: [w1, w2, w3, w4] Follower: [w1, w2, w3] (w4 not yet replicated) t1 Leader CRASHES Follower promoted to new Leader: [w1, w2, w3] New writes: [w5, w6] New Leader: [w1, w2, w3, w5, w6] t2 Old Leader recovers: [w1, w2, w3, w4] DIVERGENCE: New Leader has: [w1, w2, w3, w5, w6] (missing w4) Old Leader has: [w1, w2, w3, w4] (missing w5, w6) t3 w4 must be manually reconciled. w5 and w6 may have overwritten data that w4 also modified.

4. Write Skew

Two transactions that are individually consistent but globally inconsistent. For example, two users concurrently editing different fields of the same row, each reading stale versions of the other's field.

Detection

Method How It Works When Used
Checksum comparison Periodically compute checksums of data ranges across replicas. Mismatch indicates divergence. Batch reconciliation, anti-entropy
Version vectors Each replica maintains a vector of version numbers per writer. Compare vectors to detect missing updates. Real-time, multi-leader systems
Logical timestamps + CRC MySQL GTIDs, PostgreSQL LSNs — compare the set of applied transaction IDs across replicas. Single-leader systems
Application-level audits Compare replica data at the application layer (e.g., nightly reconciliation job). Post-incident investigation

Resolution Strategies

Last-Write-Wins (LWW)

Attach a timestamp to each write. When replicas diverge, keep the write with the latest timestamp. Simple but loses data: the earlier write is silently discarded.

Clock skew problem: LWW relies on synchronized clocks. If clocks differ by even a few milliseconds, the "later" write may actually be the earlier one. Google Spanner uses TrueTime (GPS + atomic clocks) to address this, but most systems cannot.

Conflict-Free Replicated Data Types (CRDTs)

Data structures mathematically designed to merge automatically without conflicts. Different replicas can apply updates in any order, and the merge function always produces the same result.

CRDT Type Behavior Example Use Case
G-Counter (Grow-only Counter) Each node has a local counter. Total = max of all nodes. Like counts, page views
PN-Counter Two G-Counters (one increment, one decrement). Net = increment - decrement. Balances, inventory
G-Set (Grow-only Set) Add-only. Union of all replicas. Unique visitors, registered emails
OR-Set (Observed-Remove Set) Supports add and remove. Uses unique tags per element. Shopping cart, user groups
LWW-Register Last-write-wins register with causal timestamps. User profile fields
Production use: Riak uses CRDTs for its conflict-free data types (flags, counters, sets, maps, registers). Apple uses CRDTs in CloudKit for Notes, Reminders, and shared documents across devices.

Application-Level Conflict Resolution

Present both versions to the application or user for manual resolution. Used when automatic merge is not possible or when data semantics are too complex.

  • Custom merge function: Application provides a merge handler that combines divergent writes based on business logic (e.g., merge contact information by taking the most recently updated field).
  • Manual resolution: Flag conflicting records for human review. Common in financial systems where automatic merge is unacceptable.

Anti-Entropy (Merkle Tree Comparison)

Replicas exchange Merkle trees (hash trees) of their data. Differences are pinpointed to specific leaf nodes, and only divergent keys are synchronized. Used by Cassandra and DynamoDB for background reconciliation.

Common Interview Questions

  1. "Two replicas have diverged. How do you detect and resolve this?" — Detection: checksum comparison, version vectors, or GTID comparison. Resolution: depends on data type. LWW for simple cases, CRDTs for counters/sets, application-level merge for complex data.
  2. "What is a CRDT and when would you use one?" — A data structure that guarantees convergence without coordination. Use for counters, sets, registers in multi-leader or offline-first systems. Mention OR-Set for shopping carts, PN-Counter for balances.
  3. "Why is last-write-wins problematic?" — Clock skew causes misordering. Data is silently lost. Not suitable for operations where every write matters (e.g., financial transactions, medical records).
  4. "How does Cassandra detect and repair divergence?" — Cassandra uses Merkle trees during anti-entropy repair (nodetool repair). Each node builds a hash tree of its data, exchanges with peers, and only syncs the divergent ranges.

What to Skip

Avoid CRDT mathematical proofs: Knowing the algebraic properties (commutativity, associativity, idempotence) is useful, but formal proofs are not needed for interviews.

Avoid Operational Transformation: OT (used in Google Docs) is related to conflict resolution but is a different domain (collaborative editing). Only mention it if directly comparing CRDTs vs. OT.

Avoid Byzantine fault tolerance: Divergence here assumes honest nodes. Byzantine behavior (malicious nodes forging data) is a separate research area.
