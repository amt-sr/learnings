4.3.1 Synchronous Replication

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the strongest form of replication guarantee: the leader waits for followers to acknowledge writes before committing. Know the zero-data-loss guarantee, the latency cost (writes are as slow as the slowest follower), and when to choose synchronous replication over its alternatives.

Definition

Synchronous replication means the leader does not report a write as successful (does not commit) until at least one follower has confirmed that it has received and persisted the write. The client's write request blocks until this acknowledgment is received.

This is the replication type with the strongest data durability guarantee: if the leader crashes, no committed data is lost because the data is already safely on at least one follower.

How It Works

Receive write request │ │ 2. Write to WAL locally │ │ 3. Send write to FOLLOWER (sync wait) │ │ 4. ◀── WAIT for follower ACK ── │ │ 5. Commit write │ │ 6. Send ACK to client │ │ │ └──────────────┬───────────────────────────┘ │ (step 3) │ ┌──────────────▼───────────────────────────┐ │ FOLLOWER │ │ │ │ 1. Receive write from leader │ │ 2. Write to WAL │ │ 3. Send ACK to leader ─────────────────┘ │ │ └──────────────────────────────────────────┘ Total write latency = Leader local write + Network RTT + Follower local write Client is blocked during steps 3-4 (sync wait)
Key guarantee: The write is on at least TWO machines (leader + follower) before the client gets an ACK. If the leader crashes after step 5 but before step 6, the client may timeout and retry, but no data is lost -- the follower has the committed data.

Performance Impact

The critical performance characteristic of synchronous replication is that write latency equals the slowest follower in the synchronous set.

Latency Breakdown (Synchronous Replication) ──────────────────────────────────────────── t=0ms Client sends write to leader t=2ms Leader writes to local WAL (disk I/O) t=3ms Leader sends write to follower (network send) t=6ms Follower receives write (network + 3ms RTT) t=8ms Follower writes to WAL (disk I/O) t=9ms Follower sends ACK (network send) t=12ms Leader receives ACK (network + 3ms RTT) t=12ms Leader commits and ACKs client Total latency: ~12ms Compare with asynchronous replication: t=0ms Client sends write t=2ms Leader writes to WAL t=2ms Leader ACKs client immediately (no wait) Total latency: ~2ms SYNCHRONOUS is ~6x SLOWER in this example.

The latency multiplier depends on:

FactorImpactMitigation
Network RTT to followerDominant factor; each additional millisecond of RTT adds to write latencyCo-locate followers in the same rack/DC; dedicated network
Follower disk speedSlow follower disk means slow ACKUse SSDs for all replicas; ensure similar hardware
Number of sync followersMore sync followers = wait for more ACKs = higher latencyKeep the number of required sync followers minimal (typically 1-2)
Follower loadIf follower is under heavy read load, ACKs may be delayedDedicated replica for sync (not serving reads)

When to Use Synchronous Replication

Interview Tip: The answer to "when should I use synchronous replication?" is always: "When data loss is unacceptable." Be specific about what kinds of data fall into this category.
  • Financial transactions: You cannot tell a customer "we lost your payment due to a server crash." Every committed write must be durable.
  • Account credentials and auth tokens: Lost authentication data means users are locked out of their accounts.
  • System metadata: Schema changes, configuration updates, service discovery data. Inconsistency here can crash the entire system.
  • Regulatory requirements: Financial regulations (SOX, PCI-DSS) may mandate zero data loss for certain data types.
  • Order-sensitive operations: Where the ordering and completeness of writes is critical and cannot be reconstructed.

Common Interview Questions

  1. "Explain synchronous replication."
    Answer: "In synchronous replication, the leader waits for one or more followers to confirm they have received and persisted a write before acknowledging it to the client. This means every committed write exists on at least two machines. The tradeoff is write latency: the leader is blocked until the follower responds, so total latency includes the round trip to the follower plus the follower's write time. The main benefit is zero data loss -- if the leader crashes, no committed data is lost."
  2. "What is the performance impact of synchronous replication?"
    Answer: "Write latency roughly doubles (or more) compared to asynchronous replication because the leader must wait for a network round trip to the follower and the follower's disk write. If the follower is slow or the network has high latency, the impact is worse. However, the benefit is that read-after-write consistency is guaranteed and no data can be lost on leader failure. The throughput impact is also significant because writes are serialized at the sync point -- the leader cannot pipeline writes as aggressively."
  3. "When would you choose synchronous over asynchronous replication?"
    Answer: "I choose synchronous replication when data loss is unacceptable: financial transactions, authentication data, system metadata, and any data covered by regulatory requirements. For most other data types -- user activity logs, analytics events, social feeds -- asynchronous replication is acceptable because occasional data loss on leader crash is tolerable. In practice, many systems use a hybrid: synchronous for critical data paths and asynchronous for the rest."

What to Skip

Do not go deep into:
  • PostgreSQL synchronous_standby_names configuration syntax and options.
  • Disk I/O scheduling and WAL flush mechanisms at the OS level (fsync, O_DIRECT).
  • Network transport optimization (TCP vs. RDMA for replication traffic).
  • Multi-sync-follower topologies (daisy-chaining synchronous replicas).
  • Specific storage engine internals for how writes are flushed to disk.
