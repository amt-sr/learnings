Split Brain

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand the split brain problem — one of the most dangerous failure modes in distributed systems — how it occurs during network partitions, its consequences, and the mechanisms used to prevent it.

Definition

Split brain occurs when two (or more) nodes in a distributed system both believe they are the leader simultaneously. After a network partition, each side of the partition can independently decide to promote a leader, resulting in two leaders that both accept writes. Since they cannot communicate, their data diverges — creating irreconcilable conflicts.

Why interviewers love this topic: Split brain is one of the few problems that can cause silent data corruption in production without any errors being returned to clients. Both leaders respond to writes successfully, but the data is inconsistent and may be impossible to merge.

How Split Brain Happens

NORMAL OPERATION: [Leader A] <--- heartbeat ---> [Follower B] | | Clients in AZ-1 Clients in AZ-2 All reads/writes -> A All reads -> B NETWORK PARTITION: [Leader A] <===X===X===X===> [Follower B] | | | "B is not responding!" | "A is not responding!" | Keep serving as leader | v v A stays leader (still B's failover timer expires has heartbeat timeout) B promotes itself to leader | | Clients in AZ-1 Clients in AZ-2 Write: balance=200 Write: balance=150 | | A's state: balance=200 B's state: balance=150 | | | SPLIT BRAIN: | | Two leaders, two truths, | | data has diverged! |

Consequences of Split Brain

Impact Description Severity
Data divergence Each leader writes different data for the same keys. When the partition heals, there is no automatic way to merge. Critical
Write conflicts Auto-increment IDs collide, unique constraints violate, referential integrity breaks. Critical
Lost writes When partition heals, one side must be demoted. All writes accepted by the demoted leader may be discarded. Critical
Split brain undetected Both sides return success to clients. No errors, no alerts. The problem is discovered later during reconciliation. Critical
Cascading failures Diverged data causes application errors downstream (order processing, inventory, billing). Critical

Prevention Mechanisms

1. Quorum-Based Leader Election

Require a majority (quorum) of nodes to agree before a node becomes leader. With 5 nodes, quorum is 3. During a network partition, at most one side can achieve quorum.

5-node cluster, quorum = 3 Partition splits into 3 + 2: Side A: [Node1, Node2, Node3] -> quorum=3 -> CAN elect leader Side B: [Node4, Node5] -> quorum=2 -> CANNOT elect leader (2 < 3) Result: SAFE. Only one leader. Partition splits into 2 + 2 + 1: Side A: [Node1, Node2] -> quorum=2 -> CANNOT elect leader (2 < 3) Side B: [Node3, Node4] -> quorum=2 -> CANNOT elect leader (2 < 3) Side C: [Node5] -> quorum=1 -> CANNOT elect leader (1 < 3) Result: SAFE. No leader (system is unavailable, but consistent).
Raft and Paxos use quorum-based election by design. A leader needs votes from a majority of nodes. During a network partition, only the majority partition can elect a leader. This is why odd-numbered clusters (3, 5, 7) are preferred.

2. Fencing Tokens

Assign monotonically increasing tokens to leaders. Every write must carry the current fencing token. Storage layers reject writes with tokens lower than the highest they have seen.

Coordinator assigns fencing tokens: t=0: Leader A gets fencing_token=1 A writes to disk: { key="x", value=10, token=1 } t=1: Network partition occurs B promotes itself, gets fencing_token=2 t=2: B writes: { key="x", value=20, token=2 } t=3: Network heals A (still thinks it's leader) writes: { key="x", value=30, token=1 } Disk rejects: "token=1 < last_seen_token=2, REJECTED" B wins. A's stale write is fenced out.
Important: Fencing tokens must be checked at the storage layer, not just in memory. A stale leader that has been isolated can still write to its local disk. The disk (or a shared storage like S3) must enforce the token check.

3. Lease and Heartbeat Timeouts

The leader holds a time-bound lease. If it cannot renew its lease (heartbeat from majority of nodes), it steps down. The lease is granted by a consensus system (etcd, ZooKeeper, Chubby).

  • Lease duration: Must be longer than the expected heartbeat round-trip time + jitter.
  • Lease expiry: If leader cannot contact a majority within the lease period, it voluntarily steps down.
  • New election: Only after the old lease expires can a new leader be elected.

4. Generation Numbers (Epochs)

Each leader election increments a generation number (epoch). Write operations include the epoch. Nodes reject writes from a previous epoch, effectively fencing out old leaders. This is the mechanism used by Raft (term numbers) and ZooKeeper (zxid epochs).

Real-World Examples

Incident Year What Happened
GitHub MySQL split brain 2017 A network event caused GitHub's active-passive MySQL pair to both become active. The passive node accepted writes that conflicted with the active. Required manual reconciliation.
Stack Overflow outage 2018 Network partition between primary and secondary data centers. Both sides promoted leaders. Data diverged across multiple database clusters.
AWS us-east-1 Kinesis 2020 Control plane partition caused multiple stream consumers to believe they held the lease. Duplicate processing and reordering occurred.

Common Interview Questions

  1. "What is split brain and how do you prevent it?" — Two nodes both think they're the leader after network partition. Prevention: quorum-based election, fencing tokens, lease/heartbeat timeouts, generation numbers. Explain why odd-numbered clusters matter.
  2. "Design a leader election system that avoids split brain." — Use Raft or a lease-based system with etcd. Leader must get votes from a majority. Fencing tokens prevent stale writes. Discuss the tradeoff between availability and consistency during partition.
  3. "What happens if you use a simple heartbeat-based failover without quorum?" — Split brain. Both sides can decide the other is dead and promote themselves. A 2-node cluster is fundamentally unsafe for split-brain prevention.
  4. "How does ZooKeeper prevent split brain?" — ZK uses ZAB (ZooKeeper Atomic Broadcast), which is similar to Paxos. Leaders must get a quorum of followers to acknowledge their leadership. zxid (transaction ID) includes an epoch that prevents old leaders from committing writes.

What to Skip

Avoid Byzantine fault tolerance: BFT (handling malicious nodes) is a different problem. Split brain assumes nodes are well-behaved but disconnected.

Avoid Raft log replication details: The log replication mechanism is important but separate. Focus on the leader election and fencing aspects.

Avoid STONITH details: "Shoot the other node in the head" (IPMI power control) is a hardware-level fencing technique. Mention it briefly but do not go into IPMI or fencing agents.
