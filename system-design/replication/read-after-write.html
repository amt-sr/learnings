Read After Write Consistency

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand how to guarantee that a user reads their own writes in an asynchronously replicated system — the most common consistency requirement in real-world applications.

Definition

Read-after-write consistency (also called read-your-writes consistency) guarantees that if a user writes data, any subsequent read by the same user will return that data or a newer version. This is a session-level guarantee: it only applies to the writing user, not to other users who may still see stale data.

The Problem

THE PROBLEM: User writes to Leader ---> Leader commits at t=0 | User immediately reads from Follower (routed by LB) | Follower has not yet replicated t=0 write | Returns STALE data (pre-write state) | User: "My update didn't save!" (it did save, they just can't see it) THE GOAL: User writes to Leader ---> Leader commits at t=0, returns version=42 | User reads (carries version=42 in session) | System ensures: reading replica must have version >= 42 | If replica is at version 40: route to Leader instead If replica is at version 42: safe to read from replica | Returns FRESH data (version >= 42)

Solutions

Solution 1: Read Own Writes from the Leader

The simplest approach: any data the user has recently written is always read from the leader. The application tracks which objects the user modified within a time window and routes reads for those objects to the leader.

Implementation: After a successful write, the application stores the affected resource key (e.g., user:1234:profile) and a timestamp in the user's session cache (Redis). On subsequent reads, if the key matches a recent write in the session, route to the leader. Evict after the replica lag window (e.g., 10 seconds).
Client App Server Leader Follower | | | | |-- PUT /profile ---------->| | | | |-- WRITE profile:1234 ------>| | | |<- OK (version=42) ----------| | | | | | | |-- SET session:user1234 ---->| | | | key=profile:1234 | | | | version=42 | | | | expires_at=now+10s | | |<- 200 OK -----------------| | | | | | | |-- GET /profile ---------->| | | | |-- CHECK session:user1234 | | | | profile:1234 present! | | | | --> READ from LEADER | | | |-- READ profile:1234 ------>| | | |<- { name: "Alice" } --------| | |<- 200 { name: "Alice" }--| | |

Tradeoff: This increases leader read load. At scale, only a small percentage of reads hit the leader (typically 1-5% of total reads are for a user's own recently-written data).

Solution 2: Track Write Timestamp, Wait for Follower

The leader returns a logical timestamp or LSN with each write response. The client includes this timestamp in subsequent reads. The replica checks whether it has applied all changes up to that timestamp. If not, it waits or the request is retried on another replica.

Write Phase: Client -> Leader: PUT /post/1001 Leader -> Client: 201 Created, { "version": "v7", "lsn": 5000 } Read Phase: Client -> Replica: GET /post/1001, Header: X-Read-Version: v7 Replica: internal LSN = 4998 (behind v7) Replica -> Client: 409 Conflict, Header: X-Current-Version: v6 Client -> Replica-B: GET /post/1001, Header: X-Read-Version: v7 Replica-B: internal LSN = 5002 (ahead of v7) Replica-B -> Client: 200 OK, { ... }

Solution 3: Client-Side Routing with Monotonic Reads

Maintain session consistency: once a user reads from a specific replica, all subsequent reads in that session go to the same replica. This prevents the user from going "backwards in time" by reading from a less-updated replica.

Combine with a version watermark: the client tracks the highest version it has seen. The proxy ensures the replica's applied version is at least the client's watermark before serving.

Solution 4: Hybrid — Critical Path on Leader, Non-Critical on Replicas

Not all reads need read-after-write. Classify reads by sensitivity:

Read Type Routing Example
Own profile, own posts, own settings Leader Profile page after edit
Other users' content Any replica Viewing someone else's profile
Financial balances, transaction history Leader Bank account page
Search results, feeds, analytics Any replica Timeline, search

Production Considerations

Leader read amplification: At large scale, routing all "own writes" reads to the leader can overwhelm it. Mitigations include: (1) short TTL on the read-from-leader flag (5-10s after write), (2) dedicated read replicas for leader reads that use synchronous replication, (3) caching hot user data in Redis after writes.
Instagram's approach: For the first N seconds after a user writes, reads for that user's data go to the leader. After the window expires, reads go to replicas. The window duration is based on observed P99 replica lag.

Common Interview Questions

  1. "How do you ensure read-after-write consistency in a social media app?" — Track writes in session cache with TTL. Route reads for recently-written keys to the leader. Evict after replica lag window. Mention that only own writes need this guarantee.
  2. "What's the cost of always reading from the leader?" — Leader becomes a bottleneck. Horizontal read scaling is lost. Only use leader reads for data the user recently wrote, not for all reads.
  3. "How does LinkedIn handle read-after-write?" — LinkedIn uses a similar approach where the follower tracks its replication progress. The application server reads the follower's progress, compares against the write timestamp, and decides whether to serve from the follower or redirect to the leader.
  4. "Can you guarantee read-after-write without reading from the leader?" — Yes, using timestamp-based version checks. The follower must have caught up to the write's LSN. This requires tracking follower progress and potentially waiting or retrying.

What to Skip

Avoid full causal consistency theory: Causal consistency is a broader concept. Focus on the practical read-after-write pattern unless asked about causal consistency specifically.

Avoid implementing vector clocks from scratch: Knowing the concept is sufficient. Interviewers want architecture decisions, not algorithm implementation.

Avoid monotonic read guarantees: While related (preventing reading older data after seeing newer data), it is a separate concept. Mention it briefly if relevant but do not dive deep.
