Replica Lag

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand what replica lag is, why it occurs in asynchronously replicated systems, how to measure it, and how to mitigate it in production.

Definition

Replica lag is the delay between a write being committed on the leader (primary) and that write being applied on a follower (replica). In asynchronous replication, followers pull changes from the leader at their own pace. The time gap between the leader's write position and the follower's applied position is the replica lag, typically measured in seconds or in bytes of WAL/redo log still pending.

How Replica Lag Manifests

Client Leader (Primary) Follower A Follower B | | | | |-- WRITE(id=42) ->| | | | |-- binlog entry | | | OK (42) |-- for id=42 | | |<- OK ------------| [lag ~2s] | | | | | | | |<------- PULL --------| | | | |-- send binlog ------>| | | | | |-- apply id=42 | | | | [lag ~4s] | | |<------- PULL --------------------------| | | |-- send binlog ----------------------->| | | | |-- apply id=42 | | | [lag ~6s]

In this diagram, the leader acknowledges the write immediately to the client. Follower A applies it roughly 2 seconds later, and Follower B roughly 6 seconds later. Any read routed to Follower B during that window returns stale data.

Root Causes of Replica Lag

1. Slow or Congested Network

Followers in a remote data center or availability zone experience higher round-trip latency when pulling binlog entries. Cross-region replication can add hundreds of milliseconds per round trip, compounding during burst writes.

2. Heavy Write Load on the Leader

The leader generates WAL/binlog entries faster than followers can consume them. A bulk import, schema change, or traffic spike can cause the replication queue to grow faster than it drains.

3. Disk I/O Bottleneck on the Follower

Followers must both fetch and persist changes. If a follower's disk is slow (e.g., HDD instead of NVMe, or competing workloads), applying writes falls behind.

4. Long-Running Transactions on the Follower

In statement-based replication, a slow query on the follower can block the replication thread from applying subsequent entries. Row-based replication mitigates this but does not eliminate it entirely.

5. Resource Contention

Followers may serve read queries concurrently with replication. A heavy analytical query can consume CPU or I/O, starving the replication thread.

Monitoring Replica Lag

System Metric / Command Notes
MySQL SHOW SLAVE STATUS \G — Seconds_Behind_Master Can be misleading; shows time since last applied event, not true lag if replication thread is idle.
PostgreSQL pg_stat_replication view — write_lag, flush_lag, replay_lag More granular: measures lag at WAL write, flush, and replay stages.
Cassandra nodetool netstats — pending queue depth Watch the DROPped messages counter for back-pressure signals.
MongoDB rs.printSlaveReplicationInfo() — optimeDate lag Compares each secondary's optime against the primary.
Watchdog pattern: At L5+/Staff level, you should discuss alerting thresholds. A common production setup is a warning at 5s lag and a page at 30s lag. Some teams use multi-tier thresholds: P4 at 1s (potential stale reads), P3 at 5s, P2 at 30s, P1 at 120s.

How Lag Affects Consistency

  • Eventual consistency window: Reads from a lagging follower return stale data. The system is only eventually consistent within the lag duration.
  • Read-after-write violation: A user writes a post, then reads from a follower — sees nothing. The user may retry or think the write failed.
  • Constraint violations on failover: If the leader fails and a lagging follower is promoted, committed writes on the old leader that the follower never received are lost.
  • Cascading failures: Lagging followers drain connection pools as clients wait for results. If a follower falls far behind, some systems remove it from the pool entirely, shifting more load to healthy replicas.

Remedies and Mitigations

Parallel Replication (Multi-Threaded Apply)

Modern MySQL and PostgreSQL support parallel apply of replication events on the follower. Instead of a single SQL thread applying changes serially, multiple worker threads apply changes in parallel, grouped by database or partition.

MySQL: slave_parallel_workers = 8 with slave_parallel_type = LOGICAL_CLOCK enables commit-order parallel replication. This can reduce lag by 4-8x on multi-core followers.

Connection Pooling and Read Routing

ProxySQL, PgBouncer, or application-level middleware can route reads away from lagging replicas. A common pattern: if Seconds_Behind_Master > threshold, temporarily remove the follower from the read pool.

Semi-Synchronous Replication

For critical data paths, use semi-synchronous replication where the leader waits for at least one follower to acknowledge receipt (not apply) before committing. This trades write latency for reduced data loss on failover.

Tuning the Replication Pipeline

  • Increase binlog_group_commit batch size to amortize fsync costs.
  • Use compressed replication traffic between data centers.
  • Dedicate a replication thread CPU core on the follower (CPU pinning).
  • Use faster storage (NVMe) for WAL and data directories on followers.

Common Interview Questions

  1. "You have a MySQL primary with 3 replicas. A traffic spike causes 30s replica lag. Walk me through your response." — Discuss immediate triage (remove lagging replicas from read pool, alert on-call), root cause analysis (check binlog rate, follower disk I/O, network), and mitigation (parallel replication, semi-sync for critical paths, rate limiting).
  2. "How do you design a system that can tolerate replica lag without users noticing?" — Read-after-write consistency for own writes, use Redis/leader for hot data, stale-while-revalidate pattern, background reconciliation.
  3. "What happens during failover if a replica is 10 seconds behind?"strong> — Committed writes in the last 10s are lost (durability gap). Discuss RPO and whether semi-sync is acceptable for this workload.
  4. "How does Google Spanner avoid replica lag entirely?" — TrueTime API provides synchronized clocks, allowing externally consistent transactions. Paxos-based replication commits only when a quorum acknowledges. It does not eliminate latency but eliminates inconsistency.

What to Skip

Avoid deep-dive into binlog formats: Statement-based vs. row-based vs. mixed replication internals are rarely asked. Know the difference at a high level but do not prepare binlog byte-format details.

Avoid Galera/PXC internals: Multi-master synchronous replication specifics are niche unless the role specifically targets database infrastructure.

Avoid replication topology details: Cascading replication (A -> B -> C) and circular replication are rarely tested. Focus on single-leader topology.
