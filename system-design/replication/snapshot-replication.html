Snapshot Replication

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand snapshot replication as a bulk data transfer mechanism for initial synchronization and periodic backups. Recognize why snapshots alone are insufficient for ongoing replication and how they combine with incremental change streams (WAL, binlog) for a complete replication strategy.

Definition

Snapshot replication copies the entire database (or a subset of tables) from a source to a target as a complete, point-in-time image. The target receives a full copy of all data at a specific moment. This is fundamentally different from streaming replication -- instead of continuously sending incremental changes, snapshot replication transfers the entire dataset in one bulk operation.

Key idea: Snapshot replication is the "seed" step. You use it to bootstrap a new replica, then switch to WAL/binlog streaming for ongoing incremental replication. It is not a replacement for continuous replication -- it is the starting point.

Snapshot Process

  1. Consistent snapshot: The source takes a consistent point-in-time snapshot (using FLUSH TABLES WITH READ LOCK in MySQL or pg_start_backup() in PostgreSQL). This ensures all data is from the same moment.
  2. Copy data: The full dataset is transferred. Methods vary by database:
    • Logical: mysqldump, pg_dump -- outputs SQL statements. Portable but slow for large databases.
    • Physical: pg_basebackup, Percona XtraBackup, RDS snapshots -- copies data files directly. Fast but not portable across versions.
  3. Record position: The binlog position (MySQL) or WAL LSN (PostgreSQL) at the time of the snapshot is recorded.
  4. Apply to target: The snapshot is loaded onto the follower.
  5. Start streaming: The follower begins streaming replication from the recorded position, picking up all changes that occurred during and after the snapshot.

Advantages

  • Simple and reliable: Full copy is conceptually straightforward. No incremental replay logic needed -- just copy files or dump SQL.
  • Clean starting point: The follower has a complete, consistent copy of the leader's data at a known point in time.
  • Version-agnostic (logical): Logical snapshots (SQL dumps) can be loaded into different database versions or even different database engines.
  • No lag accumulation: Unlike streaming replication, there is no concept of "falling behind." The snapshot is either complete or not.
  • Essential for disaster recovery: Regular snapshots provide recovery points. Combined with WAL/binlog, they enable point-in-time recovery.

Disadvantages

  • Expensive for large databases: Copying a 1TB database takes significant time and I/O bandwidth. The leader may be locked or I/O-bounded during the copy.
  • Not suitable for frequent replication: Taking a full snapshot every minute is impractical. Snapshots are periodic (daily, weekly), not continuous.
  • Stale data between snapshots: If using snapshots alone (without streaming), followers only receive updates at snapshot intervals. Data can be hours or days stale.
  • Resource-intensive: Full copies consume network bandwidth, disk I/O, and CPU on both source and target. Production workloads may be impacted.
  • Storage overhead: Each snapshot consumes full disk space on the target (or in backup storage).

Snapshot + Streaming: The Complete Replication Strategy

Production replication almost always combines both approaches:

  1. Initial bootstrap: Take a snapshot and copy it to the new follower. Record the replication position at snapshot time.
  2. Streaming replication: The follower starts streaming from the recorded position. All changes that occurred during the snapshot copy are applied.
  3. Periodic snapshots: Regular snapshots are taken for backup and disaster recovery. If a follower falls too far behind to catch up via streaming, a fresh snapshot is taken.
Design pattern: Snapshot = baseline. Streaming = delta. You always need both. Snapshot gets you to a starting point; streaming keeps you current. This is true for MySQL (mysqldump + binlog), PostgreSQL (pg_basebackup + WAL), and most other databases.

When Followers Fall Behind

If a follower falls too far behind (leader recycled WAL/binlog segments the follower needs), the follower cannot continue streaming and must re-sync:

  1. Stop the follower.
  2. Take a new snapshot from the leader.
  3. Copy the snapshot to the follower.
  4. Start streaming from the new snapshot's position.
Interview Tip: When asked "how do you add a new replica to an existing system?", the answer is always snapshot first (initial sync), then streaming (ongoing sync). Mention both phases and explain why snapshots alone are insufficient.

Common Interview Questions

  • "How would you create a new read replica for a 500GB MySQL database?" Answer: Take a consistent snapshot (e.g., Percona XtraBackup for minimal locking), record the binlog position, copy to the replica, configure replication starting from that binlog position, and let streaming replication handle ongoing changes.
  • "Why can't you use snapshots alone for replication?" Answer: Snapshots are expensive, infrequent, and produce stale data. For real-time replication, you need streaming (WAL/binlog) to propagate changes continuously between snapshots.
  • "What happens when a replica falls too far behind?" Answer: The leader may have recycled the WAL/binlog segments the replica needs. The replica must re-sync from a fresh snapshot.
  • "What is the difference between logical and physical snapshots?" Answer: Logical snapshots produce SQL statements (portable, slower). Physical snapshots copy raw data files (fast, not portable across versions). Physical is preferred for same-version replicas; logical for migrations.

What to Skip

Avoid going deep into:
  • mysqldump command-line flags and options. Mention --single-transaction as the key InnoDB flag; skip the rest.
  • Backup compression algorithms and encryption settings. The concept of taking backups is important; the implementation details are not.
  • Cloud-specific snapshot APIs (AWS CreateDBSnapshot, GCP snapshots). Mention cloud snapshots conceptually; skip the API calls.
