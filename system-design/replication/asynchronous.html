4.3.2 Asynchronous Replication

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the most commonly used replication mode in production systems. Know how asynchronous replication works (leader does not wait for followers), why it is the default for most databases, the data loss risk on leader failure, and the replica lag implications that follow.

Definition

Asynchronous replication means the leader writes data locally and immediately acknowledges the client's write request, without waiting for followers to receive or persist the data. The leader sends replication data to followers in the background, at its own pace. There is no guarantee that followers are up-to-date.

This is the default replication mode for most database systems because it provides the best write performance. The tradeoff is that followers may lag behind the leader, and if the leader fails, any unreplicated writes are permanently lost.

Replica Lag and Its Consequences

The defining characteristic of asynchronous replication is replica lag: the gap between the leader's state and a follower's state at any point in time.

Replica Lag Timeline ──────────────────── Leader WAL: [w1][w2][w3][w4][w5][w6][w7][w8] │ │ │ │ │ │ │ │ Time (ms): 0 5 10 15 20 25 30 35 Follower A: [w1][w2][w3][w4][w5][w6][w7][w8] (lag: 0ms) apply delay: 5ms, fully caught up Follower B: [w1][w2][w3][w4][w5] (lag: 20ms) │ currently at w5 leader is at w8 At t=35ms: Leader has: w1, w2, w3, w4, w5, w6, w7, w8 Follower A has: w1, w2, w3, w4, w5, w6, w7, w8 (up-to-date) Follower B has: w1, w2, w3, w4, w5 (20ms behind)

Interview Tip: When designing with async replication, always proactively discuss replica lag as a design concern. Interviewers expect you to identify this risk and propose mitigations (read from leader after writes, session affinity, track replication position, use causal consistency).

Data Loss on Leader Failure

The most serious risk of asynchronous replication is permanent data loss when the leader fails.

Data Loss Scenario ────────────────── Leader WAL: [w1][w2][w3][w4][w5] 💥 CRASH │ │ │ │ │ Follower: [w1][w2][w3] │ └── Follower received w1-w3 before crash w4 and w5 were ONLY on the leader After failover: Follower becomes new leader w4 and w5 are PERMANENTLY LOST Client thought these writes succeeded (got ACKs) But data was never replicated
Data Loss Window: The amount of data at risk is equal to the replication lag at the time of failure. If the follower was 100ms behind, you lose up to 100ms of writes. If it was 10 seconds behind, you lose up to 10 seconds of writes. This window is unbounded in pathological cases (follower completely disconnected).

Why Asynchronous Replication Is the Default

Despite the data loss risk, asynchronous replication is the default for most databases for practical reasons:

  • Write performance: Writes complete in the time it takes to write to local disk (~1-5ms). No network round trip to followers.
  • Write throughput: The leader can pipeline writes aggressively without waiting for sync ACKs.
  • Geographic flexibility: Followers can be in remote data centers without adding latency to every write.
  • Availability: Writes succeed even when followers are temporarily down or slow.
  • Tolerance for lag: Most applications tolerate brief replica lag (social media feeds, analytics, logging).
Interview Insight: Many interviewees say "we need synchronous replication for data safety." A more nuanced answer is: "We use asynchronous replication as the default for performance, but we can configure synchronous replication for critical data paths like payments and authentication. We also use monitoring on replica lag and alert if it exceeds our threshold."

Common Interview Questions

  1. "Explain asynchronous replication and its tradeoffs."
    Answer: "In asynchronous replication, the leader writes data locally and immediately acknowledges the client without waiting for followers. Followers receive replication data in the background. The main advantage is low write latency and high throughput -- the client only waits for the leader's local disk write. The tradeoff is replica lag: followers may be behind the leader, causing stale reads. More critically, if the leader crashes, any unreplicated writes are permanently lost. This is the default mode for MySQL, MongoDB, and most production databases because the performance benefits outweigh the data loss risk for most use cases."
  2. "What is replica lag and how do you handle it?"
    Answer: "Replica lag is the delay between a write being committed on the leader and being applied on a follower. It manifests as several consistency issues: stale reads where clients see old data, monotonic read violations where time appears to go backwards, and read-after-write inconsistencies. Mitigations include: (1) Reading from the leader after writes for critical consistency. (2) Using session affinity so a client always reads from the same replica. (3) Monitoring lag with alerts when it exceeds a threshold. (4) Breaking large transactions into smaller batches to reduce the replication payload."
  3. "What happens to unreplicated writes when the leader fails?"
    Answer: "Unreplicated writes are permanently lost. The client received an ACK for those writes (from the leader), but the data never made it to any follower. When the follower is promoted to leader, those writes do not exist. This is a silent data loss scenario -- the client believes the write succeeded. Mitigations include: semi-synchronous replication (wait for at least one follower), writing to a durable queue before the database, and using application-level idempotency so clients can safely retry after failover."

What to Skip

Do not go deep into:
  • MySQL binlog internals (statement vs. row format details, GTID internals).
  • MongoDB oplog implementation (oplog entry format, batching logic).
  • Network-level replication transport (TCP buffer management, compression).
  • Follower I/O scheduling for applying replication logs.
  • Specific monitoring tool configurations (Prometheus exporters, Grafana dashboards).
