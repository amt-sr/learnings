Stale Reads

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand why stale reads happen in replicated systems, how they degrade user experience, and how to design around them.

Definition

A stale read occurs when a client reads data from a replica that has not yet applied the latest writes from the leader. The client receives an older version of the data, potentially making incorrect decisions or seeing inconsistent state.

Timeline: How Stale Reads Happen

Time Client Leader (Primary) Follower (Replica) | | | | t0 User writes post ACCEPT WRITE | "Hello World" (id=1001) | | | COMMIT (LSN=5000) | | |-- POST /post -->| | | |<- 201 Created --| | | | | t1 User refreshes page | | |-- GET /posts -->| | | | (routed to Follower by load balancer) | | | |<-- replication delay ---| | |<- [empty list] ---| LSN=4995 (behind!) | | | | | | "Where did my post go?!" | | | t2 User refreshes again (2s later) | | |-- GET /posts -->| | | | (routed to Follower) | | | | APPLY LSN=4996..5000 | | |<- [post: "Hello World"] ---------------------| | | | | | "There it is." |

Impact on User Experience

Scenario What the User Sees Severity
Social media post User posts, refreshes, post is missing. May re-post, creating duplicates. High
Bank account balance After deposit, balance shows old amount. User may think deposit failed. Critical
Profile update User changes name, sees old name on next page load. High
Product inventory count Shows items in stock that are actually sold out. High
Analytics dashboard Metrics lag by seconds. Generally acceptable. Low
Feed timeline New items from followed users appear with slight delay. Low

Why It Happens: The Architecture Tradeoff

Stale reads are an inherent consequence of asynchronous replication, which is chosen for performance. In a single-leader system:

  • Writes go to the leader for strong consistency.
  • Reads are distributed across replicas for scalability and low latency.
  • The leader acknowledges writes immediately; replicas apply them later.
  • Any read hitting a replica before it applies the write sees stale data.
Key insight for interviews: Stale reads are not a bug — they are a deliberate tradeoff. The question is not "how do we eliminate them?" but "which reads can tolerate staleness and which cannot?"

Solutions

1. Read-After-Write Consistency

Ensure that a user always reads their own recent writes. Route reads for recently-written data back to the leader. See 4.6.3 Read After Write Consistency for full details.

2. Sticky Reads (Session Consistency)

Pin a user's session to a single replica for its duration. If the user writes, pin to the leader for the remainder of the session. This ensures the user sees a monotonically increasing version of their own data.

Session Start | v [Route to Replica A] <-- all reads go here for this session | User writes something | [Re-pin to Leader] <-- session stays pinned to leader | User reads | [Read from Leader] <-- guaranteed to see own write | Session ends | [Return to pool]

3. Follower Lag Monitoring and Adaptive Routing

Monitor each follower's replication lag in real time. If a follower's lag exceeds a threshold, stop routing reads to it. Implement this in a middleware layer (ProxySQL, HAProxy, or a custom proxy).

Production pattern: Instagram routes reads to replicas but tracks the version number (timestamp) of the user's last write. The proxy checks if the replica has caught up to that version before routing the read. If not, it falls back to the leader.

4. Client-Side Retry with Version Check

The client includes the version/timestamp of its last write with the read request. The server checks whether the serving replica has applied that version. If not, it retries on the leader or waits.

5. Causal Consistency

A stronger guarantee where replicas respect the causal order of operations. If write A causally precedes write B, no replica will return B without also having A. Implemented via vector clocks or logical timestamps.

Common Interview Questions

  1. "A user posts a comment, refreshes, and doesn't see it. What's happening and how do you fix it?" — Explain stale reads due to replica lag. Solutions: read-after-write (route own reads to leader), sticky sessions, version-aware routing.
  2. "Design a read routing layer that minimizes stale reads." — Discuss a proxy that tracks client write timestamps, checks replica progress, and adaptively routes. Mention lag thresholds and health checks.
  3. "Is eventual consistency acceptable for a banking app?" — Generally no for balance reads and transfers. Use strong consistency (read from leader or use synchronous replication) for financial operations. Eventual consistency may be acceptable for analytics, marketing emails, or statement generation.
  4. "How does Instagram handle stale reads at scale?" — They use a master-replica architecture with Redis caching for hot data. Reads for the author's own content go to the leader; reads for feed content go to replicas with eventual consistency.

What to Skip

Avoid cache invalidation deep dives: Cache staleness is related but distinct. Focus on database replica staleness unless the question explicitly bridges both.

Avoid distributed consensus details: Paxos/Raft details are covered in consensus topics. Stay focused on the read-path problem here.

Avoid multi-leader stale reads: The common interview scenario is single-leader with replicas. Multi-leader staleness is a more advanced variant covered under divergence.
