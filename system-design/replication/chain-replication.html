Chain Replication

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the chain replication topology where the leader replicates to a single follower, which replicates to the next, forming a linear chain. Recognize its bandwidth efficiency and its fragility when any single link fails.

Definition

Chain replication is a topology in which replicas are arranged in a linear sequence. The leader sends replication data to only one follower (the first link in the chain), which forwards it to the next follower, and so on down the chain. Each node in the chain has exactly one upstream and at most one downstream partner.

Key idea: The leader only ships replication data once. The burden of distributing to downstream replicas is shared across the chain. This is the replication model used by systems like Chain Replication (the academic protocol by Renesse and Schneider) and is conceptually similar to daisy-chained relay replication in MySQL.

How It Works

Write Path (Academic Chain Replication Protocol)

  1. Client sends a write request to the head (leader H0).
  2. H0 appends the operation to its log and forwards it to H1.
  3. H1 appends and forwards to H2, and so on down the chain.
  4. The tail (last node, Hn) receives the operation, appends it, and sends an acknowledgment back along the chain to H0.
  5. H0 responds to the client once the ack traverses back.

Read Path

  1. Client sends a read request to the tail (Hn).
  2. Because the write has fully propagated through the chain before the client is acked, the tail always has the latest state.
  3. This provides strong consistency for reads without additional coordination.

Advantages

  • Reduced leader bandwidth: The leader ships replication data only once, regardless of how many followers exist. This is a major advantage over star topology for large replica sets.
  • Strong read consistency: Reads from the tail are strongly consistent because writes propagate through the entire chain before acknowledgment.
  • Simple ordering: Total order of operations is naturally maintained by the chain. Each node sees operations in the same sequence.
  • Efficient for geo-replication: A single cross-region link can fan out to multiple replicas within that region.

Disadvantages

  • Single point of failure at every link: If any node in the chain fails, the chain is broken. Downstream nodes stop receiving updates. Recovery requires reconfiguring the chain.
  • High latency for writes: A write must travel the full length of the chain (H0 → H1 → ... → Hn) and the acknowledgment must travel back (Hn → ... → H0). Write latency is proportional to chain length.
  • Tail becomes a bottleneck: All reads go to the tail, making it a hotspot. Only one node serves consistent reads.
  • Cumulative lag: Each hop adds a small amount of latency. Follower 5 has 5x the propagation delay of Follower 1.
  • Reconfiguration complexity: Adding or removing a node requires breaking and re-establishing chain links, which can cause brief unavailability.

Chain vs Star Comparison

Property Star Chain
Leader outbound connections N (one per follower) 1 (only to first link)
Write latency Low (parallel fan-out) High (sequential through chain)
Read consistency from follower Eventual (each follower varies) Strong (if reading from tail)
Fault tolerance High (independent links) Low (any break halts chain)
Leader bandwidth usage O(N) data sent O(1) data sent
Adding a follower Simple (connect to leader) Requires chain reconfiguration

Practical Use Cases

MySQL Relay Log Chaining

MySQL does not implement true chain replication natively, but you can approximate it using cascading replication where a relay follower (configured with log_slave_updates) acts as an intermediate node, forwarding replication to leaf followers.

Geo-Distributed Replication

Chain topology is commonly used for cross-datacenter replication: the primary data center ships to a regional relay, which fans out locally. The WAN link is traversed only once per write.

Interview Tip: When designing a cross-region system, chain replication over WAN + star within a region is the typical pattern. The chain handles expensive WAN links; the star handles cheap intra-DC fan-out.

Failure Recovery

Chain breakage requires reconfiguration:

  1. Detect the failed node (via heartbeat timeout).
  2. Remove the failed node from the chain logically.
  3. Reconnect the upstream node to the next alive downstream node.
  4. Resume replication from the last consistent position.
Recovery challenge: If H2 fails, H1 must start shipping to H3. But H3 may be behind H2's position. H1 must either wait for H3 to catch up or ship the delta. This recovery complexity is why pure chain replication is rare in production systems.

Common Interview Questions

  • "How does chain replication differ from star replication?" Answer: In chain, the leader ships to one follower, which forwards to the next. This reduces leader bandwidth but increases write latency and creates a single point of failure at every link.
  • "When would you choose chain over star?" Answer: When the leader's outbound bandwidth is the bottleneck and you need to replicate across expensive WAN links. Chain is ideal for geo-distributed setups where a single cross-region link feeds multiple regional replicas.
  • "What happens when a node in the middle of the chain fails?" Answer: The chain breaks. Downstream nodes stop receiving updates. Recovery requires reconfiguring the chain to bypass the failed node.
  • "How does read consistency work in chain replication?" Answer: Strongly consistent reads are served from the tail node. Since writes must propagate through the entire chain before acknowledgment, the tail always has the latest data.

What to Skip

Avoid going deep into:
  • The original Chain Replication paper's formal proofs and consensus protocols. Understand the concept; skip the TLA+ specifications.
  • MySQL log_slave_updates configuration details. The concept of relay forwarding is important; the syntax is not.
  • Paxos/Raft-based chain reconfiguration algorithms. This is an advanced topic rarely asked directly.
