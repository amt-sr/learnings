Network Partition

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand network partitions as a fundamental failure mode in distributed systems, the CAP theorem implications, and the tradeoffs systems must make during a partition.

Definition

A network partition (also called a network split or partial failure) is a failure mode where messages between some nodes are lost or delayed beyond a timeout. Nodes are split into groups that can communicate within their group but not across groups. This is one of the three fundamental failures in distributed systems: crash faults, Byzantine faults, and network partitions.

Network partitions are not rare. They occur regularly in production at every scale — switch failures, routing flaps, DNS issues, misconfigurations, cross-datacenter link degradation. Any distributed system design must account for partitions.

CAP Theorem and Partitions

The CAP theorem (Brewer, 2000; Gilbert & Lynch proof) states that during a network partition, a distributed system must choose between:

Choice Behavior During Partition What You Sacrifice Examples
CP (Consistency over Availability) Stop accepting writes on the minority side. Only the majority partition serves requests. Availability: some clients see errors or timeouts ZooKeeper, etcd, HBase, MongoDB (majority write concern), CockroachDB
AP (Availability over Consistency) Both sides continue accepting writes. Data diverges. Resolved after partition heals. Consistency: clients may see stale or conflicting data Cassandra, DynamoDB, CouchDB, Riak, Eureka
Key interview point: CAP is about behavior during a partition. It does not say a system is always inconsistent or always unavailable. It says: when a partition occurs, you must pick one. In normal operation (no partition), a CP system is both consistent and available, and an AP system is both available and consistent. The tradeoff only materializes during failure.

Handling Strategies During Partition

Strategy 1: Leader Step Down (CP)

When a leader detects it cannot reach a majority of nodes, it voluntarily steps down. The minority partition goes read-only. No writes are accepted, preventing split brain and data divergence.

  • Pros: No data divergence. Strong consistency preserved.
  • Cons: Clients in the minority partition experience unavailability.

Strategy 2: Read-Only Mode with Graceful Degradation

The minority partition switches to read-only mode. It can still serve cached or replicated data for reads but rejects writes. This provides partial availability while preventing divergence.

Minority Partition During Network Partition: [Node A] [Node B] [Node C] | | | WRITE request (from client) | | | Check: quorum? NO (1/5 nodes) | | | Response: 503 Service Unavailable (or serve from local cache if acceptable) | | | READ request (from client) | | | Check: serve from local data? YES | | | Response: 200 OK (potentially stale)

Strategy 3: Continue Accepting Writes (AP)

Both partitions accept writes. Each uses version vectors or timestamps to track writes. After the partition heals, a reconciliation process merges divergent data. Conflicting writes are resolved using LWW, CRDTs, or application logic.

  • Pros: Both partitions remain available. Better for globally distributed systems where availability is critical.
  • Cons: Data diverges. Conflict resolution may silently lose data.

Strategy 4: Quorum Reads

During a partition, require reads from a quorum of replicas. If a client reads from R replicas out of N total, and W replicas acknowledged the write, the read is consistent if R + W > N. This ensures at least one replica in the read set has the latest write.

DynamoDB example: By default, DynamoDB reads from 2 of 3 replicas (R=2) and writes to 2 of 3 (W=2). Since R + W > N (2 + 2 > 3), a quorum read always returns the most recent write, even if one replica is partitioned away.

Real-World Examples

System Partition Behavior Details
etcd CP Minority partition stops serving. Leader only on majority side. Raft-based.
Cassandra AP All nodes accept writes. Divergence resolved via read repair, hinted handoff, and LWW.
CockroachDB CP Uses Raft. Minority partition goes read-only. Range leases prevent split brain.
DynamoDB AP Configurable consistency. Default: eventually consistent. Strongly consistent reads available as an option.
Netflix Eureka AP Service registry. During partition, all nodes serve stale registration data. Better to route to a stale endpoint than to fail.

Common Interview Questions

  1. "Explain the CAP theorem. What does a system do during a network partition?" — CAP says during a partition, choose C (reject writes on minority, stay consistent) or A (accept writes everywhere, risk divergence). Normal operation allows both. The choice is a design decision based on business requirements.
  2. "Would you design your system as CP or AP?" — Context-dependent. Financial: CP (cannot tolerate divergence). Social media feed: AP (availability matters more than perfect consistency). E-commerce inventory: somewhere in between (quorum reads/writes).
  3. "How does a 5-node Raft cluster behave during a 3-2 partition?" — The 3-node side has quorum, elects a leader, and serves reads and writes. The 2-node side cannot elect a leader (no quorum), goes read-only, and returns errors on write attempts. When partition heals, the 2-node side catches up from the leader.
  4. "How do you handle a network partition between two data centers?" — Discuss: cross-DC latency-aware routing, quorum settings, whether to go read-only in the minority DC, failover orchestration, and the business impact of unavailability vs. divergence.

What to Skip

Avoid PACELC deep dives: PACELC (Partition: Availability vs. Consistency, Else: Latency vs. Consistency) extends CAP but is rarely asked. Mention it briefly if you want to show depth.

Avoid specific BGP/routing protocol details: Network partitions are an abstraction. Interviewers do not expect routing protocol knowledge unless the role is network-infrastructure specific.

Avoid Gilbert-Lynch proof details: The formal proof is interesting but rarely tested. Focus on the practical implications.
