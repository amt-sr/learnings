Read Repair

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand read repair as an eventual consistency mechanism — how it detects and fixes data inconsistencies across replicas during normal read operations, without requiring a separate reconciliation process.

Definition

Read repair is a technique used in Dynamo-style distributed databases where inconsistencies across replicas are detected and corrected during read operations. When a client reads data, the coordinator reads from multiple replicas, compares the versions, resolves any conflicts, returns the most recent version to the client, and asynchronously updates (repairs) any replicas that returned stale data.

How Read Repair Works

SCENARIO: 3 replicas, one has stale data Replicas: Node-A: { key="user:42", value={name: "Alice", city: "NYC"}, v=7 } Node-B: { key="user:42", value={name: "Alice", city: "NYC"}, v=7 } Node-C: { key="user:42", value={name: "Alice", city: "Boston"}, v=5 } ^^^ STALE (missed the city update from v5->v7) READ REPAIR PROCESS: 1. Client sends: GET user:42, consistency=QUORUM (read from 2 of 3) 2. Coordinator reads from 2 replicas (say, A and C): A returns: {name: "Alice", city: "NYC"}, v=7 C returns: {name: "Alice", city: "Boston"}, v=5 3. Coordinator compares versions: v=7 > v=5 --> A is more recent 4. Return to client: {name: "Alice", city: "NYC"} (latest version) 5. BACKGROUND REPAIR (asynchronous): Coordinator sends to C: PUT user:42, value={name: "Alice", city: "NYC"}, v=7 C updates to v=7 6. All replicas now consistent: Node-A: v=7 Node-B: v=7 Node-C: v=7 (repaired!)

Read Repair in Dynamo-Style Systems

Read repair is a core mechanism in Amazon Dynamo and its derivatives (Cassandra, DynamoDB, Riak). It is part of the quorum-based read/write model:

Operation Behavior
Write at consistency level W Write sent to N replicas. Coordinator waits for W acknowledgments.
Read at consistency level R Read sent to N replicas. Coordinator waits for R responses, compares, and returns the most recent.
Read repair (triggered by read) If any of the R replicas returned stale data, the coordinator sends the latest version to all replicas that were stale.
Consistency guarantee: If R + W > N, every read is guaranteed to see the latest write because at least one of the R replicas must have received the write (since W replicas acknowledged it, and R + W > N means the read and write sets overlap).

When Read Repair Is Triggered

  • On every quorum read: In DynamoDB and Cassandra, read repair is triggered on every read at quorum consistency level. The coordinator reads from all N replicas (even though only R are needed), compares versions, and repairs any stale ones.
  • On digest mismatch: When replicas return different digests (hashes) of their data, the coordinator fetches the full data from all replicas to determine the most recent version, then repairs.
  • Background read repair: Some systems perform read repair in the background, not on the critical path of the read response. The client gets the answer immediately; repairs happen asynchronously.

Conflict Resolution During Read Repair

When replicas have truly divergent data (not just lag), read repair must resolve conflicts:

  1. Last-write-wins (LWW): Compare timestamps. The write with the latest timestamp wins. All replicas are updated to this version. Simple but may silently discard valid writes.
  2. Vector clock comparison: If one version is clearly newer (supersedes the other), use it. If versions are concurrent (neither supersedes the other), the conflict must be resolved.
  3. Application-level merge: For concurrent versions, call an application-provided merge function. E.g., for a shopping cart, merge (union) the items from both versions.
  4. Tombstones: Deleted data is handled via tombstones (special marker records). A read that encounters a tombstone ensures the tombstone is propagated to all replicas.

Limitations of Read Repair

Limitation Details
Only repairs data that is read If a key is never read, its inconsistency is never repaired. Cold data can diverge indefinitely.
Read path latency increase Reading from multiple replicas and comparing adds latency. Background repair mitigates this.
Does not replace anti-entropy Read repair is opportunistic. Systems still need periodic anti-entropy repair (Merkle tree comparison) for data that is rarely accessed.
Conflict resolution may lose data LWW silently discards one version. Application-level merge requires custom code and testing.
Complementary mechanisms: Read repair handles hot data (frequently accessed keys). Anti-entropy repair (Merkle tree comparison) handles cold data (infrequently accessed keys). Together, they provide comprehensive eventual consistency. Cassandra runs both: read repair on every read and nodetool repair (anti-entropy) as a periodic background job.

Common Interview Questions

  1. "How does DynamoDB ensure eventual consistency?" — Quorum-based reads and writes (R + W > N), read repair on every quorum read (propagate latest version to stale replicas), hinted handoff for temporarily unavailable replicas, and periodic anti-entropy for cold data.
  2. "What is read repair and when is it triggered?" &mdash> On every quorum read in Dynamo-style systems. The coordinator reads from multiple replicas, compares versions, returns the latest to the client, and asynchronously updates stale replicas. Only repairs data that is actually read.
  3. "What are the limitations of read repair?" &mdash> Only repairs read data (cold data stays inconsistent), adds read latency (mitigated by background repair), conflict resolution may lose data, and it does not replace periodic anti-entropy.
  4. "How does Cassandra handle read repair?" &mdash> On a read, Cassandra sends read requests to all replicas for the requested key. It compares versions using timestamps (LWW). If any replica is stale, it sends the latest version in a background repair. Additionally, nodetool repair runs periodic anti-entropy repair using Merkle trees.

What to Skip

Avoid Merkle tree implementation: Merkle trees are used in anti-entropy (covered separately), not in read repair. Do not confuse the two mechanisms.

Avoid DynamoDB internal storage format: The B-tree/LSM tree details of DynamoDB storage are not relevant to read repair. Stay focused on the replication and consistency layer.

Avoid read repair in single-leader systems: Read repair is specific to Dynamo-style (multi-leader / leaderless) systems. In single-leader systems, inconsistencies are resolved by replication itself, not by read repair.
