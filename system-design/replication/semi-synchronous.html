4.3.3 Semi Synchronous Replication

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the pragmatic middle ground between synchronous and asynchronous replication. Know how semi-synchronous replication reduces the data loss window while preserving acceptable write latency, the MySQL semi-sync plugin as the canonical example, and the remaining risks that still exist.

Definition

Semi-synchronous replication means the leader waits for acknowledgment from at least one follower before committing a write and responding to the client. Unlike fully synchronous replication (which waits for ALL followers), semi-sync only requires a quorum of one. The remaining followers are replicated asynchronously in the background.

Semi-synchronous replication is a practical compromise: it dramatically reduces the data loss window compared to fully asynchronous replication, while adding significantly less latency than fully synchronous replication (which must wait for all followers).

Fallback Behavior

A critical feature of semi-synchronous replication is its graceful fallback to asynchronous mode when no follower responds in time.

Semi-Sync Fallback to Async ─────────────────────────── Normal operation (Follower A responds within timeout): Client ──▶ Leader ──▶ Follower A (ACK in 3ms ✓) Client ◀── Leader ACK (semi-sync commit) Follower A is slow (ACK after 10s, timeout = 5s): Client ──▶ Leader ──▶ Follower A (no ACK in 5s) Client ◀── Leader ACK (FALLS BACK to async mode) What happens: 1. Leader sets a timeout (e.g., rpl_semi_sync_master_timeout = 5000ms) 2. If no follower ACK within timeout, leader falls back to async 3. Write is committed with only local durability 4. Data loss risk temporarily increases (same as async) 5. When follower recovers and ACKs, semi-sync mode resumes This fallback ensures writes are never blocked indefinitely by a slow or partitioned follower.
Key Point: The fallback mechanism means semi-sync is not a strict guarantee. During the fallback window, the system behaves exactly like asynchronous replication. The practical benefit is that the data loss window is reduced to the timeout period (typically 1-5 seconds) rather than the full replication lag (which could be much larger in pathological cases).

Remaining Risks

Semi-synchronous replication eliminates most but not all data loss risk:

RiskSeverityDescriptionMitigation
Timeout fallbackMediumIf no follower ACKs within the timeout, system falls back to async and loses the sync guarantee for those writesMonitor rpl_semi_sync_master_status; alert on fallback events
Both leader and sync follower crashMediumIf the leader and the one semi-sync follower crash simultaneously, committed data is on both failed machinesUse multiple semi-sync followers (wait_for_replica_count > 1)
Split-brain during failoverMediumSame split-brain risk as leader-follower; semi-sync does not prevent itFencing tokens, external consensus for failover
Non-synced follower promoted to leaderHighIf the sync follower crashes and an async follower is promoted, it may be missing writesPrefer promoting the sync follower; use GTID to track position

When to Use Semi-Synchronous Replication

Interview Tip: Semi-synchronous is the right answer when you need better durability than async but cannot afford the latency cost of full sync. Frame it as: "We use semi-synchronous replication as a practical middle ground. It reduces our data loss window from potentially seconds of replication lag to just the timeout period, while adding only one network round trip to our write latency."
  • E-commerce order processing: You cannot lose orders but can tolerate a few milliseconds of added write latency.
  • User account operations: Password changes, email updates, account settings -- important data where loss is unacceptable but not mission-critical enough for full sync.
  • General-purpose databases: A good default for production systems that want durability without the full cost of synchronous replication.
  • Multi-DC setups with cross-replication: Sync to local follower (fast RTT), async to remote follower (slow RTT). Achieves local durability without remote latency.

Common Interview Questions

  1. "What is semi-synchronous replication?"
    Answer: "Semi-synchronous replication is a middle ground between fully synchronous and asynchronous replication. The leader waits for acknowledgment from at least one follower before committing a write, but does not wait for all followers. This means committed data exists on at least two machines, reducing the data loss window compared to async. The latency cost is roughly one network round trip to the nearest follower, which is significantly less than waiting for all followers. If no follower responds within a timeout, the system falls back to async mode to avoid blocking writes indefinitely."
  2. "How does semi-sync compare to fully synchronous replication?"
    Answer: "Fully synchronous waits for ALL followers, providing zero data loss guarantee but high latency (bounded by the slowest follower). Semi-sync waits for at least ONE follower, providing a reduced but not zero data loss window, with lower latency (bounded by the fastest follower). The practical difference: semi-sync is robust against a leader crash because at least one other node has the data, but it is not robust against the simultaneous crash of both the leader and the sync follower. In most deployments, that correlated failure is unlikely enough to accept."
  3. "What happens when the semi-sync follower is slow?"
    Answer: "The leader has a configurable timeout (in MySQL, rpl_semi_sync_master_timeout). If no follower ACKs within this timeout, the leader falls back to asynchronous mode for that write. This ensures that writes are never blocked indefinitely. The tradeoff is that during the fallback period, the system has the same data loss characteristics as async replication. In practice, the timeout is set to a few seconds (5-10s), so the data loss window is bounded. Monitoring should alert when fallback events occur so operations can investigate."
