Circular Replication

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the ring-based replication topology where each node replicates to the next in a circle, recognize the absence of a single bottleneck, and identify the critical risks of replication cycles and duplicate updates.

Definition

Circular replication (also called ring replication) arranges replicas in a closed loop. Each node acts both as a replication source and a replication target. Node A replicates to B, B replicates to C, C replicates back to A. Unlike chain replication, there is no distinct head or tail -- every node is equal in the ring.

Key idea: Circular replication is fundamentally a multi-master topology. Every node can accept writes, and changes propagate around the ring. This eliminates the single-leader bottleneck but introduces conflict resolution challenges.

No single leader. No single bottleneck. But: risk of infinite replication loops.

How It Works

Write Propagation

  1. A client writes to Node A.
  2. Node A applies the write locally and adds it to its replication log.
  3. Node A ships the change to Node B (its downstream neighbor).
  4. Node B applies the change, marks it as originating from A, and ships to Node C.
  5. Node C ships to Node D, and Node D ships back to Node A.
  6. When Node A receives its own change back, it must detect and discard it to prevent infinite looping.

Conflict Detection

Since any node can accept writes concurrently, conflicts arise when two nodes modify the same data:

  • Last-write-wins (LWW): Timestamp-based resolution. Simple but can lose data.
  • Server ID filtering: Each write is tagged with the originating server's ID. Nodes skip writes that originated from themselves when they come back around the ring.
  • Application-level conflict resolution: The application defines merge semantics (e.g., CRDTs, version vectors).
Critical danger: Without proper origin tracking, writes loop forever around the ring. Each pass creates a new copy, multiplying data exponentially. This is the primary reason circular replication is rarely used in production.

Advantages

  • No single leader bottleneck: Every node can accept writes and serve reads. Write throughput scales with the number of nodes.
  • No single point of failure: Unlike chain replication, the ring has no head or tail. Any node can fail and the ring can be re-wired.
  • Low per-node outbound bandwidth: Each node ships data to only one downstream neighbor (like chain).
  • Natural load distribution: Writes from different clients can go to different nodes, distributing write load.

Disadvantages

  • Replication cycles: The closed loop means changes travel the full ring. Without careful origin tracking, writes loop infinitely.
  • Conflict resolution complexity: Multi-master means concurrent writes to the same data are possible. Conflict resolution adds application-level complexity.
  • Eventual consistency only: A write on Node A takes time to propagate through the ring. Other nodes may serve stale reads until the change arrives.
  • Split-brain risk: If the ring breaks at any point, nodes on each side cannot communicate and may diverge.
  • Difficult debugging: Tracing the origin of a change around the ring is complex. Lag compounds at each hop.
  • Not widely supported: MySQL supports circular replication configuration, but most modern systems prefer leader-follower with failover instead.

Preventing Infinite Loops

The fundamental challenge of circular replication is preventing writes from cycling forever:

Method Mechanism Limitation
Server ID filtering Each event tagged with originating server_id. Nodes skip events from themselves. Requires log_slave_updates and careful server_id configuration.
GTID (Global Transaction ID) Each transaction gets a unique ID (server_uuid:sequence). Nodes track which GTIDs they have already applied. Supported in MySQL 5.6+. Requires all nodes to use GTID mode.
Replicate-ignore-server Configure each node to ignore replication events from its own server ID. Fragile; misconfiguration causes immediate infinite loops.

Use Cases

Where Circular Replication Makes Sense

  • Multi-datacenter active-active: Each data center needs to accept writes locally. A ring topology can connect DCs where each DC is a node in the ring.
  • Write-anywhere requirements: Applications that cannot route all writes through a single leader.
  • Merge replication: Branch office scenarios where each office has a local database and changes need to sync in all directions.

Where to Avoid It

  • Most OLTP workloads -- leader-follower with failover is simpler and more reliable.
  • Systems requiring strong consistency -- circular is eventually consistent by nature.
  • Environments without mature conflict resolution -- data corruption risk is high.
Interview Tip: If asked about circular replication, immediately highlight the loop prevention problem. A strong answer discusses server ID filtering, GTID-based deduplication, and why most production systems avoid circular replication in favor of leader-follower with automated failover.

Common Interview Questions

  • "What is circular replication?" Answer: A ring topology where each node replicates to the next, forming a closed loop. Every node is both a master and a replica. Changes propagate around the ring and must be filtered to prevent infinite loops.
  • "How do you prevent infinite replication loops in circular replication?" Answer: Use server ID filtering (each node ignores events from its own ID) or GTIDs (globally unique transaction IDs that nodes track to avoid re-applying). MySQL supports replicate-same-server-id=0 and GTID-based filtering.
  • "What is the main risk of circular replication?" Answer: Data corruption from unhandled replication cycles, and conflict resolution complexity from multi-master writes to the same data.
  • "When would you choose circular over star replication?" Answer: When you need multi-master write capability across data centers and can handle eventual consistency with proper conflict resolution. Most of the time, leader-follower with failover is preferred.

What to Skip

Avoid going deep into:
  • MySQL replicate-same-server-id or log-slave-updates syntax. Understand the concept; skip the configuration details.
  • GTID internals (UUID generation, sequence numbering format). Know that GTID provides unique transaction identification; skip the binary format details.
  • Specific CRDT implementations for conflict resolution. Mention CRDTs as an option but do not implement one.
