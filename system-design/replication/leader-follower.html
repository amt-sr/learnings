4.2.1 Leader-Follower (Single-Leader Replication)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand the most widely used replication model in production systems. Know how writes flow through the leader, how followers stay synchronized, what consistency guarantees exist, and how failover works. This model is the default for most relational databases and is frequently referenced in system design interviews.

Definition

Single-leader replication (also called primary-replica, active-passive, or master-slave) is a replication model where one node is designated as the leader (or primary). All write operations must go through the leader. The leader then propagates these writes to one or more follower (replica) nodes. Followers handle read operations.

This is the simplest replication model and, consequently, the most common. Its simplicity comes from a single ordering point: the leader determines the canonical order of all writes.

Interview Tip: When designing with leader-follower, always mention that reads can go to followers for scalability, but note the replica lag problem: a follower may not yet have the latest write. This is a key tension to articulate in interviews.

Read-Your-Writes Consistency

One of the most common problems with leader-follower replication is the stale read problem. A client writes to the leader, then immediately reads from a follower, and gets an old value because the follower has not yet applied the write.

Timeline of Events: t=0ms Client writes "name = Alice" to LEADER t=1ms Leader commits write, sends ACK t=2ms Replication log entry sent to followers t=2ms Client reads "name" from FOLLOWER A └──▶ Returns "name = Bob" (STALE!) t=15ms FOLLOWER A applies replication log entry t=16ms If client reads again, gets "name = Alice" ✓ The 2ms-15ms window is the replica lag window.

Solutions to this problem:

  1. Read from leader for critical reads: After a write, route the next read to the leader. Simple but increases leader load.
  2. Track replication progress: The leader tracks how far each follower has progressed in applying the replication log. If a follower is up-to-date, route reads there; otherwise, use the leader or another follower.
  3. Client-side timestamp: The client remembers the timestamp of its last write and includes it in read requests. Followers that have applied all writes up to that timestamp can serve the read.
  4. Monotonic reads: Ensure a client always reads from the same replica (or one that is at least as up-to-date as the last read). Prevents the "time going backwards" anomaly.

Failover Process

When the leader fails, one of the followers must be promoted to become the new leader. This is a critical and surprisingly complex operation:

B │ │ 💥 DOWN │ │ (alive) │ │ (alive) │ └──────────┘ └──────────┘ └──────────┘ │ ▼ 2. DETECTION (timeout: no heartbeat for N seconds) Follower A: "Leader is unresponsive!" RECONFIGURATION - Route all writes to new leader A - Update all clients and proxies - Re-replicate from new leader to B 6. OLD LEADER RECOVERY (if it comes back) ┌──────────┐ ┌──────────┐ ┌──────────┐ │ OLD LDR │ │ LEADER A │ │Follow. B │ │ (confused)│ │ │ │ │ └──────────┘ └──────────┘ └──────────┘ - Must be demoted to follower - May need to discard unreplicated writes - RISK: split-brain if old leader thinks it's still leader
Danger: Split-Brain. The biggest risk in failover is split-brain: two nodes both believe they are the leader. If the old leader recovers but has not been informed it was replaced, it may accept writes that conflict with the new leader. This causes data corruption that is extremely hard to detect and resolve. Solutions include fencing tokens, STONITH ("Shoot The Other Node In The Head"), and lease-based approaches.

Common Interview Questions

  1. "Explain leader-follower replication."
    Answer: "One node is the leader that accepts all writes. It writes changes to a replication log and sends this log to follower nodes. Followers apply the log entries in the same order, maintaining identical copies. Reads can be served by followers for scalability. The key challenge is handling leader failure through failover and managing replica lag for read-after-write consistency."
  2. "What happens when the leader fails?"
    Answer: "First, followers detect the failure via heartbeat timeout. Then a new leader is elected -- either manually by an operator, or automatically via a consensus protocol. The elected follower starts accepting writes. The critical concern is split-brain prevention: the old leader must be fenced out. Common fencing mechanisms include fencing tokens in storage (e.g., increasing an epoch number that the storage layer checks), or network-level STONITH. After the old leader is confirmed dead or demoted, it can be brought back as a follower."
  3. "How do you handle read-after-write consistency?"
    Answer: "The simplest approach is to route reads to the leader immediately after a write. For better scalability, track replication progress per follower and route reads to a follower that has applied all writes up to the client's last write timestamp. This requires the leader to track each follower's replication position and for the client to remember its write timestamp. In practice, many systems use a session-based approach: after writing, the client's session is 'sticky' to the leader for a short window."
  4. "What is split-brain and how do you prevent it?"
    Answer: "Split-brain occurs when two nodes both believe they are the leader, typically after a network partition causes both the old and new leaders to accept writes independently. Prevention strategies include: (1) Fencing tokens -- a monotonically increasing number stored in a separate service; the higher number wins. (2) Resource fencing -- the old leader's storage access is revoked at the storage layer (e.g., SCSI reservations, cloud API calls to detach disks). (3) Leases with timeouts -- the leader must renew its lease; if it cannot, it steps down voluntarily."

What to Skip

Do not go deep into:
  • Raft/Paxos internals -- these are covered in the Consensus chapter. Know that they solve leader election, not the byte-level details.
  • Database-specific replication configuration syntax (e.g., MySQL GTID settings, PostgreSQL recovery.conf).
  • Physical vs. logical replication internals for PostgreSQL -- the distinction matters for operations but not for system design interviews.
  • Specific failover tool implementations (MHA, Patroni internals) unless you have hands-on experience.
  • Replication topology variants (circular, chained) -- these are rarely used in practice and add complexity without much interview value.
