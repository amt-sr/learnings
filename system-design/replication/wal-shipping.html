WAL Shipping

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the PostgreSQL-style write-ahead log shipping mechanism for replication. Know how the leader generates WAL segments, ships them to followers, and how followers replay them to reconstruct state. Contrast this with MySQL's statement/row-based binlog replication.

Definition

WAL (Write-Ahead Log) shipping is PostgreSQL's primary replication mechanism. The leader continuously writes all changes to its WAL -- a byte-level log of physical storage modifications. Followers receive WAL records from the leader and replay them, applying the exact same physical changes to their own storage. This produces an identical physical copy of the leader's database.

Key idea: PostgreSQL's WAL is a physical log. Unlike MySQL's binlog (which records logical statements or row changes), the WAL records the actual byte-level changes to disk pages. This makes WAL shipping a form of physical replication -- the follower is a bit-for-bit copy of the leader.

How WAL Shipping Works

1. WAL Generation (Leader)

  1. Client sends a query (INSERT, UPDATE, DELETE, DDL).
  2. PostgreSQL modifies in-memory pages in the shared buffer pool.
  3. Before the transaction is confirmed, the changes are written to the WAL buffer.
  4. The WAL buffer is flushed to WAL files on disk (fsync). This guarantees durability.
  5. WAL files are 16MB segments that rotate as they fill up.

2. WAL Streaming (Primary Mode)

  1. The follower connects to the leader and sends a IDENTIFY_SYSTEM handshake.
  2. The follower requests WAL from a specific LSN (Log Sequence Number) position.
  3. The leader's wal sender process streams WAL records to the follower over a dedicated TCP connection.
  4. The follower's wal receiver writes received WAL to its own WAL files.
  5. The follower's startup process replays WAL records, applying the physical changes to its data files.

3. File-Based Shipping (Legacy Mode)

  1. The leader writes completed WAL segments to a configurable directory (via archive_command, e.g., rsync or scp).
  2. The follower's restore_command fetches archived WAL segments.
  3. The follower replays the archived segments.
Streaming vs File-based: Streaming replication is the default in modern PostgreSQL. It provides near-real-time replication (sub-second lag in most cases). File-based shipping is a legacy approach with higher lag (seconds to minutes) and is primarily used as a fallback or for cross-site backup.

Key Concepts

LSN (Log Sequence Number)

Every position in the WAL is identified by an LSN, a 64-bit integer representing the byte offset within the WAL stream. Followers track their replication position using LSN. Example: 0/3000000.

WAL Archiving

Before a WAL segment is overwritten, it can be archived via archive_command. This provides a durable history of all changes for point-in-time recovery. Archived WAL segments are essential for restoring a follower that has fallen far behind.

Replication Slots

Replication slots ensure the leader retains WAL segments needed by a follower even if the follower disconnects. Without slots, the leader might recycle WAL segments before the follower can catch up, causing replication to break.

Replication slot warning: If a follower disconnects and never comes back, its replication slot prevents the leader from recycling WAL. This causes the leader's disk to fill up with WAL files. Monitor slot lag and drop inactive slots to prevent disk exhaustion.

Common Interview Questions

  • "How does PostgreSQL replication work?" Answer: The leader generates WAL (Write-Ahead Log) records for all changes. Followers connect via streaming replication, receive WAL records in real time, and replay them to maintain an identical physical copy.
  • "What is the difference between WAL shipping and MySQL binlog replication?" Answer: WAL is physical replication (byte-level page changes), while MySQL binlog is logical (SQL statements or row changes). WAL requires same-version PostgreSQL; binlog allows some version flexibility. WAL is faster to replay but less flexible.
  • "What is an LSN?" Answer: Log Sequence Number, a byte offset in the WAL stream that identifies a replication position. Followers use LSN to track how far they have replicated and to resume after disconnection.
  • "What happens if a PostgreSQL follower falls far behind?" Answer: The leader may recycle WAL segments the follower still needs. Replication slots prevent this by forcing the leader to retain WAL. Without slots, the follower must re-sync from a base backup.

What to Skip

Avoid going deep into:
  • WAL file format internals (XLogRecord structure, CRC checksums, page header format). Understand the concept; skip the binary layout.
  • archive_command and restore_command shell script examples. Know that these are configurable hooks; skip the implementation.
  • PostgreSQL pg_hba.conf and postgresql.conf replication settings. Mention the key parameters conceptually; skip the exact configuration syntax.
