Physical Replication

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand physical replication as the lowest-level replication approach that copies disk block and WAL byte changes directly. Know why it produces faster replication than logical, the constraints it imposes (same version, same architecture), and when to choose physical over logical replication.

Definition

Physical replication copies data at the storage level -- the exact disk block modifications recorded in the Write-Ahead Log (WAL) are shipped to the follower and applied identically. The follower is a bit-for-bit physical clone of the leader. Every page, every index, every system catalog entry is reproduced exactly as it exists on the leader. There is no interpretation, decoding, or transformation of the data -- the bytes are copied as-is.

Key idea: Physical replication is the fastest and simplest form of replication because it bypasses the SQL layer entirely. The follower does not parse statements, decode row changes, or execute queries. It simply writes bytes to the same disk locations as the leader. The tradeoff is zero flexibility -- the follower must be identical to the leader in version and architecture.

How It Works

1. WAL Generation

  1. A client modifies data on the leader (INSERT, UPDATE, DELETE).
  2. PostgreSQL modifies pages in the shared buffer pool.
  3. Before confirming the transaction, the physical page modifications are written to the WAL as byte-level records (page ID, offset, before/after image of changed bytes).

2. WAL Streaming

  1. The follower's wal receiver connects to the leader's wal sender.
  2. The wal sender streams raw WAL records (bytes) to the follower over TCP.
  3. The follower writes the WAL records to its own WAL files.

3. WAL Replay

  1. The follower's startup process reads WAL records and applies the same physical modifications to its own data files.
  2. The same page, at the same offset, gets the same bytes written to it.
  3. The result is a disk-level identical copy of the leader.

Advantages

  • Maximum performance: No decoding overhead. WAL bytes are copied and applied directly. This is the fastest replication method available.
  • Zero data interpretation: The follower does not need to understand SQL, table schemas, or data types. It just writes bytes. This eliminates entire categories of replication bugs related to type casting, character set conversion, or function evaluation.
  • Complete database replication: Everything is replicated -- all tables, all indexes, all system catalogs, all sequences. Nothing is left out. No need to configure which tables to replicate.
  • DDL replicated automatically: Schema changes (CREATE TABLE, ALTER TABLE, CREATE INDEX) are WAL operations and are replicated identically to the follower.
  • Simplicity: No publication/subscription configuration, no table-level filtering, no conflict resolution logic. The follower is a passive clone.
  • Failover readiness: Since the follower is an exact physical copy, promoting it to leader is straightforward. No schema drift, no missing sequences, no partial tables.

Limitations

  • Same major version required: The follower must run the exact same PostgreSQL major version. WAL format can change between major versions. Upgrading requires a logical dump/restore.
  • Same architecture required: Physical replication does not work across different CPU architectures (e.g., x86 to ARM) or different operating systems. The disk layout must be binary-compatible.
  • All-or-nothing replication: You cannot replicate a subset of tables. Physical replication copies the entire database cluster. If you only need one table replicated, use logical replication instead.
  • Cannot serve different workloads easily: Since indexes are replicated identically, you cannot optimize the follower with different indexes (e.g., analytics-optimized indexes vs OLTP indexes).
  • Follower cannot accept writes: In standard streaming replication, the follower is read-only. It cannot diverge from the leader by accepting its own writes.
  • Larger replication volume: Physical WAL can be larger than logical change events for the same logical change, because WAL includes index maintenance, page-level overhead, and full page images during checkpoints.

When to Choose Physical Replication

Scenario Recommendation
High-availability failover replica Physical (exact clone, fast failover)
Read replica for OLTP queries Physical (fast, simple, complete)
Cross-version upgrade replication Logical (version-independent)
Replicating specific tables to another service Logical (partial replication)
CDC pipeline to Kafka/Elasticsearch Logical (row-level events)
Disaster recovery standby Physical (complete, fast, simple)
Analytics replica with different indexes Logical (index flexibility)

Physical Replication in Other Databases

The concept of physical replication extends beyond PostgreSQL:

  • Oracle Data Guard: Ships redo log (Oracle's equivalent of WAL) to standby databases. Physical standby is an exact block-level copy.
  • MySQL Group Replication / InnoDB Cluster: Uses a combination of binlog-based logical replication. Not strictly physical, but similar concepts.
  • Cassandra: Uses SSTable-based replication (shipping sorted string tables), which is closer to physical replication than logical.
  • MongoDB replica sets: Ships oplog entries (similar to logical row changes), not physical block copies.
Interview Tip: When asked "how would you set up a PostgreSQL HA pair?", the answer is physical replication (streaming) for the standby, plus synchronous commit for zero data loss on failover. Physical replication gives you the fastest replication and simplest failover. When asked "how would you replicate specific tables to a data warehouse?", switch to logical replication for its partial replication capability.

Common Interview Questions

  • "What is physical replication?" Answer: Replication that copies disk block/WAL byte changes directly from leader to follower, producing an exact physical clone. No SQL decoding, no row-level interpretation -- just raw byte copying.
  • "What are the tradeoffs of physical vs logical replication?" Answer: Physical is faster and simpler (exact clone, automatic DDL replication) but inflexible (same version required, all-or-nothing replication). Logical is slower but flexible (cross-version, partial table replication, CDC use cases).
  • "Can I use physical replication to upgrade PostgreSQL?" Answer: No. Physical replication requires the same major version. For upgrades, use logical replication (pg_dump/pg_restore) or pg_upgrade. Mention that logical replication can help bridge versions during a rolling upgrade.
  • "How does physical replication handle DDL changes?" Answer: Automatically. DDL changes generate WAL records just like DML. Physical replication applies them identically. Logical replication does not replicate DDL.
  • "What is the right replication strategy for a PostgreSQL HA pair?" Answer: Physical streaming replication with synchronous commit. This provides the fastest replication, automatic failover (exact clone), and zero data loss if the follower confirms WAL writes before the leader commits.

What to Skip

Avoid going deep into:
  • WAL record internal format (XLogRecord, xl_heap_insert, page image compression). Understand the concept of physical byte copying; skip the binary layout.
  • pg_basebackup implementation details and tar mode vs plain mode. Know that it takes a physical backup; skip the internals.
  • Full page writes during checkpoints and their impact on WAL volume. Mention the concept briefly; skip the mechanism.
