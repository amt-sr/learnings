Star Replication

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the simplest replication topology where all followers connect directly to the leader, recognize its operational simplicity, and identify the scaling bottlenecks that make it unsuitable for large follower counts.

Definition

Star replication is a topology in which every replica (follower) establishes a direct connection to the primary (leader). The leader is responsible for shipping replication data to each follower independently. This is the default replication topology in most databases including MySQL, PostgreSQL, and Kafka.

Key idea: The leader acts as the central hub of a star. All replication traffic flows through it. Every follower is one hop away from the leader.

The leader manages N independent replication streams.

How It Works

Write Path

  1. Client sends write to the leader.
  2. Leader writes to its local storage (write-ahead log, binlog, etc.).
  3. Leader pushes the change to each follower independently over separate connections.
  4. Followers apply the change to their local storage.
  5. Leader acknowledges the write to the client (timing depends on sync mode).

Read Path

  1. Read requests can be served by the leader (always consistent).
  2. Read requests can also be served by followers (eventually consistent, possibly stale).

Advantages

  • Operational simplicity: Easy to understand, configure, and monitor. One replication source, N consumers.
  • Fast failover detection: Each follower independently monitors the leader. If the leader fails, any follower can be promoted.
  • Independent follower lag: A slow follower does not block or slow down other followers. Each replication stream is independent.
  • Easy to add/remove followers: Adding a new follower is just a matter of connecting it to the leader and starting replication from a known position.
  • Built-in support: This is the default mode for MySQL, PostgreSQL, and most relational databases.

Disadvantages

  • Leader network bottleneck: The leader must maintain N separate connections and send replication data N times. With 50+ followers, the leader's network bandwidth becomes the bottleneck.
  • Leader CPU overhead: Serializing and sending replication data to each follower consumes CPU on the leader.
  • Connection management overhead: Each follower requires an open TCP connection, memory for send buffers, and per-connection state tracking.
  • Not suitable for geo-distributed setups: Shipping data N times across WAN links is expensive. Cascading or chain topologies are better for cross-region replication.

When to Use

Scenario Suitability
Small number of followers (3-10) Ideal
All replicas in the same data center Ideal
Single-region deployment Good
Need independent per-follower failover Good
Large number of followers (50+) Poor - use cascading
Cross-region / geo-distributed Poor - use chain/cascading
Bandwidth-constrained leader Poor - use chain/cascading

Scaling Star Replication

When the leader becomes a bottleneck, there are practical mitigation strategies:

  • Compressed replication: Enable compression on the replication stream (e.g., MySQL slave_compressed_protocol). Reduces network bandwidth at the cost of CPU.
  • Binlog filtering: Only replicate the databases or tables that each follower needs (MySQL replicate-do-db, replicate-ignore-db).
  • Parallel replication threads: Followers can apply changes in parallel to catch up faster.
  • Upgrade to cascading: When follower count exceeds ~10-15, move to a cascading topology to offload the leader.
Interview Tip: When asked "how would you replicate to 100 read replicas?", the correct answer is NOT star topology. Propose cascading or use a dedicated replication relay layer. Star is the starting point, not the ending point.

Comparison with Other Topologies

Topology Leader Load Complexity Fault Isolation Best For

Circular Low Medium Low (cycle risk) Multi-master, ring
Cascading Medium Medium High Many followers, cross-region

Common Interview Questions

  • "How does MySQL default replication work?" Answer: Star topology, leader ships binlog events to each follower over a separate connection.
  • "What happens when you add 50 followers to a leader?" Answer: The leader's network and CPU become the bottleneck. Move to cascading replication where intermediate followers relay to leaf followers.
  • "Can one slow follower affect others in star replication?" Answer: No. Each replication stream is independent. A slow follower only affects its own lag, not others.
  • "How would you design replication for a global read-heavy service?" Answer: Use star within a region, cascading across regions. Leaders ship to regional relay followers, which fan out to local read replicas.

What to Skip

Avoid going deep into:
  • Specific MySQL CHANGE MASTER TO syntax or PostgreSQL connection strings. The interviewer wants architecture, not CLI commands.
  • SSL/TLS configuration for replication connections. Unless the question is specifically about security.
  • Replication filter syntax details. The concept of filtering is important; the exact syntax is not.
