4.2.2 Multi-Leader Replication

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand when and why you need multiple leaders, the fundamental problem of write conflicts in a multi-writer system, and the strategies for resolving them. This model is essential for multi-datacenter architectures and collaborative applications.

Definition

Multi-leader replication (also called multi-master or active-active) allows more than one node to accept write operations. Each leader simultaneously acts as a leader for its local clients and as a follower for the other leaders. Changes made on one leader are asynchronously replicated to all other leaders.

Write Conflict Resolution Strategies

The defining challenge of multi-leader replication is conflict resolution. When two leaders modify the same piece of data concurrently, you need a deterministic way to decide the final state.

1. Last-Write-Wins (LWW)

Last-Write-Wins assigns a timestamp to each write and discards the one with the older timestamp. The most common approach, but also the most dangerous because it silently drops data.

Implementation options for timestamps:

Timestamp SourceProsCons
Wall clock time (client or server)Simple to implementClock skew can cause incorrect ordering; NTP drift
Logical timestamps (Lamport clocks, version vectors)No clock dependencyCannot capture real-time ordering intent
Hybrid (logical + physical)Best of both worldsMore complex implementation
Warning: Last-write-wins can silently lose writes in a way that is very hard to detect. If a clock is skewed, a recent write might be discarded in favor of an older one. Use LWW only when data loss is acceptable (e.g., cached data, analytics events, user profile fields that get overwritten frequently).

2. Custom Conflict Resolution

Instead of discarding one write, merge them according to application-specific logic:

  • Application-level merge: The application provides a merge function. For example, a shopping cart merge: if two users add different items to the same cart concurrently, the merge includes both items. If the same item is modified, keep the higher quantity.
  • Conflict-free Replicated Data Types (CRDTs): Mathematical data structures designed to always merge without conflicts. Examples: G-Counter (grow-only counter), OR-Set (observed-remove set), LWW-Element-Set.
  • Operational Transformation (OT): Used in collaborative editing (Google Docs style). Each operation is transformed against concurrent operations to produce a consistent result. Complex but well-studied.
StrategyData Loss RiskComplexityBest For
Last-Write-WinsHigh (silent drops)LowAnalytics, caches, user profiles
Custom merge functionLow (app-defined)MediumShopping carts, preferences
CRDTsNone (by construction)HighCounters, sets, registers
Operational TransformationNoneVery HighCollaborative text editing

Use Cases

1. Multi-Datacenter Deployments

This is the most common use case. Each data center has its own leader to avoid cross-DC latency on every write. Inter-DC replication is asynchronous.

Interview Tip: When asked about multi-DC architecture, multi-leader is often the right answer. Compare it with the alternative: a single global leader (all writes go to one DC, causing high write latency for remote users). Multi-leader trades consistency for write latency.

2. Offline Clients and Mobile Devices

Mobile apps need to work offline. The device acts as a local leader, accepting writes while offline. When connectivity returns, changes sync back to the server (which acts as another leader). Conflict resolution merges the offline changes with any concurrent server-side changes.

Offline-First with Multi-Leader: ──────────────────────────── Phone (Offline) ┌────────────────┐ │ Local Leader │ │ ────────────── │ │ Write: Add │ │ item "X" to │ │ cart │ │ │ │ Queued writes │ └───────┬────────┘ │ │ (reconnects to network) │ ▼ ┌────────────────┐ │ Server Leader │ │ ────────────── │ │ May have other │ │ writes that │ │ conflict │ │ │ │ RESOLVE & │ │ MERGE │ └────────────────┘

3. Collaborative Editing

Applications like Google Docs, Figma, or Notion use multi-leader principles at the document level. Multiple users edit the same document simultaneously, and their changes must be merged. Google Docs uses Operational Transformation; Figma uses CRDTs.

Common Interview Questions

  1. "When would you use multi-leader instead of single-leader replication?"
    Answer: "Multi-leader is appropriate when: (1) You have multiple data centers and need low-latency writes in each DC without cross-DC round trips. (2) You have offline clients that need to accept writes locally. (3) You are building a collaborative application where multiple users edit the same data concurrently. The tradeoff is that multi-leader introduces write conflicts that single-leader avoids entirely, so the conflict resolution strategy must be carefully designed."
  2. "How do you handle write conflicts in a multi-leader system?"
    Answer: "There are several strategies: (1) Last-Write-Wins -- simplest but silently drops data; acceptable for analytics or frequently-overwritten fields. (2) Custom application-level merge -- e.g., merging shopping carts by union rather than overwriting. (3) CRDTs -- mathematical data structures that guarantee conflict-free merges. (4) Operational Transformation -- used in collaborative editing. The choice depends on the data semantics and what 'correct' means for the application."
  3. "What is a CRDT? When would you use one?"
    Answer: "CRDT stands for Conflict-free Replicated Data Type. It is a mathematical structure where all concurrent updates can be merged without conflicts by construction. Examples include G-Counters (grow-only counters), PN-Counters (increment and decrement), and OR-Sets (observed-remove sets). I would use CRDTs when: the data model is relatively simple (counters, sets, registers), I need strong eventual consistency guarantees, and I cannot tolerate silent data loss from last-write-wins."

What to Skip

Do not go deep into:
  • CRDT mathematics and formal proofs -- know what CRDTs are and when to use them, not the algebraic foundations.
  • Operational Transformation algorithm internals (the Google Wave OT algorithm is fascinating but not interview-relevant).
  • Specific multi-leader database configuration (CouchDB replication configuration, Galera cluster setup).
  • Vector clock implementation details -- know the concept, not the code.
  • Sync vs. async multi-leader replication internals -- the conflict resolution strategies matter more.
