5.6.8 Paxos

Difficulty: ⭐⭐⭐⭐⭐ Interview Frequency: ★★★★☆
Goal: Understand the Paxos consensus algorithm at a conceptual level. Know the roles (proposer, acceptor, learner), the prepare/promise/accept/learned protocol flow, and how it guarantees safety. Be able to compare Paxos with Raft and explain why Paxos is hard to understand. Know Multi-Paxos and real-world usage (Google Chubby, Spanner).

Definition

Paxos is a family of consensus protocols developed by Leslie Lamport (1989, published 1998). It is the theoretical foundation for consensus in distributed systems. Paxos guarantees that a value chosen by the protocol is the one with the highest proposal number that received a majority of promises, ensuring that all nodes agree on a single value despite concurrent proposers and node failures.

Why Paxos Is Hard to Understand

The original Paxos paper ("The Part-Time Parliament," Lamport, 1989) was rejected multiple times and rewritten. Lamport later published a more formal version ("Paxos Made Simple," 2001), but even that is notoriously difficult to follow. The difficulty comes from:

  • Abstract presentation: The algorithm is described in terms of proposers, acceptors, and learners with no obvious mapping to real system components.
  • Single-decree vs. Multi-decree: The basic protocol decides a single value. Extending it to decide a sequence of values (Multi-Paxos) is left as an exercise.
  • Minimalist design: Paxos makes the minimum assumptions needed for safety, which means it has fewer implementation hints than Raft.
  • Concurrency: Multiple proposers can compete, leading to complex interleaving scenarios.
Lamport's own admission: "There is no known Paxos algorithm, only a family of algorithms that meet the Paxos specification." This ambiguity is part of why understanding Paxos is challenging. In contrast, Raft provides a single, concrete algorithm.

Roles

Role Responsibility Typical Mapping
Proposer Proposes values and drives the consensus process Client or leader node
Acceptor Votes on proposals (promises and accepts) Server nodes in the cluster
Learner Learns the chosen value (passive observer) Replicas, clients, or non-voting nodes

In practice, a single node often plays multiple roles: a server is both an acceptor and a learner, and the leader is both a proposer and an acceptor.

The Paxos Protocol

Paxos uses a two-phase protocol to decide a single value. Each value slot (called an "instance") is decided independently.

Paxos Protocol (Single Decree) ────────────────────────────── Phase 1: PREPARE / PROMISE (Numbering) ────────────────────────────────────── Proposer Acceptor 1 Acceptor 2 Acceptor 3 │ │ │ │ │──PREPARE(n=42)─────→│ │ │ │ │ │ │ │ │ no higher n? │ │ │←──PROMISE(n=42, │ │ │ │ accepted=none)──│ │ │ │ │ │ │ │──PREPARE(n=42)────────────────────────→│ │ │ │ │ │ │ │ │ no higher n? │ │ │ │ but accepted │ │ │ │ (n=30, v=X) │ │←──PROMISE(n=42, │ │ │ │ accepted={30:X})──────────────────│ │ Phase 2: ACCEPT / ACCEPTED (Value Selection) ──────────────────────────────────────────── Proposer sees accepted value X with n=30. Proposer MUST propose X (not its own value!) │ │ │ │ │──ACCEPT(n=42, v=X)─→│ │ │ │ │ n=42 is highest │ │ │←──ACCEPTED──────────│ │ │ │ │ │ │ │──ACCEPT(n=42, v=X)────────────────────→│ │ │ │ │ │ │ │ │ n=42 is highest │ │ │ │ │ │←──ACCEPTED────────────────────────────│ │ Quorum of acceptors accepted (n=42, v=X) → VALUE X IS CHOSEN Phase 3: LEARN (Notification) ───────────────────────────── Acceptors notify learners of the chosen value.

Protocol Rules

Phase 1: Prepare/Promise

  1. A proposer picks a unique proposal number n and sends PREPARE(n) to a majority of acceptors.
  2. An acceptor that receives PREPARE(n) responds with PROMISE if: (a) n is greater than any previous proposal number it has seen, AND (b) it promises not to accept any proposal with a lower number. The promise includes any previously accepted proposal (if any).

Phase 2: Accept/Accepted

  1. If the proposer receives promises from a majority: it picks the value to propose. If any acceptor had previously accepted a value, the proposer MUST use the value with the highest proposal number. Otherwise, it may propose its own value.
  2. The proposer sends ACCEPT(n, v) to all acceptors (or just the majority that promised).
  3. An acceptor accepts if n is the highest proposal number it has seen, and sends ACCEPTED(n, v).
Critical safety rule: In Phase 2, if the proposer discovers that another value was previously accepted (with a lower proposal number), it must propose that previously accepted value, NOT its own value. This is the key to Paxos's safety guarantee -- it ensures that only one value can be chosen per instance.

Multi-Paxos

Single-decree Paxos is impractical for real systems because each value requires two round trips (prepare + accept). Multi-Paxos optimizes by electing a distinguished proposer (a "leader") that can skip Phase 1 for subsequent values:

Multi-Paxos with Leader ───────────────────────── Without leader (Single Paxos per value): Value 1: PREPARE → PROMISE → ACCEPT → ACCEPTED (2 round trips) Value 2: PREPARE → PROMISE → ACCEPT → ACCEPTED (2 round trips) Value 3: PREPARE → PROMISE → ACCEPT → ACCEPTED (2 round trips) With leader (Multi-Paxos): Value 1: PREPARE → PROMISE → ACCEPT → ACCEPTED (2 round trips, elect leader) Value 2: ACCEPT → ACCEPTED (1 round trip!) Value 3: ACCEPT → ACCEPTED (1 round trip!) Value 4: ACCEPT → ACCEPTED (1 round trip!) The leader reuses its proposal number, so it only needs Phase 1 once. As long as the leader is stable, all subsequent values are decided in a single round trip.

Google Chubby and Spanner

Paxos is used extensively in Google's infrastructure:

  • Google Chubby (2006): A lock service that uses Paxos for replication. Chubby provides distributed locking, leader election, and small-file storage. Used by GFS, Bigtable, and MapReduce for coordination.
  • Google Spanner (2012): A globally distributed database that uses Paxos for replication within each datacenter. Spanner extends Paxos with TrueTime (atomic clocks + GPS) to provide external consistency across datacenters.
Interview note: Mentioning that Google uses Paxos in production (Chubby, Spanner) demonstrates practical knowledge. Be prepared to explain why they chose Paxos over Raft (Paxos was designed earlier, and Google built significant infrastructure around it before Raft existed).

Paxos vs. Raft: Summary

Dimension Paxos Raft
First described 1989 (published 1998) 2014
Understandability Notoriously difficult Designed for understandability
Leader requirement Weak (any proposer can propose) Strong (all entries through leader)
Log structure Flexible (no required ordering) Strict (leader-ordered replicated log)
Multi-value optimization Multi-Paxos (elect stable proposer) Built-in (leader persists across entries)
Implementation correctness Many buggy implementations Reference implementation exists
Primary users Google (Chubby, Spanner) etcd, Consul, CockroachDB, TiKV

Common Interview Questions

  • Explain the Paxos consensus algorithm.
  • What are the three roles in Paxos and what do they do?
  • Walk me through the prepare/promise/accept/learned protocol.
  • Why must a proposer use a previously accepted value in Phase 2?
  • What is Multi-Paxos and how does it optimize performance?
  • Compare Paxos and Raft. Why is Raft easier to understand?
  • What real systems use Paxos?
  • What happens if two proposers compete in Paxos?
  • Why can't a basic Paxos instance guarantee liveness?
  • What is the relationship between Paxos and two-phase commit?

What to Skip

Do NOT try to reproduce Lamport's proof of Paxos correctness -- no interviewer expects that. Skip Fast Paxos, Byzantine Paxos (BPaxos), Egalitarian Paxos (EPaxos), and other Paxos variants unless specifically asked. Skip the details of Google Spanner's TrueTime integration. Do not attempt to implement Paxos from scratch in an interview (even Google engineers have found this challenging). Focus on the single-decree protocol flow, Multi-Paxos concept, and the comparison with Raft.
