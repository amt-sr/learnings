5.3.2 Eventual Consistency

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand eventual consistency as the weakest useful consistency model, how replicas converge over time, what factors affect convergence time, and when eventual consistency is the right choice.

Definition

Eventual consistency guarantees that if no new writes are made, eventually all reads will return the last written value. Replicas converge to the same state over time, but there is no guarantee of when convergence will happen or what values will be seen in the interim.

Key Point: "Eventually" is deliberately vague. It could be milliseconds or hours. The guarantee is only that convergence WILL happen — not WHEN. A system designer must understand convergence time factors to make informed decisions.

How Eventual Consistency Works

Timeline of an eventual consistency scenario: t0: Client A writes X=5 to Replica 1 (US-East) Replica 1: X=5 │ Replica 2: X=1 (EU-West) │ Replica 3: X=1 (APAC) │ │ t1: Client B reads X from Replica 2 → gets X=1 (STALE) Replica 1: X=5 │ Replica 2: X=1 │ Replica 3: X=1 │ │ t2: Replication: R1 → R2 (async) Replica 1: X=5 │ Replica 2: X=5 │ Replica 3: X=1 │ │ t3: Client C reads X from Replica 2 → gets X=5 (CONVERGED for R2) Replica 1: X=5 │ Replica 2: X=5 │ Replica 3: X=1 │ │ t4: Replication: R1 → R3 (async) Replica 1: X=5 │ Replica 2: X=5 │ Replica 3: X=5 === All replicas converged at t4 === Note: Between t0 and t4, reads could return either X=1 or X=5 depending on which replica was contacted.

Factors Affecting Convergence Time

Factor Impact Example
Network latency Slower convergence Cross-region replication (US to Asia) takes longer
Replication factor More replicas = slower convergence RF=5 converges slower than RF=3
Write rate High write rate can outrun convergence A hot key with 10K writes/sec may never converge
Conflict resolution Complex resolution slows convergence Last-write-wins is fast; merge-based is slow
Topology Varies Ring topology (gossip) vs. star topology (leader)
Partition handling Partitions stop convergence entirely During partition, divergent writes accumulate

Real-World Examples

System Eventual Consistency Mechanism Convergence Time
DynamoDB (eventual mode) Async replication across AZs Seconds (within region)
Cassandra Gossip protocol for anti-entropy Milliseconds to seconds
DNS TTL-based cache expiry + zone transfer Seconds to 48 hours (TTL-dependent)
CouchDB Multi-master with revision tree merging Seconds (replication lag)
Instagram feed Fan-out on write with eventual delivery Milliseconds to seconds

Eventual Consistency at Scale: Social Feed Example

Social Media Feed (e.g., Instagram): User A posts photo ──→ Write to Post Store (strong) │ ▼ Fan-out Service / | \ ▼ ▼ ▼ Inbox 1 Inbox 2 Inbox 3 (eventual — async) (User A) (Follower) (Follower) │ │ │ ▼ ▼ ▼ User A sees immediately Followers see within their own post seconds (fan-out latency) If fan-out is slow: - Followers see stale feed (missing the new post) - Acceptable: social feed is not a banking transaction - Next page refresh or pull-to-refresh resolves it

Eventual vs. Strong Consistency

Property Eventual Consistency Strong Consistency
Read guarantee May return stale data Always returns latest write
Write latency Low (no sync replication) High (sync replication required)
Availability High (any replica can serve) Lower (leader required)
Conflict resolution Required (divergent writes) Not needed (single source of truth)
Complexity Higher (application must handle stale reads) Lower (system handles it)
Use cases Social feeds, caches, IoT, analytics Banking, inventory, config, locking
Interview Tip: When asked "how would you design Twitter/Instagram?", eventual consistency is the right answer for feeds. Explain WHY (low latency at write time, fan-out on write, followers see posts within seconds). Then explain how you handle the edge cases (read-repair, manual refresh). This demonstrates pragmatic trade-off reasoning.

Common Interview Questions

  1. "What is eventual consistency and when would you use it?"
  2. "How long is 'eventually'? What factors affect convergence time?"
  3. "What happens when two clients write to different replicas simultaneously?"
  4. "Design a system that uses eventual consistency. How do you handle stale reads?"
  5. "Compare eventual vs. strong consistency for a social media feed."
  6. "How does DynamoDB implement eventual consistency?"

What to Skip

Avoid in interviews:
  • Gossip protocol internals (SWIM, PHI accrual failure detector details)
  • Merkle tree anti-entropy implementation specifics
  • CRDT (Conflict-free Replicated Data Type) internals unless you propose them as a solution
  • Read-repair mechanism internals beyond the basic concept
