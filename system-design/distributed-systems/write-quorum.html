Write Quorum

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand:

  • What a write quorum is and how it ensures data durability
  • The W + R > N rule and why it guarantees consistency
  • How write quorum size affects latency, durability, and availability
  • Real-world configurations in DynamoDB and Cassandra

Definition

A write quorum (W) specifies the minimum number of replicas that must acknowledge a write before it is considered successful. In a system with N replicas, a write must be stored on at least W nodes.

Write Quorum (W=2, N=3): Client writes "user:123 = Alice" │ ├──▶ Replica A (ack ✓) ├──▶ Replica B (ack ✓) ← W=2 satisfied └──▶ Replica C (may or may not receive) Write succeeds after 2 acknowledgments. Replica C may receive the write asynchronously.

W + R > N Rule

The fundamental quorum rule: if W + R > N, then every read is guaranteed to see the latest write. This is because the read set (R nodes) and write set (W nodes) must overlap in at least one node.

W + R > N guarantees read sees latest write: N=3, W=2, R=2 → 2 + 2 = 4 > 3 ✓ ┌─────────┬─────────┬─────────┐ │ Node A │ Node B │ Node C │ │ W: ✓ │ W: ✓ │ │ │ R: ✓ │ R: ✓ │ │ └─────────┴─────────┴─────────┘ ↑ Read from A or B → guaranteed to see latest write (Both A and B have the data) N=3, W=1, R=1 → 1 + 1 = 2 ≤ 3 ✗ ┌─────────┬─────────┬─────────┐ │ Node A │ Node B │ Node C │ │ W: ✓ │ │ │ │ │ R: ✓ │ │ └─────────┴─────────┴─────────┘ Read from B → might NOT see the write on A! (B might have stale data)

Quorum Trade-offs

ConfigurationConsistencyAvailabilityLatencyDurability
W=N (write all)StrongestLowestHighestStrongest
W=N/2+1 (majority)StrongModerateModerateGood
W=1 (write one)WeakestHighestLowestWeakest
Key trade-off: Higher W means stronger consistency and durability, but higher write latency (must wait for more replicas) and lower availability (more replicas must be up to accept writes). Lower W means faster writes and higher availability, but weaker consistency guarantees.

Common Configurations

# DynamoDB
# Default: N=3 (implied), consistent reads use R=W=quorum
PutItem  → W = quorum (2 of 3)
GetItem  → R = quorum (2 of 3) for consistent reads
GetItem  → R = 1 for eventual consistent reads (faster)

# Cassandra
# Write consistency levels
WRITE  CL=QUORUM  → W = ceil(N/2) + 1
WRITE  CL=ALL     → W = N (strongest, slowest)
WRITE  CL=ONE     → W = 1 (fastest, weakest)
WRITE  CL=LOCAL_QUORUM → W = ceil(rack_nodes/2) + 1

Write Path in Quorum Systems

Client → Coordinator Node → N replicas 1. Client sends write to coordinator 2. Coordinator sends write to all N replicas 3. Wait for W acknowledgments 4. If W acks received → success 5. If timeout before W acks → write fails (unavailable) Failed replicas receive write asynchronously (via hinted handoff or anti-entropy)

Common Interview Questions

  • "What is a write quorum?"
    The minimum number of replicas (W) that must acknowledge a write before it succeeds. Combined with read quorum (R), the rule W + R > N ensures every read sees at least one node with the latest write, guaranteeing strong consistency.
  • "What happens if W=1?"
    Write succeeds after just one replica acknowledges. Fastest writes, highest availability (only 1 of N nodes needs to be up), but weakest consistency — a read from a different node may return stale data. Used when write performance matters more than immediate consistency.
  • "How does W + R > N guarantee consistency?"
    If W replicas have the latest write and R replicas are read, W + R > N means R and W must overlap in at least one node. That overlapping node has the latest write, so the read will return the most recent data.

What to Skip

  • Byzantine quorum systems (assume malicious nodes)
