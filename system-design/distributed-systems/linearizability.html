5.3.4 Linearizability

Difficulty: ⭐⭐⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Master linearizability as the strictest consistency model, understand the real-time ordering requirement, know how to implement it, and be able to compare it with all other consistency models in the taxonomy.

Definition

Linearizability guarantees that every operation appears to take effect atomically at some instant between its invocation and its response. There exists a total order of all operations that is consistent with real-time (wall-clock) ordering: if operation A completes before operation B begins, then A must appear before B in the total order.

The Linearization Point: Every operation has a "linearization point" — a single instant in real time where the operation takes effect. For a read, it is the moment it returns a value. For a write, it is the moment the value is committed. This point must fall between the operation's start and end.

How Linearizability Works

Real-time timeline with linearization points: Client A: write(X=1) ═══════[LP]═══════════→ (LP = linearization point) Client B: write(X=2) ═══════[LP]════→ Client C: read(X)═════════→ returns X=2 Client D: read(X)═══════════════════════════→ returns X=2 Linearizable total order: write(X=1) → write(X=2) → read→ read(X=2) Every read after the LP of write(X=2) sees X=2. The total order respects real-time ordering. === NON-linearizable example === Client A: write(X=1) ═══════════════════════→ Client B: read(X)═════════→ returns X=2 ← IMPOSSIBLE if linearizable! Client C: write(X=2) ════════→ Why non-linearizable: - A's write completes after B's read starts - But B reads X=2 which was written AFTER B's read completed - No valid linearization point for write(X=2) before B's read

How to Implement Linearizability

Approach Description Cost
Single Leader All operations go through one node (leader). Reads must go through leader to see latest writes. Leader bottleneck, failover downtime
Consensus Protocol Raft, Paxos, ZAB — all nodes agree on every operation via majority vote. Majority round-trips per operation
Multi-Leader with Synchronization Spanner's TrueTime — uses synchronized physical clocks to order operations without coordination for reads. Expensive infrastructure (GPS, atomic clocks)
Two-Phase Commit (2PC) Coordinator asks all participants to prepare, then commit. Used for transactions across replicas. Blocking, high latency
Interview Tip: The key question to test linearizability: "Can a read return a value that was written after the read started?" If yes, the system is NOT linearizable. This simple mental model helps you reason about any consistency model quickly.

The Cost of Linearizability

Fundamental Limit (Gilbert & Lynch): Linearizability requires coordination. If there is a network partition, a linearizable system must either block or error — it cannot remain available. This is the CAP theorem: you cannot have linearizability + availability during a partition.
  • Latency: At least one round-trip to a quorum per operation (cannot be masked)
  • Availability: Partitioned nodes cannot serve linearizable reads or writes
  • Throughput: Single-leader bottleneck limits write throughput
  • Geographic: Cross-datacenter linearizability requires either high-latency coordination or expensive infrastructure (Spanner)

Common Interview Questions

  1. "What is linearizability and how does it differ from sequential consistency?"
  2. "Can you implement linearizability without a leader? How?"
  3. "What is the cost of linearizability? Why can't we always have it?"
  4. "Is ZooKeeper linearizable? Under what conditions?"
  5. "Can a system be linearizable and available during a partition?"
  6. "Compare linearizability with all other consistency models."
  7. "What is a linearization point? Explain with an example."
  8. "How does Spanner achieve linearizability across datacenters?"

What to Skip

Avoid in interviews:
  • Herlihy & Wing's formal definition with opacity and locality properties
  • Linearizability checking algorithms (WGL algorithm, PROVE)
  • Non-blocking linearizable data structure implementations (lock-free queues, CAS-based registers)
  • Distinction between linearizability and strict serializability (too academic for most interviews)
