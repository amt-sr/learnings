5.3.7 Read Your Writes

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand the read-your-writes consistency guarantee, common implementation strategies, its relationship to session consistency, and when it is the appropriate level of consistency for a system.

Definition

Read-your-writes (RYW) guarantees that after a client writes a value, any subsequent read by the same client will reflect that write (or a more recent one). The guarantee applies only to the writing client — other clients may not immediately see the write.

Intuition: "After I update my profile picture, I immediately see the new picture. My friends might see the old one for a while, but I always see my own latest version." This is the read-your-writes guarantee — it is user-centric, not system-centric.

How It Works

Read-Your-Writes Scenario: Client A writes name="Alice" ──→ Replica 1 (US-East) │ ▼ async replication Replica 2 (EU-West) — still has old name Client A reads name ──→ MUST go to Replica 1 (or wait for Replica 2 to sync) → Returns "Alice" ✓ (saw own write) Client B reads name ──→ Can go to Replica 2 → Returns old name ✗ (stale — OK under read-your-writes!) === RYW only guarantees the WRITER sees their own writes === Violation of RYW: Client A writes name="Alice" ──→ Replica 1 Client A reads name ──→ Replica 2 → returns old name ✗ → VIOLATION: Client A did not see their own write!

Implementation Strategies

1. Route Writes and Reads to the Same Replica

Approach: Session affinity (sticky sessions) Client A ──→ Load Balancer ──→ always Replica 1 Client B ──→ Load Balancer ──→ always Replica 2 Pros: Simple, no coordination Cons: Unbalanced load, session breaks on replica failure

2. Read from the Leader

Approach: All reads go through the leader after a write Client A writes ──→ Leader (Replica 1) → sync to followers Client A reads ──→ Leader (Replica 1) → guaranteed latest Pros: Always correct, no session management Cons: Leader bottleneck, higher latency for cross-region clients

3. Track Timestamps / Version Numbers

Approach: Client tracks its last write timestamp Client A writes name="Alice" at t=100 → Server returns: { value: "Alice", timestamp: 100 } → Client stores: last_write_ts = 100 Client A reads name: → Client sends: read(name, min_timestamp=100) → Replica must have applied all writes up to t=100 before responding → If replica is behind, server waits or redirects Pros: Works without sticky sessions, resilient to replica failure Cons: Requires timestamp metadata, server must wait if replica is behind
Approach Latency Availability Complexity Best For
Sticky Sessions Low High Low Web apps with stateful sessions
Read from Leader Higher Lower Low Leader-based systems (etcd, MongoDB)
Timestamp Tracking Medium Medium Higher Systems with version metadata (DynamoDB, Spanner)

Relationship to Other Consistency Models

Hierarchy: Read-your-writes is implied by (and weaker than):
  • Linearizability — all clients see latest (stronger than RYW)
  • Causal consistency — all causal dependencies preserved (stronger than RYW)
  • Session consistency — includes RYW plus monotonic reads/writes (stronger than RYW alone)
Read-your-writes is stronger than:
  • Eventual consistency — no guarantees about when reads see writes
Interview Tip: Read-your-writes is the most commonly needed consistency guarantee in practice. Most user-facing applications need RYW even when they do not need full strong consistency. When asked "what consistency does your system need?", start with RYW and explain whether you need more (causal, linearizable) or if RYW is sufficient.

Real-World Examples

  • DynamoDB: Supports read-your-writes when using ConsistentRead=true or reading from the leader
  • MongoDB causal sessions: Tracks clusterTime per operation to guarantee RYW within a session
  • Google Spanner: Reads at a timestamp at least as recent as the client's last write (bounded staleness reads)
  • Git: After you push, your next fetch reflects your push (RYW)

Common Interview Questions

  1. "What is the read-your-writes consistency guarantee?"
  2. "How would you implement read-your-writes in a distributed system?"
  3. "Compare sticky sessions vs. read-from-leader for implementing RYW."
  4. "Is read-your-writes enough for a banking system? Why or why not?"
  5. "How does RYW relate to session consistency?"

What to Skip

Avoid in interviews:
  • Specific client SDK implementation details (how DynamoDB client tracks timestamps)
  • Session store implementation (Redis session management)
  • Cookie-based vs. token-based session management (web framework detail)
  • Read repair mechanism internals (beyond the concept)
