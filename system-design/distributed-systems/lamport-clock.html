Lamport Clock

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Master Lamport's algorithm for logical timestamps: the rules, execution on a multi-process diagram, correctness properties, and critically -- what it cannot do.

Definition

A Lamport clock (also called a Lamport timestamp) is a simple, counter-based logical clock algorithm introduced by Leslie Lamport in his seminal 1978 paper. Each process maintains a single integer counter. The algorithm ensures that if event A causally precedes event B, then Lamport timestamp of A is strictly less than that of B.

Historical significance: Lamport's 1978 paper "Time, Clocks, and the Ordering of Events in a Distributed System" is one of the most cited papers in computer science. It established the theoretical foundation for all subsequent work on logical time, ordering, and consensus.

Algorithm Rules

The Three Rules

  1. R1 -- Local event (or send): Increment the counter before the event executes.
    lc = lc + 1
  2. R2 -- Receive message: When process Pj receives a message from Pi with timestamp ts, set local counter to the maximum of local counter and received timestamp, then increment:
    lc = max(lc, ts) + 1
  3. R3 -- Attach to messages: Every message sent includes the sender's current Lamport timestamp.
Key insight: Rule R2 is the crucial one. The max() operation ensures the receiver "catches up" to any causally prior events it has not yet seen. The +1 ensures the receive event itself gets a new, unique timestamp.

Example: Three Processes

Time → P0 P1 P2 | | | a: lc=1 | | | | | b: lc=2 | | | | | c: lc=3 ---------> | | | d: lc=max(0,3)+1=4 | | | | | e: lc=5 --------> | | | f: lc=max(0,5)+1=6 | | | | | g: lc=7 | | | | | h: lc=8 | | | | <--------- | | i: lc=max(3,5)+1=6 | | | | | j: lc=7 | | Events and their Lamport timestamps: a(1), b(2), c(3), d(4), e(5), f(6), g(7), h(8), i(6), j(7) Causal chains: a → b → c → d → e → f → g → h (chain from P0 to P1 to P2) a → b → c → d → e → i → j (chain back to P0)

Properties

Property 1: Causality Preservation

If A → B, then C(A) < C(B) -- where C(event) is the Lamport timestamp and → is the happens-before relation. This is the correctness guarantee of the algorithm.

Proof sketch: Consider each case of the happens-before relation:

  • Same process: LC monotonically increases, so consecutive events always have increasing timestamps
  • Send-receive: Receiver sets LC = max(local, ts) + 1 >= ts + 1 > ts, so receive timestamp > send timestamp
  • Transitivity: follows from the above two cases by induction

Property 2: Converse is NOT True

If C(A) < C(B), it does NOT imply A → B. This is the fundamental limitation of Lamport clocks. Events with ordered timestamps may be concurrent (no causal relationship).
Counterexample: P0 P1 | | a: lc=1 | | b: lc=1 | c: lc=2 | d: lc=3 e: lc=2 | | | C(b) = 1 < C(e) = 2, but b and e are CONCURRENT. There is no message or chain of messages between b and e. Lamport clock cannot detect this -- it sees C(b) < C(e) and wrongly suggests b happened before e.

Property 3: Total Order via (timestamp, process_id)

By breaking ties with process ID, we get a consistent total order across all processes:

total_order(A, B) = C(A) < C(B)  OR  (C(A) == C(B) AND pid(A) < pid(B))

This total order is consistent (all processes agree on it) but not causal -- it imposes an arbitrary order on concurrent events.

Common Interview Questions

  1. "Walk me through Lamport's algorithm." -- State the three rules: increment on local/send, max+1 on receive, always attach timestamp to messages. Walk through the three-process example.
  2. "If C(A) < C(B), does A causally precede B?" -- No. This is the most important thing to know about Lamport clocks. Give a concrete counterexample with two processes doing independent local events.
  3. "How do you create a total order from Lamport timestamps?" -- Break ties with process ID: (timestamp, pid) ordered lexicographically. This is consistent across all processes but arbitrary for concurrent events.
  4. "What is the space complexity of a Lamport clock?" -- O(1) per process (single integer). Compare to vector clocks at O(N).
  5. "How does Lamport clock compare to vector clock?" -- Lamport is simpler (O(1) space) but cannot detect concurrency. Vector clocks can detect concurrency but require O(N) space per process.
  6. "Can you use Lamport clocks for a distributed lock?" -- Yes, the Ricart-Agrawala algorithm uses them to order lock requests. Each request carries a Lamport timestamp, and processes defer to requests with higher timestamps.

What to Skip

Depth to avoid in interviews:
  • Formal proof of the clock condition (Lamport's original paper does this, but you only need the intuition)
  • Implementation details of specific Lamport clock libraries
  • The connection between Lamport clocks and the Byzantine Generals problem
  • Lamport's Bakery algorithm (a mutual exclusion algorithm, not directly about logical clocks)
  • Optimizations like buffered timestamps or piggybacking
