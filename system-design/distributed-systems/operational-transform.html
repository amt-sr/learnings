Operational Transform

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★☆☆

Goal

Understand Operational Transform (OT) -- the algorithm behind Google Docs and other collaborative editors. Learn how it transforms concurrent operations to maintain consistency, why it requires a central server, and how it compares to CRDTs.

Definition

Operational Transform (OT) is a conflict resolution algorithm for real-time collaborative editing. When two users make concurrent edits to the same document, OT transforms each operation against the other so that both users end up with the same document state. The transformation function adjusts operation positions and parameters to account for concurrent changes.

Key insight: OT's core idea is simple: if two users concurrently edit position 5 in a document, and one user inserts 3 characters before position 5, then the other user's operation at position 5 must be shifted to position 8. This "shifting" is the transformation.

How OT Works

The Transform Function

Given two concurrent operations op1 and op2, the transform function T(op1, op2) produces two new operations:

  • op1' -- op1 transformed against op2 (op1 adjusted to account for op2)
  • op2' -- op2 transformed against op1 (op2 adjusted to account for op1)

After applying both op1 and op2' (or op2 and op1'), the document reaches the same state.

Example: Two Concurrent Inserts

Initial document: "Hello World" Index: 01234567890... Client A: inserts "Beautiful " at position 6 Operation A: insert("Beautiful ", pos=6) Client B: inserts "Great " at position 6 (concurrent, didn't see A's op) Operation B: insert("Great ", pos=6) Both operations target the same position, but they're concurrent. Without OT, Client A would see "Hello Beautiful Great World" Client B would see "Hello Great Beautiful World" → INCONSISTENT With OT -- transform(opA, opB): opA': insert("Beautiful ", pos=6) → stays at 6 (A came first in our order) opB': insert("Great ", pos=15) → shifted by len("Beautiful ") = 9 Client A: apply A then B' → "Hello Beautiful Great World" Client B: apply B then A' → "Hello Beautiful Great World" → CONSISTENT (both converge to same document) The transform function T adjusts positions based on the TYPE and POSITION of concurrent operations. Insert shifts subsequent inserts; delete reduces positions of subsequent operations.

OT Architecture

OT systems typically follow a centralized server architecture:

Client A Server Client B | | | |-- op1 (local) --------->| | | | | | |<--------- op2 (local) ---| | | | | | transform(op1, op2) | | | | |<-- op2' (transformed) ---| | | | | | |-- op1' (transformed) --->| | | | Server is the SINGLE SOURCE OF TRUTH: 1. Maintains the "official" operation history 2. Transforms incoming ops against all concurrent ops 3. Broadcasts transformed ops to all clients 4. Clients apply ops in the server's order Clients maintain: - Local state (document + operation history) - Pending ops (sent to server, awaiting acknowledgment) - Server state (last known server revision)
Critical requirement: OT needs a central authority (the server) that defines a total order of operations. This is different from CRDTs, which can work in a fully decentralized manner. The server's role in OT is non-trivial -- it maintains the canonical operation history and resolves conflicts.

Insert vs Insert

insert(pos, "X") vs insert(pos, "Y") (same position)X at pos, Y at pos + len(X) (left bias wins)

Insert vs Delete

ScenarioTransform
insert(pos, "X") vs delete(pos, n) (insert inside delete range)Insert shifts the delete, delete "swallows" insert position
insert(pos, "X") vs delete(pos2, n) (pos < pos2)Insert shifts the delete: delete at pos2 + len(X)
insert(pos, "X") vs delete(pos2, n) (pos > pos2 + n)Insert shifts: insert at pos - n

Delete vs Delete

ScenarioTransform
delete(pos1, n1) vs delete(pos2, n2) (overlapping ranges)Shorten second delete to remove only un-deleted range
delete(pos1, n1) vs delete(pos2, n2) (non-overlapping)Shift positions based on which came first
Complexity explosion: The number of possible transformation combinations grows with the number of operation types. With just insert and delete, there are 9 transform cases (3x3). Adding formatting operations (bold, italic, etc.) creates dozens of cases. Getting the transform function correct for all combinations is notoriously difficult.

Industry trend: Newer collaborative systems increasingly favor CRDTs over OT. Figma uses CRDTs. Notion uses CRDTs. OT is still used by Google Docs (the original), but the complexity of implementing correct transform functions makes CRDTs more attractive for new systems.

Challenges with OT

  • Transform function correctness: The biggest challenge. There are dozens of edge cases, and getting all transform pairs correct is extremely difficult. Google's own OT implementation had subtle bugs that persisted for years.
  • Operational composition: When the server groups multiple operations together, composing them and then transforming against individual client operations is complex.
  • Undo/Redo: Standard undo/redo doesn't work directly with OT because the operations have been transformed. You need a special "inverse transform" for undo.
  • Client-server sync protocol: Managing the sync state (pending ops, acknowledged ops, server revision) is complex. Race conditions between local and remote ops are tricky.
  • Scalability: The server must process and transform every operation for every connected client, making it a bottleneck at scale.

Common Interview Questions

  1. "How does Google Docs handle concurrent edits?" -- Using Operational Transform. Each edit is an operation (insert/delete/format at a position). When concurrent edits arrive at the server, the server transforms them against each other to ensure consistency, then broadcasts the transformed versions to all clients.
  2. "What is the transform function in OT?" -- A function T(op1, op2) that takes two concurrent operations and produces transformed versions that, when applied in either order, produce the same result. It adjusts positions and parameters to account for concurrent changes.
  3. "Why does OT require a central server?" -- The server maintains the canonical operation history and provides a total ordering. Without a central authority, there's no consistent way to determine which operations are concurrent and how to transform them.
  4. "Compare OT and CRDTs for collaborative editing." -- OT requires a central server and has complex transform functions but is memory-efficient. CRDTs are decentralized and mathematically guaranteed to converge but use more memory. OT is used in Google Docs; CRDTs are used in Figma and newer collaborative tools.
  5. "How would you handle undo in an OT system?" -- Standard undo doesn't work because operations have been transformed. You need to compose an "undo operation" by inverting the original operation, then transform it against all operations that happened since.
  6. "What are the challenges of implementing OT?" -- The transform function has many edge cases (dozens of operation type combinations), undo/redo is non-trivial, the sync protocol is complex, and the server can become a bottleneck at scale.
