5.6.2 Leader Election

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★
Goal: Understand why distributed systems need leaders, how leader election works across different algorithms, and be able to walk through the Raft election process in detail. Know the failure modes and how to handle split votes.

Definition

Leader election is the process by which nodes in a distributed system agree on a single coordinator (leader) responsible for making decisions, ordering operations, or managing replication. It is one of the most common instances of the consensus problem in real systems.

Why Systems Elect Leaders

Many distributed algorithms simplify coordination by designating a single leader:

  • Single point of decision-making: The leader decides the order of operations, avoiding the complexity of coordinating multiple decision-makers.
  • Simplified replication: Followers replicate from the leader. No complex merge logic needed.
  • Client routing: Clients can send all writes to the leader, simplifying the client protocol.
  • Failure detection: The leader monitors followers; if the leader fails, a new election is triggered.
Tradeoff: Leader-based systems create a single point of throughput (all writes go through the leader) and a potential availability bottleneck if election latency is high. Leaderless systems (Dynamo) avoid this but sacrifice strong consistency.

Bully Algorithm

The simplest election algorithm. Assumes all nodes have unique IDs and communicate via reliable message passing.

Process

  1. When node N detects the leader has failed, it sends an ELECTION message to all nodes with higher IDs.
  2. If N receives no response within a timeout, it becomes the leader and sends a COORDINATOR message to all nodes.
  3. If a higher-ID node responds, that node takes over the election. N stops.
  4. The highest-ID alive node always wins.
Bully Algorithm Election ─────────────────────── Nodes: [1] [2] [3] [4] [5] (IDs) Leader (5) fails! │ Node 3 initiates election: [3] ──ELECTION──→ [4] ──ELECTION──→ [5] ✗ (dead) │ │ [4] responds "OK" │ │ │ Node 4 takes over: │ [4] ──ELECTION──→ [5] ✗ [4] ──COORDINATOR──→ [1] [2] [3] Result: Node 4 is the new leader (highest ID alive)

Weaknesses: O(n) messages per election. Requires reliable message delivery. The "bully" name comes from the fact that the highest-ID node always wins regardless of who started the election.

Raft-Style Election (Terms, Timeouts, Votes)

Raft uses a term-based election mechanism with randomized timeouts to ensure safety and liveness.

Key Concepts

  • Term: A monotonically increasing epoch number. Each term has at most one leader.
  • Election timeout: A randomized timer (150-300ms in Raft). If a follower receives no heartbeat within this timeout, it becomes a candidate.
  • Vote: A follower grants its vote to at most one candidate per term (first-come-first-served with log completeness check).
Raft Election Process ───────────────────── Term 1 Term 2 ───── ───── Leader [A] sends heartbeats [A] crashes [B] timeout fires ──→ [B] becomes Candidate (term 2) │ ├──RequestVote──→ [C] (C votes for B ✓) ├──RequestVote──→ [D] (D votes for B ✓) └──RequestVote──→ [E] (E votes for B ✓) B gets majority → B becomes Leader (term 2) B sends AppendEntries (heartbeat) to all followers State transitions: Follower ──timeout──→ Candidate ──majority votes──→ Leader ↑ │ └──higher term discovered──┘

Election Rules

Rule Description
One vote per term A follower votes for at most one candidate per term (voting for self counts)
Log completeness A candidate must have a log at least as complete as the voter's to receive a vote
Term always increases Nodes never decrease their current term; they adopt the higher term from any message
Leader uniqueness At most one leader per term (enforced by the one-vote rule + majority)

ZooKeeper Leader Election

ZooKeeper uses Zab (ZooKeeper Atomic Broadcast) for leader election, which is similar to Raft in concept but differs in implementation:

  1. Nodes create an ephemeral sequential znode under /election.
  2. The node with the smallest sequence number is the leader.
  3. Each node watches the znode with the next-lower sequence number.
  4. If the watched node disappears (leader crash), the watcher triggers and the next node becomes leader.
ZooKeeper vs. Raft: ZooKeeper's election uses the ZooKeeper service itself (bootstrap problem solved by the ensemble), while Raft's election is self-contained. ZooKeeper's approach leverages ephemeral nodes which automatically disappear when a session expires, providing built-in failure detection.

Split Vote Handling

A split vote occurs when no candidate receives a majority of votes in an election round. This can happen if two candidates start elections in the same term (e.g., because their election timeouts fired at nearly the same time).

Split Vote Scenario (5 nodes, majority = 3) ───────────────────────────────────────── Node [B] becomes candidate (term 4) Node [C] becomes candidate (term 4) │ │ B gets votes: B, D C gets votes: C, E Total: 2 votes (< 3) Total: 2 votes (< 3) ❌ No majority → Election fails → Wait for new timeout → New term (5)

Strategies to Avoid Split Votes

  • Randomized election timeouts: Raft's primary mechanism. By randomizing the timeout (e.g., 150-300ms), the probability of two nodes timing out simultaneously is low.
  • Pre-vote phase: A candidate first checks if it can win before actually incrementing the term. Avoids disrupting the current leader.
  • Priority-based voting: Some implementations give preference to nodes with more complete logs or higher priority (etcd's leadership transfer).

Election Timeout Tuning

Parameter Too Low Too High
Election timeout Unnecessary elections, leader churn, wasted bandwidth Slow failover, longer unavailability after leader crash
Heartbeat interval High network traffic, CPU overhead Followers may timeout prematurely, triggering unnecessary elections
Randomization range Higher probability of split votes Slower elections (worst-case timeout is large)
Best practice: Heartbeat interval should be significantly less than the minimum election timeout (e.g., heartbeat every 50ms, election timeout 150-300ms). The ratio of broadcast-time to election-timeout should be at least 10:1 for good reliability.

Common Interview Questions

  • Walk me through how Raft leader election works.
  • What happens if two nodes start an election at the same time?
  • How does Raft prevent split votes?
  • Why does randomized election timeout help?
  • How does ZooKeeper handle leader election differently from Raft?
  • What happens to a running system when the leader crashes?
  • How would you tune election timeouts for a system spanning multiple datacenters?
  • Can a follower reject a vote request? Under what conditions?

What to Skip

Do NOT implement the Bully algorithm in interview code -- it is rarely used in production. Skip Ring algorithm elections unless specifically asked. Do not memorize ZooKeeper's internal implementation details beyond the ephemeral sequential znode approach. Focus your preparation on Raft-style elections, which are the most commonly expected at FAANG interviews.
