5.3.6 Session Consistency

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand session consistency as a pragmatic consistency model that provides guarantees within a client session, how it is implemented with sticky sessions, and why it is a practical choice for web applications.

Definition

Session consistency (also called session guarantee or client-centric consistency) provides consistency guarantees only within the context of a session — a logical connection between a client and the system. Different sessions may see different views of the data, but a single session never observes inconsistencies.

Key Concept: A "session" is an abstraction that groups a sequence of operations from a single client. In web applications, a session typically corresponds to a user's login session (authenticated interactions). Session consistency says: "Within your session, things look consistent. Across sessions, they may not."

How Session Consistency Works

Web Application with Session Consistency: Client A (Session A) Client B (Session B) │ │ ▼ ▼ ┌──────────┐ ┌──────────┐ │ LB / GW │ │ LB / GW │ └────┬─────┘ └────┬─────┘ │ session_id=A │ session_id=B │ (sticky) │ (sticky) ▼ ▼ ┌──────────┐ ┌──────────┐ │ Replica 1│ │ Replica 2│ │ (always) │ │ (always) │ └──────────┘ └──────────┘ Session A: All reads/writes go to Replica 1 → Reads always see writes from the same session → Reads always see the same state (no going backward) Session B: All reads/writes go to Replica 2 → Same guarantee within Session B BUT: Replica 1 may have stale data compared to Replica 2 Client A and Client B may see different views of the same data → This is OK under session consistency!

Session Guarantees (Terry et al.)

Session consistency is actually a family of four related guarantees:

Guarantee Description Protects Against
Read Your Writes After writing, you read your own write Not seeing your own updates
Monotonic Reads Never read older data within a session Time appearing to go backward
Monotonic Writes Writes are applied in order within a session Write reordering
Writes Follow Reads Writes go to at least as up-to-date a replica as the last read Writing stale data

Session consistency typically implies all four of these guarantees within a session. This is why it is stronger than any individual guarantee but weaker than causal consistency (which applies across sessions).

Implementation: Sticky Sessions

Sticky Session (Session Affinity): Client connects with session_id Load balancer hashes session_id → always routes to same replica Option 1: Source IP affinity hash(client_ip) % N → replica Problem: IP may change (mobile), shared IPs (NAT) Option 2: Cookie-based affinity LB sets cookie on first request Subsequent requests carry cookie → routes to same replica Most common in web applications Option 3: Consistent hashing with session key hash(session_id) → ring position → replica Supports resharding (minimal session migration) Trade-offs: ┌──────────────────────────────────────────────────┐ │ Sticky Sessions │ │ + Simple to implement │ │ + No coordination overhead │ │ + Low latency (local reads) │ │ - Unbalanced load (hot sessions) │ │ - Replica failure = session disruption │ │ - No cross-session consistency guarantees │ └──────────────────────────────────────────────────┘

Real-World Examples

System Session Consistency Mechanism
MongoDB (causal consistency) Causal sessions: read-after-write within a session using operationTime tracking
AWS ElastiCache Sticky sessions with Redis — session affinity for web app state
Nginx / HAProxy Cookie-based or IP-based sticky session routing
AWS DynamoDB Session consistency not directly supported, but can emulate with read-after-write pattern
Interview Tip: When asked "how do you handle consistency for a web application?", session consistency via sticky sessions is often the pragmatic answer. It is cheap to implement, provides a good user experience (you always see your own writes), and is how most web applications actually work. Just be sure to mention the downsides (replica failure, unbalanced load).

Session vs. Causal Consistency

Property Session Consistency Causal Consistency
Scope Within a single session/client Across all clients (globally)
Ordering guarantee Per-session only All causally related operations
Implementation Sticky sessions (simple) Vector clocks (complex)
Cost Low Medium (metadata overhead)
Cross-session anomaly Possible (sessions see different data) Not possible (causal order preserved globally)

Common Interview Questions

  1. "What is session consistency and how does it differ from causal consistency?"
  2. "How do you implement session consistency in a web application?"
  3. "What are the downsides of sticky sessions?"
  4. "How does MongoDB implement causal session consistency?"
  5. "When would you choose session consistency over causal consistency?"

What to Skip

Avoid in interviews:
  • Load balancer configuration syntax (nginx upstream blocks, etc.)
  • Session storage backends (Redis vs. Memcached for session store)
  • Cookie security (HttpOnly, Secure flags) — that is a web security topic
  • Session fixation / hijacking attack details
