5.6.6 Membership

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆
Goal: Understand the cluster membership problem, how nodes join and leave, and the different approaches to maintaining a consistent membership view. Know how Cassandra, ZooKeeper, and consensus-based systems handle membership. Be able to discuss the tradeoffs between gossip-based and consensus-based membership.

Definition

Cluster membership is the problem of maintaining an authoritative, up-to-date list of nodes that are part of a distributed system. Every node must know who the other members are so that it can route requests, replicate data, and participate in consensus protocols. The membership list must handle nodes joining, leaving voluntarily, and failing unexpectedly.

The Membership Problem

Maintaining membership seems simple -- keep a list of nodes -- but becomes complex when nodes can fail or the network can partition:

  • Node join: How does a new node discover existing members and announce itself?
  • Node leave: How does a node gracefully deregister itself?
  • Node failure: How do remaining nodes detect and remove a failed node?
  • Split brain: During a network partition, how do nodes on each side handle membership?
  • Convergence: How do all nodes agree on the same membership list?
Why membership matters: Incorrect membership leads to routing errors, data inconsistency, and split-brain. If Node A thinks Node B is alive but B has crashed, A will route requests to B that will never be served. If two partitions each believe they are the full cluster, both may accept writes independently, creating unreconcilable conflicts.

Failure Detection + Gossip for Membership

The most common membership approach in modern distributed databases combines failure detection with gossip protocols:

Gossip-Based Membership (Cassandra-style) ───────────────────────────────────────── 1. Seed nodes: Bootstrap with known contact points New Node ──join request──→ Seed Node (known address) │ ├── Returns current membership list └── Starts gossiping new node's info 2. Gossip propagation: Every 1 second, each node: - Picks a random peer - Sends its own state + states of up to 3 other nodes - Receives peer's state - Merges: newer generation timestamp wins 3. Failure detection: Phi accrual detector monitors heartbeat intervals If phi > threshold → mark as SUSPECT After confirmation by others → mark as DEAD 4. State reconciliation: Seed: [A:gen1, B:gen1, C:gen1, D:gen1] Gossip from A: [A:gen2, B:gen1, E:gen1] (A updated, E joined) After merge: [A:gen2, B:gen1, C:gen1, D:gen1, E:gen1] 5. All nodes converge within O(log N) gossip rounds

Generation Timestamps

Each node maintains a generation timestamp that increments every time the node restarts or its state changes significantly. When two gossip messages contain different state for the same node, the one with the higher generation timestamp wins. This prevents stale information from overwriting fresh information.

Ring Membership in Cassandra

Cassandra uses a consistent hashing ring where each node owns a token range. Membership and token assignment are tightly coupled:

Cassandra Ring Membership ─────────────────────────── Token Range: 0 ─────────────────────────────────────── 2^127 Node A: owns [0, 42) ← token = 0 Node B: owns [42, 85) ← token = 42 Node C: owns [85, 170) ← token = 85 Node D: owns [170, 200) ← token = 170 Node E: owns [200, 0) ← token = 200 When Node C joins: 1. C contacts seed node via gossip 2. C is assigned token 85 (or auto-selected with vnodes) 3. C gossips: {C: token=85, gen=1} 4. Other nodes update their routing tables 5. C streams data from its predecessors for its token range When Node C fails: 1. Phi accrual detector marks C as SUSPECT 2. Gossip propagates SUSPECT status 3. After timeout, C marked as DEAD 4. Other nodes stream replicas to fill C's range 5. Hinted handoff re-delivers writes that arrived during failure

SWIM Protocol

SWIM (Scalable Weakly-consistent Infection-style Process Group Membership) combines failure detection with gossip-based membership dissemination. Used in HashiCorp Memberlist and Consul.

How SWIM Works

  1. Probing: Each node periodically probes a random target via a direct ping.
  2. Indirect probing: If the direct ping fails, the prober asks K random nodes to also ping the target (indirect probes). This reduces false positives.
  3. Suspicion: If all probes fail, the target is marked as SUSPECT. This suspicion is disseminated via gossip.
  4. Dissemination: Membership updates (joins, leaves, suspicions) are piggybacked on probe messages and gossiped through the cluster.
SWIM Indirect Probing ────────────────────── Node A probes Node B: │ [A]──ping──→[B] ✗ no response │ [A]──ping-req──→[C]──ping──→[B] ✗ no response [A]──ping-req──→[D]──ping──→[B] ✗ no response │ [A] marks [B] as SUSPECT [A] gossips: "B is suspected" │ All nodes eventually learn B is suspected After timeout, B promoted to DEAD Advantage: Reduces false positives because B is checked from multiple network paths. If A cannot reach B due to a local network issue, C or D likely can.

Consensus-Based Membership

Systems like etcd, ZooKeeper, and CockroachDB use consensus protocols (Raft or Zab) to maintain membership. The membership list is stored as part of the replicated log, ensuring strong consistency.

Viewstamped Replication for Membership

Viewstamped Replication (VR) was one of the first protocols to explicitly address membership changes. The current membership is called a view. View changes occur when:

  • A node joins or leaves the cluster.
  • The leader is suspected of failure.
  • A network partition is detected.

View changes are committed through the consensus protocol, so all nodes agree on the current view. Raft implements this as joint consensus -- a two-phase membership change where the cluster transitions through an intermediate configuration that includes both old and new members.

Gossip vs. Consensus Membership Comparison

Aspect Gossip-Based (Cassandra) Consensus-Based (etcd)
Consistency Eventual (may see stale membership briefly) Strong (all nodes agree via Raft)
Join speed Fast (contact seed + gossip) Slower (must be committed to Raft log)
Failure detection speed Fast (phi accrual) Slower (based on election timeouts)
Split brain risk Higher (each partition has its own view) Lower (only majority partition can commit)
Scalability Thousands of nodes Typically 3-9 nodes
Best for Data stores, large clusters Configuration stores, coordination services

Common Interview Questions

  • How does a new node join a Cassandra cluster?
  • What happens when a node fails in a gossip-based membership system?
  • What is the SWIM protocol and how does it reduce false positives?
  • How does etcd handle membership changes?
  • What is a generation timestamp and why is it important?
  • How do you handle a network partition in a membership system?
  • Compare gossip-based and consensus-based membership approaches.
  • What happens to data when a node leaves a Cassandra ring?

What to Skip

Do NOT memorize Cassandra's gossip message format or the internal data structures. Skip the details of the HyParView protocol for gossip overlay networks unless asked. Do not dive deep into Raft's joint consensus implementation details (covered in the Raft section). Skip the specific token allocation strategies (vnode random allocation, rack-aware allocation) unless the question specifically asks about data distribution.
