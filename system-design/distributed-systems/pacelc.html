5.2 PACELC

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand the PACELC theorem as an extension of CAP that captures the trade-off between latency and consistency even when the network is operating normally (no partition).

Definition

PACELC stands for: If there is a Partition (P), the system trades off Availability (A) vs. Consistency (C); Else (E), when running normally, the system trades off Latency (L) vs. Consistency (C).

CAP only tells you what happens during a partition. PACELC adds the second half: even when everything is fine, do you choose low latency or strong consistency? This is the trade-off that most systems actually make in practice.

Why PACELC Matters: Partitions are rare in practice. Most design decisions are made during normal operation. A CP system (like Spanner) that provides consistency during partitions must also accept higher latency during normal operation due to cross-replica coordination. An AP system (like Cassandra) avoids this coordination cost.

Interview Tip: Mentioning PACELC instead of just CAP immediately signals senior-level understanding. Most interviewers expect candidates to only know CAP. Demonstrating knowledge of PACELC shows you understand that consistency is a spectrum, not a binary choice, and that most decisions are made during normal operation, not during partitions.

PA/EL vs PC/EC vs PA/EC

PA/EL (e.g., Cassandra default): ┌────────────────────────────────────────────┐ │ Partition? → Stay Available (serve stale) │ │ Normal ops? → Minimize Latency (no sync) │ │ │ │ Reads/writes hit nearest replica directly │ │ Async gossip for replication │ │ Lowest latency, highest availability │ └────────────────────────────────────────────┘ PC/EC (e.g., MongoDB, HBase): ┌────────────────────────────────────────────┐ │ Partition? → Preserve Consistency (block) │ │ Normal ops? → Preserve Consistency (sync) │ │ │ │ All writes go through leader/follower sync │ │ Reads from leader or sync'd followers │ │ Highest consistency, highest latency │ └────────────────────────────────────────────┘ PA/EC (e.g., Cassandra with QUORUM): ┌────────────────────────────────────────────┐ │ Partition? → Stay Available (serve stale) │ │ Normal ops? → Preserve Consistency (sync) │ │ │ │ Normal: QUORUM read/write for consistency │ │ Partition: Fallback to local (ANY) │ │ Flexible: strong when healthy, weak when │ │ under partition │ └────────────────────────────────────────────┘

Why This Matters for System Design

In a system design interview, when you choose a consistency model, you are implicitly choosing from the PACELC menu:

  • "We use eventual consistency" = PA/EL: Fast reads and writes, accept stale data
  • "We use strong consistency with leader election" = PC/EC: Correct data, accept higher latency
  • "We use read-after-write consistency" = somewhere between: Partial guarantees for specific operations

The key insight is that latency and consistency are two ends of a single spectrum. You cannot have both simultaneously — every additional consistency guarantee requires an additional network round-trip, which adds latency.

Common Interview Questions

  1. "What is PACELC and how does it extend CAP?"
  2. "Why is CAP incomplete for real-world system design decisions?"
  3. "Classify Cassandra with QUORUM in the PACELC framework."
  4. "When would you choose PA/EL vs PC/EC for a new service?"
  5. "Can a system offer different PACELC profiles for different operations?"
  6. "How does Cosmos DB implement tunable consistency in PACELC terms?"

What to Skip

Avoid in interviews:
  • Formal proofs of PACELC (the Abadi 2012 paper's theoretical framework)
  • Detailed analysis of Byzantine fault tolerance in PACELC context
  • Comparing PACELC to lesser-known alternatives like BASE, HAT, or CALM theorem
  • Specific consistency level tuning parameters for Cassandra unless asked
