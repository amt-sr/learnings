5.3.5 Causal Consistency

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand causal consistency as a model that preserves the order of causally related operations, how it is implemented using vector clocks, and why it strikes an important balance between strong consistency and eventual consistency.

Definition

Causal consistency guarantees that if operation A causally precedes operation B (i.e., B depends on or was influenced by A), then all nodes see A before B. Causally independent operations can appear in any order on different nodes.

What is "Causal" Precedence? Operations are causally related if:
  • B reads a value written by A
  • B happens-after A on the same client (client ordering)
  • A and B are both causally related to C (transitivity)
If none of these hold, A and B are causally independent (concurrent).

How Causal Consistency Works

Chat / Messaging Example: Alice and Bob are chatting through different replicas: t0: Alice writes "Hi!" (Replica 1, US) t1: Alice reads "Hi!" (Replica 1, US) t2: Bob reads "Hi!" (Replica 2, EU) — must see "Hi!" before any reply t3: Bob writes "Hello!" (Replica 2, EU) — causally depends on "Hi!" t4: Alice reads "Hello!" (Replica 1, US) — must see "Hi!" before "Hello!" Causal dependency chain: "Hi!" → read by Bob → "Hello!" === Causally Consistent === All nodes agree: "Hi!" comes before "Hello!" === Concurrent (Independent) Operations === Carol writes "Hey!" at the same time as Alice's "Hi!" These are NOT causally related — different orderings OK: Node 1 may show: "Hi!" then "Hey!" Node 2 may show: "Hey!" then "Hi!" Both orderings are valid under causal consistency.

Vector Clocks: Implementing Causal Consistency

Vector Clock Example (3 replicas: R1, R2, R3): Alice writes "Hi!" at R1: R1: [1, 0, 0] R2: [0, 0, 0] R3: [0, 0, 0] Event E1: VC = [1, 0, 0] Replicate E1 to R2: R1: [1, 0, 0] R2: [1, 0, 0] R3: [0, 0, 0] Bob writes "Hello!" at R2 (read "Hi!" first, so depends on E1): Event E2: VC = [1, 1, 0] (R2's counter incremented after seeing R1's) Causal check: E2.VC = [1, 1, 0] dominates E1.VC = [1, 0, 0] → E2 causally depends on E1 ✓ Carol writes "Hey!" at R3 (no prior reads): Event E3: VC = [0, 0, 1] Causal check: E3.VC = [0, 0, 1] vs E1.VC = [1, 0, 0] Neither dominates the other → E1 and E3 are CONCURRENT === Vector Clock Comparison Rules === VC_A dominates VC_B if every element of A >= B and at least one > B If neither dominates: operations are concurrent

Vector Clock Properties

  • Each node maintains a vector clock with one entry per known node
  • On local event: increment own counter
  • On receive: merge (element-wise max) with received vector clock
  • Comparison enables causal ordering detection
  • Size grows with number of replicas (not ideal for very large clusters)

Causal Consistency in Practice

System Causal Consistency Implementation
Amazon DynamoDB Global tables use causal consistency across regions for last-writer-wins
Cassandra Client-side timestamps (LWW) approximate causal ordering
CouchDB Revision trees track causal history for conflict detection
MongoDB (causal session) Causal consistency within a session (since v3.6)
Google Sheets Operational Transformation with causal ordering for collaborative editing

Where Causal Fits in the Hierarchy

Linearizability (strongest) │ │ stronger ▼ Sequential Consistency │ │ stronger ▼ Causal Consistency ← You are here │ │ stronger ▼ Session Consistency / Read-Your-Writes / Monotonic Reads │ │ stronger ▼ Eventual Consistency (weakest) Causal consistency: - Stronger than session/read-your-writes (applies globally, not per-client) - Weaker than sequential (no total order for concurrent operations) - Significant practical benefit: prevents confusing reordering

Trade-offs

Aspect Causal Consistency
Concurrency High — concurrent operations can proceed without coordination
Confusing reordering Prevented — causal dependencies always respected
Metadata overhead Vector clocks grow with cluster size (O(N) per event)
Client-visible anomalies None for causally related operations
Implementation complexity Medium — vector clock management, merge on read/write
Interview Tip: When designing a collaborative system (e.g., Google Docs, Figma, chat), causal consistency is often the right answer. Explain that you need it because: (1) users should not see replies before the original message, (2) edits should be applied in causal order, (3) but you do not need the full cost of linearizability for every keystroke.

Common Interview Questions

  1. "What is causal consistency and how does it differ from eventual consistency?"
  2. "How do vector clocks implement causal consistency?"
  3. "Give an example of a system that needs causal consistency."
  4. "What happens when two concurrent operations conflict in a causally consistent system?"
  5. "How does causal consistency compare to linearizability?"
  6. "What is the overhead of vector clocks? How do they scale?"

What to Skip

Avoid in interviews:
  • Version vectors vs. vector clocks distinction (implementation detail)
  • Dotted version vectors or hybrid logical clocks
  • CRDT internals (OR-Set, LWW-Element-Set, etc.) unless you propose them
  • Formal causal consistency models (PBS, CPC, etc.)
