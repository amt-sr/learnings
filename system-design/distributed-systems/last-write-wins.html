Last Write Wins (LWW)

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆

Goal

Understand Last Write Wins conflict resolution: how it works, why it is deceptively simple, the data loss it causes under concurrent writes, clock skew problems, and when it is the right choice for a system.

Definition

Last Write Wins (LWW) is a conflict resolution strategy where, when two replicas have different values for the same key, the one with the higher timestamp wins. All other values are silently discarded. It is the simplest conflict resolution strategy and is used by many databases (including Cassandra, Riak, and many key-value stores) as the default or optional behavior.

Key insight: LWW trades correctness for simplicity. It always converges (no manual conflict resolution needed) but can silently lose data when concurrent writes occur. Whether this is acceptable depends entirely on your application.

How LWW Works

Replica A Replica B | | | write(X="hello", t=10) | | | | write(X="world", t=12) | | |<--- sync: B sends (X="world", t=12) --->| | | | X="world" wins (t=12 > t=10) | "hello" is PERMANENTLY LOST | | The write "hello" at t=10 was valid data written by a user. Because t=12 happened to have a higher timestamp (possibly just due to clock skew, not causal ordering), "hello" is silently discarded.

The Fundamental Problem: Clock Skew

In a distributed system, clocks are not perfectly synchronized. This means a later write with an earlier timestamp (due to clock skew) can be incorrectly discarded:

Client A (clock is 50ms behind) writes "cancel_order" at t=0950 Client B (clock is 50ms ahead) writes "confirm_order" at t=1050 Both write concurrently to the same key "order_status". LWW picks "confirm_order" (t=1050 > t=0950). But causally, Client A's cancel came FIRST. The timestamp difference is entirely due to clock skew, not causal ordering. LWW made the WRONG decision and an order was confirmed when it should have been cancelled. This is a real, production-impacting failure mode.
Critical point for interviews: LWW with wall-clock timestamps is fundamentally unreliable for causal ordering. The timestamp reflects clock skew, not causality. Two writes that happen within the clock skew window (~50-250ms with NTP) can be resolved in the wrong order.

LWW + Dotted Version Vectors

Some systems improve LWW by combining it with version vectors for causality detection:

  • If version vectors show writes are causally ordered, the later one wins (correct)
  • If version vectors show writes are concurrent, fall back to timestamp comparison (may lose data)
  • This is better than raw LWW but still loses data on concurrent writes

When Is LWW Acceptable?

ScenarioLWW Suitable?Reason
User profile updatesYesMost updates are non-concurrent; losing an old name/bio is tolerable
Configuration valuesYesSingle admin typically changes config; conflicts are rare
Session tokensYesEach user generates their own token; overwriting is expected
Bank account balanceNoEvery write must be counted; losing any write = financial loss
Inventory countNoConcurrent decrements must all be counted
Chat messagesNoLosing a message is never acceptable
Audit logsNoEvery entry must be preserved for compliance
Shopping cart contentsDependsIf clients always read-modify-write, LWW may suffice
Interview tip: A good answer about LWW always includes a discussion of what data you're protecting. "It depends" is a valid answer -- the key is explaining which scenarios tolerate data loss and which do not.

Strategies to Mitigate LWW Data Loss

  1. Use client-generated timestamps -- Let the client include the timestamp from their own clock. This ensures the order reflects the client's intent (assuming single-client writes per key).
  2. Add a tiebreaker -- When timestamps are equal (within epsilon), use a node ID or hash of the value as a deterministic tiebreaker. This ensures deterministic convergence even with identical timestamps.
  3. Use monotonic counters instead of wall-clock time -- A centralized counter or sequence number provides strict ordering without clock skew issues (but introduces a single point of coordination).
  4. Combine with version vectors -- Detect concurrent writes first; only use LWW as a last resort for true conflicts.
  5. Keep a conflict log -- Record discarded writes for potential manual reconciliation or auditing.

Common Interview Questions

  1. "What is Last Write Wins?" -- A conflict resolution strategy where the write with the higher timestamp wins. Simple, always converges, but can silently lose data from concurrent writes.
  2. "What are the problems with LWW?" -- Clock skew causes wrong ordering decisions; concurrent writes cause silent data loss; no way to detect or recover from lost data.
  3. "When would you use LWW?" -- When conflicts are rare, data loss is tolerable, or the cost of more sophisticated conflict resolution outweighs the benefit. Good for user profiles, config values, cached data.
  4. "How does Cassandra handle conflicts?" -- Cassandra uses LWW by default. The client provides a timestamp with each write. The write with the highest timestamp wins. If two writes have the same timestamp, the one from the replica with the highest replica ID wins.
  5. "How would you design a system that needs to avoid LWW data loss?" -- Use CRDTs for data types that can be merged (counters, sets, registers). Use version vectors to detect conflicts and prompt users to resolve. Use OT or merge strategies for collaborative data.
  6. "What happens if two LWW writes have the same timestamp?" -- Need a deterministic tiebreaker: node ID, hash of value, or client ID. Without a tiebreaker, the result is non-deterministic and replicas may diverge.
