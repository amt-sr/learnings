Merge Strategies

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand custom application-level merge strategies -- the approach taken when generic mechanisms like LWW, CRDTs, and OT are not sufficient. Learn three-way merge, version vectors with merge functions, and how real systems implement domain-specific conflict resolution.

Definition

A merge strategy is a custom, application-defined function that resolves conflicts between concurrent writes. Unlike LWW (which blindly picks one winner) or CRDTs (which use generic mathematical merging), merge strategies use domain-specific logic to combine conflicting updates intelligently. They are the most flexible but also the most complex approach to conflict resolution.

Key insight: Merge strategies sit at the top of the sophistication spectrum for conflict resolution. LWW is the simplest (pick one). CRDTs are data-structure-aware (merge mathematically). OT is operation-aware (transform operations). Merge strategies are application-semantic-aware (use business logic to combine).

Three-Way Merge

The most common merge pattern, borrowed from version control systems like Git:

Base (common ancestor) / \ / \ Branch A Branch B (User 1 edits) (User 2 edits) \ / \ / Three-Way Merge (detect conflicts, apply merge function) Algorithm: 1. Find the common ancestor version (Base) 2. Compare Branch A with Base → changes_A 3. Compare Branch B with Base → changes_B 4. If changes_A and changes_B don't overlap → merge both 5. If changes_A and changes_B overlap → conflict → apply merge function Example: Merging user profiles Base: {name: "Alice", email: "alice@old.com", role: "admin"} Branch A: {name: "Alice Smith", email: "alice@old.com", role: "admin"} Branch B: {name: "Alice", email: "alice@new.com", role: "editor"} changes_A: name changed ("Alice" → "Alice Smith") changes_B: email changed ("alice@old.com" → "alice@new.com"), role changed No overlapping fields → clean merge: Result: {name: "Alice Smith", email: "alice@new.com", role: "editor"}

Conflict Detection in Three-Way Merge

A conflict occurs when both branches modified the same field to different values:

  Base:      {name: "Alice", email: "alice@old.com"}
  Branch A:  {name: "Alice Smith", email: "alice@old.com"}
  Branch B:  {name: "Alice B", email: "alice@new.com"}

  Both changed "name" to different values → CONFLICT on "name"
  email: only B changed → merge B's change

  Merge result: {name: ???CONFLICT???, email: "alice@new.com"}

  → Application must resolve the conflict (prompt user, apply custom logic)
Git analogy: Three-way merge in distributed systems works exactly like Git merge. The "common ancestor" is the last synchronized version. Branches are concurrent updates from different replicas. The merge function is like a .gitattributes merge driver customized for your data type.

Version Vectors + Merge Function

A practical pattern combining version vectors for conflict detection with custom merge functions for resolution:

Data record: { key: "shopping_cart:user_42", value: { items: [ {sku: "A100", qty: 2}, {sku: "B200", qty: 1} ], total: 59.99 }, version: [replica1: 5, replica2: 3, replica3: 2] } Replica 1 update: user adds sku "C300", qty=1 version: [replica1: 6, replica2: 3, replica3: 2] Replica 2 update: user removes sku "B200" version: [replica1: 5, replica2: 4, replica3: 2] Sync: [6,3,2] vs [5,4,2] → INCOMPARABLE → CONCURRENT Merge function (application-defined): - For each SKU: keep the max qty across versions - If item exists in one but removed in another: use "tombstone wins" or "add wins" based on business rules - Recalculate total Result: items: [{sku: "A100", qty: 2}, {sku: "C300", qty: 1}] total: 39.99 version: [replica1: 6, replica2: 4, replica3: 2]

Real-World Examples

Git: The Gold Standard

Git's merge is the most widely understood three-way merge:

  • Uses content-addressable storage to find the common ancestor (merge base)
  • Line-by-line diff to detect conflicts
  • Automatic merge for non-overlapping changes
  • Manual resolution for conflicts (conflict markers in the file)
  • Custom merge drivers (e.g., for .po files, lock files)

Dynamo/Cassandra: Application-Level Merge

When version vectors detect a conflict (concurrent siblings), the application is responsible for merging:

  • Dynamo returns all conflicting versions to the client
  • Client applies domain-specific logic to merge
  • Client writes back the merged result with a new version vector
  • Example: shopping cart merge adds items from both versions, removes items with tombstones

Apache CouchDB: Revision-Based Merge

  • Every document has a revision history (revision tree)
  • Conflicts create branches in the revision tree
  • Application reads all leaf revisions and merges them
  • Write the merged document as a new revision on a new branch
  • Old conflicting revisions can be deleted after merge

iCloud: Three-Way Merge for Contacts/Notes

  • Each device syncs via a central iCloud server
  • Server maintains version vectors per record
  • On conflict, server stores both versions and flags them
  • On next sync, device downloads both and performs three-way merge
  • Merge function is per-field: take newer value for each field independently

Designing a Merge Strategy: Decision Framework

1. What is the data type? ├── Primitive (number, string, boolean) │ → Can you define a merge operation? (max, min, sum, concat) │ → If yes: use that. If no: fall back to LWW. │ ├── Collection (set, list, map) │ → Can elements be merged independently? │ │ → Yes: per-element merge (like shopping cart SKUs) │ │ → No: need CRDT or OT │ └── Is order significant? │ → No: CRDT set or map │ → Yes: CRDT sequence (RGA) or OT │ └── Complex object (nested structure) → Can fields be merged independently? │ → Yes: per-field merge with version vectors │ → No: whole-object merge (three-way merge) └→ Business rules determine the right strategy 2. What is the conflict frequency? ├── Very low: LWW is probably fine (simpler) ├── Moderate: Three-way merge with custom logic └── Very high: CRDTs (automated, no manual resolution) 3. What is the cost of a wrong merge? ├── Low: LWW or simple auto-merge ├── Medium: Three-way merge with conflict queue for review └── High (data loss, financial): Strong consistency (no merge needed)

Common Interview Questions

  1. "How would you handle conflicts in a shopping cart system?" -- Use version vectors to detect concurrent updates. Merge function: for each SKU, take the max quantity across versions. Handle add/remove with business rules (add-wins or remove-wins depending on requirements). This is the Dynamo shopping cart merge pattern.
  2. "Explain three-way merge." -- Find the common ancestor, compute diffs from both branches, merge non-overlapping changes automatically, flag overlapping changes as conflicts for resolution. Analogous to Git merge.
  3. "When would you use a custom merge strategy vs CRDTs?" -- Use custom merge when the data has complex business rules that don't fit standard CRDT operations (e.g., "if item is on sale, use sale price; otherwise, use the price the customer saw at add-time"). CRDTs are better when the merge is purely mathematical (counters, sets).
  4. "How does Dynamo handle shopping cart conflicts?" -- Version vectors detect concurrent writes. All conflicting versions are returned to the client. The client merges: adds items from both versions, respects tombstones for removals, takes max quantity per SKU. Writes back merged result with new version vector.
  5. "Compare three-way merge with OT." -- Three-way merge operates at the data level (comparing states). OT operates at the operation level (transforming operations). Three-way merge needs a common ancestor; OT needs a transformation function. Three-way merge is simpler but coarser-grained.
  6. "How would you design merge for a contact book?" -- Per-field merge with version vectors. For non-conflicting fields, take the newer value. For conflicting fields (e.g., both users changed phone number), either take the newer one or flag for manual resolution. Add phone numbers to the set (union merge). Remove by tombstone.

What to Skip

Depth to avoid in interviews:
  • Git's specific merge algorithms (recursive merge, octopus merge, etc.)
  • Darcs' patch theory (an alternative to OT with interesting theoretical properties)
  • Patience merge, weave merge, and other Git merge strategies
  • CouchDB revision tree internals (B-tree format of revision history)
  • Specific .gitattributes merge driver syntax
  • Three-way merge for binary files (images, PDFs)
