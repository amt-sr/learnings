5.1.1 Consistency (CAP Consistency)

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand what consistency means in the CAP theorem context, how it differs from ACID consistency, and why it is a critical trade-off in every distributed system design decision.

Definition

In the CAP theorem, consistency means that every node in the system sees the same data at the same time. A read always returns the most recent write or an error. Formally, this is atomic consistency (also called linearizability), which is the strongest form of consistency in distributed systems.

Key Distinction: CAP consistency is NOT the same as ACID consistency. ACID consistency refers to database constraints and invariants (e.g., "every transfer debits one account and credits another"). CAP consistency refers to every node agreeing on the same value for a given key at the same point in time.

What CAP Consistency Really Means

Client A writes X=5 ──→ Node 1 (X=5) │ Client B reads X ──→ Node 2 (X=5) ← Consistent: same value │ Client C reads X ──→ Node 3 (X=5) ← Consistent: same value === After write, ALL nodes must agree === Client A writes X=10 ──→ Node 1 (X=10) │ sync required Client B reads X ──→ Node 2 (X=??) ← Without sync: stale read! │ Client C reads X ──→ Node 3 (X=??) ← Without sync: stale read! With CAP Consistency (CP): Node 2 blocks read until X=10 is replicated Node 3 blocks read until X=10 is replicated → Every read returns the latest value (or errors)

Linearizability as the CAP "C"

The CAP theorem's consistency is equivalent to linearizability: every operation appears to take effect atomically at a single point in time. This is the gold standard of consistency guarantees.

  • Single-copy semantics: The system behaves as if there is only one copy of the data
  • Total ordering: All operations can be placed in a single, real-time-ordered sequence
  • No stale reads: A read that begins after a write completes will always see that write

Cost of CAP Consistency

Aspect Impact
Latency Increased — requires coordination between replicas before responding
Availability Reduced — a partitioned node cannot serve reads or writes
Throughput Lower — coordination overhead limits parallelism
Correctness Guaranteed — every client sees the same data

CAP Consistency vs. ACID Consistency

Property CAP Consistency ACID Consistency
Scope Replicas/nodes across the network Within a single transaction
Concern All nodes see the same data simultaneously Data satisfies invariants (constraints, rules)
Example Node A and Node B both return X=5 Bank transfer: debit + credit are both applied
Distributed? Always distributed by definition Can be single-node
Interview Tip: When asked about "consistency" in a system design interview, always clarify whether the interviewer means CAP consistency (replica agreement) or ACID consistency (transactional invariants). This distinction signals deep understanding.

Real-World Examples

  • HBase / ZooKeeper (CP): A leader coordinates all writes. Followers reject reads that are not up-to-date during a leader election.
  • Google Spanner: Uses TrueTime (GPS + atomic clocks) to provide externally consistent (linearizable) transactions across datacenters.
  • etcd: Uses the Raft consensus protocol to ensure all reads go through the leader, guaranteeing linearizability.
CP System Flow (e.g., etcd with Raft): Client ──write(X=5)──→ Leader ──AppendEntries──→ Follower 1 │ → Follower 2 │ → Follower 3 │ │ (Wait for majority ACK) │ ←── ACK from majority ──────┘ │ Client ←── success ──┘ Only AFTER majority commit: Client ──read(X)──→ Leader ──→ returns X=5 (guaranteed latest) During partition: Follower 2 is isolated → CANNOT serve reads (rejects or errors) Follower 2 is isolated → CANNOT serve writes (not the leader)

Common Interview Questions

  1. "What does the 'C' in CAP actually mean?"
  2. "How is CAP consistency different from ACID consistency?"
  3. "What is the cost of strong consistency in a distributed system?"
  4. "Give an example of a CP system and explain why it chose consistency over availability."
  5. "Can a system be fully consistent and fully available? Under what conditions?"
  6. "How does a CP system handle a network partition?"

What to Skip

Avoid in interviews:
  • Deep dives into formal proof of the CAP theorem (Brewer's conjecture, Gilbert/Lynch proof)
  • ACID transaction isolation levels (READ COMMITTED, REPEATABLE READ, etc.) unless the interviewer pivots to databases
  • Multi-version concurrency control (MVCC) internals unless directly relevant
  • Implementation details of specific consensus protocols — that is covered under consensus topics
