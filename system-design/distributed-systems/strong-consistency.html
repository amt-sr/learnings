5.3.1 Strong Consistency

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand what strong consistency means, how it is implemented, its cost in distributed systems, and when to choose it over weaker consistency models.

Definition

Strong consistency (also called linearizability or strict consistency) guarantees that once a write completes, all subsequent reads return that value or a more recent one. The system behaves as if there is a single copy of the data — equivalent to single-copy semantics.

Single-Copy Semantics: A strongly consistent system with N replicas behaves identically to a system with exactly 1 replica. Clients cannot distinguish (in terms of observable behavior) whether they are talking to one node or many.

How Strong Consistency Works

Client A writes X=1 ────────────────────────────→ t0 Client B writes X=2 ────────────────────────────→ t1 (t1 > t0) Client C reads X ──────────────────────────────→ t2 (t2 > t1) Strong Consistency Guarantee: Client C reads X=2 (the most recent write at the time of read) Implementation with Leader: ┌──────────────────────────────────────────────────────┐ │ Leader Node │ │ ┌────────────────────────────────────────────┐ │ │ │ Write X=1 ← Client A (t0) │ │ │ │ Replicate to F1, F2, F3 (wait for ACK) │ │ │ │ Write X=2 ← Client B (t1) │ │ │ │ Replicate to F1, F2, F3 (wait for ACK) │ │ │ │ Read X → Client C (t2) │ │ │ │ Return X=2 (leader has latest) │ │ │ └────────────────────────────────────────────┘ │ │ │ │ │ │ │ ▼ ▼ ▼ │ │ Follower 1 Follower 2 Follower 3 │ │ X=2 (sync) X=2 (sync) X=2 (sync) │ └──────────────────────────────────────────────────────┘

Real-World Examples

System Strong Consistency Mechanism
Google Spanner TrueTime API (GPS + atomic clocks) + Paxos for globally consistent transactions
ZooKeeper ZAB protocol (leader-based) — all reads go through leader for linearizability
etcd Raft consensus — leader serializes all reads and writes
MongoDB (primary) Primary node handles all writes; reads can go to secondaries with configurable acknowledgment
CockroachDB Raft per-range; serializable isolation with distributed transactions

Cost of Strong Consistency

Aspect Impact Why
Latency Higher Writes must be replicated to a quorum before ACK; reads may need to consult leader
Throughput Lower Coordination limits parallelism; single-leader bottleneck
Availability Lower Cannot serve writes (or reads) during leader election / partition
Geographic Reach Limited Cross-region round-trips add latency (Spanner mitigates with TrueTime)
Correctness Maximum All clients always see the latest data
Reasoning Simple Programmers can reason as if there is a single copy
Interview Tip: When designing a system, always ask: "Do I need strong consistency everywhere, or only for specific operations?" Many systems use strong consistency for critical paths (payments, inventory) and weaker consistency for non-critical paths (user profiles, analytics). This mixed approach is practical and shows nuanced thinking.

Strong vs. Eventual Consistency

Strong Consistency: Write(X=5) ──→ ALL replicas ACK ──→ Read(X) = 5 (guaranteed) Timeline: ══════════════════════════════════════ write → replicate → ACK → read = latest Eventual Consistency: Write(X=5) ──→ ONE replica ACK ──→ Read(X) = 5 or stale (maybe) Timeline: ══════════════════════════════════════ write → local ACK → read = maybe latest, maybe old │ (replication happens async) │ ══════════════→ eventually all replicas have X=5

Common Interview Questions

  1. "What is strong consistency and how does it differ from eventual consistency?"
  2. "How do you implement strong consistency in a distributed system?"
  3. "What is the cost of strong consistency?"
  4. "Name systems that provide strong consistency and explain their approach."
  5. "When would you NOT choose strong consistency?"
  6. "Can you have strong consistency across datacenters? How?"

What to Skip

Avoid in interviews:
  • Deep details of specific consensus protocols (Raft/ZAB internals — covered in consensus section)
  • TrueTime implementation details (GPS clock sync, clock uncertainty bounds)
  • Serializability vs. linearizability formal distinction (beyond the basic definition)
  • Specific ZooKeeper session management or watcher mechanics
