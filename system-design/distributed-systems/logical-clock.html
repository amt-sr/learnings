Logical Clock

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★☆

Goal

Understand why physical clocks fail for causal ordering, how logical clocks solve this by tracking causality rather than wall-clock time, and what trade-offs they introduce.

Definition

A logical clock is a mechanism for capturing the causal ordering of events in a distributed system without relying on synchronized physical clocks. Instead of tracking "what time it is," a logical clock tracks "what happened before what." Each process maintains a counter that is incremented on local events and updated upon receiving messages.

Core idea: If event A causally happened before event B (A → B), then the logical clock value of A must be less than the logical clock value of B. This is a necessary but not sufficient condition for detecting causality.

Why Logical Clocks Are Needed

Physical clocks (see 5.4.1 Physical Clock) fail for event ordering because:

  • Clock skew means two nodes can disagree on the current time
  • Network delay means a message can arrive "after" its physical timestamp on the receiver
  • There is no universal time instant that all nodes agree on

Logical clocks solve this by not trying to measure real time at all. Instead, they encode the causal relationships between events directly into the clock values.

The Happens-Before Relation (→)

The happens-before relation, defined by Lamport, is a partial order on events:

  1. Same process: If event a happens before event b in the same process, then a → b
  2. Message passing: If process Pi sends message m (event a) and Pj receives m (event b), then a → b
  3. Transitivity: If a → b and b → c, then a → c
Partial order: The happens-before relation does not order all pairs of events. Two events are concurrent (a || b) if neither a → b nor b → a. Concurrent events are events with no causal relationship -- they could have happened in any order.

Basic Counter-Based Logical Clock

Rules

  • LC1 (Local event): Before executing a local event, increment the local counter: lc = lc + 1
  • LC2 (Send): Before sending a message, increment the counter and attach it to the message: lc = lc + 1; send (message, lc)
  • LC3 (Receive): On receiving a message with timestamp lc_msg, set lc = max(lc, lc_msg) + 1

Simple Example

Process P1 Process P2 lc=0 lc=0 | (local event) | lc = 1 | | | send(msg, lc=1) ----> | receive(msg, lc=1) | lc = max(0, 1) + 1 = 2 | | | (local event) | lc = 3 | | <---- send(reply, lc=3) receive(reply, lc=3) lc = max(1, 3) + 1 = 4 | (local event) lc = 5 Causal order: P1:local(1) → send(1) → P2:recv(2) → P2:local(3) → send(3) → P1:recv(4) → P1:local(5)

How Logical Clocks Capture Causality

The fundamental property is:

Correctness guarantee: If event A happens before event B (A → B), then LC(A) < LC(B). This is always true by construction.

However, the converse is NOT true:

Limitation: If LC(A) < LC(B), it does not mean A → B. Events A and B could be concurrent (independent). A simple counter-based logical clock cannot distinguish between causal and concurrent events when their clock values are ordered.

The Concurrency Blindspot

Process P1 Process P2 Process P3 lc=0 lc=0 lc=0 | (local event) | | lc = 1 | | | (local event) | | lc = 1 | | | (local event) | | lc = 1 P1:local(1), P2:local(1), P3:local(1) -- all have the same timestamp But these events are CONCURRENT (no causal relationship). Even worse: Process P1 Process P2 lc=0 lc=0 (local) lc=1 | (local) lc=2 | (local) lc=3 (local) lc=1 P2:local(1) < P1:local(3) -- but P2:local(1) is NOT causally before P1:local(3). They are concurrent. The simple logical clock cannot detect this.

Common Interview Questions

  1. "Why do we need logical clocks in distributed systems?" -- Physical clocks have unbounded skew and cannot guarantee causal ordering. Logical clocks encode causality directly without relying on synchronized time.
  2. "What is the happens-before relation?" -- A partial order defined by Lamport: same-process ordering, message ordering, and transitivity. It captures causal dependence but leaves concurrent events unordered.
  3. "If LC(A) < LC(B), does that mean A happened before B?" -- No. This is the critical limitation. LC(A) < LC(B) is necessary but not sufficient for A → B. Concurrent events can have ordered clock values.
  4. "How do you create a total order from logical clocks?" -- Append the process ID: order by (lc, pid) lexicographically. This gives a consistent total order but is arbitrary for concurrent events.
  5. "Can logical clocks detect concurrent events?" -- A simple counter-based logical clock cannot. You need vector clocks for that.

What to Skip

Depth to avoid in interviews:
  • Formal proofs of the happens-before relation properties
  • Implementations of logical clocks in specific languages
  • The distinction between strong and weak clocks (advanced topic)
  • Matrix clocks (an extension beyond the scope of most interviews)
  • Fidge's theorem and formal lattice theory
