5.3.3 Sequential Consistency

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★☆☆

Goal

Understand sequential consistency as a consistency model where all operations appear in some total order, but without the real-time ordering requirement of linearizability. Know how it differs from linearizability and when it appears in practice.

Definition

Sequential consistency guarantees that the result of any execution is the same as if the operations of all processors were executed in some sequential order, and the operations of each individual processor appear in the order specified by its program. In other words, there exists some total ordering of all operations that respects per-client program order.

Key Distinction from Linearizability: Sequential consistency does NOT require the total order to respect real-time ordering. Two operations from different clients can appear in any order as long as each client's own operations remain in order. Linearizability additionally requires that if operation A completes before operation B starts, A must appear before B in the total order.

Sequential vs. Linearizability: The Difference

Real-time timeline: Client A: ──write(X=1)───────────────────────────────────→ Client B: ───────────────────────write(X=2)──read(X)──────→ Linearizable result: read(X) must return 2 → A's write completes before B's write starts → So in the total order: write(X=1) before write(X=2) → Read must see X=2 (latest) Sequential (but NOT linearizable) result: read(X) could return 1 → Per-client order preserved: A's write before B's write (OK) → But the total order could be: write(X=2), write(X=1), read(X) → This respects A's order and B's order, but NOT real-time → read(X) returns 1 — sequentially consistent but NOT linearizable ┌───────────────────────────────────────────────────────────┐ │ Linearizability = Sequential + Real-time ordering │ │ │ │ All linearizable executions are sequentially consistent │ │ NOT all sequentially consistent executions are linearizable│ └───────────────────────────────────────────────────────────┘

Formal Properties

  • Total order exists: All operations can be arranged in a single sequential order
  • Per-client order respected: Each client's operations appear in the order that client issued them
  • No real-time requirement: The total order does NOT need to respect wall-clock time
  • Weaker than linearizability: More behaviors are acceptable, allowing more optimizations

Where Sequential Consistency Appears

System / Context Usage
Shared memory multiprocessors Original use case (Lamport, 1979) — CPU memory models
ZooKeeper (non-linearizable reads) When clients read from followers instead of the leader, they get sequential but not linearizable reads
Cassandra with QUORUM Provides ordering but may not respect strict real-time bounds across datacenters
Version control (Git) Commits appear in some total order per branch, but timestamps may be reordered

Practical Implications

Interview Tip: Sequential consistency rarely comes up by name in system design interviews. It is most useful as a concept to contrast with linearizability. If asked "what consistency model does ZooKeeper provide when reading from followers?", the answer is sequential consistency (not linearizable). This demonstrates deep understanding.

Why It Matters

Sequential consistency is important because:

  • It is the weakest model where all nodes agree on a total order of operations
  • It allows reordering optimizations that linearizability does not (e.g., buffering writes, speculative execution)
  • Many real systems that claim "strong consistency" actually provide only sequential consistency
  • It is the basis for understanding many weaker models (causal, eventual)

Common Interview Questions

  1. "What is sequential consistency?"
  2. "How does sequential consistency differ from linearizability?"
  3. "Give an example of a scenario that is sequentially consistent but not linearizable."
  4. "When does sequential consistency appear in real systems?"
  5. "Why is sequential consistency weaker than linearizability?"

What to Skip

Avoid in interviews:
  • Lamport's original formal definition and proof (Lamport, 1979)
  • Memory model details for multi-core CPUs (x86-TSO, ARM weak ordering)
  • Optimizations enabled by sequential consistency (speculative execution, store buffers)
  • Formal verification of sequential consistency (TLA+, model checking)
