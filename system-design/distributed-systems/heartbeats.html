5.6.3 Heartbeats

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★★☆
Goal: Understand how heartbeats provide the foundation for failure detection in distributed systems. Know the relationship between heartbeat intervals, timeouts, and false positive rates. Be able to design a heartbeat system and discuss edge cases like network partitions and GC pauses.

Definition

A heartbeat is a periodic signal sent by one node to another (or broadcast to a group) to indicate that the sender is alive and functioning. The absence of expected heartbeats triggers a failure suspicion, which may initiate recovery actions such as leader election or traffic rerouting.

How Heartbeats Work

Heartbeats are the most basic form of failure detection in distributed systems. A node sends periodic "I'm alive" messages to its monitor. If the monitor does not receive a heartbeat within a configured timeout, it suspects the sender has failed.

Suspect A failed │ → Initiate leader election

Heartbeat Interval and Timeout

The two key parameters in a heartbeat-based failure detector:

  • Heartbeat interval: How often the sender transmits a heartbeat (e.g., every 50ms).
  • Failure timeout: How long the receiver waits before declaring the sender failed (e.g., 150ms).
Configuration Detection Speed False Positive Risk Network Overhead
Frequent heartbeat, short timeout Fast (low latency) High (GC pauses, network spikes) High
Infrequent heartbeat, long timeout Slow (high latency) Low (tolerates temporary delays) Low
Rule of thumb: The failure timeout should be at least 2-3x the heartbeat interval and at least 10x the expected network round-trip time (RTT). For cross-datacenter deployments with 50ms RTT, a heartbeat interval of 100ms and timeout of 300ms is typical.

Piggybacking Heartbeats

In practice, heartbeats are often piggybacked on existing messages rather than sent as standalone packets. In Raft, the leader's AppendEntries RPCs serve as heartbeats -- if there are no log entries to replicate, the leader sends empty AppendEntries messages.

// Raft-style heartbeat piggybacking
type AppendEntriesArgs struct {
    Term     int
    LeaderId int
    PrevLogIndex int
    PrevLogTerm  int
    Entries  []LogEntry  // Empty = pure heartbeat
    LeaderCommit  int
}

Failure Detection via Heartbeats

The simplest heartbeat failure detector is a binary (yes/no) decision: the node is either alive or dead. This is simple to implement but prone to errors:

  • False positive: Declaring a live node as dead (due to GC pause, network congestion, scheduler starvation).
  • False negative: Declaring a dead node as alive (due to a heartbeat arriving just before the timeout from a buffered queue).
Interview tip: When discussing heartbeats, always mention the false positive problem. A false positive is far more dangerous than a false negative -- declaring a healthy leader as dead triggers an unnecessary election, which blocks writes and could cause a split-brain if the "dead" leader is still processing requests.

Phi Accrual Failure Detector

The phi accrual detector (used in Cassandra, Akka, and Amazon Swarm) is a more sophisticated approach that measures the likelihood of failure rather than making a binary decision.

How It Works

  1. Maintain a sliding window of recent heartbeat arrival intervals.
  2. Calculate the mean and standard deviation of these intervals.
  3. When a heartbeat is late, compute phi = -log10(probability of the next heartbeat arriving this late by random chance).
  4. Higher phi = more likely the node has failed.
Phi Accrual Detector ──────────────────── Heartbeat intervals (sliding window): [45, 52, 48, 51, 50, 47, 53, 49, 50, 51] ms Mean: 49.6ms, StdDev: 2.2ms Current time since last heartbeat: ───────────────────────────────── 50ms → phi ≈ 1.0 (normal, probably alive) 60ms → phi ≈ 3.0 (suspicious) 70ms → phi ≈ 5.0 (very suspicious) 80ms → phi ≈ 8.0 (almost certainly failed) 100ms → phi ≈ 12.0 (definitely failed) Threshold: phi > 8 → declare failure Advantage: Adapts to network conditions automatically

The phi accrual detector adapts to changing network conditions: if the network becomes slower, the mean and standard deviation increase, and the detector automatically adjusts its failure threshold. This avoids false positives during transient network degradation.

Relationship to Leader Election

Heartbeats are the direct mechanism that triggers leader election:

  1. The leader sends periodic heartbeats to all followers.
  2. Each follower resets its election timer upon receiving a heartbeat.
  3. If a follower's election timer expires (no heartbeat received), it starts a new election.
  4. The new leader begins sending heartbeats to establish its authority.
Design question to consider: If the leader is under heavy load and its heartbeats are delayed (but the leader is alive), followers may trigger unnecessary elections. This is why heartbeat messages should be prioritized over data replication, and why Raft piggybacks heartbeats on AppendEntries RPCs (so the heartbeat is always the first thing processed).

Edge Cases

  • GC pauses: A long Java/C# garbage collection pause can delay all heartbeats from a node, causing followers to suspect it has failed. Mitigation: use phi accrual detector with a higher threshold, or separate the heartbeat thread from the GC-managed heap.
  • Network partitions: During a partition, the isolated nodes cannot receive heartbeats and will start their own election. This is correct behavior -- the system splits into two partitions, each with its own leader, and only the majority partition can make progress.
  • Clock skew: Heartbeat timeouts should be based on wall-clock time at the receiver, not synchronized clocks. The timeout measures "how long since I last heard from the leader," not "what time is it."

What to Skip

Do NOT deep-dive into the mathematical details of the phi accrual detector's probability distribution. Knowing the concept (phi as a probabilistic measure of suspicion) is sufficient. Skip SWIM protocol details here (covered in the Membership section).
