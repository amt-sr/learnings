Hybrid Logical Clock

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★☆☆

Goal

Understand how Hybrid Logical Clocks combine the best of physical and logical clocks to provide both causal ordering and physical time proximity, without requiring specialized hardware like GPS/atomic clocks.

Definition

A Hybrid Logical Clock (HLC) is a timestamp mechanism that combines a physical clock component with a logical counter component. It provides timestamps that are always close to physical time (so you can reason about real-time deadlines) while also tracking causality (so you can detect event ordering). Proposed by Bengtson and Bortnikov in 2013.

Motivation: Pure physical clocks can't guarantee causality (see 5.4.1). Pure logical clocks lose physical time semantics (see 5.4.2). HLC gives you the best of both worlds in a software-only solution.

HLC Timestamp Structure

Each HLC timestamp is a tuple (l, c, pid):

  • l (logical time) -- the largest physical clock value seen so far by this node (bounded by real time)
  • c (counter) -- a logical counter used to disambiguate events that occur within the same physical time unit
  • pid -- process identifier, for breaking remaining ties
HLC timestamp = (l, c, pid)

  l = max of own physical clock and all l values received
  c = incremented only when physical clock hasn't advanced
  pid = unique process identifier

Comparison:
  (l1, c1) < (l2, c2) iff l1 < l2 OR (l1 == l2 AND c1 < c2)

Algorithm

Initialization

l = 0
c = 0
pt = physical clock of this node

Rules

  1. Send / Local event: 
    pt = read_physical_clock()
if pt > l:
    l = pt
    c = 0
else:
    c = c + 1
return (l, c, pid)
  2. Receive (incoming timestamp msg_l, msg_c): 
    pt = read_physical_clock()
l_new = max(l, msg_l, pt)
if msg_l == l or msg_l == pt:
    c_new = max(c, msg_c) + 1
else:
    c_new = 0
l = l_new
c = c_new
return (l, c, pid)
Intuition: The logical time l tracks the "highest physical time we've heard about." The counter c is used only when we're "stuck" at the same physical time (multiple events happening within the same millisecond/physical tick). This keeps timestamps close to real time while still capturing causality.

Example

Node A (physical clock ~100ms) | Event 1: send pt=100, l=0, c=0 pt > l → l=100, c=0 timestamp: (100, 0, A) | |--- msg(100, 0, A) ---> | | Event 2 (local) Node B (physical clock ~102ms) pt=100, l=100, c=0 Receive msg(100, 0, A) pt == l → c = 1 pt=102, l=0, c=0 timestamp: (100, 1, A) l_new = max(0, 100, 102) = 102 msg_l != l_new → c = 0 l=102, c=0 timestamp: (102, 0, B) | Event 3 (local) | pt=101, l=100, c=1 Event 4: local pt > l → l=101, c=0 pt=102, l=102, c=0 timestamp: (101, 0, A) pt == l → c = 1 timestamp: (102, 1, B) Timeline: (100,0,A) → (102,0,B) [causal: A sent to B] (100,0,A) → (100,1,A) [causal: same node, ordered] (101,0,A) || (102,0,B) [concurrent: no causal link] All timestamps are close to physical time (~100-102ms).

Properties

Causality trackingYesLike Lamport clock: if A → B then HLC(A) < HLC(B)

HLC vs TrueTime

Both HLC (CockroachDB) and TrueTime (Spanner) solve similar problems but with different approaches:

AspectTrueTime (Spanner)HLC (CockroachDB)
Timestamp formatInterval [earliest, latest]Tuple (l, c, pid)
Hardware requiredGPS + atomic clocks per DCNone (NTP is sufficient)
Uncertainty handlingExplicit epsilon bound in intervalImplicit: epsilon bounds counter range
Commit latency impactWait for epsilon (~10ms)Wait for max-clock-skew guard (~250ms worst case, often 0)
Clock skew requirement~10 ms (with GPS)~250 ms (with NTP)
Deployment costVery high (hardware in every DC)Low (software only)
Causality trackingIndirectDirect (counter component)
Trade-off: TrueTime provides tighter bounds but requires expensive hardware. HLC is less precise but works on any cloud with standard NTP. CockroachDB chose HLC because it enables deployment on any infrastructure without GPS antennas.

Use in CockroachDB

CockroachDB uses HLC as its transaction timestamp mechanism:

  • Each transaction gets an HLC timestamp at the coordinator
  • Writes are ordered by HLC timestamps across replicas
  • The Closest-in-Time protocol ensures that if a transaction commits with timestamp T, no other transaction at a different node can have a conflicting timestamp within the clock uncertainty window
  • This provides serializable isolation without requiring TrueTime's GPS hardware

Common Interview Questions

  1. "What is a Hybrid Logical Clock and why was it invented?" -- HLC combines physical and logical clocks to get both causality tracking and physical time proximity in a single O(1) timestamp, without requiring specialized hardware.
  2. "How does HLC differ from TrueTime?" -- TrueTime returns a time interval and requires GPS/atomic clocks. HLC returns a (logical_time, counter, pid) tuple and works with just NTP. TrueTime has tighter bounds but higher deployment cost.
  3. "Can HLC detect concurrent events?" -- No. HLC provides the same causality guarantee as a Lamport clock (if A → B then HLC(A) < HLC(B)) but cannot detect concurrency. For that, you need vector clocks.
  4. "Why does CockroachDB use HLC instead of TrueTime?" -- To avoid requiring GPS/atomic clock hardware in every data center. This allows CockroachDB to run on any cloud or on-premise infrastructure with standard NTP.
  5. "What are the components of an HLC timestamp?" -- (l, c, pid): logical time (bounded by physical clock), counter (for disambiguation within same physical time), and process ID (for breaking remaining ties).

What to Skip

Depth to avoid in interviews:
  • Formal correctness proofs of the HLC algorithm
  • The specific implementation details of CockroachDB's Closest-in-Time protocol
  • HLC optimizations like bounded HLC or HLC with piggybacking
  • The interaction between HLC and CockroachDB's Raft consensus layer
  • Time synchronization daemon internals (chrony vs ntpd) as they relate to HLC
