5.6.4 Failure Detector

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆
Goal: Understand the taxonomy of failure detectors (perfect, strong eventual, weak eventual), how the phi accrual detector works in practice, and the fundamental tradeoff between false positives and false negatives. Know which systems use which detector and why.

Definition

A failure detector is a mechanism by which a node in a distributed system monitors other nodes and determines whether they have failed. Failure detectors provide the input that drives higher-level protocols like leader election, membership management, and replication failover.

The Fundamental Problem

In an asynchronous distributed system, there is no reliable way to distinguish between a node that has crashed and a node that is merely slow. Any timeout-based failure detector must choose between two types of errors:

  • False positive: Declaring a healthy node as failed. This is dangerous -- it can trigger unnecessary leader elections, data rebalancing, or traffic rerouting.
  • False negative: Failing to detect a truly crashed node. This is costly -- it means the system continues routing traffic to a dead node or waiting for a response that will never come.
Core tradeoff: You can never eliminate both errors simultaneously. Making the timeout shorter detects crashes faster but increases false positives. Making the timeout longer reduces false positives but increases detection latency. The right balance depends on the system's requirements and failure mode costs.

Failure Detector Taxonomy

Chandra and Toueg (1996) formalized two key properties of failure detectors:

Property Definition
Completeness Every crashed process is eventually suspected by some non-faulty process
Accuracy Non-faulty processes are never (or eventually never) suspected

Types of Failure Detectors

Type Completeness Accuracy Feasibility
Perfect Strong (every crash detected) Strong (never suspects alive) Impossible in async systems
Strongly eventual (Ω) Strong (every crash detected) Weak (eventually stops suspecting alive) Achievable
Weakly eventual (⋄) Weak (some process detects) Weak (eventually stops suspecting alive) Achievable

Key result: The Omega failure detector (strong eventual) is the weakest detector that can solve consensus. Raft, Paxos, and Zab all implement something equivalent to Omega.

Phi Accrual Failure Detector

The phi accrual detector (Hayashibara et al., 2004) replaces binary alive/dead decisions with a continuous suspicion level. Used in Cassandra, Akka, and Amazon Swarm.

Mechanism

Phi Accrual Detector Internals ────────────────────────────── 1. Sliding window of heartbeat inter-arrival times: [52ms, 48ms, 55ms, 50ms, 53ms, 47ms, 51ms, 49ms, 54ms, 50ms] 2. Compute statistics: mean = 50.9ms stddev = 2.6ms (Assumed normal distribution) 3. Calculate phi for current delay: phi(delay) = -log10( P(arrival > delay) ) delay = 50ms → phi ≈ 0.4 (perfectly normal) delay = 60ms → phi ≈ 3.8 (mildly suspicious) delay = 70ms → phi ≈ 7.2 (very suspicious) delay = 80ms → phi ≈ 11.0 (almost certainly dead) delay = 100ms → phi ≈ 17.0 (definitely dead) 4. Threshold comparison: if phi > PHI_THRESHOLD (typically 8): suspect(node) Key advantage: Adapts to changing network conditions automatically - Network gets slower → mean/stddev increase → phi decreases for same delay - Network gets faster → mean/stddev decrease → phi increases for same delay

Benefits Over Simple Timeout

Aspect Simple Timeout Phi Accrual
False positive rate High during network degradation Adapts to network conditions
Configuration Requires manual tuning Self-adapting
Output Binary (alive/dead) Continuous suspicion level
Implementation complexity Simple Moderate

Suspicion Levels

Some systems (like SWIM) implement a multi-level suspicion model rather than binary detection:

  1. Alive: Node is responding to heartbeats normally.
  2. Suspect: Node has missed some heartbeats but not yet declared failed. Other nodes may be consulted for corroboration.
  3. Dead (confirmed): Node has been confirmed failed by multiple nodes and is removed from membership.
Suspicion State Machine (SWIM-style) ─────────────────────────────────── heartbeat received ┌─────────┐ ────────────────── ┌─────────┐ │ ALIVE │ ←─────────────────── │ SUSPECT │ └────┬────┘ └────┬────┘ │ timeout (missed heartbeats) │ timeout (prolonged) │ │ or confirmed by others │ │ │ ┌──────▼──────┐ └───────────────────────────→│ DEAD │ failure confirmed └─────────────┘

False Positives vs. False Negatives in Practice

Interview framework for answering "how would you handle this?":
  • Scenario: Load balancer health checks. False positives = healthy backends removed from pool = reduced capacity. False negatives = dead backends receiving traffic = errors. Tolerance for false negatives is low -- use aggressive timeouts with quick re-adds.
  • Scenario: Leader election. False positives = unnecessary election = write unavailability. False negatives = zombie leader = split brain. Tolerance for false positives is low -- use conservative timeouts with phi accrual.
  • Scenario: Membership management. False positives = premature node removal = data rebalancing thrash. False negatives = stale routing = slow requests. Use suspicion levels to reduce thrashing.

Systems Using Failure Detectors

System Failure Detector Notes
Raft / etcd Simple timeout (election timeout) Binary: leader or not. Tuned via randomization.
Cassandra Phi accrual Adaptive. Configurable phi threshold (default 8).
ZooKeeper Simple timeout (session timeout) Ephemeral nodes disappear on session expiry.
Akka Cluster Phi accrual Adaptive. Separate detector per monitored node.
SWIM / HashiCorp Memberlist Multi-level suspicion Alive → Suspect → Dead. Confirmed via indirect probing.
Consul Simple timeout + suspicion Health checks with configurable intervals and timeouts.

Common Interview Questions

  • What is a failure detector and why do distributed systems need one?
  • Explain the difference between false positives and false negatives.
  • What is a phi accrual failure detector and how does it work?
  • Why is a perfect failure detector impossible in asynchronous systems?
  • How does Cassandra's failure detector adapt to changing network conditions?
  • When would you choose a simple timeout over a phi accrual detector?
  • What happens if a failure detector produces too many false positives?
  • How does the failure detector interact with leader election?

What to Skip

Do NOT memorize Chandra-Toueg's failure detector hierarchy beyond Perfect/Omega/Diamond. Skip Byzantine failure detectors (which detect arbitrary/malicious failures) -- crash-only models are the interview standard. Do not implement a phi accrual detector from scratch; understand the concept and its adaptive behavior. Skip the detailed statistical formulas for phi calculation.
