5.6.1 Why Consensus

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★
Goal: Understand what the consensus problem is, why it is fundamental to distributed systems, and what real-world problems it solves. Be able to articulate why consensus is unavoidable in many distributed system designs.

Definition

Consensus is the problem of getting a group of distributed processes to agree on a single value (or state), even when some processes may fail or the network may deliver messages late, out of order, or not at all. A correct consensus algorithm must satisfy three properties:

  • Safety: All non-faulty nodes decide on the same value.
  • Liveness: Every non-faulty node eventually decides (no indefinite deadlock).
  • Validity: The decided value must have been proposed by some node (not fabricated).

Why Consensus Is Unavoidable

In a single machine, coordination is trivial -- one CPU, one memory, one clock. In distributed systems, there is no shared memory, no global clock, and no single source of truth. When multiple nodes need to make a coordinated decision, they must communicate over an unreliable network. Consensus is the formal solution to this coordination challenge.

FLP Impossibility Result

Fischer, Lynch, and Paterson (1985): In a purely asynchronous system with even one possible crash failure, no deterministic algorithm can guarantee consensus. This means that any consensus algorithm must either make timing assumptions (e.g., partial synchrony) or use randomization to achieve liveness.

The practical implication is significant: real-world consensus algorithms (Raft, Paxos, Zab) all assume partial synchrony -- the network is usually timely, but may have arbitrary delays occasionally. This assumption allows them to make progress under normal conditions while providing safety guarantees always.

Why FLP Does Not Matter in Practice

  • Real systems have timeouts (not purely asynchronous).
  • Crash-only failures are a reasonable model for well-engineered servers.
  • Randomized election timeouts (Raft) break the symmetry that FLP exploits.
  • The FLP scenario (a perfectly timed message delay during a vote) is vanishingly unlikely in production.

Real Problems That Require Consensus

1. Leader Election

Many distributed systems (Raft, Zab, Kafka, etcd) need a single leader to coordinate writes. Electing that leader is a consensus problem -- all nodes must agree on who the leader is.

2. Mutual Exclusion / Distributed Locking

Preventing two nodes from accessing a shared resource simultaneously requires consensus. Who holds the lock? All nodes must agree, otherwise two nodes could both believe they hold the lock (split brain).

3. Atomic Broadcast / Total Order Broadcast

Ensuring all nodes see the same sequence of messages in the same order is equivalent to consensus. This is critical for replicated state machines, where every replica must apply the same operations in the same order.

4. Configuration / Membership Changes

When nodes join or leave a cluster, all nodes must agree on the current membership list. An inconsistent membership view leads to partition splits and data loss.

5. Distributed Transactions (Two-Phase Commit)

Deciding whether to commit or abort a transaction across multiple nodes requires all participants to agree on the outcome. This is a form of consensus, though 2PC is a blocking protocol (not liveness-safe under all failures).

Consensus and the CAP Theorem

Consensus sits at the heart of the consistency vs. availability tradeoff. A system that uses consensus for every operation sacrifices latency and availability (a write must be replicated to a quorum before it is acknowledged). Systems that avoid consensus (Dynamo-style) gain availability at the cost of potential inconsistencies.

Key insight for interviews: When asked "how would you design X?", ask yourself: "Do all nodes need to agree on something?" If yes, you need consensus. If you can tolerate temporary disagreement (eventual consistency), you can avoid it and gain performance and availability.

Common Interview Questions

  • What is the consensus problem and why is it hard?
  • Explain the FLP impossibility result and its practical implications.
  • Why does a distributed database need consensus?
  • What is the relationship between consensus and leader election?
  • How does consensus relate to the CAP theorem?
  • Can you have consistency without consensus? Under what conditions?
  • Name three real systems that use consensus internally.

What to Skip

Do NOT deep-dive into Byzantine consensus (BFT, PBFT) unless asked. Most interviews focus on crash-fault tolerance, not Byzantine faults. Also skip formal proofs of FLP -- knowing the result and its practical implications is sufficient. Avoid the full history of consensus algorithms; focus on Raft and Paxos which are what interviewers expect.
