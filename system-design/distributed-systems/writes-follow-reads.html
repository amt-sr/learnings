5.3.10 Writes Follow Reads

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the writes-follow-reads guarantee, the specific anomaly it prevents (writing stale data based on stale reads), its implementation, and how it completes the session consistency family of guarantees.

Definition

Writes-follow-reads guarantees that a write operation by a client will be sent to a replica that is at least as up-to-date as the replica from which the client last read. This prevents a client from writing stale data based on a stale read.

The Anomaly It Prevents: A client reads value X=5 from Replica 1 (which is behind), then writes an update based on X=5 to Replica 2 (which has X=10). The write overwrites X=10 with a value based on stale X=5, effectively "undoing" another client's update. Writes-follow-reads prevents this by ensuring the write goes to at least as fresh a replica as the read.

The Problem: Writing Based on Stale Data

Without Writes-Follow-Reads: Client A reads X from Replica 1 → X=5 (stale, latest is X=10) Client A writes X=7 (based on the X=5 they just read) Client A sends write(X=7) to Replica 2 → overwrites X=10 with X=7 Timeline: t0: Someone writes X=10 (goes to Replica 2) t1: Client A reads X from Replica 1 → X=5 (replica 1 is behind) t2: Client A computes X=7 based on X=5 t3: Client A writes X=7 to Replica 2 → REPLACES X=10 with X=7 === Lost Update === X=10 was lost because Client A read stale data and overwrote it. This is the "lost update" or "read-then-write" anomaly. With Writes-Follow-Reads: t1: Client A reads X from Replica 1 → X=5, version=5 t2: Client A wants to write X=7 → System checks: write must go to replica with version >= 5 → Replica 2 has version=10 (>= 5) → write goes to Replica 2 → Replica 2: X=10, but client expected X=5 → System detects conflict! (X changed between read and write) → Options: reject write, merge, or let application handle OR: t1: Client A reads X from Replica 1 → X=5, version=5 t2: Client A writes X=7 with precondition: version must be 5 → Replica 2 has X=10, version=10 → precondition fails → Write REJECTED (conditional write / compare-and-swap)

Implementation Strategies

1. Route Writes to the Same Replica as the Read

Approach: Sticky sessions (simplest) Client reads from Replica 1 → Replica 1 Client writes to Replica 1 → Replica 1 (same replica!) The replica has whatever the client just read, so no staleness gap. Subsequent reads from Replica 1 also see the write. ✓ Simple, no extra coordination ✗ Tied to one replica, unbalanced load

2. Version-Based Conditional Writes

Approach: Compare-and-swap / optimistic concurrency Client reads X=5, version=42 from any replica Client computes X=7 based on X=5 Client writes X=7 WITH precondition: version must be 42 At the replica: Current version is 42 → write succeeds (X=7, version=43) Current version is 45 → write FAILS (data changed since read) Client handles failure: Retry: re-read current value, recompute, rewrite Or: present conflict to user Or: use merge logic ✓ Works across replicas, no session affinity needed ✓ Prevents lost updates (detects conflicts) ✗ Client must handle retry/merge logic

3. Track Read Version on Write

Approach: System tracks the version of the last read Client reads X from Replica 2 (version=10) Client wants to write X → System ensures write goes to a replica with version >= 10 → If all replicas have version >= 10: write proceeds → If some replicas are behind: write goes to the freshest replica ✓ Stronger guarantee: write goes to fresh enough replica ✗ Requires version tracking infrastructure
Approach Guarantee Anomaly Prevention Complexity
Sticky sessions Writes follow reads (same replica) Prevents stale write target Low
Conditional writes (CAS) Detects lost updates Prevents lost updates AND stale writes Medium
Version tracking Writes to fresh replica Prevents writing to stale replica Medium

Real-World Relevance

Interview Tip: Writes-follow-reads is the guarantee behind conditional writes and compare-and-swap (CAS) operations. When asked "how do you prevent lost updates in a collaborative editor or an inventory system?", CAS is the standard answer. It is also the basis of optimistic concurrency control in most databases.
  • Version counters: Every record has a version field. Write includes "only if version = X". If version changed, write fails.
  • ETags: HTTP ETags implement writes-follow-reads at the caching layer. If-Match header ensures the resource has not changed.
  • Conditional mutations: DynamoDB conditional writes, Spanner transactions — all implement writes-follow-reads semantics.

The Complete Session Consistency Picture

Writes-follow-reads is the fourth and final guarantee in the Terry et al. session consistency family:

Session Consistency (Terry et al., 1994) consists of: ┌──────────────────────────────────────────────┐ │ 1. Read Your Writes │ │ After writing, you see your own write │ │ │ │ 2. Monotonic Reads │ │ Never read an older value (no going back) │ │ │ │ 3. Monotonic Writes │ │ Your writes are applied in order │ │ │ │ 4. Writes Follow Reads │ │ Writes go to as fresh a replica as reads │ └──────────────────────────────────────────────┘ All four together = Session Consistency Any individual guarantee is useful on its own Most real systems implement subsets depending on need

What to Skip

Avoid in interviews:
  • Optimistic concurrency control internals (version check implementations in specific databases)
  • HTTP conditional request details (If-Match, If-None-Match, If-Modified-Since syntax)
  • Lock-based concurrency control alternatives (pessimistic locking)
  • Specific DynamoDB conditional write syntax or Spanner transaction isolation levels
