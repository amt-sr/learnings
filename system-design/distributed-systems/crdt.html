CRDT

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand Conflict-Free Replicated Data Types (CRDTs) -- data structures that are mathematically guaranteed to converge across replicas without coordination. Know the major CRDT types, their trade-offs, and real-world usage.

Definition

A CRDT (Conflict-Free Replicated Data Type) is a data structure designed for distributed systems where replicas can update independently and always converge to the same state when they synchronize, without requiring coordination or consensus. Convergence is guaranteed by mathematical properties of the merge function.

Key insight: CRDTs solve the fundamental problem of LWW -- data loss. Instead of discarding writes, CRDTs merge all writes using a mathematically defined merge function. The merge is commutative (order doesn't matter), associative (grouping doesn't matter), and idempotent (applying the same update twice has no effect). These three properties guarantee convergence regardless of message ordering, delivery timing, or duplication.

Mathematical Foundation

A CRDT's merge function must satisfy three properties for any states S1, S2, S3:

PropertyDefinitionWhy it matters
Commutativemerge(S1, S2) = merge(S2, S1)Messages can arrive in any order
Associativemerge(merge(S1,S2), S3) = merge(S1, merge(S2,S3))Messages can be grouped arbitrarily
Idempotentmerge(S, S) = SDuplicate messages are harmless
Together, these three properties form a semilattice (specifically, a join-semilattice). The merge function computes the least upper bound of two states. No matter what order updates arrive in, replicas converge to the same final state. This is STRONG eventual consistency.

Types of CRDTs

CRDTs come in two flavors:

  • State-based (CvRDT) -- Each replica maintains a state and periodically sends its full state to others. Merging = computing the join of two states.
  • Operation-based (CmRDT) -- Each replica broadcasts operations (e.g., "add element X"). Operations are applied to the local state in a deterministic way.
AspectState-based (CvRDT)Op-based (CmRDT)
CommunicationSend full stateSend operations only
BandwidthHigher (full state)Lower (incremental ops)
Message deliveryNo ordering requiredCausal broadcast required
DuplicationHarmless (idempotent)Must be deduplicated

Common CRDT Data Types

G-Counter (Grow-only Counter)

Each process maintains its own counter. The global count is the sum of all process counters.

Process P0: [P0=5, P1=0, P2=0] → contributed 5 Process P1: [P0=0, P1=3, P2=0] → contributed 3 Process P2: [P0=0, P1=0, P2=7] → contributed 7 Global count = sum of all values = 5 + 3 + 7 = 15 Merge = element-wise max: merge([5,0,0], [0,3,0]) = [5,3,0] Properties: + Can only increment, never decrement + Always converges (element-wise max is commutative, associative, idempotent) + Each process can only increment its OWN slot → no coordination needed

PN-Counter (Positive-Negative Counter)

Two G-Counters: one for increments (P) and one for decrements (N). The value = sum(P) - sum(N).

Increments: [P0=5, P1=3, P2=0] → sum(P) = 8 Decrements: [N0=0, N1=2, P2=1] → sum(N) = 3 Value = sum(P) - sum(N) = 8 - 3 = 5 Allows both increment and decrement No coordination needed (each process only modifies its own slots) Merge = element-wise max on both P and N vectors independently

G-Set (Grow-only Set)

A set that only allows adding elements. An element, once added, can never be removed.

Merge = union of both sets
  Set A = {1, 2, 3}
  Set B = {3, 4, 5}
  Merge(A, B) = {1, 2, 3, 4, 5}

Limitation: No delete operation. Items are never removed.

OR-Set (Observed-Remove Set)

Supports both add and remove. Each element is tagged with a unique tag at add-time. Remove only removes tags that the replica has seen (observed).

Initial state: {} (empty set of tag-element pairs) Add "alice" on P1 → {(uuid1, "alice")} Add "alice" on P2 → {(uuid1, "alice"), (uuid2, "alice")} Remove "alice" on P1 → P1 removes uuid1 (which it observed) Remaining: {(uuid2, "alice")} → "alice" is still present! Add "bob" on P1 → {(uuid2, "alice"), (uuid3, "bob")} Merge: union of all sets by tags, keep if any tag survives Important: remove only removes tags the remover has OBSERVED. If P2 adds "alice" concurrently (uuid2) and P1 removes "alice" having only seen uuid1, then uuid2 survives the remove. This is "add-wins" semantics for concurrent add/remove.
OR-Set semantics: Concurrent add and remove of the same element results in add-wins (the element survives). This is a deliberate design choice -- it prevents the "lost update" problem where a concurrent add is silently discarded by a concurrent remove.

LWW-Element-Set

A set where each element has an associated timestamp. Add and remove are both writes with timestamps; the one with the later timestamp wins.

Each element has: {value, timestamp, is_add: bool}
  If timestamps differ: keep the one with the LATER timestamp
  If timestamps are equal: remove wins (bias toward deletion)

Trade-offs of CRDTs

Costs of CRDTs:
  • Space overhead: Most CRDTs require O(N) space where N is the number of replicas or number of operations. Tombstones accumulate and must be garbage collected.
  • Bandwidth: State-based CRDTs send full state on sync; operation-based CRDTs require causal broadcast.
  • Not all data types can be CRDTs: Complex application logic (e.g., "if balance > 0, withdraw") cannot be modeled as a simple CRDT merge.
  • Eventual consistency: CRDTs guarantee eventual consistency, not strong consistency. During a partition, replicas may temporarily diverge.
  • Garbage collection: Tombstones and deleted elements accumulate over time, requiring periodic GC.

Common Interview Questions

  1. "What is a CRDT?" -- A data structure that guarantees convergence across replicas without coordination. The merge function is commutative, associative, and idempotent (forms a semilattice), which ensures all replicas reach the same state regardless of message ordering.
  2. "How is a CRDT different from LWW?" -- LWW discards one of two conflicting writes. CRDTs merge both writes. No data is lost. CRDTs are mathematically guaranteed to converge; LWW's behavior depends on clock accuracy.
  3. "What are the two types of CRDTs?" -- State-based (CvRDT): send full state, merge via join. Operation-based (CmRDT): send operations, apply in causal order. State-based is simpler; operation-based is more bandwidth-efficient.
  4. "Explain a G-Counter." -- Each process maintains a vector of N counters (one per process). Only the owner increments its own slot. Global count = sum of all slots. Merge = element-wise max. Can only increment, never decrement.
  5. "How does an OR-Set handle concurrent add and remove?" -- Add-wins. Each add generates a unique tag. Remove only removes tags the remover has observed. If a concurrent add creates a new tag, that tag survives the remove. This prevents lost updates.
  6. "When would you NOT use a CRDT?" -- When you need strong consistency (banking), when the data type is too complex for a merge function (conditional logic), when the space overhead is prohibitive (many replicas), or when latency requirements demand synchronous coordination.
  7. "Compare CRDTs with Operational Transform." -- Both solve the multi-user editing problem. CRDTs are decentralized (no server authority needed) and mathematically converge. OT requires a central server for transformation. See 5.5.3 OT for details.

What to Skip

Depth to avoid in interviews:
  • Formal lattice theory and join-semilattice proofs
  • Delta-state CRDT optimizations (sending only deltas instead of full state)
  • Specific garbage collection algorithms for CRDT tombstones
  • RGA (Replicated Growable Array) and other sequence CRDT internals
  • CRDT-N (nested CRDT) composition rules
  • The relationship between CRDTs and CALM theorem
