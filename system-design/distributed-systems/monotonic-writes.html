5.3.9 Monotonic Writes

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the monotonic writes guarantee, the specific anomaly it prevents (write reordering), implementation strategies, and how it complements read-your-writes and other client-centric consistency models.

Definition

Monotonic writes guarantees that writes from a single client are applied to replicas in the same order the client issued them. A later write from the same client will never be applied before an earlier write from that same client.

The Anomaly It Prevents: Without monotonic writes, a client sends write(X=5) then write(X=10), but the system might apply write(X=10) first, then write(X=5), leaving the value at X=5 — the older value. The final state does not reflect the client's intent. Monotonic writes prevents this reordering.

The Problem: Write Reordering

Without Monotonic Writes: Client sends: write(X=5) at t0 Client sends: write(X=10) at t1 Network paths: write(X=5) → Replica 2 (slow path, arrives at t3) write(X=10) → Replica 1 (fast path, arrives at t1) Replica 1 applies: X=10 (at t1) Replica 2 applies: X=5 (at t3, arrives LATE) Now Replica 1 has X=10, Replica 2 has X=5 A read from Replica 2 returns X=5 — the OLD value wins! → The later write (X=10) is lost on Replica 2 === Monotonic Writes Violation === Write(X=10) was issued AFTER write(X=5), but X=5 was applied AFTER X=10 on Replica 2, effectively "undoing" the client's later write. With Monotonic Writes: Option A: Writes go through a sequencer (leader) write(X=5) → Leader → sequences at position 1 → replicates in order write(X=10) → Leader → sequences at position 2 → replicates in order All replicas apply: position 1 then position 2 Option B: Client tags writes with sequence numbers write(X=5, seq=1) → all replicas buffer until seq=1 applied write(X=10, seq=2) → all replicas buffer until seq=1 applied, then apply seq=2

Implementation Strategies

1. Write Through a Single Sequencer

All writes from a client go through the same ordering point: Client writes ──→ Leader / Sequencer ──→ Ordered replication │ ▼ Replica 1 (X=5, then X=10) Replica 2 (X=5, then X=10) Replica 3 (X=5, then X=10) The sequencer ensures: all replicas see writes in the same order This is the most common approach (leader-based systems)

2. Sequence Numbers or Timestamps

Each write is tagged with a client-specific sequence number or timestamp. Replicas buffer out-of-order writes and apply them in sequence order.

3. FIFO Channels to Replicas

Ensure that writes from the same client travel over a FIFO (first-in-first-out) channel to each replica. TCP provides this guarantee naturally — as long as the client sends all writes to the same replica.

Approach How It Works Guarantee Limitation
Single leader/sequencer All writes pass through one node Global monotonic writes Leader bottleneck, failover complexity
Client sequence numbers Client tags writes with increasing numbers Per-client monotonic writes Replicas must buffer and reorder
FIFO / TCP to same replica Client writes to one replica over ordered channel Per-replica monotonic writes Tied to one replica

Relationship to Read-Your-Writes

Key Relationship: Monotonic writes is necessary (but not sufficient) for read-your-writes. If writes can be reordered, then even reading from the leader might not show the client's latest write (the latest write might have been applied before the earlier one). Together, monotonic writes + read-your-writes form the core of session consistency.
  • Monotonic writes + Read-your-writes = Session consistency (approximately, also needs monotonic reads and writes-follow-reads)
  • Read-your-writes does NOT automatically imply monotonic writes — you could read your write but have your writes applied out of order
  • Monotonic writes does NOT imply read-your-writes — writes are ordered, but you might read from a replica that has not received them

Real-World Examples

System Monotonic Writes Mechanism
Kafka Partition leader sequences all writes; consumers read in order within partition
MongoDB Primary node serializes all writes; secondaries apply in oplog order
Redis (cluster) Single leader per shard; writes are sequenced per key by the leader
MySQL (binlog) Binary log provides total order of all writes; replicas replay in order
Interview Tip: Monotonic writes is usually taken for granted in leader-based systems but becomes critical in multi-leader or leaderless systems. When asked "what happens if two writes from the same client arrive at replicas out of order?", monotonic writes is the answer. This is particularly relevant for multi-region write scenarios.

What to Skip

  • Formal verification of write ordering properties
