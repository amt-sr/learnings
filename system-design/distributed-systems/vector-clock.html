Vector Clock

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Master vector clocks -- the gold standard for capturing causal relationships in distributed systems. Understand how they detect concurrency (which Lamport clocks cannot), their use in production databases, and their scalability trade-offs.

Definition

A vector clock is an array of N logical clocks, one per process in the system. Each process maintains its own vector clock and updates it on local events and message exchanges. Vector clocks provide an exact characterization of causality: they can determine whether two events are causally ordered or concurrent.

Key insight: Vector clocks solve the fundamental limitation of Lamport clocks. With a Lamport clock, C(A) < C(B) does NOT imply A → B. With vector clocks, V(A) < V(B) DOES imply A → B. This bidirectional property makes vector clocks the definitive tool for causal analysis.

Algorithm Rules

Each process Pi maintains a vector VC[i] of length N (N = total number of processes).

  1. VC1 -- Local event: Increment own component:
    VC[i][i] = VC[i][i] + 1
  2. VC2 -- Send: Before sending a message, increment own component and attach the full vector:
    VC[i][i] = VC[i][i] + 1; send(message, VC[i])
  3. VC3 -- Receive: On receiving a message with vector VC_msg, perform element-wise max then increment own component:
    VC[i][j] = max(VC[i][j], VC_msg[j]) for all j; then VC[i][i] = VC[i][i] + 1

Comparing Vector Clocks

The comparison operation on vector clocks determines causality:

ConditionMeaning
V(A) < V(B): all components of A are <= B, and at least one is strictly lessA happened before B (A → B)
V(A) > V(B): all components of A are >= B, and at least one is strictly greaterB happened before A (B → A)
V(A) || V(B): some components of A are greater, some are lessA and B are concurrent (A || B)
V(A) == V(B): all components equalSame event (only possible for the same logical event)
Correctness guarantee: V(A) < V(B) if and only if A → B. This is a biconditional -- both directions hold. This is strictly stronger than Lamport clocks (which only guarantee one direction).

Example: Three Processes with Concurrency Detection

P0 P1 P2 | | | a:[1,0,0] | | | b:[0,1,0] | | | | c:[2,0,0] ------->| | | d:[2,2,0] | | | | | e:[2,3,0] -------> | | | f:[2,3,1] | | | | g:[2,4,0] | | | h:[2,4,0] | <-----| | | | | | | i:[0,0,1] | | | | j:[2,5,0] -------> | | | k:[2,5,1] Causal relationships (V < means happened-before): a < c < d < e < g < j (causal chain P0→P1) d < e < f < k (causal chain P1→P2) c < d < g < h (causal chain P0→P1→P2) Concurrent events (V || means concurrent): b || a → [0,1,0] || [1,0,0] (incomparable) b || i → [0,1,0] || [0,0,1] (incomparable) i || a → [0,0,1] || [1,0,0] (incomparable) h || j → [2,4,0] || [2,5,0] BUT h < j (P2 received from P1 first) Note how event 'i' on P2 is completely independent from events on P0/P1. Vector clocks correctly identify i as concurrent with all of them.

Detecting Conflicts in Dynamo-Style Systems

Amazon Dynamo and Apache Cassandra use version vectors (a practical form of vector clocks) for conflict detection:

Client A writes key "cart" with value "X" → Server stores: cart = "X", version = [server1: 1, server2: 0, server3: 0] Client B writes key "cart" with value "Y" (concurrently, didn't see X) → Server stores: cart = "Y", version = [server1: 0, server2: 1, server3: 0] On reconciliation: [1,0,0] vs [0,1,0] → INCOMPARABLE → CONCURRENT → CONFLICT Both versions are kept as siblings: cart = {"X" [1,0,0], "Y" [0,1,0]} Client reads and resolves (application-level merge): cart = "X+Y", version = [1,1,0]
Dynamo's optimization: Instead of using process IDs directly, Dynamo uses (node, counter) pairs -- a technique called "dotted version vectors." This allows dynamic membership (nodes joining/leaving) without requiring a fixed-size vector.

Scalability Concerns

FactorImpact
Space per processO(N) -- grows linearly with number of processes
Message sizeO(N) -- every message carries the full vector
Comparison costO(N) -- element-wise comparison
Dynamic membershipProblematic -- adding/removing processes changes vector size
Typical practical limit~100-200 processes before overhead becomes prohibitive
Scalability bottleneck: In systems with thousands of nodes, full vector clocks become impractical. Solutions include:
  • Version vectors with server IDs -- only track replica nodes, not all processes
  • Dotted version vectors -- compress using (node, counter) pairs
  • Scalable vector clocks -- only grow vectors on new causal paths (bounded growth)
  • Hybrid Logical Clocks -- trade some causality tracking for O(1) space (see 5.4.5 HLC)

Common Interview Questions

  1. "What is a vector clock and how is it different from a Lamport clock?" -- Vector clock is an array of N logical clocks (one per process). Unlike Lamport clocks, it can detect concurrent events: if V(A) and V(B) are incomparable (some components larger, some smaller), the events are concurrent.
  2. "How do you compare two vector clocks?" -- Element-wise comparison. V(A) < V(B) if every element of A is <= the corresponding element of B, and at least one is strictly less. If some elements are larger and some smaller, the events are concurrent.
  3. "How does Dynamo use vector clocks?" -- Each object version has a version vector. When a coordinator receives writes, it merges version vectors. If two versions have incomparable vectors, they are concurrent conflicts. The client must resolve on read.
  4. "What are the scalability problems with vector clocks?" -- O(N) space and message overhead, where N is the number of processes. For large systems, this becomes prohibitive. Discuss approaches like dotted version vectors, version vectors with server IDs, or scalable vector clocks.
  5. "Design a system to detect conflicting writes across replicas." -- Use version vectors: each write carries a vector clock. On read, compare vectors. If concurrent, return all conflicting versions to the client for resolution. This is exactly how Dynamo/Cassandra work.
  6. "Can vector clocks handle dynamic node membership?" -- Not directly with simple vectors. Explain (node, counter) pairs / dotted version vectors as the solution.

What to Skip

Depth to avoid in interviews:
  • Formal lattice-theoretic foundations of vector clocks (unless you're interviewing for a research role)
  • Implementations of scalable vector clocks (the algorithm is complex and rarely asked)
  • Tree-crystal mechanism for compression
  • Merged vector clocks and their optimizations
  • The relationship between vector clocks and STRONG eventual consistency
