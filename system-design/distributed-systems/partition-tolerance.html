5.1.3 Partition Tolerance

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand why partition tolerance is a mandatory property of distributed systems, what network partitions look like in practice, and how the CAP theorem's trade-off between consistency and availability plays out when partitions occur.

Definition

Partition tolerance means the system continues to operate despite an arbitrary number of messages being dropped or delayed between nodes. In other words, the network may split into groups of nodes that cannot communicate with each other, and the system must still function.

Why P Is Mandatory: In any distributed system with multiple nodes communicating over a network, partitions will happen. Network cables get cut, switches fail, DNS goes down, congestion causes timeouts. You cannot opt out of partitions — you can only choose how to handle them: sacrifice consistency (AP) or sacrifice availability (CP).

What a Network Partition Looks Like

Normal Operation: ┌──────────┐ ┌──────────┐ │ Node A │◄───────►│ Node B │ └──────────┘ └──────────┘ ↕ ↕ ┌──────────┐ ┌──────────┐ │ Node C │◄───────►│ Node D │ └──────────┘ └──────────┘ All nodes communicate freely === Network Partition === ┌──────────┐ ╳╳╳╳╳ ┌──────────┐ │ Node A │ ╳ LINK ╳ │ Node B │ └──────────┘ ╳ DOWN ╳ └──────────┘ ↕ ╳╳╳╳╳ ↕ ┌──────────┐ ┌──────────┐ │ Node C │◄───────►│ Node D │ └──────────┘ └──────────┘ Left group: {A, C} — can communicate with each other Right group: {B, D} — can communicate with each other Cross-group: NO COMMUNICATION Question: What does each group do when it cannot reach the other?

The CAP Theorem Explained

Consistency (C) Availability (A) ┌────────────┐ ┌────────────┐ │ All nodes │ │ All requests│ │ see same │ │ get a │ │ data │ │ response │ └──────┬─────┘ └──────┬─────┘ │ │ │ You MUST pick one │ │ during a partition: │ │ │ ▼ ▼ ┌──────────────────────────────────┐ │ Partition Tolerance (P) │ │ Network splits are inevitable │ │ In distributed systems │ └──────────────────────────────────┘ CAP Theorem: In a distributed system, when a partition occurs, you must choose between C and A. You cannot have both. CA is impossible in distributed systems (single-node only). CP: Preserve consistency, sacrifice availability (error on partition). AP: Preserve availability, sacrifice consistency (stale reads ok).

CP vs. AP Trade-off

Aspect CP (Consistency + Partition Tolerance) AP (Availability + Partition Tolerance)
During Partition Minority partition rejects requests All nodes continue serving requests
Data Guarantee Always correct (or error) May return stale data
Response May return errors/timeouts Always returns a response
Post-Partition Resume normally (no divergence) Conflict resolution / merge required
Examples HBase, ZooKeeper, etcd, MongoDB (default), Spanner Cassandra, DynamoDB (eventual), CouchDB, Eureka
Best For Financial systems, metadata, leader election Social feeds, shopping carts, IoT telemetry

Decision Framework

Choose CP when: Incorrect data is worse than no data (banking, inventory, configuration, locking, leader election).
Choose AP when: No response is worse than stale data (user timelines, product catalogs, caching, logging, analytics).

Partitions in Practice

Partitions are not just network cable cuts. They include:

  • Network failures: Switch failures, routing issues, misconfigured firewalls
  • GC pauses: A long garbage collection pause makes a node "invisible" to others — effectively a partition
  • CPU throttling: A throttled node cannot respond fast enough, causing timeouts that look like partitions
  • DNS failures: Nodes cannot resolve each other's addresses
  • Asymmetric partitions: Node A can send to Node B, but Node B cannot send to Node A (one-way links)
Interview Tip: Mentioning GC pauses and CPU throttling as "hidden partitions" demonstrates operational maturity. These are the most common causes of production "partitions" in real-world Java/Go systems, not cable cuts.

Common Interview Questions

  1. "Why is partition tolerance mandatory in distributed systems?"
  2. "Explain the CAP theorem in your own words."
  3. "When would you choose CP over AP?"
  4. "Is CA a valid choice? Explain."
  5. "What are some non-obvious causes of network partitions?"
  6. "How does a CP system handle a partition between its leader and followers?"
  7. "Give an example of a system that dynamically switches between CP and AP."

What to Skip

Avoid in interviews:
  • Mathematical proof details of the CAP theorem (Gilbert & Lynch proof specifics)
  • Network-level partition detection algorithms (Phi Accrual failure detector internals)
  • Specific TCP retransmission or network layer details
  • Arguments about whether CAP is "dead" or "misunderstood" — acknowledge nuance, but do not debate semantics
