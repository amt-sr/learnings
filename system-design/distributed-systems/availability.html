5.1.2 Availability (CAP)

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★★★

Goal

Understand what availability means in the CAP theorem, how it differs from operational uptime or SLA metrics, and why "available" does not mean "correct."

Definition

In the CAP theorem, availability means that every request receives a non-error response — without guaranteeing that the response contains the most recent write. The system always returns some data, even if it is stale, during a partition.

Formal Definition (Gilbert & Lynch): For every request received, a non-error response is returned, without guaranteeing that the response contains the most recent write. The key phrase is "non-error response" — the system must not reject or timeout requests during a partition.

Availability vs. Uptime / SLA

Property CAP Availability Uptime / SLA
Definition Every non-failing node returns a response to every request System is operational for X% of time (e.g., 99.99%)
Scope Per-request guarantee Aggregated over time window
Data Freshness May return stale data Not a factor in the metric
Context During network partitions All operations (maintenance, failures, etc.)
Measurement Binary: responds or errors Percentage: nines (99.9%, 99.99%, etc.)
Interview Tip: "Five nines" (99.999% uptime) is about 5 minutes of downtime per year. But CAP availability is not the same thing. A system with 99.999% uptime can still violate CAP availability if it returns errors during partitions. Conversely, a system that always responds (even with stale data) satisfies CAP availability but might return incorrect results.

"Available" Does Not Mean "Correct"

Scenario: Node 1 and Node 2 are partitioned Before partition: Node 1: balance = $100 Node 2: balance = $100 (consistent) During partition: Client A → Node 1: write(balance = $50) → Node 1: balance = $50 Client B → Node 2: read(balance) → Node 2: balance = $100 ← STALE but AVAILABLE CAP-Available: Node 2 returns $100 (not an error) CAP-Consistent: Node 2 would return ERROR (cannot guarantee latest value) === The fundamental trade-off === AP: Always respond, even with stale data → Users can transact, but see old data CP: Block until you can guarantee correct → Users may get errors, but data is right

Real-World Example: Shopping Cart

An e-commerce site during a partition: Do you let users add items to their cart (AP — available but the cart might be stale) or do you show an error page (CP — consistent but users cannot shop)?

  • Amazon chose AP: In 2001, Gilbert & Lynch noted that Amazon's "1-click" checkout chose high availability over consistency — a customer could always place an order, even if inventory data was slightly stale.
  • Payment systems choose CP: You cannot tolerate double-spending. An error is better than an incorrect charge.

Examples of AP Systems

System AP Behavior
Cassandra Any replica can serve reads/writes; uses tunable consistency but defaults to AP
DynamoDB (eventual) Serves reads from any replica; eventual consistency mode is the default for many workloads
Eureka (Netflix) Service discovery registry that works even when nodes cannot reach each other
CouchDB Multi-master replication; every node accepts writes, resolves conflicts later
DNS Caches propagate over time; no global lock, always resolves (with potential staleness)

Common Interview Questions

  1. "What does the 'A' in CAP actually mean?"
  2. "Is 'available' the same as '99.99% uptime'? Explain the difference."
  3. "Give an example where you would choose availability over consistency."
  4. "How does an AP system handle a network partition?"
  5. "What happens to stale data in an AP system after a partition heals?"
  6. "Can you name real-world systems that are AP and explain their choice?"

What to Skip

Avoid in interviews:
  • SLA/SLO calculation exercises (that is an SRE topic, not system design)
  • Load balancer health check mechanisms in depth unless directly relevant
  • Service mesh retry/backoff configuration details
  • Specific cloud provider uptime guarantees (AWS SLA tables, etc.)
