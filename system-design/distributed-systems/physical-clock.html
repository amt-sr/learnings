Physical Clock

Difficulty: ⭐⭐☆☆☆ Interview Frequency: ★★★☆☆

Goal

Understand how physical clocks work in distributed systems, why they are insufficient for event ordering, and what solutions exist to improve their accuracy for production use.

Definition

A physical clock is a real-world time source (e.g., wall clock,石英 crystal oscillator) that measures time based on physical phenomena. In distributed systems, each node has its own physical clock, but these clocks inevitably drift apart due to hardware imperfections, temperature variations, and network delays.

Key insight: In a single machine, ordering is trivial -- CPU provides a total order. In distributed systems with multiple nodes, there is no shared clock and no universal "now." Physical clocks provide an approximation, but approximation is not enough for correctness.

Clock Drift

Even high-quality quartz oscillators drift at a rate of roughly 10-6 seconds per second (about 1 second per ~11.6 days). Cheaper hardware drifts even faster. After synchronizing two clocks, they will gradually diverge.

Server A: ----[t=0]--------[t=100]--------[t=200]---- (crystal oscillator) Server B: ----[t=0]---[t=101]---------[t=203]------- (slightly faster crystal) After sync, B's clock runs ~1% faster than A's clock. Over 1 hour, B gains ~36 seconds relative to A. After 1 day, the drift becomes ~864 seconds (14.4 minutes).

Network Time Protocol (NTP)

NTP is the most widely used clock synchronization protocol. It organizes servers into a hierarchy (strata) with GPS/atomic clocks at the top (stratum 1).

NTP Synchronization Process

Client NTP Server | | | -- T1: Send request (record local time t0) --> | | | | <-- T2: Receive reply (record local time t3) -- | | (server recorded send time t1 & recv time t2) | | | | | Round-trip time = (t3 - t0) - (t2 - t1) | Clock offset = ((t1 - t0) + (t2 - t3)) / 2 | Client adjusts its clock by the calculated offset

Why Physical Clocks Aren't Sufficient for Ordering

Critical limitation: Physical clocks cannot guarantee ordering correctness. Even with NTP, two events that are causally related can receive timestamps that appear out of order due to clock skew and network delay.

Consider this scenario:

Process A (clock ahead by 50ms) Process B | | | Event X (t=A+100ms) | | ---- send message -----------> | | | Event Y (t=B+60ms) | | | Physical timestamps: | | X = A's clock = 100ms | | Y = B's clock = 60ms | | | | Problem: X physically happened | | before Y (causally), but | | timestamp(X=100) > timestamp(Y=60)

This means:

  • Two causally related events can have out-of-order timestamps
  • Physical time provides no guarantee of causal consistency
  • Events with close timestamps could be in any order -- you cannot distinguish causality from concurrency

NTP Limitations for Distributed Systems

  • Asymmetric network delays -- the return path may differ from the forward path, introducing error in offset calculation
  • Clock slew rate limits -- NTP gradually adjusts clocks (slewing) to avoid discontinuities, meaning synchronization is never instantaneous
  • No bounded error guarantee -- NTP does not provide a strict upper bound on clock error, making it unsuitable as a correctness primitive
  • Vulnerability to network partitions -- during partitions, clocks can drift significantly before re-synchronization
  • Leap seconds -- NTP must handle leap seconds, which can cause system-level bugs (see the Cloudflare 2017 outage)

GPS Clocks in Data Centers

Major cloud providers and database systems install GPS-based atomic clocks (rubidium or cesium) in data centers to achieve microsecond-level synchronization.

Why GPS? GPS satellites carry atomic clocks and broadcast precise time signals. A GPS receiver with a clear sky view can achieve ~10-100 ns accuracy. Each data center gets a GPS antenna + atomic clock (holds time if GPS signal is lost).

TrueTime (Google Spanner)

Spanner introduces TrueTime, an API that returns a time interval rather than a single timestamp:

TrueTime.now() returns [earliest, latest]

  earliest = actual time - epsilon
  latest   = actual time + epsilon

where epsilon accounts for worst-case clock uncertainty between nodes.

Spanner uses GPS + atomic clocks to keep epsilon small (typically ~1-10 ms). The interval-based approach means:

  • If commit_timestamp + epsilon < TrueTime.now().earliest, the transaction is definitely committed
  • Before that point, transactions must wait (commit-wait) to ensure linearizability
  • This trades latency for strong consistency guarantees

Common Interview Questions

  1. "How does NTP work?" -- Explain the request/reply exchange, round-trip time calculation, and clock offset formula. Mention stratum hierarchy.
  2. "Why can't we just use wall-clock time for ordering events?" -- Clock drift, network delay, and the fundamental impossibility of knowing the exact offset between two clocks. Give the out-of-order timestamp example.
  3. "How does Spanner handle time uncertainty?" -- TrueTime returns an interval. Spanner performs commit-wait: after committing at timestamp ts, it waits until ts + epsilon has passed before making data visible.
  4. "What's the difference between NTP and PTP?" -- PTP (IEEE 1588) is designed for local networks with hardware timestamping support, achieving microsecond accuracy vs NTP's millisecond accuracy.
  5. "How do you handle leap seconds in a distributed database?" -- Smearing (Google, Amazon: stretch the last second of the day over several hours), disabling leap seconds (POSIX time), or using TAI (no leap seconds).
  6. "Why does CockroachDB use HLC instead of TrueTime?" -- TrueTime requires GPS hardware in every data center. HLC achieves similar properties in software-only, making it cheaper to deploy on any cloud.

What to Skip

Depth to avoid in interviews:
  • Stratum 1/2 server topology details -- just know the hierarchy concept
  • Chrony vs ntpd configuration internals
  • GPS signal processing (ephemeris calculations, ionospheric delay correction)
  • PTP Best Master Clock algorithm (BMCA) details
  • Spanner's Paxos implementation details when discussing TrueTime
