RBAC (Role-Based Access Control)

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand RBAC in depth: the role-permission model, role hierarchy, assignment of roles to users, and how RBAC works in both monolithic and microservices architectures. This builds on the ABAC/RBAC overview in the Authorization topic (14.2).

Definition

RBAC assigns permissions to roles, and roles to users. Instead of managing individual user permissions (which does not scale), you define roles that bundle related permissions and assign users to those roles. A user's permissions are the union of all permissions from all their assigned roles.

RBAC Data Model

Users ──▶ Role Assignments ──▶ Roles ──▶ Permissions ┌──────────┐ ┌──────────────────┐ ┌──────────┐ ┌─────────────────┐ │ Users │ │ User_Roles │ │ Roles │ │ Role_Permissions│ ├──────────┤ ├──────────────────┤ ├──────────┤ ├─────────────────┤ │ alice │────▶│ user: alice │────▶│ admin │────▶│ role: admin │ │ bob │────▶│ role: admin │ │ │ │ perm: users:read │ │ carol │────▶│ │ │ editor │────▶│ role: admin │ │ │ │ user: bob │────▶│ │ │ perm: users:write│ │ │ │ role: editor │ │ viewer │────▶│ role: admin │ │ │ │ │ │ │ │ perm: users:delete│ │ │ │ user: carol │────▶│ │ │ │ │ │ │ role: viewer │ │ │ │ role: editor │ └──────────┘ └──────────────────┘ └──────────┘ │ perm: posts:read │ │ role: editor │ │ perm: posts:write │ └─────────────────┘ alice = admin → users:read, users:write, users:delete, posts:read, posts:write bob = editor → posts:read, posts:write carol = viewer → posts:read

Role Hierarchy

super_admin (inherits all) │ ├── admin │ │ │ ├── editor │ │ │ │ │ └── viewer │ │ │ └── moderator │ └── auditor Permission resolution: - viewer: posts:read - editor: posts:read, posts:write (inherits viewer) - admin: all permissions (inherits editor, viewer, moderator) - super_admin: all permissions + system:config (inherits admin, auditor)
Trade-off: Role hierarchies reduce management overhead but increase complexity. Deep hierarchies (5+ levels) are hard to reason about. Keep hierarchies flat (2-3 levels) and use ABAC for fine-grained decisions.

RBAC in Microservices

Approach: Embed roles in JWT + enforce in each service 1. Auth Service issues JWT: { "sub": "alice", "roles": ["admin"], "exp": 1516242622 } 2. API Gateway checks: role present? (coarse-grained) "User has role 'admin' → allow to /admin/* routes" 3. Each service checks permissions (fine-grained) OrderService: requires "admin" role for DELETE /orders/:id OrderService: requires "viewer" role for GET /orders/:id ┌──────────┐ JWT: {roles: ["admin"]} ┌──────────────┐ │ Client │──────────────────────────────▶│ API Gateway │ └──────────┘ │ Check: admin? │ └──────┬───────┘ │ ┌────────────┼────────────┐ ▼ ▼ ┌──────────────┐ ┌──────────────┐ │ Order Svc │ │ User Svc │ │ Check: admin?│ │ Check: admin?│ └──────────────┘ └──────────────┘

Design Considerations

ConcernRecommendation
Role namingUse descriptive names tied to job function, not people (not "alice_role")
Least privilegeDefault to no permissions. Grant only what is needed.
Separation of dutiesPrevent users from having conflicting roles (e.g., cannot be both "auditor" and "accountant")
Principle of least rolesUsers should have the minimum roles needed for their job function.
Role explosionIf you have 50+ roles, you probably need ABAC. Role explosion signals over-granularity.
Dynamic rolesConsider time-bound roles (temporary admin access) and context-based activation.

When RBAC Falls Short

RBAC limitations: Cannot express "user X can edit document Y only if user X owns it" — that requires ABAC. Cannot express "users in department A can access resources in department A" without creating N department roles. For multi-tenant, contextual, or fine-grained access control, move to ABAC.
Interview tip: "I would use RBAC for the base access control model — roles like admin, editor, viewer are intuitive and cover 80% of use cases. For the remaining 20% (resource-level permissions, multi-tenant access), I would add ABAC policies evaluated by a central policy engine (OPA). This gives me simplicity where possible and flexibility where needed."

Common Interview Questions

  1. How does RBAC work?
  2. How do you design the database schema for RBAC?
  3. What is role hierarchy and how does it work?
  4. How would you implement RBAC in a microservices architecture?
  5. What happens when RBAC does not provide fine-grained enough access control?
  6. How would you handle "separation of duties" with RBAC?
  7. What is "role explosion" and how do you prevent it?

What to Skip

Avoid diving into: Specific database schema SQL, ORMs, or framework-specific role implementations (Spring Security @PreAuthorize). Focus on the conceptual model, role hierarchy, and the microservices integration pattern.
