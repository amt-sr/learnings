JWT (JSON Web Token)

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Understand JWT structure (header, payload, signature), signing algorithms, verification process, token expiry and rotation strategies, and when to use JWT versus opaque tokens.

Definition

JWT (JSON Web Token) is a compact, URL-safe, self-contained token that carries claims (data) in a signed format. It consists of three Base64URL-encoded parts separated by dots: header, payload, and signature. The signature ensures the token has not been tampered with.

JWTs are commonly used as access tokens in OAuth 2.0 flows. Because they are self-contained (the signature can be verified without a database lookup), they enable stateless authentication — a key advantage for distributed systems.

JWT Structure

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.   ← HEADER
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwicm9sZSI6ImFkbWluLCJpYXQiOjE1MTYyMzkwMjJ9.   ← PAYLOAD
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c   ← SIGNATURE

Header (decoded):
{
  "alg": "HS256",
  "typ": "JWT"
}

Payload (decoded):
{
  "sub": "1234567890",
  "name": "John Doe",
  "role": "admin",
  "iat": 1516239022,
  "exp": 1516242622,
  "scope": "read write"
}

Signature = HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key
)

Parts Explained

PartPurposeKey Fields
HeaderDefines the signing algorithm and token typealg (HS256, RS256, ES256), typ (JWT)
PayloadCarries claims (data about the user/session)sub (subject), exp (expiry), iat (issued at), role, scope
SignatureEnsures integrity — proves the token was not modifiedHMAC or RSA signature over header + payload
Common misconception: The payload is encoded (Base64URL), NOT encrypted. Anyone can decode a JWT and read its contents. Never put sensitive data (passwords, SSNs, credit cards) in the payload. The signature prevents tampering, not reading.

Signing Algorithms

AlgorithmTypeHow It WorksUse Case
HS256Symmetric (HMAC)Same secret for signing and verificationSingle-service apps (one secret holder)
RS256Asymmetric (RSA)Private key signs, public key verifiesMicroservices (distributed verification)
ES256Asymmetric (ECDSA)Private key signs, public key verifiesMicroservices (smaller signatures than RSA)
For microservices: Use RS256 or ES256 (asymmetric). The auth service holds the private key and signs tokens. Every other service only needs the public key to verify — no shared secret needed. This is the key advantage for distributed systems.

Verification Process

Token received by service: 1. Check structure: three parts separated by dots 2. Decode header and payload (Base64URL) 3. Verify signature: - Reconstruct signature input: header + "." + payload - Compute HMAC/RSA with the signing key - Compare with the signature in the token 4. Validate claims: - exp: token not expired - iat: issued in the past (not future-dated) - iss: issued by a trusted authority - aud: intended for this service 5. Extract identity/claims for authorization decisions

JWT vs Opaque Tokens

DimensionJWT (self-contained)Opaque (random string)
VerificationLocal (verify signature, no DB)Network call to auth server
SizeLarge (typically 1-2KB)Small (16-32 bytes)
RevocationDifficult (need blacklist)Instant (delete from store)
StatelessnessFully statelessStateful (requires store)
FlexibilityClaims fixed at issue timeCan change server-side
Best forShort-lived access tokens (15 min)Refresh tokens, long-lived sessions

Security Best Practices

  • Short expiry: Access tokens should expire in 15-60 minutes. Use refresh tokens for long-lived sessions.
  • Never use HS256 with untrusted clients: If a client can obtain the shared secret, it can forge tokens.
  • Validate ALL claims: Not just the signature. Check exp, iat, iss, aud, nbf.
  • Use the kid (Key ID) header: Enables key rotation without breaking existing tokens.
  • Do not put sensitive data in the payload: JWT payload is readable by anyone who intercepts the token.
  • Store in memory, not localStorage for SPAs: XSS attacks can read localStorage. Use httpOnly cookies instead.
Interview tip: "I would use RS256 JWTs with 15-minute expiry for access tokens. The auth service signs with a private key; all downstream services verify with the public key — no shared secret or database lookup needed. For long-lived sessions, I use opaque refresh tokens stored in the database, rotated on each use."

Common Interview Questions

  1. What is a JWT and what are its three parts?
  2. Is a JWT encrypted? Can anyone read its contents?
  3. Compare symmetric (HS256) vs asymmetric (RS256) signing.
  4. How do you revoke a JWT?
  5. Compare JWT with opaque tokens.
  6. What security best practices would you follow for JWTs?
  7. How do you handle key rotation with JWTs?

What to Skip

Avoid diving into: JWE (encrypted JWT), JWK (key management), JWS (signed JWT specification details), or specific library implementations (jsonwebtoken in Node, pyjwt). Focus on the structure, signing, verification, and the JWT vs opaque token trade-off.
