System Design / Security / 14.2 Authorization

Authorization (AuthZ)

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★

Goal

Understand authorization as the process of determining what an authenticated user can do, the three main models (ACL, RBAC, ABAC), and how to design an authorization system for a microservices architecture.

Definition

Authorization (AuthZ) is the process of determining what actions an authenticated user (or service) is permitted to perform. It answers the question "what can you do?" and enforces access control based on identity, role, attributes, or policies.

Authentication establishes identity. Authorization establishes permissions. Both are required for security, but they serve different purposes. A user can be authenticated but not authorized (valid login, but no access to a resource).

ACL (Access Control List)

ACL associates each resource with a list of users (or groups) who have access. Every resource maintains its own access list. 
# File-level ACL:
document_123:
  - user_456: read, write
  - user_789: read
  - group_managers: read, write, delete

# Check: Can user_789 write to document_123?
# → Look up document_123's ACL → user_789 only has read → DENY
ACL limitation: ACLs are resource-centric, not user-centric. To find all resources a user can access, you must scan every resource's ACL. This does not scale when you have millions of resources.

RBAC (Role-Based Access Control)

RBAC assigns permissions to roles, and roles to users. Instead of managing permissions per-user, you assign roles (admin, editor, viewer) that bundle related permissions.
RBAC Model: ┌──────────┐ assigned to ┌──────────┐ has permissions ┌──────────┐ │ User │──────────────▶│ Role │─────────────────▶│Permission│ │ "alice" │ │ "admin" │ │ read:users│ └──────────┘ └──────────┘ │ write:users│ │ delete:users│ User "alice" is assigned role "admin" Role "admin" has permissions: read, write, delete on users resource Therefore: alice can read, write, delete users Roles hierarchy: super_admin ──▶ admin ──▶ editor ──▶ viewer (inherits all permissions from child roles)

ABAC (Attribute-Based Access Control)

ABAC evaluates access based on attributes of the user, resource, action, and environment. Instead of static roles, policies use conditions: user.department == resource.owner_department AND resource.status == "approved". 
# ABAC Policy example:
Policy: "Users can edit documents they own, or if they are managers"

IF user.role == "manager"
   OR user.id == document.owner_id
THEN allow action="edit" resource="document"

# Contextual attributes:
IF user.department == resource.department
   AND time.hour BETWEEN 9 AND 17
   AND user.ip IN approved_networks
THEN allow action="view" resource="financial_reports"

Authorization in Microservices

Approach 1: Centralized Policy Engine ┌──────────┐ ┌──────────────┐ ┌──────────────┐ │ Service A │────▶│ Policy Engine│────▶│ Policy Store │ └──────────┘ │ (OPA / Casbin)│ │ (rules DB) │ ┌──────────┐ └──────────────┘ └──────────────┘ │ Service B │────▶│ ┌──────────┐ │ │ Service C │────▶│ Each service calls: "Can user X do action Y on resource Z?" Policy engine evaluates rules and returns ALLOW/DENY Approach 2: Token-based (claims in JWT) ┌──────────┐ JWT with claims: ┌──────────┐ │ Client │ { role: "admin", │ Service A │ ← validates claims locally │ │ dept: "engineering"│ │ └──────────┘ } └──────────┘ Each service extracts claims from JWT and checks locally. No network call needed. Trade-off: less flexible (claims fixed at token issue time)
Interview tip: "For a microservices architecture, I would use RBAC for coarse-grained access (roles in JWT) and ABAC via a central policy engine (OPA) for fine-grained decisions. The API gateway handles coarse-grained checks (is this user authenticated and has a valid role?), and each service handles fine-grained checks (can this user access this specific resource?)."

Common Anti-Patterns

  • Only checking at the API gateway: If the gateway checks auth but internal services trust all intra-cluster traffic, a compromised service bypasses all authorization.
  • Hardcoding roles in code: if user.role == "admin" scattered across code. Changes require redeployment.
  • Ignoring service-to-service auth: Services calling other services should also authenticate and authorize.

Common Interview Questions

  1. What is authorization and how is it different from authentication?
  2. Compare ACL, RBAC, and ABAC.
  3. How would you implement authorization in a microservices architecture?
  4. What is OPA and how does it help with authorization?
  5. How would you handle authorization at the API gateway vs within each service?
  6. What happens when a user's role changes and they have a valid JWT?

What to Skip

Avoid diving into: OPA Rego syntax, specific Casbin policy definitions, or SAML/OIDC protocol details. Focus on the three models (ACL, RBAC, ABAC), when to use each, and the microservices authorization architecture.
