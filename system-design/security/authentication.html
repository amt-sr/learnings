System Design / Security / 14.1 Authentication

Authentication (AuthN)

Difficulty: ⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Understand authentication as the process of verifying identity ("who are you?"), the evolution from passwords to MFA, session-based vs token-based auth, and the OAuth flow for third-party authentication.

Definition

Authentication (AuthN) is the process of verifying that a user or system is who they claim to be. It answers the question "who are you?" and produces a verifiable identity that can be used for subsequent authorization decisions ("what can you do?").

Authentication is the first gate in any secured system. Without reliable identity verification, authorization is meaningless because you cannot make access decisions based on unverified identity.

Password Storage

# NEVER store plain text passwords
# NEVER use MD5 or SHA-1 (fast, easily brute-forced)

# BEST PRACTICE: bcrypt with cost factor 12+
# bcrypt is intentionally slow (100-500ms) to make brute-force impractical
hash = bcrypt(password, cost_factor=12, random_salt)
# Example: $2b$12$LJ3m4yxK2YbQZ.0qn.jXuO1JXMdJQv.yYQJKVMbX7J3f4wX.1m2C

# Argon2id is even better (winner of Password Hashing Competition)
hash = argon2id(password, salt, memory_cost=65536, time_cost=3, parallelism=4)

# Key rules:
# 1. Use a unique random salt per password (prevents rainbow table attacks)
# 2. Use adaptive hashing (increase cost as hardware gets faster)
# 3. Never roll your own crypto

MFA (Multi-Factor Authentication)

MFA requires two or more independent verification factors: something you know (password), something you have (phone/hardware key), or something you are (biometric). MFA blocks 99.9% of automated attacks even if the password is compromised.

MFA Implementation Flow

1. User submits username + password
2. Server validates credentials
3. Server prompts for second factor:
   - TOTP: User enters 6-digit code from authenticator app
   - SMS: Server sends code, user enters it
   - Push: Server sends push notification, user approves
   - WebAuthn: User touches hardware security key
4. Server validates second factor
5. Server issues session/token
Modern guidance: Prefer TOTP (Google Authenticator, Authy) or WebAuthn (YubiKey) over SMS OTP. SMS is vulnerable to SIM swap attacks and SS7 protocol exploits. Apple and Google are deprecating SMS for 2FA.

Common Interview Questions

  1. What is authentication and how is it different from authorization?
  2. How would you securely store user passwords?
  3. Compare session-based and token-based authentication.
  4. What is MFA and why is it important?
  5. How does OAuth work for third-party authentication?
  6. What are the risks of SMS-based 2FA?
  7. How would you implement "remember me" functionality?
  8. What is credential stuffing and how do you prevent it?

What to Skip

Avoid diving into: Specific OAuth grant type protocol details (covered in 14.3), JWT internals (covered in 14.4), or password hashing library API. Focus on the high-level authentication concepts and the session vs token trade-off.
