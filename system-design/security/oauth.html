OAuth 2.0

Difficulty: ⭐⭐⭐⭐ Interview Frequency: ★★★★★

Goal

Understand OAuth 2.0 as a delegation framework (not authentication), the authorization code flow with PKCE, the roles (client, resource owner, auth server, resource server), tokens (access token, refresh token), and security considerations.

Definition

OAuth 2.0 is an authorization framework that allows a third-party application to access a user's resources on another service without sharing the user's credentials. It delegates access by issuing tokens with limited scope and lifetime.
OAuth is NOT authentication. OAuth 2.0 delegates authorization (access to resources). OpenID Connect (OIDC) is the authentication layer built on top of OAuth 2.0. This is a common interview trap — always clarify the distinction.

PKCE (Proof Key for Code Exchange)

PKCE (pronounced "pixie") prevents authorization code interception attacks. The client generates a random code_verifier, hashes it (SHA-256) to create a code_challenge, and sends the challenge in the auth request. When exchanging the code for a token, the client sends the original verifier. The server verifies SHA256(verifier) == challenge.
PKCE is mandatory for all OAuth flows. Originally designed for public clients (mobile apps, SPAs) that cannot securely store a client secret, PKCE is now recommended for all OAuth 2.0 flows. The OAuth 2.1 draft makes it required.

Access Token vs Refresh Token

DimensionAccess TokenRefresh Token
PurposeAccess resources (API calls)Obtain new access tokens
LifetimeShort (15 min - 1 hour)Long (days, weeks, months)
ContentOpaque or JWT with claims/scopesOpaque (random string stored server-side)
StorageMemory or short-lived storageEncrypted, long-term storage (DB)
ExposureHigh (sent with every API call)Low (sent only to token endpoint)
RevocationEffective until expiryCan be revoked immediately

Token Security

  • Always use HTTPS: Tokens transmitted in plain text over HTTP can be intercepted.
  • Short-lived access tokens: Limit blast radius if stolen. 15 minutes is a good default.
  • Use refresh token rotation: Each time a refresh token is used, issue a new one and invalidate the old. Detects token theft.
  • Bind refresh tokens to client: If a refresh token is used from a different client fingerprint, revoke it.
  • Scope limitation: Tokens should have the minimum scopes needed (principle of least privilege).
Interview tip: "For my OAuth implementation, I would use the Authorization Code flow with PKCE for all clients. Access tokens would be JWTs with 15-minute expiry. Refresh tokens would be stored as opaque values in the database with rotation enabled. If a refresh token is used from a different IP or user-agent, I would revoke it and force re-authentication."

Common Interview Questions

  1. What is OAuth 2.0 and is it authentication or authorization?
  2. Explain the authorization code flow step by step.
  3. What is PKCE and why is it needed?
  4. Compare access tokens and refresh tokens.
  5. What are the OAuth 2.0 grant types and when would you use each?
  6. How would you secure tokens in a mobile app?
  7. What is refresh token rotation and why does it matter?
  8. What is the difference between OAuth 2.0 and OpenID Connect?

What to Skip

Avoid diving into: Client credentials flow details (service-to-service), device authorization grant (IoT), or specific provider implementations (Google OAuth console setup). Focus on the authorization code flow with PKCE and token security.
