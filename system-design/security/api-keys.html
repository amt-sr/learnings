System Design / Security / 14.6 API Keys

API Keys

Difficulty: ⭐⭐ Interview Frequency: ★★★

Goal

Understand API key authentication, generation, storage, rotation, rate limiting per key, and the limitations of API keys compared to OAuth 2.0. Know when API keys are appropriate and when they are insufficient.

Definition

API key is a static string (alphanumeric token) that identifies a client application or user for API access. The client includes the key in every request (typically in the Authorization header or query parameter), and the server validates it against a key-value store.

API keys are the simplest form of API authentication. They identify the caller but do not carry information about the user or their permissions. They are a shared secret between the client and server.

How API Keys Work

Client → API Gateway → Services Request: GET /api/v1/users Authorization: ApiKey sk_live_abc123def456 (or) GET /api/v1/users?api_key=sk_live_abc123def456 API Gateway: 1. Extract API key from header/query 2. Look up key in database (hash the key, compare) 3. Check: key exists? active? not expired? rate limit OK? 4. If valid: identify client, attach client_id to request 5. Route to downstream service API Keys Table: ┌──────────┬──────────┬───────────┬──────────┬──────────────┐ │ key_hash │ client_id│ scopes │ status │ rate_limit │ ├──────────┼──────────┼───────────┼──────────┼──────────────┤ │ sha256(ab│ acme_corp│ read,write│ active │ 1000/min │ │ c123...) │ │ │ │ │ │ sha256(xy│ startup_a│ read │ active │ 100/min │ │ z789...) │ pp │ │ │ │ └──────────┴──────────┴───────────┴──────────┴──────────────┘ NEVER store plain text API keys. Store only the hash. The key itself is shown to the user ONCE at creation.

Key Generation

# API Key format: prefix + random string
# Prefix identifies the key type:
#   sk_live_  = secret, production
#   sk_test_  = secret, test environment
#   pk_live_  = public (publishable)

# Generation:
prefix = "sk_live_"
random = crypto.random_bytes(32)     # 256 bits of entropy
api_key = prefix + base62_encode(random)
# Example: sk_live_4eC39HqLyjWDarjtT1zdp7dc

# Store in database:
api_key_hash = SHA256(api_key)       # Only store the hash
client_id = "acme_corp"
scopes = ["read", "write"]
created_at = now()

# Return to user (ONE TIME ONLY):
return api_key  # sk_live_4eC39HqLyjWDarjtT1zdp7dc
# Never shown again. User cannot recover it.

API Key vs OAuth 2.0

DimensionAPI KeyOAuth 2.0
IdentityIdentifies the application/clientIdentifies the user
User contextNo user identity (or fixed)Full user context with consent
ScopeLimited (predefined per key)Fine-grained, user-consented scopes
ExpiryNo built-in expiry (must implement)Built-in token expiry + refresh
RevocationManual (delete from DB)Built-in (revoke token)

ComplexitySimple (key in header)Complex (OAuth flow, PKCE, tokens)
Best forService-to-service, developer APIs, internal toolsUser-facing apps, third-party integrations

API Key Rotation

API keys should be rotated periodically. Since they are static and have no expiry, a compromised key provides access indefinitely. Rotation is the primary security control.
Key Rotation Flow: ┌──────────────────────────────────────────────────────┐ │ Phase 1: Create new key │ │ - Generate new API key alongside existing key │ │ - Both keys are valid simultaneously │ │ - Client starts using new key │ └──────────────────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────┐ │ Phase 2: Monitor (1-4 weeks) │ │ - Check if old key is still being used │ │ - Alert if old key has traffic after migration │ └──────────────────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────┐ │ Phase 3: Revoke old key │ │ - Delete old key from database │ │ - Any remaining traffic on old key gets 401 │ └──────────────────────────────────────────────────────┘

API Key Security Best Practices

  • Never expose in client-side code: API keys in JavaScript/mobile apps can be extracted. Use them only for server-to-server or developer-to-API communication.
  • Use HTTPS exclusively: API keys transmitted over HTTP can be intercepted.
  • Store only hashes: Never store the plain text key. If the database is compromised, only hashes are exposed.
  • Use prefixes: sk_live_, sk_test_ prefixes help identify keys accidentally committed to repos.
  • Rate limit per key: Each key should have its own rate limit to prevent abuse.
  • Scope keys: Limit each key to specific APIs (read-only, write, admin).
  • Audit key usage: Log which key was used for each request for forensic analysis.
Interview tip: "For developer APIs and service-to-service communication, I would use API keys with per-key rate limiting and scopes. Keys are hashed before storage. For user-facing authentication, I would use OAuth 2.0 — API keys are not suitable for user authentication because they do not carry user identity."

Common Interview Questions

  1. What is an API key and how does it work?
  2. Compare API keys with OAuth 2.0.
  3. How would you securely store and validate API keys?
  4. What is API key rotation and why is it needed?
  5. Why should API keys never be used in client-side JavaScript?
  6. How would you implement rate limiting per API key?
  7. What is the difference between a public and secret API key?

What to Skip

Avoid diving into: Specific cloud provider API key management (AWS IAM access keys, GCP service account keys), API gateway rate limiting configuration, or webhook authentication patterns. Focus on the API key lifecycle, security considerations, and comparison with OAuth.
