Three Phase Commit (3PC)

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★☆☆

Goal

Understand how 3PC extends 2PC by adding a pre-commit phase to eliminate the blocking problem (under reliable network assumptions), why this matters theoretically, and why it is rarely used in practice.

Definition

Three-Phase Commit (3PC) is a non-blocking distributed commit protocol that adds a Pre-Commit phase between the Prepare and Commit phases of 2PC. The key insight is that once any participant enters the pre-commit state, all participants are guaranteed to have voted YES, so a surviving participant can always determine the outcome even if the coordinator fails.

Key Insight: 3PC is non-blocking if the network is reliable (no partial failures, no network partitions). Under real-world conditions with partitions, 3PC can still block. This theoretical limitation is why it is almost never used in production.

Protocol Phases

Phase 1: CanCommit (Voting Phase)

  1. Coordinator sends CAN_COMMIT to all participants
  2. Each participant checks if it can commit and votes YES or NO
  3. Participants do not acquire locks yet (unlike 2PC)

Phase 2: PreCommit (Preparation Phase)

  1. If all votes are YES, coordinator sends PRE_COMMIT to all participants
  2. If any vote is NO, coordinator sends ABORT
  3. Participants receiving PRE_COMMIT acquire locks, write to log, acknowledge
  4. Critical property: Once a participant ACKs pre-commit, it knows all others also voted YES

Phase 3: DoCommit (Completion Phase)

  1. Coordinator sends DO_COMMIT to all participants
  2. Participants commit and release locks
  3. Participants ACK the coordinator
3PC Protocol Flow Coordinator Participant A Participant B | | | PHASE 1: CAN_COMMIT (Voting) | | | |-- CAN_COMMIT -------->| | |<-- VOTE YES ----------| (no locks yet) | | | | |-- CAN_COMMIT --------------------------------->| |<-- VOTE YES ----------------------------------| | | | PHASE 2: PRE_COMMIT (Preparation) | | | |-- PRE_COMMIT -------->| | | |-- acquire locks | | |-- write log | |<-- ACK ---------------| | | | | |-- PRE_COMMIT --------------------------------->| | | |-- acquire locks | | |-- write log |<-- ACK ---------------------------------------| | | | PHASE 3: DO_COMMIT (Completion) | | | |-- DO_COMMIT --------->| | | |-- commit + release | |<-- ACK ---------------| | | | | |-- DO_COMMIT --------------------------------->| | | |-- commit + release |<-- ACK ---------------------------------------|

Why 3PC Solves Blocking (Theoretically)

The blocking problem in 2PC arises because a participant that voted YES does not know how others voted. In 3PC:

  1. If a participant is in the Pre-Commit state: It knows all participants voted YES (because the coordinator only sends PRE_COMMIT when unanimous). It can safely timeout and commit.
  2. If a participant is in the Abortable state (voted YES but no PRE_COMMIT): It knows the transaction cannot have committed (commit requires PRE_COMMIT first). It can safely timeout and abort.
Non-Blocking Property: In 3PC, a participant can always determine the correct action (commit or abort) on its own after a timeout, because the three-phase protocol ensures no ambiguity about the global state.

Why 3PC Is Rarely Used in Practice

Interview Answer: "3PC solves the theoretical blocking problem but assumes a reliable network. In real distributed systems, network partitions happen. Under a partition, 3PC can still block, which defeats its primary purpose. This is why production systems use 2PC with robust recovery, or avoid atomic commit entirely with Saga patterns and eventual consistency."
  • FLP Impossibility: No deterministic protocol can achieve consensus in an asynchronous system with even one crash failure. 3PC assumes synchronous network with bounded delays, which does not hold in reality.
  • Network partitions still cause blocking: If participants are partitioned from each other, they cannot communicate to determine the outcome.
  • Extra latency: Three round trips vs two for 2PC, with marginal benefit under real failure conditions.
  • Operational complexity: More state to manage, more logs, more recovery paths.
  • Better alternatives exist: Paxos/Raft-based consensus, Saga patterns, and modern databases with TrueTime (Spanner) provide more practical solutions.

3PC State Machine

Participant State Transitions CAN_COMMIT | v [Abortable]----NO---->[ABORTED] | YES | v [PreCommit]----ABORT->[ABORTED] | ACK | v [Committed] Timeout from Abortable --> ABORT (safe: commit not possible) Timeout from PreCommit --> COMMIT (safe: all voted YES) Timeout from Committed --> re-query or assume COMMITTED

Common Interview Questions

  1. How does 3PC solve the blocking problem of 2PC?
  2. Under what conditions can 3PC still block?
  3. Why isn't 3PC used in production systems?
  4. How does 3PC relate to the FLP impossibility result?
  5. Compare the latency of 2PC vs 3PC. Is the extra latency worth it?
  6. Can a participant that has entered the PreCommit state safely decide to commit on its own? Why?
  7. If you were designing a distributed transaction system, would you choose 2PC or 3PC? Why?

What to Skip

For L5/L6/Staff interviews, skip:
  • Formal proofs of 3PC non-blocking property (Skeen's original paper details)
  • Optimized 3PC variants (reducing communication by piggybacking)
  • Implementation-level recovery log formats
  • Exotic extensions like four-phase commit or hierarchical 3PC
