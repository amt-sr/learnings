System Design / Section 9: Distributed Transactions / 9.2.4 Compensation

Compensation

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand compensation as the mechanism for reversing partial distributed transactions. This is the backbone of the Saga pattern and a critical concept for handling failures in distributed workflows where atomic rollback is not possible.

Definition

Compensation is the process of undoing the effects of a completed step in a distributed transaction when a subsequent step fails. Unlike traditional database rollbacks (which reverse all changes atomically), compensation is a semantic reversal -- each step must define its own compensating action that logically undoes the business effect of the original step. Compensation is fundamental to the Saga pattern because there is no global transaction to roll back; each local transaction must be reversed individually.

Key Insight: Compensation is not a rollback. A database rollback undoes all changes atomically using the write-ahead log. Compensation applies a new operation that logically reverses the business effect. The original operation may have had side effects (emails sent, logs written, external API calls made) that cannot be truly reversed. This is why compensation is called "semantic" -- it restores business correctness, not database state.

Semantic vs Physical Compensation

PropertyPhysical (DB Rollback)Semantic (Compensation)
ScopeDatabase onlyAny system (DB, API, external service)
GranularityAll changes in a transactionPer-step reversal
Side effectsNone (complete undo)May leave residual effects (logs, sent emails)
IsolationRollback is invisible to othersIntermediate state visible to other transactions
TimingImmediate (same transaction)Delayed (separate transactions)
IdempotencyN/A (rollback is idempotent by nature)Must be explicitly designed as idempotent

Compensation in a Saga

Saga with Compensation Flow HAPPY PATH: [T1: Create Order] --> [T2: Reserve Inventory] --> [T3: Charge Payment] | | | v v v Order created Stock reserved Card charged FAILURE PATH (Payment fails): [T1: Create Order] --> [T2: Reserve Inventory] --> [T3: Charge Payment] | | | v v [FAIL] Order created Stock reserved | | | | | v | | [C2: Release Inventory] | | | | v v v [C1: Cancel Order] <--------------------------------------+ | v All steps compensated (system returned to consistent state)

Compensation Strategies

Strategy 1: Full Reversal

Each compensating transaction completely undoes the original operation.

  • Reserve inventory --> Release inventory reservation
  • Charge credit card --> Refund credit card
  • Activate subscription --> Deactivate subscription

Strategy 2: Partial Reversal with Business Logic

The compensation may not be a simple inverse; it may involve business rules.

  • Applied discount code --> "Release" discount (but the code may have been time-limited)
  • Reserved seat on flight --> Cancel reservation (but cancellation fee may apply)
  • Sent welcome email --> Cannot "unsend" email; send a follow-up correction instead

Strategy 3: No Compensation Needed

Some operations don't need compensation because they have no reversible business impact.

  • Logging / auditing
  • Analytics events
  • Cache warming
Design Principle: When designing a saga, ask for each step: "If this step succeeds but a later step fails, what operation would logically undo the business effect?" If there is no reasonable compensation, consider whether the step should be in the saga or if it should be deferred to after the saga completes.

Compensation Challenges

1. Compensation Failure

The compensating transaction itself can fail. For example, the refund API might be down.

Compensation Failure: Cascading Problem [T3 fails] --> Compensate T2 --> [C2 FAILS: Refund service down] | v Retry C2 (exponential backoff) | v [C2 still fails after max retries] | v Record in Dead Letter Queue Alert operations team Manual intervention required
Mitigation Strategies:
  • Retries with exponential backoff: Most transient failures resolve with retries
  • Compensation as a saga itself: If C2 fails, compensate C2 with C2a (escalation saga)
  • Dead Letter Queue: Persist failed compensations for manual or automated resolution
  • Idempotent compensations: Ensure retries are safe (same effect if run multiple times)
  • Time-based resolution: Auto-expire reservations that were not confirmed within a timeout

2. Observability of Compensations

You need to track compensation state across the saga:

  • Which steps completed successfully
  • Which compensations have been triggered
  • Which compensations are still pending or failed
CREATE TABLE saga_instance (
    saga_id        UUID PRIMARY KEY,
    saga_type      VARCHAR(255) NOT NULL,
    status         VARCHAR(50) NOT NULL,   -- PENDING, COMPLETED, COMPENSATING, FAILED
    current_step   INTEGER NOT NULL DEFAULT 0,
    created_at     TIMESTAMP NOT NULL,
    updated_at     TIMESTAMP NOT NULL
);

CREATE TABLE saga_step (
    id              UUID PRIMARY KEY,
    saga_id         UUID REFERENCES saga_instance(id),
    step_number     INTEGER NOT NULL,
    action          VARCHAR(255) NOT NULL,   -- e.g., "reserve_inventory"
    status          VARCHAR(50) NOT NULL,   -- PENDING, COMPLETED, COMPENSATING, COMPENSATED, FAILED
    compensation    VARCHAR(255),            -- e.g., "release_inventory"
    executed_at     TIMESTAMP,
    compensated_at  TIMESTAMP
);

3. Idempotent Compensations

Compensating transactions must be idempotent because:

  • The orchestrator may retry a failed compensation
  • Network issues may cause duplicate compensation messages
  • Recovery after a crash may re-execute a compensation
Best Practice: Design compensating operations to be naturally idempotent. For example: "Set order status to cancelled" is idempotent (setting it again has no effect). "Refund $100" is NOT idempotent unless you track refund IDs. Always check if a compensation has already been applied before executing it.

4. Compensating External Services

When a saga step calls an external service (third-party API), compensation is harder:

  • External API may not support compensation (no "undo" endpoint)
  • External API may impose fees or rate limits on compensation
  • Time between operation and compensation may affect what reversal is possible
Design Rule: For external services, design the saga so that the external call happens as late as possible. Move all internal operations before the external call. This minimizes the number of compensations needed when the external call fails.

Compensation vs Rollback: When to Use What

ScenarioUse CaseReason
Single database transactionDatabase rollbackFast, atomic, complete undo
Multiple services, same companySaga + compensationNo cross-service transaction possible
Cross-company boundariesSaga + compensationNo shared transaction coordinator
Long-running processSaga + compensationCannot hold locks for duration
High-throughput systemSaga + compensationAvoid cross-service lock contention

Common Interview Questions

  1. What is compensation? How does it differ from a database rollback?
  2. Why must compensating transactions be idempotent?
  3. What happens when a compensating transaction fails? How do you handle it?
  4. How do you track compensation state across a saga?
  5. Design a compensation strategy for an e-commerce order. What are the forward and reverse steps?
  6. How do you handle compensation for external services that don't support "undo"?
  7. What is a Dead Letter Queue and how does it relate to compensation?
  8. How would you test that your compensation logic is correct?
  9. When should you use compensation vs a traditional database rollback?
  10. How do you handle the case where compensation partially reverses the state (e.g., cancellation fee)?
