Outbox Pattern

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★☆

Goal

Understand the Transactional Outbox pattern as the solution to the dual-write problem when you need to atomically update a database and publish a message. This is one of the most practically important patterns in distributed systems.

Definition

The Transactional Outbox pattern solves the dual-write problem: when a service needs to update its database and send a message (to a message broker or another service) as a single atomic operation. Since databases and message brokers cannot participate in the same distributed transaction (without 2PC), the outbox pattern writes the message to a special outbox table within the same database transaction as the business data. A separate process (polling relay or CDC) then publishes the outbox messages to the message broker.

Key Insight: The outbox pattern converts a distributed transaction problem into a single-database transaction followed by an async relay. It trades latency for reliability: the message is published with a small delay, but the system guarantees that if the database write succeeded, the message will eventually be published.

The Dual-Write Problem

The Problem: Two Resources, One Transaction Service: "Create order and publish OrderCreated event" Option 1: Write DB first, then publish message [Write Order to DB] -> [Publish to MQ] | [FAIL] -> DB updated, but no message published -> Inconsistent state! Option 2: Publish message first, then write DB [Publish to MQ] -> [Write Order to DB] | [FAIL] -> Message published, but no DB record -> Ghost message, inconsistent state! Option 3: Use 2PC (XA transaction) [BEGIN XA] -> [Write DB] -> [Publish MQ] -> [COMMIT] Works, but: blocking, slow, couples services

Outbox Pattern Solution

Outbox Table Schema

CREATE TABLE outbox (
    id            UUID PRIMARY KEY,
    aggregate_type VARCHAR(255) NOT NULL,
    aggregate_id  VARCHAR(255) NOT NULL,
    event_type    VARCHAR(255) NOT NULL,
    payload       JSONB NOT NULL,
    created_at    TIMESTAMP NOT NULL DEFAULT NOW(),
    processed_at  TIMESTAMP NULL,        -- NULL = not yet published
    version       BIGINT NOT NULL         -- for idempotent processing
);

CREATE INDEX idx_outbox_unprocessed ON outbox (processed_at)
    WHERE processed_at IS NULL;          -- efficient polling

Message Relay Strategies

Strategy 1: Polling Relay

A background worker polls the outbox table for unprocessed rows and publishes them.

  • Pros: Simple to implement, works with any database
  • Cons: Polling latency (usually 100ms-1s), load on database from frequent queries
  • Optimization: Use processed_at IS NULL with an index, batch polling, or listen/notify

Strategy 2: Change Data Capture (CDC)

A CDC tool (Debezium, Maxwell) reads the database's transaction log (WAL/Binlog) and publishes outbox events.

  • Pros: Near real-time, no polling load on database, more efficient
  • Cons: Additional infrastructure (Debezium, Kafka Connect), operational complexity
  • How: Debezium captures inserts to the outbox table from the WAL, transforms them, and publishes to Kafka

Guarantees Provided

  • At-least-once delivery: If the relay fails before marking processed, it retries. The message may be delivered multiple times.
  • No lost messages: As long as the DB transaction committed, the message will eventually be published.
  • Event ordering: If using CDC with WAL position, ordering is preserved for a single aggregate.
Important: The outbox guarantees at-least-once delivery. Consumers must be idempotent (use the Inbox Pattern from Section 9.2.2) to handle duplicate messages correctly.

Common Interview Questions

  1. What is the dual-write problem? Why can't you just write to DB and publish to Kafka in sequence?
  2. How does the outbox pattern solve the dual-write problem?
  3. What is the outbox table schema? What columns does it need?
  4. Compare polling relay vs CDC for the outbox pattern. When would you choose each?
  5. What delivery guarantee does the outbox provide? What must consumers handle?
  6. How does the outbox pattern relate to the Saga pattern?
  7. What happens if the message relay crashes? Are messages lost?
  8. How do you clean up old outbox messages? What retention policy would you use?
  9. Design an order service that needs to publish events reliably.
