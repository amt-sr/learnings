Inbox Pattern

Difficulty: ⭐⭐⭐☆☆ Interview Frequency: ★★★☆☆

Goal

Understand the Inbox pattern as the consumer-side complement to the Outbox pattern, providing idempotent message processing through deduplication. This is essential for handling at-least-once delivery guarantees.

Definition

The Inbox pattern ensures that a message consumer processes each message exactly once even when the delivery mechanism provides at-least-once guarantees. The consumer maintains an inbox table in its local database. Before processing a message, it checks if the message ID already exists in the inbox. If it does, the message is skipped (duplicate). If not, the message is processed and its ID is inserted into the inbox within the same database transaction.

Key Insight: The inbox pattern turns at-least-once delivery into effectively-once processing. It is the missing piece that makes the outbox pattern safe to use in production: the outbox guarantees at-least-once delivery, and the inbox guarantees idempotent consumption.

Why Idempotency Matters

When using the outbox pattern (or any at-least-once messaging system), messages can be delivered multiple times due to:

  • Message relay retries after failure
  • Consumer processing fails after acknowledging the message
  • Network retransmissions
  • Kafka consumer rebalancing (committed offset lag)

Without the inbox pattern, processing a duplicate message could cause:

  • Double-charging a customer
  • Duplicate order creation
  • Double inventory deduction
  • Corrupted state that is hard to detect and repair

How the Inbox Pattern Works

BEGIN TRANSACTION | | | 3. SELECT from inbox | WHERE message_id = ? | | | ┌-- EXISTS ──> [SKIP] (duplicate) | | return 200 OK | | | NOT EXISTS | | | 4. Process message | (business logic) | | | 5. Inbox Table Schema

CREATE TABLE inbox (
    message_id    VARCHAR(255) PRIMARY KEY,  -- unique message identifier
    aggregate_type VARCHAR(255),             -- e.g., "order", "payment"
    aggregate_id  VARCHAR(255),              -- e.g., "order-123"
    event_type    VARCHAR(255),              -- e.g., "OrderCreated"
    payload       JSONB,                    -- original message payload
    processed_at  TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_inbox_aggregate ON inbox (aggregate_type, aggregate_id);

Processing Logic (Pseudocode)

def process_message(message):
    # Use advisory lock to prevent concurrent processing of same message
    # (PostgreSQL: SELECT pg_advisory_xact_lock(hashtext(message.id)))

    if inbox.exists(message.id):
        log.info(f"Duplicate message skipped: {message.id}")
        return  # Already processed, safely skip

    try:
        db.begin_transaction()

        # Process the business logic
        handle_order_created(message.payload)

        # Record that we processed this message
        inbox.insert(
            message_id=message.id,
            aggregate_type=message.aggregate_type,
            aggregate_id=message.aggregate_id,
            payload=message.payload
        )

        db.commit()
    except Exception:
        db.rollback()
        raise  # Will trigger message redelivery

Cleanup

  • Inbox entries can be cleaned up after a retention period (e.g., 7-30 days)
  • Use a background job to DELETE WHERE processed_at < now() - interval
  • Keep longer if audit requirements exist

Concurrency Control

  • Use database advisory locks to prevent two consumers from processing the same message simultaneously
  • PostgreSQL: SELECT pg_advisory_xact_lock(hashtext(message_id))
  • Alternatively, rely on the UNIQUE constraint on message_id and handle duplicate key errors
