Two Phase Commit (2PC)

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Understand the two-phase commit protocol as the foundational atomic commit protocol for distributed transactions, including its flow, failure modes, and the fundamental blocking problem that limits its practical use at scale.

Definition

Two-Phase Commit (2PC) is a distributed atomic commit protocol where a coordinator node ensures all participants either commit or abort a transaction atomically. It provides atomicity across multiple nodes but sacrifices availability under certain failure conditions, making it a classic example of the CAP theorem tradeoffs.

Key Insight: 2PC is a blocking protocol. If the coordinator crashes after participants vote "yes" but before sending the final decision, participants must block (hold locks) until the coordinator recovers. This is the single most important thing to understand about 2PC.

Protocol Phases

Phase 1: Prepare (Voting Phase)

  1. Coordinator sends PREPARE to all participants
  2. Each participant acquires locks on required resources, writes undo/redo logs, and votes:
    • YES — participant is ready to commit (locks held)
    • NO — participant cannot commit (aborts locally, releases locks)
  3. If any participant votes NO, coordinator decides to ABORT

Phase 2: Commit (Decision Phase)

  1. Coordinator collects all votes
  2. If all votes are YES, coordinator sends COMMIT to all participants
  3. If any vote is NO, coordinator sends ABORT to all participants
  4. Participants execute the decision and release locks
  5. Participants send ACK back to coordinator

The Blocking Problem

The blocking problem is the critical weakness of 2PC. Once a participant votes YES in Phase 1, it cannot unilaterally decide to commit or abort. It must wait for the coordinator's decision. If the coordinator crashes:

  • Participants cannot release locks (commit or abort might happen)
  • Resources remain unavailable to other transactions
  • System throughput degrades under coordinator failures
  • Only recovery after coordinator restart resolves the block

Interview Tip: When asked about 2PC in an interview, always mention the blocking problem first. Then discuss how real systems mitigate it: persistent transaction logs, coordinator election, and why most production systems prefer Saga or eventual consistency over 2PC.

Where 2PC Appears in Real Systems

  • X/Open XA: Standard API for distributed transactions (used by Java EE, JTA)
  • Database internal replication: Raft and Paxos use 2PC-like protocols for log replication commits
  • Spanner: Uses 2PC (TrueTime-assisted) for cross-database transactions
  • MySQL XA: Supports XA transactions across storage engines
  • Oracle/Microsoft DTC: Distributed Transaction Coordinators

2PC in System Design Interviews

Common Design Pattern: In an interview, if you propose 2PC, the interviewer will likely challenge you on availability and latency. Use 2PC only when you can justify the tradeoff: small number of participants, low-latency network, and strong consistency is a hard requirement (e.g., money transfers between accounts).
  • Money transfer between two banks: atomic debit and credit
  • Order fulfillment: inventory deduction + payment processing
  • Inventory held across multiple warehouses (few participants)

Common Interview Questions

  1. What happens when the coordinator crashes after Phase 1? How do participants recover?
  2. Why is 2PC considered a blocking protocol? What does blocking mean here?
  3. How does 2PC relate to the CAP theorem? What does it sacrifice?
  4. Compare 2PC with the Saga pattern. When would you choose one over the other?
  5. How would you design a money transfer system using 2PC? What are the failure modes?
  6. How does 2PC affect system throughput? Why doesn't it scale?
  7. What is the XA protocol and how does it implement 2PC?
  8. How do databases like Spanner or CockroachDB use 2PC?

What to Skip

For L5/L6/Staff interviews, skip:
  • Deep implementation details of XA transaction manager APIs (JTA interfaces, XAResource)
  • Legacy protocols like LU 6.2 syncpoint or CICS recovery
  • Exotic variants (presumed abort, presumed commit optimizations) unless asked
  • Formal proofs of 2PC correctness (FLP impossibility not needed here)
  • Two-phase locking (2PL) concurrency control -- different concept, don't confuse them
