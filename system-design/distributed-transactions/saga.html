Saga Pattern

Difficulty: ⭐⭐⭐⭐☆ Interview Frequency: ★★★★★

Goal

Master the Saga pattern as the go-to approach for managing distributed transactions in microservices architectures. Understand choreography vs orchestration, compensating transactions, and the tradeoffs compared to 2PC.

Definition

A Saga is a sequence of local transactions where each step publishes an event or sends a message that triggers the next step. If any step fails, the Saga executes compensating transactions (reverse operations) to undo the effects of all previously completed steps. Unlike 2PC, a Saga does not hold locks across participants and does not require atomicity -- it achieves eventual consistency through compensation.

Key Insight: The Saga pattern trades atomicity for availability and performance. Instead of one big atomic transaction, you have a series of smaller local transactions with a defined rollback plan. This is the dominant pattern for distributed transactions in microservices.

How a Saga Works

Order Fulfillment Example

Saga: Order Fulfillment Flow [Create Order] | v [Reserve Inventory] -- T1: INSERT reservation | success v [Charge Payment] -- T2: debit customer account | success v [Ship Order] -- T3: create shipment record | success v [Complete Order] -- T4: update order status --- If [Charge Payment] fails --- | v [Refund Inventory] -- C2: cancel reservation (compensate T1)

Compensating Transactions

Each forward step has a corresponding compensating (reverse) step:

Forward TransactionCompensating Transaction
Create orderCancel order
Reserve inventoryRelease inventory reservation
Charge paymentRefund payment
Create shipmentCancel shipment
Send confirmation emailSend cancellation email
Design Consideration: Not all operations need compensation. Side effects like sending emails or logging metrics typically don't need reversal. Focus compensation on operations that affect state: inventory, payments, reservations, and allocations.

Coordination Styles

Choreography (Decentralized)

Each service publishes events. Other services react to events and publish their own events. No central coordinator.

Choreography: Event-Driven Saga Order Service Inventory Service Payment Service | | | |-- OrderCreated ------->| | | (event) | | | |-- process reservation | | | | |<-- InventoryReserved --| | | (event) | | | | | |-------------------------------------------------| | PaymentRequested ------->| | (event) | | |-- charge card | <--- PaymentCompleted ---| | (event) |

Orchestration (Centralized)

A central orchestrator (saga manager) coordinates the flow, sending commands to each service and handling responses.

Orchestration: Centralized Saga Manager Saga Orchestrator | Order Service --| 1. createOrder() | --> OrderCreated | | 2. reserveInventory(orderId) Inventory Service | --> InventoryReserved | | 3. chargePayment(orderId) Payment Service | --> PaymentCompleted (or failed) | | 4. shipOrder(orderId) Shipping Service | --> OrderShipped On failure at step 3: | | 3a. releaseInventory(orderId) Inventory Service | --> InventoryReleased

Choreography vs Orchestration

PropertyChoreographyOrchestration
Control flowDecentralized (events)Centralized (commands)
CouplingLow (services don't know about each other)Higher (orchestrator knows all services)
ComplexityGrows with saga length (hard to trace)Contained in orchestrator
TestingHarder (event-driven)Easier (single orchestrator logic)
Single point of failureNo SPOFOrchestrator is SPOF (mitigate with HA)

Interview Answer: "For 2-4 step sagas with simple flows, I'd use choreography for its lower coupling. For complex business processes with 5+ steps, branching logic, or external integrations, I'd use orchestration for better manageability and observability."

Challenges with Sagas

1. Lack of Isolation (ABA Problem)

Between saga steps, other transactions can read and modify the same data. Example: Inventory is reserved in Step 1, but another order might snatch the last item before Step 2 (payment) completes.

Mitigation: Use semantic locks (marking resources as "pending"), pessimistic locking for critical resources, or serializable isolation levels within individual services.

2. Compensating Transaction Failures

If a compensating transaction fails, you have a cascade problem. The saga may need manual intervention or a retry with exponential backoff.

3. Idempotency Requirement

Since messages can be delivered more than once, every saga step (both forward and compensating) must be idempotent. Use the inbox pattern (Section 9.2.2) to guarantee this.

4. Observability

Debugging a saga that spans 5 services and 10+ events requires distributed tracing, correlation IDs, and centralized logging. This is a significant operational concern.

When to Use Sagas

  • E-commerce order fulfillment: Order -> Inventory -> Payment -> Shipping
  • Travel booking: Flight reservation -> Hotel booking -> Car rental -> Payment
  • Account onboarding: Create account -> Provision resources -> Send welcome email -> Activate
  • Money transfer (non-blocking): Debit source -> Credit destination, with compensation on failure
