AddressSanitizer & Tools

Sanitizers are compiler instrumentation tools that detect runtime errors in C and C++ code. Unlike static analysis tools, sanitizers run as part of the compiled program and catch errors during actual execution. They add a 2x memory overhead and 2-5x CPU overhead but provide extremely precise error detection with near-zero false positive rates.

Compiler Flags Overview

SanitizerFlag (GCC/Clang)Detects
AddressSanitizer (ASan)-fsanitize=addressBuffer overflows, use-after-free, memory leaks
MemorySanitizer (MSan)-fsanitize=memoryUse of uninitialized memory (Clang only)
ThreadSanitizer (TSan)-fsanitize=threadData races, lock order inversions
UndefinedBehaviorSanitizer (UBSan)-fsanitize=undefinedUndefined behavior (signed overflow, null deref)
LeakSanitizer (LSan)-fsanitize=leakMemory leaks (included with ASan)

Important: Use only one memory-related sanitizer at a time (ASan, MSan, or TSan). They each instrument memory differently and conflict with each other. However, UBSan can be combined with any of them.

Compiling with Sanitizers

# GCC / Clang
g++ -fsanitize=address -fno-omit-frame-pointer -g -O1 src/main.cpp -o main
g++ -fsanitize=thread -g -O1 src/main.cpp -o main
g++ -fsanitize=undefined -g -O1 src/main.cpp -o main

# Combine UBSan with ASan
g++ -fsanitize=address,undefined -g -O1 src/main.cpp -o main

# CMake
cmake -DCMAKE_CXX_FLAGS="-fsanitize=address -fno-omit-frame-pointer -g" \
      -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address" -B build

What ASan Detects

  • Heap buffer overflow (reading/writing past allocated size)
  • Stack buffer overflow (writing past local array bounds)
  • Use-after-free (accessing memory after it was freed)
  • Use-after-return (accessing stack variables after function returns)
  • Double-free (calling delete twice on the same pointer)
  • Memory leaks (via integrated LeakSanitizer)

ThreadSanitizer (TSan)

TSan detects data races in multi-threaded programs. It instruments every memory access and uses a happens-before analysis to find unsynchronized concurrent access:

$ ./server
==================
WARNING: ThreadSanitizer: data race (pid=12345)
  Write of size 8 at 0x7b4a00000000 by thread T1:
    #0 increment_counter() src/stats.cpp:15
    #1 void std::thread::_State_impl<...>::_M_run() (libstdc++)

  Previous write of size 8 at 0x7b4a00000000 by main thread:
    #0 reset_counters() src/stats.cpp:22
    #1 main src/main.cpp:10

  Location is global variable 'packet_count' declared at src/stats.cpp:5
SUMMARY: ThreadSanitizer: data race src/stats.cpp:15

UndefinedBehaviorSanitizer (UBSan)

UBSan catches undefined behavior at runtime, complementing static UB detection:

$ ./main
src/math.cpp:42:18: runtime error: signed integer overflow: 2147483647 + 1
src/parser.cpp:78:5: runtime error: null pointer dereference
src/convert.cpp:33:22: runtime error: shift exponent 64 is too large
src/float.cpp:15:10: runtime error: division by zero
src/cast.cpp:22:7: runtime error: member access within null pointer

Interpreting Reports

Report ElementMeaning
heap-use-after-freeReading or writing memory that was already freed
heap-buffer-overflowAccess beyond the bounds of a heap allocation
stack-buffer-overflowAccess beyond the bounds of a stack variable
data raceTwo threads accessing same memory without synchronization
signed integer overflowArithmetic operation overflows a signed integer
null pointer dereferenceDereferencing a pointer that is null

Best practice: Run sanitizers in CI with -fno-omit-frame-pointer -g -O1 for readable stack traces. Use ASAN_OPTIONS=detect_leaks=1:halt_on_error=0 to control behavior. For debugging, use GDB with sanitizers -- set breakpoints on __asan_report_error to catch issues live.
