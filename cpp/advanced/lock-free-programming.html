Lock-Free Programming

Lock-free programming uses atomic operations instead of mutexes to synchronize threads. The goal is to avoid the overhead and pitfalls of locks: priority inversion, deadlocks, convoy effects, and the cost of kernel-mode transitions for contended locks. This topic builds directly on Memory Model & Atomics.

Compare-Exchange (CAS)

The fundamental building block of lock-free algorithms is compare_exchange. It atomically checks whether a value equals an expected value and, if so, replaces it with a desired value:

#include <atomic>
#include <iostream>

int main() {
    std::atomic<int> value{10};
    int expected = 10;
    int desired = 20;

    // CAS succeeds: value was 10, now becomes 20
    bool success = value.compare_exchange_strong(expected, desired);
    std::cout << "Success: " << success << ", value: " << value.load() << "\n";
    // Success: true, value: 20

    // CAS fails: expected was updated to 20 (the actual value)
    success = value.compare_exchange_strong(expected, 99);
    std::cout << "Success: " << success << ", expected (updated): " << expected << "\n";
    // Success: false, expected: 20 (not 99 — expected is updated to actual)
}

compare_exchange_strong vs compare_exchange_weak

VariantBehaviorUse Case
_strongGuarantees success when value equals expected (may spin on false spurious failures)Simpler loops, correctness-critical
_weakMay fail spuriously even when value equals expectedLoops that retry anyway (performance)
// Both produce the same result in a loop, but weak may be faster on some architectures
std::atomic<int> counter{0};

void increment_weak() {
    int expected = counter.load(std::memory_order_relaxed);
    while (!counter.compare_exchange_weak(expected, expected + 1,
                                          std::memory_order_relaxed)) {
        // expected is automatically updated to current value
    }
}

void increment_strong() {
    int expected = counter.load(std::memory_order_relaxed);
    while (!counter.compare_exchange_strong(expected, expected + 1,
                                             std::memory_order_relaxed)) {
        // same loop, but no spurious failures
    }
}

Spinlock Implementation

A spinlock is the simplest lock-free primitive (technically "busy-wait lock"). It polls an atomic flag in a tight loop:

#include <atomic>
#include <thread>

class SpinLock {
    std::atomic<bool> locked_{false};

public:
    void lock() noexcept {
        while (true) {
            // Try to set locked_ from false to true
            bool expected = false;
            if (locked_.compare_exchange_weak(expected, true,
                                               std::memory_order_acquire)) {
                return;  // acquired the lock
            }
            // Spinning — optional: add a small pause to reduce contention
            // std::this_thread::yield(); or _mm_pause() on x86
        }
    }

    void unlock() noexcept {
        locked_.store(false, std::memory_order_release);
    }
};

// Usage
SpinLock lock;
std::atomic<int> shared_data{0};

void worker() {
    lock.lock();
    shared_data.fetch_add(1, std::memory_order_relaxed);
    lock.unlock();
}

Spinlock caveats: A spinlock is not "lock-free" in the strict academic sense (one thread's failure to acquire does not prevent another thread's progress). True lock-free algorithms guarantee system-wide progress. Spinlocks also waste CPU cycles while spinning. Use them only for very short critical sections where the lock hold time is less than the cost of a context switch.

Treiber Stack — Lock-Free Stack

#include <atomic>
#include <memory>

template<typename T>
class LockFreeStack {
    struct Node {
        T data;
        Node* next;
        explicit Node(T val) : data(std::move(val)), next(nullptr) {}
    };

    std::atomic<Node*> head_{nullptr};

public:
    void push(T value) {
        Node* new_node = new Node(std::move(value));
        new_node->next = head_.load(std::memory_order_relaxed);

        // Retry until we successfully link the new node
        while (!head_.compare_exchange_weak(
                   new_node->next,
                   new_node,
                   std::memory_order_release,
                   std::memory_order_relaxed)) {
            // new_node->next is updated to current head on failure
        }
    }

    bool pop(T& result) {
        Node* old_head = head_.load(std::memory_order_acquire);
        while (old_head) {
            Node* next = old_head->next;
            if (head_.compare_exchange_weak(
                    old_head, next,
                    std::memory_order_release,
                    std::memory_order_relaxed)) {
                result = std::move(old_head->data);
                delete old_head;  // WARNING: ABA problem (see below)
                return true;
            }
            // old_head is updated to current head on failure
        }
        return false;  // stack was empty
    }

    ~LockFreeStack() {
        T dummy;
        while (pop(dummy)) {}
    }
};

The ABA Problem

The ABA problem is the classic pitfall of CAS-based algorithms. It occurs when:

  1. Thread 1 reads value A from a shared variable
  2. Thread 2 changes A to B, then back to A
  3. Thread 1's CAS succeeds because the value is still A, but the underlying data may have changed

In the Treiber stack: Between Thread 1 loading old_head and performing CAS, Thread 2 could pop old_head, delete it, allocate a new node that reuses the same memory address, and push it. Thread 1's CAS succeeds, but it now holds a dangling pointer to the deleted node's memory.

Solutions to ABA

// Solution 1: Hazard pointers (each thread publishes pointers it's accessing)
// Solution 2: Epoch-based reclamation
// Solution 3: Tagged pointer (append a counter to the pointer)

// Tagged pointer solution for ABA:
#include <atomic>
#include <cstdint>

template<typename T>
class TaggedPointer {
    static constexpr std::uintptr_t kPointerBits = 48;  // 48-bit address space
    static constexpr std::uintptr_t kCounterBits = 16;
    static constexpr std::uintptr_t kPointerMask =
        (std::uintptr_t{1} << kPointerBits) - 1;

    std::atomic<std::uintptr_t> packed_{0};

public:
    // Pack pointer and counter into a single atomic word
    void set(T* ptr, std::uint16_t counter) {
        std::uintptr_t packed =
            reinterpret_cast<std::uintptr_t>(ptr) |
            (static_cast<std::uintptr_t>(counter) << kPointerBits);
        packed_.store(packed, std::memory_order_release);
    }

    T* pointer() const {
        return reinterpret_cast<T*>(packed_.load(std::memory_order_acquire) & kPointerMask);
    }

    std::uint16_t counter() const {
        return static_cast<std::uint16_t>(
            packed_.load(std::memory_order_acquire) >> kPointerBits);
    }
};

Lock-Free Queue (MPSC)

#include <atomic>
#include <optional>

// A simple multi-producer, single-consumer (MPSC) queue
template<typename T>
class MpscQueue {
    struct Node {
        T data;
        std::atomic<Node*> next{nullptr};

        explicit Node(T val) : data(std::move(val)) {}
    };

    std::atomic<Node*> head_{nullptr};  // producers push here
    Node* tail_ = nullptr;               // consumer reads from here

    // Sentinel node simplifies empty-state handling
    void init() {
        tail_ = new Node(T{});
        head_.store(tail_, std::memory_order_relaxed);
    }

public:
    MpscQueue() { init(); }

    void push(T value) {
        Node* node = new Node(std::move(value));
        Node* expected = head_.load(std::memory_order_acquire);
        do {
            node->next.store(expected, std::memory_order_relaxed);
        } while (!head_.compare_exchange_weak(
                     expected, node,
                     std::memory_order_release,
                     std::memory_order_relaxed));
    }

    std::optional<T> try_pop() {
        // Consumer only — no CAS needed since single consumer
        if (tail_ == head_.load(std::memory_order_acquire)) {
            return std::nullopt;
        }
        Node* node = tail_;
        tail_ = node->next.load(std::memory_order_relaxed);
        T result = std::move(node->data);
        delete node;
        return result;
    }

    ~MpscQueue() {
        while (try_pop().has_value()) {}
        delete tail_;  // delete sentinel
    }
};

When to Use Lock-Free

Use lock-free when: Latency requirements are extremely tight (sub-microsecond), lock contention is measured and problematic, the algorithm is well-known and verified, or you are building infrastructure (queues, pools) used by many threads.

Avoid lock-free when: The algorithm is complex and unverified, you are optimizing prematurely, your team lacks concurrency expertise, or a mutex would be simpler and fast enough. A correct mutex-based solution is always preferable to a buggy lock-free one.

Telecom Relevance

In high-throughput telecom (5G UPF, base station datapath), packet processing pipelines process millions of packets per second. Lock-free queues between processing stages eliminate lock contention on the critical path. Lock-free counters for statistics avoid cache-line bouncing from mutex acquisition.
