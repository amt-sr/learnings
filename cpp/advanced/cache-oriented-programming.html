Cache-Oriented Programming

Modern CPUs can execute instructions far faster than they can fetch data from main memory. The CPU cache hierarchy (L1, L2, L3) bridges this gap, but only if your data access patterns are cache-friendly. Cache-oriented programming structures data and code to maximize cache hit rates and minimize cache misses, false sharing, and unnecessary cache line transfers. This topic follows Placement New & Memory Pools and leads into Advanced constexpr.

Key principle: A cache miss to main memory costs 200-400 cycles. In that time, the CPU could have executed 100-200 instructions from L1 cache. Structure your data so that hot paths hit L1/L2, not main memory.

Cache Lines

Memory is transferred between CPU caches and main memory in fixed-size units called cache lines (typically 64 bytes on x86-64, 128 bytes on some ARM). The entire cache line is loaded even if you only read one byte.

#include <cstddef>

// A typical cache line size
static constexpr std::size_t CACHE_LINE = 64;

// Padding to prevent false sharing
template<typename T>
struct CacheLinePadded {
    alignas(CACHE_LINE) T value{};
    char padding[CACHE_LINE - ((sizeof(T) + CACHE_LINE - 1) % CACHE_LINE)];
};

False Sharing

False sharing occurs when two threads modify independent variables that happen to reside on the same cache line. The hardware cache coherency protocol (MESI) treats the entire line as a single unit, causing the line to bounce between cores:

#include <atomic>
#include <thread>
#include <vector>
#include <iostream>
#include <chrono>

// BAD: Both counters likely on the same cache line
struct CountersBad {
    std::atomic<int64_t> counter_a{0};  // offset 0
    std::atomic<int64_t> counter_b{0};  // offset 8 (same cache line!)
};

// GOOD: Each counter on its own cache line
struct alignas(64) CountersGood {
    std::atomic<int64_t> counter_a{0};
    char pad_a[64 - sizeof(std::atomic<int64_t>)];  // padding
    std::atomic<int64_t> counter_b{0};
    char pad_b[64 - sizeof(std::atomic<int64_t>)];  // padding
};

template<typename C>
void benchmark(const char* name, int iterations) {
    C counters;
    constexpr int N = 1'000'000;

    auto start = std::chrono::high_resolution_clock::now();

    std::thread t1([&]() {
        for (int i = 0; i < N; ++i) counters.counter_a.fetch_add(1, std::memory_order_relaxed);
    });
    std::thread t2([&]() {
        for (int i = 0; i < N; ++i) counters.counter_b.fetch_add(1, std::memory_order_relaxed);
    });
    t1.join();
    t2.join();

    auto end = std::chrono::high_resolution_clock::now();
    auto ms = std::chrono::duration_cast<std::chrono::microseconds>(end - start).count();
    std::cout << name << ": " << ms << " us\n";
}

int main() {
    benchmark<CountersBad>("False sharing (bad) ", 0);
    benchmark<CountersGood>("Padded (good)       ", 0);
    // Padded version is typically 3-10x faster
}

Data Layout Optimization

Array of Structs (AoS) vs Struct of Arrays (SoA)

#include <vector>
#include <cstddef>

constexpr std::size_t N = 10'000'000;

// AoS: Cache-friendly for accessing all fields of one object
struct ParticleAoS {
    float x, y, z;    // position
    float vx, vy, vz;  // velocity
    float mass;
};
std::vector<ParticleAoS> particles_aos(N);

// SoA: Cache-friendly for accessing one field of many objects
struct ParticleSoA {
    std::vector<float> x, y, z;      // position arrays
    std::vector<float> vx, vy, vz;   // velocity arrays
    std::vector<float> mass;
};
ParticleSoA particles_soa;
particles_soa.x.resize(N); particles_soa.y.resize(N); particles_soa.z.resize(N);
particles_soa.vx.resize(N); particles_soa.vy.resize(N); particles_soa.vz.resize(N);
particles_soa.mass.resize(N);

// When updating only velocities (not positions), SoA is much faster:
// - AoS: loads entire ParticleAoS (28 bytes, spanning ~0.5 cache lines per particle)
// - SoA: loads only vx, vy, vz (12 bytes, fits in one cache line for ~5 particles)

void update_velocities_aos() {
    for (auto& p : particles_aos) {
        p.vx += p.ax * dt;  // loads x, y, z, vx, vy, vz, mass — wastes bandwidth
        p.vy += p.ay * dt;
        p.vz += p.az * dt;
    }
}

void update_velocities_soa() {
    for (std::size_t i = 0; i < N; ++i) {
        particles_soa.vx[i] += ax * dt;  // loads only vx, vy, vz — tight loop
        particles_soa.vy[i] += ay * dt;
        particles_soa.vz[i] += az * dt;
    }
}

Rule: Use AoS when you access most/all fields of each object together. Use SoA when you process one field across all objects. In telecom: per-UE processing is typically AoS; per-metric aggregation across all UEs benefits from SoA.

Prefetching

#include <cstddef>

// Manual prefetch hint (compiler-specific, x86 example)
#if defined(__x86_64__)
#include <x86intrin.h>
#define PREFETCH(addr) _mm_prefetch(reinterpret_cast<const char*>(addr), _MM_HINT_T0)
#elif defined(__aarch64__)
#define PREFETCH(addr) __builtin_prefetch(addr, 1, 3)
#else
#define PREFETCH(addr)
#endif

// Example: prefetching in a linked-list traversal
struct Node {
    int data;
    Node* next;
};

void process_list(Node* head) {
    for (Node* curr = head; curr != nullptr; curr = curr->next) {
        // Prefetch the NEXT node's cache line before processing current
        if (curr->next) {
            PREFETCH(curr->next);
        }
        // Process current node
        process(curr->data);
    }
}

Column-Major vs Row-Major Traversal

constexpr std::size_t ROWS = 1024;
constexpr std::size_t COLS = 1024;

int matrix[ROWS][COLS];

// Row-major traversal: sequential memory access — FAST
void sum_rows() {
    long sum = 0;
    for (std::size_t r = 0; r < ROWS; ++r) {
        for (std::size_t c = 0; c < COLS; ++c) {
            sum += matrix[r][c];  // sequential: matrix[0][0], [0][1], [0][2], ...
        }
    }
}

// Column-major traversal: stride-ROWS access — SLOW (cache miss per access)
void sum_cols() {
    long sum = 0;
    for (std::size_t c = 0; c < COLS; ++c) {
        for (std::size_t r = 0; r < ROWS; ++r) {
            sum += matrix[r][c];  // strided: matrix[0][0], [1][0], [2][0], ...
        }
    }
}

Telecom Relevance

Packet processing in a 5G base station involves iterating over millions of packets per second. Each packet header is parsed and forwarded. If packet buffers are cache-line aligned and headers are compact, a single cache line fetch can deliver an entire header. SoA layout for per-UE statistics allows efficient aggregation of metrics across all UEs without loading unnecessary data. Avoiding false sharing between per-core statistics counters is essential for scaling to many cores.

Optimization checklist: (1) Profile before optimizing, (2) Use Valgrind/cachegrind to measure cache misses, (3) Ensure hot data fits in L1/L2, (4) Eliminate false sharing with padding or thread-local data, (5) Align frequently accessed structures to cache-line boundaries, (6) Use sequential access patterns where possible.
