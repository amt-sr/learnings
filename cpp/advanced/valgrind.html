Valgrind

Valgrind is a suite of instrumentation tools for detecting memory errors, threading bugs, and performance bottlenecks in C and C++ programs. Unlike AddressSanitizer (which requires recompilation), Valgrind works on unmodified binaries by running them on a synthetic CPU. This makes it ideal for analyzing programs you cannot recompile.

Valgrind Tools Overview

ToolInvocationPurpose
Memcheckvalgrind ./myapp (default)Detect memory errors: leaks, invalid access, uninitialized reads
Callgrindvalgrind --tool=callgrind ./myappProfiling: call graph, instruction counts, cache simulation
Cachegrindvalgrind --tool=cachegrind ./myappCache miss analysis: L1, LL cache hits/misses
Helgrindvalgrind --tool=helgrind ./myappThread error detection: data races, lock ordering
DRDvalgrind --tool=drd ./myappAlternative thread error detector (different algorithm)
Massifvalgrind --tool=massif ./myappHeap profiling: memory usage over time

Memcheck -- Memory Error Detection

Memcheck is the most commonly used Valgrind tool. It detects every byte of memory in the program and tracks whether it has been initialized, freed, or is still accessible.

Running Memcheck

# Basic run
valgrind ./myapp

# Full leak check with leak origins
valgrind --leak-check=full --show-leak-kinds=all --track-origins=yes ./myapp

# Report to file
valgrind --leak-check=full --log-file=valgrind-report.txt ./myapp

# Suppress known false positives
valgrind --suppressions=myapp.supp ./myapp

Suppression Files

Suppress known false positives or third-party library issues:

# myapp.supp
{
    # Suppress leak from known library issue
    lib_some_leak
    Memcheck:Leak
    match-leak-kinds: reachable
    fun:malloc
    fun:lib_init_function
    ...
}
{
    # Suppress uninitialized read in third-party code
    third_party_uninit
    Memcheck:Value8
    fun:third_party_parse
    ...
}

# Run with suppression file
valgrind --suppressions=myapp.supp --leak-check=full ./myapp

Callgrind -- Profiling

# Run with Callgrind
valgrind --tool=callgrind --callgrind-out-file=callgrind.out ./myapp

Valgrind vs AddressSanitizer

AspectValgrindASan
Recompilation neededNo (works on any binary)Yes (-fsanitize=address)
Performance overhead10-50x slower2x slower
Memory overhead3-5x more RAM2x more RAM
Use-after-freeYesYes
Uninitialized readsYes (Memcheck)Yes (MSan, Clang only)
Data racesYes (Helgrind)Yes (TSan)
Stack analysisLimitedExcellent (buffer overflows)
PlatformLinux, macOS (limited)All (GCC/Clang)

Recommendation: Use ASan for everyday development (fast, catches most memory bugs). Use Valgrind Memcheck for cases where you cannot recompile or need to detect uninitialized memory reads on GCC. Use Helgrind as a complement to TSan. See GDB Debugging for interactive debugging and Memory Leak Detection for practice.
