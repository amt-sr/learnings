Modules (C++20)

Modules are a replacement for the traditional #include / header-file system. They provide better encapsulation, faster compilation (no redundant text inclusion), and eliminate macro pollution across translation units. Modules are one of the most significant build-system changes in C++ history.

Why Modules Replace Headers

AspectHeaders (#include)Modules (import)
Compilation modelText substitution per TUCompiled once, reused
Build timeO(n * includes) — quadraticO(n) — linear
Macro isolationNone — macros leak everywhereFull isolation between modules
EncapsulationOnly #pragma once or include guardsExplicit export declarations
PreprocessorRuns on every TUMinimal preprocessor in module units
Error isolationErrors in headers affect all TUsErrors are isolated to the module

Module Declaration Types

// ====== File: math_utils.cppm (Module Interface Unit) ======
export module math_utils;  // declares the module "math_utils"

export int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);
}

export double degrees_to_radians(double deg) {
    return deg * 3.14159265358979323846 / 180.0;
}

// Not exported — internal to this module
namespace detail {
    constexpr double pi = 3.14159265358979323846;
    double radians_to_degrees(double rad) {
        return rad * 180.0 / pi;
    }
}

export double pi_value() { return detail::pi; }

// ====== File: main.cpp (Consumer) ======
import math_utils;
import std.core;  // or import <iostream> (compiler-dependent)

int main() {
    std::cout << "5! = " << factorial(5) << "\n";
    std::cout << "90 deg = " << degrees_to_radians(90.0) << " rad\n";

    // detail::radians_to_degrees(1.0);  // ERROR: not exported
}

Module Partitions

Large modules can be split into partitions for organizational purposes. Each partition is a separate translation unit that contributes to the same module.

// ====== File: protocol.cppm (Primary Module Interface) ======
export module protocol;

// Import partitions
import :rrc;   // the RRC partition
import :nas;   // the NAS partition
import :ngap;  // the NGAP partition

// Re-export everything from partitions
export import :rrc;
export import :nas;
export import :ngap;

// ====== File: protocol_rrc.cppm (Partition Module Interface) ======
module protocol:rrc;  // partition "rrc" of module "protocol"

export class RrcMessage {
    int msg_type = 0;
    std::uint8_t buffer[4096]{};
public:
    void set_type(int type) { msg_type = type; }
    [[nodiscard]] int type() const { return msg_type; }
    std::size_t serialize(std::uint8_t* dst) const;
};

// ====== File: protocol_nas.cppm (Partition Module Interface) ======
module protocol:nas;

export class NasMessage {
    int procedure_code = 0;
public:
    void set_procedure(int code) { procedure_code = code; }
    [[nodiscard]] int procedure() const { return procedure_code; }
};

// ====== File: protocol_ngap.cppm (Partition Module Interface) ======
module protocol:ngap;

export class NgapMessage {
    int pdu_type = 0;
public:
    void set_pdu_type(int type) { pdu_type = type; }
    [[nodiscard]] int pdu_type() const { return pdu_type; }
};

// ====== File: protocol_rrc_impl.cpp (Module Implementation Unit) ======
module protocol:rrc;  // same partition, implementation file

// This file is NOT exported. It provides definitions for
// declarations in the partition's interface.

std::size_t RrcMessage::serialize(std::uint8_t* dst) const {
    std::memcpy(dst, buffer, sizeof(buffer));
    return sizeof(buffer);
}

Module Implementation Units

Implementation units (.cpp files) belong to a module but do not export anything. They provide definitions for declarations made in the module interface:

// ====== File: math_utils_impl.cpp ======
module math_utils;  // belongs to module, but exports nothing

// This definition is visible inside the module but not to importers.
// Only the exported declarations in math_utils.cppm are visible externally.

Partition naming: Module partition names start with :. The primary module interface must export-import all partitions. Implementation units use the module name without export.

Importing Standard Library

How you import the standard library depends on your compiler and its level of C++23 module support:

// GCC 14+ with C++23 modules
import std;

// Clang with modules
import std.core;
import std.io;
import std.algorithm;

// MSVC (Visual Studio 2022+)
import std.core;
import std.filesystem;

// Fallback: mixing headers and modules is allowed
import math_utils;
#include <chrono>  // still works

Global Module Fragment

When a module needs to use macros or include traditional headers, use the global module fragment:

module;  // global module fragment starts here
#include <cstring>  // still needed for strcpy, etc.
#include <cstdint> // std::uint8_t etc.

export module network_utils;  // module declaration ends global fragment

export void copy_buffer(std::uint8_t* dst, const std::uint8_t* src, std::size_t n) {
    std::memcpy(dst, src, n);  // available from global module fragment
}

Modules vs Headers — Decision Guide

Use modules when: Starting a new project, refactoring a large legacy codebase, building library boundaries, or when build times are a critical concern. Modules give you encapsulation by default.

Use headers when: Interfacing with C libraries, supporting older compilers, or working in a codebase that hasn't migrated. Headers remain fully supported and will not be removed.

Compiler support (2026): GCC 14+, Clang 18+, MSVC 19.38+ all support C++20 modules. C++23 adds import std; as a single import for the entire standard library. Check your toolchain before migrating.

Telecom Relevance

In a 5G protocol stack, modules map naturally to protocol layers. Each layer (PHY, MAC, RLC, PDCP, RRC, NGAP) can be a module with clear export boundaries. Internal implementation details (e.g., buffer management, encoding routines) stay hidden. This prevents accidental coupling between layers — a common source of bugs in monolithic telecom codebases.
