Undefined Behavior & Pitfalls

Undefined behavior (UB) is the result of executing code that the C++ standard does not define. When UB occurs, the compiler is free to assume it never happens and may optimize code in ways that produce unexpected, dangerous, or seemingly impossible results. UB is not just "undefined" -- it is actively dangerous because the compiler uses the assumption of no-UB to make optimization decisions.

Key insight: UB does not mean "the program crashes." It means "the program may do literally anything." The compiler may eliminate code paths that would only be reachable through UB, produce incorrect results silently, or introduce security vulnerabilities. UB is a time bomb that can go off years later when a compiler update changes optimization heuristics.

Why UB Is Dangerous: Compiler Optimization Example

// Example: Compiler assumes signed overflow never happens
int compute(int x) {
    if (x + 1 > x) {
        return x + 1;    // This path is kept
    }
    return 0;              // Compiler may eliminate this path entirely
}

// The compiler reasons:
// "Signed overflow is UB. If x + 1 overflows, that is UB.
//  Therefore x + 1 > x is ALWAYS true for well-defined programs.
//  I can eliminate the else branch."
//
// Result: the function always returns x + 1, even when x == INT_MAX
// which is UB, but the compiler removed the safety check!

Detailed Examples

Signed Integer Overflow

#include <climits>

int sum_array(const int* arr, int n) {
    int sum = 0;
    for (int i = 0; i < n; ++i) {
        sum += arr[i];  // UB if sum overflows INT_MAX
    }
    return sum;
}

// Fix: Use wider type or checked arithmetic
#include <cstdint>
int64_t sum_array_safe(const int* arr, int n) {
    int64_t sum = 0;
    for (int i = 0; i < n; ++i) {
        sum += arr[i];
    }
    return sum;
}

Use-After-Free

#include <memory>

void use_after_free_example() {
    auto* ptr = new int{42};
    delete ptr;
    std::cout << *ptr << "\n";  // UB: use-after-free
}

// Fix: Use smart pointers (RAII)
void safe_example() {
    auto ptr = std::make_unique<int>(42);
    std::cout << *ptr << "\n";  // Safe: unique_ptr owns the memory
    // ptr is automatically deleted when it goes out of scope
}

Data Races

#include <thread>

int counter = 0;  // Non-atomic shared variable

void bad_increment() {
    for (int i = 0; i < 1000; ++i) {
        counter++;  // DATA RACE -- concurrent non-atomic access
    }
}

// Fix: Use std::atomic (see memory-model-atomics)
#include <atomic>
std::atomic<int> safe_counter{0};

void safe_increment() {
    for (int i = 0; i < 1000; ++i) {
        safe_counter.fetch_add(1, std::memory_order_relaxed);
    }
}

Compile with Maximum UB Detection

# GCC / Clang: Enable sanitizer + warnings + traps
g++ -std=c++17 -Wall -Wextra -Wpedantic \
    -fsanitize=address,undefined \
    -fno-omit-frame-pointer \
    -ftrapv \
    -g -O1 \
    src/main.cpp -o main

# Ftrapv causes signed overflow to trap instead of wrapping
# Useful for debugging, but adds runtime overhead

Best practice: Treat compiler warnings as errors (-Werror). Use sanitizers in test suites. Use smart pointers to prevent use-after-free and double-free. Use std::atomic to prevent data races. Never rely on undefined behavior, even if it "works" on your current compiler.
