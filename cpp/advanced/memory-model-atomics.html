Memory Model & Atomics

The C++ memory model defines how threads interact through shared memory. It specifies the guarantees the compiler and hardware must provide about the visibility and ordering of memory operations. std::atomic is the language primitive that enables lock-free communication between threads without data races.

Critical distinction: A data race (two threads accessing the same non-atomic variable, at least one writing, without synchronization) is undefined behavior. An race condition (logic bug from unexpected interleaving) is a logic error but not UB. Atomics prevent data races; they do not automatically prevent race conditions. See Undefined Behavior for more details.

std::atomic Basics

#include <atomic>
#include <thread>
#include <iostream>

std::atomic<int> counter{0};

void increment(int times) {
    for (int i = 0; i < times; ++i) {
        counter.fetch_add(1, std::memory_order_relaxed);
        // equivalent: counter++; (uses memory_order_seq_cst by default)
    }
}

int main() {
    constexpr int N = 1'000'000;
    std::thread t1(increment, N);
    std::thread t2(increment, N);
    t1.join();
    t2.join();
    std::cout << "Counter = " << counter.load() << "\n";
    // Always prints 2000000 — no data race
}

Memory Orders

Memory orders control the synchronization guarantees between threads. Each level provides stronger guarantees at the cost of potentially more expensive instructions:

Memory OrderGuaranteeCostUse Case
memory_order_relaxedNo ordering — only atomicityCheapestCounters, statistics
memory_order_acquirePrevents subsequent reads/writes from reordering before this loadModerateReading shared state
memory_order_releasePrevents prior reads/writes from reordering after this storeModeratePublishing shared state
memory_order_acq_relBoth acquire and release semanticsExpensiveRead-modify-write ops
memory_order_seq_cstTotal order across all threads (default)Most expensiveGeneral purpose

Acquire-Release Pattern

#include <atomic>
#include <thread>
#include <iostream>
#include <cassert>

std::atomic<bool> ready{false};
int payload = 0;

void producer() {
    payload = 42;                          // (1) ordinary write
    ready.store(true,                      // (2) release store
                 std::memory_order_release);
    // (1) is guaranteed to be visible before (2) to any thread
    // that synchronizes via this release store
}

void consumer() {
    while (!ready.load(std::memory_order_acquire)) {
        // spin until ready
    }
    // acquire load synchronizes with release store
    // guaranteeing payload == 42 is visible here
    assert(payload == 42);
    std::cout << "Payload: " << payload << "\n";
}

int main() {
    std::thread t1(producer);
    std::thread t2(consumer);
    t1.join();
    t2.join();
}

Relaxed Ordering — When It Is Safe

#include <atomic>
#include <thread>
#include <vector>
#include <iostream>

// Relaxed is fine for independent counters — no ordering needed
std::atomic<std::uint64_t> packets_received{0};
std::atomic<std::uint64_t> packets_sent{0};

void process_packets(int id) {
    for (int i = 0; i < 1000; ++i) {
        packets_received.fetch_add(1, std::memory_order_relaxed);
        // ... do work ...
        packets_sent.fetch_add(1, std::memory_order_relaxed);
    }
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 8; ++i) {
        threads.emplace_back(process_packets, i);
    }
    for (auto& t : threads) t.join();
    std::cout << "Sent: " << packets_sent.load() << "\n";
    std::cout << "Received: " << packets_received.load() << "\n";
}

Warning: Never use relaxed ordering when one thread's read of a variable depends on another thread's write of a different variable. The relaxed thread may see the variables updated in the wrong order.

C++20 Atomic Wait/Notify

#include <atomic>
#include <thread>
#include <iostream>
#include <vector>

std::atomic<int> signal{0};

void waiter(int id) {
    int expected = 0;
    signal.wait(expected, std::memory_order_acquire);
    std::cout << "Thread " << id << " woke up! Signal = "
              << signal.load() << "\n";
}

void notifier() {
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
    signal.store(1, std::memory_order_release);
    signal.notify_all();  // wake all waiting threads
}

int main() {
    std::vector<std::thread> threads;
    for (int i = 0; i < 4; ++i) {
        threads.emplace_back(waiter, i);
    }
    threads.emplace_back(notifier);
    for (auto& t : threads) t.join();
}

Telecom Relevance

In telecom software, atomics are used for lock-free statistics (packet counters, throughput meters), state flags (connection status, thread lifecycle), and notification between threads in a pipeline architecture. A 5G base station processes millions of packets per second — using the wrong memory order can cause subtle data races that only manifest under load on specific hardware architectures.

Rule of thumb: Start with seq_cst (the default). Only optimize to relaxed or acquire/release after profiling shows the atomic operations are a bottleneck and you can prove the weaker ordering is correct for your specific access pattern.
