Singleton Pattern

The Singleton pattern ensures a class has exactly one instance and provides a global point of access to it. While widely used (and often misused), the Singleton is one of the GoF creational patterns. In modern C++, there are several implementation strategies with different trade-offs in thread safety, lazy initialization, and lifetime management.

Prerequisites: Observer Pattern, Factory Pattern, RAII Pattern, Mutex & Locks.

Meyer's Singleton (Recommended)

#include <iostream>
#include <string>
#include <mutex>

// Meyer's Singleton: thread-safe in C++11+
// The compiler generates guard variables with atomic checks
class Logger {
    std::string name_;
    int log_count_ = 0;

    // Private constructor — prevents external instantiation
    Logger() : name_("GlobalLogger") {
        std::cout << "Logger initialized\n";
    }

public:
    // Delete copy and move operations
    Logger(const Logger&) = delete;
    Logger& operator=(const Logger&) = delete;
    Logger(Logger&&) = delete;
    Logger& operator=(Logger&&) = delete;

    // The static local variable is initialized on first call
    // C++11 guarantees this is thread-safe (magic statics)
    static Logger& instance() {
        static Logger logger;
        return logger;
    }

    void log(const std::string& msg) {
        ++log_count_;
        std::cout << "[" << log_count_ << "] " << name_ << ": " << msg << "\n";
    }

    int count() const { return log_count_; }
};

int main() {
    Logger::instance().log("Application started");
    Logger::instance().log("Processing data");
    Logger::instance().log("Application finished");

    std::cout << "Total logs: " << Logger::instance().count() << "\n";
}

Thread-Safe Singleton with call_once

#include <iostream>
#include <mutex>

class Config {
    int max_connections_ = 100;
    int timeout_ms_ = 5000;
    bool debug_mode_ = false;

    static Config* instance_;
    static std::once_flag init_flag_;

    Config() = default;
public:
    Config(const Config&) = delete;
    Config& operator=(const Config&) = delete;

    static Config& instance() {
        std::call_once(init_flag_, []() {
            instance_ = new Config();
            // Register atexit for cleanup
            std::atexit([]() { delete instance_; });
        });
        return *instance_;
    }

    // Getters/setters
    int max_connections() const { return max_connections_; }
    void set_max_connections(int n) { max_connections_ = n; }
};

Config* Config::instance_ = nullptr;
std::once_flag Config::init_flag_;

int main() {
    // Safe to call from multiple threads
    Config::instance().set_max_connections(200);
    std::cout << "Max connections: " << Config::instance().max_connections() << "\n";
}

Singleton Pros and Cons

Advantages

  • Controlled access to a single instance — no accidental duplicates
  • Lazy initialization — created only when first needed
  • Global access point without global variables
  • Can be subclassed (though this is rarely done)

Disadvantages

  • Hidden dependencies: Functions that use Singleton::instance() hide their dependency on the global state
  • Testing difficulty: Cannot easily substitute a mock instance in unit tests
  • Global state: Shared mutable state makes concurrent reasoning harder
  • Initialization order: Static destruction order is the reverse of construction — singletons may be destroyed before other objects that depend on them
  • Violates dependency injection: Better to pass dependencies explicitly

Testing difficulty: Singletons make unit testing hard because tests share the same global instance. If test A modifies the singleton, test B sees the modification. Use dependency injection instead — pass the singleton as a reference parameter or use an interface with Factory Pattern for test doubles.

When to use Singleton: True singletons in a system are rare. Good candidates: a logging subsystem, a hardware device wrapper, or a configuration manager. Bad candidates: objects that happen to have one instance "for now" but may need multiple instances later. See Singleton Pattern Practice for exercises.

Dependency Injection Alternative

The main problem with Singletons is hidden dependencies and testability. Dependency injection (DI) solves both by passing dependencies explicitly through constructors:

#include <iostream>
#include <memory>
#include <string>

// Interface for logging (allows test doubles)
class ILogger {
public:
    virtual ~ILogger() = default;
    virtual void log(const std::string& msg) = 0;
    virtual int count() const = 0;
};

// Production implementation
class ConsoleLogger : public ILogger {
    int log_count_ = 0;
public:
    void log(const std::string& msg) override {
        ++log_count_;
        std::cout << "[" << log_count_ << "] " << msg << "\n";
    }
    int count() const override { return log_count_; }
};

// Test double (mock)
class MockLogger : public ILogger {
    int log_count_ = 0;
public:
    void log(const std::string&) override { ++log_count_; }
    int count() const override { return log_count_; }
    std::string last_message;
};

// Services receive logger via constructor injection
class DatabaseService {
    std::shared_ptr<ILogger> logger_;
public:
    explicit DatabaseService(std::shared_ptr<ILogger> logger)
        : logger_(std::move(logger)) {}

    void connect() {
        logger_->log("Database connected");
    }

    void query(const std::string& sql) {
        logger_->log("Executing: " + sql);
    }
};

int main() {
    // Production: real logger
    auto real_logger = std::make_shared<ConsoleLogger>();
    DatabaseService db_prod(real_logger);
    db_prod.connect();

    // Testing: inject mock to verify behavior
    auto mock_logger = std::make_shared<MockLogger>();
    DatabaseService db_test(mock_logger);
    db_test.connect();
    db_test.query("SELECT * FROM users");

    std::cout << "Log count in test: " << mock_logger->count() << "\n";
    // Output: Log count in test: 2
}

Key advantage: With DI, tests inject a MockLogger that records calls without side effects. With a Singleton, tests would share the same global logger instance and could not verify specific interactions.
