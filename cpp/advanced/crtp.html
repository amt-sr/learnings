Curiously Recurring Template Pattern (CRTP)

CRTP is a C++ idiom where a class derives from a base class that is templated on the derived class itself. This enables static polymorphism: the base class can call methods on the derived type without virtual dispatch. CRTP provides the benefits of polymorphism (code reuse, interface contracts) without the runtime cost of virtual functions (vtable lookups, indirect calls).

Prerequisites: Templates, Inheritance & Polymorphism, Advanced constexpr.

Basic CRTP Pattern

#include <iostream>
#include <string>

// CRTP base class
template<typename Derived>
class Serializer {
public:
    // Base class calls derived class method — static dispatch
    std::string serialize() const {
        const Derived& d = static_cast<const Derived&>(*this);
        return d.to_string();
    }

    void print() const {
        std::cout << serialize() << "\n";
    }
};

// Derived class: provides the actual implementation
class PacketHeader : public Serializer<PacketHeader> {
    int src_id_ = 0;
    int dst_id_ = 0;
    int type_ = 0;

public:
    PacketHeader(int src, int dst, int type)
        : src_id_(src), dst_id_(dst), type_(type) {}

    std::string to_string() const {
        return "Packet[src=" + std::to_string(src_id_)
             + " dst=" + std::to_string(dst_id_)
             + " type=" + std::to_string(type_) + "]";
    }
};

class MeasurementReport : public Serializer<MeasurementReport> {
    double value_ = 0.0;
    std::string unit_ = "dBm";

public:
    MeasurementReport(double val, std::string unit)
        : value_(val), unit_(std::move(unit)) {}

    std::string to_string() const {
        return "Measurement[" + std::to_string(value_) + " " + unit_ + "]";
    }
};

int main() {
    PacketHeader p(1, 2, 0);
    p.print();  // Packet[src=1 dst=2 type=0]

    MeasurementReport m(-85.5, "dBm");
    m.print();  // Measurement[-85.5 dBm]

    // No vtable — serialize() is inlined
}

AspectCRTP (Static)Virtual (Dynamic)
Dispatch mechanismCompile-time (static_cast)Runtime (vtable lookup)
Indirection costNone (can be inlined)One indirect call per method
Object sizeNo vptr overheadOne vptr (8 bytes) per object
Heterogeneous containersNot directly possibleEasy (std::vector<Base*>)

Mixin Pattern

CRTP is commonly used to add functionality (mixins) to a class without virtual functions:

#include <iostream>
#include <string>

// Mixin: adds comparison operators
template<typename Derived>
class Comparable {
    [[nodiscard]] bool operator!=(const Derived& other) const {
        return !(static_cast<const Derived&>(*this) == other);
    }
    [[nodiscard]] bool operator>(const Derived& other) const {
        return other < static_cast<const Derived&>(*this);
    }
    [[nodiscard]] bool operator<=(const Derived& other) const {
        return !(static_cast<const Derived&>(*this) > other);
    }
    [[nodiscard]] bool operator>=(const Derived& other) const {
        return !(static_cast<const Derived&>(*this) < other);
    }
};

// Mixin: adds serialization
template<typename Derived>
class Serializable {
public:
    std::string debug_string() const {
        return static_cast<const Derived&>(*this).describe();
    }
};

// Mixin: adds reference counting
template<typename Derived>
class ReferenceCounted {
    int ref_count_ = 1;
public:
    void add_ref() { ++ref_count_; }
    void release() { if (--ref_count_ == 0) delete static_cast<Derived*>(this); }
    [[nodiscard]] int ref_count() const { return ref_count_; }
};

// Combine multiple mixins
class UeContext : public Comparable<UeContext>,
                  public Serializable<UeContext> {
    int ue_id_ = 0;
    std::string state_ = "idle";

public:
    UeContext(int id, std::string state) : ue_id_(id), state_(std::move(state)) {}

    int ue_id() const { return ue_id_; }

    // Required by Comparable (only == is needed; !=, >, <=, >= are auto-generated)
    bool operator==(const UeContext& other) const {
        return ue_id_ == other.ue_id_;
    }

    // Required by Serializable
    std::string describe() const {
        return "UE[id=" + std::to_string(ue_id_) + " state=" + state_ + "]";
    }
};

int main() {
    UeContext ue1(1, "connected");
    UeContext ue2(2, "idle");

    // Mixin-provided operators
    std::cout << (ue1 != ue2) << "\n";  // true (from Comparable)
    std::cout << (ue1 > ue2) << "\n";   // false (from Comparable)
    std::cout << ue1.debug_string() << "\n";  // UE[id=1 state=connected]
}

CRTP with Override Enforcement

A common CRTP pitfall: the derived class forgets to override a method, and the base class's version is called, leading to infinite recursion. Use static_assert to enforce the contract:

#include <type_traits>

template<typename Derived>
class Enforcer {
public:
    void process() {
        // Compile-time check: Derived must implement process_impl
        static_assert(
            requires(const Derived& d) { d.process_impl(); },
            "Derived must implement process_impl()"
        );

        static_cast<Derived*>(this)->process_impl();
    }

protected:
    // This will trigger a linker error if Derived doesn't override
    void process_impl() {
        static_assert(!std::is_same_v<Derived, Enforcer<Derived>>,
                      "Derived class must override process_impl()");
    }
};

CRTP vs Type Erasure

CRTP and Type Erasure solve the same problem (interface-based dispatch) at opposite ends of the flexibility spectrum:

  • CRTP: Static dispatch, no runtime polymorphism, best performance, but the concrete type must be known at compile time
  • Type Erasure: Runtime polymorphism without virtual functions, value semantics, can store heterogeneous types, but with some runtime overhead

When to use CRTP: When the derived type is known at the point of use, you need maximum performance, and you want to add reusable behavior (mixins) to a class hierarchy. Avoid CRTP for heterogeneous containers — use type erasure or traditional virtual dispatch instead.

Telecom Relevance

In protocol stacks, CRTP provides zero-overhead interfaces for message handling. A base MessageHandler<Derived> can define the common processing pipeline, while each derived handler (RRC, NGAP, NAS) provides the protocol-specific parsing. Since the handler type is known at compile time (each protocol has its own processing function), CRTP eliminates virtual dispatch on the per-packet path.
