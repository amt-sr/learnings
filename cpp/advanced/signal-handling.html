Signal Handling

Signals are software interrupts delivered to a process by the kernel or other processes. They provide an asynchronous notification mechanism for events like user interrupts (Ctrl+C), segmentation faults, child process termination, and timer expirations. Proper signal handling is critical for building robust server processes that can shut down gracefully.

Prerequisites: Socket Programming, Shared Memory IPC, std::thread, Exception Handling.

sigaction — The Right Way to Handle Signals

Prefer sigaction() over the legacy signal() function. sigaction provides reliable behavior across platforms and more control over signal handling semantics:

#include <iostream>
#include <csignal>
#include <cstring>
#include <atomic>
#include <unistd.h>

// Global flag for graceful shutdown
std::atomic<bool> g_running{true};

// Signal handler — must be async-signal-safe
void signal_handler(int signum) {
    switch (signum) {
        case SIGINT:
            std::cout << "\nReceived SIGINT, shutting down...\n";
            g_running.store(false);
            break;
        case SIGTERM:
            std::cout << "Received SIGTERM, shutting down...\n";
            g_running.store(false);
            break;
    }
}

// Register signal handlers using sigaction
void setup_signal_handlers() {
    struct sigaction sa{};
    sa.sa_handler = signal_handler;
    sigemptyset(&sa.sa_mask);

    // SA_RESTART: automatically restart interrupted system calls
    sa.sa_flags = SA_RESTART;

    if (sigaction(SIGINT, &sa, nullptr) != 0) {
        std::cerr << "sigaction(SIGINT) failed: " << strerror(errno) << "\n";
    }
    if (sigaction(SIGTERM, &sa, nullptr) != 0) {
        std::cerr << "sigaction(SIGTERM) failed: " << strerror(errno) << "\n";
    }
}

int main() {
    setup_signal_handlers();

    std::cout << "Server running (PID=" << getpid() << ")\n";
    std::cout << "Press Ctrl+C to stop\n";

    while (g_running.load()) {
        // Main event loop
        // (socket operations, processing, etc.)
        sleep(1);
    }

    std::cout << "Cleaning up resources...\n";
    // Close sockets, cleanup shared memory, flush buffers
    std::cout << "Shutdown complete\n";
    return 0;
}

Ignoring SIGPIPE

SIGPIPE is sent when a process writes to a socket whose remote end has closed the connection. The default action (terminate) is almost never what you want for a server. Instead, ignore it and handle the EPIPE errno from write()/send():

#include <csignal>

// Ignore SIGPIPE globally — check errno instead
void ignore_sigpipe() {
    signal(SIGPIPE, SIG_IGN);
}

// When writing to a socket:
ssize_t n = send(sockfd, data, size, 0);
if (n < 0 && errno == EPIPE) {
    // Remote end disconnected — handle gracefully
    std::cout << "Connection closed by peer\n";
}

Signal-Safe Functions

Only async-signal-safe functions may be called from a signal handler. Calling non-safe functions (like malloc, std::cout, most libc functions) causes undefined behavior:

Safe (async-signal-safe)NOT Safe
_exit()exit()
write() (to FDs)printf(), std::cout
sigaction(), kill()malloc(), new
sigprocmask()pthread_mutex_lock()
atomic_flag operationsstd::cout, std::cerr

Critical rule: Signal handlers are async-signal-safe context. Only write to variables of type volatile std::sig_atomic_t or std::atomic with specific guarantees. Use the write() system call for output in handlers, not std::cout. Set a flag in the handler, handle the signal in the main loop.

Signals in Multithreaded Programs

#include <iostream>
#include <csignal>
#include <pthread.h>

std::atomic<bool> g_shutdown{false};

void handler(int signum) {
    g_shutdown.store(true);
}

void setup_signals() {
    struct sigaction sa{};
    sa.sa_handler = handler;
    sigemptyset(&sa.sa_mask);
    sa.sa_flags = SA_RESTART;
    sigaction(SIGTERM, &sa, nullptr);
}

// Signal delivery in multi-threaded program:
// - Kernel picks ANY thread in the process to deliver the signal
// - You cannot predict which thread receives it
// - Best practice: block all signals in worker threads,
//   handle signals only in the main thread
void block_signals_in_workers() {
    sigset_t set;
    sigfillset(&set);
    // Don't block SIGKILL/SIGSTOP (cannot be blocked)
    sigdelset(&set, SIGKILL);
    sigdelset(&set, SIGSTOP);
    pthread_sigmask(SIG_BLOCK, &set, nullptr);
}

int main() {
    setup_signals();
    block_signals_in_workers();

    // Spawn worker threads...
    // Worker threads will NOT receive signals

    // Main thread handles signals via the flag
    while (!g_shutdown.load()) {
        // Event loop
    }
}

Best practice: Block signals in all worker threads using pthread_sigmask. Designate a single thread (usually the main thread) to handle signals. Use std::atomic<bool> flags for communication between the signal handler and main loop. Never do complex work inside a signal handler.

Deadlock risk: Never lock a mutex inside a signal handler. If the main program holds the same mutex when the signal arrives, the process deadlocks. This is one of the most common signal-related bugs in server software.
