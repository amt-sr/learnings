Static Code Analysis Overview

Static analysis is the process of examining source code without executing it to find bugs, security vulnerabilities, and code quality issues. Unlike dynamic testing, static analysis can detect problems across all possible execution paths, including edge cases that are difficult to reach with tests alone.

Why it matters: Studies consistently show that finding and fixing bugs early in development is orders of magnitude cheaper than fixing them in production. Static analysis catches entire classes of errors (memory leaks, null pointer dereferences, buffer overflows) at compile time, before the code ever runs.

Types of Static Analysis

TechniqueDescriptionStrengthsLimitations
Pattern MatchingMatches known bug patterns (regex, AST patterns) against source codeFast, low false positive rate for known patternsCannot discover novel bugs
Data Flow AnalysisTracks how data propagates through the program (taint analysis, reaching definitions)Finds use of uninitialized variables, tainted data flowsMay not model all possible paths
Symbolic ExecutionExplores all feasible paths by substituting symbolic values for inputsDeep bug detection, proves absence of certain bug classesPath explosion, scalability issues
Abstract InterpretationModels program behavior using abstract domains (intervals, congruences)Sound analysis with guaranteed coverageMay report false positives from over-approximation
Type-Based AnalysisLeverages the type system and type inference to find inconsistenciesFast, precise, leverages compiler infrastructureLimited to type-related errors

False Positives and False Negatives

Every static analyzer must balance two competing goals:

  • False positive -- The tool reports a bug that does not actually exist. High false positive rates cause alert fatigue and developers start ignoring warnings.
  • False negative -- The tool misses a real bug. Reducing false negatives requires deeper analysis but often increases false positives.

Practical advice: A tool with 10% false positives is usable. A tool with 50% false positives will be disabled by the team. Prioritize configuring tools to minimize noise while catching the most critical bugs. See Code Scanning Best Practices for strategies on managing alerts.

Integrating Static Analysis into CI/CD

Static analysis is most effective when it runs automatically on every commit or pull request. A typical CI/CD pipeline includes:

# Example GitHub Actions workflow for static analysis
name: Static Analysis
on: [push, pull_request]

jobs:
  analyze:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Run cppcheck
        run: |
          cppcheck --enable=all --error-exitcode=1 \
            --suppress=missingInclude \
            --xml 2> cppcheck-report.xml \
            src/

      - name: Run clang-tidy
        run: |
          cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON
          run-clang-tidy -p build -j$(nproc) src/

      - name: Run sanitizers in tests
        run: |
          cmake -B build-san -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined"
          cmake --build build-san
          ctest --test-dir build-san --output-on-failure

Best practice: Use multiple complementary tools. cppcheck catches what clang-tidy misses and vice versa. Add sanitizers for runtime coverage. See Code Scanning Best Practices for a complete pipeline setup.

Bugs That Static Analysis Catches

Static analyzers can detect entire categories of bugs that compilers miss. Here are common C++ bugs and what the tools report:

// Bug 1: Null pointer dereference (caught by clang-tidy, Clang SA)
void processUser(User* user) {
    int age = user->getAge();  // Warning: potential null dereference
    // Fix: check for nullptr first
    if (!user) { return; }
    int name_len = user->getName().length();
}

// Bug 2: Memory leak (caught by cppcheck, LeakSanitizer)
void processData() {
    auto* data = new int[1024];  // cppcheck: "Memory leak: data"
    if (someCondition()) {
        throw std::runtime_error("failed");  // leaks data
    }
    delete[] data;  // never reached on exception path
    // Fix: use std::vector<int> or std::unique_ptr<int[]>
}

// Bug 3: Uninitialized variable (caught by cppcheck, Clang SA)
void computeAverage() {
    int sum;                    // Warning: variable is uninitialized
    int count = 5;
    for (int i = 0; i < count; ++i) {
        sum += i;               // UB: using uninitialized sum
    }
    double avg = sum / count;   // result is undefined
    // Fix: int sum = 0;
}

// Bug 4: Use-after-free (caught by ASan at runtime, Clang SA statically)
void useAfterFree(std::vector<int>& v) {
    int* ptr = &v[0];
    v.push_back(42);           // may reallocate, invalidating ptr
    std::cout << *ptr << "\n";  // Warning: potential use-after-free
}

Compilers vs static analyzers: GCC and Clang catch syntax errors and basic type mismatches, but they do not check data flow for null pointers, uninitialized variables, or memory leaks. Static analysis tools like cppcheck and clang-tidy bridge this gap by performing deeper analysis.
