C++20 Concepts

Concepts are named constraints on template parameters. They improve template error messages, enable overload resolution based on type properties, and make generic code self-documenting. Concepts are one of the four major language features added in C++20, alongside Ranges, Coroutines, and Modules.

Prerequisites: Familiarity with templates, function overloading, and SFINAE (SFINAE is the predecessor to concepts).

Why Concepts?

Before C++20, template constraints relied on SFINAE with std::enable_if, void_t, and convoluted decltype expressions. This produced unreadable code and nearly incomprehensible error messages when constraints were violated.

// Before concepts (C++17 SFINAE) — hard to read, bad errors
template<typename T, typename = std::enable_if_t<std::is_integral_v<T>>>
void process(T value);

// After concepts (C++20) — clear intent, great errors
template<std::integral T>
void process(T value);

Defining Concepts

A concept is a constexpr bool value that evaluates whether a type satisfies a set of requirements. Use the concept keyword to define one.

#include <concepts>
#include <type_traits>

// A concept requiring the type to have a .size() member
template<typename T>
concept HasSize = requires(T t) {
    { t.size() } -> std::convertible_to<std::size_t>;
};

// A concept requiring arithmetic operations
template<typename T>
concept Addable = requires(T a, T b) {
    { a + b } -> std::same_as<T>;
};

// A concept with multiple requirements
template<typename T>
concept SortableContainer = requires(T container) {
    requires std::ranges::range<T>;
    requires std::totally_ordered<typename T::value_type>;
    { container.begin() } -> std::input_or_output_iterator;
    { container.end() } -> std::input_or_output_iterator;
};

requires Clause Syntax

There are four ways to use concepts for constraining templates:

// 1. Constrained template parameter (terse syntax)
template<std::integral T>
T square(T x) { return x * x; }

// 2. requires clause after template parameter list
template<typename T>
    requires std::integral<T>
T square(T x) { return x * x; }

// 3. Trailing requires clause
template<typename T>
T square(T x) requires std::integral<T> { return x * x; }

// 4. Constrained auto (non-template, abbreviated function template)
void process(std::integral auto x) { /* ... */ }

requires Expression

The requires expression checks whether a series of statements are valid for a given type:

template<typename T>
concept Streamable = requires(T t, std::ostream& os) {
    // Valid expression (type must support << operator)
    { os << t } -> std::same_as<std::ostream&>;
};

template<typename T>
concept Container = requires(T t, typename T::value_type v) {
    // Type requirements
    typename T::value_type;
    typename T::iterator;
    typename T::const_iterator;

    // Expression requirements
    { t.begin() } -> std::same_as<typename T::iterator>;
    { t.end() } -> std::same_as<typename T::iterator>;
    { t.size() } -> std::convertible_to<std::size_t>;
    { t.empty() } -> std::convertible_to<bool>;

    // Nested requirements
    requires std::input_iterator<typename T::iterator>;
};

Custom Concepts in Practice

Network Protocol Concept

#include <string>
#include <string_view>
#include <array>
#include <concepts>
#include <cstdint>

// Concept for any type that can be serialized to/from bytes
template<typename T>
concept Serializable = requires(T t, std::uint8_t* buf, std::size_t buf_size) {
    { t.serialize(buf, buf_size) } -> std::same_as<std::size_t>;
    { T::deserialize(buf, buf_size) } -> std::same_as<T>;
    { T::serialized_size() } -> std::convertible_to<std::size_t>;
};

// Concept for a network message handler
template<typename Handler, typename MsgType>
concept MessageHandler = requires(Handler h, MsgType msg) {
    { h.handle(msg) } -> std::same_as<void>;
    { h.on_error(std::string_view{}) } -> std::same_as<void>;
};

// A serializable message
struct RrcSetupRequest {
    std::array<std::uint8_t, 6> gnb_id{};
    std::uint16_t tac = 0;
    std::uint8_t cause = 0;

    static constexpr std::size_t serialized_size() { return 10; }

    std::size_t serialize(std::uint8_t* buf, std::size_t /*buf_size*/) const {
        std::copy(gnb_id.begin(), gnb_id.end(), buf);
        buf[6] = static_cast<std::uint8_t>(tac >> 8);
        buf[7] = static_cast<std::uint8_t>(tac & 0xFF);
        buf[8] = cause;
        return serialized_size();
    }

    static RrcSetupRequest deserialize(const std::uint8_t* buf, std::size_t /*size*/) {
        RrcSetupRequest req{};
        std::copy(buf, buf + 6, req.gnb_id.begin());
        req.tac = (static_cast<std::uint16_t>(buf[6]) << 8) | buf[7];
        req.cause = buf[8];
        return req;
    }
};

static_assert(Serializable<RrcSetupRequest>, "Must be serializable");

Concept-Based Overloading

Concepts participate in overload resolution. The compiler selects the most constrained overload, enabling clean dispatch based on type properties:

#include <concepts>
#include <iostream>
#include <string>
#include <vector>
#include <map>

// Most constrained: integral types
template<std::integral T>
std::string classify(T value) {
    return "integral (" + std::to_string(value) + ")";
}

// Less constrained: any arithmetic type (includes floating-point)
template<std::floating_point T>
std::string classify(T value) {
    return "floating-point (" + std::to_string(value) + ")";
}

// For any container of arithmetic elements
template<std::ranges::range R>
    requires std::integral<typename std::ranges::range_value_t<R>>
std::string classify(const R& range) {
    return "container of integrals (size=" + std::to_string(range.size()) + ")";
}

int main() {
    std::cout << classify(42) << "\n";          // integral
    std::cout << classify(3.14) << "\n";         // floating-point
    std::cout << classify(std::vector<int>{1,2,3}) << "\n";  // container of integrals
}

Partial ordering: When multiple concepts could match, the compiler uses partial ordering to pick the most specific one. If neither is more specific, the call is ambiguous. Use requires clauses with boolean combinations to resolve ambiguity.

Combining Concepts with Boolean Logic

#include <concepts>

// Any numeric type that is NOT a bool
template<typename T>
concept NumberButNotBool = std::arithmetic<T> && !std::same_as<T, bool>;

// Either integral or a pointer type
template<typename T>
concept IntegralOrPointer = std::integral<T> || std::is_pointer_v<T>;

// Combines multiple constraints: must be a range of copyable elements
template<typename R>
concept CopyableRange = std::ranges::range<R>
    && std::copyable<std::ranges::range_value_t<R>>;

// Use in function
template<CopyableRange R>
R duplicate(const R& source) {
    R copy = source;  // guaranteed valid because element type is copyable
    return copy;
}

Best practice: Prefer concepts over SFINAE in all new code. Concepts are more readable, produce better error messages, and integrate with Ranges and the standard library. See SFINAE Advanced for the legacy patterns you may still encounter.

Telecom Relevance

In telecom stack development (5G RAN, Core Network), concepts enforce protocol correctness at compile time. A message processor can require that all message types implement a serialize interface. A scheduler can require that tasks model std::invocable. This eliminates an entire class of runtime errors where the wrong type is passed to a protocol handler.
