Code Scanning Best Practices

Setting up an effective code scanning pipeline requires more than just running a tool. It requires careful configuration, team buy-in, and a strategy for managing alerts over time. This guide covers best practices for integrating static analysis into your C++ development workflow.

Setting Up the Pipeline

A robust code scanning pipeline runs at multiple stages of development, catching issues as early as possible:

StageToolsSpeedScope
Pre-commit (local)clang-tidy (incremental)< 5 secondsChanged files only

Pre-Commit Hooks

# .githooks/pre-commit (make executable: chmod +x .githooks/pre-commit)
#!/bin/bash
set -euo pipefail

CHANGED_FILES=$(git diff --cached --name-only --diff-filter=ACM \
    | grep -E '\.(cpp|cc|cxx|h|hpp|hxx)$' || true)

if [ -z "$CHANGED_FILES" ]; then
    exit 0
fi

echo "Running static analysis on changed files..."

# clang-tidy on changed files
for file in $CHANGED_FILES; do
    if [ -f "$file" ]; then
        clang-tidy --config-file=.clang-tidy "$file" 2>&1 || {
            echo "ERROR: clang-tidy found issues in $file"
            echo "Fix with: clang-tidy --fix $file"
            exit 1
        }
    fi
done

# cppcheck on changed files
cppcheck --enable=error,warning,performance \
    --std=c++17 \
    --suppress=missingInclude \
    $CHANGED_FILES || {
        echo "ERROR: cppcheck found issues"
        exit 1
    }

echo "Static analysis passed!"

# Configure git to use custom hooks
git config core.hooksPath .githooks

CI/CD Integration

# .github/workflows/code-quality.yml
name: Code Quality

on:
  pull_request:
  push:
    branches: [main]

jobs:
  static-analysis:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Configure
        run: cmake -B build -DCMAKE_EXPORT_COMPILE_COMMANDS=ON

      - name: cppcheck
        run: |
          cppcheck --enable=all --error-exitcode=1 \
            --suppress=missingInclude \
            --std=c++17 --xml 2> cppcheck.xml src/

      - name: clang-tidy
        run: run-clang-tidy -p build -j$(nproc) src/

  sanitizers:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build with ASan
        run: |
          cmake -B build-asan \
            -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined -fno-omit-frame-pointer"
          cmake --build build-asan

      - name: Run tests with sanitizers
        run: ctest --test-dir build-asan --output-on-failure

Suppressing False Positives

When a tool reports a false positive, suppress it explicitly with a reason. Never suppress blindly:

// Inline suppression with reason (NOLINT for clang-tidy)
int legacy_offset = 42;  // NOLINT(readability-magic-numbers)
                         // Legacy offset constant, do not change

// cppcheck inline suppression
void legacy_handler(char* buf) {
    // cppcheck-suppress nullPointer
    *buf = '\0';  // Caller guarantees non-null (legacy API contract)
}

// .clang-tidy suppression
# Suppress for specific file
Checks: '*, -modernize-use-trailing-return-type'

Rule: Every suppression must include a comment explaining why it is a false positive and whether it should be revisited. Use TODO(safety): tags for temporary suppressions that need fixing.

Combining Tools Effectively

# Recommended tool combination for C++ projects:

# 1. Fast, always-on (pre-commit + CI)
cppcheck --enable=error,warning,performance --std=c++17 src/
clang-tidy --checks='bugprone-*,modernize-*,performance-*' src/

# 2. Deep analysis (nightly CI)
scan-build cmake --build build     # Clang Static Analyzer

# 3. Runtime verification (test suite)
cmake -DCMAKE_CXX_FLAGS="-fsanitize=address,undefined" \
      -DCMAKE_EXE_LINKER_FLAGS="-fsanitize=address" -B build-san
ctest --test-dir build-san

# 4. Code style enforcement
clang-format --dry-run --Werror src/**/*.cpp src/**/*.h

Key principles: (1) Never introduce new violations. (2) Fix existing violations incrementally -- do not try to fix everything at once. (3) Automate everything -- if it is not in CI, it will not be enforced. (4) Make alerts actionable -- developers should understand each finding and know how to fix it.
