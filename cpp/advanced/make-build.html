Make & Build Process

Make is a build automation tool that reads Makefile descriptions to determine which parts of a program need to be recompiled. It uses file timestamps to rebuild only what has changed, making incremental builds fast. While CMake generates Makefiles as its backend, understanding Make directly is essential for debugging build issues and for smaller projects.

Makefile Anatomy

A Makefile consists of rules. Each rule has a target, prerequisites (dependencies), and a recipe (commands to run):

# target: prerequisites
#     recipe (must be indented with TAB, not spaces)

# Simple rule
hello: main.cpp
    g++ -std=c++17 -Wall -o hello main.cpp

Makefile for a C++ Project

# Makefile for a multi-file C++ project
# ================================

# Variables
CXX      = g++
CXXFLAGS = -std=c++17 -Wall -Wextra -Wpedantic -g -O2
LDFLAGS  = -lpthread

# Directories
SRC_DIR  = src
BUILD_DIR = build
TARGET   = ran_simulator

# Source files and objects
SRCS     = $(wildcard $(SRC_DIR)/*.cpp)
OBJS     = $(patsubst $(SRC_DIR)/%.cpp,$(BUILD_DIR)/%.o,$(SRCS))
DEPS     = $(OBJS:.o=.d)

# Default target
all: $(BUILD_DIR) $(TARGET)

# Create build directory
$(BUILD_DIR):
    mkdir -p $(BUILD_DIR)

# Link executable from object files
$(TARGET): $(OBJS)
    $(CXX) $(CXXFLAGS) -o $@ $^ $(LDFLAGS)
    @echo "Build complete: $(TARGET)"

# Pattern rule: compile .cpp to .o
$(BUILD_DIR)/%.o: $(SRC_DIR)/%.cpp | $(BUILD_DIR)
    $(CXX) $(CXXFLAGS) -MMD -MP -c $< -o $@
    @echo "Compiled: $<"

# Include auto-generated dependency files
-include $(DEPS)

# Phony targets (always run, not file-based)
.PHONY: all clean debug test memcheck tidy

clean:
    rm -rf $(BUILD_DIR) $(TARGET)
    @echo "Clean complete"

debug: CXXFLAGS += -O0 -DDEBUG -fsanitize=address,undefined
debug: clean all

test: all
    ./$(TARGET) --run-tests

memcheck: all
    valgrind --leak-check=full --error-exitcode=1 ./$(TARGET) --run-tests

tidy:
    clang-tidy $(SRC_DIR)/*.cpp -- -std=c++17

Key Concepts

Variables

# Simple assignment (= lazy evaluation, expanded on use)
CC = gcc

# Immediate assignment (:= expanded at definition time)
OBJS := $(SRC:.cpp=.o)

# Append (+= adds to existing variable)
CXXFLAGS += -Wall

# Conditional assignment (?= only set if not already defined)
CC ?= gcc  # Use existing CC, or default to gcc

# Automatic variables (inside recipes)
# $@ = target name
# $< = first prerequisite
# $^ = all prerequisites
# $* = stem of pattern rule match

Pattern Rules

# Pattern rule: any .o file depends on matching .cpp
%.o: %.cpp %.h
    $(CXX) $(CXXFLAGS) -c $< -o $@

# More specific pattern rule (overrides generic)
debug/%.o: %.cpp
    $(CXX) $(CXXFLAGS) -DDEBUG -c $< -o $@

Phony Targets

# .PHONY tells Make these are not real files
# Without .PHONY, "make clean" would do nothing if a file named "clean" exists
.PHONY: all clean install test format

clean:
    rm -f *.o $(TARGET)

test: $(TARGET)
    ./$(TARGET) --test

format:
    clang-format -i src/*.cpp include/*.h

Automatic Dependency Tracking

# Generate header dependency files with -MMD -MP flags
# -MMD generates .d files listing header dependencies
# -MP adds phony targets for headers to avoid errors when headers are deleted

$(BUILD_DIR)/%.d: $(SRC_DIR)/%.cpp
    @$(CXX) $(CXXFLAGS) -MM -MG -MF $@ $<

# Include dependency files (silently ignore missing files with -include)
-include $(DEPS)

Make vs CMake Comparison

FeatureMakeCMake
Cross-platformLimited (GNU Make mostly)Full (generates Make, Ninja, VS, Xcode)
Learning curveSimpler for small projectsSteeper initial learning
Dependency discoveryManual or -MMDAutomatic
External librariesManual LDFLAGS/CXXFLAGSfind_package()
IDE integrationBasicFull (VS, Xcode, CLion)
Large projectsComplex, error-proneScales well
SpeedFast (direct build)Fast with Ninja backend
Build type managementManual flagsCMAKE_BUILD_TYPE

Recommendation: Use CMake for projects of any significant size. Use Make directly for quick scripts, one-off builds, or when CMake is overkill. Understanding Make is still valuable since CMake generates Makefiles by default, and you may need to read or debug them with GDB.
