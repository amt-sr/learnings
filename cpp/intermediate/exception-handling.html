Exception Handling

Exceptions provide a structured mechanism for handling runtime errors. When something goes wrong (file not found, out of memory, invalid argument), a function can throw an exception instead of returning an error code. The exception propagates up the call stack until it is caught by a matching handler. This separates error-handling code from normal flow.

try / catch / throw Basics

#include <iostream>
#include <stdexcept>
#include <vector>

double safe_divide(double a, double b) {
    if (b == 0.0) {
        throw std::invalid_argument("Division by zero");
    }
    return a / b;
}

int element_at(const std::vector<int>& v, size_t idx) {
    if (idx >= v.size()) {
        throw std::out_of_range("Index " + std::to_string(idx)
            + " out of range (size: " + std::to_string(v.size()) + ")");
    }
    return v[idx];
}

void exception_basics() {
    try {
        std::cout << safe_divide(10.0, 3.0) << "\n";  // 3.33333
        std::cout << safe_divide(10.0, 0.0) << "\n";  // throws
    }
    catch (const std::invalid_argument& e) {
        std::cerr << "Invalid argument: " << e.what() << "\n";
    }
    catch (const std::exception& e) {
        std::cerr << "Standard exception: " << e.what() << "\n";
    }
    catch (...) {
        std::cerr << "Unknown exception\n";
    }
}

Catch by Reference

Always catch by reference: catch (std::exception& e) -- not by value (slicing) and not by pointer (memory management). Catching by reference avoids object slicing when derived exceptions are thrown, preserves polymorphic behavior, and avoids unnecessary copies.

// BAD: catch by value — derived exceptions are sliced
catch (std::exception e) { ... }

// BAD: catch by pointer — you must manage lifetime
catch (std::exception* e) { ... }

// GOOD: catch by reference
catch (const std::exception& e) { ... }

// GOOD: catch by const reference (preferred)
catch (const std::runtime_error& e) { ... }

Custom Exceptions

#include <stdexcept>
#include <string>
#include <iostream>

class NetworkError : public std::runtime_error {
    int error_code_;
public:
    explicit NetworkError(const std::string& msg, int code)
        : std::runtime_error(msg), error_code_(code) {}

    int error_code() const noexcept { return error_code_; }
};

class ValidationError : public std::logic_error {
    std::string field_name_;
public:
    explicit ValidationError(const std::string& msg, const std::string& field)
        : std::logic_error(msg), field_name_(field) {}

    const std::string& field() const noexcept { return field_name_; }
};

void custom_exception_demo() {
    try {
        throw NetworkError("Connection refused", 10061);
    }
    catch (const NetworkError& e) {
        std::cerr << "Network error " << e.error_code()
                  << ": " << e.what() << "\n";
    }
    catch (const std::exception& e) {
        std::cerr << "Error: " << e.what() << "\n";
    }
}

Stack Unwinding

When an exception is thrown and not caught in the current function, the function exits. All local objects in that function are destroyed (their destructors called) in reverse order of construction. This process continues up the call stack until a matching catch is found.

#include <iostream>

class ResourceGuard {
    std::string name_;
public:
    explicit ResourceGuard(const std::string& name) : name_(name) {
        std::cout << "Acquire " << name_ << "\n";
    }
    ~ResourceGuard() {
        std::cout << "Release " << name_ << "\n";
    }
};

void inner_function() {
    ResourceGuard r3("inner");
    throw std::runtime_error("Something went wrong");
    // r3 is destroyed during stack unwinding
}

void outer_function() {
    ResourceGuard r1("outer-1");
    ResourceGuard r2("outer-2");
    inner_function();  // throws
    // r2 destroyed first, then r1
}

void stack_unwind_demo() {
    try {
        outer_function();
    } catch (const std::exception& e) {
        std::cout << "Caught: " << e.what() << "\n";
    }
}
// Output:
//   Acquire outer-1
//   Acquire outer-2
//   Acquire inner
//   Release inner
//   Release outer-2
//   Release outer-1
//   Caught: Something went wrong

RAII + exceptions: This is why RAII is so important. If resources are managed by RAII objects (smart pointers, file handles, locks), stack unwinding automatically releases them. No explicit cleanup code is needed in catch blocks.
