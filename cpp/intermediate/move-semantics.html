Move Semantics

Move semantics is one of the most important features introduced in C++11. It allows resources (heap memory, file handles, network sockets) to be transferred from one object to another instead of being copied. This eliminates unnecessary deep copies and dramatically improves performance for objects that manage resources.

Lvalues vs Rvalues

Understanding move semantics requires understanding value categories:

CategoryDefinitionExamples
LvalueHas a name, persists beyond a single expressionint x = 5; — x is an lvalue
RvalueTemporary, expires at end of expression42, x + y, std::string("hi")
int x = 10;           // x is lvalue, 10 is rvalue
int& lr = x;          // lvalue reference — binds to lvalues
int&& rr = 10;         // rvalue reference — binds to rvalues (C++11)
// int&& rr2 = x;     // ERROR: cannot bind rvalue ref to lvalue
int&& rr3 = x + 10;  // OK: x + 10 is an rvalue

std::move

std::move does not move anything. It is an unconditional cast to an rvalue reference. It tells the compiler: "I am done with this object; you may steal its resources."

#include <utility>
#include <string>
#include <iostream>

void move_basics() {
    std::string a = "Hello, World!";
    std::string b = std::move(a);  // a's internal buffer is transferred to b

    std::cout << "a: '" << a << "' (likely empty or in valid but unspecified state)\n";
    std::cout << "b: '" << b << "'\n";

    // After move, 'a' is in a valid but unspecified state.
    // It can be assigned to or destroyed, but don't assume its value.
    a = "New value";  // OK: assign to moved-from object
    std::cout << "a after reassignment: '" << a << "'\n";
}

Common misconception: std::move does not guarantee a move will occur. It merely casts to rvalue reference. Whether a move actually happens depends on whether a move constructor/move assignment operator exists and is selected by overload resolution.

Move Constructor and Move Assignment

Custom classes managing resources should define move operations. The move constructor steals resources from the source; the move assignment operator does the same after cleaning up existing resources.

#include <cstring>
#include <utility>
#include <iostream>

class Buffer {
    char* data_;
    size_t size_;
public:
    // Constructor
    explicit Buffer(size_t size) : size_(size), data_(new char[size]) {
        std::cout << "Allocated " << size_ << " bytes\n";
    }

    // Destructor
    ~Buffer() {
        delete[] data_;
        std::cout << "Freed buffer (" << size_ << " bytes)\n";
    }

    // Move constructor — steal resources from other
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;  // leave source in valid state
        other.size_ = 0;
        std::cout << "Move-constructed (" << size_ << " bytes)\n";
    }

    // Move assignment operator
    Buffer& operator=(Buffer&& other) noexcept {
        if (this != &other) {
            delete[] data_;          // free our existing resource
            data_ = other.data_;      // steal from source
            size_ = other.size_;
            other.data_ = nullptr;   // leave source in valid state
            other.size_ = 0;
            std::cout << "Move-assigned (" << size_ << " bytes)\n";
        }
        return *this;
    }

    // Delete copy operations (this class exclusively owns its buffer)
    Buffer(const Buffer&) = delete;
    Buffer& operator=(const Buffer&) = delete;

    size_t size() const { return size_; }
};

void move_constructor_demo() {
    Buffer a(1024);             // "Allocated 1024 bytes"
    Buffer b = std::move(a);    // "Move-constructed (1024 bytes)"
    // a's data_ is now nullptr — safe to destroy
    std::cout << "b.size() = " << b.size() << "\n";  // 1024
}

Rule of Five: If a class defines any of: destructor, copy constructor, copy assignment, move constructor, or move assignment — it should usually define all five. Alternatively, if copying makes no sense, delete the copy operations and default the move operations (see RAII).

Move Semantics with std::vector

#include <vector>
#include <string>
#include <iostream>

void vector_move_demo() {
    std::vector<std::string> source = {"alpha", "beta", "gamma", "delta"};
    std::cout << "source size: " << source.size() << "\n";  // 4

    // Move the entire vector — O(1), just swaps internal pointers
    std::vector<std::string> dest = std::move(source);
    std::cout << "source size after move: " << source.size() << "\n";  // 0
    std::cout << "dest size: " << dest.size() << "\n";  // 4

    // Insert via move — avoids copying the string
    std::string temp = "epsilon";
    dest.push_back(std::move(temp));  // temp is moved into the vector
}

Perfect Forwarding

Perfect forwarding allows a function template to pass its arguments to another function with their original value categories (lvalue/rvalue) and const qualifications preserved. It uses std::forward.

#include <utility>
#include <iostream>
#include <string>

class Logger {
    std::string name_;
public:
    explicit Logger(std::string name) : name_(std::move(name)) {
        std::cout << "Logger '" << name_ << "' created\n";
    }
};

// Factory using perfect forwarding
template<typename T>
Logger make_logger(T&& arg) {
    // std::forward preserves whether arg was an lvalue or rvalue
    return Logger(std::forward<T>(arg));
}

void perfect_forwarding_demo() {
    std::string name = "main";

    // Passing an lvalue — forwarded as lvalue, Logger copies it
    auto l1 = make_logger(name);
    std::cout << "name after lvalue pass: '" << name << "'\n";  // still "main"

    // Passing an rvalue — forwarded as rvalue, Logger moves it
    auto l2 = make_logger(std::string("temp"));
}

How std::forward Works

// When T is deduced from an lvalue:
//   T = std::string&  (lvalue reference)
//   T&& collapses to std::string& (lvalue reference)
//   std::forward<T> returns an lvalue reference

// When T is deduced from an rvalue:
//   T = std::string    (plain type)
//   T&& = std::string&& (rvalue reference)
//   std::forward<T> returns an rvalue reference

When to Use Move

Move candidates:

  • Returning large local objects by value (or rely on copy elision)
  • Passing objects to functions that will take ownership (e.g., unique_ptr)
  • Swapping objects (always use std::swap which uses moves)
  • Inserting into containers: vec.push_back(std::move(obj))

Do not move:

  • Objects you still need to use afterward
  • const objects — std::move on a const object produces a const T&&, which will fall back to copy
  • Small types like int, float — copying is cheaper than moving
