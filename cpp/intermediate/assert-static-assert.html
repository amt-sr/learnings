Assert / Static Assert

Assertions are sanity checks that verify conditions that should always be true. assert() checks conditions at runtime (debug builds only), while static_assert checks at compile time. Both catch bugs early -- static_assert catches them before the program even runs, and assert catches them during development.

Runtime assert

assert(expr) from <cassert> checks expr at runtime. If false, it prints an error message and calls abort(). Assertions are disabled in release builds by defining NDEBUG.

#include <cassert>
#include <vector>
#include <iostream>

int safe_divide(int a, int b) {
    assert(b != 0 && "Division by zero in safe_divide");
    return a / b;
}

class RingBuffer {
    std::vector<int> data_;
    size_t head_ = 0;
    size_t count_ = 0;
public:
    explicit RingBuffer(size_t capacity) : data_(capacity) {}

    void push(int val) {
        assert(count_ < data_.size() && "Buffer overflow");
        data_[head_] = val;
        head_ = (head_ + 1) % data_.size();
        ++count_;
    }

    int pop() {
        assert(count_ > 0 && "Buffer underflow");
        --count_;
        size_t idx = (head_ + data_.size() - 1) % data_.size();
        return data_[idx];
    }

    size_t size() const { return count_; }
};

void assert_demo() {
    std::cout << safe_divide(10, 2) << "\n";  // 5
    // safe_divide(10, 0);  // ASSERTION FAILURE in debug, undefined in release
}

NDEBUG and Release Builds

// When compiling with -DNDEBUG (typical for release):
//   assert(expr) is completely removed by the preprocessor
//   No runtime cost, no code in the binary

// Debug build:
//   g++ -g -O0 program.cpp          (asserts active)
// Release build:
//   g++ -O2 -DNDEBUG program.cpp    (asserts disabled)

// To keep asserts even in release builds, use:
if (condition) {
    // This code is NOT removed in release
    // Use for truly critical checks that must always run
}

Do not use assert for input validation: Assertions check for programmer bugs (internal invariants), not for invalid user input. User input should produce graceful error handling (exceptions, error codes), not program termination. If the check should always run in production, it is not an assert.

static_assert (Compile-Time)

static_assert verifies conditions at compile time. If the condition is false, compilation fails with a custom error message. This is the most powerful form of assertion because it catches bugs before the program runs.

#include <type_traits>
#include <cstdint>
#include <string_view>

// C++17: static_assert with message is optional (message defaults to condition)
template<typename T>
class NumericBuffer {
    static_assert(std::is_arithmetic_v<T>,
        "NumericBuffer requires an arithmetic type");
    static_assert(sizeof(T) <= 64,
        "NumericBuffer element type must be 64 bits or less");
    T* data_;
    size_t size_;
public:
    explicit NumericBuffer(size_t n) : data_(new T[n]), size_(n) {}
    ~NumericBuffer() { delete[] data_; }
};

// Platform-specific compile-time checks
static_assert(sizeof(void*) == 8, "This code requires a 64-bit platform");
static_assert(sizeof(int) == 4, "int must be 32 bits");
static_assert(CHAR_BIT == 8, "char must be 8 bits");

// Check struct layout
struct PacketHeader {
    uint32_t type;
    uint16_t length;
    uint16_t checksum;
};

static_assert(sizeof(PacketHeader) == 8, "PacketHeader must be 8 bytes");
static_assert(offsetof(PacketHeader, checksum) == 6,
    "checksum must be at offset 6");

static_assert with Type Traits

#include <type_traits>
#include <iterator>

// Validate template constraints at compile time
template<typename Container>
auto first_element(const Container& c)
    -> decltype(*c.begin())
{
    static_assert(std::is_same_v<
        typename std::iterator_traits<decltype(c.begin())>::iterator_category,
        std::random_access_iterator_tag>,
        "first_element requires a random-access container");

    return c.front();
}

// Ensure a class meets size constraints
template<typename T>
constexpr bool is_small_object_v = sizeof(T) <= 64 && alignof(T) <= 16;

struct BigData {
    char buffer[256];
};

static_assert(is_small_object_v<int>);
// static_assert(is_small_object_v<BigData>);  // ERROR: too large

When to Use Which

Use static_assert for:

  • Template type constraints (e.g., "T must be integral")
  • Struct/class size and layout verification
  • Platform assumptions (pointer size, endianness)
  • Any condition that can be evaluated at compile time

Use assert for:

  • Function preconditions that indicate programmer bugs
  • Internal invariants (e.g., "this list should never be empty here")
  • Postconditions ("the result should always be positive")
  • Loop invariants in complex algorithms

Assertions complement exception handling for different error categories. static_assert works naturally with type traits and constexpr for compile-time validation. In C++20, concepts provide a more powerful alternative to static_assert for template constraints.
