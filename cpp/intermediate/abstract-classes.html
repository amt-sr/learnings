Abstract Classes

An abstract class is a class that contains at least one pure virtual function and cannot be instantiated directly. It serves as an interface or a partial implementation that derived classes must complete. Abstract classes are the foundation of interface-based design in C++.

Pure Virtual Functions

A pure virtual function is declared with = 0 and has no implementation in the base class (or a default implementation that derived classes can explicitly call).

#include <iostream>
#include <memory>
#include <string>
#include <vector>

// Abstract base class — cannot be instantiated
class Renderer {
public:
    virtual ~Renderer() = default;

    // Pure virtual — must be overridden by derived classes
    virtual void render(const std::string& content) = 0;

    // Virtual with default implementation — can be overridden
    virtual void clear() {
        std::cout << "[Renderer] Clearing canvas\n";
    }

    // Pure virtual with a default implementation
    virtual void initialize() = 0;
};

void abstract_basics() {

    // auto r = std::make_unique<Renderer>();  // ERROR

    // Can have pointers/references to abstract class

}

Concrete Derived Classes

class HTMLRenderer : public Renderer {
public:
    void render(const std::string& content) override {
        std::cout << "<div>" << content << "</div>\n";
    }

    void initialize() override {
        std::cout << "[HTML] Initializing HTML renderer\n";
    }
};

class JSONRenderer : public Renderer {
public:
    void render(const std::string& content) override {
        std::cout << "{\"output\": \"" << content << "\"}\n";
    }

    void initialize() override {
        std::cout << "[JSON] Initializing JSON renderer\n";
    }

    void clear() override {
        std::cout << "[JSON] Clearing JSON buffer\n";
    }
};

void concrete_demo() {
    std::vector<std::unique_ptr<Renderer>> renderers;
    renderers.push_back(std::make_unique<HTMLRenderer>());
    renderers.push_back(std::make_unique<JSONRenderer>());

    for (const auto& renderer : renderers) {
        renderer->initialize();
        renderer->render("Hello, World!");
        renderer->clear();
    }
}

Calling Pure Virtual Base Implementation

A pure virtual function can have a body. Derived classes can call it explicitly using scope resolution.

class Database {
public:
    virtual ~Database() = default;

    // Pure virtual — but has a default implementation
    virtual void connect() = 0;

    virtual void disconnect() {
        std::cout << "Closing connection\n";
    }
};

// Pure virtual with body
void Database::connect() {
    std::cout << "Establishing base connection\n";
}

class PostgreSQL : public Database {
public:
    void connect() override {
        Database::connect();  // Call base implementation explicitly
        std::cout << "  PostgreSQL-specific setup\n";
    }
};

void call_base_demo() {
    PostgreSQL db;
    db.connect();
    // Output:
    //   Establishing base connection
    //     PostgreSQL-specific setup
}

Interface Classes (Pure Abstract Classes)

A pure abstract class (interface) has only pure virtual functions and no data members. This is the closest C++ equivalent to Java/C# interfaces.

// Interface: no data members, all functions pure virtual
class ISerializable {
public:
    virtual ~ISerializable() = default;
    virtual std::string serialize() const = 0;
    virtual void deserialize(const std::string& data) = 0;
};

class IComparable {
public:
    virtual ~IComparable() = default;
    virtual int compare(const IComparable& other) const = 0;
};

class Document : public ISerializable, public IComparable {
    std::string title_;
    int version_;
public:
    Document(std::string title, int ver) : title_(std::move(title)), version_(ver) {}

    std::string serialize() const override {
        return "{\"title\": \"" + title_ + "\", \"version\": " + std::to_string(version_) + "}";
    }

    void deserialize(const std::string& data) override {
        // Simplified — real code would parse JSON
        title_ = "deserialized";
        version_ = 1;
    }

    int compare(const IComparable& other) const override {
        // Dynamic cast to access Document-specific members
        auto& other_doc = dynamic_cast<const Document&>(other);
        return version_ - other_doc.version_;
    }
};

void interface_demo() {
    Document doc("Design Doc", 3);
    std::cout << doc.serialize() << "\n";
    // {"title": "Design Doc", "version": 3}
}

Interface naming convention: In many C++ codebases, pure abstract classes are prefixed with I (e.g., IDrawable, ISerializable). This is a convention, not a language rule. Others prefer suffixes like ...Interface or no prefix at all.

Abstract Class Design Implications

NVId Rule

If a class has a virtual destructor, it probably needs to be abstract. The Non-Virtual Interface (NVI) pattern keeps public methods non-virtual and calls protected virtual methods.

class TextProcessor {
public:
    // Public non-virtual interface — enforces invariants
    void process(const std::string& input) {
        validate(input);
        auto result = do_process(input);
        log_result(result);
    }

    virtual ~TextProcessor() = default;

protected:
    // Protected virtual hooks — extensible by derived classes
    virtual void validate(const std::string& input) {
        if (input.empty())
            throw std::invalid_argument("empty input");
    }

    virtual std::string do_process(const std::string& input) = 0;

private:
    void log_result(const std::string& result) {
        std::cout << "[LOG] Result: " << result << "\n";
    }
};

class UpperCaseProcessor : public TextProcessor {
protected:
    std::string do_process(const std::string& input) override {
        std::string result = input;
        for (auto& c : result) c = std::toupper(c);
        return result;
    }
};

NVI pattern: The public non-virtual interface ensures base class invariants are always enforced (validation, logging, locking), while derived classes only implement the core logic through protected virtual hooks. This prevents derived classes from accidentally skipping important base class behavior.
