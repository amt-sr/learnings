Virtual Functions & vtable

Virtual functions are the mechanism behind runtime polymorphism in C++. When a function is declared virtual in a base class, derived classes can override it. Calls through a base class pointer or reference are dispatched to the correct derived class implementation at runtime using a vtable (virtual function table).

Basic Virtual Functions

#include <iostream>
#include <memory>
#include <vector>
#include <string>

class Shape {
public:
    virtual double area() const = 0;          // pure virtual (see abstract classes)
    virtual std::string name() const {        // virtual with default
        return "Shape";
    }
    virtual ~Shape() = default;               // virtual destructor — critical!
};

class Circle : public Shape {
    double radius_;
public:
    explicit Circle(double r) : radius_(r) {}

    double area() const override {
        return 3.14159265358979 * radius_ * radius_;
    }

    std::string name() const override {
        return "Circle (r=" + std::to_string(radius_) + ")";
    }
};

class Rectangle : public Shape {
    double width_, height_;
public:
    Rectangle(double w, double h) : width_(w), height_(h) {}

    double area() const override {
        return width_ * height_;
    }

    std::string name() const override {
        return "Rectangle (" + std::to_string(width_) + "x" + std::to_string(height_) + ")";
    }
};

void virtual_demo() {
    // Polymorphic collection via base class pointers
    std::vector<std::unique_ptr<Shape>> shapes;
    shapes.push_back(std::make_unique<Circle>(5.0));
    shapes.push_back(std::make_unique<Rectangle>(3.0, 4.0));
    shapes.push_back(std::make_unique<Circle>(2.0));

    // Calls dispatched through vtable at runtime
    for (const auto& shape : shapes) {
        std::cout << shape->name() << ": area = " << shape->area() << "\n";
    }
}
// Output:
//   Circle (r=5.000000): area = 78.5398
//   Rectangle (3.000000x4.000000): area = 12
//   Circle (r=2.000000): area = 12.5664

How the vtable Works

Every class with virtual functions has an associated vtable — an array of function pointers. Each object of such a class contains a hidden pointer (vptr) to its class's vtable.

// Conceptual model of the vtable:

// Shape's vtable:
//   [0] -> Shape::area      (pure virtual — typically __cxa_pure_virtual)
//   [1] -> Shape::name
//   [2] -> Shape::~Shape

// Circle's vtable:
//   [0] -> Circle::area     (overridden)
//   [1] -> Circle::name     (overridden)
//   [2] -> Circle::~Circle  (overrides base destructor)

// When you call shape->area():
//   1. Compiler looks up vptr inside the Shape object
//   2. Follows vptr to Circle's vtable (since the object is actually a Circle)
//   3. Calls the function at slot [0] — Circle::area()

vtable layout:

  • One vtable per class (not per object)
  • Each object has one vptr (typically 8 bytes on 64-bit systems)
  • The vptr is set by the constructor during object creation
  • Virtual call dispatch is typically one pointer dereference + one indirect call

Virtual Destructor

Critical rule: If a class has any virtual functions, its destructor must also be virtual. Failing to do so causes undefined behavior when deleting a derived class object through a base class pointer.

class Base {
public:
    virtual void do_work() { std::cout << "Base work\n"; }
    virtual ~Base() { std::cout << "~Base\n"; }
};

class Derived : public Base {
    int* data_;
public:
    Derived() : data_(new int[100]) {}
    ~Derived() override {
        delete[] data_;
        std::cout << "~Derived (freed data_)\n";
    }
    void do_work() override { std::cout << "Derived work\n"; }
};

void virtual_destructor_demo() {
    Base* ptr = new Derived();
    ptr->do_work();   // "Derived work" — correct dispatch
    delete ptr;        // Correctly calls ~Derived() then ~Base()

    // If ~Base() were NOT virtual:
    //   delete ptr;  // Would only call ~Base() — memory leak!
}

Override Keyword

class Animal {
public:
    virtual void speak() const { std::cout << "...\n"; }
    virtual void eat(const std::string& food) { /* ... */ }
    virtual void move() { /* ... */ }
};

class Dog : public Animal {
public:
    // override — compiler checks that a base class virtual function exists
    void speak() const override { std::cout << "Woof!\n"; }

    // Error catching: override catches signature mismatches
    // void speak() override;        // ERROR: missing const
    // void eat(std::string food);   // ERROR: should be const ref
    // void run() override;           // ERROR: no base class function 'run'

    // final — prevents further overriding in derived classes
    void move() override final {
        std::cout << "Dog running\n";
    }
};

Best practice: Always use override when overriding a virtual function. It catches bugs at compile time rather than creating a new overloaded function that silently fails to override the base version.

When to Use Virtual Functions

Use virtual functions when:

  • You need runtime polymorphism through base class pointers/references
  • You have a collection of related types with a common interface (abstract classes)
  • The set of types is open-ended (new derived classes can be added)

Avoid virtual functions when:

  • Performance is critical and polymorphism can be resolved at compile time
  • The type set is closed and known — consider std::variant with std::visit (optional and variant)
  • You are working in a deeply embedded system where the vptr overhead matters

Virtual functions are fundamental to OOP in C++. They are used heavily with smart pointers for polymorphic ownership and with abstract classes for defining interfaces. Next, we explore abstract classes and pure virtual functions.
