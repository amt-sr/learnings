Copy Elision

Copy elision is an optimization where the compiler omits copy or move constructors, constructing the object directly in the target memory location. In C++17, certain cases of copy elision became mandatory, meaning the copy/move constructor may not even exist and the program must still compile.

RVO — Return Value Optimization

When a function returns a local object by value, the compiler can construct the result directly in the caller's storage, eliminating the copy/move entirely.

#include <iostream>
#include <string>

class BigObject {
    std::string data_;
public:
    explicit BigObject(const std::string& s) : data_(s) {
        std::cout << "  Constructor: " << data_ << "\n";
    }

    BigObject(const BigObject& other) : data_(other.data_) {
        std::cout << "  Copy constructor: " << data_ << "\n";
    }

    BigObject(BigObject&& other) noexcept : data_(std::move(other.data_)) {
        std::cout << "  Move constructor: " << data_ << "\n";
    }

    ~BigObject() { std::cout << "  Destructor: " << data_ << "\n"; }
};

// The compiler constructs 'obj' directly in the caller's memory
BigObject create_object(const std::string& name) {
    std::cout << "Inside create_object\n";
    return BigObject(name);  // No copy/move — RVO
}

void rvo_demo() {
    std::cout << "Before create_object\n";
    BigObject obj = create_object("my_object");
    std::cout << "After create_object\n";
}
// Output (with RVO — typical):
//   Before create_object
//   Inside create_object
//     Constructor: my_object
//   After create_object
//     Destructor: my_object
//
// Note: Only ONE constructor call and ONE destructor call.
// No copy or move constructor is called.

NRVO — Named Return Value Optimization

When the returned object has a name, NRVO can still elide the copy, but it was optional before C++17.

// NRVO — the returned variable 'result' is named
BigObject make_named_object(const std::string& a, const std::string& b) {
    std::cout << "Inside make_named_object\n";
    BigObject result(a + b);   // named local variable
    // ... do some processing ...
    return result;  // NRVO may elide the copy/move
}

void nrvo_demo() {
    std::cout << "Before make_named_object\n";
    BigObject obj = make_named_object("hello", "world");
    std::cout << "After make_named_object\n";
}
// Output (with NRVO — typical):
//   Before make_named_object
//   Inside make_named_object
//     Constructor: helloworld
//   After make_named_object
//     Destructor: helloworld

NRVO is still optional: Unlike RVO for prvalues (C++17 mandatory), NRVO is still an optimization the compiler may or may not perform. If NRVO does not happen, the move constructor is used (if available). NRVO can fail if there are multiple return paths with different named objects.

Mandatory Copy Elision (C++17)

C++17 made copy elision mandatory in two cases involving prvalues (pure rvalues).

// C++17: This object is guaranteed to be constructed directly
// in 'obj' — no copy, no move, and the copy/move constructors
// need not even exist or be accessible.

class NoCopy {
public:
    NoCopy() = default;
    NoCopy(const NoCopy&) = delete;   // deleted copy
    NoCopy(NoCopy&&) = delete;        // deleted move
};

// C++17: This works! Mandatory copy elision means the deleted
// constructors are never needed.
NoCopy make_no_copy() {
    return NoCopy();  // C++17 OK: mandatory elision
}

void mandatory_elision_demo() {
    // NoCopy is constructed directly — no copy/move attempted
    NoCopy obj = make_no_copy();

    // Direct initialization from prvalue — also mandatory elision
    NoCopy obj2 = NoCopy();  // C++17 OK

    // This is still a copy initialization from an lvalue — ERROR
    // NoCopy obj3 = obj2;  // ERROR: uses deleted copy constructor
}

Copy Elision and std::move — Do Not Return std::move

Anti-pattern: Do not use std::move on return values. It prevents RVO/NRVO and can actually force a move when elision would have been better.

// BAD: std::move on return prevents NRVO
BigObject bad_create(const std::string& s) {
    BigObject obj(s);
    return std::move(obj);   // Prevents NRVO! Forces move constructor.
}

// GOOD: just return the local variable
BigObject good_create(const std::string& s) {
    BigObject obj(s);
    return obj;  // NRVO applies — no copy/move at all
}

When Copy Elision Matters

#include <vector>
#include <iostream>

class Matrix {
    std::vector<double> data_;
    size_t rows_, cols_;
public:
    Matrix(size_t rows, size_t cols)
        : data_(rows * cols, 0.0), rows_(rows), cols_(cols) {
        std::cout << "  Constructing " << rows << "x" << cols
                  << " matrix (" << data_.size() << " elements)\n";
    }

    Matrix(const Matrix& other) : data_(other.data_), rows_(other.rows_), cols_(other.cols_) {
        std::cout << "  Copying " << rows_ << "x" << cols_
                  << " matrix (" << data_.size() << " elements)\n";
    }

    Matrix(Matrix&& other) noexcept
        : data_(std::move(other.data_)), rows_(other.rows_), cols_(other.cols_) {
        std::cout << "  Moving " << rows_ << "x" << cols_
                  << " matrix (" << data_.size() << " elements)\n";
    }
};

// Without copy elision: constructing a 1000x1000 matrix,
// then copying it (copying 8 million doubles!) would be catastrophic.
// With RVO: one construction, zero copies.
Matrix create_identity(size_t n) {
    Matrix m(n, n);
    // ... set diagonal elements ...
    return m;  // RVO/NRVO — no copy
}

void performance_demo() {
    std::cout << "Creating identity matrix:\n";
    Matrix m = create_identity(1000);
    std::cout << "Done.\n";
    // Output:
    //   Creating identity matrix:
    //     Constructing 1000x1000 matrix (1000000 elements)
    //   Done.
    // (No copy or move constructor called!)
}

Copy elision complements move semantics — when elision happens, moves are unnecessary. When elision cannot happen (NRVO failure), move semantics provide the fallback. Together with RAII, they form C++'s resource management story. Next, we explore template specialization.
