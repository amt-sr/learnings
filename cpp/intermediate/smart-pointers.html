Smart Pointers

Smart pointers are objects that manage the lifetime of dynamically allocated memory automatically. They are the modern C++ replacement for raw new/delete and are defined in the <memory> header. Using smart pointers eliminates entire classes of bugs: memory leaks, double frees, and dangling pointers.

Why Smart Pointers?

Raw pointer dangers: Forgetting to delete, deleting twice, accessing freed memory, and exception safety issues when allocations are mixed with throwing operations.

// RAW POINTER PITFALLS (avoid this pattern)
void raw_pointer_problems() {
    int* p = new int(42);
    if (some_condition()) {
        throw std::runtime_error("leak!"); // p is never freed
    }
    delete p;
}

// SMART POINTER SOLUTION
void smart_pointer_solution() {
    auto p = std::make_unique<int>(42);
    if (some_condition()) {
        throw std::runtime_error("safe!"); // p is freed automatically
    }
    // no delete needed
}

std::unique_ptr

std::unique_ptr represents exclusive ownership of a dynamically allocated object. Only one unique_ptr can own a given resource at a time. It cannot be copied, only moved.

#include <memory>
#include <iostream>

class Widget {
    std::string name;
public:
    Widget(std::string n) : name(std::move(n)) {
        std::cout << "Widget '" << name << "' created\n";
    }
    ~Widget() { std::cout << "Widget '" << name << "' destroyed\n"; }
    void greet() const { std::cout << "Hello from " << name << "\n"; }
};

void unique_ptr_basics() {
    // create with make_unique (C++14) — preferred over new
    auto w1 = std::make_unique<Widget>("Alice");

    w1->greet();          // access via ->
    (*w1).greet();         // dereference with *

    // cannot copy: auto w2 = w1;          // ERROR
    // can move:
    auto w2 = std::move(w1);  // w1 is now nullptr
    // w1->greet();                        // CRASH — w1 is empty

    std::cout << "w1 is " << (w1 ? "not null" : "null") << "\n";
    std::cout << "w2 is " << (w2 ? "not null" : "null") << "\n";
}
// Output:
//   Widget 'Alice' created
//   Hello from Alice
//   Hello from Alice
//   w1 is null
//   w2 is not null
//   Widget 'Alice' destroyed

unique_ptr with Arrays

void unique_ptr_arrays() {
    // unique_ptr to arrays uses partial specialization
    auto arr = std::make_unique<int[]>(5);
    for (int i = 0; i < 5; ++i)
        arr[i] = i * 10;

    for (int i = 0; i < 5; ++i)
        std::cout << arr[i] << " ";  // 0 10 20 30 40
    std::cout << "\n";
} // calls delete[] arr

Best practice: Always prefer std::make_unique over std::unique_ptr<T>(new T(...)). It is safer (no exception leaks), more concise, and can be slightly faster due to better cache locality.

std::shared_ptr

std::shared_ptr represents shared ownership. Multiple shared_ptr instances can own the same resource. The resource is freed when the last shared_ptr is destroyed. Internally it uses a reference-counted control block.

#include <memory>
#include <iostream>

class Resource {
public:
    Resource() { std::cout << "Resource acquired\n"; }
    ~Resource() { std::cout << "Resource released\n"; }
    void use() { std::cout << "Using resource\n"; }
};

void shared_ptr_basics() {
    std::cout << "use_count: " << sp1.use_count() << "\n";  // 1

    {
        auto sp2 = sp1;  // copy — ref count increments
        std::cout << "use_count: " << sp1.use_count() << "\n";  // 2
        sp2->use();
    }
    // sp2 destroyed — ref count decrements

    std::cout << "use_count: " << sp1.use_count() << "\n";  // 1
}
// sp1 destroyed — Resource released

Performance note: shared_ptr has overhead: the control block requires a heap allocation (unless you use make_shared which combines them), and every copy involves atomic reference count operations. Use unique_ptr when shared ownership is not needed.

make_shared vs make_unique

// make_shared performs a single allocation for object + control block
auto sp = std::make_shared<Widget>("Bob");

// Constructing from new performs two allocations
auto sp2 = std::shared_ptr<Widget>(new Widget("Carol"));  // less efficient

// The aliasing constructor: shared_ptr to part of an object
struct Node {
    int value;
    std::shared_ptr<Node> next;
};

auto node = std::make_shared<Node>();
node->value = 42;
// shared_ptr to just the value, sharing ownership with node
std::shared_ptr<int> val_ptr(node, &node->value);

std::weak_ptr

std::weak_ptr is a non-owning observer of a shared_ptr. It does not contribute to the reference count and is used to break circular references.

#include <memory>
#include <iostream>

class B;  // forward declaration

class A {
public:
    std::shared_ptr<B> b_ptr;
    ~A() { std::cout << "A destroyed\n"; }
};

class B {
public:
    // weak_ptr breaks the cycle — A and B can both be freed
    std::weak_ptr<A> a_ptr;
    ~B() { std::cout << "B destroyed\n"; }
};

void circular_reference_demo() {
    auto a = std::make_shared<A>();
    auto b = std::make_shared<B>();
    a->b_ptr = b;
    b->a_ptr = a;

    std::cout << "a.use_count = " << a.use_count() << "\n";  // 1 (b holds weak_ptr)
    std::cout << "b.use_count = " << b.use_count() << "\n";  // 2 (a + b shared)

    // Access weak_ptr's object via lock()
    if (auto locked = b->a_ptr.lock()) {
        std::cout << "A is still alive\n";
    }
}
// Output:
//   a.use_count = 1
//   b.use_count = 2
//   A is still alive
//   A destroyed
//   B destroyed

Without weak_ptr: If both A and B held shared_ptr to each other, neither would ever be destroyed — a classic memory leak via reference counting. weak_ptr breaks this cycle.

Common Patterns

Factory Functions

class DatabaseConnection {
    DatabaseConnection(const std::string& conn_str)
        : connection_string(conn_str) {}
public:
    std::string connection_string;

    // Factory returns unique_ptr — hides implementation
    static std::unique_ptr<DatabaseConnection> create(const std::string& conn_str) {
        return std::make_unique<DatabaseConnection>(conn_str);
    }
};

void factory_demo() {
    // Caller cannot see DatabaseConnection's constructor
    auto db = DatabaseConnection::create("postgresql://localhost/mydb");
    std::cout << db->connection_string << "\n";
}

Unique_ptr as Function Parameters

// Pass by reference if you don't transfer ownership
void use_widget(const std::unique_ptr<Widget>& w) {
    w->greet();
}

// Pass by value to TRANSFER ownership (caller gives up the pointer)
void consume_widget(std::unique_ptr<Widget> w) {
    w->greet();
    // w is destroyed here — Widget is freed
}

void parameter_demo() {
    auto w = std::make_unique<Widget>("Dave");
    use_widget(w);           // w still valid after this
    consume_widget(std::move(w));  // w is now nullptr
}

When to Use Which

Decision guide:

  • Default to unique_ptr for all dynamically allocated objects.
  • Use shared_ptr only when ownership must truly be shared (e.g., caches, observer patterns).
  • Use weak_ptr to observe a shared_ptr without extending its lifetime, or to break circular references.

Smart pointers work hand-in-hand with RAII (Resource Acquisition Is Initialization) and complement the move semantics model. Understanding them is essential before diving into move semantics, where we will explore how unique_ptr achieves its move-only behavior.
