Async / Futures

std::async, std::future, and std::promise provide a higher-level abstraction for asynchronous programming than raw threads. Instead of manually managing threads and shared state, you launch a task and get back a future -- a handle to a result that will be available later.

std::async

std::async launches an asynchronous task and returns a std::future that holds the result. It may run the task in a new thread or defer execution until get() is called.

#include <future>
#include <iostream>
#include <vector>
#include <numeric>

long compute_sum(size_t start, size_t end) {
    long sum = 0;
    for (size_t i = start; i < end; ++i) {
        sum += static_cast<long>(i);
    }
    return sum;
}

void async_demo() {
    // Launch async (may create a new thread)
    std::future<long> f1 = std::async(std::launch::async,
        compute_sum, 1, 5000000);

    // Launch async with lambda
    std::future<long> f2 = std::async(std::launch::async, []() {
        return compute_sum(5000000, 10000001);
    });

    // Do other work while tasks run in background
    std::cout << "Main thread working...\n";

    // get() blocks until result is ready
    long sum1 = f1.get();
    long sum2 = f2.get();

    std::cout << "Total: " << (sum1 + sum2) << "\n";
}

Launch Policies

PolicyBehavior
std::launch::asyncForces creation of a new thread
std::launch::deferredDefers execution until get()/wait() is called (lazy)
std::launch::async | std::launch::deferredImplementation chooses (default)

Default launch policy: When you do not specify a policy, the implementation may choose either async or deferred. This makes timing unpredictable. For guaranteed parallelism, always use std::launch::async.

std::promise and std::future

std::promise is the producer side of a future -- you set a value or exception, and the corresponding std::future retrieves it. This gives you fine-grained control over when results are ready.

#include <future>
#include <iostream>
#include <thread>
#include <cmath>

void compute_factorial(int n, std::promise<unsigned long long> prom) {
    unsigned long long result = 1;
    for (int i = 2; i <= n; ++i) {
        result *= i;
        if (result == 0) {  // overflow
            prom.set_exception(
                std::make_exception_ptr(
                    std::overflow_error("Factorial overflow")));
            return;
        }
    }
    prom.set_value(result);
}

void promise_demo() {
    std::promise<unsigned long long> prom;
    std::future<unsigned long long> fut = prom.get_future();

    std::thread t(compute_factorial, 20, std::move(prom));

    try {
        unsigned long long result = fut.get();
        std::cout << "20! = " << result << "\n";
    } catch (const std::exception& e) {
        std::cerr << "Error: " << e.what() << "\n";
    }

    t.join();
}

std::shared_future

Unlike std::future (which can only be moved, not copied), std::shared_future can be copied. Multiple threads can share a single result.

#include <future>
#include <iostream>
#include <thread>
#include <vector>

void shared_future_demo() {
    std::promise<int> prom;
    std::shared_future<int> sfut = prom.get_future().share();

    // Multiple readers can wait on the same result
    std::vector<std::thread> readers;
    for (int i = 0; i < 5; ++i) {
        readers.emplace_back([sfut, i]() {
            int result = sfut.get();  // all threads get the same value
            std::cout << "Reader " << i << " got: " << result << "\n";
        });
    }

    // Producer sets the value -- all readers wake up
    prom.set_value(42);

    for (auto& t : readers) t.join();
}

wait_for and wait_until

#include <future>
#include <iostream>

void timeout_demo() {
    std::future<int> fut = std::async(std::launch::async, []() {
        std::this_thread::sleep_for(std::chrono::seconds(2));
        return 42;
    });

    // Wait with timeout
    auto status = fut.wait_for(std::chrono::milliseconds(500));
    if (status == std::future_status::ready) {
        std::cout << "Result: " << fut.get() << "\n";
    } else if (status == std::future_status::timeout) {
        std::cout << "Still running...\n";
    } else {
        std::cout << "Deferred\n";
    }
    // Output: Still running...
}

Async vs Thread

Return valuestd::future (easy result retrieval)Must use shared state/pointers
Exception propagationAutomatic (re-thrown on get())Must handle manually

Thread poolImplementation may reuseNew thread each time
OverheadPossible (shared state)Lower per-thread overhead
Best forTasks with return valuesLong-running background work

Best practice: Use std::async when you need to launch a task and wait for its result. Use std::thread for long-running background work (e.g., event loops, I/O threads). Use std::promise/std::future when you need to pass results between manually managed threads.

Async/futures build on std::thread and use mutexes internally for shared state. For waiting on conditions, see condition variables.
