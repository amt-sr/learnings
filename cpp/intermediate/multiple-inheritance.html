Multiple Inheritance

C++ allows a class to inherit from multiple base classes. While powerful, multiple inheritance introduces complexity: the diamond problem, ambiguity in member access, and complications with virtual function dispatch. Understanding these issues is essential for designing robust class hierarchies.

Basic Multiple Inheritance

#include <iostream>
#include <string>

class Printable {
public:
    virtual ~Printable() = default;
    virtual void print() const = 0;
};

class Storable {
public:
    virtual ~Storable() = default;
    virtual void save() const = 0;
    virtual void load() = 0;
};

// Document inherits from both Printable and Storable
class Document : public Printable, public Storable {
    std::string content_;
public:
    explicit Document(std::string content) : content_(std::move(content)) {}

    void print() const override {
        std::cout << content_ << "\n";
    }

    void save() const override {
        std::cout << "Saving: " << content_ << "\n";
    }

    void load() override {
        content_ = "loaded content";
        std::cout << "Loaded document\n";
    }
};

void basic_mi_demo() {
    Document doc("Hello, Multiple Inheritance!");
    doc.print();
    doc.save();

    // Interface-based MI is generally safe and well-accepted
    Printable& p = doc;
    Storable& s = doc;
    p.print();
    s.save();
}

The Diamond Problem

The diamond problem occurs when a class inherits from two classes that both inherit from the same base class.

#include <iostream>

//     Person
//     /     \
// Student   Employee
//     \     /
//   TeachingAssistant

class Person {
protected:
    std::string name_;
public:
    explicit Person(std::string name) : name_(std::move(name)) {
        std::cout << "Person(" << name_ << ") constructed\n";
    }
    virtual ~Person() = default;

    void greet() const { std::cout << "I am " << name_ << "\n"; }
};

class Student : public Person {
public:
    explicit Student(const std::string& name) : Person(name) {
        std::cout << "Student(" << name << ") constructed\n";
    }
};

class Employee : public Person {
public:
    explicit Employee(const std::string& name) : Person(name) {
        std::cout << "Employee(" << name << ") constructed\n";
    }
};

// WITHOUT virtual inheritance — diamond problem
class TeachingAssistant : public Student, public Employee {
public:
    TeachingAssistant(const std::string& name)
        : Student(name), Employee(name) {
        // Two separate Person subobjects exist!
        // name_ is ambiguous
    }

    void introduce() {
        // greet();  // ERROR: ambiguous — Student::Person::greet or Employee::Person::greet?
        Student::greet();   // OK: specify which
        Employee::greet();  // OK: specify which
    }
};

void diamond_problem_demo() {
    TeachingAssistant ta("Alice");
    // Output:
    //   Person(Alice) constructed    (Student's Person)
    //   Student(Alice) constructed
    //   Person(Alice) constructed     (Employee's Person)
    //   Employee(Alice) constructed

    // ta.greet();  // ERROR: ambiguous
    ta.Student::greet();     // "I am Alice"
    ta.Employee::greet();    // "I am Alice"

    // Two separate name_ members — wasteful and confusing
}

Virtual Inheritance — Solving the Diamond

// With virtual inheritance — only ONE Person subobject
class Person2 {
protected:
    std::string name_;
public:
    explicit Person2(std::string name) : name_(std::move(name)) {
        std::cout << "Person2(" << name_ << ") constructed\n";
    }
    virtual ~Person2() = default;

    void greet() const { std::cout << "I am " << name_ << "\n"; }
};

class Student2 : virtual public Person2 {
public:
    explicit Student2(const std::string& name) : Person2(name) {
        std::cout << "Student2(" << name << ") constructed\n";
    }
};

class Employee2 : virtual public Person2 {
public:
    explicit Employee2(const std::string& name) : Person2(name) {
        std::cout << "Employee2(" << name << ") constructed\n";
    }
};

class TeachingAssistant2 : public Student2, public Employee2 {
public:
    // Most-derived class constructs the virtual base
    explicit TeachingAssistant2(const std::string& name)
        : Person2(name), Student2(name), Employee2(name) {
        // Only ONE Person2 subobject exists
        std::cout << "TA2(" << name << ") constructed\n";
    }

    void introduce() {
        greet();  // No ambiguity — only one Person2::greet
    }
};

void virtual_inheritance_demo() {
    TeachingAssistant2 ta2("Bob");
    // Output:
    //   Person2(Bob) constructed      (only once!)
    //   Student2(Bob) constructed
    //   Employee2(Bob) constructed
    //   TA2(Bob) constructed

    ta2.greet();  // No ambiguity: "I am Bob"
}

Virtual inheritance cost: Virtual inheritance adds an extra level of indirection (a virtual base pointer) to access the shared base subobject. The most-derived class is responsible for constructing the virtual base. This adds complexity and slight runtime overhead. Use it only when you actually need it.

Ambiguity Resolution

class A {
public:
    void do_work() { std::cout << "A::do_work\n"; }
    void process() { std::cout << "A::process\n"; }
};

class B : public A {
public:
    void do_work() override { std::cout << "B::do_work\n"; }
};

class C : public A {
public:
    void do_work() override { std::cout << "C::do_work\n"; }
};

class D : public B, public C {
public:
    void call_both() {
        B::do_work();   // calls A::do_work as overridden by B
        C::do_work();   // calls A::do_work as overridden by C

        // process() — inherited only through B and C, but B's and C's
        // process() both refer to A::process, so this is ambiguous:
        // process();  // ERROR: ambiguous

        B::process();  // OK
        C::process();  // OK — same function, different paths
    }
};

Multiple Inheritance Design Guidelines

Safe uses of multiple inheritance:

  • Interface inheritance — inheriting from multiple pure abstract classes (no data members). This is the most common and recommended pattern.
  • Mixins — small classes providing a specific capability (e.g., Serializable, Observer).

Risky uses of multiple inheritance:

  • Multiple base classes with data members (stateful bases)
  • Deep inheritance hierarchies with virtual inheritance
  • Frameworks where you cannot control all base classes
// Safe: Interface-only MI
class ISerializable {
public:
    virtual ~ISerializable() = default;
    virtual std::string serialize() const = 0;
};

class IObservable {
public:
    virtual ~IObservable() = default;
    virtual void notify() = 0;
};

class Widget : public ISerializable, public IObservable {
    // Concrete implementation
    std::string serialize() const override { return "widget_data"; }
    void notify() override { std::cout << "Widget changed\n"; }
};

Alternative: Composition

// Instead of inheriting from multiple base classes, consider composition
class Logger {
public:
    void log(const std::string& msg) { std::cout << "[LOG] " << msg << "\n"; }
};

class Timer {
public:
    void start() { std::cout << "Timer started\n"; }
    void stop() { std::cout << "Timer stopped\n"; }
};

class Service {
    Logger logger_;
    Timer timer_;
public:
    void run() {
        timer_.start();
        logger_.log("Service running");
        // do work
        logger_.log("Service done");
        timer_.stop();
    }
};
// Composition: less coupling, no ambiguity, easier to test and modify

Multiple inheritance is most commonly used with abstract classes (interface inheritance). It can also involve operator overloading complexities when base classes define the same operators. Next, we explore operator overloading.
