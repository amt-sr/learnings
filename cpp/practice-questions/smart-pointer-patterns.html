Design Patterns with Smart Pointers

Topics Covered: Smart Pointers, Design Patterns, Memory Management, RAII

Problem Statement

Implement three classic design patterns using C++ smart pointers to demonstrate proper ownership semantics:

  1. Factory Pattern with unique_ptr: A factory that creates objects and transfers exclusive ownership to the caller.
  2. Observer Pattern with weak_ptr: An observer system where subjects hold weak references to observers to avoid dangling pointers when observers are destroyed.
  3. Tree Structure with shared_ptr: A binary tree where child nodes are shared (parent and children reference each other via shared/weak pointers to break cycles).

Pattern 1: Factory with unique_ptr

Click to reveal Solution

#include <iostream>
#include <memory>
#include <string>
#include <unordered_map>

// Base interface
class Message {
public:
    virtual ~Message() = default;
    virtual std::string serialize() const = 0;
    virtual std::string type() const = 0;
};

// Concrete message types
class TextMessage : public Message {
    std::string text_;
public:
    explicit TextMessage(std::string text) : text_(std::move(text)) {}
    std::string serialize() const override { return "TEXT:" + text_; }
    std::string type() const override { return "TextMessage"; }
};

class BinaryMessage : public Message {
    std::vector<uint8_t> data_;
public:
    explicit BinaryMessage(std::vector<uint8_t> data) : data_(std::move(data)) {}
    std::string serialize() const override { return "BIN:size=" + std::to_string(data_.size()); }
    std::string type() const override { return "BinaryMessage"; }
};

// Factory using unique_ptr: exclusive ownership transfer
class MessageFactory {
public:
    enum class Type { Text, Binary };

    // Returns unique_ptr: caller owns the object exclusively
    std::unique_ptr<Message> create(Type type, const std::string& content) {
        switch (type) {
            case Type::Text:
                return std::make_unique<TextMessage>(content);
            case Type::Binary: {
                std::vector<uint8_t> data(content.begin(), content.end());
                return std::make_unique<BinaryMessage>(std::move(data));
            }
        }
        return nullptr;
    }
};

int main() {
    MessageFactory factory;

    auto msg1 = factory.create(MessageFactory::Type::Text, "Hello World");
    auto msg2 = factory.create(MessageFactory::Type::Binary, "\x01\x02\x03");

    std::cout << msg1->type() << ": " << msg1->serialize() << "\n";
    std::cout << msg2->type() << ": " << msg2->serialize() << "\n";

    // unique_ptr is move-only: transfer ownership
    auto msg3 = std::move(msg1);
    // msg1 is now nullptr (moved-from state)

    return 0;
}

Explanation:

  • std::make_unique creates the object on the heap and wraps it in a unique_ptr.
  • unique_ptr represents exclusive ownership — only one unique_ptr can own the object at any time.
  • The factory returns by value (RVO/move), transferring ownership to the caller.
  • This eliminates new/delete at the call site and makes ownership semantics explicit.

Pattern 2: Observer with weak_ptr

Click to reveal Solution

#include <iostream>
#include <memory>
#include <string>
#include <vector>
#include <algorithm>

class Observer;

// Subject holds weak_ptr to observers (breaks ownership cycle)
class Subject {
public:
    void addObserver(const std::shared_ptr<Observer>& observer) {
        observers_.push_back(observer);  // stores weak_ptr internally
    }

    void removeObserver(const std::shared_ptr<Observer>& observer) {
        std::weak_ptr<Observer> weak = observer;
        observers_.erase(
            std::remove_if(observers_.begin(), observers_.end(),
                [&weak](const std::weak_ptr<Observer>& w) {
                    return w.lock() == weak.lock();
                }),
            observers_.end());
    }

    void notify(const std::string& event) {
        // Clean up expired observers
        observers_.erase(
            std::remove_if(observers_.begin(), observers_.end(),
                [](const std::weak_ptr<Observer>& w) {
                    return w.expired();
                }),
            observers_.end());

        // Notify surviving observers
        for (const auto& weakObs : observers_) {
            if (auto obs = weakObs.lock()) {
                obs->onNotify(event);
            }
        }
    }

private:
    std::vector<std::weak_ptr<Observer>> observers_;
};

class Observer : public std::enable_shared_from_this<Observer> {
    std::string name_;
public:
    explicit Observer(std::string name) : name_(std::move(name)) {}
    virtual ~Observer() { std::cout << "Observer '" << name_ << "' destroyed\n"; }
    virtual void onNotify(const std::string& event) {
        std::cout << name_ << " received: " << event << "\n";
    }
    std::string name() const { return name_; }
};

int main() {
    auto subject = std::make_shared<Subject>();

    // Create observers (shared ownership)
    auto obs1 = std::make_shared<Observer>("Logger");
    auto obs2 = std::make_shared<Observer>("UI");
    auto obs3 = std::make_shared<Observer>("DB");

    subject->addObserver(obs1);
    subject->addObserver(obs2);
    subject->addObserver(obs3);

    subject->notify("Data received");
    // All three observers receive notification

    // Observer destroyed (shared_ptr count drops to 0)
    obs2.reset();
    std::cout << "After UI observer destroyed:\n";
    subject->notify("Update");
    // Only Logger and DB receive notification (UI is gone)

    return 0;
}

Explanation:

  • The Subject stores weak_ptr to observers, not shared_ptr. This prevents reference cycles (Subject → Observer → Subject).
  • weak_ptr::lock() returns a shared_ptr if the object is still alive, or nullptr if it has been destroyed.
  • weak_ptr::expired() checks if the object has been destroyed without creating a temporary shared_ptr.
  • Before each notification, we clean up expired observers to keep the vector tidy.
  • Observers derive from enable_shared_from_this to safely get a shared_ptr to themselves when needed.

Pattern 3: Tree with shared_ptr / weak_ptr

Click to reveal Solution

#include <iostream>
#include <memory>
#include <string>
#include <vector>

class TreeNode : public std::enable_shared_from_this<TreeNode> {
    int value_;
    std::shared_ptr<TreeNode> left_;
    std::shared_ptr<TreeNode> right_;
    std::weak_ptr<TreeNode> parent_;  // weak: breaks cycle

public:
    explicit TreeNode(int value)
        : value_(value) {}

    void setLeft(std::shared_ptr<TreeNode> child) {
        left_ = std::move(child);
        if (left_) left_->parent_ = shared_from_this();
    }

    void setRight(std::shared_ptr<TreeNode> child) {
        right_ = std::move(child);
        if (right_) right_->parent_ = shared_from_this();
    }

    int value() const { return value_; }
    std::shared_ptr<TreeNode> left() const { return left_; }
    std::shared_ptr<TreeNode> right() const { return right_; }
    std::shared_ptr<TreeNode> parent() const { return parent_.lock(); }

    void printTree(int depth = 0) const {
        std::string indent(depth * 2, ' ');
        std::cout << indent << "Node(" << value_ << ")";
        auto p = parent_.lock();
        if (p) std::cout << " [parent=" << p->value_ << "]";
        std::cout << "\n";
        if (left_)  left_->printTree(depth + 1);
        if (right_) right_->printTree(depth + 1);
    }
};

int main() {
    // Build a tree:
    //       1
    //      / \
    //     2   3
    //    / \
    //   4   5

    auto root = std::make_shared<TreeNode>(1);
    auto n2   = std::make_shared<TreeNode>(2);
    auto n3   = std::make_shared<TreeNode>(3);
    auto n4   = std::make_shared<TreeNode>(4);
    auto n5   = std::make_shared<TreeNode>(5);

    root->setLeft(n2);
    root->setRight(n3);
    n2->setLeft(n4);
    n2->setRight(n5);

    std::cout << "Tree structure:\n";
    root->printTree();

    // Demonstrate navigation
    std::cout << "\nParent of node 4: ";
    auto parent = n4->parent();
    if (parent) std::cout << parent->value() << "\n";  // 2

    // Drop root's left subtree
    root->setLeft(nullptr);
    // n2, n4, n5 are now destroyed (ref count drops to 0)

    std::cout << "\nAfter dropping left subtree:\n";
    root->printTree();

    return 0;
}

Explanation:

  • shared_ptr for parent→child links: the parent "owns" its children. When a parent is destroyed, its children are destroyed too (unless shared elsewhere).
  • weak_ptr for child→parent links: avoids the reference cycle (parent → child → parent) that would cause memory leaks with shared_ptr alone.
  • enable_shared_from_this is needed for a node to safely create a shared_ptr to itself (required when setting the parent pointer from within the node).
  • When root->setLeft(nullptr) is called, the shared_ptr to n2 is released. Since n4 and n5 are only held by n2, they are destroyed as well.

Smart Pointer Selection Guide:

Pointer TypeUse WhenOwnership
unique_ptrExclusive ownership, single ownerMoves only, non-copyable
shared_ptrShared ownership, multiple ownersReference counted, copyable
weak_ptrNon-owning reference, break cyclesNo ownership, must lock() to use

Follow-up Questions

Variations to Try

  1. Abstract Factory: Extend the factory to support families of related objects (e.g., different message protocols).
  2. Signal/Slot with weak_ptr: Implement a signal-slot system where slots can be disconnected automatically when the listener is destroyed.
  3. DAG (Directed Acyclic Graph): Implement a DAG with shared_ptr for edges and weak_ptr for back-references. Add cycle detection.
  4. Polymorphic factory: Register types at runtime using a map of factory functions, enabling plugins to add new types.

Common Mistakes:

  • Creating a shared_ptr from this directly causes double-free. Always use enable_shared_from_this::shared_from_this().
  • Using shared_ptr for both sides of a parent-child relationship creates reference cycles that are never freed.
  • Never store a weak_ptr and forget to check expired() or lock() before using it.
