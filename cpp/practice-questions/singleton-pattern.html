Thread-Safe Singleton Pattern

Topics Covered: Design Patterns, Multithreading, Memory Management

Problem Statement

Implement the Singleton pattern in C++ using two approaches:

  1. Meyer's Singleton (C++11 guaranteed thread-safe, using a local static variable).
  2. Double-Checked Locking (manual thread-safe implementation with std::mutex and std::call_once).

Discuss when to use each approach and explain why the naive singleton is not thread-safe.

Meyer's Singleton (Preferred)

Click to reveal Solution

#include <iostream>
#include <string>
#include <thread>
#include <vector>
#include <mutex>
#include <syncstream>  // C++20, for osyncstream

// Meyer's Singleton: thread-safe by C++11 standard guarantee
class Logger {
public:
    // Delete copy and move operations
    Logger(const Logger&) = delete;
    Logger& operator=(const Logger&) = delete;
    Logger(Logger&&) = delete;
    Logger& operator=(Logger&&) = delete;

    // Thread-safe access point
    static Logger& instance() {
        // C++11 guarantees: local static is initialized exactly once,
        // even in the presence of concurrent access
        static Logger logger;
        return logger;
    }

    void log(const std::string& message) {
        std::lock_guard<std::mutex> lock(logMutex_);
        ++logCount_;
        std::cout << "[Log #" << logCount_ << "] " << message << "\n";
    }

    int logCount() const {
        std::lock_guard<std::mutex> lock(logMutex_);
        return logCount_;
    }

private:
    Logger() : logCount_(0) {
        std::cout << "Logger initialized (thread: "
                  << std::this_thread::get_id() << ")\n";
    }
    ~Logger() = default;

    mutable std::mutex logMutex_;
    int logCount_;
};

// Demo: concurrent access from multiple threads
int main() {
    const int numThreads = 4;
    std::vector<std::thread> threads;

    for (int i = 0; i < numThreads; ++i) {
        threads.emplace_back([i]() {
            // Every thread gets the same instance
            Logger& logger = Logger::instance();
            logger.log("Message from thread " + std::to_string(i));
        });
    }

    for (auto& t : threads) t.join();

    std::cout << "\nTotal log count: "
              << Logger::instance().logCount() << "\n";

    return 0;
}

Explanation:

  • Meyer's Singleton leverages the C++11 guarantee that the initialization of function-local static variables is thread-safe.
  • The compiler generates invisible code to ensure the static Logger logger is constructed exactly once, even if multiple threads call instance() concurrently.
  • This approach is the simplest and most efficient: no explicit mutex needed for initialization. The compiler handles the synchronization.
  • The Singleton's own methods (log) still need their own mutex for thread safety if they modify shared state.
  • Construction is lazy (on first use) and destruction is automatic (at program exit).

Double-Checked Locking + std::call_once

Click to reveal Solution

#include <iostream>
#include <mutex>
#include <memory>
#include <thread>
#include <vector>

// Approach A: Double-Checked Locking (DCLP)
class ConfigManager {
public:
    static ConfigManager& instance() {
        // First check: no lock (fast path for already-initialized case)
        ConfigManager* tmp = instance_.load(std::memory_order_acquire);
        if (!tmp) {
            // Second check: with lock (only one thread enters here)
            std::lock_guard<std::mutex> lock(initMutex_);
            tmp = instance_.load(std::memory_order_relaxed);
            if (!tmp) {
                tmp = new ConfigManager();
                instance_.store(tmp, std::memory_order_release);
            }
        }
        return *tmp;
    }

    void set(const std::string& key, const std::string& value) {
        std::lock_guard<std::mutex> lock(dataMutex_);
        data_[key] = value;
    }

    std::string get(const std::string& key) const {
        std::lock_guard<std::mutex> lock(dataMutex_);
        auto it = data_.find(key);
        return (it != data_.end()) ? it->second : "";
    }

private:
    ConfigManager() { std::cout << "ConfigManager initialized\n"; }
    ~ConfigManager() = default;

    ConfigManager(const ConfigManager&) = delete;
    ConfigManager& operator=(const ConfigManager&) = delete;

    static std::atomic<ConfigManager*> instance_;
    static std::mutex initMutex_;
    mutable std::mutex dataMutex_;
    std::unordered_map<std::string, std::string> data_;
};

// Static member definitions
std::atomic<ConfigManager*> ConfigManager::instance_{nullptr};
std::mutex ConfigManager::initMutex_;

// Approach B: std::call_once (cleaner, recommended over manual DCLP)
class Database {
public:
    static Database& instance() {
        std::call_once(initFlag_, []() {
            instance_.reset(new Database());
        });
        return *instance_;
    }

    void connect() { std::cout << "Database connected\n"; }

private:
    Database() { std::cout << "Database initialized\n"; }
    ~Database() = default;

    Database(const Database&) = delete;
    Database& operator=(const Database&) = delete;

    static std::unique_ptr<Database> instance_;
    static std::once_flag initFlag_;
};

std::unique_ptr<Database> Database::instance_;
std::once_flag Database::initFlag_;

int main() {
    // DCLP approach
    ConfigManager::instance().set("host", "localhost");
    ConfigManager::instance().set("port", "5432");
    std::cout << "host = " << ConfigManager::instance().get("host") << "\n";

    // call_once approach
    Database::instance().connect();

    // Concurrent test
    std::vector<std::thread> threads;
    for (int i = 0; i < 8; ++i) {
        threads.emplace_back([i]() {
            ConfigManager::instance().set("thread", std::to_string(i));
        });
    }
    for (auto& t : threads) t.join();

    return 0;
}

Explanation:

  • DCLP uses an atomic pointer for the first check (no synchronization overhead on the fast path) and a mutex for the second check (only one thread creates the instance).
  • Memory ordering is critical: acquire on the first read and release on the store ensure all threads see the fully-constructed object.
  • std::call_once is the recommended C++11 approach for manual initialization: it guarantees the callable executes exactly once, regardless of how many threads call it.
  • Both approaches are useful when you cannot use local statics (e.g., when initialization depends on runtime parameters).

Comparison:

ApproachProsCons
Meyer's SingletonSimple, no manual sync, guaranteed by standardCannot control init order
Double-Checked LockingFast path avoids lock, explicit controlComplex, subtle memory ordering bugs
std::call_onceClean, standard, explicit controlSlightly more code than Meyer's

Variations to Try

  1. Singleton with parameter: Implement a singleton that accepts runtime parameters for initialization (e.g., a Logger with a configurable file path).
  2. Singleton destruction order: What happens if a Singleton's destructor references another Singleton that has already been destroyed? (The "dead reference" problem.)
  3. Singleton vs. dependency injection: Discuss why Singletons are often considered an anti-pattern and how dependency injection solves the same problems with better testability.
  4. Thread-safe lazy initialization: Implement a generic Lazy<T> wrapper that provides thread-safe lazy initialization for any type.

When to Avoid Singletons:

  • Hidden dependencies: Singletons create invisible coupling between components.
  • Testing difficulty: Global state makes unit tests unreliable (order-dependent).
  • Initialization order: In multi-threaded or dynamic loading scenarios, init order can be unpredictable.
  • Consider using dependency injection (pass references explicitly) instead of Singletons for better testability and modularity.
