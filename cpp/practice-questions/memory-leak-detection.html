Find and Fix Memory Leaks

Topics Covered: Memory Management, Smart Pointers, RAII

Problem Statement

Below are several code snippets containing common memory leak patterns. For each snippet:

  1. Identify the leak.
  2. Explain why it occurs.
  3. Fix it using proper C++ techniques (smart pointers, RAII, etc.).

Also demonstrate how to use Valgrind and AddressSanitizer to detect leaks at runtime.

Leak 1: Exception in Constructor

Click to reveal Solution

// LEAKY CODE:
class Buffer {
public:
    Buffer(std::size_t size) : size_(size), data_(new int[size]) {
        // If process() throws, data_ is never freed
        process();
    }
    ~Buffer() { delete[] data_; }
private:
    void process() { throw std::runtime_error("Init failed"); }
    std::size_t size_;
    int* data_;
};

// FIXED CODE:
class BufferFixed {
public:
    BufferFixed(std::size_t size)
        : data_(std::make_unique<int[]>(size)), size_(size) {
        // If process() throws, unique_ptr cleans up automatically
        process();
    }
private:
    void process() { throw std::runtime_error("Init failed"); }
    std::unique_ptr<int[]> data_;
    std::size_t size_;
};

// Alternative: use std::vector instead of raw array
class BufferVector {
public:
    BufferVector(std::size_t size) : data_(size), size_(size) {
        process();  // vector cleans up on throw
    }
private:
    void process() { throw std::runtime_error("Init failed"); }
    std::vector<int> data_;
    std::size_t size_;
};

Explanation:

  • In the leaky version, if process() throws, the destructor is never called (the object was never fully constructed). The raw new int[size] allocation is leaked.
  • With unique_ptr or vector, the member's destructor runs during stack unwinding, freeing the memory even if the constructor throws.

Leak 2: Copy Semantics with Raw Pointers

Click to reveal Solution

// LEAKY CODE: Shallow copy causes double-free
class StringHolder {
public:
    StringHolder(const char* s) {
        data_ = new char[strlen(s) + 1];
        strcpy(data_, s);
    }
    ~StringHolder() { delete[] data_; }  // DOUBLE FREE on copy!

private:
    char* data_;
    // Default copy constructor copies the pointer, not the string
};

// FIXED: Rule of Three
class StringHolderFixed {
public:
    StringHolderFixed(const char* s) {
        std::size_t len = strlen(s) + 1;
        data_ = new char[len];
        strcpy(data_, s);
    }

    ~StringHolderFixed() { delete[] data_; }

    // Custom copy constructor (deep copy)
    StringHolderFixed(const StringHolderFixed& other) {
        data_ = new char[strlen(other.data_) + 1];
        strcpy(data_, other.data_);
    }

    // Custom copy assignment
    StringHolderFixed& operator=(const StringHolderFixed& other) {
        if (this != &other) {
            char* newData = new char[strlen(other.data_) + 1];
            strcpy(newData, other.data_);
            delete[] data_;
            data_ = newData;
        }
        return *this;
    }
private:
    char* data_;
};

// BEST FIX: Use std::string (no manual memory management)
class StringHolderBest {
    std::string data_;
public:
    explicit StringHolderBest(const char* s) : data_(s) {}
    // Copy, move, destruction all handled by std::string
};

Explanation:

  • The leaky version has a compiler-generated copy constructor that does a shallow copy (copies the pointer). When both copies are destroyed, delete[] is called twice on the same address.
  • The Rule of Three (or Rule of Five) states: if you define a destructor, copy constructor, or copy assignment, you should define all three.
  • The best fix is to use std::string, which handles all memory management internally.

Leak 3: Container of Raw Pointers

Click to reveal Solution

// LEAKY CODE:
void processData() {
    std::vector<Widget*> widgets;
    for (int i = 0; i < 10; ++i) {
        widgets.push_back(new Widget(i));
    }
    // If we return early or throw, all Widgets leak
    // Even on normal exit, vector destructor doesn't delete pointers
}

// FIX 1: Manual cleanup (error-prone)
void processDataManual() {
    std::vector<Widget*> widgets;
    try {
        for (int i = 0; i < 10; ++i) {
            widgets.push_back(new Widget(i));
        }
        // ... use widgets ...
    } catch (...) {
        for (auto* w : widgets) delete w;
        throw;
    }
    for (auto* w : widgets) delete w;
}

// FIX 2: unique_ptr (recommended)
void processDataSmart() {
    std::vector<std::unique_ptr<Widget>> widgets;
    for (int i = 0; i < 10; ++i) {
        widgets.push_back(std::make_unique<Widget>(i));
    }
    // All Widgets freed when vector is destroyed,
    // even on early return or exception
}

Explanation:

  • A vector<T*> only frees the pointer values, not the objects they point to.
  • Manual cleanup is verbose and error-prone: you must remember to delete in every exit path (normal, exception, early return).
  • vector<unique_ptr<T>> is self-cleaning: the vector destructor destroys each unique_ptr, which deletes the object.

Follow-up Questions

Variations to Try

  1. Circular reference leak: Create two objects with shared_ptr pointing to each other. Show that they are never freed, then fix with weak_ptr.
  2. Virtual destructor leak: Show what happens when you delete a derived class object through a base class pointer that has no virtual destructor.
  3. Custom leak tracker: Implement a simple allocation tracker by overloading operator new and operator delete to log all allocations and detect mismatches.
  4. RAII wrapper: Create a generic ScopeGuard class that executes a cleanup function when it goes out of scope.

Key Takeaways:

  • The golden rule: every new must have a matching delete. Better yet, avoid new/delete entirely.
  • Use std::unique_ptr, std::shared_ptr, std::vector, and std::string for automatic memory management.
  • The Rule of Five (or Rule of Zero) prevents leaks from copy/move semantics.
  • Always run your code through AddressSanitizer or Valgrind during testing.
  • Exceptions are the most common trigger for leaks: ensure RAII cleans up during stack unwinding.
