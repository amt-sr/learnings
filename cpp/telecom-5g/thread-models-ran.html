Thread Models for RAN

The choice of threading model in a RAN (Radio Access Node) directly impacts throughput, latency, and scalability. Different models trade off simplicity against performance. Production gNB/vRAN systems use thread-per-core with CPU pinning and NUMA awareness for deterministic data-path processing.

Threading Models

ModelDescriptionProsCons
PipelineOne thread per protocol layer (PHY, MAC, RLC, PDCP)Simple, low contentionLatency accumulates across stages
Worker PoolPool of threads picks tasks from a shared queueLoad balancing, good for bursty loadContention on queue, cache bouncing
PartitionedUEs partitioned across cores, each core handles full stackBest cache locality, scalableImbalanced load per UE
Thread-Per-CoreDedicated thread pinned to each physical coreNo contention, deterministicComplex assignment, wasted cores

Thread-Per-Core Model (Production Choice)

  Thread-Per-Core with UE Partitioning:

  Core 0: Control thread (RRC, NGAP, system)
  Core 1: UE 1-100 (PHY -> MAC -> RLC -> PDCP -> F1-U)
  Core 2: UE 101-200 (PHY -> MAC -> RLC -> PDCP -> F1-U)
  Core 3: UE 201-300 (PHY -> MAC -> RLC -> PDCP -> F1-U)
  ...
  Core N: Timer, stats, monitoring

  Each core:
  - Pinned (sched_setaffinity)
  - Own memory pools (no sharing)
  - Own packet ring buffers (SPSC between cores)
  - No mutexes in data path

CPU Pinning and NUMA

  NUMA layout (typical dual-socket server):
  Socket 0 (NUMA 0):         Socket 1 (NUMA 1):
  Core 0  [Control]          Core 8  [Control backup]
  Core 1  [UE pool A]        Core 9  [UE pool C]
  Core 2  [UE pool A]        Core 10 [UE pool C]
  Core 3  [UE pool B]        Core 11 [UE pool D]
  Core 4  [UE pool B]        Core 12 [UE pool D]
  Core 5  [S1/X2/NG]         Core 13 [F1-U]
  Core 6  [F1-U]              Core 14 [Monitor]
  Core 7  [Monitor]           Core 15 [Reserved]

  Rule: UE data stays on same NUMA node across its lifetime.
  Inter-UE communication via rings (no shared memory).

NUMA penalty: A memory access to a remote NUMA node costs 40-80ns extra. In a tight PHY processing loop running billions of iterations, this overhead is devastating. Always keep UE context, buffers, and lookup tables on the local NUMA node.

C++ Thread Pool Skeleton

#include <thread>
#include <vector>
#include <atomic>
#include <functional>
#include <sched.h>
#include <unistd.h>

// Task for a worker thread
struct RanTask {
    std::function<void()> func;
    uint32_t ue_id;        // For partitioning
    uint8_t  priority;     // Task priority
};

// Partitioned RAN thread pool
class RanThreadPool {
public:
    explicit RanThreadPool(int num_workers)
        : running_(true), next_core_(0) {
        workers_.reserve(num_workers);
        for (int i = 0; i < num_workers; ++i) {
            workers_.emplace_back(&RanThreadPool::workerLoop, this, i);
        }
    }

    ~RanThreadPool() {
        running_.store(false);
        for (auto& w : workers_) {
            if (w.joinable()) w.join();
        }
    }

    // Submit task to specific worker (partitioned by UE)
    void submitToWorker(int worker_id, RanTask task) {
        if (worker_id < 0 ||
            worker_id >= static_cast<int>(workers_.size()))
            return;
        worker_queues_[worker_id].push(std::move(task));
    }

    // Submit to least-loaded worker (work-stealing fallback)
    void submit(RanTask task) {
        int id = next_core_.fetch_add(1) % workers_.size();
        submitToWorker(id, std::move(task));
    }

    [[nodiscard]] int numWorkers() const {
        return static_cast<int>(workers_.size());
    }

private:
    static constexpr int MAX_WORKERS = 32;
    std::vector<std::thread> workers_;
    std::atomic<bool> running_;
    std::atomic<int> next_core_;

    // Per-worker SPSC task queue (simplified; use lock-free ring)
    CircularTaskQueue worker_queues_[MAX_WORKERS];

    void workerLoop(int worker_id) {
        // Pin to specific core
        pinToCore(worker_id);

        // Set real-time priority
        setRtPriority(90 - worker_id);

        // Allocate per-core memory pool
        initWorkerMemoryPool(worker_id);

        while (running_.load(std::memory_order_relaxed)) {
            // Poll for tasks (busy-wait in data path)
            auto task = worker_queues_[worker_id].tryPop();
            if (task.has_value()) {
                task->func();
            }
            // Optional: brief sleep if no work
        }
    }

    static void pinToCore(int core_id) {
        cpu_set_t cpuset;
        CPU_ZERO(&cpuset);
        CPU_SET(core_id, &cpuset);
        pthread_setaffinity_np(pthread_self(),
                                sizeof(cpuset), &cpuset);
    }

    static void setRtPriority(int priority) {
        struct sched_param param{};
        param.sched_priority = priority;
        sched_setscheduler(0, SCHED_FIFO, &param);
    }

    void initWorkerMemoryPool(int) {}

    // Simplified task queue (use lock-free circular buffer)
    struct CircularTaskQueue {
        void push(RanTask task) {}
        std::optional<RanTask> tryPop() { return std::nullopt; }
    };
};
