CU-DU Split

The Central Unit (CU) / Distributed Unit (DU) split is a key innovation in 5G that decouples higher-layer protocol functions into a centralized CU and lower-layer real-time functions into a distributed DU. This split is connected via the F1 interface and enables centralized resource management with distributed radio processing.

Functional Split Options (3GPP TS 38.401)

The CU-DU split point determines which protocol functions reside in the CU versus the DU. Different split options trade off fronthaul bandwidth vs processing centralization:

Split OptionSplit PointCU FunctionsDU FunctionsFronthaul BW
Option 1RRC/PDCPRRCPDCP downVery Low
Option 2PDCP/RLCRRC, PDCPRLC, MAC, PHYLow
Option 3RLC (intra-PDCP)RRC, PDCP, RLC ARQRLC segmentation, MAC, PHYLow
Option 4RLC/MACRRC, PDCP, RLCMAC, PHYMedium
Option 5MAC (RLC header split)RRC, PDCP, RLC, MAC schedMAC HARQ, PHYMedium
Option 6MAC (HARQ/sched split)RRC, PDCP, RLC, MACMAC HARQ, PHYMedium
Option 7MAC/PHY (split 7-1/7-2/7-3)RRC, PDCP, RLC, MACPHY (partial)High
Option 8PHY (RF split)All L1+RF onlyVery High

Key Point: 3GPP standardized Option 2 (PDCP/RLC boundary) as the default CU-DU split. O-RAN uses a 7-2 split (between precoding and FFT/CP insertion) for the O-DU/O-RU boundary. See O-RAN Architecture for details.

C++ Split Point Configuration

#include <string>
#include <array>
#include <iostream>

// 3GPP TS 38.401 functional split options
enum class FunctionalSplit : uint8_t {
    SPLIT_1  = 1,  // RRC/PDCP boundary
    SPLIT_2  = 2,  // PDCP/RLC boundary (3GPP default)
    SPLIT_3  = 3,  // Intra-PDCP (higher PDCP / lower PDCP)
    SPLIT_4  = 4,  // RLC/MAC boundary
    SPLIT_5  = 5,  // Intra-MAC (RLC header)
    SPLIT_6  = 6,  // Intra-MAC (HARQ/scheduler)
    SPLIT_7a = 7,  // MAC/PHY (FFT/PRACH)
    SPLIT_7b = 8,  // MAC/PHY (precoding) -- O-RAN 7-2x
    SPLIT_7c = 9,  // MAC/PHY (CSI)
    SPLIT_8  = 10  // PHY/RF boundary
};

struct SplitCharacteristics {
    std::string name;
    std::string split_point;
    uint32_t    fronthaul_bw_mbps;  // Approximate per-cell
    uint32_t    latency_tolerance_us;
    bool        is_3gpp_standard;
};

class SplitOptionRegistry {
public:
    static constexpr size_t NUM_SPLITS = 10;

    void registerSplit(FunctionalSplit split,
                       const SplitCharacteristics& chars) {
        splits_[static_cast<size_t>(split)] = chars;
    }

    const SplitCharacteristics& get(FunctionalSplit split) const {
        return splits_[static_cast<size_t>(split)];
    }

    void printAll() const {
        const char* names[] = {
            "1: RRC/PDCP", "2: PDCP/RLC", "3: Intra-PDCP",
            "4: RLC/MAC",  "5: Intra-MAC-RLC",
            "6: Intra-MAC-HARQ", "7a: MAC/PHY-FFT",
            "7b: MAC/PHY-Precoder", "7c: MAC/PHY-CSI",
            "8: PHY/RF"
        };
        for (size_t i = 0; i < NUM_SPLITS; ++i) {
            std::cout << "Split " << names[i]
                      << " | BW: " << splits_[i].fronthaul_bw_mbps
                      << " Mbps\n";
        }
    }

private:
    std::array<SplitCharacteristics, NUM_SPLITS> splits_{};
};

// Factory function to create CU/DU based on split option
class ICentralUnit {
public:
    virtual ~ICentralUnit() = default;
    virtual void handleRrcMessage(const uint8_t* data, size_t len) = 0;
    virtual void handlePdcpPdu(const uint8_t* data, size_t len) = 0;
    virtual void sendToDu(const uint8_t* data, size_t len) = 0;
};

class IDistributedUnit {
public:
    virtual ~IDistributedUnit() = default;
    virtual void handleMacPdu(const uint8_t* data, size_t len) = 0;
    virtual void scheduleResources() = 0;
    virtual void sendToCu(const uint8_t* data, size_t len) = 0;
};

// Factory: creates CU and DU with correct protocol stacks
// based on selected split point
struct CuDuPair {
    std::unique_ptr<ICentralUnit>    cu;
    std::unique_ptr<IDistributedUnit> du;
};

CuDuPair createCuDu(FunctionalSplit split) {
    CuDuPair pair;
    switch (split) {
        case FunctionalSplit::SPLIT_2:
            // CU: RRC + PDCP, DU: RLC + MAC + PHY
            pair.cu = std::make_unique<CuSplit2>();
            pair.du = std::make_unique<DuSplit2>();
            break;
        case FunctionalSplit::SPLIT_7b:
            // CU: RRC + PDCP + RLC + MAC, DU: PHY (partial)
            pair.cu = std::make_unique<CuSplit7b>();
            pair.du = std::make_unique<DuSplit7b>();
            break;
        default:
            throw std::runtime_error("Unsupported split option");
    }
    return pair;
}

CU-CP and CU-UP Separation

The CU itself is further split into a control plane part (CU-CP) and a user plane part (CU-UP), connected via the E1 interface:

               +----------+          +----------+
               |  CU-CP   |--- E1 -->|  CU-UP   |
               | (RRC,    |          | (SDAP,   |
               |  F1-C)   |<-- E1 ---|  PDCP,   |
               +----+-----+          |  F1-U)   |
                    |                +-----+----+
                 F1-C |                    | F1-U
                    |                    |
               +----+----+          +----+----+
               |   DU    |          |   DU    |
               | (RLC,   |          | (RLC,   |
               |  MAC,   |          |  MAC,   |
               |  sched) |          |  sched) |
               +---------+          +---------+

Real-Time Consideration: Lower split options (closer to Option 2) require less fronthaul bandwidth but shift more processing to the DU. Higher split options (closer to Option 8) increase fronthaul load but centralize more processing. The choice impacts CPU allocation and thread model design.
