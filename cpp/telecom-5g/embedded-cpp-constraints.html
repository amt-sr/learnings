Embedded C++ Constraints

Telecom embedded systems (BBU, RRH, ASIC firmware) operate under strict constraints that differ from general-purpose software. Determinism, limited resources, and safety requirements force developers to use a constrained subset of C++ features. This topic covers what is allowed and what is restricted in telecom embedded code.

No Exceptions in Telecom

Exceptions banned: Exception handling in C++ requires unwinding the stack, which involves non-deterministic memory allocation and execution time. In real-time telecom code, a thrown exception could miss a deadline. All telecom coding standards (MISRA, CERT, JSF) either ban or strongly discourage exceptions in the data path.

// BAD: Using exceptions in data path
void processData(uint8_t* data, size_t len) {
    if (len < MIN_SIZE) throw std::runtime_error("Too short");
    if (len > MAX_SIZE) throw std::length_error("Too long");
    // ...
}

// GOOD: Return error codes or use std::expected (C++23)
// or error codes with constexpr
enum class ErrorCode : uint8_t {
    SUCCESS = 0,
    BUFFER_TOO_SHORT,
    BUFFER_TOO_LONG,
    INVALID_PARAMETER
};

ErrorCode processData(uint8_t* data, size_t len) {
    if (len < MIN_SIZE) return ErrorCode::BUFFER_TOO_SHORT;
    if (len > MAX_SIZE) return ErrorCode::BUFFER_TOO_LONG;
    // ... processing ...
    return ErrorCode::SUCCESS;
}

// C++23 alternative (if available):
// std::expected<void, ErrorCode> processData(uint8_t*, size_t);

Limited STL Usage

Standard library containers may use heap allocation internally. In the data path, you must either avoid them or use them with custom allocators backed by pre-allocated memory pools.

No Dynamic Allocation in Data Path

// Data path buffer handling (CORRECT):
// All buffers come from pre-allocated pools
class DataPathProcessor {
public:
    // Initialize with pre-allocated buffers (at startup)
    void init(MemoryPool& pool) {
        // Reserve fixed work buffers
        work_buffer_ = pool.allocate();
        output_buffer_ = pool.allocate();
    }

    // Process: all allocation already done
    ErrorCode process(const uint8_t* input, size_t len) {
        // Use pre-allocated buffers only
        memcpy(work_buffer_->data, input, len);
        work_buffer_->length = len;

        // Process in-place or to output buffer
        transform(work_buffer_, output_buffer_);

        return ErrorCode::SUCCESS;
    }

private:
    PacketBuffer* work_buffer_{nullptr};
    PacketBuffer* output_buffer_{nullptr};

    void transform(PacketBuffer* src, PacketBuffer* dst) {}
};

Static analysis: In production telecom code, static analysis tools (cppcheck, clang-tidy, Coverity, Polyspace) are run as part of CI/CD. These tools catch violations of coding standards, memory leaks, undefined behavior, and concurrency bugs before code reaches production.

Deterministic Behavior

// Deterministic loop: fixed iteration count
void processSlot(const uint8_t* symbols, size_t n_symbols) {
    for (size_t i = 0; i < n_symbols; ++i) {  // Fixed bound
        processSymbol(symbols + i * SYMBOL_SIZE);
    }
}

// Non-deterministic: variable iteration
void processData(const uint8_t* data) {
    while (*data != 0xFF) {  // Unpredictable termination
        data++;
    }
}

Key principle: In real-time telecom, the execution time of every function in the data path must be bounded and predictable. Avoid: unbounded loops, dynamic allocation, recursive functions (stack overflow risk), and system calls that may block.
