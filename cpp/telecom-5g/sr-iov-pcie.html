SR-IOV & PCIe

SR-IOV (Single Root I/O Virtualization) is a PCIe standard that allows a single physical NIC to present multiple virtual NICs to the host or virtual machines. In vRAN and O-RAN, SR-IOV enables multiple virtual DU functions or containerized RAN components to share physical NIC hardware with near-line-rate performance and isolation.

Physical Functions vs Virtual Functions

  SR-IOV Architecture:

  +------------------------------------------+
  | Physical NIC (PCIe Device)              |
  |  +------------------------------------+  |
  |  | Physical Function (PF)               |  |
  |  | - Full configuration control        |  |
  |  | - Creates/manages VFs               |  |
  |  | - Used by hypervisor/host            |  |
  |  +------------------------------------+  |
  |  +--------+ +--------+ +--------+      |  |
  |  | VF 0   | | VF 1   | | VF 2   | ...  |  |
  |  | (light)| | (light)| | (light)|      |  |
  |  +--------+ +--------+ +--------+      |  |
  +------------------------------------------+

  PF = Physical Function (full NIC, manages VFs)
  VF = Virtual Function (lightweight NIC, data only)
  - Each VF has its own MAC, queues, MSI-X vectors
  - VFs appear as independent PCIe devices
  - Can be assigned to VMs or processes directly
  - Typical: 1 PF + 64-256 VFs per NIC

PCIe TLP (Transaction Layer Packet)

  PCIe Data Path:
  CPU/VM -> Memory Write TLP -> NIC -> DMA Read/Write TLP -> Memory

  TLP types relevant to networking:
  - Memory Write (MWr): CPU sends packet data to NIC
  - Memory Read (MRead): NIC reads descriptor from host
  - Completion (Cpl): NIC returns completed status

  DMA (Direct Memory Access):
  - NIC reads packet descriptors from host memory (doorbell)
  - NIC writes received packets directly to host memory (rx buffers)
  - Zero-copy: data stays in user-space buffers (with DPDK + hugepages)

SR-IOV in Telecom (vRAN/O-RAN)

Use CaseHow SR-IOV Helps
vRAN DU virtualizationMultiple VFs assigned to different DU VMs/containers
O-RAN RU-DU splitVF for eCPRI fronthaul, PF for management
5G UPFMultiple VF groups for different slices/networks
Multi-tenant RANEach tenant gets dedicated VFs with bandwidth limits

Key advantage: SR-IOV bypasses the hypervisor for data traffic. VMs/containers access VFs directly via DMA, eliminating the software switching overhead. This provides near bare-metal throughput while maintaining isolation between tenants.

C++ SR-IOV Configuration

#include <fstream>
#include <string>
#include <vector>
#include <iostream>

class SrIovConfigurator {
public:
    struct VfInfo {
        int vf_id;
        int pci_bus;
        int pci_slot;
        int pci_func;
        std::string mac;
        int vlan_id;
        uint32_t max_tx_rate;  // Mbps (0 = unlimited)
    };

    // Enable SR-IOV and create VFs
    bool enableVfs(const std::string& pf_pci_addr, int num_vfs) {
        // Enable SR-IOV via sysfs
        std::ofstream sriov_numvfs(
            "/sys/bus/pci/devices/" + pf_pci_addr
            + "/sriov_numvfs");
        if (!sriov_numvfs.is_open()) return false;

        // Set number of VFs (must be 0 before changing)
        sriov_numvfs << "0";
        sriov_numvfs.flush();
        sriov_numvfs << std::to_string(num_vfs);

        return sriov_numvfs.good();
    }

    // Set VF MAC address
    bool setVfMac(const std::string& pf_pci_addr,
                  int vf_id, const std::string& mac) {
        std::ofstream vf_mac(
            "/sys/bus/pci/devices/" + pf_pci_addr
            + "/virtfn" + std::to_string(vf_id) + "/mac");
        if (!vf_mac.is_open()) return false;
        vf_mac << mac;
        return vf_mac.good();
    }

    // Set VF max TX rate (in Mbps)
    bool setVfRate(const std::string& pf_pci_addr,
                   int vf_id, uint32_t rate_mbps) {
        std::ofstream vf_rate(
            "/sys/bus/pci/devices/" + pf_pci_addr
            + "/virtfn" + std::to_string(vf_id) + "/max_tx_rate");
        if (!vf_rate.is_open()) return false;
        vf_rate << std::to_string(rate_mbps * 1000); // in Kbps
        return vf_rate.good();
    }

    // Get VF PCI addresses
    std::vector<std::string> getVfPciAddrs(
            const std::string& pf_pci_addr) {
        std::vector<std::string> vfs;
        std::ifstream virtfn_list(
            "/sys/bus/pci/devices/" + pf_pci_addr
            + "/virtfn?/pci_address");
        std::string line;
        while (std::getline(virtfn_list, line)) {
            vfs.push_back(line);
        }
        return vfs;
    }
};

// Example: Setup vRAN NIC with 4 VFs
void setupVranNic() {
    SrIovConfigurator cfg;
    std::string pf_addr = "0000:81:00.0";

    // Enable 4 VFs
    cfg.enableVfs(pf_addr, 4);

    // Configure each VF
    for (int i = 0; i < 4; ++i) {
        cfg.setVfMac(pf_addr, i,
            "02:00:00:00:00:" + std::to_string(i + 1));
        cfg.setVfRate(pf_addr, i, 10000);  // 10 Gbps each
    }

    // Get VF PCI addresses for DPDK binding
    auto vfs = cfg.getVfPciAddrs(pf_addr);
    for (const auto& vf : vfs) {
        // Bind VFs to DPDK vfio-pci driver:
        // dpdk-devbind.py --bind=vfio-pci <vf_addr>
    }
}

DPDK with SR-IOV

DPDK + SR-IOV: Bind VFs to the vfio-pci driver (not igb_uio for security). DPDK uses each VF as an independent port with its own RX/TX queues. The PF manages overall NIC settings (link state, global statistics) while each VF processes packets independently.

  DPDK SR-IOV setup:
  # 1. Load vfio-pci driver
  modprobe vfio-pci

  # 2. Enable IOMMU (required for VFIO)
  intel_iommu=on in kernel boot params

  # 3. Bind VFs to vfio-pci
  dpdk-devbind.py --bind=vfio-pci 0000:81:00.1  # VF 0
  dpdk-devbind.py --bind=vfio-pci 0000:81:00.2  # VF 1

  # 4. In DPDK app, VFs appear as regular ports:
  # rte_eth_dev_configure(port=0, ...)  -> uses VF 0
  # rte_eth_dev_configure(port=1, ...)  -> uses VF 1
