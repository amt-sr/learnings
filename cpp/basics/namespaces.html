1.2.6 Namespaces

Namespaces provide a way to group related declarations (classes, functions, variables) under a named scope. They prevent name collisions when different libraries or modules define entities with the same name. Every name in the C++ standard library lives inside the std namespace.

What Namespaces Are

A namespace is a declarative region that provides a scope to the identifiers inside it. You have already been using the std namespace since the first Hello World program:

#include <iostream>

// Defining a custom namespace
namespace math {
    constexpr double PI = 3.14159265358979;

    double square(double x) {
        return x * x;
    }

    double circle_area(double radius) {
        return PI * square(radius);
    }
}

int main() {
    // Use fully qualified name
    std::cout << "PI = " << math::PI << std::endl;
    std::cout << "square(5) = " << math::square(5.0) << std::endl;
    std::cout << "circle_area(3) = " << math::circle_area(3.0) << std::endl;

    return 0;
}

Using Directive vs Using Declaration

C++ provides two mechanisms to avoid repeating the namespace prefix. They differ significantly in scope:

Featureusing namespace (directive)using (declaration)
Syntaxusing namespace math;using math::square;
ScopeImports all names from the namespaceImports only one specific name
Collision riskHigh — all names are visibleLow — only the selected name is visible
RecommendedRarely — only in small .cpp filesYes — preferred approach
#include <iostream>
#include <vector>

namespace util {
    void print(const std::string& msg) {
        std::cout << "[util] " << msg << std::endl;
    }
}

namespace logging {
    void print(const std::string& msg) {
        std::cout << "[log] " << msg << std::endl;
    }
}

int main() {
    // Using declaration: import only what you need
    using util::print;
    print("Hello");       // calls util::print

    // Qualified call: always unambiguous
    logging::print("World");  // calls logging::print

    // using namespace util;  // DANGEROUS: would hide logging::print
    // using namespace logging;  // DANGEROUS: conflicts with util::print

    return 0;
}

Best practice: Prefer using declarations for specific names over using namespace directives. This makes it clear exactly which names you are importing and avoids unexpected collisions.

Nested Namespaces

Namespaces can be nested to organize code hierarchically. C++17 introduced a cleaner syntax for nested namespaces:

// C++11 style (pre-C++17)
namespace telecom {
    namespace ran {
        namespace cu {
            class CellConfig {
            public:
                int cell_id;
                double frequency;
            };
        }
    }
}

// C++17 style (much cleaner)
namespace telecom::ran::du {
    class PrachConfig {
    public:
        int preamble;
        double power;
    };
}

int main() {
    telecom::ran::cu::CellConfig cu_config{101, 3.5e9};
    telecom::ran::du::PrachConfig du_config{64, -80.0};

    // Using declaration for a deeply nested type
    using PrachConfig = telecom::ran::du::PrachConfig;
    PrachConfig prach{64, -85.0};

    return 0;
}

Anonymous Namespaces

An anonymous (unnamed) namespace restricts visibility to the current translation unit (the .cpp file). It is the C++ equivalent of C's static for file-level linkage:

// config.cpp
#include <iostream>

namespace {
    // These are only visible within this file
    const int MAX_RETRIES = 3;
    const std::string DEFAULT_HOST = "localhost";

    void log_internal(const std::string& msg) {
        std::cout << "[internal] " << msg << std::endl;
    }
}

void connect() {
    log_internal("Connecting to " + DEFAULT_HOST);
    // MAX_RETRIES is accessible here
    for (int i = 0; i < MAX_RETRIES; ++i) {
        // retry logic
    }
}

Tip: Use anonymous namespaces in .cpp files for helper functions and constants that should not be visible outside the file. This provides better encapsulation than making them static and works for types as well (anonymous namespaces can contain classes, whereas static cannot be applied to class definitions).

Namespace Aliasing

Namespace aliases create short names for long namespace paths, improving readability:

#include <iostream>

namespace company {
    namespace project {
        namespace subsystem {
            void process() {
                std::cout << "Processing..." << std::endl;
            }
        }
    }
}

// Create a short alias
namespace cps = company::project::subsystem;

int main() {
    cps::process();  // much shorter than the full path

    // Aliases are also useful for std library types
    using StringVec = std::vector<std::string>;

    return 0;
}

Why using namespace std in Headers is Bad

This is one of the most important namespace rules in C++:

Never put using namespace std; (or any using namespace directive) in a header file. A header file is included into every translation unit that uses it. A using namespace directive in a header pollutes the global namespace of every file that includes it, causing potentially hard-to-debug name collisions.

// BAD: header file with using directive
// utils.h
#pragma once
using namespace std;   // NEVER DO THIS in a header

void print(const string& msg);  // 'string' pollutes every includer's namespace


// GOOD: header file with explicit qualification
// utils.h
#pragma once
#include <string>

void print(const std::string& msg);  // explicit and safe


// Acceptable: using directive in a .cpp file
// utils.cpp
#include "utils.h"
#include <iostream>

using namespace std;  // OK in .cpp (still not ideal but acceptable)

void print(const string& msg) {
    cout << msg << endl;
}

Example of collision: Imagine your project defines a function called distance and includes a third-party header that has using namespace std;. If std::distance leaks into the global namespace, it could silently shadow or conflict with your distance function. These bugs are extremely difficult to track down.

Namespaces and Classes

Namespaces and classes both provide scoping, but they serve different purposes:

FeatureNamespaceClass
PurposeGroup related entities, prevent name collisionsDefine a type with data and behavior
Can containClasses, functions, variables, typedefsMember variables, member functions, typedefs
InstanceCannot create instancesCan create objects
Access controlNo private/protected (all public)Yes: public, protected, private
InheritanceNoYes
Closing braceNo semicolon neededSemicolon required
