1.3.1 Stack vs Heap Memory

C++ gives you direct control over memory allocation. Understanding the difference between the stack and the heap is fundamental to writing correct, efficient, and bug-free C++ programs. Every variable, object, and data structure lives in one of these two regions.

Stack Memory

The stack is a region of memory that grows and shrinks automatically as functions are called and return. It follows a Last-In, First-Out (LIFO) discipline. Local variables, function parameters, and return addresses are all stored on the stack.

  • Automatic management: Memory is allocated when a function is entered and deallocated when it returns. No manual cleanup needed.
  • Fast: Allocation is just incrementing a stack pointer (a single CPU instruction).
  • Contiguous: Stack memory is always contiguous, providing excellent cache locality.
  • Limited size: Typically 1–8 MB depending on the OS and compiler settings.
  • Deterministic lifetime: Variables are destroyed in reverse order of construction.
#include <iostream>

void process() {
    int local_var = 42;        // Allocated on stack when process() is called
    int arr[1000];             // 4000 bytes on stack (sizeof(int) * 1000)
    std::cout << "local_var = " << local_var << std::endl;
    // local_var and arr are deallocated when process() returns
}

int main() {
    int a = 10;                // On main's stack frame
    int b = 20;                // On main's stack frame
    process();
    // a and b still valid here
    std::cout << "a = " << a << ", b = " << b << std::endl;
    return 0;
}

Heap Memory

The heap is a large pool of memory available for dynamic allocation at runtime. Unlike the stack, the heap is not automatically managed — you control when memory is allocated and freed.

  • Dynamic allocation: Memory is allocated with new and freed with delete (or via smart pointers).
  • Slower: The allocator must search for a free block of the right size and update bookkeeping data structures.
  • Large: Limited only by available virtual memory (gigabytes on modern systems).
  • Fragmentation: Repeated allocation and deallocation can fragment free memory over time.
  • Manual lifetime: You must free memory explicitly, or use RAII/smart pointers.
#include <iostream>

int main() {
    // Allocate a single int on the heap
    int* ptr = new int{42};
    std::cout << "*ptr = " << *ptr << std::endl;
    delete ptr;               // Must free manually

    // Allocate an array on the heap
    int* arr = new int[1000];
    for (int i = 0; i < 1000; ++i) {
        arr[i] = i * i;
    }
    delete[] arr;             // Use delete[] for arrays

    return 0;
}

Warning: Forgetting to delete heap memory causes a memory leak. The memory remains allocated but unreachable, gradually consuming all available RAM. In long-running applications (servers, daemons), memory leaks are critical bugs. See new/delete for proper techniques and smart pointers for automatic cleanup.

Process Memory Layout

Both stack and heap occupy specific regions in a process's virtual address space. Understanding this layout clarifies why pointers are dangerous and why stack overflows occur.

Low Addresses (0x0)
+-------------------+
|       Text        |  Program code (instructions)     -- read-only
|      Segment      |
+-------------------+
|    Initialized    |  Global/static variables with
|      Data         |  explicit initial values          -- read-write
+-------------------+
|       BSS         |  Uninitialized globals/statics   -- zero-initialized
|     Segment       |
+-------------------+
|                   |
|       Heap        |  Grows upward toward stack       -- new/malloc
|        |          |     (brk/sbrk or mmap)
|        v          |
+-------------------+
|                   |
|      (free)       |
|                   |
+-------------------+
|        ^          |
|        |          |
|      Stack        |  Grows downward toward heap       -- local vars
|                   |
+-------------------+
High Addresses

Key insight: The stack and heap grow toward each other. On most platforms the stack grows downward (from high to low addresses) and the heap grows upward (from low to high). When they meet, the program is out of memory. See Memory Layout for a deeper look at each segment.

Stack Overflow

A stack overflow occurs when the stack exceeds its allocated size. This typically happens due to:

  • Infinite or deep recursion: Each recursive call adds a new stack frame. Too many nested calls exhaust the stack.
  • Very large local variables: Allocating large arrays on the stack (e.g., int arr[1000000];) can immediately overflow.
  • Excessive call depth: Deeply nested function calls without proper base cases.
#include <iostream>

// Infinite recursion: causes stack overflow
void infinite_recursion(int n) {
    std::cout << "n = " << n << std::endl;
    infinite_recursion(n + 1);  // No base case!
    // Run with: ulimit -s unlimited  (Unix) to increase stack
}

// Large stack allocation: can cause overflow
void large_stack_array() {
    // 10 million ints = ~40 MB -- may exceed stack size
    int huge[10'000'000];
    huge[0] = 1;  // May crash before reaching this line
}

int main() {
    // Uncomment to test (will crash):
    // infinite_recursion(0);
    // large_stack_array();
    std::cout << "Program completed safely." << std::endl;
    return 0;
}

Note: On Linux, check stack size with ulimit -s (typically 8192 KB). Increase it with ulimit -s unlimited. On Windows, stack size is set in the linker settings. However, increasing stack size is a band-aid — fix the root cause (deep recursion or large locals).

When to Use Stack vs Heap

Rule of thumb: Always prefer the stack. Only use the heap when you must. The stack is faster, safer, and automatic.

Use the stack when:

  • The object's lifetime is tied to a function scope.
  • The object is small (a few bytes to a few kilobytes).
  • You do not need to share ownership across functions.
  • Performance is critical and allocation overhead matters.

Use the heap when:

  • The object is large (megabytes of data, large arrays, images).
  • The object must outlive the function that creates it.
  • The size is not known at compile time (dynamic arrays).
  • You need polymorphic behavior with virtual functions (objects stored via base pointers).
  • You need shared ownership across multiple parts of the program.
#include <iostream>
#include <memory>   // for smart pointers
#include <vector>

// Stack: appropriate for small, scope-bound objects
void use_stack() {
    int counter = 0;
    double result = 3.14;
    char buffer[256];
    // All cleaned up automatically when function returns
}

// Heap: needed for large, dynamic, or shared data
void use_heap() {
    // Large array: heap
    int* big_data = new int[1'000'000];
    // ... use big_data ...
    delete[] big_data;

    // Smart pointer (recommended): heap with automatic cleanup
    auto ptr = std::make_shared<int>(100);

    // STL container (internally uses heap): stack variable, heap data
    std::vector<int> nums;
    nums.reserve(1'000'000);  // Allocates on heap internally
}

int main() {
    use_stack();
    use_heap();
    std::cout << "Done." << std::endl;
    return 0;
}

Common Pitfalls

#include <iostream>

int* dangling_pointer() {
    int x = 42;       // x lives on the stack
    return &x;        // WARNING: returns address of a stack variable!
    // x is destroyed here -- the pointer is now dangling
}

int* correct_version() {
    int* p = new int{42};  // Allocated on heap
    return p;              // Safe: caller must delete this
}

int main() {
    int* bad = dangling_pointer();
    std::cout << *bad << std::endl;  // Undefined behavior!

    int* good = correct_version();
    std::cout << *good << std::endl; // OK: prints 42
    delete good;                  // Must free heap memory

    return 0;
}

Danger: Never return a pointer or reference to a local (stack) variable. The variable is destroyed when the function returns, leaving a dangling pointer. Accessing it is undefined behavior — it may appear to work sometimes but can crash unpredictably.
