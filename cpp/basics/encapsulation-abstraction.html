1.2.5 Encapsulation & Abstraction

Encapsulation and abstraction are two fundamental principles of object-oriented design. Encapsulation bundles data and methods into a single unit while restricting direct access to internal details. Abstraction hides complex implementation details behind a simple interface. Together, they make code more maintainable, robust, and easier to reason about.

What is Encapsulation

Encapsulation is the practice of keeping data (member variables) private or protected and providing controlled access through public member functions. This prevents external code from putting the object into an invalid state:

#include <iostream>
#include <string>

class BankAccount {
private:                    // data is hidden
    std::string owner;
    double balance;
    int account_number;

public:                     // behavior is exposed
    BankAccount(const std::string& name, int acc_num, double initial)
        : owner(name), account_number(acc_num), balance(initial) {
        if (balance < 0) balance = 0;
    }

    // Controlled access: no one can set a negative balance
    void deposit(double amount) {
        if (amount > 0) {
            balance += amount;
        }
    }

    bool withdraw(double amount) {
        if (amount > 0 && amount <= balance) {
            balance -= amount;
            return true;
        }
        return false;
    }

    double get_balance() const { return balance; }
    std::string get_owner() const { return owner; }
};

int main() {
    BankAccount acc{"Alice", 12345, 1000.0};

    acc.deposit(500.0);
    bool ok = acc.withdraw(200.0);

    // acc.balance = -9999;  // ERROR: private, cannot access directly
    // acc.balance += 1000000;  // ERROR: no direct manipulation possible

    std::cout << acc.get_owner() << ": $" << acc.get_balance() << std::endl;

    return 0;
}

Benefit: The BankAccount class enforces invariants: the balance can never go negative through its public interface. If the internal representation changes (e.g., tracking transactions), the public interface remains unchanged.

Access LevelWho Can AccessTypical Use

protectedThe class and its derived classesDetails needed by derived classes

class Device {
private:
    int internal_id;          // only Device can access
    void internal_reset() {}  // internal helper

protected:
    bool is_active;           // derived classes can access
    void log_status() {}      // shared utility for derived classes

public:
    void start() { is_active = true; internal_reset(); }
    void stop()  { is_active = false; }
};

Getters and Setters

Getters and setters provide controlled read and write access to private member variables. They allow you to add validation, logging, or compute values on the fly:

#include <iostream>

class Temperature {
private:
    double celsius;  // internal: always stored in Celsius

public:
    // Constructor with validation
    Temperature(double c) : celsius(c) {
        // Absolute zero is -273.15 C
        if (celsius < -273.15) celsius = -273.15;
    }

    // Getter: read access
    double get_celsius() const { return celsius; }
    double get_fahrenheit() const { return celsius * 9.0 / 5.0 + 32.0; }
    double get_kelvin() const { return celsius + 273.15; }

    // Setter: write access with validation
    void set_celsius(double c) {
        if (c >= -273.15) celsius = c;
    }

    void set_fahrenheit(double f) {
        double c = (f - 32.0) * 5.0 / 9.0;
        if (c >= -273.15) celsius = c;
    }

    void display() const {
        std::cout << celsius << " C / "
                  << get_fahrenheit() << " F / "
                  << get_kelvin() << " K" << std::endl;
    }
};

int main() {
    Temperature t{100.0};
    t.display();  // 100 C / 212 F / 373.15 K

    t.set_fahrenheit(32.0);
    t.display();  // 0 C / 32 F / 273.15 K

    t.set_celsius(-300.0);  // rejected (below absolute zero)
    t.display();            // still 0 C (validation prevented change)

    return 0;
}

Avoid trivial getters/setters: If a getter or setter does nothing except return or set a value without validation or computation, consider whether the member should simply be public. However, in practice, trivial accessors are common because they preserve the ability to add validation later without changing the interface.

Abstraction vs Encapsulation

While often confused, encapsulation and abstraction address different concerns:

AspectEncapsulationAbstraction
GoalProtect data from unauthorized accessHide complexity behind a simple interface
Mechanismprivate/protected access + getters/settersAbstract classes, interfaces, pure virtual functions
FocusData integrity and invariantsSimplifying how clients use a component
AnalogyA safe with a locked doorA steering wheel that hides the engine

Interface vs Implementation

Abstraction separates the interface (what a class does) from the implementation (how it does it). Clients of a class only need to know the public interface:

#include <iostream>
#include <vector>
#include <memory>

// Interface (abstract class)
class DataStore {
public:
    virtual void store(const std::string& key, const std::string& value) = 0;
    virtual std::string retrieve(const std::string& key) const = 0;
    virtual ~DataStore() = default;
};

// Implementation 1: in-memory map
class InMemoryStore : public DataStore {
    // Private implementation details hidden from the user
    std::vector<std::pair<std::string, std::string>> data;
public:
    void store(const std::string& key, const std::string& value) override {
        data.push_back({key, value});
    }
    std::string retrieve(const std::string& key) const override {
        for (const auto& p : data) {
            if (p.first == key) return p.second;
        }
        return "";
    }
};

// Implementation 2: could be a file-based store, database, etc.
// class FileStore : public DataStore { ... };

int main() {
    // User code depends only on the interface
    DataStore* store = new InMemoryStore{};

    store->store("name", "Alice");
    store->store("role", "Engineer");

    std::cout << "name: " << store->retrieve("name") << std::endl;
    std::cout << "role: " << store->retrieve("role") << std::endl;

    delete store;
    return 0;
}

Benefit: The DataStore interface lets you swap implementations (in-memory, file, database) without changing client code. This is a key advantage of abstraction for testability and maintainability.

Practical Guidelines

  • Make all member variables private by default. Use protected only when derived classes genuinely need direct access.
  • Expose only the minimum necessary interface through public methods.
  • Validate inputs in setters and constructors to enforce invariants.
  • Prefer const member functions for any method that does not modify the object's state.
  • Use namespaces to group related classes and prevent name collisions at a higher level.
  • Use friend functions only when two classes are tightly coupled and need direct access to each other's internals.
