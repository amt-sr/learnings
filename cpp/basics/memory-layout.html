1.3.3 Memory Layout

When a C++ program runs, the operating system allocates it a virtual address space divided into several segments. Each segment has a specific purpose, permissions, and behavior. Understanding this layout is crucial for debugging pointer errors, optimizing performance, and avoiding security vulnerabilities.

Text Segment (Code Segment)

The text segment contains the compiled machine code of your program — all the CPU instructions that the program executes. This segment is typically read-only to prevent the program from accidentally modifying its own code.

#include <iostream>

void say_hello() {
    std::cout << "Hello!" << std::endl;
    // The body of this function lives in the text segment
    // as compiled CPU instructions
}

int main() {
    say_hello();  // CPU jumps to the text segment to execute
    return 0;
}

Security: Modifying the text segment (e.g., via buffer overflow) is a classic attack vector. Modern OSes enforce W^X (Write XOR Execute) — memory is either writable or executable, never both. This prevents injecting and running shellcode.

Initialized Data Segment

This segment holds global and static variables that are explicitly initialized to a non-zero value. They persist for the entire lifetime of the program.

#include <iostream>

// Initialized Data: lives in the initialized data segment
int global_counter = 42;
double pi = 3.14159265;
const int max_size = 1024;

void demo() {
    // Static local variable: also in initialized data segment
    static int call_count = 0;
    call_count++;
    std::cout << "Called " << call_count << " times" << std::endl;
}

int main() {
    std::cout << "global_counter = " << global_counter << std::endl;
    demo();
    demo();
    demo();
    return 0;
}

BSS Segment (Block Started by Symbol)

The BSS segment stores global and static variables that are not explicitly initialized (or initialized to zero). The OS zeros this entire segment at program startup, which is more efficient than storing zeros in the executable file.

#include <iostream>

// BSS segment: zero-initialized by the OS at startup
int uninit_global;           // Zero-initialized to 0
static int static_var;       // Zero-initialized to 0
int big_array[10000];        // 40 KB zeroed at startup (no space in .exe)
int zero_init = 0;          // Also in BSS (initialized to zero)

int main() {
    std::cout << "uninit_global = " << uninit_global << std::endl;  // 0
    std::cout << "static_var   = " << static_var << std::endl;    // 0
    std::cout << "zero_init    = " << zero_init << std::endl;      // 0
    return 0;
}

Why BSS matters: A variable like int big_array[1000000] would take 4 MB of space in the executable if stored in initialized data. By placing it in BSS, the OS only needs to store a note ("zero 4 MB at this address"), and the actual zeroing happens at load time. This keeps binary sizes small.

Heap Segment

The heap is used for dynamic memory allocation via new (C++) or malloc (C). It grows upward from the BSS segment toward the stack. The C runtime manages heap allocation using algorithms that track free and occupied blocks.

#include <iostream>
#include <vector>

int main() {
    // These allocations live in the heap
    int* p1 = new int{10};
    int* p2 = new int{20};

    // STL containers internally use the heap for storage
    std::vector<int> vec = {1, 2, 3, 4, 5};
    // The vector object itself may be on the stack,
    // but its internal array of 5 ints is on the heap

    std::cout << "Heap ptr p1: " << p1 << std::endl;
    std::cout << "Heap ptr p2: " << p2 << std::endl;
    std::cout << "Vector data: " << vec.data() << std::endl;

    delete p1;
    delete p2;

    return 0;
}

Stack Segment

The stack holds local variables, function parameters, return addresses, and other per-call data. Each function call creates a stack frame. The stack grows downward (toward lower addresses) on most architectures.

#include <iostream>

int add(int a, int b) {
    // a and b are on this function's stack frame
    int result = a + b;
    return result;  // result is copied to caller, then frame destroyed
}

int main() {
    int x = 10;      // On main's stack frame
    int y = 20;      // On main's stack frame
    int sum = add(x, y);  // After add() returns, its frame is popped

    std::cout << x << " + " << y << " = " << sum << std::endl;
    return 0;
}

Why This Matters for Pointers

Understanding the memory layout is essential for writing correct pointer code. The most common bugs — dangling pointers, buffer overflows, and use-after-free — all stem from misunderstanding where data lives and how long it persists.

#include <iostream>

int global = 10;  // Data segment -- lives forever

int* dangerous() {
    int local = 20;    // Stack -- dies when function returns
    return &local;      // Dangling pointer! Points to dead stack memory
}

int* safe() {
    static int persistent = 30;  // Data segment -- lives forever
    return &persistent;         // Safe: variable persists
}

int* also_safe() {
    return new int{40};  // Heap -- lives until explicitly deleted
    // Caller MUST delete this!
}

int main() {
    std::cout << global << std::endl;              // 10 (OK: data segment)
    // std::cout << *dangerous() << std::endl;     // BUG: dangling pointer

    int* p1 = safe();
    std::cout << *p1 << std::endl;                  // 30 (OK: static storage)

    int* p2 = also_safe();
    std::cout << *p2 << std::endl;                  // 40 (OK: heap)
    delete p2;                                       // Must free

    return 0;
}

Remember: Pointers do not know which segment their target lives in. A pointer is just an address. It is up to the programmer to ensure the pointed-to memory is still valid when accessed. See Pointers for pointer operations and Stack vs Heap for lifetime rules.

Inspecting Memory Layout

#include <iostream>

int global_init = 1;       // Initialized data
int global_uninit;          // BSS

int main() {
    int local = 2;           // Stack
    int* heap = new int{3};  // Heap

    std::cout << "Text segment:  " << (void*)&main << std::endl;
    std::cout << "Init data:    " << (void*)&global_init << std::endl;
    std::cout << "BSS:          " << (void*)&global_uninit << std::endl;
    std::cout << "Heap:         " << (void*)heap << std::endl;
    std::cout << "Stack:        " << (void*)&local << std::endl;

    delete heap;
    return 0;
}
