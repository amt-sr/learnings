1.3.2 new / delete

C++ provides the new and delete operators for allocating and deallocating memory on the heap at runtime. While modern C++ encourages smart pointers over raw new/delete, understanding how these operators work is essential for grasping C++ memory management.

Allocating a Single Object

Use new to allocate a single object on the heap. It returns a pointer to the allocated memory. Use delete to free the memory when done.

#include <iostream>

int main() {
    // Allocate a single int on the heap, initialized to 42
    int* p = new int{42};
    std::cout << "Value: " << *p << std::endl;   // 42
    std::cout << "Address: " << p << std::endl;   // e.g., 0x55a3b4

    // Free the memory
    delete p;
    p = nullptr;  // Good practice: avoid dangling pointer

    return 0;
}

Initialization with new

new supports several initialization styles:

#include <iostream>
#include <string>

int main() {
    // Default initialization (indeterminate value for built-in types)
    int* p1 = new int;            // *p1 is uninitialized

    // Value initialization (zero-initialized for built-in types)
    int* p2 = new int();          // *p2 == 0
    int* p3 = new int{};         // *p3 == 0 (uniform init, C++11)

    // Direct initialization
    int* p4 = new int{100};      // *p4 == 100

    // Works with any type, including user-defined classes
    std::string* s = new std::string{"Hello"};
    std::cout << *s << std::endl;   // Hello

    delete p1;
    delete p2;
    delete p3;
    delete p4;
    delete s;

    return 0;
}

Tip: Always initialize heap memory at the point of allocation. Uninitialized int, double, etc. contain garbage values. Use new int{0} or new int() to zero-initialize.

Allocating Arrays: new[] and delete[]

To allocate an array of objects on the heap, use new[]. You must use delete[] (with brackets) to free it. Mismatching new[] with delete (without brackets) is undefined behavior.

#include <iostream>

int main() {
    // Allocate an array of 10 ints on the heap
    int* arr = new int[10];     // Elements are uninitialized

    // Initialize
    for (int i = 0; i < 10; ++i) {
        arr[i] = i * i;
    }

    // Print
    for (int i = 0; i < 10; ++i) {
        std::cout << arr[i] << " ";
    }
    std::cout << std::endl;
    // Output: 0 1 4 9 16 25 36 49 64 81

    // MUST use delete[] for arrays
    delete[] arr;

    // Value-initialized array
    double* scores = new double[5]{};  // All elements initialized to 0.0
    delete[] scores;

    return 0;
}

Critical rule: Always match allocation with the correct deallocation:
new ↔ delete (single object)
new[] ↔ delete[] (array)
Mismatching them causes undefined behavior — the program may crash, corrupt the heap, or appear to work while silently corrupting memory.

Memory Leaks

A memory leak occurs when heap-allocated memory is no longer referenced by any pointer but has not been freed. The memory remains allocated for the lifetime of the process, gradually consuming available RAM.

#include <iostream>

void leak_example() {
    int* p = new int{100};  // Allocated on heap
    // Function returns without delete -- memory is leaked!
    // p goes out of scope, but the int on the heap remains
}

void double_leak() {
    int* p1 = new int{10};
    int* p2 = new int{20};
    p1 = p2;  // Original int{10} is now leaked (no pointer to it)
    delete p2; // Only frees int{20}
}

void correct_cleanup() {
    int* p = new int{100};
    // ... use p ...
    delete p;    // Properly freed
    p = nullptr; // Avoid dangling pointer
}

int main() {
    leak_example();     // Leaks 4 bytes each call
    double_leak();      // Leaks 4 bytes
    correct_cleanup();  // No leak

    std::cout << "Leaks not visible here -- use tools like Valgrind." << std::endl;
    return 0;
}

Detecting leaks: Use Valgrind on Linux (valgrind --leak-check=full ./program), AddressSanitizer (-fsanitize=address), or Visual Studio's Diagnostic Tools. See Memory Leak Detection practice questions for exercises.

Handling Allocation Failure

If new cannot allocate the requested memory (e.g., out of memory), it throws std::bad_alloc by default. Use nothrow to get nullptr instead.

#include <iostream>
#include <new>      // for std::nothrow

int main() {
    // Default: throws std::bad_alloc on failure
    try {
        int* huge = new int[1'000'000'000'000];  // Very large
        delete[] huge;
    } catch (const std::bad_alloc& e) {
        std::cout << "Allocation failed: " << e.what() << std::endl;
    }

    // Nothrow version: returns nullptr on failure
    int* p = new (std::nothrow) int[1'000'000'000'000];
    if (p == nullptr) {
        std::cout << "Allocation failed (nothrow)" << std::endl;
    } else {
        delete[] p;
    }

    return 0;
}

Smart Pointers: The Modern Alternative

Raw new/delete is error-prone. Modern C++ strongly prefers smart pointers, which automatically free memory when it is no longer needed.

#include <iostream>
#include <memory>   // unique_ptr, shared_ptr

int main() {
    // unique_ptr: sole ownership, automatically deleted
    std::unique_ptr<int> up = std::make_unique<int>(42);
    std::cout << *up << std::endl;  // 42
    // No delete needed -- freed when 'up' goes out of scope

    // shared_ptr: shared ownership, freed when last reference dies
    std::shared_ptr<int> sp1 = std::make_shared<int>(100);
    std::shared_ptr<int> sp2 = sp1;  // Both point to same int
    std::cout << *sp1 << " " << *sp2 << std::endl;  // 100 100
    // Freed when sp1 and sp2 both go out of scope

    return 0;
}

Guideline: In modern C++ code, avoid raw new and delete entirely. Use std::make_unique() (C++14) or std::make_shared() (C++11) instead. See Smart Pointers and RAII for the modern approach to resource management.

Best Practices Summary

  1. Prefer stack allocation over heap when possible.
  2. Use std::make_unique and std::make_shared instead of raw new.
  3. If you must use raw new, pair it with delete in the same scope or use RAII.
  4. Always initialize pointers to nullptr after delete.
  5. Never mix new/delete[] or new[]/delete.
  6. Use std::vector instead of new[] for dynamic arrays.
  7. Run memory-checking tools regularly in development.
