1.1.8 References

A reference is an alias for another variable. Once a reference is bound to a variable, it acts as an alternative name for that same variable. Unlike pointers, references cannot be reassigned to refer to a different variable and cannot be null.

Declaration Syntax

A reference is declared by placing an & after the type. It must be initialized at the point of declaration:

#include <iostream>

int main() {
    int x = 10;
    int& ref = x;  // ref is now an alias for x

    std::cout << "x = " << x << std::endl;    // 10
    std::cout << "ref = " << ref << std::endl;  // 10

    ref = 20;  // this modifies x
    std::cout << "x after modification = " << x << std::endl;  // 20

    x = 30;    // this also changes ref
    std::cout << "ref after modification = " << ref << std::endl;  // 30

    std::cout << "&x == &ref: " << (&x == &ref ? "true" : "false") << std::endl;  // true

    return 0;
}

Warning: A reference must be initialized when declared. There is no such thing as an uninitialized reference. Writing int& ref; without initialization is a compile-time error.

References vs Pointers

References and pointers both provide indirect access to variables, but they differ in important ways:

AspectReferencePointer
InitializationMust be initialized at declarationCan be declared uninitialized (dangerous)
ReassignmentCannot be reassignedCan point to different variables
Null valueCannot be nullCan be nullptr
SyntaxUse directly: ref = 10Dereference to access: *ptr = 10
Address&ref gives the original variable's address&ptr gives the pointer's own address
SafetySafer — always refers to a valid objectCan be dangling, wild, or null
ImplementationTypically implemented as a pointer internallyStores the address directly
Sizeofsizeof(ref) == sizeof(T)sizeof(ptr) == sizeof(T*) (typically 8 bytes)

Rule of thumb: Prefer references over pointers when possible. References provide cleaner syntax and stronger safety guarantees. Use pointers when you need nullable values, reassignment, or pointer arithmetic (e.g., with arrays).

Pass by Reference vs Pass by Value

When passing arguments to functions, you have three main choices:

#include <iostream>
#include <string>

// Pass by value: a copy is made (safe but potentially slow for large objects)
void by_value(std::string s) {
    s += " (modified)";
    std::cout << "Inside by_value: " << s << std::endl;
} // s is destroyed; original is unchanged

// Pass by reference: no copy, original can be modified
void by_ref(std::string& s) {
    s += " (modified)";
    std::cout << "Inside by_ref: " << s << std::endl;
}

// Pass by const reference: no copy, original cannot be modified
void by_const_ref(const std::string& s) {
    std::cout << "Inside by_const_ref: " << s << std::endl;
    // s += " (modified)";  // ERROR: s is const
}

int main() {
    std::string msg = "Hello";

    by_value(msg);
    std::cout << "After by_value: " << msg << std::endl;
    // Output: "Hello" (unchanged)

    by_ref(msg);
    std::cout << "After by_ref: " << msg << std::endl;
    // Output: "Hello (modified)"

    by_const_ref(msg);

    return 0;
}

Best practice: For non-primitive types (strings, vectors, structs, classes), pass by const& to avoid expensive copies while preventing modification. For small types (int, double, bool), pass by value. Use non-const references only when the function needs to modify the argument.

Const References

A const reference prevents modification of the referenced object through the reference. It also allows binding to temporary values (rvalues):

#include <iostream>

int main() {
    int x = 10;
    const int& cref = x;       // cannot modify x through cref
    // cref = 20;               // ERROR: cannot assign to const reference

    // Const references can bind to temporaries (rvalues)
    const int& r = 42;          // OK: binds to a temporary int
    const double& d = 3.14;     // OK: binds to a temporary double

    std::cout << "r = " << r << ", d = " << d << std::endl;

    return 0;
}

Note: A non-const reference cannot bind to a temporary. int& r = 42; is a compile error because 42 is an rvalue. Only const int& can extend the lifetime of a temporary. See Move Semantics for more on rvalue references.

Reference as Function Return Type

Returning a reference from a function allows callers to modify the returned object directly. This is commonly used for operator overloading and chained calls:

#include <iostream>

struct Score {
    int value;
};

// Return a reference to allow modification
int& get_value(Score& s) {
    return s.value;  // returns a reference to s.value
}

// Returning a const reference prevents modification
const int& peek_value(const Score& s) {
    return s.value;
}

int main() {
    Score s{100};

    get_value(s) = 200;  // modifies s.value through the returned reference
    std::cout << "s.value = " << s.value << std::endl;  // 200

    // peek_value(s) = 300;  // ERROR: returning const ref prevents assignment

    return 0;
}

Danger: Never return a reference to a local variable. The local is destroyed when the function returns, making the reference a dangling reference:

int& bad_function() {
    int local = 42;
    return local;  // WARNING: dangling reference!
} // local is destroyed; returned reference is invalid

References and Polymorphism

References preserve polymorphic behavior when working with derived classes, just like pointers:

#include <iostream>

class Animal {
public:
    virtual void speak() const { std::cout << "..." << std::endl; }
};

class Dog : public Animal {
public:
    void speak() const override { std::cout << "Woof!" << std::endl; }
};

void make_speak(const Animal& animal) {
    animal.speak();  // calls the correct override based on actual type
}

int main() {
    Dog dog;
    make_speak(dog);  // Output: "Woof!"
    return 0;
}
